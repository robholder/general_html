Hacking With Swift - 118 - Moonshot: Wrap up

You’ve now finished Moonshot, which is the first of our projects that started to get difficult - it took longer to explain.

We used custom SwiftUI layouts, and I even snuck in a few advanced Swift features. Not all our future projects are going to be difficult, but it’s certainly going to be the case that they are more complex; which means the resulting apps are more interesting, and more representative of real-world app building.

As the complexity rises so do the odds of making mistakes, and Swift is pretty unforgiving. Even one small mistake on line 20 can result in a random error appearing on line 5, and this can be disheartening.

Well, I hope today’s quote will inspire you. I’ve chosen it specially for today for reasons best left to the reader, but it’s this: don’t panic! These sorts of problems are common, and the easiest way to solve them right now is to comment out whatever code you most recently added, and keep doing so until your code works. You can then slowly re-introduce code until you find the part that causes your compile to break, then fix it.

So, don’t panic - you can do this!

Challenge

One of the best ways to learn is to write your own code as often as possible, so here are three ways you should try extending this app to make sure you fully understand what’s going on.

  • Add the launch date to MissionView, below the mission badge. You might choose to format this differently given that more space is available, but it’s down to you.
  • Extract one or two pieces of view code into their own new SwiftUI views - the horizontal scroll view in MissionView is a great candidate, but if you followed my styling then you could also move the Rectangle dividers out too.
  • For a tough challenge, add a toolbar item to ContentView that toggles between showing missions as a grid and as a list.

Tip: For that last one, your best bet is to make all your grid code and all your list code two separate views, and switch between them using an if condition in ContentView. You can’t attach SwiftUI modifiers to an if condition, but you can wrap that condition in a Group then attach modifiers to there, like this: 

Group {
    if showingGrid {
        GridLayout(astronauts: astronauts, missions: missions)
    } else {
        ListLayout(astronauts: astronauts, missions: missions)
    }
}
.navigationTitle("My Group")

You might hit some speed bumps trying to style your list, because they have a particular look and feel on iOS by default. Try attaching .listStyle(.plain) to your list, then something like .listRowBackground(Color.darkBackground) to the contents of your list row - that should get you a long way towards your goal.

AI Background

Moving code into a custom view is one of the best ways to keep your ContentView from becoming a "wall of text." In SwiftUI, we call this Refactoring.

To do this, you create a new struct that conforms to the View protocol.

1. The Custom View Struct

You can place this code either at the bottom of your current file or in a brand-new Swift file. 

import SwiftUI

struct CustomDivider: View {
    var body: some View {
        Rectangle()
            .frame(height: 2)
            .foregroundStyle(.lightBackground)
            .padding(.vertical)
    }
}

2. How to use it

Now, instead of writing those four lines of code over and over, you just "call" the struct like a regular SwiftUI component: 

VStack {
    Text("Mission Description")
    
    CustomDivider() // This replaces your old code!
    
    Text(mission.description)
}

3. Why this is powerful

  • Consistency: If you decide later that the divider should be 3 points thick instead of 2, you only have to change it in one place (the CustomDivider struct), and every screen in your app will update automatically.
  • Readability: Your main view logic becomes much easier to read because it describes what is on the screen (a divider) rather than how to draw a rectangle.

4. Advanced: Customizing on the fly

If you want to be able to change the color or padding occasionally, you can add properties to your struct: 

struct CustomDivider: View {
    let color: Color // Allow the caller to pick a color

    var body: some View {
        Rectangle()
            .frame(height: 2)
            .foregroundStyle(color) // Use the property here
            .padding(.vertical)
    }
}

// Usage:
CustomDivider(color: .red)
CustomDivider(color: .lightBackground)

More Complex Custom View Example

This is the perfect candidate for a custom view. When a view gets nested deeply - like a NavigationLink inside a ForEach inside an HStack inside a ScrollView - it becomes very hard to see the "forest for the trees."

To extract this, you need to decide what data the new view needs to do its job. In this case, it needs the entire CrewMember object.

1. The New Custom View: CrewView

You can create a new file or add this to the bottom of your current one. Notice how we pass in crewMember as a property. 

struct CrewView: View {
    // This is the 'argument' the view needs to function
    let crewMember: MissionView.CrewMember

    var body: some View {
        HStack {
            Image(crewMember.astronaut.id)
                .resizable()
                .frame(width: 104, height: 72)
                .clipShape(.capsule)
                .overlay(
                    Capsule()
                        .strokeBorder(.white, lineWidth: 1)
                )

            VStack(alignment: .leading) {
                Text(crewMember.astronaut.name)
                    .foregroundStyle(.white)
                    .font(.headline)
                Text(crewMember.role)
                    .foregroundStyle(.white.opacity(0.5))
            }
        }
        .padding(.horizontal)
    }
}

2. The Updated (Clean) Usage

Now, look at how much more readable your MissionView becomes. You’ve separated the layout (ScrollView/HStack) from the content (the individual crew row). 

ScrollView(.horizontal, showsIndicators: false) {
    HStack {
        ForEach(crew, id: \.role) { crewMember in
            NavigationLink {
                AstronautView(astronaut: crewMember.astronaut)
            } label: {
                // We just pass the current member into our new view
                CrewView(crewMember: crewMember)
            }
        }
    }
}

3. Why this specific "Argument" choice?

You might ask: "Why not pass in the name, image, and role as separate strings?"

  • Passing the whole object: CrewView(crewMember: crewMember) is simpler. If you later decide the CrewView should also show the astronaut's description, you don't have to change the arguments; the view already has access to the whole astronaut struct inside the crewMember.
  • Namespace Tip: If you defined CrewMember inside MissionView, you refer to its type as MissionView.CrewMember.

4. Handling the Preview

When you extract a view that requires an argument, the #Preview block will complain because it doesn't have a CrewMember to display. You’ll need to "mock" one: 

#Preview {
    // We have to load the data just like we did for the other previews
    let astronauts: [String: Astronaut] = Bundle.main.decode("astronauts.json")
    let missions: [Mission] = Bundle.main.decode("missions.json")
    
    // Create a single mock crew member for the preview
    let tempCrew = MissionView.CrewMember(role: "Commander", astronaut: astronauts["aldrin"]!)

    return CrewView(crewMember: tempCrew)
        .background(.darkBackground) // Add background so white text is visible
}

The "Aha!" Moment

By doing this, you've turned a 20-line block of code into a single, reusable component. If you decide to add a "Crew List" to a different part of the app later, you can just use CrewView again!

MissionView.swift

import SwiftUI

struct MissionView: View {
    
    struct CrewMember {
        let role: String
        let astronaut: Astronaut
    }
    
    let mission: Mission
    let crew: [CrewMember] // Combined data
    
    var body: some View {
        ScrollView {
            VStack {
                Image(mission.image)
                    .resizable()
                    .scaledToFit()
                    .containerRelativeFrame(.horizontal) { width, axis in
                        width * 0.6
                    }
                    .padding(.top)
                Text("Launch: \(mission.longLaunchDate)")
                    .padding(10)
                
                VStack(alignment: .leading) {
                    
                    // Custom divider
                    CustomDivider()
                    
                    Text("Mission Highlights")
                        .font(.title.bold())
                        .padding(.bottom, 5)
                    
                    Text(mission.description)
                    
                    // Custom divider
                    CustomDivider()
                    
                    Text("Crew")
                        .font(.title.bold())
                        .padding(.bottom, 5)
                    
                }
                .padding(.horizontal)
                
                ScrollView(.horizontal, showsIndicators: false) {
                    HStack {
                        ForEach(crew, id: \.role) { crewMember in
                            NavigationLink {
                                AstronautView(astronaut: crewMember.astronaut)
                            } label: {
                                // We just pass the current member into our new view
                                CrewView(crewMember: crewMember)
                            }
                        }
                    }
                }
            }
            .padding(.bottom)
        }
        .navigationTitle(mission.displayName)
        .navigationBarTitleDisplayMode(.inline)
        .background(.darkBackground)
    }
    init(mission: Mission, astronauts: [String: Astronaut]) {
        self.mission = mission
        
        self.crew = mission.crew.map { member in
            if let astronaut = astronauts[member.name] {
                return CrewMember(role: member.role, astronaut: astronaut)
            } else {
                fatalError("missing \(member.name)")
            }
        }
    }
}

Challenge 3: Allow User to Alternate between Grid & List Views

This is a significant refactor, but it follows the same "conveyor belt" and "packaging" logic we discussed earlier. You are essentially taking that big block of ScrollView code and moving it into its own specialized "box" (a subview), then creating a second box for the List version.

1. Create the GridLayout Subview

First, we move your existing ScrollView logic out of ContentView. Notice that we have to pass in astronauts and missions as arguments so the view has data to work with. 

struct GridLayout: View {
    let astronauts: [String: Astronaut]
    let missions: [Mission]

    let columns = [
        GridItem(.adaptive(minimum: 150))
    ]

    var body: some View {
        ScrollView {
            LazyVGrid(columns: columns) {
                ForEach(missions) { mission in
                    NavigationLink {
                        MissionView(mission: mission, astronauts: astronauts)
                    } label: {
                        VStack {
                            Image(mission.image)
                                .resizable()
                                .scaledToFit()
                                .frame(width: 100, height: 100)
                                .padding()

                            VStack {
                                Text(mission.displayName)
                                    .font(.headline)
                                    .foregroundStyle(.white)
                                Text(mission.formattedLaunchDate)
                                    .font(.caption)
                                    .foregroundStyle(.white.opacity(0.8))
                            }
                            .padding(.vertical)
                            .frame(maxWidth: .infinity)
                            .background(.lightBackground)
                        }
                        .clipShape(.rect(cornerRadius: 10))
                        .overlay(
                            RoundedRectangle(cornerRadius: 10)
                                .stroke(.lightBackground)
                        )
                    }
                }
            }
            .padding([.horizontal, .bottom])
        }
    }
}

2. Create the ListLayout Subview

For the "List" version, we use a standard List (or a LazyVStack inside a ScrollView). This is usually much simpler code: 

struct ListLayout: View {
    let astronauts: [String: Astronaut]
    let missions: [Mission]

    var body: some View {
        List(missions) { mission in
            NavigationLink {
                MissionView(mission: mission, astronauts: astronauts)
            } label: {
                HStack {
                    Image(mission.image)
                        .resizable()
                        .scaledToFit()
                        .frame(width: 50, height: 50)
                        .padding(.trailing, 5)

                    VStack(alignment: .leading) {
                        Text(mission.displayName)
                            .font(.headline)
                        Text(mission.formattedLaunchDate)
                            .font(.caption)
                    }
                }
            }
            .listRowBackground(Color.darkBackground) // Keep it dark
        }
        .listStyle(.plain)
    }
}

3. Putting it all together in ContentView

Now we add the @State variable to track which view to show and a .toolbar button to flip that switch. 

struct ContentView: View {
    let astronauts: [String: Astronaut] = Bundle.main.decode("astronauts.json")
    let missions: [Mission] = Bundle.main.decode("missions.json")

    // 1. The Toggle State
    @State private var showingGrid = true

    var body: some View {
        NavigationStack {
            // 2. The Conditional Switch
            Group {
                if showingGrid {
                    GridLayout(astronauts: astronauts, missions: missions)
                } else {
                    ListLayout(astronauts: astronauts, missions: missions)
                }
            }
            .navigationTitle("Moonshot")
            .background(.darkBackground)
            .preferredColorScheme(.dark)
            // 3. The Toolbar Button
            .toolbar {
                Button(showingGrid ? "Show List" : "Show Grid") {
                    showingGrid.toggle()
                }
            }
        }
    }
}

Group

The reason we put that if/else inside a Group is that SwiftUI's body property expects to return one single thing. Without the Group, Swift sees two possible branches and gets "Type Confusion." The Group acts like a single envelope containing two different letters; Swift is happy just holding the envelope.

What changed?

  • Encapsulation: ContentView no longer cares how the missions are drawn; it only cares which layout is active.
  • The Toolbar: We used a ternary operator showingGrid ? "Show List" : "Show Grid" inside the button label so the button text changes depending on what the user sees.
  • Group: The Group is necessary here because if/else logic inside a View can confuse Swift unless it's wrapped in a container that can hold "either/or."
