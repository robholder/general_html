Challenge 2 - Color Code Values

AI Reply to a Question - Is a Ternary Conditional advisable for three conditions?

For three or more distinct states, a nested ternary usually becomes a "code smell"—it's hard to read, easy to mess up with parentheses, and a nightmare to debug later.

Since you are working in SwiftUI, here are the three best ways to handle this, ranked from "clean" to "expert."

1. The "Computed Property" (Best for Clean Code)

Instead of cramming logic inside your body view, move it into a computed property. This keeps your view code readable and allows you to use standard if/else logic. 

var amountColor: Color {
    if amount <= 10 {
        return .green
    } else if amount < 100 {
        return .orange
    } else {
        return .red
    }
}

// In your view:
Text(amount, format: .currency(code: "USD"))
    .foregroundStyle(amountColor)

1. What does the Color word mean?

In the line var amountColor: Color {, the word Color is the Return Type.

It is telling the Swift compiler: "I am creating a variable named amountColor, and I promise that whenever you ask for its value, you will receive a piece of data that is a SwiftUI Color object."

  • If you tried to return 5 (an Int): Xcode would give you an error because 5 isn't a Color.
  • If you tried to return "Red" (a String): Xcode would complain because a word isn't a Color object.

Because you explicitly told Swift that this property is a Color, you can use it anywhere a color is expected, like .foregroundStyle(amountColor).

2. The switch Statement (Most Robust)

If you want to be very precise, a switch with ranges is technically the most "Swift-like" way to handle numerical boundaries. 

var amountColor: Color {
    switch amount {
    case ..<10:       // Everything up to 10
        return .green
    case 10..<100:    // 10 to 99.99
        return .orange
    default:          // 100 and above
        return .red
    }
}

3. The "Nested Ternary" (The "Quick & Dirty" Way)

If you absolutely must do it inline, this is how it looks. Notice how quickly the readability drops: 

Text("$\(amount)")
    .foregroundStyle(amount <= 10 ? .green : (amount < 100 ? .orange : .red))

Why this is risky: If you forget one colon or parenthesis, Xcode’s error messages for complex ternaries are notoriously vague, often just saying "Failed to produce diagnostic in reasonable time."

Comparison of Logic Mechanisms

Mechanism Readability Scalability Best Use Case
Ternary Low Poor Simple "A or B" choices.
If/Else Property High Good 3-4 conditions based on logic.
Switch Statement Excellent Excellent Complex ranges or many categories.

Pro-Tip: Extensions

If you plan on using this "Currency Color" logic in multiple places in your app, you can add it as an extension to Double: 

extension Double {
    var currencyStatusColor: Color {
        if self <= 10 { return .green }
        if self < 100 { return .orange }
        return .red
    }
}

// Now you can just use:
Text("$\(amount)").foregroundStyle(amount.currencyStatusColor)

The Power (and Danger) of Extensions

If you add an extension to Double in your project, every single Double in your entire app now has that property available to it. 

extension Double {
    var currencyStatusColor: Color {
        self < 10 ? .green : .red
    }
}

// Now look at the impact:
let myPrice: Double = 5.0
let temp: Double = 98.6
let batteryLevel: Double = 0.5

print(myPrice.currencyStatusColor)     // Works (and makes sense)
print(temp.currencyStatusColor)        // Works (but is weird/nonsensical)
print(0.99.currencyStatusColor)        // Works (even on literal numbers!)

Is this a good thing?

  • The Pro: It makes your code very "clean" and readable. amount.currencyStatusColor is easy to understand.
  • The Con: It "pollutes" the Double type. Every time you type a period after a number, that property will show up in Xcode’s autocomplete list, even if you are working on something completely unrelated to currency.

The "Best Practice" Middle Ground

If you want the convenience of an extension without "polluting" the entire Double type everywhere, you have two choices:

  • Keep it local: Just use the computed property inside your View (like we did in the previous step). It only exists inside that one screen.
  • Use a more specific name: Instead of just .color, name it .amountColor or .expenseTierColor. This makes it clear that this property is specifically for your currency logic.

Summary

  • Color is the "Contract"—it defines what the property actually is.
  • Extension is a "Global Upgrade"—it gives a new superpower to every instance of that type in your code.

Complications in this App

Since we are iterating through a list of ExpenseItem objects, the logic needs to know which item it’s looking at. You could add a function inside the View, but there is an even more "Swift-like" way to do this given your setup.

You have three solid options based on where you want the logic to live.

Option 1: The "Data-Driven" Way (Best for Architecture)

Since the color depends entirely on the amount inside the ExpenseItem, you can put the logic directly inside the ExpenseItem struct. This keeps the data and its styling rules together. 

struct ExpenseItem: Identifiable, Codable {
    var id = UUID()
    let name: String
    let type: String
    let amount: Double
    
    // Add this computed property here
    var color: Color {
        if amount <= 10 {
            return .green
        } else if amount < 100 {
            return .orange
        } else {
            return .red
        }
    }
}

In your ContentView: 

Text(item.amount, format: .currency(code: Locale.current.currency?.identifier ?? "USD"))
    .foregroundStyle(item.color) // Simple and clean!

Option 2: The View Function

If you prefer to keep your Data Struct "pure" (only holding data, no UI colors), you can write a helper function inside your ContentView.

Inside your ContentView struct: 

func color(for amount: Double) -> Color {
    switch amount {
    case ..<10: return .green
    case 10..<100: return .orange
    default: return .red
    }
}

In your body: 

Text(item.amount, format: .currency(code: ...))
    .foregroundStyle(color(for: item.amount))

Option 3: The "Local View" Method

If you only ever plan to use this color logic inside this one List, you can define a small helper variable inside the ForEach loop. 

ForEach(expenses.items) { item in
    // Define a local constant for the color right here
    let amountColor: Color = if item.amount <= 10 {
        .green
    } else if item.amount < 100 {
        .orange
    } else {
        .red
    }

    HStack {
        // ... (your other code)
        Text(item.amount, format: .currency(code: ...))
            .foregroundStyle(amountColor)
    }
}

Which one should you choose?

Method Recommendation Why?
Option 1 (Struct) Highly Recommended It makes the most sense. An expense "knows" its own status. If you ever show these expenses in a different view, the color logic goes with them automatically.
Option 2 (Function) Good Best if you feel that a Data Struct shouldn't know about SwiftUI Color types (useful if you were sharing the code with a non-UI framework).
Option 3 (Inline) Least Favorite It clutters your body code. SwiftUI views get long very quickly; keeping the logic out of the body is usually better for your sanity!

A Quick Tip on Color and Codable

If you choose Option 1, you might notice that ExpenseItem is Codable. Since Color is not easily codable, making it a computed property (using var color: Color { ... }) is perfect because computed properties are ignored by the JSON Encoder. It won't mess up your UserDefaults saving!

Where Should They Be Placed?

Here is where the logic should live. Think of the area above the body as the "brain" and the body itself as the "skeleton." 

struct ContentView: View {
    // 1. State Variables (The Data)
    @State private var amount: Double = 55.0

    // 2. Computed Properties (The Logic / "The Brain")
    // This stays inside the struct, but outside the body.
    var amountColor: Color {
        switch amount {
        case ..<10:
            return .green
        case 10..<100:
            return .orange
        default:
            return .red
        }
    }

    // 3. The Body (The Layout / "The Skeleton")
    var body: some View {
        VStack {
            Text(amount, format: .currency(code: "USD"))
                .font(.largeTitle)
                .foregroundStyle(amountColor) // Using the logic here
        }
    }
}

Why this is the preferred spot:

  • Readability: Your body stays clean. Instead of seeing a messy switch or if/else block inside your UI layout, you just see a descriptive name like amountColor.
  • Performance: SwiftUI is highly optimized for computed properties. Every time amount changes, SwiftUI re-calculates amountColor and updates only the parts of the view that need it.
  • Encapsulation: By keeping the logic inside the ContentView struct, the logic is "privately owned" by this view. It doesn't clutter up the rest of your app.

A Note on "The Gap"

You can technically put these properties above or below the body view, as long as they are inside the struct curly braces. However, the standard convention is:

  • @State variables at the top.
  • Computed properties next.
  • The body view last.

When would it go OUTSIDE the struct?

Only if you want that logic to be shared across many different views. In that case, you would move it into an extension (like the Double extension I mentioned earlier) or a separate ViewModel class. But for a single screen, keeping it right above the body is the "Sweet Spot."

Challenge 3 - Separate Expense Type Lists

This is a classic architectural pivot in SwiftUI. While you could create two separate arrays in your Expenses class, that often leads to "data desync" (where you delete from one list but forget to update the other).

The most elegant way to handle this—and the way Apple usually recommends—is to use computed properties to filter your single "source of truth" array into two views.

The "Filtered View" Approach

You keep your one items array in the Expenses class, but in your ContentView, you create two "virtual" lists.

1. Create the Filtered Properties

Add these inside your ContentView struct (right above the body). They will automatically re-run whenever expenses.items changes. 

var personalExpenses: [ExpenseItem] {
    expenses.items.filter { $0.type == "Personal" }
}

var businessExpenses: [ExpenseItem] {
    expenses.items.filter { $0.type == "Business" }
}

2. Update the List with Section

Now, instead of one big ForEach, you use Section views. This gives you the nice iOS-native headers for free. 

List {
    Section("Personal") {
        ForEach(personalExpenses) { item in
            ExpenseRow(item: item) // Let's clean this up (see below)
        }
        .onDelete { offsets in
            removeItems(at: offsets, in: personalExpenses)
        }
    }

    Section("Business") {
        ForEach(businessExpenses) { item in
            ExpenseRow(item: item)
        }
        .onDelete { offsets in
            removeItems(at: offsets, in: businessExpenses)
        }
    }
}

The "Deletion" Challenge

When you use filtered arrays, onDelete gives you the index of the item in that filtered list, not the main array. If you delete the 1st "Business" item, it might actually be the 10th item in your main array!

To fix this, update your removeItems function to find the "real" ID: 

func removeItems(at offsets: IndexSet, in source: [ExpenseItem]) {
    for offset in offsets {
        let item = source[offset]
        if let index = expenses.items.firstIndex(where: { $0.id == item.id }) {
            expenses.items.remove(at: index)
        }
    }
}

Pro-Tip: The "Sub-View" Refactor

Since you are now using the same HStack code twice (once for Personal, once for Business), your body is going to get messy. Create a small helper view inside the same file to keep things tidy: 

struct ExpenseRow: View {
    let item: ExpenseItem
    // You can move your color logic here too!

    var body: some View {
        HStack {
            VStack(alignment: .leading) {
                Text(item.name).font(.headline)
                Text(item.type)
            }
            Spacer()
            Text(item.amount, format: .currency(code: Locale.current.currency?.identifier ?? "USD"))
        }
    }
}

Why this is the "Best" way:

  • Single Source of Truth: You still only have one items array to save to UserDefaults.
  • Automatic Updates: Because Expenses is @Observable, any time you add an item in AddView, these filtered properties re-calculate instantly.
  • Native Feel: Using Section in a List provides the standard iOS grouping that users expect.

Here is the complete picture of how your ContentView should look with the Filtered Properties, the Sections, and the Corrected Deletion Logic.

The Updated ContentView

struct ContentView: View {
    @State private var expenses = Expenses()
    @State private var showingAddExpense = false

    // 1. Filtered Computed Properties
    var personalExpenses: [ExpenseItem] {
        expenses.items.filter { $0.type == "Personal" }
    }

    var businessExpenses: [ExpenseItem] {
        expenses.items.filter { $0.type == "Business" }
    }

    var body: some View {
        NavigationStack {
            List {
                // 2. Personal Section
                Section("Personal Expenses") {
                    ForEach(personalExpenses) { item in
                        expenseRow(for: item)
                    }
                    .onDelete { offsets in
                        removeItems(at: offsets, in: personalExpenses)
                    }
                }

                // 3. Business Section
                Section("Business Expenses") {
                    ForEach(businessExpenses) { item in
                        expenseRow(for: item)
                    }
                    .onDelete { offsets in
                        removeItems(at: offsets, in: businessExpenses)
                    }
                }
            }
            .navigationTitle("iExpense")
            .toolbar {
                Button("Add Expense", systemImage: "plus") {
                    showingAddExpense = true
                }
            }
            .sheet(isPresented: $showingAddExpense) {
                AddView(expenses: expenses)
            }
        }
    }

    // 4. Helper View Function (to avoid repeating the HStack code)
    func expenseRow(for item: ExpenseItem) -> some View {
        HStack {
            VStack(alignment: .leading) {
                Text(item.name)
                    .font(.headline)
                Text(item.type)
                    .font(.caption)
                    .foregroundStyle(.secondary)
            }
            Spacer()
            Text(item.amount, format: .currency(code: Locale.current.currency?.identifier ?? "USD"))
                .foregroundStyle(color(for: item.amount))
        }
    }

    // 5. Updated Deletion Logic
    func removeItems(at offsets: IndexSet, in source: [ExpenseItem]) {
        for offset in offsets {
            let item = source[offset]
            if let index = expenses.items.firstIndex(where: { $0.id == item.id }) {
                expenses.items.remove(at: index)
            }
        }
    }

    func color(for amount: Double) -> Color {
        if amount < 10 { return .primary }
        if amount < 100 { return .blue }
        return .red
    }
}

Why this works:

  • The Delete Logic: This is the most critical part. When you delete the second item in the "Business" section, offsets says "Index 1." But in your main expenses.items array, that might actually be index 5. The new removeItems function looks at the UUID of the item at "Index 1 of Business" and finds where that exact UUID lives in the master list.
  • The Section Header: Using Section("Title") inside a List automatically gives you the native iOS grouped look, which handles the spacing and font styling for you.
  • The Helper Function: I moved the HStack into a function called expenseRow. This means if you want to change the font or layout later, you only have to change it in one place instead of updating both the Personal and Business blocks.

One final warning: SwiftUI can sometimes be a bit "lazy" with onDelete inside multiple sections. If you find the swipe-to-delete animation looks a little jumpy, it’s usually just the Simulator being the Simulator. It should be buttery smooth on a real device!

Integrate the Color function into the ExpenseRow Struct

To integrate the color(for:) logic into the ExpenseRow structure, you have two main paths. You can either keep it as a function inside the struct or turn it into a computed property.

Since the color depends entirely on the item.amount, turning it into a computed property is the most "SwiftUI" way to handle it—it makes the color feel like a natural attribute of the row.

Option 1: The Computed Property (Cleanest)

By placing the logic inside the ExpenseRow, you make the row self-sufficient. It doesn't need to ask the ContentView for instructions on how to color itself. 

struct ExpenseRow: View {
    let item: ExpenseItem

    // This property lives inside the row and handles the logic
    var amountColor: Color {
        switch item.amount {
        case ..<10:
            return .primary // or .black
        case 10..<100:
            return .green
        default:
            return .red
        }
    }

    var body: some View {
        HStack {
            VStack(alignment: .leading) {
                Text(item.name)
                    .font(.headline)
                Text(item.type)
                    .font(.caption)
                    .foregroundStyle(.secondary)
            }
            Spacer()
            Text(item.amount, format: .currency(code: Locale.current.currency?.identifier ?? "USD"))
                .foregroundStyle(amountColor) // Now just call the property
        }
    }
}

Option 2: The Logic Inside the Model (Best for Reusability)

If you moved the color logic into your ExpenseItem struct earlier (as discussed a few steps ago), the row becomes incredibly simple because you are just accessing data: 

struct ExpenseRow: View {
    let item: ExpenseItem

    var body: some View {
        HStack {
            // ... name and type views ...
            Spacer()
            Text(item.amount, format: .currency(code: "USD"))
                .foregroundStyle(item.color) // Logic lives inside the ExpenseItem struct
        }
    }
}

Why move it into the ExpenseRow?

When you keep the color function in ContentView, you have to "pass" the logic down or call it explicitly. By moving it into the ExpenseRow (Option 1):

  • Encapsulation: The ContentView doesn't need to know how an expense is colored; it just needs to display the row.
  • Readability: Your List code stays focused on the hierarchy (Sections and Rows) rather than the nitty-gritty details of currency thresholds.
  • Maintenance: If you decide that ＄50 should be the new "red" threshold, you only change it in the ExpenseRow struct, and every list in your app updates automatically.

How to use it in your List

Now that the logic is tucked away inside the row, your ForEach in the ContentView becomes a one-liner: 

Section("Personal") {
    ForEach(personalExpenses) { item in
        ExpenseRow(item: item) // All the color magic happens inside here now!
    }
    .onDelete { offsets in
        removeItems(at: offsets, in: personalExpenses)
    }
}

Quick catch: If you move the logic into the ExpenseRow struct, remember to delete the func color(for amount: Double) from your ContentView to avoid having duplicate code in two places!

Minimap & Jump Bar

Note on the use of // MARK: in the following annotations.

In the world of Xcode, // MARK: is a special "instruction" for the editor's Minimap and Jump Bar. It acts as a digital bookmark to help you navigate long files.

Think of it as a Table of Contents for your code.

1. Where do you see it?

If you look at the very top of your Xcode editor window, there is a breadcrumb path (usually showing Project Name > Folder > FileName > ...).

  • If you click the last item in that path (the name of the current struct or function), a dropdown menu appears.
  • Your MARK: comments will appear there in bold, creating clear categories so you can jump straight to "DATA MODEL" or "ACTIONS" without scrolling.

2. The Syntax Variations

There are two main ways to use it:

  • The Label: // MARK: Description:
    • This creates a bold heading in the jump bar.
  • The Divider: // MARK: - Description (Note the hyphen):
    • Adding that hyphen tells Xcode to draw a thin horizontal line above the heading in the menu, which is great for visually separating your State variables from your Body logic.

3. Why it’s so useful in SwiftUI

Because SwiftUI views tend to get very "nested" with many trailing braces, a file can quickly become a "wall of text."

  • Use MARK: - UI COMPONENTS to separate your main view from your helper views.
  • Use MARK: - LOGIC to separate your functions (like removeItems) from your UI code.

4. Pro-Tips for Navigation

Since you're using Vim Mode, you're already a fan of efficiency. Combine MARK: with these shortcuts:

  • Quick Jump: Press <Ctrl> 6. This opens that "Jump Bar" menu instantly via keyboard. You can then type a few letters of your MARK name and hit Enter to fly to that section.
  • The Minimap: If you have the Minimap enabled (on the right side of the editor), your MARK: labels will actually be rendered in tiny text on the map, allowing you to see the "landmarks" of your file as you scroll.

It’s basically a way of saying to yourself (or other developers): "Hey, this specific section of the file is for these specific things." It's a hallmark of professional-grade Swift code!

import SwiftUI

// MARK: - DATA MODEL
// Structs are "Value Types." Use them to define the shape of your data.
struct ExpenseItem: Identifiable, Codable {
    // Identifiable: Lets the List know which item is which using a unique ID.
    // Codable: Allows this data to be converted to JSON for saving (encoding/decoding).
    
    var id = UUID() // Generates a unique "Universally Unique Identifier."
    let name: String
    let type: String
    let amount: Double
}

// MARK: - DATA STORAGE (THE "BRAIN")
// @Observable: A macro that tells SwiftUI to watch this class for changes.
// Classes are "Reference Types." Use them to share data across different views.
@Observable
class Expenses {
    // The "Source of Truth" array.
    var items = [ExpenseItem]() {
        // didSet is a "Property Observer." It runs every time the array changes.
        didSet {
            // Convert the array into JSON data (encoding).
            if let encoded = try? JSONEncoder().encode(items) {
                // Save that JSON to the phone's permanent storage (UserDefaults).
                UserDefaults.standard.set(encoded, forKey: "Items")
            }
        }
    }
    
    // The initializer runs when the app first creates this object.
    init() {
        // Check if there is data saved under the key "Items."
        if let savedItems = UserDefaults.standard.data(forKey: "Items") {
            // Try to convert the JSON data back into an array of ExpenseItems (decoding).
            if let decodedItems = try? JSONDecoder().decode([ExpenseItem].self, from: savedItems) {
                items = decodedItems
                return
            }
        }
    }
}

// MARK: - COMPONENT VIEW (A REUSABLE ROW)
// Extracting logic into a sub-view keeps the main layout clean.
struct ExpenseRow: View {
    let item: ExpenseItem
    
    // Computed Property: Logic that returns a specific type (Color).
    var amountColor: Color {
        switch item.amount {
        case ..<10: return .primary      // Standard text color
        case ..<100: return .green      // Mid-range
        default: return .red            // High expense
        }
    }
    
    var body: some View {
        HStack {
            VStack(alignment: .leading) {
                Text(item.name)
                    .font(.headline)
                Text(item.type)
                    .font(.subheadline)
                    .foregroundStyle(.secondary) // Faded grey look
            }
            
            Spacer() // Pushes the following content to the far right
            
            // Format the Double as a currency based on the user's phone settings.
            Text(item.amount, format: .currency(code: Locale.current.currency?.identifier ?? "USD"))
                .foregroundStyle(amountColor) // Applies our custom switch-case color
        }
    }
}

// MARK: - MAIN INTERFACE
struct ContentView: View {
    // @State: Tells SwiftUI that this view "owns" this data. 
    // If 'expenses' changes, the view will re-draw.
    @State private var expenses = Expenses()
    
    // Controls whether the AddView sheet is currently visible.
    @State private var showingAddExpense = false
    
    // Filtered Computed Properties: 
    // These create "virtual lists" based on the master 'items' array.
    var personalExpenses: [ExpenseItem] {
        expenses.items.filter { $0.type == "Personal" }
    }
    var businessExpenses: [ExpenseItem] {
        expenses.items.filter { $0.type == "Business" }
    }
    
    var body: some View {
        // NavigationStack: Provides the header area and manages view transitions.
        NavigationStack {
            // List: Provides the scrolling, grouped look and swipe actions.
            List {
                // Section: Groups rows with a header.
                Section("Personal") {
                    ForEach(personalExpenses) { item in
                        ExpenseRow(item: item)
                    }
                    .onDelete { offsets in
                        // Pass the offsets and the specific filtered list.
                        removeItems(at: offsets, in: personalExpenses)
                    }
                }
                
                Section("Business") {
                    ForEach(businessExpenses) { item in
                        ExpenseRow(item: item)
                    }
                    .onDelete { offsets in
                        removeItems(at: offsets, in: businessExpenses)
                    }
                }
            }
            .navigationTitle("iExpense") // Sets the text in the NavigationStack
            .toolbar {
                // Button with a system icon (SF Symbols).
                Button("Add Expense", systemImage: "plus") {
                    showingAddExpense = true
                }
            }
            // A Modal Sheet that slides up when 'showingAddExpense' becomes true.
            .sheet(isPresented: $showingAddExpense) {
                // Passes the 'expenses' class so AddView can add items to it.
                AddView(expenses: expenses)
            }
        }
    }
    
    // FUNCTION: Handles item deletion across sections.
    func removeItems(at offsets: IndexSet, in source: [ExpenseItem]) {
        for offset in offsets {
            let item = source[offset]
            // We find the ID of the filtered item within the MASTER array.
            if let index = expenses.items.firstIndex(where: { $0.id == item.id }) {
                expenses.items.remove(at: index)
            }
        }
    }
}

// MARK: - PREVIEW

    ContentView()
}

Key Concepts to Remember:

  • Property Observers (didSet): Great for "silent" tasks like saving data every time a variable changes.
  • Computed Properties: Use these for UI logic (like colors or filtering) so your body stays focused on the layout.
  • Identifiable: Without id = UUID(), the ForEach wouldn't know which item you are trying to delete.
  • Locale.current: Using this makes your app "World-Ready"—it will show Euros for a user in France or Yen for a user in Japan automatically.

You’ve essentially built a mini "Model-View-ViewModel" (MVVM) architecture here:

  • The Model: ExpenseItem (The data)
  • The View: ContentView and ExpenseRow (The interface)
  • The ViewModel: Expenses class (The logic that manages and saves the data)

By keeping these layers separate, your code stays "modular." If you decided tomorrow to save your data to the Cloud instead of UserDefaults, you’d only have to change the code inside the Expenses class—the rest of your app wouldn't even need to know anything had changed!

Now that you have this annotated "Master Script," you have a perfect template for almost any data-entry app you want to build in the future.

The AddView Module

While ContentView is about reading data, this module is all about creating it and safely passing it back to the "Source of Truth."

Here is the annotated walk-through for your AddView.swift file: 

import SwiftUI

struct AddView: View {
    // MARK: - ENVIRONMENT & STATE
    
    // @Environment: Accesses system-wide values. 
    // Here, we grab the "dismiss" action, which allows the view to close itself.
    @Environment(\.dismiss) var dismiss
       
    // @State: Local "Single Source of Truth" for the form fields.
    // These hold the data temporarily until the user hits "Save."
    @State private var name = ""
    @State private var type = "Personal"
    @State private var amount = 0.0
    
    // Data source for our Picker.
    let types = ["Business", "Personal"]
    
    // MARK: - EXTERNAL DATA
    
    // This receives the 'Expenses' class instance from ContentView.
    // Because it is a class (Reference Type), adding an item here 
    // updates the same list seen in ContentView automatically.
    // This is a variable awaiting population from ContentView as an argument:
    // See: .sheet(isPresented: $showingAddExpense) {
    //            AddView(expenses: expenses)
    //        }
    // Left expenses refers to this unassigned variable, the right expenses is
    // the variable in ContenView.
    var expenses: Expenses
    
    // MARK: - USER INTERFACE
    var body: some View {
        NavigationStack {
            // Form: A specialized container that automatically styles rows 
            // to look like standard iOS settings or input screens.
            Form {
                // $name: The '$' creates a "Two-Way Binding."
                // 1. The TextField shows what's in the variable.
                // 2. Typing in the field updates the variable instantly.
                TextField("Name", text: $name)
                
                // Picker: Linking the 'selection' to our 'type' state.
                Picker("Type", selection: $type) {
                    // \.self: Tells SwiftUI to use the String itself as the unique ID.
                    ForEach(types, id: \.self) {
                        Text($0) // '$0' is shorthand for "the current string in the loop"
                    }
                }
                
                // TextField for numbers: Uses 'value' and 'format' instead of 'text'.
                TextField("Amount", value: $amount, format: .currency(code: "USD"))
                    .keyboardType(.decimalPad) // Forces the numeric keypad with a decimal point
            }
            .navigationTitle("Add new expense")
            
            // MARK: - ACTIONS
            .toolbar {
                Button("Save") {
                    // 1. Package the local @State into a new ExpenseItem struct.
                    let item = ExpenseItem(name: name, type: type, amount: amount)
                    
                    // 2. Append the new item to the shared Expenses class.
                    expenses.items.append(item)
                    
                    // 3. Dismiss the view. 
                    // This triggers the 'isPresented' binding in ContentView to become false.
                    dismiss()
                }
            }
        }
    }
}

// MARK: - PREVIEW
#Preview {
    // The Preview needs a "dummy" version of Expenses to render correctly.
    AddView(expenses: Expenses())
}

Key Logic Annotations:

  • The Power of dismiss(): This is much cleaner than passing a boolean binding back and forth. You are telling the system, "I’m done with whatever modal state I’m in; please take me back."
  • Two-Way Bindings (＄): This is a core SwiftUI concept. Without that ＄, the TextField could read the value but wouldn't be able to change it.
  • Form vs. List: While they look similar, a Form is optimized for input (handling focus, keyboard avoidance, and specific row alignment), whereas a List is optimized for displaying many rows of data.
  • Keyboard Type: .decimalPad is a "Quality of Life" feature. It prevents users from accidentally typing letters into your Double variable, which would cause the save to fail or behave oddly.

The "Swift Way" to Calculate Totals:

Instead of using a for loop to add numbers manually, Swift uses a powerful function called reduce. You can add this as a computed property inside your Expenses class or your ContentView: 

var totalAmount: Double {
    items.reduce(0) { $0 + $1.amount }
}
  • The 0 is the starting point.
  • ＄0 is the "running total."
  • ＄1 is the "next item" in the list.

It combines them all into one final number in a single line of code!
