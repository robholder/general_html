Hacking With Swift - 58 - Views as properties

There are lots of ways to make it easier to use complex view hierarchies in SwiftUI, and one option is to use properties – to create a view as a property of your own view, then use that property inside your layouts.

For example, we could create two text views like this as properties, then use them inside a VStack: 

struct ContentView: View {
    let motto1 = Text("Draco dormiens")
    let motto2 = Text("nunquam titillandus")

    var body: some View {
        VStack {
            motto1
            motto2
        }
    }
}

You can even apply modifiers directly to those properties as they are being used, like this: 

VStack {
    motto1
        .foregroundStyle(.red)
    motto2
        .foregroundStyle(.blue)
}

Creating views as properties can be helpful to keep your body code clearer – not only does it help avoid repetition, but it can also get more complex code out of the body property.

Swift doesn’t let us create one stored property that refers to other stored properties, because it would cause problems when the object is created. This means trying to create a TextField bound to a local property will cause problems.

However, you can create computed properties if you want, like this: 

var motto1: some View {
    Text("Draco dormiens")
}

This is often a great way to carve up your complex views into smaller chunks, but be careful: unlike the body property, Swift won’t automatically apply the @ViewBuilder attribute here, so if you want to send multiple views back you have three options.

First, you can place them in a stack, like this: 

var spells: some View {
    VStack {
        Text("Lumos")
        Text("Obliviate")
    }
}

If you don’t specifically want to organize them in a stack, you can also send back a Group. When this happens, the arrangement of your views is determined by how you use them elsewhere in your code: 

var spells: some View {
    Group {
        Text("Lumos")
        Text("Obliviate")
    }
}

The third option is to add the @ViewBuilder attribute yourself, like this: 

@ViewBuilder var spells: some View {
    Text("Lumos")
    Text("Obliviate")
}

Of them all, I prefer to use @ViewBuilder because it mimics the way body works, however I’m also wary when I see folks cram lots of functionality into their properties – it’s usually a sign that their views are getting a bit too complex, and need to be broken up. Speaking of which, let’s tackle that next…

What's the Difference - VStack & Group?

At first glance, the VStack and Group examples above look identical in the Canvas, but their "DNA" is completely different.

The VStack is a Layout container; Group is a Logical container.

1. The Differences

Feature VStack Group
Layout Forces views to be stacked vertically. Does not have a layout of its own.
Spacing Adds default system spacing between items. No spacing (items just exist).
Styling Passing a modifier to a VStack styles the container. Passing a modifier to a Group styles every item inside individually.
Limit Can hold up to 10 views (standard). Used to bypass the 10-view limit or group logic.

2. "Will the text lines always be stacked?"

No.

  • In your VStack version, they are hardcoded to be vertical. No matter where you put that spells property, it will always be a vertical stack.
  • In your Group version, the layout is inherited from wherever you put it.
    • If you put spells inside a VStack, they stack.
    • If you put spells inside an HStack, they go side-by-side.
    • If you put spells inside a List, they become two separate rows.

A Group is a chameleon; a VStack is a skyscraper.

3. How would you actually use the Group version?

You use Group when you want to apply a "batch" of settings to multiple views without changing how they are positioned.

Use Case A: Batch Styling

Instead of writing .foregroundStyle(.secondary) on five different Text views, you wrap them in a Group: 

var spells: some View {
    Group {
        Text("Lumos")
        Text("Obliviate")
    }
    .font(.title)
    .foregroundStyle(.blue) // Both texts turn blue
}

Use Case B: Conditional Logic

This is the most common use. If you want to return some View but have an if/else statement, SwiftUI gets grumpy because it wants a single return type. Group fixes this: 

var spells: some View {
    Group {
        if userIsWizard {
            Text("Lumos")
        } else {
            Image(systemName: "muggle")
        }
    }
}

Use Case C: The "10 View Limit"

SwiftUI (historically and internally) can only handle 10 views inside a single parent. If you have 15 Text views, you wrap the first 10 in a Group and the next 5 in another Group to keep the compiler happy.

Summary

Think of a VStack as a physical box—it has dimensions and keeps things in order. A Group is like a rubber band—it holds things together so you can move them as one, but it doesn't have a shape of its own.

QUESTION

Does that mean that this wouldn't work: 

var spells: some View {
    VStack {
        Text("Lumos")
        Text("Obliviate")
    }
    .font(.title)
    .foregroundStyle(.blue)
}

Gemini said

Actually, it would work perfectly!

In SwiftUI, modifiers like .font() and .foregroundStyle() are environment-driven. When you apply them to a container like a VStack, the VStack says to its children, "Hey, unless you have your own specific settings, use these."

So, if you want everything in the stack to be blue and large, your VStack code is exactly right.

So why bother with Group?

The distinction comes down to Layout vs. Logistics.

1. The Layout Difference

If you use the VStack, those two spells are trapped in a vertical arrangement. If you later decide you want them to be horizontal in a different part of your app, you have to rewrite the property. If you use a Group, you can drop spells into an HStack somewhere else, and they will automatically sit side-by-side.

2. The "Ghost" Container

Sometimes you want to apply a modifier to multiple views, but you don't want them to be in a stack.

Imagine you are building your flag buttons. You might want 3 flags to have the same shadow and border, but they are already inside a complex VStack/HStack layout. You can't wrap them in another VStack without potentially messing up your spacing. You wrap them in a Group instead. It applies the styles but stays "invisible" to the layout engine.

The "Nesting" Trick

Here is where it gets interesting. You can use them together: 

VStack {
    Text("Spells:") // This stays black/default
        .font(.headline)

    Group {
        Text("Lumos")
        Text("Obliviate")
    }
    .foregroundStyle(.blue) // Only these two turn blue
}

Summary

Your VStack version is like a uniformed squad: they are standing in a specific formation (vertical) and wearing the same clothes (blue).

The Group version is like a group of friends: they are wearing the same clothes, but they can stand in a line, a circle, or spread out across the room depending on where the "party" (the parent container) is happening.
