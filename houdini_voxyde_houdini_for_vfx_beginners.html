Attributes

Quote:

Houdini attributes are the most important part of Houdini. Your ability to create and modify attributes is the single most important thing that will determine your success as a VFX artist. An attribute - in the simplest terms - is just a piece of information that you can attach to geometry. This attribute once you created will transfer down over to all of the other nodes.

The Attribute Adjust Float and Attribute Noise Vector are introduce, showing how you can supply the Attribute Name and perform various changes to the attribute.

The Attribute Create node was shown (to lay down an attribute on a network), but you don't really need to do that for the most part, as the 'adjust' nodes can also create an attribute if the named attribute didn't already exist.

Interesting tip: if the attribute name field has what looks like a Google Maps location marker (an inverted rain drop shape), clicking it on will act as a visualizer for the Viewport and show the attribute in an appropriate manner.

Also, if you have an attribute node selected in the Network Editor, pressing <enter> in the Viewport will give access to the Handle for that adjustment.

The Attribute Adjust Float example shows an object, and this node is set to Operation of Line, and in the Viewport, the Handle is evoked, revealing a start and end point for a linear gradation (shown as a spectrum when visualized). Looking at the Geometry Spreadsheet reveals the values of zero to one being applied - in this case to a new attribute called 'mask'.

An Attribute Noise node is then used to illustrate how the 'mask' attribute can be added into the Blend field, to drive the efficacy of the noise.

Binding Attributes

I haven't got the full picture, but on the Poly Extrude node, the tutorial shows how a created attribute ('zscale') is used to drive the distance of each poly extrude. Instead of a Maya style 'binding' on the distance slider, there is a section lower down the parameter pane (Local Attributes) to activate an attribute assignment to the parameters (and in this case there is a 'Direction' field. The name of the attribute is entered into the field, and since the poly extrude works on primitives, the attribute was created as a primitive attribute (even though the default was 'points').

Attribute VOP

Typing Attribute in the Tab (create node menu) shows a ton of 'Attribute <blah>' nodes.

Instead of learning all these nodes, the tutorial suggest all can be replaced with Attribute VOP, and understanding this node will be all you need to know.

The previously mentioned Attribute nodes (above) can all be replaced with the 'Attribute VOP'. However, sometimes, the pre-built variations can still be used, and often save 'set up' time.

This [Attribute VOP] is the single most important node you have to learn in Houdini.

While VOPs means 'vector attributes', the various 'attribute vop' nodes can operate on floats, ints, vectors or any king of attribute.

Inside

Inside the Attribute VOP node, there are two nodes by default:

  • Input: geometryvopglobal
  • Output: geometryvopoutput

Attributes of parameter input/output widgets are 'colored' according to the data type:

  • green is Vector
  • teal is Float
  • blue is Integer
  • gray (slightly orange) can be custom external Inputs

VOPs Operations

You can re-create the functionality of the Attribute node varieties by connecting nodes inside the Attribute VOP node.

If you use a Noise node, the Turbulent Noise seems to be the most commonly used.

Voxyde shows a default Turbulent Noise node connected directly to the P (position) input node of the Output node, and shows how it doesn't work (the geometry disappers). The reason is that the noise attribute is a 'float' and the 'P' (position) input is expecting a 'vector' data type - and this is were the color codes come into play. The teal (float) doesn't necessarily play well with the green (vector) input.

To fix this, on the Turbulent Noise node, the Signature has to be switched (via drop-down) to 3D Noise. (The color of the 'nose' output widget changes to green.

Creating an Add node enables creating a similar action to the Attribute Noise node.

  • P goes from the input node through the turbulent noise, this then goes into input 2 of the add, and on to the output (P)
  • Add also takes in P from the input node (to its input 1), and the 'sum' output goes onwards to the output node's 'P' input.

The tutorial shows how nodes like the Constant can be substitutes for the sliders on attributes of nodes by plugging into the slots available.

TIP: <shift> R reverses the order of the inputs.

TIP: MMB on an input parameter and choose Constant to automatically create a Constant node hooked up to the parameter.

Also, Constant nodes can deliver values of numerous types, not just floats! The Constant Type drop-down will reveal a ton of options (on the Constant node's parameter pane).

Using the Y value of Position (Y-grad anyone?)

Inside the Attribute VOP node:

  • From the Input node (geometryvopglobal):
    • Take the P (position) output
    • Plug into a Vector to Float node (since we need to split the 3 coords from the Vector).
  • The values from the 'P' parameter include negative, so we need to 'fit' to positive numbers:
    • Plug the fval2 (y-coord) from the Vector to Float
    • Into a Fit Range:
      • Source Min will be the extent into negative y (so in the tutorial case, -3)
      • Source Max will be 3 (as the highest point on the object is 3 in 'y').
  • You can then 'expose' or promote these to the SOP level (Attribute VOP parameters).

This particular set up can replace a set up using Attribute Adjust Float in tandem with an Attribute Noise Vector (another example shown in the tutorial).

In fact, if you go inside one of the custom Houdini attribute nodes, you'll see how they are constructed and which parameters have been promoted to the node itself.

Attribute VOP node is like a For each loop

The tutorial argues that the contents of an Attribute VOP is like a For/each loop, acting on each elements specified (point, primitive, etc.).

The Compare node is used as an illustration, with one set to Greater Than and zero as the input value. This node is piped from the Vector to Float from the 'P' value, and piped into the Cd (color) parameter as output. This means, everything under zero in 'Y' is black, and above zero is white.

'Attribute VOP' vs 'Attribute Wrangle'

The Attribute VOP and Attribute Wrangle are essentially doing the same thing, but Attribute Wrangle does it using VEX and VEXpression language rather than node based.

Using VEX to move an object upward 2 units (at point level): 

@P += 2;

DOPs - Dynamic Operators

DOPs are the Dynamics of Houdini.

The tutorial creates a POP Network to illustration that in fact the network is a DOP Network. (This is apparent when you look at it in the Network Editor. Particles in Houdini are literally just points, so controlling them is a DOP responsibility.

Inside the DOP Network

Initial DOPs setup: 

                                ┌────────────────────┐   
                                │ source_first_input │   
   ┌───────────┐                └──────────┬─────────┘   
   │ popobject │                           │             
   └───┬───────┘                           │             
       │                        ┌──────────┴──────────┐  
       │                        │ wire_pops_into_here │  
       └─────────────┐          │       MERGE         │  
                     │          └─────────┬───────────┘  
                     │                    │              
                     │    ┌───────────────┘              
                  ┌──┴────┼───┐                          
                  │ popsolver │                          
                  └───────────┘                          
                       │                                 
                       │                                 
                   ┌───┴────┐                            
                   │ output │                            
                   └────────┘

The merge can be disregarded when understanding the network.

The remaining 4 nodes are a basis for understanding most simulations.

  • Pop Object:
    • The points (particles) container.
  • Source Input:
    • The source object or geometry from which the points are scattered.
  • Pop Solver:
    • Engine that tells the points/particles what to do at each frame.
  • Output:
    • The conduit through which the result emerge.
    • The Output works even if other nodes have display focus (unlike a custom Null 'OUT' node for example).

Solvers

The Solvers are what makes DOPs or separates DOPs from SOPs.

There are many varieties of Solvers:

  • Pop solvers:
    • for particle simulations
  • Flip solvers:
    • for fluid simulations
  • Pyro solvers:
    • for smoke simulations
  • Rigid body solvers:
    • for collapsing, destruction, interaction simulations
  • Vellum Solvers:
    • for cloth and fabrics

Each solver 'engine' will tell particles to move and behave like the subject of the solver.

Each solver uses unique algorithms to achieve the specific type of simulations.

Objects

If we look at the first Object node needed to feed into a Solver, we'll see a pattern:

  • Pop:
    • Pop Object
    • Pop Object > Pop Solver > Output
  • Flip:
    • Flip Object
    • Flip Object > Flip Solver > Output
  • Pyro:
    • Smoke Object
    • Smoke Object > Pyro Solver > Output
  • RBD:
    • RBD Packed Object
    • RBD Packed Object > Rigid Body Solver > Output
  • Vellum:
    • Vellum Object
    • Vellum Object > Vellum Solver > Output

*These Objects are essentially 'boxes' or 'containers' for the simulation objects.*

Second Input

The second input is the geometry base.

The same 'second input' node (Pop Source node) can be used on the Flip simulations. (This is demonstrated in the tutorial using a cube.)

SOP Solver

To get to know the functionality of solvers, the tutorial goes to a simple solver, a 'SOP Solver' Solver node.

Looking inside, we see this setup: 

                                                              
                                                              
                    ┌───────┐ ┌───────┐ ┌───────┐ ┌───────┐   
                    │input_1│ │input_2│ │input_3│ │input_4│   
                    └───────┘ └───────┘ └───────┘ └───────┘   
                                                              
    ┌────────────┐                                            
    │ Prev_Frame │                                            
    └────┬───────┘                                            
         │                                                    
         │                                                    
      ┌──┴──┐                                                 
      │ OUT │                                                 
      └─────┘

The Input nodes correspond to the inputs seen on the exterior (Solver) node - the input nodules.

The Prev_Frame node takes the state of the object input (input 0) as it was on the frame before the current. Adding a Transform node between the Prev_Frame and OUT node, and putting a value in y, will result in an ascending motion in playback, as the transform value is added to each frame.

We are accumulating changes rather than applying changes.

NOTE: to see the effect of the SOP Solver we have to pin the viewport at the upper level while working in the Solver. Other simulation solvers work even when inside the DOP Network.

Solver with Noise

NOTE: when the timeline bar is orange for a simulation, you have to Reset Simulation on the containing node to proceed and ingest any changes you've made.

The tutorial shows several examples of purely Attribute VOP particle effects and then a similar thing in Solver, where the values build on the previous frame's calculation. The Solver version results in more organic morphing motion.

The tutorial adds a Curl Noise node inside the Attribute VOP node (added to 'P' position) and after altering the Curl Noise parameters gets a smoke like motions. Adding an offset to the 'y' parameter of the Curl Noise offset results in a slight lifting of the particles as it proceeds.

A transform is also added after the Attribute VOP to add more vertical movement (since we are still pulling from the Prev_Frame at the start).

The take-away is that this is the basis of the solvers.

VOPs in DOPs and Conclusion

The tutorial introduces the notion of adding a Geometry VOP inside a POP Network. (It's added after the source_first_input (POP Source) node.

I think the take away is that you can accumulate VOP results, so adding a constant to position for example will result in that constant being added EVERY frame, and not just once.
