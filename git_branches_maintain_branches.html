Git Main, MacMini & MacBook Branches

QUESTION

I want to have a main branch in a repo, then two branches (one for each macOS machine). These are config files.

I want to be able to adjust configs on each machine (on their respective branches) but when it seems like a good Idea, I'd like to pull the changes from one or other of the 'machine' branches into main. Is this possible while maintaining two distinct branches that are always in parallel to main?

ANSWER

This is a classic "Dotfiles" workflow. It is absolutely possible and is actually the recommended way to manage machine-specific configurations (like WezTerm or Starship settings) while sharing a common core.

1. The Strategy: "The Core and the Flavors"

  • main branch: Contains the "Generic" config that works on both Macs (e.g., your Vim mappings, Git aliases).
  • mac-work & mac-home branches: These stay "downstream" from main. They contain the hostname-specific colors/icons we discussed.

2. The Workflow

To keep everything in sync without creating a mess, you follow a one-way flow for shared features and a specific pull for "promoting" features.

A. When you improve a "shared" config (on main):

You want both machines to get this update.

  • Commit to main.
  • Switch to your machine branch: git switch mac-work.
  • Rebase on main: git rebase main.
    • This "lifts" your machine-specific colors and places them on top of the new shared core.

B. When you have a "great idea" on a machine branch:

If you wrote a cool new function on mac-home and want it on both machines: Cherry-pick that specific commit into main: 

git checkout main
git cherry-pick <commit-hash-from-home-branch>

Now that it's in main, follow Step A above to pull it into the mac-work branch.

3. Maintaining "Parallel" Distinctness

To ensure your host-specific variables don't get overwritten when you merge, you have two options:

  • Option 1 (Git Attributes): You can tell Git to "always keep my local version" of a specific file (like hostname_config.lua) during merges using a merge=ours strategy in a .gitattributes file.
  • Option 2 (Modularization - Recommended):
  • Keep wezterm.lua identical on main.
  • Have it require('host_config').
  • On mac-work, host_config.lua defines Blue.
  • On mac-home, host_config.lua defines Green.
  • Never merge host_config.lua into main.

4. How to prevent accidental merges

To make sure you don't accidentally merge your "Work" colors into "Main," you can use a .gitignore on the main branch that includes the host_config.lua file, but force add it only on the specific machine branches.

The Pro-Tip for Vim/Mac Users:

Since you use WezTerm, you can actually skip the "two branches" headache entirely by using the Lua logic we wrote earlier. If your wezterm.lua detects the hostname and sets the color automatically, you only need one branch (main) for both machines.

Do you prefer the logic-based approach (one branch, code detects the machine) or the branch-based approach (two separate versions of the file)?
