Git - Unstaging

The Git Tag System: A Concise Guide

1. What is a Tag?

A tag is a permanent bookmark attached to a specific commit. While branch names (like main) move forward as you add new code, a tag stays exactly where you put it forever.

2. Creating Tags

There are two main types, but you should almost always use Annotated Tags because they store the "Why" (the comment).

Annotated vs. Lightweight Tags

Keep in mind that Git has two types of tags:

  • Annotated Tags: Created with git tag -a. These store comments, tagger info, and dates. The commands above work best for these.
  • Lightweight Tags: Created with git tag (no flags). These are just pointers to a commit and do not have comments. If your tags aren't showing comments, they might be lightweight tags!

Tag the current spot:

Annotated: 

git tag -a v1.0 -m "Finished Moonshot basic layout"

Lightweight: 

git tag v1.0

Tag a spot in the past:

git tag -a v0.9 <commit-hash> -m "Before starting NavigationStack refactor"

3. Viewing Tags & Comments

Tags are slightly "hidden" in the standard log, so use these commands to see them:

  • List all labels:
    • git tag
  • List labels + their comments:
    • git tag -n
  • See the full "Plaque" (Date, Author, Comment, and Code):
    • git show v1.0

In Git, the commands for viewing tags vary depending on whether you want a quick list or a deep dive into the metadata (like the author, date, and comments) of those tags.

The Quick List (Names Only)

If you just need to see the names of all tags in your repository, use:

  • git tag

Showing Tags with Comments

To see the tag message (comment) alongside the tag name, you have two primary options:

Option A: Using -n (The "Clean" List)

The -n flag tells Git to display n lines of the tag comment.

  • Command: git tag -n1

Result: This lists every tag followed by the first line of its comment. You can increase the number (e.g., -n5) to see more lines of a multi-line comment.

Option B: Using git show (The "Deep Dive")

If you want to see everything—the tagger, the date, the full comment, and the specific commit diff—use git show.

  • Command: git show <tag_name>

Result: This provides the full metadata for a specific tag.

3. Formatting a Custom List

If you want a prettier, more readable output that includes the comment and perhaps the date or hash, you can use the --format flag: 

git tag -l --format='%(refname:short) | %(contents:subject) | %(creatordate:short)'
Placeholder Description
%(refname:short) The name of the tag (e.g., v1.0.2).
%(contents:subject) The first line of the tag comment.
%(creatordate:short) The date the tag was created.

4. Navigating with Tags

This is the "Time Travel" part. Use tags to jump to specific lesson milestones without hunting for hashes.

  • Jump to a tag:
    • git switch --detach v1.0
  • Jump back to today:
    • git switch main

5. Managing Tags

Mistyped a tag? Delete it:

git tag -d v1.0

Sharing with GitHub:

Tags are not sent to the cloud by default when you push. You must send them explicitly: 

git push origin --tags

Why Use This For Tutorials Like 100 Days Of Swiftui?

  • Challenge Reference:
    • Tag your code before starting a "Challenge." If you get stuck, you can easily switch back to the clean version to see where you went wrong.
  • Code Evolution:
    • Tag moonshot-classic-nav and moonshot-modern-nav. You can then git show both to compare exactly how the NavigationLink logic changed.

Pro-Tip: If you're looking at a tagged version (--detach) and you accidentally start typing code, Git won't save it to a branch. Always switch back to main before you continue your actual development!
