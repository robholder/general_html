The Git Tag System: A Concise Guide

1. What is a Tag?

A tag is a permanent bookmark attached to a specific commit. While branch names (like main) move forward as you add new code, a tag stays exactly where you put it forever.

2. Creating Tags

There are two main types, but you should almost always use Annotated Tags because they store the "Why" (the comment).

Tag the current spot:

git tag -a v1.0 -m "Finished Moonshot basic layout"

Tag a spot in the past:

git tag -a v0.9 <commit-hash> -m "Before starting NavigationStack refactor"

3. Viewing Tags & Comments

Tags are slightly "hidden" in the standard log, so use these commands to see them:

  • List all labels:
    • git tag
  • List labels + their comments:
    • git tag -n
  • See the full "Plaque" (Date, Author, Comment, and Code):
    • git show v1.0

4. Navigating with Tags

This is the "Time Travel" part. Use tags to jump to specific lesson milestones without hunting for hashes.

  • Jump to a tag:
    • git switch --detach v1.0
  • Jump back to today:
    • git switch main

5. Managing Tags

Mistyped a tag? Delete it:

git tag -d v1.0

Sharing with GitHub:

Tags are not sent to the cloud by default when you push. You must send them explicitly: 

git push origin --tags

Why Use This For Tutorials Like 100 Days Of Swiftui?

  • Challenge Reference:
    • Tag your code before starting a "Challenge." If you get stuck, you can easily switch back to the clean version to see where you went wrong.
  • Code Evolution:
    • Tag moonshot-classic-nav and moonshot-modern-nav. You can then git show both to compare exactly how the NavigationLink logic changed.

Pro-Tip: If you're looking at a tagged version (--detach) and you accidentally start typing code, Git won't save it to a branch. Always switch back to main before you continue your actual development!
