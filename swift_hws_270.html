Hacking With Swift - 270 - Understanding frames and coordinates inside GeometryReader

SwiftUI’s GeometryReader allows us to use its size and coordinates to determine a child view’s layout, and it’s the key to creating some of the most remarkable effects in SwiftUI.

You should always keep in mind SwiftUI’s three-step layout system when working with GeometryReader:

  • Parent proposes a size for the child,
  • The child uses that to determine its own size,
  • Parent uses that to position the child appropriately.

In its most basic usage, what GeometryReader does is let us read the size that was proposed by the parent, then use that to manipulate our view. For example, we could use GeometryReader to make a text view have 90% of all available width regardless of its content: 

struct ContentView: View {
    var body: some View {
        GeometryReader { proxy in
            Text("Hello, World!")
                .frame(width: proxy.size.width * 0.9)
                .background(.red)
        }
    }
}

That proxy parameter that comes in is a GeometryProxy, and it contains the proposed size, any safe area insets that have been applied, plus a method for reading frame values that we’ll look at in a moment.

GeometryReader has an interesting side effect that might catch you out at first: the view that gets returned has a flexible preferred size, which means it will expand to take up more space as needed. You can see this in action if you place the GeometryReader into a VStack then put some more text below it, like this: 

struct ContentView: View {
    var body: some View {
        VStack {
            GeometryReader { proxy in
                Text("Hello, World!")
                    .frame(width: proxy.size.width * 0.9, height: 40)
                    .background(.red)
            }

            Text("More text")
                .background(.blue)
        }
    }
}

You’ll see "More text" gets pushed right to the bottom of the screen, because the GeometryReader takes up all remaining space. To see it in action, add background(.green) as a modifier to the GeometryReader and you’ll see just how big it is. Note: This is a preferred size, not an absolute size, which means it’s still flexible depending on its parent.

Coordinate Spaces

When it comes to reading the frame of a view, GeometryProxy provides a frame(in:) method rather than simple properties.

This is because the concept of a "frame" includes X and Y coordinates, which don’t make any sense in isolation - do you want:

  • the view’s absolute X and Y coordinates, or
  • their X and Y coordinates compared to their parent?

SwiftUI calls these options coordinate spaces, and those two in particular are called:

  • the global space (measuring our view’s frame relative to the whole screen), and
  • the local space (measuring our view’s frame relative to its parent).

We can also create custom coordinate spaces by attaching the coordinateSpace() modifier to a view - any children of that can then read its frame relative to that coordinate space.

To demonstrate how coordinate spaces work, we could create some example views in various stacks, attach a custom coordinate space to the outermost view, then add an onTapGesture to one of the views inside it so it can print out the frame globally, locally, and using the custom coordinate space.

Try this code: 

struct OuterView: View {
    var body: some View {
        VStack {
            Text("Top")
            InnerView()
                .background(.green)
            Text("Bottom")
        }
    }
}

struct InnerView: View {
    var body: some View {
        HStack {
            Text("Left")
            GeometryReader { proxy in
                Text("Center")
                    .background(.blue)
                    .onTapGesture {
                        print("Global center: \(proxy.frame(in: .global).midX) x \(proxy.frame(in: .global).midY)")
                        print("Custom center: \(proxy.frame(in: .named("Custom")).midX) x \(proxy.frame(in: .named("Custom")).midY)")
                        print("Local center: \(proxy.frame(in: .local).midX) x \(proxy.frame(in: .local).midY)")
                    }
            }
            .background(.orange)
            Text("Right")
        }
    }
}

struct ContentView: View {
    var body: some View {
        OuterView()
            .background(.red)
            .coordinateSpace(name: "Custom")
    }
}

Click on the Center text (on the upper left of the center view).

The output you get in the Xcode debug area when that code runs depends on the device you’re using, but here’s what I got:

  • Global center: 191.33 x 440.60
  • Custom center: 191.33 x 381.60
  • Local center: 153.66 x 350.63

Those sizes are mostly different, so hopefully you can see the full range of how these frame work:

  • A global center X of 191 means that the center of the geometry reader is 191 points from the left edge of the screen.
  • A global center Y of 440 means the center of the geometry reader is 440 points from the top edge of the screen. This isn’t dead in the center of the screen because there is more safe area at the top than the bottom.
  • A custom center X of 191 means the center of the geometry reader is 191 points from the left edge of whichever view owns the "Custom" coordinate space, which in our case is OuterView because we attach it in ContentView. This number matches the global position because OuterView runs edge to edge horizontally.
  • A custom center Y of 381 means the center of the geometry reader is 381 points from the top edge of OuterView. This value is smaller than the global center Y because OuterView doesn’t extend into the safe area.
  • A local center X of 153 means the center of the geometry reader is 153 points from the left edge of its direct container.
  • A local center Y of 350 means the center of the geometry reader is 350 points from the top edge of its direct container.

Which coordinate space you want to use depends on what question you want to answer:

  • Want to know where this view is on the screen?
    • Use the global space.
  • Want to know where this view is relative to its parent?
    • Use the local space.
  • What to know where this view is relative to some other view?
    • Use a custom space.

Coordinate Spaces

This lesson is introducing Coordinate Spaces.

The text gets confusing because it throws three different coordinate systems and a wall of numbers at you all at once.

Let’s clean it up by using a simple real-world analogy, and then map it directly to the code.

1. The Core Concept: The "Where Am I?" Analogy

Imagine you are sitting in a seat on an airplane. If someone asks you, "Where are you?", your answer changes completely depending on your frame of reference (your coordinate space):

  • Local Space: You are in Row 12, Seat B. (You are measuring relative only to the inside of your immediate cabin).
  • Global Space: You are at 35,000 feet above Chicago. (You are measuring relative to the entire planet).
  • Custom Space: You are 50 feet away from the pilot. (You are measuring relative to a specific landmark on the plane).

In SwiftUI, a view's X and Y coordinates don't mean anything by themselves. You always have to tell SwiftUI which map you are using to measure them.

2. Visualizing the Three Coordinate Spaces

SwiftUI gives you three distinct maps to choose from when you ask a GeometryProxy for a view's frame via proxy.frame(in:).

🗺️ .global (The Entire Screen)

  • What it measures: The distance from the absolute top-left corner of the physical device screen to your view.
  • When to use it: When you are creating parallax scrolling effects, or checking if a dragging item has collided with a trash can icon at the bottom of the screen.

📦 .local (The Immediate Parent)

  • What it measures: The distance from the top-left corner of the view's direct parent container (like the specific HStack it is trapped inside).
  • When to use it: When you only care about how your view fits inside its local box, completely ignoring where that box happens to sit on the screen.

🏷️ .named("Custom") (A Specific Landmark View)

  • What it measures: The distance from the top-left corner of a specific chosen ancestor view further up the hierarchy.
  • When to use it: When you want to calculate how far a scroll view has moved relative to the entire app page background, ignoring minor inner stacks.

3. Breaking Down the Code Mechanics

To use a custom space, you have to complete a two-step handshake in your code:

  • Name the Landmark and
  • Read the Landmark.

Step 1: Name the Landmark View

You attach the .coordinateSpace modifier to an outer view to tag it as your landmark. 

OuterView()
    .background(.red)
    // We are putting a billboard on OuterView saying: "Your name is 'Custom'"
    .coordinateSpace(name: "Custom")

Step 2: Read the Frame Inside the GeometryReader

Deep inside your layout, you tell your GeometryReader's proxy to look up coordinates using that specific billboard name: 

.onTapGesture {
    // 1. Where is this view relative to the whole screen?
    let globalY = proxy.frame(in: .global).midY
    
    // 2. Where is this view relative to the OuterView landmark?
    let customY = proxy.frame(in: .named("Custom")).midY
    
    // 3. Where is this view relative to its immediate orange container?
    let localY = proxy.frame(in: .local).midY
}

Summary: The Developer's Cheat Sheet

When writing your own UI animations later, memorize this simple guide to choose your coordinate space:

If you want to know... Use this space: Syntax:
"Where is my view on the physical glass of the iPhone?" Global proxy.frame(in: .global)
"Where is my view inside its own immediate container?" Local proxy.frame(in: .local)
"Where is my view relative to a specific outer container?" Custom proxy.frame(in: .named("YourName"))

A Look At midX, midY Properties

.midX and .midY are highly convenient, built-in properties that let you instantly grab the geometric center of a view without doing any manual math.

They are part of a very old, robust suite of tools belonging to a core layout struct in Apple development called CGRect (Core Graphics Rectangle).

Whenever you ask a GeometryReader for a frame by writing proxy.frame(in: .global), it hands you back a CGRect.

Let’s break down exactly what properties are available inside that box.

1. Visualizing the CGRect Geometry Suite

Every frame on an Apple device screen is treated as a 2D bounding box. To make your life easy, Apple populates every single box with 9 standard point properties. You can access any of them instantly using dot-notation.

The Horizontal Axis (X) properties:

  • .minX: The absolute left edge of the view box.
  • .midX: The exact horizontal center point of the view box ((left+right)÷2).
  • .maxX: The absolute right edge of the view box.

The Vertical Axis (Y) properties:

  • .minY: The absolute top edge of the view box.
  • .midY: The exact vertical center point of the view box ((top+bottom)÷2).
  • .maxY: The absolute bottom edge of the view box.

The list contains the 6 foundational coordinate properties of a rectangle.

The Other 3 Properties

The other 3 properties that complete the "suite of 9" are structural shortcuts that define the rectangle's size and origins as a whole.

Here are the final three properties that live inside that same CGRect box:

  • .width: The total horizontal size of the view (calculated automatically as maxX - minX).
  • .height: The total vertical size of the view (calculated automatically as maxY - minY).
  • .origin: A combined coordinate point (CGPoint) containing both the x and y position of the top-left corner. It is a quick shortcut to read .minX and .minY simultaneously.

Summary of the Complete 9-Property Suite

When you ask a GeometryReader for a frame, you get a box equipped with all 9 of these tools ready to use instantly without any math required on your part:

Position Sub-suite Size Sub-suite Combo Shortcut
.minX / .minY (Top-Left) .width (Total Width) .origin (Top-Left Point)
.midX / .midY (Dead Center) .height (Total Height)  
.maxX / .maxY (Bottom-Right)    

2. Deep Dive Into the Lesson's Code Line

Now let’s look at exactly what is happening under the hood on that complex print line from Paul's lesson: 

print("Global center: \(proxy.frame(in: .global).midX) x \(proxy.frame(in: .global).midY)")

Here is the exact step-by-step chain reaction that happens when that line runs:

  • proxy: Inside the GeometryReader, you look at the view's dimensions.
  • .frame(in: .global): You say, "Give me the bounding box (CGRect) of this view relative to the entire phone screen."
  • .midX: SwiftUI calculates the exact horizontal center of that box and spits out a pixel number (e.g., 191.33).
  • .midY: SwiftUI calculates the exact vertical center of that box and spits out another pixel number (e.g., 440.60).

3. Quick Reference: When to use which property?

When you build advanced animations or scrolling effects later on, you will use this suite of properties constantly. Here is a quick cheat sheet of how you will use them in practice: 

let frame = proxy.frame(in: .global)

// Want to anchor an arrow pointing to the top-center of a card?
let topArrowAnchorX = frame.midX
let topArrowAnchorY = frame.minY

// Want to check if a scrolling list has completely disappeared off the top of the screen?
if frame.maxY < 0 { 
    print("The bottom of the view has moved past the top of the screen. It's fully hidden!")
}

// Want to spawn a particle effect popping out of the exact center of a button?
let spawnPoint = CGPoint(x: frame.midX, y: frame.midY)
