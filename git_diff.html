Git Diff

The git diff command is a powerful way to see changes in your working repository that haven't yet been committed. It allows you to review what’s different between commits, the staging area (index), or your working directory, making it an essential tool in your Git workflow. Here are the primary ways to use it:

Compare Working Directory to Staging Area

Command: 

git diff

What it does: This shows the differences between your current working directory and the staging area (i.e., what has not yet been added via git add). It highlights which lines have been modified, added, or removed in each file.

Compare Staging Area to the Last Commit

Command: 

git diff --staged

(or equivalently, git diff --cached)

  • What it does: Use this to review the changes you have staged and are ready to commit. This is useful for a final check before you run git commit.

Compare Two Commits, Branches, or Tags

What it does: This form lets you see the differences between two arbitrary points in your repository’s history.

Command: 

git diff <commit1> <commit2>

Using relative pointers for the last two commits: 

git diff HEAD~1 HEAD

or: 

git diff HEAD^

To get short SHAs (commit ids): 

git log --oneline -n 2

...this prints the last two commits in short form.

You might compare a feature branch with the main branch: 

git diff main feature-branch

This is ideal for code reviews or understanding what changes a branch would introduce if merged.

Compare a File or Directory

Command: 

git diff -- <file_or_directory>
  • What it does: By including a path, you can narrow the diff to changes in a particular file or directory. For example: 
git diff -- path/to/file.txt

This is very handy when you need to inspect changes in a specific part of your project.

Additional Options

  • Ignore Whitespace: Using the -w flag can make the diff ignore changes in whitespace: 
git diff -w
  • Output as a Patch: To save a diff as a patch file (which could be applied later), redirect the output: 
git diff > changes.patch

How to Read the Output

The diff output uses a unified diff format:

  • Lines starting with a minus (-) represent content that was removed.
  • Lines starting with a plus (+) represent content that was added.
  • Unchanged context is shown to help you understand where the changes occur.

This granular review is invaluable for ensuring that only the intended changes are being committed, and that you have full insight into what’s been modified.

Example Workflow

  • Make changes to your files.
  • Run git diff to see what’s been modified in your working directory.
  • Stage some changes with git add <file>.
  • Use git diff --staged to review your staged changes.
  • Finally, commit your changes with git commit -m "Describe what you've changed".

Each of these commands leverages git diff to help you confidently manage and review your changes throughout your development process.

Can you git diff a single file that hasn't been staged?

Yes, you can. To view unstaged changes for a single file in your repository, simply specify the file name with the git diff command. For example, if you want to see the differences for a file named example.txt that hasn't been staged, you can run: 

git diff example.txt

This command shows the differences between your working directory version of example.txt and the version in the index (what you're about to commit). If the file is already tracked by Git and you have unstagged modifications, you'll see a diff detailing the changes.

A couple of notes:

  • Tracked Files: This works for tracked files where changes have been made but not staged. If the file is new and untracked, git diff won’t produce any output unless you force Git to compare it (for instance, by using --no-index).
  • Comparing to HEAD: If you want to see the differences between your current working file and the last committed version (HEAD), you can do: 
git diff HEAD example.txt

This can be useful if you’ve made changes that aren't staged but want to see how they differ from what’s in the latest commit.

Using these commands, you can focus on a specific file to review changes before deciding to add or commit them.

Deciphering Diff Output

The git diff command shows the differences between various states of your files. Understanding its output is crucial for tracking changes in your code.

Example

git diff origin/master

Output Breakdown

Header:

diff --git a/file1 b/file2

File Metadata:

index f3e63d7..e8f44ba 100644

File Markers:

--- a/file1
+++ b/file2

Hunk Header:

@@ -1,8 +1,9 @@

The hunk header is a line in a unified diff output that marks the beginning of a "hunk" (a contiguous block of changes) and provides information about the line numbers and range of the changes in both the original and modified files. It is enclosed by double at signs (@@).

A typical hunk header looks like this:

@@ -m,n +p,q @@ optional context info

Here is a breakdown of the components:

  • @@ ... @@: These symbols act as delimiters that mark the start and end of the hunk header line itself.
  • -m,n: This part describes the original file (the "from" file).
    • m is the starting line number in the original file.
    • n is the number of lines the hunk spans in the original file, including context lines. The comma and n are omitted if the count is only one line.
  • +p,q: This part describes the new, modified file (the "to" file).
    • p is the starting line number in the new file.
    • q is the number of lines the hunk spans in the new file, including context lines. The comma and q are omitted if the count is only one line.
  • optional context info: After the @@ symbols, git diff often includes an excerpt of the code (like a function or class name) to help identify the location and context of the changes, which is a useful aid for human review. Git uses regular expressions to determine this context string, which can be customized via .gitattributes files

Another Run Through of the Hunk Header

In a unified diff output, a hunk header is a line that marks the beginning of a "hunk"—a specific block of changed lines surrounded by context. It serves to orient the user by specifying where the changes occur and, often, providing a textual hint about the surrounding code.

Structure of a Hunk Header

A standard hunk header is enclosed in double at-signs (@@) and follows this format: @@ -start,len +start,len @@ [optional context]

  • -start,len: Refers to the original (old) file.
    • start: The line number where the hunk begins.
    • len: The total number of lines shown for this version, including context and removed lines.
  • +start,len: Refers to the modified (new) file.
    • start: The line number in the new version where the hunk begins.
    • len: The total number of lines in the new version, including context and added lines.
  • [optional context]: Text following the closing @@, which Git or diff generates by searching backward for a representative line, such as a function or class definition, to help you identify where you are in the source code.

Example Breakdown

If you see @@ -24,7 +24,7 @@ class TicketPdf:

  • The original file hunk starts at line 24 and covers 7 lines.
  • The new file hunk starts at line 24 and also covers 7 lines (meaning lines were replaced rather than just added or removed).
  • The change is located within the class TicketPdf block.

Key Details

  • Atomic Changes: A hunk is considered the "atomic element" of a change. In tools like Git, you can choose to "stage" or "discard" individual hunks rather than the entire file.
  • Context Lines: By default, hunks include three lines of unchanged context before and after the actual differences to help the patch tool apply them correctly.
  • Customization: You can use the .gitattributes file to define custom regex patterns for the hunk header context, ensuring it correctly identifies functions in specific languages like Ruby or C++.

Manually Edit A Hunk Header For A Custom Patch

To manually edit a hunk or navigate between them, you primarily use the git add --patch (or git add -p) command. This opens an interactive session where you can handle each change individually.

Navigating Between Hunks

When Git presents a hunk, it provides a prompt like Stage this hunk [y,n,q,a,d,e,?]?. You can use these keys to navigate:

  • y / n: Stage (yes) or skip (no) the current hunk.
  • j / k: Leave this hunk undecided and move to the next (j) or previous (k) undecided hunk.
  • s: Split the current hunk into smaller hunks if they are separated by unchanged lines.
  • g: Go to a specific hunk by number.
  • /: Search for a hunk matching a specific regex.
  • q: Quit the session; nothing further is staged.

Manually Editing a Hunk

If a hunk contains some lines you want and others you don't (and s cannot split them), press e to open your default text editor.

  • To stage a removal: Keep the line starting with -.
  • To NOT stage a removal: Change the - at the start of the line to a space ( ).
  • To stage an addition: Keep the line starting with +.
  • To NOT stage an addition: Delete the entire line starting with +.
  • The Hunk Header: When you save and exit, Git automatically recalculates the line numbers in the hunk header (@@ -start,len +start,len @@) for you. You do not need to manually edit the header numbers unless you are creating a raw patch file outside of Git's interactive tool.

Important Tips

  • Abort Edit: If you make a mistake while editing, delete every line in the editor and save. Git will abort the edit and return you to the prompt.
  • Verify Changes: After staging, use git diff --cached to see exactly what is in your staging area before committing.

Would you like to see how to use git checkout -p to interactively discard changes instead of staging them?

Interactive Change Discards

To interactively discard changes (un-apply them) rather than stage them, use the command git checkout -p (or its modern equivalent, git restore -p).

This process works as a "reverse" of the git add -p workflow you just used.

How it Works

When you run git checkout -p, Git compares your working directory against the index (staged area) and asks if you want to discard specific hunks.

  • y (yes): Discard this hunk. The changes in your file will be permanently overwritten by the version in the index.
  • n (no): Keep this hunk. Your local changes stay exactly as they are.
  • q (quit): Stop the process; no more hunks will be discarded.
  • s (split): Break the hunk into smaller pieces if possible.
  • e (edit): Manually choose which lines to discard.

Editing to Discard (e)

Editing during a checkout -p is the opposite of add -p because you are choosing what to discard:

  • To discard a change: Leave it in the editor as a + or - line.
  • To keep a change: Remove the line from the editor or change it to a context line (starting with a space).

Modern Alternative: git restore

Since Git 2.23, it is recommended to use the git restore command for this purpose, as it separates "file restoration" from "branch switching":

  • Discard unstaged changes: git restore -p
  • Unstage changes: git restore --staged -p

Warning: Changes discarded via checkout -p or restore -p are not stored in Git's history and cannot be easily recovered.
