Git Diff

The git diff command is a powerful way to see changes in your working repository that haven't yet been committed. It allows you to review what’s different between commits, the staging area (index), or your working directory, making it an essential tool in your Git workflow. Here are the primary ways to use it:

Compare Working Directory to Staging Area

Command: 

git diff

What it does: This shows the differences between your current working directory and the staging area (i.e., what has not yet been added via git add). It highlights which lines have been modified, added, or removed in each file.

Compare Staging Area to the Last Commit

Command: 

git diff --staged

(or equivalently, git diff --cached)

  • What it does: Use this to review the changes you have staged and are ready to commit. This is useful for a final check before you run git commit.

Compare Two Commits, Branches, or Tags

What it does: This form lets you see the differences between two arbitrary points in your repository’s history.

Command: 

git diff <commit1> <commit2>

Using relative pointers for the last two commits: 

git diff HEAD~1 HEAD

or: 

git diff HEAD^

To get short SHAs (commit ids): 

git log --oneline -n 2

...this prints the last two commits in short form.

You might compare a feature branch with the main branch: 

git diff main feature-branch

This is ideal for code reviews or understanding what changes a branch would introduce if merged.

Compare a File or Directory

Command: 

git diff -- <file_or_directory>
  • What it does: By including a path, you can narrow the diff to changes in a particular file or directory. For example: 
git diff -- path/to/file.txt

This is very handy when you need to inspect changes in a specific part of your project.

Additional Options

  • Ignore Whitespace: Using the -w flag can make the diff ignore changes in whitespace: 
git diff -w
  • Output as a Patch: To save a diff as a patch file (which could be applied later), redirect the output: 
git diff > changes.patch

How to Read the Output

The diff output uses a unified diff format:

  • Lines starting with a minus (-) represent content that was removed.
  • Lines starting with a plus (+) represent content that was added.
  • Unchanged context is shown to help you understand where the changes occur.

This granular review is invaluable for ensuring that only the intended changes are being committed, and that you have full insight into what’s been modified.

Example Workflow

  • Make changes to your files.
  • Run git diff to see what’s been modified in your working directory.
  • Stage some changes with git add <file>.
  • Use git diff --staged to review your staged changes.
  • Finally, commit your changes with git commit -m "Describe what you've changed".

Each of these commands leverages git diff to help you confidently manage and review your changes throughout your development process.

Can you git diff a single file that hasn't been staged?

Yes, you can. To view unstaged changes for a single file in your repository, simply specify the file name with the git diff command. For example, if you want to see the differences for a file named example.txt that hasn't been staged, you can run: 

git diff example.txt

This command shows the differences between your working directory version of example.txt and the version in the index (what you're about to commit). If the file is already tracked by Git and you have unstagged modifications, you'll see a diff detailing the changes.

A couple of notes:

  • Tracked Files: This works for tracked files where changes have been made but not staged. If the file is new and untracked, git diff won’t produce any output unless you force Git to compare it (for instance, by using --no-index).
  • Comparing to HEAD: If you want to see the differences between your current working file and the last committed version (HEAD), you can do: 
git diff HEAD example.txt

This can be useful if you’ve made changes that aren't staged but want to see how they differ from what’s in the latest commit.

Using these commands, you can focus on a specific file to review changes before deciding to add or commit them.
