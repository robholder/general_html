Swift - @Model Macro

Ai Insight

To understand how SwiftData works, looking at @Query is only seeing half the equation. If @Query is the claw crane that reaches into the machine to pull out your data, @Model is the factory blueprint that defines exactly what those data prizes look like.

However, there is one critical vocabulary distinction to make right out of the gate: @Model is a Macro, not a Property Wrapper. While a property wrapper (like @Query) manages an individual variable inside a view, a macro (like @Model) completely transforms an entire Swift class into a living database table.

Here is the blueprint layout of how you use it to set up your SwiftData storage.

The Anatomy of a @Model Class

To turn any standard Swift class into a managed SwiftData storage object, you simply type @Model directly above the class definition.

Here is a complete, annotated blueprint for a hypothetical Student storage object: 

import Foundation
import SwiftData

@Model 
class Student {
    // 1. Storage attributes (Configurations)
    @Attribute(.unique) var studentID: String
    
    // 2. Standard persistent properties
    var firstName: String
    var lastName: String
    var enrollmentDate: Date
    
    // 3. Relationships (Linking tables together)
    @Relationship(deleteRule: .cascade) var grades: [Grade] = []
    
    // 4. A standard Swift initializer is required
    init(studentID: String, firstName: String, lastName: String, enrollmentDate: Date = .now) {
        self.studentID = studentID
        self.firstName = firstName
        self.lastName = lastName
        self.enrollmentDate = enrollmentDate
    }
}

What @Model Is Doing Behind the Scenes

When the compiler reads that @Model tag, your code-generation robot goes to work. It radically alters how that class behaves in your app's memory:

1. Automatic Table Schema Generation

SwiftData looks at every variable inside your class (String, Date, Int, etc.) and automatically constructs a corresponding column inside a hidden SQLite database file. You don't have to write a single line of SQL database schema setup.

2. Conformance to PersistentModel

The macro forces your class to conform to SwiftData's internal PersistentModel protocol. This injects hidden tracking mechanics into your variables. The exact millisecond you change a student's name in your app (student.firstName = "Alex"), SwiftData detects the mutation and queues it up to be saved to disk automatically.

3. Conformance to Identifiable

It automatically synthesizes a unique internal database identifier (PersistentIdentifier) for every single object instance. This is why you can pass a @Model array straight into a SwiftUI List or ForEach block without having to manually define an id: \.id parameter!

The Companion Companions: Core Companions for @Model

When configuring properties inside your @Model class, you have three major structural modifiers available to fine-tune your storage database engine:

Modifier / Macro What It Instantly Tells the Database Engine
@Attribute Set specific constraints or rules on a column (like .unique for tracking IDs, or .externalStorage for binary image optimization).
@Relationship Connects this table to another class table (e.g., matching a Student to an array of Grade objects). The .cascade rule means if a student is deleted, all their grades are automatically scrubbed too!
@Transient Tells SwiftData to completely ignore this specific property when saving to disk. It lives temporarily in RAM only while the app is actively running.  

How @Model and @Query Talk to Each Other

The entire system relies on these two macros matching perfectly.

When you write @Query var students: [Student], SwiftUI looks at the type inside the array brackets (Student). It checks the database directory for a schema registered under that exact name by the @Model macro, pulls out the data rows, converts them back into native Swift objects, and streams them smoothly into your screen layouts!

Blueprints into Storage

To turn those blueprints into real physical storage on the user's device, you need modelContainer and modelContext. These two components set up the active database factory environment that links @Model and @Query together.

The SwiftData Trinity

To make SwiftData work in any app, you need three interconnected components. Think of them as a factory hierarchy:

  • @Model (The Blueprint): Defines what the data structure looks like.
  • ModelContainer (The Factory Building): The actual persistent storage database file on the hard drive. It reads your blueprints and creates the real tables on disk.
  • ModelContext (The Assembly Line Worker): The active, running memory space where you add, edit, or delete items. It watches for changes and pushes them into the Container.

Step 1: Setting up the Factory (.modelContainer)

Before any view can use @Query to fetch data, you must initialize the backend database file. You do this at the absolute root of your app using the .modelContainer(for:) view modifier.

Open your main app file (the one with the @main attribute) and pass your @Model classes into the window group: 

import SwiftData
import SwiftUI

@main
struct StudentTrackerApp: App {
    var body: some Scene {
        WindowGroup {
            ContentView()
                // This builds the database file on disk for the Student blueprint
                .modelContainer(for: Student.self) 
        }
    }
}

What this modifier does automatically:

  • It looks at the Student class layout and provisions a physical SQLite database file inside your app's protected sandbox directory.
  • It sets up an active ModelContext assembly line.
  • It injects that entire database environment deep into the SwiftUI environment tree, making it available to ContentView and every single subview down the line.

Step 2: Modifying Data via the Assembly Line (modelContext)

Once the container is running at the top of your app, @Query will seamlessly start pulling data out of it. But what happens when you want to add a new student or delete an existing one?

You need to call upon the context (the worker). You grab the active context out of SwiftUI's environment using the \.modelContext key path: 

import SwiftData
import SwiftUI

struct ContentView: View {
    // 1. Grab the active worker out of the environment
    @Environment(\.modelContext) var modelContext
    
    // 2. Stream the live data out of the container
    @Query var students: [Student]

    var body: some View {
        NavigationStack {
            List(students) { student in
                Text("\(student.firstName) \(student.lastName)")
            }
            .toolbar {
                Button("Add Student", systemImage: "plus") {
                    addExampleStudent()
                }
            }
        }
    }

    func addExampleStudent() {
        // Create a new instance using your @Model blueprint
        let newStudent = Student(studentID: "123", firstName: "Harry", lastName: "Potter")
        
        // 3. Hand the new object to the worker to insert it into the database container
        modelContext.insert(newStudent)
        
        // No need to call a "save()" method! The context automatically syncs 
        // changes to disk periodically behind the scenes.
    }
}

Tracking Multiple Classes With SwiftData

The Code Syntax

Instead of passing a single class like User.self, you pass an array literal [User.self, Post.self, Order.self] like this: 

import SwiftUI
import SwiftData

@main
struct MyApp: App {
    var body: some Scene {
        WindowGroup {
            ContentView()
        }
        // Pass all of your @Model classes inside an array here:
        .modelContainer(for: [User.self, Post.self, Order.self])
    }
}

Pro-Tip: The "Relationship" Shortcut

SwiftData is smart. If your models have relationships with one another (for example, if your User class has a property that links to a Post class), you only need to list the parent class. 

@Model
class User {
    var id: String
    var posts: [Post] // Relationship links User to Post
}

@Model
class Post {
    var title: String
}

If your models are linked like the example above, you can actually just write: 

.modelContainer(for: User.self)

SwiftData will automatically scan the User class, see that it relies on Post, and implicitly load the Post table into the exact same database container for you!

However, explicitly listing them all in an array (like the first example) is considered best practice because it makes it obvious to anyone reading the code exactly what models the app is managing.

Summary Checklist

Without setting up these layers, trying to use @Query will cause your app to crash immediately with a message like: "No ModelContext found in the environment."

  • @Model: Put this above your class to turn it into a database blueprint.
  • .modelContainer(for:): Attach this to your main App struct file window to spin up the actual database engine on disk.
  • @Environment(\.modelContext): Pull this variable into any view where you need to manually .insert() or .delete() data entries!
