Hacking With Swift - 165 - Challenge 2 - Adding a sort menu

Adding a Sort Menu to the iExpense App

Starting state

ContentView.swift

import SwiftData
import SwiftUI

@Model
class ExpenseItem {
    
    var name: String
    var type: String
    var amount: Double
    
    init(name: String, type: String, amount: Double) {
        self.name = name
        self.type = type
        self.amount = amount
    }
}


struct ExpenseRow: View {
    let item: ExpenseItem
    
    var amountColor: Color {
        switch item.amount {
        case ..<10: return .primary
        case ..<100: return .green
        default: return .red
        }
    }
    
    var body: some View {
        HStack {
            VStack(alignment: .leading) {
                Text(item.name)
                    .font(.headline)
                Text(item.type)
                    .font(.subheadline)
                    .foregroundStyle(.secondary)
            }
            
            Spacer()
            
            Text(item.amount, format: .currency(code: Locale.current.currency?.identifier ?? "USD"))
                .foregroundStyle(amountColor)
        }
    }
}

struct ContentView: View {
    @Environment(\.modelContext) var modelContext
    @Query var items: [ExpenseItem]
    
    var personalExpenses: [ExpenseItem] {
        items.filter { $0.type == "Personal" }
    }
    var businessExpenses: [ExpenseItem] {
        items.filter { $0.type == "Business" }
    }
    
    var body: some View {
        NavigationStack {
            List {
                Section("Personal") {
                    ForEach(personalExpenses) { item in
                        ExpenseRow(item: item)
                    }
                    .onDelete { offsets in
                        removeItems(at: offsets, in: personalExpenses)
                    }
                }
                
                Section("Business") {
                    ForEach(businessExpenses) { item in
                        ExpenseRow(item: item)
                    }
                    .onDelete { offsets in
                        removeItems(at: offsets, in: businessExpenses)
                    }
                }
            }
            .navigationTitle("iExpense")
            .toolbar {
                NavigationLink {
                    AddView()
                } label: {
                    Label("Add Expense", systemImage: "plus")
                }
            }
        }
    }
    
    func removeItems(at offsets: IndexSet, in source: [ExpenseItem]) {
        for offset in offsets {
            let item = source[offset]
            modelContext.delete(item)
        }
    }
}

Adding the Sort Functionality

We Need a New ExpenseListView

We need to extract the List from ContentView to a new ExpenseListView to control the sorting facility.

We'll end up with two Views:

  • ContentView
  • ExpenseListView

Here is a conceptual guide to understanding why we have to refactor the view, followed by a step-by-step strategic checklist you can reuse in future projects.

The Core Strategy: Why Two Views?

The problem boils down to a strict limitation in SwiftUI: @Query properties are locked in place once a View is created. They cannot listen to changes from a @State variable inside the same struct.

To bypass this roadblock, we use an architectural pattern called View Extraction.

By splitting the code into a Parent View (ContentView) and a Child View (ExpenseListView), we change the data flow:

  • The Parent tracks what the user clicked using @State.
  • When the user changes the sort picker, the Parent forces the Child to completely destroy and recreate itself.
  • During recreation, the Child's custom init() function interceptively rebuilds the @Query with the updated rules before any data hits the screen.

Step-by-Step Refactoring Checklist

Whenever you need to add dynamic sorting or filtering to a SwiftData @Query, use this exact 4-step workflow:

Step 1: Create the Child Struct

Create an entirely new View structure. Cut the data-dependent UI elements (usually the List) out of your main view and paste them into this new subview.

Step 2: Define the Child's Custom Initializer

Instead of declaring a standard @Query, you leave the query uninitialized. You use a custom init method to receive the sorting rule from the parent and inject it straight into the underlying property wrapper using the underscore (_items) syntax. 

struct MyListView: View {
    @Query var items: [ExpenseItem] // 1. Uninitialized query
    
    init(sortOrder: [SortDescriptor<ExpenseItem>]) {
        // 2. Underscore targets the Query wrapper itself, configuring it dynamically
        _items = Query(sort: sortOrder) 
    }
    // ... UI goes here
}

Step 3: Set Up Parent State and Menu UI

Inside your main ContentView, create a @State variable containing a default SortDescriptor. Build your Menu and Picker in the toolbar, assigning corresponding array tags to match your data types. 

@State private var sortOrder = [SortDescriptor(\ExpenseItem.name)]

// Picker Items use arrays matching the State type:
Text("Sort by Amount").tag([SortDescriptor(\ExpenseItem.amount)])

Step 4: Pass State from Parent to Child

Finally, drop your newly created child view into the body of your parent view, passing the @State variable forward as a normal parameter. 

var body: some View {
    NavigationStack {
        MyListView(sortOrder: sortOrder) // State automatically flows downward
    }
}

Why this is highly repeatable

This layout scales cleanly. If you later decide to add a Search Bar or a Category Filter, you do not need to rewrite your views again. You simply expand the child view's initializer to accept a filter string or predicate alongside the sorting rules, allowing ContentView to remain a lightweight control dashboard while the subview handles the database heavy lifting.

In One Refactored File

import SwiftData
import SwiftUI

// ==========================================
// DATA MODEL
// ==========================================
@Model
class ExpenseItem {
    var name: String
    var type: String
    var amount: Double
    
    init(name: String, type: String, amount: Double) {
        self.name = name
        self.type = type
        self.amount = amount
    }
}

// ==========================================
// ROW VISUAL COMPONENT
// ==========================================
struct ExpenseRow: View {
    let item: ExpenseItem
    
    var amountColor: Color {
        switch item.amount {
        case ..<10: return .primary
        case ..<100: return .green
        default: return .red
        }
    }
    
    var body: some View {
        HStack {
            VStack(alignment: .leading) {
                Text(item.name)
                    .font(.headline)
                Text(item.type)
                    .font(.subheadline)
                    .foregroundStyle(.secondary)
            }
            
            Spacer()
            
            Text(item.amount, format: .currency(code: Locale.current.currency?.identifier ?? "USD"))
                .foregroundStyle(amountColor)
        }
    }
}

// ==========================================
// PARENT VIEW (Step 3 & 4)
// Acts as the control dashboard for State and UI controls
// ==========================================
struct ContentView: View {
    // STEP 3a: Define a @State variable to store the active sorting rule.
    // We use an array of SortDescriptors to allow for fallback sorting (e.g., if amounts are equal, sort by name).
    @State private var sortOrder = [
        SortDescriptor(\ExpenseItem.name),
        SortDescriptor(\ExpenseItem.amount)
    ]
    
    var body: some View {
        NavigationStack {
            // STEP 4: Inject the child view into the hierarchy, passing our active @State forward.
            // Whenever 'sortOrder' changes here, SwiftUI automatically recreates ExpenseListView.
            ExpenseListView(sortOrder: sortOrder)
                .navigationTitle("iExpense")
                .toolbar {
                    NavigationLink {
                        AddView()
                    } label: {
                        Label("Add Expense", systemImage: "plus")
                    }
                    
                    // STEP 3b: Build the Menu and Picker UI.
                    // The Picker selection binds directly to our @State property.
                    Menu("Sort", systemImage: "arrow.up.arrow.down") {
                        Picker("Sort", selection: $sortOrder) {
                            // The .tag array types must match the type of our @State property exactly.
                            Text("Sort by Name")
                                .tag([
                                    SortDescriptor(\ExpenseItem.name),
                                    SortDescriptor(\ExpenseItem.amount),
                                ])
                            Text("Sort by Amount")
                                .tag([
                                    SortDescriptor(\ExpenseItem.amount),
                                    SortDescriptor(\ExpenseItem.name)
                                ])
                        }
                    }
                }
        }
    }
}

// ==========================================
// CHILD VIEW (Step 1 & 2)
// Handles the database query execution and data layout
// ==========================================
struct ExpenseListView: View {
    @Environment(\.modelContext) var modelContext
    
    // STEP 1a: Cut the original @Query out of ContentView and paste it here.
    // Leave it uninitialized so it can be dynamically configured in the init method.
    @Query var items: [ExpenseItem]
    
    // STEP 2: Create a custom initializer to intercept the sorting data from the parent.
    init(sortOrder: [SortDescriptor<ExpenseItem>]) {
        // The underscore (_) bypasses the wrapped array data and targets the underlying Query structure itself.
        // This is where the live database stream gets configured with the fresh sorting rules.
        _items = Query(sort: sortOrder)
    }
    
    // STEP 1b: Paste the data-dependent computed properties inside the child view.
    var personalExpenses: [ExpenseItem] {
        items.filter { $0.type == "Personal" }
    }
    var businessExpenses: [ExpenseItem] {
        items.filter { $0.type == "Business" }
    }
    
    var body: some View {
        // STEP 1c: Paste the original List structural layout here.
        List {
            Section("Personal") {
                ForEach(personalExpenses) { item in
                    ExpenseRow(item: item)
                }
                .onDelete { offsets in
                    removeItems(at: offsets, in: personalExpenses)
                }
            }
            
            Section("Business") {
                ForEach(businessExpenses) { item in
                    ExpenseRow(item: item)
                }
                .onDelete { offsets in
                    removeItems(at: offsets, in: businessExpenses)
                }
            }
        }
    }
    
    // STEP 1d: Paste the database mutator function inside the child view.
    func removeItems(at offsets: IndexSet, in source: [ExpenseItem]) {
        for offset in offsets {
            let item = source[offset]
            modelContext.delete(item)
        }
    }
}

Previews

The additions to the #Preview blocks (from those showing in Paul Hudson's examples so far) are the difference between a preview canvas that crashes or stays blank, and one that actually simulates your app flawlessly while you type.

There are two major improvements added here: Naming the Canvas and Injecting an In-Memory Database.

1. Naming the Canvas (The String Labels)

By adding a string like "Main Dashboard" right next to #Preview, you are giving that specific preview block a custom title.

When your file starts growing and contains multiple preview blocks (like having one for the full view and one for just the list row), Xcode will display tabs or titles at the top of your canvas pane. Labeling them makes it incredibly easy to switch between different view states without guessing which preview is which.

2. The Database Injector: .modelContainer(..., inMemory: true)

This is the most critical addition. If you try to run your original preview blocks, Xcode will likely throw a preview crash or display a blank screen.

The Logic Behind It

Remember earlier when we discussed how @Query and @Environment(\.modelContext) work? They rely entirely on a database container existing at the root of the app (iExpensesApp.swift).

However, the Xcode Preview canvas doesn't run your App struct. It treats ContentView or ExpenseListView as the absolute root of the universe.

Because of this, when your code hits @Query or tries to fetch the modelContext, it looks up into that "environment bucket"—and finds nothing. The app crashes because the database layer doesn't exist inside the preview sandbox.

What .modelContainer(..., inMemory: true) solves:

By attaching this modifier to the view inside the preview block, you are manually dropping a temporary database into the environment bucket just for the canvas.

  • for: ExpenseItem.self: Tells the preview canvas to prepare tables for your expense objects.
  • inMemory: true: This is a brilliant safety setting. It tells SwiftData to build the database completely inside the computer's temporary RAM, not on the hard drive.

Why inMemory: true is crucial for development:

If you don't use inMemory: true, your preview engine will attempt to read and write to your actual simulator's database file. This can pollute your real testing data with weird "dummy text" you typed while testing layouts, or cause file-locking conflicts if your simulator is running at the same time.

With an in-memory container, every time the canvas refreshes, you get a pristine, completely blank slate that vanishes the second you close Xcode.

Summary Checklist for Future Previews

Whenever you are creating a SwiftUI file that utilizes SwiftData (@Query or modelContext), your preview blocks must mirror this exact strategy:

  • Always pass dependencies: If your child view requires initialization variables (like sortOrder), pass sample ones into the preview call.
  • Always attach a container: Append .modelContainer(for: YourClass.self, inMemory: true) to ensure the preview engine doesn't crash from an empty environment bucket.
