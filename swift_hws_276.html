Hacking With Swift - 276 - Challenge 3

Challenge 3

  • For a real challenge make the views change color as you scroll. For the best effect, you should create colors using the Color(hue:saturation:brightness:) initializer, feeding in varying values for the hue.

Ai Insight

For this challenge, we can ditch the static colors array entirely. The Color(hue:saturation:brightness:) initializer takes values from 0.0 to 1.0. Conveniently, you already have a perfect value that ranges from 0.0 to 1.0 dynamically as you scroll: your screen ratio (midY / fullView.size.height).

By feeding that exact ratio directly into the hue parameter, the row's color will smoothly shift through the entire rainbow spectrum based on where it sits on the screen.

The Solution

We just need to replace your .background(colors[index % 7]) modifier with a dynamic background calculation: 

struct ContentView: View {
    // We don't need the hardcoded colors array anymore!
    
    var body: some View {
        GeometryReader { fullView in
            ScrollView(.vertical) {
                VStack(spacing: 0) {
                    ForEach(0..<50) { index in
                        GeometryReader { proxy in
                            // 1. Calculate the Y-position ratio (0.0 at top, 1.0 at bottom)
                            let currentMidY = proxy.frame(in: .global).midY
                            let screenHeight = fullView.size.height
                            let positionRatio = currentMidY / screenHeight
                            
                            // 2. Generate the dynamic color based on the scroll position
                            let dynamicColor = Color(
                                hue: Double(positionRatio),
                                saturation: 0.8,
                                brightness: 0.9
                            )
                            
                            Text("Row #\(index)")
                                .font(.title)
                                .frame(maxWidth: .infinity)
                                // 3. Apply the dynamic color here
                                .background(dynamicColor)
                                .scaleEffect(
                                    CGFloat(0.5 + (0.5 * positionRatio))
                                )
                                .rotation3DEffect(
                                    .degrees(Double(currentMidY - screenHeight / 2) / 3),
                                    axis: (x: 0, y: 1, z: 0)
                                )
                                .opacity(
                                    max(0.0, 1.0 - Double(abs(currentMidY - screenHeight / 2) / 200.0))
                                )
                        }
                        .frame(height: 40)
                    }
                }
            }
        }
    }
}

Why This Works Beautifully

  • Clean Code: Creating currentMidY, screenHeight, and positionRatio variables at the top of the GeometryReader means we can reuse them across all your modifiers. It keeps the math readable instead of constantly calling proxy.frame(in: .global).midY.
  • The Color Cycle: In the HSB (Hue, Saturation, Brightness) color space, a hue of 0.0 is red, 0.3 is green, 0.6 is blue, and it wraps right back around to red at 1.0. As you scroll a row from the bottom to the top, it will shift dynamically across the entire rainbow spectrum!

QUESTION

Is any way of extracting those out of the view?

I've been led to consider taking math OUT of the ContentView struct/view, perhaps by making a dedicated struct/view that can be placed as a view within the ContentView (to simplify the view body).

Am I correct in assuming this is desirable? If so, how can we best achieve it?

ANSWER

You are 100% correct in assuming this is desirable. In fact, it's one of the best habits you can form in SwiftUI.

When a single view handles layout, scrolling, geometry math, text rendering, and style mutations all in one place, it turns into what developers call a "Massive View." It makes the code harder to read, harder to debug, and forces SwiftUI to do extra work tracking body changes.

By extracting the row into its own dedicated subview, we achieve two major wins:

  • Separation of Concerns: ContentView handles the scrolling list container. The subview handles the visual presentation of a single row.
  • Readability: Your code instantly reads like a clean layout blueprint.

Here is the best way to refactor this by creating a dedicated ScrollRow subview.

The Refactored Solution

import SwiftUI

struct ContentView: View {
    var body: some View {
        GeometryReader { fullView in
            ScrollView(.vertical) {
                VStack(spacing: 0) {
                    ForEach(0..<50) { index in
                        // We pass the parent container's total height down to each row
                        ScrollRow(index: index, parentHeight: fullView.size.height)
                            .frame(height: 40)
                    }
                }
            }
        }
    }
}

// MARK: - Extracted Subview
struct ScrollRow: View {
    let index: Int
    let parentHeight: CGFloat
    
    var body: some View {
        GeometryReader { proxy in
            // All the heavy math is cleanly calculated right here
            let currentMidY = proxy.frame(in: .global).midY
            let positionRatio = currentMidY / parentHeight
            
            let dynamicColor = Color(
                hue: Double(positionRatio),
                saturation: 0.8,
                brightness: 0.9
            )
            
            Text("Row #\(index)")
                .font(.title)
                .frame(maxWidth: .infinity)
                .background(dynamicColor)
                .scaleEffect(
                    CGFloat(0.5 + (0.5 * positionRatio))
                )
                .rotation3DEffect(
                    .degrees(Double(currentMidY - parentHeight / 2) / 3),
                    axis: (x: 0, y: 1, z: 0)
                )
                .opacity(
                    max(0.0, 1.0 - Double(abs(currentMidY - parentHeight / 2) / 200.0))
                )
        }
    }
}

#Preview {
    ContentView()
}

Why this approach works best

  • Passing the Parent Height: The inner GeometryReader inside ScrollRow only knows about its own tiny frame (40 points high). It has no idea how big the screen is. By passing fullView.size.height as a dependency (parentHeight), the subview has everything it needs to calculate the perfect screen ratio.
  • Encapsulation: Notice how ContentView is now incredibly clean. If you ever want to change how the text looks, add an image, or adjust the animations, you only have to look at the isolated ScrollRow struct.
