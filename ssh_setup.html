Setting Up SSH

The following steps, based on GitHub Docs, will guide you through creating an SSH key and adding it to your GitHub account. 

Step 1: Open Terminal 

Open the Terminal application on your Mac (you can use Spotlight search by pressing Command + Space and typing "Terminal"). 

Step 2: Generate a New SSH Key 

In the Terminal, run the following command to generate a new SSH key pair, replacing "your_email@example.com" with your email address associated with your GitHub account:  

ssh-keygen -t ed25519 -C "your_email@example.com"

• When prompted to "Enter a file in which to save the key," press Enter to accept the default file location (/Users/YOU/.ssh/id_ed25519). • When prompted for a passphrase, you can type a secure passphrase or press Enter to leave it empty (for no passphrase). Using a passphrase adds more security but requires you to enter it whenever you use the

Step 3: Add Your SSH Key to the ssh-agent 

The ssh-agent manages your keys and remembers your passphrase. 

  • Start the ssh-agent in the background: 
eval "$(ssh-agent -s)"

Add your private key to the ssh-agent

If you set up a passphrase (I normally don't), store your passphrase in the keychain using the --apple-use-keychain option: 

ssh-add --apple-use-keychain ~/.ssh/id_ed25519

If you did not use a passphrase, you can run the command without the --apple-use-keychain option: 

ssh-add ~/.ssh/id_ed25519

Step 4: Copy Your Public Key to the Clipboard 

Display the content of your public key file and copy it using the pbcopy command:  

pbcopy < ~/.ssh/id_ed25519.pub

This command copies the entire content of the id_ed25519.pub file to your clipboard. 

Step 5: Add the Public Key to Your GitHub Account 

  • Navigate to GitHub and sign in to your account.
  • Click your profile picture in the upper-right corner, then click Settings.
  • In the left sidebar, click SSH and GPG keys under the "Access" section.
  • Click the New SSH key or Add SSH key button.
  • In the "Title" field, add a descriptive label for the new key (e.g., "Personal MacBook Pro").
  • In the "Key" field, paste your public key from the clipboard.
  • Click Add SSH key. You may need to confirm access to your

Step 6: Test Your SSH Connection 

Finally, verify the connection by running the following command in Terminal:  

ssh -T git@github.com

You should receive a message confirming that you have successfully authenticated to GitHub, which does not provide shell access.

If you get an error featuring a line like this: 

Error
The authenticity of host 'github.com (140.82.113.4)' can't be established.

Do this: 

ssh-keyscan github.com >> ~/.ssh/known_hosts

Then repeat: 

ssh -T git@github.com

You might get something like: 

Hi <username>! You've successfully authenticated, but GitHub does not provide shell access.

Optional Config File

To use the keychain setup above, and not be prompted each time for a passphrase, make this config file: 

~/.ssh/config

...and add this to the contents: 

# Rename <key_name> to the ssh key/file:
Host *
  UseKeychain yes
  AddKeysToAgent yes
  IdentityFile ~/.ssh/<key_name>

More on Know Hosts

When you connect to a new server, SSH checks its "fingerprint" against your ~/.ssh/known_hosts file, you will see the standard security prompt: 

The authenticity of host 'example.com' can't be established... Are you sure you want to continue connecting (yes/no/[fingerprint])?

How to automate the known_hosts part (Optional)

If you want to bypass that prompt as well (typically for scripts or trusted internal networks), you would need to add one of these lines to your Host * block:sh 

StrictHostKeyChecking accept-new: Automatically adds the key for any new host you connect to, but will still alert/block you if an existing host's key changes.

StrictHostKeyChecking no: Automatically adds any new key and ignores changes to old ones. This is less secure as it makes you vulnerable to man-in-the-middle attacks.

Managing multiple SSH keys and Git accounts

It can sometimes be useful to have multiple SSH keys for the same Git hosts, which requires some additional SSH configuration.

Example

A typical example involves using the same host (GitHub) with multiple accounts on the same computer. Since each account should use its own SSH key, we need to use a combination of the SSH configuration and Git remotes to determine which SSH key to use.

Configuration

1. Generate a unique SSH key for each use case

SSH keys should follow the following naming format: [Host]-[Use] (ie. github-personal).

  • Host - Host the key will be associated with
    • ie. github for GitHub repositories
  • Use - What kind of work the key will be used for
    • ie. personal for personal work 
~/.ssh/
  id_rsa.github-personal
  id_rsa.github-personal.pub

Keys can be generated using the following command: ssh-keygen -t rsa -C "[EMAIL_ADDRESS]" -b 4096

Add the key to the ssh agent: 

eval $(ssh-agent -s)
ssh-add ~/.ssh/other_id_rsa

2. Modify your SSH config file (typically ~/.ssh/config) with the required hosts and identity files

The SSH config file is typically located at ~/.ssh/config.

  • Host - Host name for referencing later (in Git config)
  • HostName - Actual host name
  • User - User who will be using the credentials
  • IdentityFile - SSH private key file
  • IdentitiesOnly - Prevents falling back to the default key 
# ~/.ssh/config

Host github-personal
  HostName github.com
  User git
  IdentityFile ~/.ssh/id_rsa.github-personal
  IdentitiesOnly yes

Host github-work
  HostName github.com
  User git
  IdentityFile ~/.ssh/id_rsa.github-work
  IdentitiesOnly yes

3. Modify the Git remote urls

The new SSh configuration will be used when attempting to connect to a Git remote, meaning that if the Git remote URL is updated to match the configuration's Host parameter it will use the nested information. 

remote["origin"]
  git@github-personal:username/repository.git

4. Add the SSH Key to Remote

SSH keys should follow the following naming format when added to the remote: [Host]-[Use]-[Location] (ie. github-personal-macbook).

  • Host - Host the key will be associated with
    • ie. github for GitHub repositories
  • Use - What kind of work the key will be used for
    • ie. personal for personal work
  • Location - Physical location of the key (one place only)
    • ie. macbook for key created on a MacBook

Naming SSH Keys (Directly Or Renaming)

You can rename SSH keys at any time, and you can also specify a custom name the moment you create them.

1. Naming keys during creation

When you run ssh-keygen, it will prompt you: Enter a file in which to save the key. You can type a custom path and name there.

Alternatively, you can skip the prompt entirely by using the -f (file) flag to name it directly in the command: 

ssh-keygen -t ed25519 -f ~/.ssh/my_custom_key_name

This will automatically create two files: my_custom_key_name (private) and my_custom_key_name.pub (public).

2. Renaming existing keys

SSH keys are just standard files. You can rename them using the mv (move) command in your terminal.

Important: You must rename both the private and public key files to the same base name for them to continue working together easily. 

# Rename the private key
mv ~/.ssh/id_rsa ~/.ssh/new_key_name

# Rename the public key
mv ~/.ssh/id_rsa.pub ~/.ssh/new_key_name.pub

3. Critical follow-up steps

Renaming or using custom names "breaks" the default behavior where SSH automatically looks for id_rsa. To make your custom-named keys work:

  • Update your Config:
    • You must update the IdentityFile path in your ~/.ssh/config to point to the new filename.
  • Update the Agent:
    • If you use ssh-agent, you may need to re-add the key using its new name: ssh-add ~/.ssh/new_key_name.
  • Permissions: Ensure the private key still has restricted permissions (usually 600) after moving or renaming it.

Permissions

To ensure SSH doesn't reject your renamed keys for being "too open," you need to set strict permissions. SSH will refuse to use a private key if it's readable by other users on your system.

1. Fix Permissions for All Keys

Run these commands in your terminal to secure the entire folder and its contents: 

# Set the .ssh folder so only YOU can read/write/enter it
chmod 700 ~/.ssh

# Set all PRIVATE keys so only YOU can read/write them
# (This handles all files in the folder; SSH will ignore the .pub files)
chmod 600 ~/.ssh/*

2. Verify the Permissions

You can check if it worked by listing the files with ls -l ~/.ssh. You want to see -rw------- for your private keys: 

ls -l ~/.ssh
# Output should look like this:
# -rw-------  1 yourname  staff  2602 Feb 21 19:00 my_custom_key_name
# -rw-r--r--  1 yourname  staff   571 Feb 21 19:00 my_custom_key_name.pub

3. Clear and Re-add to Agent (The "Fresh Start")

If you renamed your keys, the ssh-agent might still be looking for the old names. Clear it and add your new ones: 

# Remove all identities currently in the agent
ssh-add -D

# Add your specific new key (and save to macOS Keychain)
ssh-add --apple-use-keychain ~/.ssh/my_custom_key_name

Pro Tip: The "Why"

  • 600 (-rw-------): Means Read & Write for you, and Nothing for everyone else.
  • 700 (-drwx------): Means Read, Write, & Execute (access) for you, and Nothing for everyone else.

If you ever see the error "Permissions 0644 for '~/.ssh/id_rsa' are too open", these commands are your immediate fix.
