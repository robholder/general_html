Swift - Result

Understanding the Result Type in Swift

In Swift, functions that can fail usually use the throws keyword, forcing you to handle errors immediately using a do-catch block. However, there are times when you don't want to handle an error right away - you want to store the success or failure in a variable and pass it around your app to be unpacked later.

This is exactly what the Result type is for. It captures either a successful value or a failure error into a single, neat package.

What is the Result Type?

At its core, Result is a standard library enum with two cases:

  • .success(Value) – Contains the data you actually wanted.
  • .failure(Error) – Contains the error that occurred.

Because it is an enum, it enforces compile-time safety. You cannot accidentally access the success data if the operation failed, and you cannot ignore the error if it occurred. 

// Under the hood, Swift defines it like this:
enum Result<Success, Failure: Error> {
    case success(Success)
    case failure(Failure)
}

Why and When to Use It

While Swift's modern async/await has reduced the need for Result in simple network calls, Result is still incredibly useful in specific scenarios:

  • Asynchronous Completion Handlers: Passing the outcome of a background task back to the main thread (very common in older Swift code and external libraries).
  • Pre-loading and Storing States: Saving the outcome of an operation in a @State variable to dictate what the UI should look like (e.g., showing a loading spinner, an error view, or the actual data).
  • Delayed Error Handling: Catching an error now, but waiting to process or present it to the user until a later time.

The Fundamental Lifecycle: Creating and Unpacking

Using Result always follows a three-step pattern: Define, Generate, and Unpack.

1. Generating a Result

You can manually create a Result by returning .success or .failure. Alternatively, Swift allows you to initialize a Result directly from a throwing expression using Result { try ... }.

2. Unpacking a Result

To get the data out of a Result, you must "unwrap" it. This is most commonly done using a switch block (the cleanest and most idiomatic way) or a do-catch block combined with .get().

Here is a complete, standalone example demonstrating both methods. 

import Foundation

// Define a custom error type
enum NetworkError: Error {
    case badConnection
    case dataCorrupted
}

// 1. A function that returns a Result type
func fetchUserData(isOnline: Bool) -> Result<String, NetworkError> {
    if isOnline {
        return .success("User Profile: Jane Doe")
    } else {
        return .failure(.badConnection)
    }
}

// Generate a result instance
let fetchOutcome = fetchUserData(isOnline: false)

Unpacking Method A: The switch Block (Recommended)

The switch statement is the preferred way to unpack a Result because it forces you to handle both cases explicitly. 

switch fetchOutcome {
case .success(let profileData):
    print("Success! Data received: \(profileData)")
    
case .failure(let error):
    switch error {
    case .badConnection:
        print("Please check your internet connection.")
    case .dataCorrupted:
        print("The data received from the server was bad.")
    }
}

Unpacking Method B: The do-catch Block

If you prefer not to use a switch statement, Result has a built-in .get() method. This method will either instantly return the success value, or it will throw the error, allowing you to use a standard do-catch routine. 

do {
    // .get() will either extract the string or throw the NetworkError
    let profileData = try fetchOutcome.get()
    print("Success! Data received: \(profileData)")
} catch let error as NetworkError {
    print("Failed with network error: \(error)")
} catch {
    print("An unknown error occurred: \(error.localizedDescription)")
}

The Big Picture: Optionals vs. Result

Think of an Optional as a mystery gift box. When you open it, you either get a prize (.some), or the box is completely empty (.none / nil). It tells you that something failed, but it can never tell you why.

A Result box is the upgraded version. It can never be empty. When you open a Result box, you are guaranteed to find one of two things:

  • .success: A shiny green card with your data (like a String or an Int).
  • .failure: A bright red card explaining exactly what went wrong (the specific error).
Box Type What's inside Case A? What's inside Case B? Syntactic Sugar?
Optional A real value Absolutely nothing (nil) Yes (if let, ??, ?.)
Result A real value A specific error message No (Must use switch or methods)
Note: The text mentions "syntactic sugar" - that just means Swift gives optionals nice shortcuts like if let because we use them constantly. Result doesn't get those shortcuts, so it feels a bit more manual.

Feature 1: Turning a Result into a do/catch (The get() method)

Sometimes you have a Result box, but you prefer working with standard Swift error handling (do, try, catch).

You can rip open the box using .get().

  • If the box contains a .success, .get() hands you the value directly.
  • If the box contains a .failure, .get() immediately throws the error, kicking you straight into the catch block. 
// Opening the box with get()
do {
    let finalData = try myResultBox.get() 
    print("Success: \(finalData)")
} catch {
    print("The box contained an error, so I ended up here!")
}

Feature 2: Turning a throwing function into a Result

You can also go the exact opposite direction. If you have a sketchy function that throws errors, you can wrap it in a Result box right at the birth site. 

// This automatically creates a Result box for you
let myResultBox = Result { try String(contentsOf: websiteURL) }

Swift runs the code inside the braces. If the website loads, it automatically labels the box .success(text). If the internet drops, it catches the error and labels the box .failure(internetError). No do/catch required on this line!

Feature 3: Transforming inside the box (map and mapError)

Just like arrays, Result boxes have a map() tool. It allows you to modify what's inside the box without actually opening it yet.

map() changes the Success

Imagine you have a Result box containing the string "100". You want to turn it into the integer 100.

  • If it's a .success: map reaches inside, converts "100" to 100, and seals the box back up as a success.
  • If it's a .failure: map looks inside, sees an error, closes the lid, and ignores your conversion entirely. The error remains untouched.

Use map when you want to change the successful data inside the box, but leave any errors completely untouched. 

// A Result box containing a String ("42"), or a NetworkError
let stringResult: Result<String, NetworkError> = .success("42")

// map() reaches inside and converts the String "42" into the Int 42
let integerResult: Result<Int, NetworkError> = stringResult.map { stringValue in
    Int(stringValue) ?? 0
}

// NOTE: If stringResult had been a .failure, map() would have ignored 
// the integer conversion entirely and just passed the error along.

mapError() changes the Failure

This does the exact opposite. It ignores successful data, but if it finds an error, it lets you translate a messy system error (like Code 404) into a clean, readable app error (like .pageNotFound).

Use mapError when you don't care about the successful data, but you want to take a messy, low-level error and translate it into a clean, custom error that your app understands. 

// Custom, clean errors for our app's UI
enum AppUIError: Error {
    case connectionLost
}

// A Result box containing a String, but holding a messy system URLSession error
let systemResult: Result<String, Error> = .failure(URLError(.notConnectedToInternet))

// mapError() ignores success, but catches the system error and swaps it out
let cleanResult: Result<String, AppUIError> = systemResult.mapError { systemError in
    // Turn any internet-related system error into our clean .connectionLost case
    return AppUIError.connectionLost
}
