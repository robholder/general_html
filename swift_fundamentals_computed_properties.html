Swift - Computed Properties

An elegant understanding of computed properties is a superpower in SwiftUI. Because SwiftUI views automatically refresh whenever state changes, computed properties act as the perfect "live connectors" to clean, transform, and prepare data for your user interface without cluttering your main view hierarchy.

📘 Computed Properties Cheat Sheet

A computed property is a variable that doesn’t actually store a value in memory. Instead, it runs a block of code to calculate a value on the fly every single time you read it.

The Anatomy: Stored vs. Computed

// 💾 STORED PROPERTY: Takes up space in memory. Holds a fixed piece of data.
@State private var sides = 6 

// 🧮 COMPUTED PROPERTY: Takes up zero memory. Recalculates dynamically.
var objectWord: String {
    sides == 2 ? "Coin" : "Die"
}

1. The Single vs. Multi-Line Rule (Implicit vs. Explicit Return)

This is one of the most common syntax errors developers encounter when moving code around.

The Single-Line Rule (Implicit Return)

If the property contains exactly one single expression of code, Swift automatically infers that the result of this expression should be returned. You must not or do not need to use the return keyword. 

var actionWord: String {
    sides == 2 ? "Flip" : "Roll" // No 'return' needed!
}

The Multi-Line Rule (Explicit Return)

The moment you introduce a second statement (like creating a local variable, printing a log, or running a calculation step first), Swift stops guessing. You must explicitly use the return keyword to point out the final payload. 

var diceImageName: String {
    // Statement 1: Setup work
    let formattedNumber = String(format: "%02d", sides) 
    
    // Statement 2: The actual output
    return "dice_sides_\(formattedNumber)" // 'return' is MANDATORY here!
}

2. Advanced: Read-Only vs. Read-Write Properties

By default, almost every computed property you write in SwiftUI will be Read-Only (Getters). However, Swift allows you to build properties that can be written to as well.

Read-Only (Standard Shortcut)

When you write a basic computed property, you are secretly writing a get block. Swift lets you omit the word get entirely to save space: 

// What you write:
var doubleSides: Int { sides * 2 }

// What Swift sees under the hood:
var doubleSides: Int {
    get {
        return sides * 2
    }
}

Read-Write (Getters & Setters)

Rarely, you might want to update a stored property by changing a computed property. To do this, you provide both a get block and a set block. The setter receives a magic keyword variable called newValue. 

@State private var radius: Double = 10.0

var diameter: Double {
    get {
        radius * 2
    }
    set {
        // If someone sets view.diameter = 50, radius becomes 25
        radius = newValue / 2 
    }
}

3. Why Computed Properties are Essential for SwiftUI

Computed properties are the secret to keeping your SwiftUI views light, clean, and readable.

❌ The Anti-Pattern: Messy Inline Views

Don't clutter your layout engine with complex logic strings. It makes the UI code hard to read and debug. 

Text("\(sides == 2 ? "Flip" : "Roll") the \(sides == 2 ? "Coin" : "\(sides)-Sided Die")")

The SwiftUI Clean-Code Pattern

Move that logic up into computed properties. Your layout code transforms into beautiful, self-documenting human sentences: 

Text("\(actionWord) the \(objectWord)")

Key Takeaway for Notes: If your UI needs data transformed, formatted, or evaluated based on a @State variable, always use a computed property. It calculates instantly, leaves no footprint in memory, and keeps your layout code pristine.

Computed Property vs Function

Here is a great rule of thumb to add to your notes for deciding which one to use:

📐 Use a Computed Property (var) when:

  • It returns a value (like a String, Int, or View).
  • It is fast and cheap to run (takes O(1) time complexity).
  • It has no side effects - meaning just reading it doesn't alter, delete, or change state data elsewhere in the app. 
    var actionWord: String {
    sides == 2 ? "Flip" : "Roll"
}

⚙️ Use a Function (func) when:

  • It performs an action or executes a command (like saving, deleting, or loading).
  • It does heavy lifting or loops through data (like checking database bounds).
  • It has side effects - meaning it actively modifies or deletes data in your database or memory. 
    func enforceHistoryLimit() {
    // Actively mutates and deletes items from the database!
}
