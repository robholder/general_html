Git - Amend (Change) a Commit (including the message)

Message Change Command:

git -commit --amend -m "New commit message"

Change in text editor:

git commit --amend

It is important to know that Git isn't actually "editing" the old commit. It is replacing it.

  • The old commit is tossed aside.
  • A brand new commit is created with a new SHA (ID).
  • Your branch pointer moves to this new commit.

⚠️ The Golden Rule: Only do this if you have not pushed the commit to a shared repository (like GitHub). If you amend a commit that others have already pulled, you will create a "split timeline" that can be a nightmare to merge later.

Changing the Commit, but Keep the Message

If you realize you forgot to add a small change (like a missing padding or a typo fix) to your last commit, you can:

  • Save the file.
  • git add .
  • git commit --amend --no-edit

The --no-edit flag tells Git: "Take these new changes, shove them into the last commit, but keep the message exactly the same."

It keeps your git history looking very clean and professional.

In Depth

Even if you don't change a single letter of the commit message, using --amend (with or without --no-edit) always generates a brand-new commit hash.

Why the hash changes

In Git, a commit hash (like a1b2c3d) is a digital fingerprint calculated from three things:

  • The Snapshot: Every file and folder in your project at that moment.
  • The Metadata: The author, the date/timestamp, and the commit message.
  • The Parent: The hash of the commit that came before it.

When you use git commit --amend --no-edit:

  • You are changing the Snapshot (by adding your new changes).
  • You are changing the Metadata (the "Commit Date" is updated to now, even if the "Author Date" stays the same).

Because the input data has changed, the resulting fingerprint (the SHA-1 hash) must change.

Why this matters (The "Safety" Reminder)

Because the hash changes, Git views this as a completely different object than the one you had 30 seconds ago.

  • If you haven't pushed: This is a "silent" replacement. No one ever needs to know the old hash existed.
  • If you HAVE pushed: GitHub still has the old hash (a1b2c3d). If you try to push your new "amended" hash (e5f6g7h), GitHub will get confused and reject it because the histories don't match. This is why you would have to "Force Push."

The "Ghost" Commit

The old commit isn't actually deleted immediately; it becomes a "dangling" commit. It stays in Git’s database for a while (visible via git reflog) but is no longer part of your branch's timeline. After about 30 days, Git's "Garbage Collection" finally scrubs it away.

The takeaway: Think of --amend --no-edit as a "Do-Over." It lets you fix your mistakes while pretending to the rest of the world that you got it right the first time!
