Hacking With Swift - 221 - Challenge

Challenge

Have you ever been to a conference or a meetup, chatted to someone new, then realized seconds after you walk away that you’ve already forgotten their name? You’re not alone, and the app you’re building today will help solve that problem and others like it.

Your goal is to build an app that asks users to import a picture from their photo library, then attach a name to whatever they imported. The full collection of pictures they name should be shown in a List, and tapping an item in the list should show a detail screen with a larger version of the picture.

Breaking it down, you should:

  • Use PhotosPicker to let users import a photo from their photo library.
  • Detect when a new photo is imported, and immediately ask the user to name the photo.
  • Save that name and photo somewhere safe.
  • Show all names and photos in a list, sorted by name.
  • Create a detail screen that shows a picture full size.
  • Decide on a way to save all this data.

Remember to import the user's photo as Data, so you can write it out easily.

You can use SwiftData for this project if you want to, but it isn’t required - a simple JSON file written out to the documents directory is fine, although you will need to add a custom conformance to Comparable to get array sorting to work.

If you do choose to use SwiftData, here's an important tip: when storing large data like images or movies in your models, define them using a special @Attribute macro like this: 

@Attribute(.externalStorage) var photo: Data

That tells SwiftData not to save the image data directly inside its database, but to put it alongside instead - it's much more efficient.

Remember, you already know everything you need to make this work - good luck!

Ai Insight

@Attribute

Let’s unpack this line completely. This single line of code is actually packed with a ton of advanced Swift architecture.

To understand what each component is doing, we have to look at how SwiftData transforms a standard variable into a managed database column.

1. Breaking Down the Components

Here is the role of every single word in that line: 

    @Attribute(.externalStorage) var  photo: Data
//  └───┬───┘ └──────┬───────┘  └─┬─┘ └─┬─┘ └─┬┘
//    Macro      Configuration Variable Name Type

@Attribute (The Macro)

This is a Macro (introduced in Swift 5.9). Think of a macro as a code-generation robot. When you compile your app, Xcode reads @Attribute and automatically writes dozens of lines of hidden code behind the scenes. It intercepts this property and teaches SwiftData how to map it to a database database schema.

(.externalStorage) (The Configuration Option)

This is an enum case passed as a parameter into the macro. It configures the macro's behavior. Because Swift knows that @Attribute expects an option from the Attribute.Options enum, you can use the shorthand dot-syntax .externalStorage instead of typing out the full Attribute.Options.externalStorage.

var photo: Data (The Property)

This is the standard Swift part you already know. It creates a mutable variable named photo that holds raw binary data (Data), which is how computers represent files like images, audio, or PDFs.

2. What exactly is .externalStorage?

Mechanically, .externalStorage is a static property/enum case acting as a configuration flag.

Conceptually, it completely changes where your image is saved on the user's device.

Without .externalStorage (The Default Behavior)

By default, SwiftData saves everything inside a single, monolithic database file (a SQLite file) in your app's sandbox. If you import ten 5MB photos, your database file suddenly swells to 50MB.

  • The Problem: Databases are built to search text and numbers quickly. They get incredibly slow and bloated when they have to lug giant blobs of raw image binaries around every time you perform a simple text search or sort a list.

With .externalStorage

When you add this flag, you are telling SwiftData: "Keep this property in my data model, but do not stuff the raw image bytes inside the SQL database file."

Instead, SwiftData does two things automatically:

  • It carves out a private, secure folder right next to your database on the iPhone's hard drive.
  • It saves your image as a standalone file in that folder, and then writes a tiny, lightweight text path (a reference pointer) inside the actual database.

Why this is a Superpower

When you scroll through your List of named photos, SwiftData only has to read the text names from the database file, making your scrolling buttery smooth. It only reaches out to read the heavy image file from the external disk at the exact millisecond that specific row is drawn on the screen! You get all the organizational benefits of a database without any of the performance lag.

More on @Attribute Configuration Options

Here is a quick bulleted breakdown of the most common and useful configuration options you can pass into the @Attribute macro in SwiftData:

  • .unique Ensures that no two entries in the database can share the same value for this property. If a duplicate is inserted, it can trigger an overwrite or a conflict crash. Perfect for user IDs, email addresses, or SKU numbers.
  • .encrypt Tells SwiftData to automatically scramble and encrypt the data when writing it to disk, and decrypt it on the fly when loading it into memory. Ideal for highly sensitive user information like passwords or personal health metrics.
  • originalName: "string" Allows you to map a Swift property name to a different column name inside the underlying database file. This is incredibly helpful if you are renaming a variable in your code but need to map it to an older database to prevent losing existing user data.
  • .preserveValueOnDeletion Tells SwiftData to keep a record of this specific property's historical data in the persistent store's undo/redo history logs, even if the model instance itself gets completely deleted by the user.
  • .transient: (Though sometimes used via @Transient directly) Tells SwiftData to completely ignore this variable when saving to the database. It exists purely while the app is actively running in memory (like a temporary calculated layout state).

Ai Build

Names

This is the classic "Name that Photo" challenge from the 100 Days of SwiftUI! It ties together a lot of advanced skills:

  • Handling permissions with PhotosPicker,
  • Managing asynchronous data loading,
  • Prompting the user with an alert text field, and
  • Persisting files.

Since the challenge tip mentions it, using SwiftData is by far the most elegant and modern way to handle this. It handles both the database storage and the sorting requirements automatically, meaning we don't have to manually code JSON document directory paths or write a custom Comparable extension.

Here is a complete, working implementation split into the Data Model, the Detail View, and the Main View.

1. The Data Model (NamedPhoto.swift)

First, we create our SwiftData model. As the challenge suggested, we use @Attribute(.externalStorage) on the image data so the database stays small and performant.

NamedPhoto.swift

import Foundation
import SwiftData

@Model
class NamedPhoto {
    var id: UUID
    var name: String
    
    // This tells SwiftData to save the raw image data as a separate file outside the DB
    @Attribute(.externalStorage) var photoData: Data
    
    init(name: String, photoData: Data) {
        self.id = UUID()
        self.name = name
        self.photoData = photoData
    }
}

2. The Detail Screen (PhotoDetailView.swift)

This simple view takes the selected item and displays the image at a larger size, making use of safe area rules to look great.

PhotoDetailView.swift

import SwiftUI

struct PhotoDetailView: View {
    let namedPhoto: NamedPhoto
    
    var body: some View {
        VStack {
            if let uiImage = UIImage(data: namedPhoto.photoData) {
                Image(uiImage: uiImage)
                    .resizable()
                    .scaledToFit()
                    .clipShape(.rect(cornerRadius: 12))
                    .padding()
            } else {
                ContentUnavailableView("Loading Image Failed", systemImage: "xmark.circle")
            }
            
            Spacer()
        }
        .navigationTitle(namedPhoto.name)
        .navigationBarTitleDisplayMode(.inline)
    }
}

3. The Main Screen (ContentView.swift)

This is where the magic happens. It coordinates the PhotosPicker, watches for a selection change, intercepts the raw data, triggers a naming prompt alert, and saves the final product to SwiftData.

ContentView.swift

import PhotosUI
import SwiftData
import SwiftUI

struct ContentView: View {
    @Environment(\.modelContext) var modelContext
    
    // Automatically fetches items from SwiftData, pre-sorted alphabetically by name
    @Query(sort: \NamedPhoto.name) var photos: [NamedPhoto]
    
    // PhotosPicker States
    @State private var selectedPickerItem: PhotosPickerItem? = nil
    @State private var temporaryPhotoData: Data? = nil
    
    // Alert States for naming the photo
    @State private var showingNamePrompt = false
    @State private var newPhotoName = ""
    
    var body: some View {
        NavigationStack {
            List(photos) { item in
                NavigationLink(destination: PhotoDetailView(namedPhoto: item)) {
                    HStack(spacing: 16) {
                        if let uiImage = UIImage(data: item.photoData) {
                            Image(uiImage: uiImage)
                                .resizable()
                                .scaledToFill()
                                .frame(width: 60, height: 60)
                                .clipShape(.rect(cornerRadius: 8))
                        }
                        
                        Text(item.name)
                            .font(.headline)
                    }
                }
            }
            .navigationTitle("Photo Collector")
            .overlay {
                if photos.isEmpty {
                    ContentUnavailableView("No Photos", systemImage: "photo.on.rectangle.angled", description: Text("Tap the button below to import your first photo!"))
                }
            }
            .toolbar {
                ToolbarItem(placement: .bottomBar) {
                    // 1. The Picker Button
                    PhotosPicker(selection: $selectedPickerItem, matching: .images) {
                        Label("Import Photo", systemImage: "photo.badge.plus")
                            .font(.headline)
                    }
                }
            }
            // 2. Listen for when the user successfully chooses a photo from the library
            .onChange(of: selectedPickerItem) { _, newItem in
                Task {
                    if let data = try? await newItem?.loadTransferable(type: Data.self) {
                        temporaryPhotoData = data
                        newPhotoName = "" // Reset text field
                        showingNamePrompt = true // Trigger the naming alert immediately
                    }
                }
            }
            // 3. Intercept the flow and ask the user to input a name
            .alert("Name Your Photo", isPresented: $showingNamePrompt) {
                TextField("Enter name", text: $newPhotoName)
                
                Button("Save") {
                    saveItem()
                }
                .disabled(newPhotoName.trimmingCharacters(in: .whitespacesAndNewlines).isEmpty)
                
                Button("Cancel", role: .cancel) {
                    temporaryPhotoData = nil
                }
            } message: {
                Text("Please give a descriptive name to the picture you just imported.")
            }
        }
    }
    
    // 4. Commit the paired Name and Data into SwiftData permanent memory
    func saveItem() {
        guard let data = temporaryPhotoData else { return }
        
        let newPhoto = NamedPhoto(name: newPhotoName, photoData: data)
        modelContext.insert(newPhoto)
        
        // Clean up state tracking values
        temporaryPhotoData = nil
        selectedPickerItem = nil
    }
}

4. Don't Forget Your App Entrance Setup!

Because we are using SwiftData, your core App file (YourAppNameApp.swift) must inject the model container context into the view tree window, like this:

NamesApp.swift

import SwiftData
import SwiftUI

@main
struct PhotoCollectorApp: App {
    var body: some Scene {
        WindowGroup {
            ContentView()
                .modelContainer(for: NamedPhoto.self) // Sets up the DB environment
        }
    }
}
