Resetting to Previous Commit, and Deleting Subsequent Commits

To delete the latest commit and completely erase its existence—including any code changes you wrote—you use the git reset command with the --hard flag.

The Command

{{bash git reset --hard HEAD~1 }}}

Breaking Down the Syntax

  • git reset: The command used to move the current branch pointer to a specific state.
  • --hard: This is the "scorched earth" flag. It tells Git to discard all changes in your working directory and the staging area. Your files will revert exactly to how they looked in the previous commit.
  • HEAD~1: This tells Git to look at the commit immediately before the current one (HEAD).

Understanding the "Reset" Levels

If you find yourself needing to do this often, it helps to understand how Git's different "reset" modes affect your work:

Mode Pointer Moves? Files in Folder Stay? Staged Changes Stay?
--soft Yes Yes Yes
--mixed (Default) Yes Yes No
--hard Yes No (Deleted) No

Important Precautions

  • Permanent Loss: Unlike deleting a file on your computer, a hard reset doesn't go to the Recycle Bin. Those changes are difficult (and sometimes impossible) to recover.
  • Shared Branches: If you have already pushed that commit to GitHub/GitLab, running a reset locally will put your local repo out of sync with the remote. You would have to use git push -f (force push) to overwrite the remote history, which can be dangerous if you're collaborating with others.
  • Check your position: It’s always a good habit to run git log --oneline first to make sure you know exactly which commit you are about to delete.

Pro-Tip: If you just want to "undo" the commit but keep the code you wrote so you can edit it and commit again, use git reset --soft HEAD~1 instead. That way, your work stays in your files, but the "Commit" itself is gone.
