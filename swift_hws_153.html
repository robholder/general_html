Hacking With Swift - 153 - Sorting SwiftData queries using SortDescriptor

When you use @Query to pull objects out of SwiftData, you get to specify how you want the data to be sorted.

  • Should it be alphabetically by one of the fields?
  • Numerically with the highest numbers first?

Regardless of what you choose, it's always a good idea to choose something so your users have a predictable experience.

In this project we have various fields that might be useful for sorting purposes:

  • the title of the book,
  • the author,
  • the rating

All are sensible and would be good choices, but we don't have to rely on just one - you can specify multiple, so that you might ask for highest-rated books first, then use their names for a tiebreaker.

Query Sorting

Query sorting can be done in two ways: a simple option that allows just one sort field, and a more advanced version that allows an array of a new type called SortDescriptor.

Using the simple version, we might ask for our books to be provided in alphabetical order based on their title: 

@Query(sort: \Book.title) var books: [Book]

Or we could ask for them to be sorted by rating, highest to lowest: 

@Query(sort: \Book.rating, order: .reverse) var books: [Book]

That works well when you want just one single field, but generally I'd say it's a better idea to have a backup field too - to say "sort by rating, then by title" adds an extra level of predictability to your app, which is always a good thing.

SortDescriptor

This is done using the SortDescriptor type, which we can create from either one or two values:

  • the property we want to sort on,
  • and optionally whether it should be reversed or not.

For example, we can alphabetically sort on the title property like this: 

@Query(sort: [SortDescriptor(\Book.title)]) var books: [Book]

Like the simpler approach to sorting, sorting results using SortDescriptor is done in ascending order by default, meaning alphabetical order for text, but if you wanted to reverse the sort order you’d use this instead: 

@Query(sort: [SortDescriptor(\Book.title, order: .reverse)]) var books: [Book]

You can specify more than one sort descriptor, and they will be applied in the order you provide them.

For example, if the user added the book “Forever” by Pete Hamill, then added “Forever” by Judy Blume - an entirely different book that just happens to have the same title - then specifying a second sort field is helpful.

So, we might ask for book title to be sorted ascending first, followed by book author ascending second, like this: 

@Query(sort: [
    SortDescriptor(\Book.title),
    SortDescriptor(\Book.author)
]) var books: [Book]

Having a second or even third sort field has little to no performance impact unless you have lots of data with similar values. With our books data, for example, almost every book will have a unique title, so having a secondary sort field is more or less irrelevant in terms of performance.

AI Insight

The backslash in the SortDescriptor parameter assignment indicates a KeyPath.

In Swift, a KeyPath is a way of referring to a property itself rather than the value inside that property.

The Logic: Property vs. Value

Think of it this way:

  • book.title: This is you asking for the actual string (e.g., "The Great Gatsby").
  • \Book.title: This is you pointing at the "column" in the database titled "Title."

When you use SortDescriptor, you aren't telling Swift to sort by a specific book's name. You are giving Swift a map that says: "Look at the Book blueprint, find the title property, and use that as the instruction for how to arrange the list."

Breaking down the Syntax: \Book.title

  • \ (The Backslash): This tells Swift, "Start a KeyPath."
  • Book (The Type): This tells Swift which class or struct to look at.
  • .title (The Property): This is the specific "path" to the data field.

Why do we need it here?

SortDescriptor is designed to be flexible. It doesn't want the data right now; it wants to know how to get the data later when it’s actually performing the sort.

If you tried to pass just Book.title (without the backslash), Swift would get confused and think you’re trying to access a static property. If you passed book.title (from a specific instance), it would only have one string to look at, which doesn't help it sort a whole list of books.

Type Inference Shortcut

Since you already told Swift that the variable is an array of books (var books: [Book]), Swift is smart enough to "guess" that the KeyPath starts at the Book model.

Because of this, you will often see Paul Hudson (and other developers) leave out the type name to keep the code clean: 

// This is exactly the same as what you wrote:
@Query(sort: [SortDescriptor(\.title)]) var books: [Book]

By using \.title, you are saying: "Start at the root of whatever type is in this array and look for the title property."

Summary

  • Without backslash: "Give me the value of this property right now."
  • With backslash: "Here is a reference to this property so you can use it later."

It's essentially the difference between giving someone a glass of water and giving them directions to the well. SortDescriptor needs the directions!
