Swift - Protocols - Enum - String

String Raw-Value Enums

When you append a type like String to an enum declaration (e.g., enum StorageKeys: String), you are giving the enum a Raw Value Backing Type.

This reference guide breaks down what a String raw value is, what the compiler does automatically, and how it transforms abstract types into tangible data.

What is a Raw-Value Enum?

Normally, Swift enum cases are pure abstract symbols. They have no textual meaning, cannot be easily printed, and cannot be saved directly to data stores like UserDefaults.

Assigning : String instructs the Swift compiler to generate a stable, underlying string twin for every single case, accessible via the .rawValue property.

Implicit Raw Values (The Compiler Magic)

If you do not explicitly assign text strings to your cases, Swift's compiler automatically writes the boilerplate strings for you behind the scenes, matching your case names exactly. 

// What you type:
enum Sport: String {
    case skiing
    case snowboarding
}

// What Swift generates under the hood:
enum Sport: String {
    case skiing = "skiing"
    case snowboarding = "snowboarding"
}

If you ever want a case to display differently than its code name, you can override the implicit behavior by manually providing a string: 

enum Sport: String {
    case crossCountry = "Cross Country Skiing" // Explicit raw value
    case downhill = "downhill"                 // Implicit raw value
}

The 2 Core Benefits of Raw Values

1. Data Serialization

It bridges the gap between strong Swift types and simple text. This allows you to immediately pass your type-safe enum into storage systems, web requests, or text labels: 

// Evaluates to the literal text string "skiing"
let keyText = Sport.skiing.rawValue 

UserDefaults.standard.set(Sport.skiing.rawValue, forKey: "selected_sport")

2. Failable Initializers (Reconstructing from Storage)

Because the enum is backed by strings, you can attempt to recreate the safe enum type using a raw string retrieved from a file or web server via init(rawValue:).

This initializer is failable (returns an optional), meaning if the string doesn't match any known case, it returns a safe nil. 

// 1. Success: returns Optional(Sport.skiing)
let userSport = Sport(rawValue: "skiing") 

// 2. Failure: returns nil (protects your app from corrupted data)
let brokenSport = Sport(rawValue: "invalid_text")

Let’s break this down completely.

In Swift, a failable initializer is an initialization function that is allowed to say, "I tried my best, but the input you gave me makes no sense, so I am returning nothing (nil) instead of crashing."

In code, a failable initializer is written as init?() (notice the question mark). It outputs an Optional value.

The Core Problem it Solves

Imagine your app downloads a text string from a web server or reads a string from UserDefaults that tells you what sport a user likes.

You want to convert that raw string into your type-safe Sport enum so you can use it safely in your code. 

enum Sport: String {
    case skiing
    case snowboarding
}

When you use Sport(rawValue:), you are passing a completely unpredictable string into a strict system.

Scenario A: The Happy Path (Success)

You read the text "skiing" out of your database and hand it over to the enum: 

let userSport = Sport(rawValue: "skiing")
  • Swift looks inside the Sport enum and asks: "Does any case have a raw value matching the string 'skiing'?"
  • It finds case skiing.
  • It packages it up and returns Optional(Sport.skiing).

Because it's wrapped in an optional, you can safely unwrap it using your standard Swift tools: 

if let safeSport = userSport {
    print("The user loves \(safeSport)!") // Prints: The user loves skiing!
}

Scenario B: The Bad Data Path (Failure)

Now imagine the web server has a bug, or a user tampers with your save file, and the string your app reads out of storage is "tennis". 

let brokenSport = Sport(rawValue: "tennis")
  • Swift looks inside the Sport enum and asks: "Does any case have a raw value matching the string 'tennis'?"
  • It looks at skiing (no match) and snowboarding (no match).
  • Instead of crashing your app or forcing "tennis" into a container where it doesn't belong, the initializer fails gracefully.
  • It returns nil.

Why this protects your app

By returning nil, Swift forces you to handle the bad data safely before it can cause a bug or a crash deeper in your app: 

if let safeSport = Sport(rawValue: textFromStorage) {
    // This block gets completely skipped because brokenSport is nil
    goToSportScreen(for: safeSport) 
} else {
    // Your app gracefully recovers!
    print("Warning: Unknown sport data received. Defaulting to home screen.")
    goToHomeScreen()
}

QUESTION

So for String protocol, does Swift use init?() behind the scenes? I don't see it in the syntax.

ANSWER

Yes, exactly! Swift is completely hiding the boilerplates from your eyes, but init?(rawValue:) is absolutely running behind the scenes.

When you add : String to an enum, Swift automatically conforms that enum to a built-in system protocol called RawRepresentable.

You don't see init?(rawValue:) in your code because the Swift compiler generates it automatically for you during the build process.

What the Compiler Secretly Writes For You

When you write this short snippet: 

enum Sport: String {
    case skiing
    case snowboarding
}

The Swift compiler expands your code into a full implementation that looks like this: 

enum Sport: RawRepresentable {
    // 1. Sets the RawValue type alias to String
    typealias RawValue = String

    // 2. Secretly generates the read-only property
    var rawValue: String {
        switch self {
        case .skiing: return "skiing"
        case .snowboarding: return "snowboarding"
        }
    }

    // 3. Secretly generates the failable initializer!
    init?(rawValue: String) {
        switch rawValue {
        case "skiing": self = .skiing
        case "snowboarding": self = .snowboarding
        default: return nil // 👈 This is where the failure happens!
        }
    }
}

Where You See the Syntax

Because the compiler wrote that init?(rawValue:) function for you, you get to call it explicitly whenever you need to jump from a text string back to your safe enum: 

// You are explicitly invoking the hidden failable initializer here:
let mySport = Sport(rawValue: "skiing")

Because there is a hidden question mark (init?) inside that synthesized function, Swift treats mySport as an Optional<Sport>, forcing you to unwrap it safely.
