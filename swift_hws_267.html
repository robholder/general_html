Hacking With Swift - 267 - How to create a custom alignment guide

SwiftUI gives us alignment guides for the various edges of our views (.leading, .trailing, .top, and so on) plus .center and two baseline options to help with text alignment. However, none of these work well when you’re working with views that are split across disparate views - if you have to make two views aligned the same when they are in entirely different parts of your user interface.

To fix this, SwiftUI lets us create custom alignment guides, and use those guides in views across our UI. It doesn’t matter what comes before or after these views; they will still line up.

For example, here’s a layout that shows my Twitter account name and my profile picture on the left, and on the right shows "Full name:" plus "Paul Hudson" in a large font: 

struct ContentView: View {
    var body: some View {
        HStack {
            VStack {
                Text("@twostraws")
                Image(.paulHudson)
                    .resizable()
                    .frame(width: 64, height: 64)
            }

            VStack {
                Text("Full name:")
                Text("PAUL HUDSON")
                    .font(.largeTitle)
            }
        }
    }
}

If you want "@twostraws" and "Paul Hudson" to be vertically aligned together, you’ll have a hard time right now. The horizontal stack contains two vertical stacks inside it, so there’s no built-in way to get the alignment you want - things like HStack(alignment: .top) just won’t come close.

Custom Layout

To fix this we need to define a custom layout guide. This should be an extension on either VerticalAlignment or HorizontalAlignment, and be a custom type that conforms to the AlignmentID protocol.

When I say "custom type" you might be thinking of a struct, but it’s actually a good idea to implement this as an enum instead as I’ll explain shortly. The AlignmentID protocol has only one requirement, which is that the conforming type must provide a static defaultValue(in:) method that accepts a ViewDimensions object and returns a CGFloat specifying how a view should be aligned if it doesn’t have an alignmentGuide() modifier. You’ll be given the existing ViewDimensions object for the view, so you can either pick one of those for your default or use a hard-coded value.

Let’s write out the code so you can see how it looks: 

extension VerticalAlignment {
    struct MidAccountAndName: AlignmentID {
        static func defaultValue(in context: ViewDimensions) -> CGFloat {
            context[.top]
        }
    }

    static let midAccountAndName = VerticalAlignment(MidAccountAndName.self)
}

You can see I’ve used the .top view dimension by default, and I’ve also created a static constant called midAccountAndName to make the custom alignment easier to use.

Now, I mentioned that using an enum is preferable to a struct, and here’s why: we just created a new struct called MidAccountAndName, which means we could (if we wanted) create an instance of that struct even though doing so doesn’t make sense because it doesn’t have any functionality. If you replace struct MidAccountAndName with enum MidAccountAndName then you can’t make an instance of it any more - it becomes clearer that this thing exists only to house some functionality.

Regardless of whether you choose an enum or a struct, its usage stays the same: set it as the alignment for your stack, then use alignmentGuide() to activate it on any views you want to align together. This is only a guide: it helps you align views along a single line, but doesn’t say how they should be aligned. This means you still need to provide the closure to alignmentGuide() that positions the views along that guide as you want.

For example, we could update our Twitter code to use .midAccountAndName, then tell the account and name to use their center position for the guide. To be clear, that means "align these two views so their centers are both on the .midAccountAndName guide".

Here’s how that looks in code: 

HStack(alignment: .midAccountAndName) {
    VStack {
        Text("@twostraws")
            .alignmentGuide(.midAccountAndName) { d in d[VerticalAlignment.center] }
        Image(.paulHudson)
            .resizable()
            .frame(width: 64, height: 64)
    }

    VStack {
        Text("Full name:")
        Text("PAUL HUDSON")
            .alignmentGuide(.midAccountAndName) { d in d[VerticalAlignment.center] }
            .font(.largeTitle)
    }
}

That will make sure they are vertically aligned regardless of what comes before or after. I suggest you try adding some more text views before and after our examples - SwiftUI will reposition everything to make sure the two we aligned stay that way.

Rehash of the lesson.

It is completely fair to find this text overwhelming. Paul is mixing

  • high-level layout philosophy,
  • Swift type-safety choices (structs vs. enums),
  • and syntax implementation all into one unbroken stream.

It reads like word salad because he's trying to explain the "why" and the "how" simultaneously.

Let’s strip away the fluff and break this down into a clean, structured blueprint:

  • The Problem,
  • The Strategy, and
  • The 2-Step Execution.

1. The Core Problem (The "What")

Normally, alignment works inside a single container.

  • If you have a VStack, you can line up its items.
  • If you have an HStack, you can line up its items.

But look at your UI structure in this lesson:

  • You have an outer HStack.
  • Inside it, you have two VStack views:
    • VStack A (on the left) and
    • VStack B (on the right).

You want the:

  • text inside VStack A (@twostraws)

to line up perfectly on a horizontal line with the

  • text inside VStack B (PAUL HUDSON).

Because they are trapped inside separate vertical columns, standard tools like HStack(alignment: .top) will only align the tops of the columns themselves, not the specific text lines deep inside them.

2. The Strategy: The "Invisible String"

Custom Tethering Point

To fix this, we are going to create a custom tethering point.

Think of it like drilling a hole through both VStack columns at the exact height of the text views, running an invisible piece of string through them, and tying them together.

No matter what other content shrinks or grows around them, those two text fields are anchored to that string.

3. The 2-Step Technique (The "How")

This process is broken down into two distinct phases:

  • Creating the string (Boilerplate) and
  • Tying the knots (UI Implementation).

Step 1: Create the Custom Alignment (The Infrastructure)

You need to tell iOS that this new alignment type exists. We do this by extending VerticalAlignment. 

extension VerticalAlignment {
    // 1. Define the unique ID for our alignment string
    enum MidAccountAndName: AlignmentID {
        static func defaultValue(in context: ViewDimensions) -> CGFloat {
            context[.top] // If a view doesn't specify, default to its top edge
        }
    }

    // 2. Make a clean, lowercase shortcut to use in our views
    static let midAccountAndName = VerticalAlignment(MidAccountAndName.self)
}

Deciphering Paul'S Side-Note

Paul spends a lot of words talking about struct vs enum.

All you need to know is that using enum is a safety trick. It prevents you from accidentally writing let blank = MidAccountAndName() elsewhere in your app, because an enum with no cases cannot be instantiated. It's code housekeeping, not layout logic.

Step 2: Use it in your UI (The Technique)

Now that .midAccountAndName is a real alignment option, you apply it to the outer container, and then tell the specific children to latch onto it. 

// A. Tell the parent container to listen for our custom alignment string
HStack(alignment: .midAccountAndName) {
    
    VStack {
        Text("@twostraws")
            // B. Tie this child's center point to our string
            .alignmentGuide(.midAccountAndName) { d in d[VerticalAlignment.center] }
        Image(.paulHudson)
            .resizable()
            .frame(width: 64, height: 64)
    }

    VStack {
        Text("Full name:")
        Text("PAUL HUDSON")
            .font(.largeTitle)
            // C. Tie this child's center point to the exact same string
            .alignmentGuide(.midAccountAndName) { d in d[VerticalAlignment.center] }
    }
}

Summary of what you are actually doing:

  • You 'turn on' the custom guide on the parent: HStack(alignment: .midAccountAndName).
  • You say to the first text view: "Find your center point (d[VerticalAlignment.center]), and snap it to that guide."
  • You say to the second text view: "Find your center point, and snap it to that exact same guide."

SwiftUI handles all the layout calculation behind the scenes. Even if you add 5 massive images above Full Name:, it will physically shift the entire right stack up or down just to ensure those two specific text fields remain perfectly aligned.

QUESTION

I don't understand the whole extension's syntax. 

extension VerticalAlignment {
    // 1. Define the unique ID for our alignment string
    enum MidAccountAndName: AlignmentID {
        static func defaultValue(in context: ViewDimensions) -> CGFloat {
            context[.top] // If a view doesn't specify, default to its top edge
        }
    }

    // 2. Make a clean, lowercase shortcut to use in our views
    static let midAccountAndName = VerticalAlignment(MidAccountAndName.self)
}

ANSWER

It is completely natural that this feels like hocus pocus. You are staring at a very specific Swift programming pattern called "using a caseless enum as a namespace." It completely breaks the mental model of what a standard enum is supposed to be.

Let’s demystify this piece by piece. Once you see why the code is written this way, the pattern will make perfect sense, and you’ll be able to write your own easily.

1. Why is an enum being used like a struct?

You are 100% correct: standard enums are meant to be a list of choices (like enum Direction { case north, south }).

However, in Swift, enums have a superpower: They can conform to protocols and hold functions, just like a struct.

Look closely at Paul's enum: 

enum MidAccountAndName: AlignmentID { ... }

Notice there are no case statements inside it. It's empty. Why do this instead of a struct?

  • If you make a struct Dummy {}, someone could accidentally write let x = Dummy() in their code.
  • If you make an enum Dummy {} with no cases, it is physically impossible to instantiate it. You can't type let x = Dummy(). The compiler will block you.

Swift developers use empty enums purely as a "storage box" to hold functions or IDs securely. It’s a pure type-safety trick. You can completely pretend it's a struct that has a lock on it.

2. What does defaultValue(in:) actually do?

The protocol AlignmentID is a rulebook created by Apple. It says: "If you want to create a custom alignment, you must provide a box that contains a defaultValue function."

SwiftUI needs this because of a simple logical rule: If you tell an HStack to use a custom alignment, every single child view must participate, even if they don't have a .alignmentGuide() modifier attached to them.

The defaultValue function is your fallback answer. You are telling SwiftUI:

"Hey, if I forget to explicitly tell a view how to align to this custom guide, just use its .top edge as the default fallback."

3. Demystifying the Whole Blueprint

Let’s look at the syntax again, but rewritten with clear annotations of what each part actually is to SwiftUI: 

extension VerticalAlignment {
    
    // STEP A: Create a unique "Identity Card" for your layout.
    // SwiftUI requires a unique Type to identify your alignment.
    enum MidAccountAndName: AlignmentID {
        static func defaultValue(in context: ViewDimensions) -> CGFloat {
            return context[.top] // Fallback rule
        }
    }

    // STEP B: Create the actual alignment tool.
    // We instantiate a real VerticalAlignment object, passing it our Identity Card (.self).
    static let midAccountAndName = VerticalAlignment(MidAccountAndName.self)
}

Think of Step A as manufacturing a unique lock, and Step B as cutting the key that you actually use in your code (.midAccountAndName).

The "Copy-Paste" Blueprint for Your Own Alignments

Because this is boilerplate, you don't need to reinvent the wheel. When you want to make your own custom alignment tomorrow, you just follow this exact formula.

Say you want to create a custom vertical alignment called iconBaseline that defaults to the center of a view. You just swap out three words: 

extension VerticalAlignment {
    // 1. Change the name of the "Identity Box"
    enum IconBaseline: AlignmentID {
        static func defaultValue(in context: ViewDimensions) -> CGFloat {
            return context[.center] // 2. Change your default fallback choice
        }
    }

    // 3. Change your lowercase shortcut name
    static let iconBaseline = VerticalAlignment(IconBaseline.self)
}

Now, in your UI, you can instantly use HStack(alignment: .iconBaseline) and .alignmentGuide(.iconBaseline).

QUESTION

What is happening on these function lines? Why 'in' and 'context', and what is 'ViewDimensions', and why is it output as a CGFloat? And '.center' must be drawing on a selection of options (.center, .top, .leading etc.). 

static func defaultValue(in context: ViewDimensions) -> CGFloat {
    return context[.center]

ANSWER

This single line of code is packed with advanced Swift syntax, which makes it incredibly dense to read. Let’s unpack it word-by-word.

1. The Parameter: (in context: ViewDimensions)

This line defines the input that SwiftUI hands to the function.

  • in: This is an argument label. It is only there so that when someone calls the function, it reads like a natural English sentence: defaultValue(in: someDimensions).
  • context: This is the actual variable name used inside the function.
  • ViewDimensions: This is the data type. It is a special SwiftUI struct that contains the exact physical measurements of the view currently being analyzed. It knows the view's width, its height, and where its edges are.

2. The Return Type: -> CGFloat

This tells Swift what kind of data the function must spit out.

  • CGFloat stands for Core Graphics Float. It is a decimal number (like 12.5 or 140.0) used across Apple platforms to represent physical screen pixels.
  • SwiftUI alignments are not abstract concepts; they are explicit pixel coordinates. The function must return a CGFloat because it is telling SwiftUI: "Place the alignment line exactly X pixels down from the top of this view."

3. The Body: return context[.center]

This is where the magic happens, and it uses a special Swift feature called a subscript (the square brackets [ ]). 

context[.center]

You are absolutely right that .center comes from a list of predefined options!

The ViewDimensions object (context) allows you to look up pixel values using these standard edges. When you pass .center into the square brackets, the ViewDimensions object does some quick math behind the scenes: it takes the view's total height and divides it by 2.

So, if a text view is 30 pixels tall, context[.center] returns 15.0.

Putting It All Together into a Plain English Translation

If we translate that entire function into a conversational sentence, it means:

"Hey SwiftUI, here is a function called defaultValue. When you run it, I am going to hand you the physical dimensions of a view (the context). I want you to look up where the exact middle of that view is (context[.center]), and return that exact pixel number (CGFloat) to use as our alignment anchor."

QUESTION

Can you list some other ViewDimensions? (.top, .leading etc.?)

ANSWER

To see the full list of options, we actually have to split them into two separate categories: Vertical and Horizontal.

Because ViewDimensions needs to return a single pixel coordinate, you can only look up vertical options if you are making a VerticalAlignment, and horizontal options if you are making a HorizontalAlignment. Here is the complete menu of standard alignment options you can plug into context[...]:

1. Vertical Dimensions (For VerticalAlignment)

These measure distances from the top edge of the view downward (in pixels).

  • .top: The absolute top edge of the view (returns 0.0).
  • .center: The exact vertical midpoint (returns height / 2).
  • .bottom: The absolute bottom edge of the view (returns the total height).
  • .firstTextBaseline: The invisible line that the very first line of text sits on inside the view. (Incredibly useful for aligning an icon next to a multi-line paragraph so the icon lines up with the first sentence).
  • .lastTextBaseline: The invisible line that the very last line of text sits on inside the view.

2. Horizontal Dimensions (For HorizontalAlignment)

These measure distances from the leading (left) edge of the view across to the right (in pixels).

  • .leading: The left edge of the view (returns 0.0).
  • .center: The exact horizontal midpoint (returns width / 2).
  • .trailing: The right edge of the view (returns the total width).

A Quick Note on "Leading" vs "Left": Apple uses .leading and .trailing instead of .left and .right because of internationalization. If a user has their phone set to a right-to-left language like Arabic or Hebrew, SwiftUI automatically flips .leading to mean the right side of the screen, keeping your app layout perfectly adapted without you rewriting code!

How it looks in your code:

If you were building a custom Horizontal alignment tomorrow, your blueprint would look like this: 

extension HorizontalAlignment {
    enum MyHorizontalGuide: AlignmentID {
        static func defaultValue(in context: ViewDimensions) -> CGFloat {
            return context[.trailing] // Looks up the right-edge pixel value
        }
    }
    static let myHorizontalGuide = HorizontalAlignment(MyHorizontalGuide.self)
}

More on Enum (Comparisons)

QUESTION

As a reminder, can you show me a simple regular enum to compare?

ANSWER

Here is a standard, everyday Swift enum (Enumeration) to compare.

Think of a regular enum as a menu of fixed choices. You use it when a variable should only ever hold one of a few specific options - like days of the week, loading states, or compass directions. 

// A regular enum defines a list of explicit cases
enum CompassDirection {
    case north
    case south
    case east
    case west
}

// How you actually use it in your code:
var currentHeading = CompassDirection.north

// You can change it to any other valid case on the menu
currentHeading = .south 

// You can check it using a switch statement
switch currentHeading {
case .north:
    print("Heading North!")
case .south:
    print("Heading South!")
case .east, .west:
    print("Heading East or West!")
}

The Big Visual Comparison

Look at how completely different their jobs are:

Feature Regular Enum (CompassDirection) Layout Identity Enum (MidAccountAndName)
What's inside? A list of explicit choices (case north, etc.) No cases. Just a protocol rule and a function.
Can you create it? Yes. let go = CompassDirection.north No. Typing let x = MidAccountAndName() is a compiler error.
What is its job? To store data that represents a specific state or choice. To act as a safe, un-instantiable "box" to hold code for SwiftUI.

When you look at Paul's code, you are seeing a trick where developers use an empty enum purely as a container, exploiting the fact that it can conform to protocols (AlignmentID) without needing to actually hold any data choices.

QUESTION

Can you explain this d in d[VerticalAlignment.center] line? 

VStack {
    Text("@twostraws")
        .alignmentGuide(.midAccountAndName) { d in d[VerticalAlignment.center] }

ANSWER

This line of code is written in a shorthand syntax called a Closure (Swift's name for an anonymous, on-the-fly function).

When you see { d in d[...] }, you are looking at an input/output machine. Let’s break down exactly what that d means, why it appears twice, and why VerticalAlignment.center is written out fully.

1. What is the d?

The letter d is just a variable name chosen by Paul Hudson. It stands for Dimensions.

When SwiftUI renders your layout, it automatically measures the exact width and height of your @twostraws text view. It then hands those measurements to your closure. The d is the nickname we use to access those measurements.

You could easily rename d to dimensions to make it more readable: 

.alignmentGuide(.midAccountAndName) { dimensions in dimensions[VerticalAlignment.center] }

2. Why does d appear twice?

The in keyword splits the closure into two halves:

  • The Input (on the left) and
  • The Code Body (on the right). 
    { d in d[VerticalAlignment.center] }
  │    │
  │    └─► 2. The Code Body: "Take that input and look up its center pixel value."
  │
  └──────► 1. The Input: "SwiftUI, hand me this view's dimensions and call them 'd'."

Think of it like writing a quick math function: f(x)=x+2. You have to declare x as your input variable before you can use it in your equation.

3. Why write VerticalAlignment.center instead of just .center?

Normally, SwiftUI lets you use dot-shorthand (like .center) because it can easily guess what enum you are talking about.

However, inside a custom alignment guide closure, SwiftUI gets confused because there are too many different types of "centers." If you just write d[.center], the compiler panics because it doesn't know if you mean:

  • HorizontalAlignment.center (the left-to-right middle)
  • VerticalAlignment.center (the top-to-bottom middle)
  • Alignment.center (the absolute dead-center of a 2D box)

To prevent the compiler from throwing an error, you have to spell it out explicitly: VerticalAlignment.center. This clarifies that you are looking up the vertical midpoint pixel value.

Visualizing the Whole Mechanism

When the code runs, SwiftUI performs this exact calculation loop:

  • SwiftUI measures the @twostraws text view and discovers it is 20 pixels tall.
  • It packs that info into a ViewDimensions object and hands it to your closure as d.
  • Your code asks for d[VerticalAlignment.center].
  • The closure calculates 20 ÷ 2, and returns the number 10.0.
  • SwiftUI takes that 10.0 pixel coordinate and lines it up perfectly with the custom .midAccountAndName alignment string.
