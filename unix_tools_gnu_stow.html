GNU Stow - Dotfile Management

Install on MacOS

brew install stow

Verify with... 

stow --version

Basic Setup for Dotfiles

See this YouTube video:

  • How To Easily Manage Your Dotfiles With GNU Stow

Once installed, a common workflow for managing configuration files involves:

  • Create a Directory: mkdir ~/.dotfiles.
  • Organize Configs: Move your existing configs (e.g., .zshrc) into subdirectories within ~/.dotfiles.
  • Symlink: Run stow . from inside your dotfiles directory to symlink them back to your home folder. 
    brew install stow
mkdir ~/.dotfiles
# move your configs or dot-file config files into the ~/.dotfiles dir (in the same structure)
git init
stow .

Create Ignore List

Much like Git, you can use an ignore list to force Stow to NOT symlink certain entities in the directory from which it's run.

Stow already ignores common files/dirs (like .git), but you can make your own ignore list.

There are two varieties (see later for more insight):

  • .stow-local-ignore
  • .stow-global-ignore

Here are some suggestions (with some 'safer' or more robust additions to the default): 

^README\.md$
^install\.sh$
\.DS_Store

Original List 

# Comments and blank lines are allowed.

RCS
.+,v

CVS
\.\#.+       # CVS conflict files / emacs lock files
\.cvsignore

\.svn
_darcs
\.hg

\.git
\.gitignore
\.gitmodules

.+~          # emacs backup files
\#.*\#       # emacs autosave files

^/README.*
^/LICENSE.*
^/COPYING

The Local Ignore

This is a per directory ignore list, and is stored in:

  • .stow-local-ignore

It applies to only the directory from which stow is run.

You only need a .stow-local-ignore inside a specific subdirectory if that app has unique "garbage" files you don't want linked.

Example:

In your vim folder, you might want to ignore a temp_notes.txt that exists there, but you still want notes.txt in your zsh folder to be linked.

The Global Ignore

Stow has a "Global" ignore feature that applies to every subdirectory (package) in your dotfiles folder. This is much better than copying .stow-local-ignore into every app folder.

Why using a global ignore list is often better than a local list:

  • Stow will automatically check this file for any package you try to stow (wezterm, vim, zsh, etc.), keeping your subdirectories clean of "boilerplate" files.

Create a file named .stow-global-ignore in the root of your ~/.dotfiles directory: 

# ~/.dotfiles/.stow-global-ignore
\.DS_Store
^README\.md$
^LICENSE.*
\.git$

Since you are probably putting the dotfiles directory under Git control, make sure you also have a .gitignore in the root of ~/.dotfiles.

  • Stow ignore prevents files from being linked to your home directory.
  • Git ignore prevents files from being uploaded to your repo.

You generally want .DS_Store in both.

The "Nuclear" Option (Shell Alias)

If you want to be 100% sure nothing weird gets stowed, you can bake the ignore pattern directly into a stow alias in your ~/.zshrc: 

alias stow='stow --ignore=".DS_Store|README.md|.git"'

More on the Git Ignore

To keep your .dotfiles repository clean and professional, you need a .gitignore that handles three main categories: macOS clutter, Vim/Editor noise, and Shell history.

Place this .gitignore file in the root of your ~/.dotfiles directory: 

# ==============================================================================
# .gitignore for .dotfiles Repo
# ==============================================================================

# --- macOS Metadata ---
.DS_Store
.AppleDouble
.LSOverride
Icon?
._*
.Spotlight-V100
.Trashes

# --- Vim / Editor Junk ---
# Ignore swap and recovery files
[._]*.s[a-v][a-z]
[._]*.sw[a-p]
[._]s[a-rt-v][a-z]
[._]ss[a-gi-z]
*.swp
*.swo

# Ignore Vim undo and backup files
*~
*.un~
.netrwhist

# --- App Specific Bloat ---
# Ignore the actual downloaded plugins (only track your vimrc!)
.vim/plugged/
.vim/bundle/
.vim/pack/
.vim/undo/

# Ignore local zsh/shell histories
.zcompdump*
.zsh_history
.bash_history
node_modules/
.python-version

# --- Stow Specific ---
# If you use a local ignore file, you might want to keep it out of some setups
# .stow-local-ignore

Why this is a "Pro" Configuration:

  • Vim Swap Files: The [._]*.s... patterns are the "standard" way to catch all variations of Vim swap files (like .swp, .swo, .swn) so they never get committed while you're editing.
  • Plugin Isolation: By ignoring .vim/plugged/, your repo stays under 1MB. When you clone to a new machine, your vimrc auto-install logic (we set up earlier) will download them fresh.
  • Shell History: You definitely don't want to push your .zsh_history to GitHub, as it often contains sensitive commands or passwords you've typed.

A Critical Step for Stow

Now that you have a .gitignore, make sure your .stow-global-ignore (the one we talked about previously) matches it.

Stow doesn't automatically read your .gitignore, so if you ignore .DS_Store in Git but not in Stow, it will still try to symlink those invisible macOS files to your Home directory!

Remove Stow Setup

To reverse the process and "un-link" your dotfiles from your home directory, you use the Delete flag.

The Reverse Command

Navigate to your ~/.dotfiles folder and run: 

stow -D <folder_name>

...or 

stow -D .

What this does:

  • Removes the Symlink: It finds the symbolic link in your home directory (e.g., ~/.zshrc) that points to your dotfiles folder and deletes it.
  • Safety First: Stow is smart—it will not delete the actual file inside your ~/.dotfiles folder. It only breaks the connection.
  • Folder Cleanup: If Stow created any empty parent directories to house those links, it will try to clean those up as well.

How to fully "Revert" to local files:

If you want to move the files back to your home directory permanently (no longer using Stow):

  • Un-stow: From inside the .dotfiles directory -> stow -D wezterm.
  • Move them back: cp -r ~/.dotfiles/wezterm/.config/wezterm ~/.config/
    • Here is exactly how that command behaves:
    • The Source: It targets the wezterm directory located inside your dotfiles structure.
    • The Action (-r): The "recursive" flag tells macOS to grab the folder, all its sub-folders, and every file inside (like your wezterm.lua).
    • The Destination: It places that entire folder into ~/.config/.
    • The Result
      • After running that, you will have a real, physical directory at: ~/.config/wezterm/
  • Verify: Run ls -la ~/.config/wezterm to ensure they are now regular files and not "arrows" pointing elsewhere.

A Tiny "macOS Gotcha"

If ~/.config/wezterm/ already exists (even as a broken symlink), running that cp command might try to nest the folder inside itself (creating ~/.config/wezterm/wezterm/).

To be safe, do this sequence: 

# 1. Remove the existing link/folder first
rm -rf ~/.config/wezterm

# 2. Copy the fresh version over
cp -r ~/.dotfiles/wezterm/.config/wezterm ~/.config/

Pro-Tip: The "Restow"

If you just made a mistake or renamed a file and want to refresh the links without deleting everything first, use: 

stow -R <folder_name>

This performs a Delete and an Install in one quick motion.

Adding Vim Configs

To back up your Vim setup to a repo using Stow, the "gold standard" is to keep your configuration (the "brains") in Git, but let a plugin manager handle the heavy lifting of downloading the "paraphernalia" (the "brawn").

1. Structure your Dotfiles for Stow

Move your .vim directory into your ~/.dotfiles folder. To work with Stow, the internal structure must mimic your home directory: 

~/.dotfiles/
└── vim/
    └── .vim/
        ├── vimrc
        ├── colors/
        ├── snippets/
        └── (Don't move 'plugged' or 'bundle' here yet!)

2. The .gitignore Strategy (Omit the Bloat)

You are right—omitting plugins is essential. They take up space and are easily re-downloaded. Create a ~/.dotfiles/.gitignore file and add these: 

# Ignore the actual plugin files
.vim/plugged/
.vim/bundle/
.vim/pack/

# Ignore swap and backup files
*.swp
*.swo
.netrwhist

# Ignore the undo and view history
.vim/undo/
.view/

3. Use a Plugin Manager for "One-Command" Restore

In your vimrc, ensure you use something like vim-plug. This allows you to restore your entire setup on a new machine with just your vimrc and one command.

Example vimrc block: 

call plug#begin('~/.vim/plugged')
Plug 'nordtheme/vim'
" ... other plugins
call plug#end()

4. The Workflow to Backup

Once your folder is structured:

  • Initialize Git: cd ~/.dotfiles && git init
  • Stow it: stow vim (This creates the link ~/.vim -> ~/.dotfiles/vim/.vim)
  • Commit: git add . && git commit -m "Add Vim config"
  • Push: Link to a private GitHub/GitLab repo.

Why this is the best way:

  • Portability: On a new Mac, you just git clone, stow vim, and run :PlugInstall inside Vim.
  • Cleanliness: Your repo stays tiny (likely under 1MB) because it only contains your custom logic, not thousands of lines of third-party plugin code.
  • Safety: If a plugin update breaks your setup, you can just delete the plugged/ folder and run :PlugInstall to redownload them fresh.
    Do you want to see a minimal vimrc that handles the automatic installation of the plugin manager itself if it’s missing?

Adding to the Vimrc

This snippet is a lifesaver for a portable setup. It checks if vim-plug exists; if not, it uses curl to download it and then tells Vim to run :PlugInstall automatically. Add this to the very top of your vimrc: 

" 1. AUTO-INSTALL vim-plug if missing
let data_dir = expand('~/.vim/autoload/plug.vim')
if empty(glob(data_dir))
  silent execute '!curl -fLo '.data_dir.' --create-dirs https://raw.githubusercontent.com'
  autocmd VimEnter * PlugInstall --sync | source $MYVIMRC
endif

" 2. PLUGIN SECTION
call plug#begin('~/.vim/plugged')

" Appearance (Matches your WezTerm!)
Plug 'nordtheme/vim'

" Navigation & Editing
Plug 'tpope/vim-sensible'
Plug 'preservim/nerdtree'
Plug 'junegunn/fzf', { 'do': { -> fzf#install() } }

call plug#end()

" 3. BASIC CONFIG
syntax on
colorscheme nord
set number
set relativenumber

Why this is the "Gold Standard" for your Dotfiles:

  • Zero Manual Setup: When you move to a new machine, you just stow vim and open Vim. It handles the rest.
  • Isolation: By keeping plugins in ~/.vim/plugged, and ignoring that folder in Git, your repo stays tiny and clean [1].
  • Stow Friendly: Since this script installs autoload/plug.vim directly into your ~/.vim folder (which is now a symlink to your dotfiles), the plugin manager itself becomes part of your tracked setup.

A Pro-Tip for your .gitignore:

Since you're using GNU Stow, make sure your .gitignore in your ~/.dotfiles folder specifically ignores the undo and swap directories inside .vim so you don't accidentally commit your editing history [1][2].

Git Alias to Backup Configs

To keep your .dotfiles sync as painless as possible, you can create a single Git alias that adds, commits, and pushes in one go.

If you're using Zsh, the best way to do this is with a shell function in your ~/.zshrc. This allows you to include a custom commit message (or use a default one) with a single word.

1. The "Auto-Push" Alias

Add this to your ~/.zshrc: 

# Dotfiles sync: usage 'dot sync "your message"'
dot() {
  if [ "$1" = "sync" ]; then
    local msg="${2:-Update configs}"
    cd ~/.dotfiles && \
    git add . && \
    git commit -m "$msg" && \
    git push
    cd - > /dev/null
  else
    echo "Usage: dot sync \"commit message\""
  fi
}

2. How to use it:

After making changes to your wezterm.lua or vimrc:

  • With a message: dot sync "added font cycle"
  • Fast (default message): dot sync

Why this is great for your setup:

  • Safety: The cd - > /dev/null at the end instantly teleports you back to the directory you were working in, so your workflow isn't interrupted.
  • Consistency: Since you've already set up your .gitignore to omit the heavy .vim plugins, this command will only push your lightweight text configs.
  • Stow-Ready: Because it runs git add . from the root of ~/.dotfiles, it will catch any new folders you decide to Stow later.

Pro-Tip: The "Lazy" Check

If you're using eza, you can add a quick check to see what’s changed before you sync: 

alias dots='eza -lT --git ~/.dotfiles'

This lets you run dots to see exactly which config files have "Modified" (M) or "New" (N) status in your tree before you push.

Avoid 'dot' in Names of Stowed Dotfiles

If you prefix your actual config file with dot- instead of and actual dot, you can call the --dotfiles flag with stow to have the symlink (symbolic link) get named with an actual dot. 

# in ~/dotfile/dot-config
# run:
stow --dotfiles .
## This will create a symlink in the home directory:
.config

Don't forget to use the --dotfiles flag when using other functions (like the -D delete flag). 

# in ~/dotfile/dot-config
# run
stow -D --dotfiles
## This will delete a symlink called .config

Finding Your Symlinks in Zsh

Using ls: 

ls -la | grep "^l"

Using eza: 

eza -la --icons | grep "^l"

Stow Flags

adopt

In Gnu's Stow utility, what does the --adopt flag do?

--adopt is one of Stow’s most powerful (and slightly dangerous) flags, and once you understand what it does, it becomes an incredible tool for converting an existing system into a clean, Stow‑managed dotfiles setup.

Here’s the clearest way to think about it.

🌟 What --adopt does (in plain English)

--adopt tells Stow:

“If a file already exists in the target directory, move it into the Stow package and replace it with a symlink.”

In other words:

  • It absorbs existing files into your dotfiles repo
  • It moves them into the correct package directory
  • It creates symlinks in their place
  • It does not overwrite anything — it reorganizes it

This is perfect when you're migrating an existing system into Stow.

🧠 A concrete example

Suppose you have: 

~/.zshrc   (a real file)
~/dotfiles/zshrc/dot-zshrc   (empty or missing)

If you run: 

stow --dotfiles --adopt -t ~ zshrc

Stow will: Move your real file into the package: 

~/.zshrc → ~/dotfiles/zshrc/dot-zshrc

Replace it with a symlink: 

~/.zshrc → dotfiles/zshrc/dot-zshrc

Now your .zshrc is fully managed by Stow.

🌟 Why this is useful

--adopt is perfect when:

  • You already have dotfiles in your home directory
  • You want to migrate them into your Stow repo
  • You don’t want to manually copy each file into the right package
  • You want Stow to reorganize everything for you

It’s basically a “convert my system into a Stow setup” button.

⚠️ Why it’s powerful (and risky)

Because --adopt moves files, not copies them.

If you run it on the wrong package, you can accidentally move files into the wrong place.

That’s why your bootstrap script only uses --adopt when you explicitly pass: 

--adopt

And why it’s good to pair it with: 

--dry-run

Example: 

~/dotfiles/bootstrap.sh --adopt --dry-run

This shows you exactly what would happen without touching anything.

🎯 Summary

--adopt:

  • Moves existing files into your Stow package
  • Replaces them with symlinks
  • Helps migrate an existing system into a Stow‑managed dotfiles repo
  • Should be used carefully, ideally with a dry run first

It’s one of the best features Stow has — as long as you respect its power.

Migrate Using Adopt

If you want to migrate an existing home directory into a clean, Stow‑managed dotfiles setup without losing anything, there’s a safe, predictable workflow that works beautifully. Think of it as a “dotfiles refactor” where Stow does the heavy lifting and you stay in control the whole time.

Here’s the approach I recommend — it’s the same one I use when helping people convert years of accumulated config files into a tidy, version‑controlled structure.

🌟 The Safest Workflow for Using stow --adopt

This method ensures:

  • nothing gets overwritten
  • everything gets backed up
  • you always know what Stow is about to do
  • your repo ends up clean and organized

We’ll do it in five steps.

1️⃣ Start with a clean Stow directory structure

Inside ~/dotfiles, create one directory per “package”: 

dotfiles/
  zshrc/
  vim/
  wezterm/
  starship/
  robsaliases/

Inside each package, use your dot-* naming convention: 

dotfiles/zshrc/dot-zshrc
dotfiles/zshrc/dot-zprofile
dotfiles/vim/dot-vimrc
dotfiles/wezterm/dot-config/wezterm/wezterm.lua
...

You can leave these files empty for now — Stow will fill them.

2️⃣ Dry‑run the adoption first (critical)

Before moving anything, preview what Stow would adopt: 

stow --dotfiles --adopt -nvt ~ zshrc

This shows:

  • which files Stow would move into the package
  • which symlinks it would create
  • whether anything unexpected would happen

Do this for each package: 

stow --dotfiles --adopt -nvt ~ vim
stow --dotfiles --adopt -nvt ~ wezterm
...

If anything looks wrong, you can fix your package layout before touching real files.

3️⃣ Back up your entire home directory (optional but smart)

This gives you a safety net: 

mkdir -p ~/backup-pre-stow
cp -a ~/. ~/backup-pre-stow/

Now you can always restore anything.

4️⃣ Run the real adoption (one package at a time)

Once the dry‑run looks good: 

stow --dotfiles --adopt -vt ~ zshrc

What happens:

  • ~/.zshrc is moved into dotfiles/zshrc/dot-zshrc
  • A symlink is created: ~/.zshrc → dotfiles/zshrc/dot-zshrc

Repeat for each package: 

stow --dotfiles --adopt -vt ~ vim
stow --dotfiles --adopt -vt ~ wezterm
stow --dotfiles --adopt -vt ~ starship
stow --dotfiles --adopt -vt ~ robsaliases

This is the magic moment where your entire home directory becomes Stow‑managed.

5️⃣ Commit everything to Git

Once adoption is complete: 

cd ~/dotfiles
git add .
git commit -m "Adopt existing dotfiles into Stow structure"

Now your dotfiles are:

  • version‑controlled
  • reproducible
  • portable
  • cleanly organized

🌟 Bonus: Using your bootstrap script with --adopt

Your script already supports: 

~/dotfiles/bootstrap.sh --adopt

This will:

  • detect all packages
  • run stow --adopt on each
  • back up conflicts if --force is used
  • show everything in color
  • keep the process safe

For maximum safety, combine: 

~/dotfiles/bootstrap.sh --adopt --dry-run

Then, when everything looks good: 

~/dotfiles/bootstrap.sh --adopt --force

This is the “big migration” button.

🎯 Summary: The safest way to migrate into Stow

Step Action
1 Create package directories in ~/dotfiles
2 Dry‑run adoption (--adopt -nvt)
3 Back up your home directory
4 Adopt packages one by one
5 Commit everything to Git

This workflow is safe, reversible, and clean.
