The Master Guide to GPS Data in iOS: Live Hardware & Photo EXIF

This document serves as your definitive reference blueprint for working with location data in SwiftUI. It covers both vectors of location acquisition:

  • Live GPS Hardware Streaming (via CoreLocation) and
  • Static Metadata Extraction (via Image EXIF data),

...culminating in rendering those coordinates onto an interactive map.

The Architectural Mindset: One Target, Two Pipelines

No matter how you obtain a coordinate in iOS, it always ends up formatted as a standard CLLocationCoordinate2D object. This uniformity means your user interface (like a Map) doesn't care how the coordinate was found; it treats them exactly the same way.

PART 1: The Live GPS Hardware Pipeline

Live tracking requires interacting with the phone's physical antenna chips. Because this touches user privacy, it requires an explicit system authorization handshake.

The CoreLocation Wrapper Engine

This reusable class inherits from NSObject to maintain Objective-C compatibility and adopts the CLLocationManagerDelegate protocol contract to receive automated satellite stream updates. 

import Foundation
import CoreLocation
import Observation // Required for the @Observable macro system

@Observable 
class LiveLocationEngine: NSObject, CLLocationManagerDelegate {
    private let manager = CLLocationManager()
    
    // These are now automatically observable, normal Swift variables!
    var lastKnownLocation: CLLocationCoordinate2D? = nil
    var authorizationStatus: CLAuthorizationStatus = .notDetermined

    override init() {
        super.init()
        manager.delegate = self
        manager.desiredAccuracy = kCLLocationAccuracyBest
        self.authorizationStatus = manager.authorizationStatus
    }

    func startTracking() {
        manager.requestWhenInUseAuthorization()
        manager.startUpdatingLocation()
    }
    
    func stopTracking() {
        manager.stopUpdatingLocation()
    }

    // DELEGATE PROTOCOL METHODS
    func locationManager(_ manager: CLLocationManager, didUpdateLocations locations: [CLLocation]) {
        guard let latestLocation = locations.first else { return }
        
        // UI updates must happen on the Main Actor
        DispatchQueue.main.async {
            self.lastKnownLocation = latestLocation.coordinate
        }
    }
    
    func locationManagerDidChangeAuthorization(_ manager: CLLocationManager) {
        DispatchQueue.main.async {
            self.authorizationStatus = manager.authorizationStatus
        }
    }
}

PART 2: The Photo EXIF Pipeline

EXIF data extraction requires zero hardware activation and zero location privacy prompt permissions because the coordinate is already cooked inside the file metadata bytes.

The Static File Extraction Engine

This utility reads raw image file blocks directly, drills past global image properties, and isolates the localized GPS directory layer. 

import Foundation
import CoreLocation
import ImageIO

struct PhotoMetadataEngine {
    static func extractGPS(from imageRawData: Data) -> CLLocationCoordinate2D? {
        guard let imageSource = CGImageSourceCreateWithData(imageRawData as CFData, nil) else { return nil }
        guard let containerProperties = CGImageSourceCopyPropertiesAtIndex(imageSource, 0, nil) as? [String: Any] else { return nil }
        guard let gpsDictionary = containerProperties[kCGImagePropertyGPSDictionary as String] as? [String: Any] else { return nil }
        
        guard let latVal = gpsDictionary[kCGImagePropertyGPSLatitude as String] as? Double,
              let latRef = gpsDictionary[kCGImagePropertyGPSLatitudeRef as String] as? String,
              let lonVal = gpsDictionary[kCGImagePropertyGPSLongitude as String] as? Double,
              let lonRef = gpsDictionary[kCGImagePropertyGPSLongitudeRef as String] as? String else { return nil }
        
        let absoluteLatitude = (latRef == "S") ? -latVal : latVal
        let absoluteLongitude = (lonRef == "W") ? -lonVal : lonVal
        
        return CLLocationCoordinate2D(latitude: absoluteLatitude, longitude: absoluteLongitude)
    }
}

PART 3: The Universal Map View Blueprint

This unified structural layout contains both architectures. Users can select whether they want to locate themselves live using the hardware toggle, or drop a pin down by choosing an existing photo from their album.

Unified Map Coordinate View

import SwiftUI
import MapKit
import PhotosUI

struct MapAnchorPoint: Identifiable {
    let id = UUID()
    let coordinate: CLLocationCoordinate2D
    let label: String
}

struct UniversalLocationMasterView: View {
    // Pipeline 1: Instantiated with modern @State instead of @StateObject!
    @State private var liveGPS = LiveLocationEngine()
    
    // Pipeline 2 state
    @State private var selectedPhotoItem: PhotosPickerItem? = nil
    
    // Unified Map States
    @State private var currentPins: [MapAnchorPoint] = []
    @State private var mapCamera: MapCameraPosition = .automatic
    
    var body: some View {
        NavigationStack {
            VStack(spacing: 20) {
                
                // UNIFIED MAP DRAWER
                Map(position: $mapCamera) {
                    ForEach(currentPins) { pin in
                        Marker(pin.label, coordinate: pin.coordinate)
                            .tint(pin.label.contains("Live") ? .green : .purple)
                    }
                }
                .frame(height: 350)
                .clipShape(.rect(cornerRadius: 16))
                .padding(.horizontal)
                
                // CONTROLS
                VStack(spacing: 12) {
                    Button(action: activateLiveHardwareCapture) {
                        Label("Use Current Live GPS", systemImage: "location.fill")
                            .frame(maxWidth: .infinity)
                            .padding(.vertical, 4)
                    }
                    .buttonStyle(.borderedProminent)
                    .tint(.green)
                    
                    PhotosPicker(selection: $selectedPhotoItem, matching: .images) {
                        Label("Extract From Photo EXIF", systemImage: "photo.fill")
                            .frame(maxWidth: .infinity)
                            .padding(.vertical, 4)
                    }
                    .buttonStyle(.borderedProminent)
                    .tint(.purple)
                }
                .padding(.horizontal)
                
                Spacer()
            }
            .navigationTitle("Modern GPS Guide")
            .onChange(of: selectedPhotoItem) { _, pickedItem in
                if let pickedItem { processPhotoExif(pickedItem) }
            }
        }
    }
    
    private func activateLiveHardwareCapture() {
        liveGPS.startTracking()
        
        if let currentHardwareCoordinate = liveGPS.lastKnownLocation {
            updateMapTarget(coordinate: currentHardwareCoordinate, tag: "Live Hardware Location")
        }
    }
    
    private func processPhotoExif(_ item: PhotosPickerItem) {
        Task {
            if let extractedRawBytes = try? await item.loadTransferable(type: Data.self) {
                if let extractedCoordinate = PhotoMetadataEngine.extractGPS(from: extractedRawBytes) {
                    await MainActor.run {
                        updateMapTarget(coordinate: extractedCoordinate, tag: "Photo EXIF Location")
                    }
                }
            }
        }
    }
    
    private func updateMapTarget(coordinate: CLLocationCoordinate2D, tag: String) {
        let newAnchor = MapAnchorPoint(coordinate: coordinate, label: tag)
        self.currentPins = [newAnchor]
        
        withAnimation(.easeInOut) {
            let areaRegion = MKCoordinateRegion(
                center: coordinate,
                span: MKCoordinateSpan(latitudeDelta: 0.03, longitudeDelta: 0.03)
            )
            self.mapCamera = .region(areaRegion)
        }
    }
}

Mandatory Framework Settings Checklist

Before compiling this project on a real physical device, you must configure your app targets to explicitly state your hardware intents to the security operating system:

  • Open your project settings in Xcode and navigate to the Info tab.
  • Hover over any line and click the + icon to add a new key string entry.
  • Select: Privacy - Location When In Use Usage Description.
  • Set the value to a clear descriptive string for your user (e.g., "This app requires connection to your phone's GPS arrays to plot your active coordinates onto the display map.").
Note: If this key configuration string is missing when your code executes manager.requestWhenInUseAuthorization(), the app runtime will crash instantly.

Observations On The Minutiae

It is completely normal to look at a production-grade class and feel like you've been hit with a tidal wave of new keywords. What you are looking at is the machinery that turns a standard Swift class into a SwiftUI-compatible data stream.

Let's demystify these mechanics one by one so you can understand the exact architectural purpose behind the minutiae.

  1. Why use private let manager instead of bare var?

In your tutorial, writing let manager = CLLocationManager() is completely valid code, but adding private is an industry best practice known as Encapsulation.

Swift private let manager = CLLocationManager() By marking the manager as private, you are telling the compiler: "Nobody outside of this specific class file is allowed to touch or modify the GPS hardware instance directly."

Imagine if you didn't use private. Another developer working on a different screen in your app could write code like this:

Swift // Catastrophe! Another screen turns off your hardware tracking without you knowing universalView.liveGPS.manager.stopUpdatingLocation() Using private locks the front door. If another view wants to interact with the location manager, it is forced to use your official, secure methods like .startTracking().

Additionally, changing var to let makes the variable a constant. Since you only ever need one engine chip controller instance for the lifetime of this class, making it a constant prevents your code from accidentally destroying or re-instantiating the hardware pipeline later.

  1. What is the @Published property wrapper?

Inside a standard Swift class, variables are silent. If a variable changes from nil to a GPS coordinate, your SwiftUI views have absolutely no idea that anything happened, so the screen remains blank.

The @Published wrapper turns a variable into a Broadcaster.

Whenever the location hardware feeds a new coordinate into lastKnownLocation, @Published instantly rings a giant alarm bell that yells to the entire app: "Hey! This data just changed! Anyone displaying this coordinate needs to redraw themselves right now!"

SwiftUI hears that broadcast, automatically re-runs your view's body layout property, and smoothly snaps the new pin onto your map file layout.

  1. The override init() and super.init() Handshake

Your guess is spot on! This is exactly how Swift manages the initialization chain when inheriting from Apple's base classes.

Swift override init() { 

super.init()
manager.delegate = self

} Because LiveLocationEngine inherits from NSObject, it inherits all of NSObject’s internal setup instructions.

override init(): This tells the compiler, "I know my parent class (NSObject) has its own blueprint setup instructions, but I need to add some custom, extra steps of my own right here."

super.init(): This tells Swift to jump up to the parent class (super) and run all of its original foundational memory allocation code first.

Think of it like building an extension onto a house. You call super.init() to lay down the original concrete foundation, and then you add your custom wall framing (manager.delegate = self) immediately afterward.

  1. What is the @StateObject property wrapper?

To understand @StateObject, we have to look at how SwiftUI views work under the hood. SwiftUI views are lightweight structs that are destroyed and recreated constantly whenever the screen updates or layout shifts occur.

If you initialized your tracking engine like a normal variable inside a view, your engine would get wiped out and restarted from scratch every single time the view redrew itself:

Swift // ❌ CRASH LOOP: Every view update destroys and recreates this tracker! var liveGPS = LiveLocationEngine() @StateObject creates a permanent storage vault inside the device memory that is entirely decoupled from the view's volatile layout lifecycle.

Swift @StateObject private var liveGPS = LiveLocationEngine() When SwiftUI recreates your view layout struct for a redraw, it looks at the @StateObject tag and says, "Ah, don't allocate a new engine. Keep pointing to that exact same live tracking instance we put in the vault earlier." It guarantees your data stream remains unbroken, continuous, and safe from unexpected memory deallocations!
