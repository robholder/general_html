@ViewBuilder

At its core, @ViewBuilder is a special compiler attribute (specifically, a custom result builder) that allows SwiftUI to interpret a list of separate views and combine them into a single, cohesive layout structure.

Whenever you write a list of views inside curly braces { }, @ViewBuilder is secretly working behind the scenes to glue them together.

1. Why do we need @ViewBuilder?

To understand why it exists, you have to look at standard Swift code vs. SwiftUI code.

In standard Swift, a function can only return one single thing. If you try to write three strings or integers back-to-back without separating them, Swift gets confused and throws a compiler error: 

// ❌ Standard Swift Error: "Consecutive statements on a line must be separated by ';'"
func getWords() -> String {
    "Hello"
    "World"
}

However, in SwiftUI, you do this constantly with views inside containers like VStack, HStack, or Group: 

//  This works perfectly! But why?
VStack {
    Text("Hello")
    Text("World")
}

The reason this works is that the closure { } attached to a VStack is secretly marked with the @ViewBuilder attribute.

Behind the scenes, the Swift compiler intercepts that list of views and automatically rewrites it into a special container type called a TupleView (e.g., TupleView<(Text, Text)>). It turns your separate view declarations into a single return type that SwiftUI can draw on the screen.

2. Where is it used by default?

You are using @ViewBuilder every day without even typing it, because Apple baked it into all the standard layout containers. If you command-click on a VStack initializer in Xcode, you will see its definition looks roughly like this: 

init(..., @ViewBuilder content: () -> Content)

The body property of every SwiftUI view also has an implicit @ViewBuilder wrapped around it. That is why you can stack multiple views inside a body property if they are wrapped in a container, or use if/else logic freely.

3. When do YOU need to type @ViewBuilder?

You will need to explicitly type @ViewBuilder yourself whenever you start building custom, reusable container views.

For example, imagine you want to create a custom "Card" view wrapper that gives whatever content you put inside it a rounded corner, a white background, and a subtle drop shadow.

The Code Example:

import SwiftUI

// A custom reusable card container
struct CustomCardView<Content: View>: View {
    let title: String
    
    // 1. We mark our custom content closure with @ViewBuilder
    @ViewBuilder let content: () -> Content
    
    var body: some View {
        VStack(alignment: .leading, spacing: 10) {
            Text(title)
                .font(.headline)
                .foregroundColor(.blue)
            
            // 2. Execute the view builder block to inject the views
            content()
        }
        .padding()
        .background(Color(.systemBackground))
        .cornerRadius(12)
        .shadow(radius: 5)
    }
}

// How you use it in your app:
struct ContentView: View {
    var body: some View {
        VStack {
            // Because 'content:' is a @ViewBuilder, we can pass ANY list of views inside the braces!
            CustomCardView(title: "User Profile") {
                Text("John Doe")
                Text("Software Engineer")
                Button("Follow") { }
            }
            
            CustomCardView(title: "System Status") {
                HStack {
                    Image(systemName: "checkmark.circle.fill").foregroundColor(.green)
                    Text("All systems operational")
                }
            }
        }
        .padding()
    }
}

4. Key Rules of @ViewBuilder

  • The 10-View Limit (Historically): In older versions of SwiftUI, a @ViewBuilder could only hold a maximum of 10 child views inside a single block. If you went to 11, Xcode threw a cryptic error. Modern Swift has mostly eliminated this restriction, but if you have a massive layout, it's still best practice to break it chunks using Group or sub-views.
  • Logic Support: @ViewBuilder natively understands if, if/else, and switch statements. It does not natively support complex loops like for-in inside its blocks (which is why we use ForEach instead).

Summary Rule of Thumb

Think of @ViewBuilder as the invisible construction worker of SwiftUI. It takes individual, loose pieces of layout material (your views) and bundles them up into a single package so SwiftUI can safely render them onto the screen.
