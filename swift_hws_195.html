Hacking With Swift - 195 - Using Touch ID and Face ID with SwiftUI

The vast majority of Apple’s devices come with biometric authentication as standard, which means they use fingerprint, facial, and even iris recognition to unlock. This functionality is available to us too, which means we can make sure that sensitive data can only be read when unlocked by a valid user.

This is another Objective-C API, but it’s only a little bit unpleasant to use with SwiftUI, which is better than we’ve had with some other frameworks we’ve looked at so far.

FaceID - Setup Key

Before we write any code, you need to add a new key to your project options, explaining to the user why you want access to Face ID. For reasons known only to Apple, we pass the Touch ID request reason in code, and the Face ID request reason in project options.

So, select your current target:

  • Click on top level of project,
  • Click on TARGETS section,
  • Go to the Info tab,
  • Right-click on an existing key (any in the list):
    • Choose Add Row.
    • Or: Click on last row's + icon to create new key.
  • Scroll through the list of keys until you find "Privacy - Face ID Usage Description"
  • and give it the Value (value column):
    • We need to unlock your data

Now head back to ContentView.swift, and add this import near the top of the file: 

import LocalAuthentication

And with that, we’re all set to write some biometrics code.

I mentioned earlier this was “only a little bit unpleasant”, and here’s where it comes in: Swift developers use the Error protocol for representing errors that occur at runtime, but Objective-C uses a special class called NSError. We need to be able to pass that into the function and have it changed inside the function rather than returning a new value - although this was the standard in Objective-C, it’s quite an alien way of working in Swift so we need to mark this behavior specially by using &.

We’re going to write an authenticate() method that isolates all the biometric functionality in a single place. To make that happen requires four steps:

  • Create instance of LAContext, which allows us to query biometric status and perform the authentication check.
  • Ask that context whether it’s capable of performing biometric authentication - this is important because iPod touch has neither Touch ID nor Face ID.
  • If biometrics are possible, then we kick off the actual request for authentication, passing in a closure to run when authentication completes.
  • When the user has either been authenticated or not, our completion closure will be called and tell us whether it worked or not, and if not what the error was.

Please go ahead and add this method to ContentView: 

func authenticate() {
    let context = LAContext()
    var error: NSError?
    
    // check whether biometric authentication is possible
    if context.canEvaluatePolicy(
        .deviceOwnerAuthenticationWithBiometrics,
        error: &error) {
        // it's possible, so go ahead and use it
        // Match the value key from Target > Info key
        let reason = "We need to unlock your data"
        
        context.evaluatePolicy(
            .deviceOwnerAuthenticationWithBiometrics,
            localizedReason: reason) { success, authenticationError in
            // authentication has no completed
            if success {
                //authenticated successfully
            } else {
                // there was a problem
            }
        }
    } else {
        // no biometrics
    }
}

That method by itself won’t do anything, because it’s not connected to SwiftUI at all. To fix that we need to add some state we can adjust when authentication is successful, and also an onAppear() modifier to trigger authentication.

So, first add this property to ContentView: 

@State private var isUnlocked = false

That simple Boolean will store whether the app is showing its protected data or not, so we’ll flip that to true when authentication succeeds. Replace the // authenticated successfully comment with this: 

isUnlocked = true

Finally, we can show the current authentication state and begin the authentication process inside the body property, like this: 

VStack {
    if isUnlocked {
        Text("Unlocked")
    } else {
        Text("Locked")
    }
}
.onAppear(perform: authenticate)

If you run the app there’s a good chance you just see “Locked” and nothing else. This is because the simulator isn’t opted in to biometrics by default, and we didn’t provide any error messages, so it fails silently.

To take Face ID for a test drive in the simulator:

  • go to the Features menu and choose:
    • Face ID > Enrolled,
  • Then launch the app again.

This time you should see the Face ID prompt appear, and you can trigger successful or failed authentication by going back to the Features menu and choosing

  • Face ID > Matching Face
  • Face ID >Non-matching Face.

All being well you should see the Face ID prompt go away, and underneath it will be the “Unlocked” text view - our app has detected the authentication, and is now open to use.

Important: When working with biometric authentication, you should always look for a backup plan that lets users authenticate without biometrics. This usually means adding a screen that prompts for a passcode then providing that as a fallback if biometrics fail, but this is something you need to build yourself.

In Context

ContentView.swift

import LocalAuthentication
import SwiftUI

struct ContentView: View {
    @State private var isUnlocked = false

    var body: some View {
        VStack {
            if isUnlocked {
                Text("Unlocked")
            } else {
                Text("Locked")
            }
        }
        .onAppear(perform: authenticate)
    }
    
    func authenticate() {
        let context = LAContext()
        var error: NSError?
        
        // check whether biometric authentication is possible
        if context.canEvaluatePolicy(
            .deviceOwnerAuthenticationWithBiometrics,
            error: &error) {
            // it's possible, so go ahead and use it
            // Match the value key from Target > Info key
            let reason = "We need to unlock your data"
            
            context.evaluatePolicy(
                .deviceOwnerAuthenticationWithBiometrics,
                localizedReason: reason) { success, authenticationError in
                // authentication has no completed
                if success {
                    //authenticated successfully
                    isUnlocked = true
                } else {
                    // there was a problem
                }
            }
        } else {
            // no biometrics
        }
    }
}

Ai Insight - Offering PIN Access on FaceID Fail

QUESTION

I have this SwiftUI ContentView.swift code (above), and while it asks for FaceID authentication, there is no prevision for a backup (like an 'enter pin' option). Can you suggest how this facility could be incorporated into this code?

ANSWER

This is a very common requirement when working with LocalAuthentication. Currently, you are using the policy .deviceOwnerAuthenticationWithBiometrics. This policy strictly tells iOS: "Use Face ID or Touch ID. If they fail or aren't available, stop right there."

To automatically get the "Enter Passcode/PIN" backup option provided by iOS, you need to change your authentication policy to .deviceOwnerAuthentication.

Here is how you can update your code to seamlessly support both biometrics and the device passcode fallback.

The Updated Code

import LocalAuthentication
import SwiftUI

struct ContentView: View {
    @State private var isUnlocked = false

    var body: some View {
        VStack {
            if isUnlocked {
                Text("Unlocked")
            } else {
                Text("Locked")
            }
        }
        .onAppear(perform: authenticate)
    }
    
    func authenticate() {
        let context = LAContext()
        var error: NSError?
        
        // 1. Change policy to allow BOTH biometrics and device passcode fallback
        if context.canEvaluatePolicy(
            .deviceOwnerAuthentication, 
            error: &error
        ) {
            let reason = "We need to unlock your data"
            
            // 2. Evaluate with the same fallback-supported policy
            context.evaluatePolicy(
                .deviceOwnerAuthentication, 
                localizedReason: reason
            ) { success, authenticationError in
                
                // 3. Make sure to update UI on the Main Thread
                DispatchQueue.main.sync {
                    if success {
                        isUnlocked = true
                    } else {
                        // The user cancelled, or failed both FaceID and Passcode
                        if let error = authenticationError as? LAError {
                            print("Authentication failed: \(error.localizedDescription)")
                        }
                    }
                }
            }
        } else {
            // No biometrics AND no passcode set up on this device at all
            print("No authentication methods available.")
        }
    }
}

#Preview {
    ContentView()
}

What Changed Under the Hood?

1. The Policy Swap

By switching from .deviceOwnerAuthenticationWithBiometrics to .deviceOwnerAuthentication, you change how iOS behaves when Face ID fails or when the user opts out:

  • First Try: iOS will attempt to use Face ID or Touch ID normally.
  • The Backup: If Face ID fails (or if the user taps a button on the alert), iOS will automatically replace the alert with the standard system screen asking for the device's lock screen PIN/Passcode.

2. Thread Safety (DispatchQueue.main.sync)

You'll notice I wrapped isUnlocked = true inside DispatchQueue.main.sync.

LocalAuthentication runs its checks on a background thread. In SwiftUI, whenever you change a @State variable that updates the user interface (like switching text from "Locked" to "Unlocked"), you must do that work on the Main Thread. Without this, Xcode might throw a runtime warning or your app's UI might freeze.

A Note on Testing in the Simulator

When testing this in the Xcode Simulator:

  • Go to the Simulator menu bar and choose Features > Face ID > Enrolled to mock a device with Face ID.
  • Trigger the authentication. If you want to test the passcode fallback, select Features > Face ID > Matching Face (to succeed) or Non-matching Face (to trigger the fallback prompt).

What Happens When FaceID Fails - Physical Device

When Face ID "fails" (meaning it looks at your face but doesn't recognize you), iOS intentionally does not drop you straight into the PIN passcode screen. Instead, it gives you a second chance to try again.

How Face ID Behaves on a Real Device

If you run the multi-stage code on your iPhone 14 Pro, the workflow proceeds through these deliberate phases:

  • First Failure: You look at the phone, Face ID fails to recognize you, and the little padlock icon shakes. The Face ID prompt stays on screen, displaying a button that says "Try Face ID Again".
  • Second Failure (or tapping the screen): If it fails a second time, a new button labeled "Use Passcode" (the text we customized via localizedFallbackTitle) appears on the prompt.
  • The Trigger: The PIN dialog only appears after you explicitly tap that "Use Passcode" button.

Apple designs it this way because jumping directly to a full-screen passcode view every time Face ID misses a glance would become incredibly intrusive.
