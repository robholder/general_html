Hacking With Swift - 220 - Key points

Key Points

There are two topics I want to dive into more today, both of which are quite advanced but both are things that I know will really help deepen your understanding of what was covered in the previous three projects.

Operator overloading

When we added a custom conformance to Comparable we need to add one method called <. This in turn allows Swift to compare expressions such as a < b, which is what gave us access to a version of sorted() that took no parameters.

This is called operator overloading, and it’s what allows us to add two integers or join two strings using the same + operator. You can define your own operators if you want, but it’s also easy to extend existing operators to do new things.

As an example, we could add some extensions to Int that lets us multiply an Int and a Double - something that Swift doesn’t allow by default, which can be annoying: 

extension Int {
    static func *(lhs: Int, rhs: Double) -> Double {
        return Double(lhs) * rhs
    }
}

Pay particular attention to the parameters: it uses an Int as the left-hand operand and a Double as the right-hand operand, which means it won’t work if you swap those two around. So, if you want to be complete - if you want either order to work - you need to define the method twice.

However, if you want to be really complete then extending Int is the wrong choice: we should go up to a protocol that wraps Int as well as other integer types such as the Int16 we used with Core Data. Swift puts all integer types into a single protocol called BinaryInteger, and if we write an extension on that then Self (with a capital S) refers to whatever specific type is being used. So, if it’s used on an Int then Self means Int, and if it’s used on an Int16 then it means that instead.

Here’s an extension that adds * between any kind of integer and Double, regardless of whether the integer is on the left or on the right: 

extension BinaryInteger {
    static func *(lhs: Self, rhs: Double) -> Double {
        return Double(lhs) * rhs
    }

    static func *(lhs: Double, rhs: Self) -> Double {
        return lhs * Double(rhs)
    }
}

If you were wondering, there is a reason Swift doesn’t enable these operators for us: it’s not guaranteed to be as accurate as you might hope. As a simple example, try this: 

let exampleInt: Int64 = 50_000_000_000_000_001
print(exampleInt)

let result = exampleInt * 1.0
print(String(format: "%.0f", result))

That creates a 64-bit integer holding 50 quadrillion and one. It then multiplies it by the Double 1.0 using our custom extension, which in theory means the number that comes back should be identical to the Int. However, that String(format:_:) call asks for the number to be printed with no decimal places, and you’ll see it isn’t the same as the integer: it is 50 quadrillion without the 1. Now, you might ask “what’s 1 when you’re working with 50 quadrillion”, and that’s fine - I’m not here to tell you what’s right and wrong, only to say that if you want absolute accuracy you should avoid using these kinds of helper methods.

More generally, I want to give you a warning about operator overloading. When I introduced it back in project 14 I said operator overloading “can be both a blessing and a curse,” and I want to touch briefly on why that is.

Consider code like this: 

let paul = User()
let swift = Language()

let result = paul + swift

What type of data is result? I can think of several possibilities:

  • Maybe it’s another User object, now modified so that it has Swift in its array of known languages.
  • Maybe it’s a Programmer object combining a user and a language.
  • Maybe it’s a weird remake of the classic horror movie The Fly.

The point is that we don’t know, and we can’t really tell without reading into the source code for the relevant + operator.

Now consider this code: 

let paul = User()
let swift = Language()
paul.learn(swift)

This is much clearer, I think: we’re now running a simple method on an object, and you’d probably guess that we’re mutating paul to include Swift in the array of programming languages.

Any good developer will tell you that clarity is one of the most important features of well-written code - we need to make our meaning clear in what we write, because it will be read dozens or hundreds of times in the future.

So, by all means add operator overloading to your arsenal of skills that you can deploy to solve problems, and indeed I go into way more detail about them in my book Pro Swift, but always use them with care.

Custom property wrappers

You’ve seen that property wrappers are really just some sleight of hand: they take a simple value and wrap it in another another value so that some extra functionality can be added. This might be the way SwiftUI uses @State to store values elsewhere, or how it uses @Environment to read values from a shared data source, but the principle is the same: it takes a simple value and gives it super-powers somehow.

We can use property wrappers in our own code, and there are lots of reasons why you might want to do so. As with operator overloading you’ll understand more about how things work if you try them out, but it’s also worth using them thoughtfully: if they are the first thing you reach for you’re probably making a mistake.

To demonstrate property wrappers, I want to start with a simple struct that wraps some kind of BinaryInteger value. We’re going to give this thing some custom code when it comes to setting its wrapped value, so that if the new value is below 0 we instead make it exactly 0 so that this struct can never be negative.

Ai Insight - BinaryIntegers

The Flexible BinaryInteger

By changing your extension to use BinaryInteger instead of Int, you break down a barrier. You are telling Swift: "I don't care how many bits this number is, or if it's signed or unsigned. As long as it's a whole number, allow it."

Here is how you write that using generics: 

extension View {
    // T can be an Int, Int16, UInt64, UInt8, etc.
    func itemBadge<T: BinaryInteger>(count: T) -> some View {
        Text("\(Int(count))") // We convert it inside the engine, so the user doesn't have to
            .padding(5)
            .background(.red)
            .clipShape(.circle)
    }
}

Why this is a superpower:

Now, your layout code is completely agnostic to the underlying database architecture. A developer can write all three of these lines cleanly without a single compiler error: 

ItemRow().itemBadge(count: standardInt)       // Works! (Int)
ItemRow().itemBadge(count: smallCartQuantity) // Works! (Int16)
ItemRow().itemBadge(count: massiveDatabaseID) // Works! (UInt64)

Summary Checklist

  • Use Int when you are declaring actual variables, storing properties in your models/ViewModels, or doing standard math loop calculations.
  • Look for / Use BinaryInteger when you are writing reusable utility functions, math extensions, or architectural framework tools that need to accept diverse integer types seamlessly.

Our code would look like this:

Back to tutorial

struct NonNegative<Value: BinaryInteger> {
    var value: Value

    init(wrappedValue: Value) {
        if wrappedValue < 0 {
            value = 0
        } else {
            value = wrappedValue
        }
    }

    var wrappedValue: Value {
        get { value }
        set {
            if newValue < 0 {
                value = 0
            } else {
                value = newValue
            }
        }
    }
}

Ai Insight (the struct above)

The struct above is a textbook example of Generics! In fact, it is an incredibly clever and useful one.

You are looking at the foundational pattern for a custom Property Wrapper (similar to how @State or @AppStorage work under the hood) that automatically safeguards numbers from ever dropping below zero.

Let’s demystify that exact syntax <Value: BinaryInteger> so you can see why it's written that way and how the "placeholder" mechanics work.

Breaking Down the Syntax:

Think of this syntax as a two-part contract for a structural blueprint: 

struct NonNegative<Value: BinaryInteger> { ... }
//                └──┬─┘ └──────┬──────┘
//             Placeholder    The Constraint (Rule)

1. The Placeholder (Value)

The word Value inside the angle brackets is a generic type placeholder. It doesn't exist yet as a real data type. It is simply a temporary label saying: _"A real whole-number type will be plugged in here later when this struct is actually born."

Note: You can name this anything! You could name it <T>, <Number>, or <Eggplant>. Paul Hudson and Apple developers tend to use Value or T by convention.

2. The Type Constraint (: BinaryInteger)

The colon : followed by BinaryInteger acts as a strict guardrail. It tells Swift: "The developer can plug in any data type they want for Value, BUT it absolutely must conform to the BinaryInteger protocol."

Without this guardrail, Swift would refuse to compile your code. Why? Look inside your initializer: 

if wrappedValue < 0 { ... }

If Value could be absolutely anything, someone might try to create a NonNegative<String> or a NonNegative<CUPCAKE>. Swift looks at that and asks: "How am I supposed to check if a String or a Cupcake is less than zero (< 0)?" By constraining Value to BinaryInteger, you guarantee to the Swift compiler that whatever type is passed in will natively understand basic mathematical comparisons like < 0.

How it works in practice (The Blueprint Multiplier)

Because of generics, you only had to write this code once, but you can now use it to manufacture completely different concrete structures based on your data needs.

Behind the scenes, when you use it, Swift substitutes your Value placeholder with a real, concrete type:

Example A: A standard Player Score

// Swift reads this and replaces every instance of 'Value' with 'Int'
var score = NonNegative<Int>(wrappedValue: 10)
score.wrappedValue -= 15 
print(score.wrappedValue) // Output: 0 (Safely clamped!)

Example B: A tiny space-saving Bluetooth age tracker

// Swift reads this and replaces every instance of 'Value' with 'UInt8'
var userAge = NonNegative<UInt8>(wrappedValue: 25)

Why this is so elegant

If Swift didn't have generics combined with BinaryInteger, you would have to copy and paste this exact same logic over and over again to support different number architectures:

  • struct NonNegativeInt
  • struct NonNegativeInt16
  • struct NonNegativeUInt32

Instead, your single generic NonNegative struct acts as a chameleon blueprint, adapting itself perfectly to whatever integer type your app's architecture requires!

Back to tutorial

We can now create that with an integer, but if that integer drops below 0 then it will be clamped to 0. So, this will print 0: 

var example = NonNegative(wrappedValue: 5)
example.wrappedValue -= 10
print(example.wrappedValue)

What property wrappers let us do is use that for any kind of property in a struct or class. Even better, it only takes one step: writing @propertyWrapper before the NonNegative struct, like this: 

@propertyWrapper
struct NonNegative<Value: BinaryInteger> {

That’s it - we now have our own property wrapper!

In case you hadn’t guessed from their name, property wrappers can only be used on properties rather than plain variables or constants, so to try ours out we’re going to put it inside a User struct like this: 

struct User {
    @NonNegative var score = 0
}

And now we can create a user and add or remove points freely, knowing for sure the score will never go below 0: 

var user = User()
user.score += 10
print(user.score)

user.score -= 20
print(user.score)

As you can see, there really is no magic here: not only are property wrappers just syntactic sugar that causes one piece of data to be wrapped around another, but we can also make them ourselves if needed.

Ai Insight - value & wrappedValue

In the generic struct above (with 'value' and 'wrappedValue'), it looks like a redundant double-handling of the incoming parameter at first glance, but there is a brilliant mechanical reason for separating value from wrappedValue.

To understand why we need both, we have to look at where the data is stored versus how the outside world interacts with it.

1. What is the purpose of var value? (The Storage Vault)

var value: Value is the actual, private storage tank where the raw number lives inside the computer's memory.

The structure cannot just have wrappedValue. In Swift, a computed property (a variable with a get and set block) cannot store data itself. It is just a gateway - a piece of logic. It requires a physical, underlying variable to hold the actual bits in RAM.

  • var value: The actual physical safe inside the house.
  • var wrappedValue: The front door with a security guard standing next to it.

2. How does the structure know where to apply the incoming parameter?

The bridge between the incoming parameter and your physical storage is built entirely inside the initializer (init).

When you instantiate this struct, you pass a number into the init(wrappedValue:) front door. The initializer contains the logic that intercepts that number, checks it, and then physically writes it to your storage tank (value): 

init(wrappedValue: Value) {
    if wrappedValue < 0 {
        self.value = 0          // 1. If incoming is negative, clamp the storage to 0
    } else {
        self.value = wrappedValue // 2. Otherwise, save the incoming number directly into storage
    }
}

This is the exact moment the layout engine maps the incoming raw material directly into the internal value variable.

3. Why not just change the incoming value directly?

You ask: Why not just modify the incoming value instead of maintaining two separate variables?

The problem happens after birth, when the app is running and the user tries to update the number later (like losing points in a game or changing an age).

Because wrappedValue has a set block, it intercepts future changes seamlessly. Look at what happens if a developer tries to set a player's score to a negative number later on in the app's execution: 

// Outside in your ContentView:
score.wrappedValue = -10

This triggers the set block inside your struct: 

set {
    if newValue < 0 {
        value = 0          // Intercepted! The safe gets a 0.
    } else {
        value = newValue    // Safe gets the clean positive number.
    }
}

If you didn't have the set block acting as a gatekeeper to protect a hidden value storage tank, the outside world could easily bypass your rules and write a -10 directly into your memory, breaking your "never negative" guarantee.

The Complete Picture

Here is the exact division of labor between the two properties when the struct is running:

  • var value (The Storage): Completely private. It holds the pristine, validated, legally allowed number.
  • var wrappedValue (The Interface): Public. This is what SwiftUI and your external views look at. Its get block simply reaches into the safe to read value, and its set block acts as a filter to make sure no illegal numbers make it inside!

Ai Insight - Declaring Output Types

QUESTION

Why do some functions have output types declared, and others don't? 

func itemBadge<T: BinaryInteger>(count: T) -> some View {

init(wrappedValue: Value) { // <-- No arrow here!

ANSWER

This is an excellent question that touches on the fundamental difference between Functions (which do actions) and Initializers (which build things).

The short answer is: Functions must declare an output type because they return a piece of data you can use elsewhere, whereas initializers (like init) exist solely to set up a brand-new object, so an output type is already implied.

Let’s look at the mechanics of why the itemBadge function has -> some View but the init in NonNegative has nothing.

1. Functions: The Factory Workers

Think of a standard function as a factory machine. You feed it raw materials (parameters), it processes them, and it spits out a completely new product (the return value).

In your itemBadge example: 

func itemBadge<T: BinaryInteger>(count: T) -> some View { ... }

Because this is a standalone function, the Swift compiler needs to know what this machine is going to spit out when it's done. The -> some View syntax is you promising the compiler: "When you run this function, it will return a UI element that conforms to the View protocol."

Without that declaration, Swift wouldn't know what to do with the result when you call it in your layouts: 

// Swift needs to know that itemBadge returns a View so it can draw it on screen!
Circle()
    .itemBadge(count: 5)

2. Initializers: The Assembly Line

Now let's look at the init inside your generic NonNegative struct. Notice that it completely lacks an arrow (->): 

struct NonNegative<Value: BinaryInteger> {
    var value: Value

    init(wrappedValue: Value) { // <-- No arrow here!
        // Configuration code...
    }
}

An init is an initializer. Its only job in the entire universe is to construct an instance of the struct it lives inside. It is the assembly line that prepares the data inside the object.

  • It never returns an independent value. Its output is always implicitly a brand-new instance of the struct itself (NonNegative).
  • Adding a return type like -> NonNegative would be redundant, because an initializer cannot physically return anything else.

3. What about Computed Properties?

You might have also noticed this block in your generic code: 

var wrappedValue: Value {
    get { value }
    set { ... }
}

This is a computed property. It doesn't use a function arrow (->), but it does declare a type (: Value).

This acts exactly like a mini-read/write function. The : Value colon is required here to tell Swift: "Whenever someone tries to read (get) or modify (set) this property, they will receive or must provide a piece of data matching our generic Value type."

Summary Checklist

Code Element Does it have a Type Arrow (->)? Why or Why Not?
Functions (func) Yes (e.g., -> some View) It performs an operation and returns a new piece of data.
Initializers (init) No It only configures the current struct/class. Its output type is always itself.
Properties (var) No (Uses a colon : instead) It declares the specific type of data that the variable holds or computes.
