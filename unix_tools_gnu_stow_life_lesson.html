GNU Stow - A Life Lesson

I want to use the --dotfiles flag with stow, so i've named some files and directories with the dot- prefix. I though perhaps, I could use this structure so that I could back this all to a git repo, and when cloned, run stow on each app's subdirectory as needed (wezterm, zshrc, etc.).

NOTE: The --dotfiles flag in stow interprets dot-zshrc as .zshrc when naming the symlink.

QUESTION

...would this work? 

dotfiles
├── README.md
├── robsaliases
│   └── dot-robsaliases
├── starship
│   ├── gruvbox_rainbow.starship.toml
│   ├── pastel_powerline.starship.toml
│   ├── starship.toml
│   └── tokyo_night.starship.toml
├── vim
│   └── dot-vim
├── wezterm
│   └── wezterm.lua
└── zshrc
    ├── dot-zprofile
    └── dot-zshrc

ANSWER

Your structure can work with Stow’s --dotfiles mode — but a couple of tweaks will make it behave exactly the way you want.

Let’s walk through what Stow expects, what your layout implies, and how to make it smooth and predictable.

✅ How --dotfiles actually works

When you run: 

stow --dotfiles <package>

Stow interprets any file or directory beginning with: 

dot-<name>

...as something that should become: 

.<name>

...in the target directory.

So: 

dot-zshrc     →   ~/.zshrc
dot-vim       →   ~/.vim
dot-robsaliases → ~/.robsaliases

This part of your plan is correct.

📁 Your proposed structure (annotated)

dotfiles
├── README.md
├── robsaliases
│   └── dot-robsaliases        ✔ becomes ~/.robsaliases
├── starship
│   ├── gruvbox_rainbow.starship.toml
│   ├── pastel_powerline.starship.toml
│   ├── starship.toml          ✘ this will NOT become ~/.config/starship.toml
│   └── tokyo_night.starship.toml
├── vim
│   └── dot-vim                ✔ becomes ~/.vim
├── wezterm
│   └── wezterm.lua            ✘ becomes ~/wezterm.lua (not ~/.config/wezterm/wezterm.lua)
└── zshrc
    ├── dot-zprofile           ✔ becomes ~/.zprofile
    └── dot-zshrc              ✔ becomes ~/.zshrc

So far, the dot- prefixed items will work perfectly.

But the Starship and WezTerm directories need a different layout if you want them to land in the correct XDG config paths.

❗ The issue: Stow only mirrors your directory structure

Stow does not magically know that:

  • Starship config belongs in ~/.config/starship/starship.toml
  • WezTerm config belongs in ~/.config/wezterm/wezterm.lua

If you want Stow to place files there, your repo must mirror that structure.

✅ Recommended layout for your repo

Here’s the version that will work exactly as intended: 

dotfiles
├── robsaliases
│   └── dot-robsaliases
├── starship
│   └── .config
│       └── starship
│           ├── starship.toml
│           ├── gruvbox_rainbow.starship.toml
│           ├── pastel_powerline.starship.toml
│           └── tokyo_night.starship.toml
├── vim
│   └── dot-vim
├── wezterm
│   └── .config
│       └── wezterm
│           └── wezterm.lua
└── zshrc
    ├── dot-zprofile
    └── dot-zshrc

Then you run: 

stow --dotfiles -t ~ starship
stow --dotfiles -t ~ wezterm
stow --dotfiles -t ~ zshrc
stow --dotfiles -t ~ vim
stow --dotfiles -t ~ robsaliases

This produces: 

~/.config/starship/starship.toml
~/.config/wezterm/wezterm.lua
~/.zshrc
~/.zprofile
~/.vim
~/.robsaliases

Exactly what you want.

🧠 Why this works

Stow simply:

  • Takes the directory you give it (e.g., starship/)
  • Mirrors its internal structure into the target directory
  • Applies the dot- → . transformation if --dotfiles is used

So the key is: your repo must reflect the final desired directory layout.

  • ✔ Yes, your idea works

  • ✔ Everything with dot- will behave correctly
  • ✔ This setup is perfect for a Git-backed dotfiles repo

QUESTION

From where in the structure would you run Stow from?

Like this: 

stow --dotfiles -t ~ starship
stow --dotfiles -t ~ wezterm
stow --dotfiles -t ~ zshrc
stow --dotfiles -t ~ vim
stow --dotfiles -t ~ robsaliases

ANSWER

You always run stow from the top‑level directory that contains all your packages — in your case, the dotfiles/ directory.

That’s the whole design of Stow: the directory you run it from becomes the “stow directory,” and each subdirectory becomes a package.

So with your structure: 

dotfiles/
├── robsaliases/
├── starship/
├── vim/
├── wezterm/
└── zshrc/

...you should run: 

cd ~/dotfiles
stow --dotfiles -t ~ starship
stow --dotfiles -t ~ wezterm
stow --dotfiles -t ~ zshrc
stow --dotfiles -t ~ vim
stow --dotfiles -t ~ robsaliases

Running Stow inside a package directory (e.g., inside starship/) will cause the exact problem you saw earlier — symlinks ending up one directory above where you want them.

Why this works

When you run: 

cd ~/dotfiles
stow --dotfiles -t ~ starship

Stow interprets:

  • stow directory → ~/dotfiles
  • package → starship/
  • target → ~

So it mirrors the structure of: 

dotfiles/starship/

...into: 

$HOME/

...with dot- prefixes converted to leading dots.

Quick mental model

  • Run Stow from the parent directory of your packages.
  • Each subdirectory is a package.
  • Use -t ~ to force installation into your home directory.
  • Use --dotfiles to convert dot-foo → .foo.

This keeps everything predictable and clean.

QUESTION

When I tried a wezterm setup as follows (inside a directory in the home directory called dotfiles): 

>cd ~/dotfiles

>eza -Ta 

wezterm
└── dot-config
    └── wezterm
        └── wezterm.lua

...and ran 

stow --dotfiles -t ~ wezterm

I got this: 

 .config -> dotfiles/wezterm/dot-config/

ANSWER

You actually uncovered a very normal Stow behavior — and the good news is nothing is “wrong” with your setup. What you’re seeing happens because of how Stow treats directories when using --dotfiles.

Let me walk you through exactly why you got: 

.config -> dotfiles/wezterm/dot-config/

...instead of the nested structure you expected.

✅ Why Stow created a symlink to .config instead of .config/wezterm/wezterm.lua

Your structure: 

dotfiles/
└── wezterm/
    └── dot-config/
        └── wezterm/
            └── wezterm.lua

When you run: 

stow --dotfiles -t ~ wezterm

Stow sees:

  • A package named wezterm
  • Inside it, a top‑level directory named dot-config

Because of --dotfiles, Stow rewrites: 

dot-config → .config

And since .config is the top-level item inside the package, Stow creates a single symlink: 

~/.config → dotfiles/wezterm/dot-config

This is expected behavior.

Stow only creates nested symlinks when the top-level directory already exists in the target.

🧠 Why this happens

Stow has two modes of operation:

1. If the target directory does NOT exist

Stow creates a single symlink to the entire directory.

Example: 

~/.config does NOT exist

Then Stow does: 

ln -s dotfiles/wezterm/dot-config ~/.config

2. If the target directory DOES exist

Stow creates individual symlinks inside it.

Example: 

mkdir -p ~/.config

Then run: 

stow --dotfiles -t ~ wezterm

Now Stow will create: 

~/.config/wezterm → dotfiles/wezterm/dot-config/wezterm

And inside that: 

~/.config/wezterm/wezterm.lua → dotfiles/.../wezterm.lua

🎯 The fix: ensure ~/.config exists before running Stow

Just create the directory: 

mkdir -p ~/.config

Then re-run: 

stow --dotfiles -t ~ wezterm

Now you’ll get the structure you intended: 

~/.config/wezterm/wezterm.lua

🧩 Why this is the correct approach

Stow is designed to:

  • Symlink whole directories when the target doesn’t exist
  • Populate existing directories when they do

This prevents Stow from accidentally overwriting or replacing important directories.

👍 Final answer

Yes — your structure is correct.

You just need to ensure the target directory exists: 

mkdir -p ~/.config
stow --dotfiles -t ~ wezterm

Then Stow will place: 

~/.config/wezterm/wezterm.lua

...instead of symlinking .config itself.
