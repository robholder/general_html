Hacking With Swift - 117 - Finishing up with one last view

To finish this program we’re going to make a third and final view to display astronaut details; shown by tapping one of the astronauts in the mission view.

Start by making a new SwiftUI view called AstronautView.

This will have a single Astronaut property so it knows what to show, then it will lay that out using a similar ScrollView/VStack combination as we had in MissionView.

Give it this code: 

struct AstronautView: View {
    let astronaut: Astronaut

    var body: some View {
        ScrollView {
            VStack {
                Image(astronaut.id)
                    .resizable()
                    .scaledToFit()

                Text(astronaut.description)
                    .padding()
            }
        }
        .background(.darkBackground)
        .navigationTitle(astronaut.name)
        .navigationBarTitleDisplayMode(.inline)
    }
}

Once again we need to update the preview so that it creates its view with some data: 

#Preview {
    let astronauts: [String: Astronaut] = Bundle.main.decode("astronauts.json")

    return AstronautView(astronaut: astronauts["aldrin"]!)
        .preferredColorScheme(.dark)
}

The ! Force Unwrap

In Swift, that exclamation mark after ["aldren"] is called the Force Unwrap Operator.

It is you telling the compiler: "I know for a fact that a value exists here. Don't worry about checking it—just rip it open and give me the data."

1. The "Why" behind the symbol

Whenever you look something up in a Dictionary—like astronauts["aldrin"], Swift returns an Optional.

Swift is being cautious. It thinks: "You're asking for 'aldrin', but what if that key doesn't exist in the file? What if it's misspelled as 'alldrin'?" If it can't find the key, it returns nil.

Because AstronautView expects a real, solid Astronaut (not an optional one), the code won't compile unless you "unwrap" that optional.

2. The Danger Zone

The reason it’s called "Force" unwrapping is because if you are wrong—if "aldrin" is actually missing from your JSON, the app will crash instantly. It is the equivalent of telling Swift: "I'm so certain 'aldrin' is in that file that I'm willing to bet the life of the app on it."

3. Why is it okay to use here? (The Preview Context)

You will notice Paul Hudson (and most developers) use ! in two specific places:

  • Previews: Since this code only runs while you are designing in Xcode, a crash isn't a big deal. You'd rather the preview crash and tell you "Hey, you misspelled Aldrin" than have to write complex error-handling code just for a mockup.
  • Tests: Similar to previews, if a test fails because of a missing hardcoded value, that's a signal you need to fix your data.

In your actual App logic (like in ContentView ), you should almost never use !. There, you would use if let or guard let to handle the "what if" safely.

4. Comparison of Unwrapping

Method Syntax Result if value is missing
Force Unwrap astronauts["aldrin"]! CRASH
Optional Chaining astronauts["aldrin"]? The app continues, but returns nil
Nil Coalescing ... ?? defaultAstronaut The app uses a fallback value
Safe Unwrapping if let astronaut = ... The app executes code only if found

Adding the Call to AstronautView in MissionView

Now we can present that from the NavigationLink inside MissionView. This points to Text("Astronaut details") right now, but we can update it to point to our new AstronautView instead: 

AstronautView(astronaut: crewMember.astronaut)

If you run the app now you’ll see how natural it makes our user interface feel - we start at the broadest level of information, showing all our missions, then tap to select one specific mission, then tap to select one specific astronaut. iOS takes care of animating in the new views, and provides back buttons and swipes to return to previous views.
