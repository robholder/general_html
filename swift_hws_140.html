Hacking With Swift - 140 - Taking basic order details

The first step in this project will be to create an ordering screen that takes the basic details of an order:

  • how many cupcakes they want,
  • what kind they want,
  • and whether there are any special customizations.

Before we get into the UI, we need to start by defining the data model.

Previously we’ve mixed structs and classes to get the right result, but here we’re going to take a different solution: we’re going to have a single class that stores all our data, which will be passed from screen to screen. This means all screens in our app share the same data, which will work really well as you’ll see.

Creating The Class

For now this class won’t need many properties:

  • Type:
    • The type of cakes, plus a static array of all possible options.
  • Count:
    • How many cakes the user wants to order.
  • Special Requests:
    • Whether the user wants to make special requests, which will show or hide extra options in our UI.
  • Frosting:
    • Whether the user wants extra frosting on their cakes.
  • Sprinkles:
    • Whether the user wants to add sprinkles on their cakes.

Each of those need to update the UI when changed, which means we need to make sure the class uses the @Observable macro.

So, please make a new Swift file called Order.swift, change its Foundation import to SwiftUI, and give it this code: 

@Observable
class Order {
    static let types = ["Vanilla", "Strawberry", "Chocolate", "Rainbow"]

    var type = 0
    var quantity = 3

    var specialRequestEnabled = false
    var extraFrosting = false
    var addSprinkles = false
}

We can now create a single instance of that inside ContentView by adding this property: 

@State private var order = Order()

That’s the only place the order will be created - every other screen in our app will be passed that property so they all work with the same data.

We’re going to build the UI for this screen in three sections - starting with cupcake type & quantity.

First Section

This first section will show a picker letting users choose from:

  • Vanilla,
  • Strawberry,
  • Chocolate
  • Rainbow

...cakes, then a stepper with the range 3 through 20 to choose the amount.

All that will be wrapped inside a form, which is itself inside a navigation stack so we can set a title.

There’s a small speed bump here: our cupcake topping list is an array of strings, but we’re storing the user’s selection as an integer - how can we match the two? One easy solution is to use the indices property of the array, which gives us a position of each item that we can then use with as an array index. This is a bad idea for mutable arrays because the order of your array can change at any time, but here our array order won’t ever change so it’s safe.

Put this into the body of ContentView now: 

NavigationStack {
    Form {
        Section {
            Picker("Select your cake type", selection: $order.type) {
                ForEach(Order.types.indices, id: \.self) {
                    Text(Order.types[$0])
                }
            }

            Stepper("Number of cakes: \(order.quantity)", value: $order.quantity, in: 3...20)
        }
    }
    .navigationTitle("Cupcake Corner")
}

Second Section

The second section of our form will hold three toggle switches bound to specialRequestEnabled, extraFrosting, and addSprinkles respectively. However, the second and third switches should only be visible when the first one is enabled, so we’ll wrap then in a condition.

Add this second section now: 

Section {
    Toggle("Any special requests?", isOn: $order.specialRequestEnabled)

    if order.specialRequestEnabled {
        Toggle("Add extra frosting", isOn: $order.extraFrosting)

        Toggle("Add extra sprinkles", isOn: $order.addSprinkles)
    }
}

Go ahead and run the app again, and try it out.

However, there’s a bug, and it’s one of our own making: if we enable special requests then enable one or both of “extra frosting” and “extra sprinkles”, then disable the special requests, our previous special request selection stays active. This means if we re-enable special requests, the previous special requests are still active.

This kind of problem isn’t hard to work around if every layer of your code is aware of it - if the app, your server, your database, and so on are all programmed to ignore the values of extraFrosting and addSprinkles when specialRequestEnabled is set to false. However, a better idea - a safer idea - is to make sure that both extraFrosting and addSprinkles are reset to false when specialRequestEnabled is set to false.

We can make this happen by adding a didSet property observer to specialRequestEnabled. Add this now: 

var specialRequestEnabled = false {
    didSet {
        if specialRequestEnabled == false {
            extraFrosting = false
            addSprinkles = false
        }
    }
}

Third Section

Our third section is the easiest, because it’s just going to be a NavigationLink pointing to the next screen. We don’t have a second screen, but we can add it quickly enough: create a new SwiftUI view called “AddressView”, and give it an order property like this: 

struct AddressView: View {
    var order: Order

    var body: some View {
        Text("Hello World")
    }
}

#Preview {
    AddressView(order: Order())
}

We’ll make that more useful shortly, but for now it means we can return to ContentView.swift and add the final section for our form. This will create a NavigationLink that points to an AddressView, passing in the current order object.

Please add this final section now: 

Section {
    NavigationLink("Delivery details") {
        AddressView(order: order)
    }
}

To be precise, here is what is happening in AddressView(order: order).

The order word is used twice, but they represent two different things.

  • order: (The Label): This is the parameter name defined inside AddressView. It’s like a hole in a puzzle piece that says "I need an object of type Order to fit here."
  • order (The Value): This is the actual variable existing in your current view (the one containing the NavigationLink). You are taking your local copy of the data and passing it into the next screen.

Since your Order is a class (marked with @Observable), you aren't actually sending a "copy" of the data. You are sending a pointer (a reference) to the same data.

  • If the user types their address in AddressView, they are modifying the exact same order object that exists in the parent view.
  • When they hit "Back," the parent view will already have all that new information because they were both looking at the same instance in memory.

That completes our first screen, so give it a try one last time before we move on - you should be able to select your cake type, choose a quantity, and toggle all the switches just fine.

Additional AI Insight (for #Preview)

In SwiftUI, a #Preview block is basically a small sandbox that runs your view so you can see it in the canvas. Because your AddressView has a property (var order: Order) that doesn't have a default value, the view cannot be created without being given an Order instance.

By writing AddressView(order: Order()), you are:

  • Instantiating: Creating a brand-new, temporary instance of the Order class.
  • Injecting: Passing that instance into the AddressView.

Why is this necessary?

Think of AddressView like a recipe that requires an ingredient (the order). You can't "cook" the view for the preview window unless you provide that ingredient.

How it works behind the scenes

When the Xcode Preview provider runs, it executes the code inside the #Preview braces.

  • It sees Order().
  • It allocates memory for a new Order object (using whatever default values you set up in that class, like name = "Taylor").
  • It hands that object to AddressView.

Two things to keep in mind:

1. It's a "Mock" Object

The Order() created in the preview is totally separate from the one your real app will use when it's running on a phone. This is great because you can change the preview code to show different states, like:

  • AddressView(order: Order(isPriority: true))

2. State Sharing

If Order is a class (which it usually is in these tutorials), the AddressView is receiving a reference to that object. If you had a TextField in your view that changed the order's name, the preview would stay in sync because both the view and the underlying object are looking at the same spot in memory.

Summary of the syntax

Code Purpose
AddressView The name of the structure (the blueprint).
(order: ...) The parameter name required by the blueprint.
Order() The actual "live" object being built to fill that requirement.
