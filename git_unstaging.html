Git - Unstaging

In Git, unstaging means removing files from the staging area (index) while keeping your changes intact in the working directory. This is useful when you’ve accidentally staged files with git add but don’t want them in the next commit.

Example:


# Check current status
git status

# Unstage a single file
git restore --staged example.txt

# Or using older syntax
git reset example.txt

# Unstage all staged files
git restore --staged
# or
git reset

After running these commands, the files will appear as modified but not staged when you run git status again.

Unstaging a Single File Use git restore --staged <file> (Git ≥ 2.23) to remove a file from the staging area without discarding changes: 

git restore --staged app.js

Older equivalent: 

git reset app.js

This keeps your edits in the working directory while removing them from the index.

Unstaging All Files To clear the entire staging area: 

git restore --staged

or: 

git reset

Both commands leave your working directory untouched.

Special Case – Removing from Tracking If you want Git to stop tracking a file entirely (e.g., after adding it by mistake), use: 

git rm --cached config.json

This removes it from the index but leaves it in your local filesystem. If committed and pushed, it will be deleted from others’ working copies.

Key Notes:

  • git restore --staged is the modern, more explicit command for unstaging.
  • git reset <file> is older but still widely used.
  • git rm --cached is for untracking files, not just unstaging.
  • Always verify with git status after unstaging to ensure the correct files are affected.

These commands give you precise control over what goes into your commits, helping maintain a clean and intentional commit history.
