Hacking With Swift - 96 - Deleting items using onDelete()

SwiftUI gives us the onDelete() modifier for us to use to control how objects should be deleted from a collection.

In practice, this is almost exclusively used with List and ForEach: we create a list of rows that are shown using ForEach, then attach onDelete() to that ForEach so the user can remove rows they don’t want.

This is another place where SwiftUI does a heck of a lot of work on our behalf, but it does have a few interesting quirks as you’ll see.

First, let’s construct an example we can work with: a list that shows numbers, and every time we tap the button a new number appears. Here’s the code for that: 

struct ContentView: View {
    // Create empty integer array
    @State private var numbers = [Int]()
    // Set starting number of '1'
    @State private var currentNumber = 1

    var body: some View {
        VStack {
            List {
                ForEach(numbers, id: \.self) {
                    Text("Row \($0)")
                }
            }

            Button("Add Number") {
                // Populate the array by appending the current number (1 is added on first click)
                numbers.append(currentNumber)
                // Increment current number by '1', ready for next click
                currentNumber += 1
            }
        }
    }
}

ForEach vs List only quirk

Now, you might think that the ForEach isn’t needed – the list is made up of entirely dynamic rows, so we could write this instead: 

List(numbers, id: \.self) {
    Text("Row \($0)")
}

That would also work, but here’s our first quirk: the onDelete() modifier only exists on ForEach, so if we want users to delete items from a list we must put the items inside a ForEach. This does mean a small amount of extra code for the times when we have only dynamic rows, but on the flip side it means it’s easier to create lists where only some rows can be deleted.

Lists can be made of Static and Dynamic items, and delete will only work on non-constant dynamic items. That's why we need to use ForEach to make deletion possible.

Add Deletion Facility

In order to make onDelete() work, we need to implement a method that will receive a single parameter of type IndexSet. This is a bit like a set of integers, except it’s sorted, and it’s just telling us the positions of all the items in the ForEach that should be removed.

Because our ForEach was created entirely from a single array, we can actually just pass that index set straight to our numbers array – it has a special remove(atOffsets:) method that accepts an index set.

So, add this method to ContentView now: 

func removeRows(at offsets: IndexSet) {
    numbers.remove(atOffsets: offsets)
}

When you first see this function, it looks like a tongue-twister of "offsets" and "atOffsets," but there is a very logical reason for that repetition.

Let's pull the mask off this function to see which parts are Swift's requirements and which parts you chose.

1. Keywords vs. User Terms

// 1          // 2         // 3          // 4
func removeRows(at offsets: IndexSet) {
    numbers.remove(atOffsets: offsets)
//     ^ 5                ^ 6
}
  • func: Keyword. This tells Swift you are defining a block of logic.
  • removeRows: User Term. You can name this anything! You could call it func deleteTheThing.
  • at: Keyword (Argument Label). This is a "label" that makes the function read like English when called: removeRows(at: someIndex).
  • offsets: User Term (Parameter Name). This is the "internal" name you use to refer to the data inside the function.
  • numbers: User Term. This is the name of your specific array (the @State variable you created earlier).
  • atOffsets: Keyword (SwiftUI Method). This is a built-in command specifically designed for arrays in SwiftUI to handle deletions.

2. What is an IndexSet?

Think of a standard Index (an Int) as a single finger pointing at one item in a list (e.g., "Item #3").

An IndexSet is like a hand that can point to multiple items at once.

In a list, a user might use "Edit Mode" to select three different rows and hit delete. SwiftUI doesn't want to run your function three separate times (which would be slow and might cause the list to shift around). Instead, it bundles all those locations into an IndexSet and hands it to you in one go.

3. Why the "atOffsets: offsets" part?

You might be wondering: "Why can't I just use numbers.remove(at: 1)?"

If you have a list of numbers: [10, 20, 30, 40]

And you delete the first item, the list becomes: [20, 30, 40]

Now, "Item #2" (which was 30) has moved! If you try to delete multiple items one-by-one using standard indices, the list shifts and you'll end up deleting the wrong things.

remove(atOffsets:) is a specialized tool. It looks at the IndexSet, calculates all the positions at once, and performs the deletion safely so that the remaining items don't get "misaligned" during the process.

The "Plain English" Translation

If you read your function out loud, it says:

"I'm making a function called removeRows. When it's called, it will be handed a collection of positions called offsets. I will then tell my numbers array to remove whatever is located at those offsets."

Why at offsets?

You’ll notice that when you attach this to a list, you just write: 

.onDelete(perform: removeRows)

Swift is smart enough to know that onDelete automatically provides an IndexSet. It "plugs" that IndexSet into your offsets variable for you.

Finally, we can tell SwiftUI to call that method when it wants to delete data from the ForEach, by modifying it to this: 

ForEach(numbers, id: \.self) {
    Text("Row \($0)")
}
.onDelete(perform: removeRows)

In Context

import SwiftUI

struct ContentView: View {
    @State private var numbers = [Int]()
    @State private var currentNumber = 1
    
    var body: some View {
        NavigationStack {
            VStack {
                List {
                    ForEach(numbers, id: \.self) {
                        Text("Row \($0)")
                    }
                    // Now we can swipe away (delete) rows
                    .onDelete(perform: removeRows)
                }
                
                Button("Add Number") {
                    numbers.append(currentNumber)
                    currentNumber += 1
                }
            }
        }
    }
}
    
    func removeRows(at offsets: IndexSet) {
        numbers.remove(atOffsets: offsets)
    }

}

#Preview {
    ContentView()
}

Now go ahead and run your app, then add a few numbers. When you’re ready, swipe from right to left across any of the rows in your list, and you should find a delete button appears. You can tap that, or you can also use iOS’s swipe to delete functionality by swiping further.

Edit Facility

Given how easy that was, I think the result works really well. But SwiftUI has another trick up its sleeve: we can add an Edit/Done button to the navigation bar, that lets users delete several rows more easily.

First, wrap your VStack in a NavigationStack, then add this modifier to the VStack: 

.toolbar {
    EditButton()
}

In Context

import SwiftUI

struct ContentView: View {
    @State private var numbers = [Int]()
    @State private var currentNumber = 1
    
    var body: some View {
        NavigationStack {
            VStack {
                List {
                    ForEach(numbers, id: \.self) {
                        Text("Row \($0)")
                    }
                    // We still need this with the edit menu option:
                    .onDelete(perform: removeRows)
                }
                
                Button("Add Number") {
                    numbers.append(currentNumber)
                    currentNumber += 1
                }
            }
            // Add this to the VStack within the NavigateStack to add an edit button
            .toolbar {
                EditButton()
        }
    }
}
    // We still need this with the edit menu option: 
    func removeRows(at offsets: IndexSet) {
        numbers.remove(atOffsets: offsets)
    }

}

#Preview {
    ContentView()
}

That’s literally all it takes – if you run the app you’ll see you can add some numbers, then tap Edit to start deleting those rows. When you’re ready, tap Done to exit editing mode. Not bad, given how little code it took!
