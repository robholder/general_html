Linux - No Sudo Setup Ideas

Let’s walk through how you can still build a surprisingly powerful environment entirely inside your home directory.

Below is a clear, practical roadmap — starting with zsh.

🧭 Your Constraints (and Opportunities)

In a restricted corporate Linux environment, you typically can:

  • Use your $HOME directory freely
  • Compile software from source into \(HOME/bin or \)HOME/.local/bin
  • Use static binaries when available
  • Modify your shell startup files (.bashrc, .profile, .zshrc, etc.)
  • Use SSH keys stored in $HOME/.ssh
  • Use Python, Node, or other runtimes if they’re already installed

You typically cannot:

  • Install system packages (apt install, yum install, etc.)
  • Use sudo
  • Modify /usr/bin, /usr/local/bin, /etc
  • Change your login shell via chsh

But that’s fine — you can still run your own shell without being the system’s default shell.

🥇 Step 1: Install zsh Without sudo

You have two realistic approaches:

Option A — Download a prebuilt static zsh binary (easiest)

Some people publish static builds of zsh. If you find one that matches your architecture (likely x86_64 or aarch64), you can: 

mkdir -p $HOME/bin
tar -xf zsh-static.tar.gz -C $HOME/bin

Then run it: 

$HOME/bin/zsh --version

And make it your default interactive shell by adding to .bashrc: 

exec $HOME/bin/zsh -l

This does not require chsh or sudo.

Option B — Build zsh from source in your home directory

This works surprisingly often, even on locked-down systems.

1. Download the source:

wget https://sourceforge.net/projects/zsh/files/latest/download -O zsh.tar.xz
tar -xf zsh.tar.xz
cd zsh-*/

2. Configure it to install into your home directory:

./configure --prefix=$HOME/.local
make
make install

This gives you: 

$HOME/.local/bin/zsh
$HOME/.local/share/zsh

3. Make bash auto‑launch zsh

Add to ~/.bashrc: 

if [ -x "$HOME/.local/bin/zsh" ]; then
    exec $HOME/.local/bin/zsh -l
fi

Now every time you log in, bash immediately hands control to zsh.

🧪 Step 2: Install Starship (no sudo needed)

Starship is perfect for restricted environments because it’s a single static binary. 

curl -fsSL https://starship.rs/install.sh | bash -s -- --bin-dir $HOME/bin --yes

Then in your .zshrc: 

eval "$(starship init zsh)"

🖥️ Step 3: Install WezTerm (optional inside VM)

Inside a VM, you may not need WezTerm, but if you want it:

  • WezTerm provides portable AppImage builds
  • AppImages require no installation or sudo

✔ Option 1: Manually grab the correct URL from the release page

Go to:

  • https://github.com/wez/wezterm/releases (github.com in Bing)

Find the latest release, then download the file named something like: 

WezTerm-20240203-110809-5046fc22-Ubuntu22.04.AppImage

Then: 

chmod +x WezTerm-*.AppImage

This works because the wildcard is now local, not remote.

✔ Option 2: Use a pattern that GitHub does support (API + grep)

If you want a semi‑automated approach that still fits your “hands‑on curation” philosophy, you can do: 

curl -s https://api.github.com/repos/wez/wezterm/releases/latest \
  | grep browser_download_url \
  | grep AppImage \
  | cut -d '"' -f 4 \
  | wget -i -

This:

  • fetches the latest release metadata
  • filters for the AppImage
  • extracts the URL
  • downloads it

Then: 

chmod +x WezTerm-*.AppImage

This is still portable and doesn’t assume anything about the system.

You can store it in $HOME/bin if you want.

🔠 Step 4: Nerd Fonts (user‑local install)

You can install fonts locally without sudo. 

mkdir -p ~/.local/share/fonts
cd ~/.local/share/fonts
wget https://github.com/ryanoasis/nerd-fonts/releases/download/v3.2.1/FiraCode.zip
unzip FiraCode.zip
fc-cache -fv

WezTerm or your terminal emulator will now see them.

🔧 Step 5: Tmux (build from source)

Tmux also builds cleanly in $HOME.

Dependencies: libevent and ncurses — but you can build those locally too.

Build libevent:

wget https://github.com/libevent/libevent/releases/download/release-2.1.12-stable/libevent-2.1.12-stable.tar.gz

tar -xf libevent*.tar.gz
cd libevent*
./configure --prefix=$HOME/.local
make
make install

Build tmux:

wget https://github.com/tmux/tmux/releases/download/3.4/tmux-3.4.tar.gz
tar -xf tmux*.tar.gz
cd tmux*
./configure --prefix=$HOME/.local CFLAGS="-I$HOME/.local/include" LDFLAGS="-L$HOME/.local/lib"
make
make install

🔐 Step 6: SSH access to GitHub (fully allowed as a normal user)

SSH keys live in $HOME/.ssh, so no privileges needed. 

ssh-keygen -t ed25519 -C "your_email@example.com"

Then: 

cat ~/.ssh/id_ed25519.pub

Copy to GitHub → Settings → SSH Keys.

Test:

ssh -T git@github.com

🧼 What I mean by a “clean .zshrc”

A clean .zshrc is one that:

  • ✔ avoids assumptions about system-wide paths
    • You can’t rely on /usr/share/zsh, /etc/zshenv, /usr/local/bin, or system-installed plugins.
  • ✔ loads only what you installed in $HOME
    • Everything must be self-contained.
  • ✔ avoids slowdowns
    • Many people accumulate years of plugins, oh-my-zsh layers, and PATH mutations. In a restricted environment, you want something lean and predictable.
  • ✔ is portable
    • It should work on:
      • your UTM VM
      • a corporate RHEL box
      • a Debian server
      • a random Ubuntu machine
  • ✔ doesn’t break if something is missing
    • A clean .zshrc checks for tools before enabling them.

🧽 Example of a “clean” portable .zshrc

# ----- PATH -----
export PATH="$HOME/bin:$HOME/.local/bin:$PATH"

# ----- Starship -----
if command -v starship >/dev/null 2>&1; then
    eval "$(starship init zsh)"
fi

# ----- Aliases -----
alias ll='ls -alF'
alias la='ls -A'
alias l='ls -CF'

# ----- Completion -----
if [ -d "$HOME/.local/share/zsh" ]; then
    fpath=($HOME/.local/share/zsh/site-functions $fpath)
    autoload -Uz compinit
    compinit -u
fi

# ----- Tmux auto-start -----
if command -v tmux >/dev/null 2>&1 && [ -z "$TMUX" ]; then
    tmux attach || tmux new
fi

This file:

  • never assumes system paths
  • gracefully enables features only if available
  • works on any Linux distro
  • is fast and predictable

Then unpack it on a new machine and it just works.

  • ✔ You can bootstrap a new machine with a single script
    • Even without sudo.

📁 Why zsh and tmux go in \(HOME/.local/bin, but starship goes in \)HOME/bin

This is a subtle but important point.

🧩 $HOME/.local/bin

This is the XDG standard location for user-installed executables.

It’s where software built from source normally installs when you use: 

./configure --prefix=$HOME/.local

So tools like:

  • zsh (compiled)
  • tmux (compiled)
  • libevent/ncurses (support libs)
  • fzf (if built)
  • neovim (if built)

belong here.

🧩 $HOME/bin

This is a traditional user bin directory, often used for:

  • manually downloaded binaries
  • scripts
  • AppImages
  • standalone tools

Starship’s installer defaults to \(HOME/.cargo/bin, but when you override it, \)HOME/bin is a natural place.

Why not put everything in one place?

You can, but separating them gives you:

  • clarity (compiled vs. downloaded)
  • easier maintenance
  • easier backups
  • easier PATH management

If you prefer a single directory, that’s fine — just be consistent.

ANSWER

Planning for portability is one of the smartest things you can do — and it’s absolutely possible to build a setup that works beautifully on your UTM VM and on a locked‑down corporate Linux box. But it’s also worth being honest about the trade‑offs so you can design something that’s powerful without becoming fragile.

Let’s zoom out and look at the big picture.

🌍 The Big Picture: A Portable, Self‑Contained Linux Environment

A portable environment means:

  • Your shell (zsh)
  • Your terminal tools (tmux, fzf, ripgrep, etc.)
  • Your fonts
  • Your configs (zshrc, tmux.conf, wezterm.lua)
  • Your plugins
  • Your dotfiles (managed by Stow or similar)

…all live inside your $HOME directory and do not depend on system packages or sudo.

This gives you a “personal Linux layer” that sits on top of whatever corporate Linux you’re given.

🧠 Are there downsides to planning for portability?

There are a few — none fatal, but worth knowing so you can design around them.

⚠️ 1. You become responsible for updates

System packages normally get security updates automatically. Your portable environment does not.

You’ll need to:

  • rebuild tmux occasionally
  • update zsh manually
  • refresh fonts
  • update plugins

This is manageable, but it’s a shift in responsibility.

⚠️ 2. You may need to compile things more often

Corporate systems often lack:

  • development headers
  • compilers
  • build tools

If you can’t compile on the target machine, you may need to:

  • compile on a similar machine
  • or use static binaries
  • or use prebuilt releases

This is solvable, but it’s something to plan for.

⚠️ 3. Some tools don’t behave perfectly when installed non‑system‑wide

Examples:

  • zsh completion paths may need manual configuration
  • tmux may need custom library paths
  • fzf may need manual installation of keybindings
  • wezterm may not integrate with system menus

Again, all solvable — but not “plug and play”.

⚠️ 4. PATH and environment variables become more important

You’ll need to ensure:

  • \(HOME/bin and \)HOME/.local/bin come first
  • \(LD_LIBRARY_PATH includes \)HOME/.local/lib
  • $XDG_* variables point to your configs

A portable environment is more sensitive to ordering.

⚠️ 5. You must avoid assumptions about the host system

For example:

  • don’t assume /usr/bin/zsh exists
  • don’t assume /etc/zshenv is readable
  • don’t assume /usr/share/fonts is writable
  • don’t assume system Python or Node versions

This forces you to write cleaner, more defensive config files — which is actually a good thing.

🌱 But the upsides are huge

A portable environment gives you:

✔ A consistent shell everywhere

  • Corporate RHEL?
  • Ubuntu server?
  • Your UTM VM?
  • A random cloud VM?

Your environment behaves the same.

✔ Zero reliance on system admins

✔ Easy migration

✔ Full control over your tools

You choose versions, plugins, themes, fonts — not the system.

✔ A deeper understanding of Linux

You learn how shells, paths, libraries, and configs really work.

🧭 The Philosophy: “Portable, Defensive, and Hands‑On”

A robust portable environment should:

  • ✔ Assume nothing about the host system
    • No assumptions about compilers, package managers, shells, fonts, or permissions.
  • ✔ Prefer prebuilt binaries when possible
    • Compiling is a luxury, not a guarantee.
  • ✔ Keep each component optional
    • If zsh fails, bash should still work.
    • If tmux fails, the shell should still work.
    • If starship fails, your prompt should still work.
  • ✔ Use conditional loading everywhere
    • Your config should never error out if something is missing.
  • ✔ Keep the directory structure simple and predictable
    • You should be able to debug it in minutes.
  • ✔ Allow manual intervention
    • You want to be able to fix or replace one component without touching the rest.
    • This is the opposite of a fragile bootstrap script that assumes a perfect world.

🧱 The Architecture: A Portable, Defensive Home Layout

Here’s a structure that supports your goals: 

$HOME/

  .local/

A portable .zshrc should:

  • never assume anything exists
  • never error out
  • load features only if available
  • keep PATH sane
  • avoid slowdowns
  • avoid system-specific paths

Here’s the skeleton: 

# ----- PATH -----
export PATH="$HOME/bin:$HOME/.local/bin:$PATH"

# ----- Completion -----
if [ -d "$HOME/.local/share/zsh" ]; then

alias la='ls -A'
alias l='ls -CF'

This file cannot break.

It only enhances.

🧠 Why this approach works

Because it’s built on defensive layering:

  • If zsh fails → bash still works
  • If starship fails → prompt still works
  • If tmux fails → shell still works
  • If wezterm fails → system terminal still works
  • If neovim fails → vim still works

Nothing is allowed to be a single point of failure.

This is exactly what you want in unpredictable environments.

eza fits beautifully into this “portable, defensive, zero‑assumptions” philosophy, and it’s actually one of the easiest tools to integrate because it’s distributed as a single static binary for Linux.

Let’s fold it into your overall plan in a way that keeps everything resilient and portable.

🧩 Adding eza to Your Portable Environment

There are three realistic installation paths for eza in a restricted, no‑sudo environment. Each one aligns with your “hands‑on curation, defensive, portable” strategy.

1️⃣ Preferred: Use the official static binary

eza publishes musl‑linked static binaries, which run on almost any Linux system without dependencies.

You simply:

  • download the binary
  • put it in $HOME/bin
  • make it executable

This is the most portable and least fragile option.

Why $HOME/bin?

Because this is where you keep manually downloaded standalone binaries (WezTerm, starship, scripts).

It keeps them separate from compiled tools in $HOME/.local/bin.

Defensive install steps:

mkdir -p $HOME/bin
wget https://github.com/eza-community/eza/releases/download/.../eza_x86_64-unknown-linux-musl.tar.gz
tar -xf eza_*.tar.gz -C $HOME/bin
chmod +x $HOME/bin/eza

If the binary doesn’t run on a particular system → no harm done, your shell still works.

2️⃣ Fallback: Build eza from source (if the system allows it)

This is your “Plan B” when:

  • the static binary doesn’t match the CPU architecture
  • the system has a compiler
  • or you want to optimize it yourself

eza is written in Rust, so you need cargo.

If the system doesn’t have Rust installed, you can install Rust locally (no sudo) using rustup.

But this is optional — and in a locked‑down environment, often impossible.

So this is a nice‑to‑have, not a core part of your portable strategy.

3️⃣ Last resort: Use the system’s ls

Your defensive configuration should never assume eza exists.

If it’s missing, you fall back to ls automatically.

🧼 Defensive Integration Into Your .zshrc

Here’s how you add eza without risking breakage: 

# ----- eza -----
if command -v eza >/dev/null 2>&1; then
    alias ls="eza"
    alias ll="eza -l"
    alias la="eza -la"
else
    alias ll="ls -alF"
    alias la="ls -A"
fi

This gives you:

  • eza when available
  • ls when not
  • no errors
  • no broken aliases

This is exactly the kind of defensive layering you want.
