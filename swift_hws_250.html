Hacking With Swift - 250 - Supporting specific accessibility needs with SwiftUI

SwiftUI gives us a number of environment properties that describe the user’s custom accessibility settings, and it’s worth taking the time to read and respect those settings.

Back in project 15 we looked at accessibility labels and hints, traits, groups, and more, but these settings are different because they are provided through the environment. This means SwiftUI automatically monitors them for changes and will reinvoke our body property whenever one of them changes.

Differentiate Without Color

For example, one of the accessibility options is "Differentiate without color", which is helpful for the 1 in 12 men who have color blindness. When this setting is enabled, apps should try to make their UI clearer using shapes, icons, and textures rather than colors.

To use this, just add an environment property like this one: 

@Environment(\.accessibilityDifferentiateWithoutColor) var differentiateWithoutColor

That will be either true or false, and you can adapt your UI accordingly. For example, in the code below we use a simple green background for the regular layout, but when Differentiate Without Color is enabled we use a black background and add a checkmark instead: 

struct ContentView: View {
    @Environment(\.accessibilityDifferentiateWithoutColor) var differentiateWithoutColor

    var body: some View {
        HStack {
            if differentiateWithoutColor {
                Image(systemName: "checkmark.circle")
            }

            Text("Success")
        }
        .padding()
        .background(differentiateWithoutColor ? .black : .green)
        .foregroundStyle(.white)
        .clipShape(.capsule)
    }
}

You can test that in the simulator by going to the Settings app (the Settings app on the virtual phone, not the simulator's macOS menus!) and choosing Accessibility > Display & Text Size > Differentiate Without Color.

Reduce Motion

Another common option is Reduce Motion, which again is available in the simulator (on board Systems App) under Accessibility > Motion > Reduce Motion. When this is enabled, apps should limit the amount of animation that causes movement on screen. For example, the iOS app switcher makes views fade in and out rather than scale up and down.

With SwiftUI, this means we should restrict the use of withAnimation() when it involves movement, like this: 

struct ContentView: View {
    @Environment(\.accessibilityReduceMotion) var reduceMotion
    @State private var scale = 1.0

    var body: some View {
        Button("Hello, World!") {
            if reduceMotion {
                scale *= 1.5
            } else {
                withAnimation {
                    scale *= 1.5
                }
            }

        }
        .scaleEffect(scale)
    }
}

withAnimation

I don’t know about you, but I find that rather annoying to use. Fortunately we can add a little wrapper function around withAnimation() that uses UIKit’s UIAccessibility data directly, allowing us to bypass animation automatically:

Notice the 'generic parameter'. 

func withOptionalAnimation<Result>(_ animation: Animation? = .default, _ body: () throws -> Result) rethrows -> Result {
    if UIAccessibility.isReduceMotionEnabled {
        return try body()
    } else {
        return try withAnimation(animation, body)
    }
}

So, when Reduce Motion Enabled is true the closure code that’s passed in is executed immediately, otherwise it’s passed along using withAnimation(). The whole throws/rethrows thing is more advanced Swift, but it’s a direct copy of the function signature for withAnimation() so that the two can be used interchangeably.

Use it like this: 

struct ContentView: View {
    @State private var scale = 1.0

    var body: some View {
        Button("Hello, World!") {
            withOptionalAnimation {
                scale *= 1.5
            }

        }
        .scaleEffect(scale)
    }
}

Using this approach you don’t need to repeat your animation code every time.

Reduce Transparency

One last option you should consider supporting is Reduce Transparency, and when that’s enabled apps should reduce the amount of blur and translucency used in their designs to make doubly sure everything is clear.

Reduce Transparency is here:

Accessibility > Display & Text Size > Reduce Transparency

For example, this code uses a solid black background when Reduce Transparency is enabled, otherwise using 50% transparency: 

struct ContentView: View {
    @Environment(\.accessibilityReduceTransparency) var reduceTransparency

    var body: some View {
        Text("Hello, World!")
            .padding()
            .background(reduceTransparency ? .black : .black.opacity(0.5))
            .foregroundStyle(.white)
            .clipShape(.capsule)
    }
}

That’s the final technique I want you to learn ahead of building the real project, so please reset your project back to its original state so we have a clean slate to start on.

Ai Insight

QUESTION

I can't recall what the angle brackets represent in the first line of the function here. Can you help me understand what's going on?

ANSWER

Those angle brackets <Result> represent a Generic Parameter.

In Swift, generics allow you to write a single, flexible function that can work with any data type (like a String, an Int, a custom struct, or even Void if it returns nothing), without having to rewrite the function over and over again for each type.

Here is a breakdown of exactly what is happening in that function signature.

The Anatomy of

Think of <Result> as a placeholder label that you define before the function's arguments. You are essentially telling the Swift compiler:

"Hey, this function is going to deal with a data type. I don't know what type it is yet, but for the rest of this signature, I'm going to refer to that type as Result."

If we track how Result is used across that single line of code, its purpose becomes clear: 

func withOptionalAnimation<Result>(
    _ animation: Animation? = .default, 
    _ body: () throws -> Result          // 1. The closure must return this placeholder type
) rethrows -> Result {                  // 2. The entire function will output that same type
  • _ body: () throws -> Result: This tells Swift that whatever code block (closure) the user passes into this function, that block must execute and return something (the Result type).
  • -> Result: This guarantees that the overall withOptionalAnimation function will pass that exact same data type right back to you when it finishes.

Why is this useful? (A Concrete Example)

Imagine you are fetching data inside an animation. Sometimes you are fetching a user's String profile name, and other times you are calculating an Int score.

Because of the generic <Result> placeholder, you can use this exact same function for both scenarios:

Scenario A: The closure returns a String

Swift looks at your closure, sees it returns text, and silently replaces every instance of <Result> with <String>. 

let profileName = withOptionalAnimation {
    return "Alex" // Swift knows 'Result' is a String here
}

Scenario B: The closure returns an Int

Swift looks at your closure, sees it returns a number, and silently replaces every instance of <Result> with <Int>. 

let finalScore = withOptionalAnimation {
    return 100 // Swift knows 'Result' is an Int here
}

Why not just use Any?

You might wonder why we don't just use Apple's catch-all Any type instead of generics.

If the function returned Any, you would lose all type safety. Every time you used withOptionalAnimation, you would have to manually type-cast the output (e.g., as! String), which is dangerous and can cause your app to crash if you guess wrong. Generics provide absolute flexibility while keeping strict, compile-time type safety.
