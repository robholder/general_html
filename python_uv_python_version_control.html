UV - Python Version Control

UNTESTED - COPILOT

UV can replace pyenv, and it does so in a way that’s faster, simpler, and more integrated with the rest of your Python workflow. But the way uv handles Python versions is different enough that it helps to see the mental model clearly.

Below is a complete, practical guide to how uv replaces pyenv, how to use it to install and switch Python versions, and how it fits into the broader Python‑tooling ecosystem.

🎯 When uv Is a Better Choice Than pyenv

  • You want speed (uv is extremely fast)
  • You want one tool instead of many
  • You want cleaner, explicit version control
  • You want cross‑platform consistency
  • You want integrated package + environment + interpreter management

🚀 What uv Is (and Why It Replaces So Many Tools)

According to Astral, uv is designed as “a single tool to replace pip, pip-tools, pipx, poetry, pyenv, twine, virtualenv, and more” . That includes Python version management, which is the part pyenv traditionally handled.

So uv isn’t just a package manager — it’s a full Python toolchain manager.

🧠 How uv Replaces pyenv Conceptually

Here’s the key difference:

Tool How it manages Python versions
pyenv Installs Python builds into ~/.pyenv/versions and shims your PATH to select versions
uv Installs Python builds into its own global cache and uses explicit commands to run or create environments with a specific version

uv does not use shims.

Instead, uv gives you direct commands to install, run, or select Python versions.

This makes it simpler and faster, and avoids the PATH‑hacking pyenv relies on.

🛠️ Installing Python Versions with uv

uv has a built‑in Python installer: 

uv python install 3.12
uv python install 3.11
uv python install 3.10

You can also install patch versions explicitly: 

uv python install 3.12.2

This is the uv equivalent of: 

pyenv install 3.12.2

🧪 Checking Installed Versions

uv python list

This shows all Python interpreters uv manages.

▶️ Running a Specific Python Version (pyenv local/global equivalent)

Instead of shims, uv lets you run a specific version directly: 

uv python run 3.12 --version

Or run a script with a specific interpreter: 

uv python run 3.11 script.py

This is the uv equivalent of: 

pyenv local 3.11
python script.py

📦 Creating Virtual Environments with a Specific Python Version

This is where uv really shines. 

uv venv --python 3.12

This creates a venv using uv’s managed Python 3.12.

You can also specify a path: 

uv venv .venv --python 3.11

This replaces: 

pyenv local 3.11
python -m venv .venv

🧰 Using uv to Run Tools with a Specific Python Version

uv can run Python‑based tools without installing them globally: 

uv tool run black --version

Or with a specific Python version: 

uv tool run --python 3.12 ruff check .

This replaces pyenv + pipx.

🔄 Switching From pyenv to uv

Several guides describe the migration process, including removing pyenv shims and switching to uv’s Python manager. They emphasize that uv provides faster performance and more integrated workflows than pyenv .

Another migration guide highlights that uv’s uv python command fully manages interpreters and that users can replace pyenv workflows with uv equivalents for installing interpreters, creating venvs, and running tools .

🧩 How uv Chooses Which Python Version to Use

uv follows this priority order:

  • Explicit version flags
    • e.g., uv python run 3.12 script.py
  • Virtual environment Python
    • If you’re inside a venv, uv uses that interpreter.
  • Project configuration
    • uv supports pyproject.toml metadata that can specify Python requirements.
  • System Python
    • If nothing else is specified.

This is simpler and more predictable than pyenv’s shim‑based PATH manipulation.

🧨 What uv Does Not Do (Compared to pyenv)

uv does not:

  • Override your system python command
  • Insert shims into your PATH
  • Automatically switch Python versions based on directory

Instead, uv encourages explicit version usage, which avoids many of pyenv’s common pitfalls.

If you want automatic version switching, you can simulate it with shell aliases or direnv, but uv intentionally avoids this complexity.

🏁 Practical “pyenv → uv” Cheat Sheet

pyenv command uv equivalent
pyenv install 3.12.2 uv python install 3.12.2
pyenv versions uv python list
pyenv local 3.11 Use uv venv --python 3.11 or uv python run 3.11
pyenv global 3.12 Use explicit version flags or venvs
pyenv which python uv python find (if needed)
pyenv uninstall 3.10.9 uv python uninstall 3.10.9

Project: End to End

Here’s a clean, modern end‑to‑end Python project workflow using uv — the way it’s meant to be used. This will show you how uv replaces pyenv, venv, pip, pip-tools, pipx, and even parts of Poetry, all while staying fast and predictable.

I’ll walk you through:

  • Creating a new project
  • Managing Python versions
  • Creating a virtual environment
  • Adding dependencies
  • Locking dependencies
  • Running scripts and tools
  • Packaging and publishing

By the end, you’ll have a full mental model of how uv wants you to work.

🧱 1. Create a New Project

You can start from scratch: 

mkdir myproject
cd myproject

Initialize a pyproject.toml: 

uv init

This creates: 

myproject/
  pyproject.toml
  README.md
  src/myproject/__init__.py

uv uses the src layout by default — a good practice.

🐍 2. Choose Your Python Version (pyenv replacement)

Install the Python version you want: 

uv python install 3.12

Check installed versions: 

uv python list

🧪 3. Create a Virtual Environment

Create a venv using that Python version: 

uv venv --python 3.12

Activate it: 

source .venv/bin/activate

(Windows: .\.venv\Scripts\activate)

This replaces:

  • pyenv local
  • python -m venv .venv

📦 4. Add Dependencies

Let’s say you want requests and rich: 

uv add requests rich

uv will:

  • Install them
  • Update pyproject.toml
  • Generate a lockfile (uv.lock)

This replaces:

  • pip install
  • pip-tools
  • poetry add

🔒 5. Lock Dependencies (automatically handled)

uv automatically maintains a lockfile: 

uv.lock

If you ever want to regenerate it: 

uv lock

▶️ 6. Run Your Code

Create a script: 

src/myproject/main.py

Run it: 

uv run src/myproject/main.py

Or run a module: 

uv run myproject

This replaces:

  • python script.py
  • poetry run

🧰 7. Run Tools Without Installing Them Globally (pipx replacement)

Example: run Ruff without installing it: 

uv tool run ruff check .

Or specify a Python version: 

uv tool run --python 3.12 black .

This replaces:

  • pipx run
  • pipx install

🧪 8. Add Dev Dependencies

uv supports dev groups: 

uv add --dev pytest ruff black

Run tests: 

uv run pytest

📦 9. Build Your Package

uv can build wheels and sdists: 

uv build

Outputs go into: 

dist/

This replaces:

  • python -m build
  • poetry build

🚀 10. Publish to PyPI

uv can publish too: 

uv publish

This replaces:

  • twine upload

🧭 Full Workflow Summary

Here’s the entire flow in one place: 

# 1. Create project
mkdir myproject
cd myproject
uv init

# 2. Install Python version
uv python install 3.12

# 3. Create venv
uv venv --python 3.12
source .venv/bin/activate

# 4. Add dependencies
uv add requests rich

# 5. Add dev dependencies
uv add --dev pytest ruff black

# 6. Run code
uv run src/myproject/main.py

# 7. Run tools
uv tool run ruff check .

# 8. Build package
uv build

# 9. Publish
uv publish

🎯 Why This Workflow Is So Good

  • No pyenv — uv manages Python versions directly
  • No pip — uv installs packages faster
  • No venv — uv creates environments instantly
  • No pip-tools — uv handles locking
  • No pipx — uv runs tools in isolated environments
  • No Poetry — uv handles project metadata, dependencies, and builds
  • No Twine — uv publishes packages

It’s the first tool that truly unifies the Python ecosystem.
