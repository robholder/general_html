UV - Python Version Control

UNTESTED - COPILOT

UV can replace pyenv, and it does so in a way that’s faster, simpler, and more integrated with the rest of your Python workflow. But the way uv handles Python versions is different enough that it helps to see the mental model clearly.

Below is a complete, practical guide to how uv replaces pyenv, how to use it to install and switch Python versions, and how it fits into the broader Python‑tooling ecosystem.

🎯 When uv Is a Better Choice Than pyenv

  • You want speed (uv is extremely fast)
  • You want one tool instead of many
  • You want cleaner, explicit version control
  • You want cross‑platform consistency
  • You want integrated package + environment + interpreter management

🚀 What uv Is (and Why It Replaces So Many Tools)

According to Astral, uv is designed as “a single tool to replace pip, pip-tools, pipx, poetry, pyenv, twine, virtualenv, and more” . That includes Python version management, which is the part pyenv traditionally handled.

So uv isn’t just a package manager — it’s a full Python toolchain manager.

🧠 How uv Replaces pyenv Conceptually

Here’s the key difference:

Tool How it manages Python versions
pyenv Installs Python builds into ~/.pyenv/versions and shims your PATH to select versions
uv Installs Python builds into its own global cache and uses explicit commands to run or create environments with a specific version

uv does not use shims.

Instead, uv gives you direct commands to install, run, or select Python versions.

This makes it simpler and faster, and avoids the PATH‑hacking pyenv relies on.

🛠️ Installing Python Versions with uv

uv has a built‑in Python installer: 

uv python install 3.12
uv python install 3.11
uv python install 3.10

You can also install patch versions explicitly: 

uv python install 3.12.2

This is the uv equivalent of: 

pyenv install 3.12.2

🧪 Checking Installed Versions

uv python list

This shows all Python interpreters uv manages.

▶️ Running a Specific Python Version (pyenv local/global equivalent)

Instead of shims, uv lets you run a specific version directly: 

uv python run 3.12 --version

Or run a script with a specific interpreter: 

uv python run 3.11 script.py

This is the uv equivalent of: 

pyenv local 3.11
python script.py

📦 Creating Virtual Environments with a Specific Python Version

This is where uv really shines. 

uv venv --python 3.12

This creates a venv using uv’s managed Python 3.12.

You can also specify a path: 

uv venv .venv --python 3.11

This replaces: 

pyenv local 3.11
python -m venv .venv

🧰 Using uv to Run Tools with a Specific Python Version

uv can run Python‑based tools without installing them globally: 

uv tool run black --version

Or with a specific Python version: 

uv tool run --python 3.12 ruff check .

This replaces pyenv + pipx.

🔄 Switching From pyenv to uv

Several guides describe the migration process, including removing pyenv shims and switching to uv’s Python manager. They emphasize that uv provides faster performance and more integrated workflows than pyenv .

Another migration guide highlights that uv’s uv python command fully manages interpreters and that users can replace pyenv workflows with uv equivalents for installing interpreters, creating venvs, and running tools .

🧩 How uv Chooses Which Python Version to Use

uv follows this priority order:

  • Explicit version flags
    • e.g., uv python run 3.12 script.py
  • Virtual environment Python
    • If you’re inside a venv, uv uses that interpreter.
  • Project configuration
    • uv supports pyproject.toml metadata that can specify Python requirements.
  • System Python
    • If nothing else is specified.

This is simpler and more predictable than pyenv’s shim‑based PATH manipulation.

🧨 What uv Does Not Do (Compared to pyenv)

uv does not:

  • Override your system python command
  • Insert shims into your PATH
  • Automatically switch Python versions based on directory

Instead, uv encourages explicit version usage, which avoids many of pyenv’s common pitfalls.

If you want automatic version switching, you can simulate it with shell aliases or direnv, but uv intentionally avoids this complexity.

🏁 Practical “pyenv → uv” Cheat Sheet

pyenv command uv equivalent
pyenv install 3.12.2 uv python install 3.12.2
pyenv versions uv python list
pyenv local 3.11 Use uv venv --python 3.11 or uv python run 3.11
pyenv global 3.12 Use explicit version flags or venvs
pyenv which python uv python find (if needed)
pyenv uninstall 3.10.9 uv python uninstall 3.10.9

Project: End to End

Here’s a clean, modern end‑to‑end Python project workflow using uv — the way it’s meant to be used. This will show you how uv replaces pyenv, venv, pip, pip-tools, pipx, and even parts of Poetry, all while staying fast and predictable.

I’ll walk you through:

  • Creating a new project
  • Managing Python versions
  • Creating a virtual environment
  • Adding dependencies
  • Locking dependencies
  • Running scripts and tools
  • Packaging and publishing

By the end, you’ll have a full mental model of how uv wants you to work.

🧱 1. Create a New Project

You can start from scratch: 

mkdir myproject
cd myproject

Initialize a pyproject.toml: 

uv init

This creates: 

myproject/
  pyproject.toml
  README.md
  src/myproject/__init__.py

uv uses the src layout by default — a good practice.

🐍 2. Choose Your Python Version (pyenv replacement)

Install the Python version you want: 

uv python install 3.12

Check installed versions: 

uv python list

🧪 3. Create a Virtual Environment

Create a venv using that Python version: 

uv venv --python 3.12

Activate it: 

source .venv/bin/activate

(Windows: .\.venv\Scripts\activate)

This replaces:

  • pyenv local
  • python -m venv .venv

📦 4. Add Dependencies

Let’s say you want requests and rich: 

uv add requests rich

uv will:

  • Install them
  • Update pyproject.toml
  • Generate a lockfile (uv.lock)

This replaces:

  • pip install
  • pip-tools
  • poetry add

🔒 5. Lock Dependencies (automatically handled)

uv automatically maintains a lockfile: 

uv.lock

If you ever want to regenerate it: 

uv lock

▶️ 6. Run Your Code

Create a script: 

src/myproject/main.py

Run it: 

uv run src/myproject/main.py

Or run a module: 

uv run myproject

This replaces:

  • python script.py
  • poetry run

🧰 7. Run Tools Without Installing Them Globally (pipx replacement)

Example: run Ruff without installing it: 

uv tool run ruff check .

Or specify a Python version: 

uv tool run --python 3.12 black .

This replaces:

  • pipx run
  • pipx install

🧪 8. Add Dev Dependencies

uv supports dev groups: 

uv add --dev pytest ruff black

Run tests: 

uv run pytest

📦 9. Build Your Package

uv can build wheels and sdists: 

uv build

Outputs go into: 

dist/

This replaces:

  • python -m build
  • poetry build

🚀 10. Publish to PyPI

uv can publish too: 

uv publish

This replaces:

  • twine upload

🧭 Full Workflow Summary

Here’s the entire flow in one place: 

# 1. Create project
mkdir myproject
cd myproject
uv init

# 2. Install Python version
uv python install 3.12

# 3. Create venv
uv venv --python 3.12
source .venv/bin/activate

# 4. Add dependencies
uv add requests rich

# 5. Add dev dependencies
uv add --dev pytest ruff black

# 6. Run code
uv run src/myproject/main.py

# 7. Run tools
uv tool run ruff check .

# 8. Build package
uv build

# 9. Publish
uv publish

🎯 Why This Workflow Is So Good

  • No pyenv — uv manages Python versions directly
  • No pip — uv installs packages faster
  • No venv — uv creates environments instantly
  • No pip-tools — uv handles locking
  • No pipx — uv runs tools in isolated environments
  • No Poetry — uv handles project metadata, dependencies, and builds
  • No Twine — uv publishes packages

It’s the first tool that truly unifies the Python ecosystem.

Migrating Projects

Migrating an existing Python project to uv is surprisingly smooth once you understand the moving parts. The goal is to replace your old stack (pyenv, venv, pip, pip-tools, pipx, Poetry, etc.) with uv’s unified workflow — without breaking your project or surprising your teammates.

Below is a clear, practical, end‑to‑end migration guide that works whether your project uses:

  • requirements.txt
  • requirements.in + pip-tools
  • pyproject.toml (Poetry, PDM, Hatch, etc.)
  • venv or virtualenv
  • pyenv‑managed Python versions

Let’s walk through it in a way that feels natural and keeps your project stable.

🧭 Step 1 — Install uv’s Python Version (pyenv replacement)

If your project expects Python 3.11, for example: 

uv python install 3.11

Check what’s installed: 

uv python list

You don’t need pyenv shims anymore — uv keeps interpreters isolated and fast.

🧱 Step 2 — Create a New uv Virtual Environment

Inside your project folder: 

uv venv --python 3.11

Activate it: 

source .venv/bin/activate

This replaces:

  • pyenv local
  • python -m venv .venv
  • poetry env use

📦 Step 3 — Convert Your Dependencies

This is the part people worry about, but uv makes it painless.

If you have requirements.txt: 

uv add --from-requirements requirements.txt

uv will:

  • Install everything
  • Create a pyproject.toml
  • Generate a uv.lock

If you have requirements.in (pip-tools):

uv add --from-requirements requirements.in

uv handles dependency resolution itself — no more pip-compile.

If you already have a pyproject.toml (Poetry/PDM/Hatch):

Just run: 

uv sync

uv will:

  • Read your existing pyproject.toml
  • Resolve dependencies
  • Create a uv.lock
  • Install everything into your venv

You can delete old lockfiles like:

  • poetry.lock
  • pdm.lock
  • hatch.lock
  • requirements.txt (optional)

🧪 Step 4 — Add Dev Dependencies (if needed)

If your project has dev tools like pytest, ruff, black: 

uv add --dev pytest ruff black

This replaces:

  • pip install -r dev-requirements.txt
  • poetry add --group dev

▶️ Step 5 — Run Your Project with uv

Instead of:

  • python script.py
  • poetry run python script.py

Use: 

uv run src/myproject/main.py

Or run a module: 

uv run myproject

uv automatically uses your venv’s interpreter.

🧰 Step 6 — Run Tools Without Installing Them Globally

This is where uv feels magical.

Instead of:

  • pipx run black
  • poetry run ruff

You can do: 

uv tool run ruff check .
uv tool run black .

Tools run in isolated environments — no global installs.

🔒 Step 7 — Lock Dependencies

uv maintains a lockfile automatically, but you can regenerate it: 

uv lock

This replaces:

  • pip-compile
  • poetry lock

📦 Step 8 — Build Your Package (if applicable)

If your project is a library: 

uv build

Outputs go to dist/.

This replaces:

  • python -m build
  • poetry build

🚀 Step 9 — Publish to PyPI (optional)

uv publish

This replaces:

  • twine upload

🧹 Step 10 — Remove Old Tooling (optional but satisfying)

Depending on your project, you can now delete:

  • .python-version (pyenv)
  • requirements.txt (if you want)
  • requirements.in
  • poetry.lock
  • Pipfile / Pipfile.lock
  • .venv/ (if you want uv to recreate it)
  • Any pip-tools scripts

Your project now runs on:

  • pyproject.toml
  • uv.lock
  • .venv/

Clean, modern, and fast.

🎁 Bonus: A Minimal pyproject.toml After Migration

uv keeps things simple: 

[project]
name = "myproject"
version = "0.1.0"
requires-python = ">=3.11"
dependencies = [
    "requests",
    "rich",
]

[tool.uv]
dev-dependencies = [
    "pytest",
    "ruff",
    "black",
]

uv doesn’t force a complex structure — it stays out of your way.

🏁 Final Summary: Migration in 10 Seconds

uv python install 3.11
uv venv --python 3.11
source .venv/bin/activate
uv add --from-requirements requirements.txt   # or uv sync for pyproject
uv add --dev pytest ruff black
uv run src/myproject/main.py
uv lock
uv build
uv publish

Your project is now fully uv‑powered.

What is the Lock?

Understanding what uv.lock actually does is one of the keys to appreciating why uv is such a big deal. Let’s break it down in a way that’s clear, practical, and grounded in how modern Python tooling works.

🔒 What is uv.lock?

uv.lock is uv’s lockfile, similar in spirit to:

  • poetry.lock
  • pdm.lock
  • Pipfile.lock
  • requirements.txt (sort of)

Its job is simple but crucial:

It records the exact, fully‑resolved versions of every dependency (and sub‑dependency) in your project so that future installs are 100% reproducible.

That means:

  • Same versions on every machine
  • Same versions in CI
  • Same versions for teammates
  • Same versions months or years later

No surprises, no “works on my machine”, no dependency drift.

🧠 What exactly is being “locked”?

When uv resolves your dependencies, it determines:

  • The exact version of each package
  • The exact version of each transitive dependency
  • The source of each package (PyPI, URL, Git repo, local path)
  • The hashes of each downloaded artifact
  • The Python version compatibility
  • Platform‑specific wheels (macOS, Linux, Windows)

All of that gets written into uv.lock.

So what’s being locked?

✔️ Direct dependencies

(e.g., requests, numpy)

✔️ Transitive dependencies

(e.g., urllib3, certifi, charset-normalizer)

✔️ Exact versions

(e.g., requests==2.31.0)

✔️ Hashes

To prevent tampering or supply‑chain attacks.

✔️ Environment markers

(e.g., sys_platform == "win32")

✔️ Python version constraints

(e.g., >=3.10,<3.13)

✔️ Wheel selection

uv knows which wheel to install for each platform.

This is why uv can install your environment blazingly fast — it doesn’t need to re-resolve anything.

🧪 Why do you need a lockfile at all?

Without a lockfile, dependency resolution is:

  • slow
  • unpredictable
  • different on every machine
  • vulnerable to upstream changes

With a lockfile:

  • installs are deterministic
  • CI is stable
  • teammates get identical environments
  • builds are reproducible
  • dependency resolution happens once, not every time

uv’s philosophy is:

Resolve once. Install instantly forever.

🛠️ When does uv use the lockfile?

✔️ uv sync

Installs exactly what’s in uv.lock.

✔️ uv add

Updates the lockfile after adding a dependency.

✔️ uv lock

Regenerates the lockfile from scratch.

✔️ uv run

Uses the locked environment if one exists.

🧩 How is uv.lock different from requirements.txt?

Feature requirements.txt uv.lock
Exact versions Yes Yes
Transitive deps No Yes
Hashes Optional Always
Platform-specific No Yes
Python-version-specific No Yes
Reproducible installs Weak Strong
Dependency resolution Manual Automatic
  • requirements.txt is basically a flat list.
  • uv.lock is a complete dependency graph.

🧨 Do you commit uv.lock to version control?

Yes — absolutely.

Just like:

  • poetry.lock
  • package-lock.json
  • Cargo.lock
  • go.sum

The lockfile is part of your project’s reproducibility.

🎯 When does uv not use the lockfile?

If you run: 

uv pip install something

or 

uv tool run black

Those operate outside your project environment and don’t use the lockfile.

🏁 Summary: What uv.lock is for

uv.lock ensures:

  • Reproducible environments
  • Fast installs
  • Stable CI
  • Predictable builds
  • Secure dependency integrity

It locks:

  • Every dependency version
  • Every transitive dependency
  • Every hash
  • Every platform-specific wheel
  • Every Python compatibility constraint

It’s the backbone of uv’s “one tool to rule them all” approach.
