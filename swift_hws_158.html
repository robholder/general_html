Hacking With Swift - 158 - Editing SwiftData model objects

SwiftData's model objects are powered by the same observation system that makes @Observable classes work, which means changes to your model objects are automatically picked up by SwiftUI so that our data and our user interface stay in sync.

This support extends to the @Bindable property wrapper we looked at previously, which means we get delightfully straightforward object editing.

To demonstrate this, we could create a simple User class with a handful of properties.

Create a new file called User.swift, add an import at the top for SwiftData, then give it this code: 

import SwiftData
import SwiftUI


@Model
class User {
    var name: String
    var city: String
    var joinDate: Date

    init(name: String, city: String, joinDate: Date) {
        self.name = name
        self.city = city
        self.joinDate = joinDate
    }
}

Now we can create a model container and model context for that by adding another import SwiftData in the App struct file then using modelContainer() like this: 

WindowGroup {
    ContentView()
}
.modelContainer(for: User.self)

Ai Insight

That single line (.modelContainer(for: User.self) is doing a lot of heavy lifting behind the scenes. Here is the concise breakdown of how it creates your database environment:

1. The Model Container (The "Storage")

Think of the Model Container as the physical "warehouse" where your data lives. When you write .modelContainer(for: User.self), SwiftData:

  • Sets up the database: It creates a persistent storage file (usually a SQLite database) on the device.
  • Defines the schema: It looks at your User class (marked with @Model) to understand what the data "table" should look like.

2. The Model Context (The "Workpad")

Once the warehouse is built, you need a way to move items in and out. SwiftData automatically creates a Model Context and places it into the SwiftUI environment.

  • The "Scratchpad": The context is where you track changes, insert new objects, or delete old ones.
  • Communication: It acts as the middleman between your SwiftUI views and the storage file.

3. The Injection (The "Environment")

By attaching it to the WindowGroup, you are "injecting" this setup into the entire app.

  • Because ContentView() is inside the WindowGroup, it (and every view it navigates to) can now access the database using @Environment(\.modelContext).

The Summary Analogy

  • Model: The blueprint for your data.
  • Model Container: The filing cabinet (the actual file on disk).
  • Model Context: The person currently standing at the cabinet adding or removing files (handling the active changes).

Without that one line in your App struct, your views would have a "manager" (context) but nowhere to actually store the "paperwork" (the container).

Back to tutorial

When it comes to editing User objects, we would create a new view called something like EditUserView, then use the @Bindable property wrapper to create bindings for it. So, something like this: 

struct EditUserView: View {
    @Bindable var user: User

    var body: some View {
        Form {
            TextField("Name", text: $user.name)
            TextField("City", text: $user.city)
            DatePicker("Join Date", selection: $user.joinDate)
        }
        .navigationTitle("Edit User")
        .navigationBarTitleDisplayMode(.inline)
    }
}

That's identical to how we used a regular @Observable class, and yet SwiftData still takes care of automatically writing out all our changes to permanent storage - it's completely transparent to us.

Ai Insight

The @Bindable Property Wrapper is the bridge that allows you to create two-way bindings (like $user.name) for properties on classes that use @Observable or @Model.

Here is the breakdown of why it exists and how it works:

The "Missing Link"

In the past, we used @StateObject or @ObservedObject, which provided bindings automatically. With the new @Observable and @Model systems, the objects are much simpler, but they don't automatically provide a way to "bind" a specific property to a UI element (like a TextField or Toggle).

  • @Observable / @Model makes the class "visible" so the UI updates when data changes.
  • @Bindable creates the "pathway" for the UI to write data back into that class.

How to use it

You typically use @Bindable inside a view when you've passed an object in, but you need to edit its properties.

NOTE: See notes on this 'Property' version and the 'Inline' version below. 
struct EditUserView: View {
    // You receive the model object here
    var user: User 

    var body: some View {
        // You wrap it in @Bindable to get access to $ syntax
        @Bindable var user = user
        
        Form {
            TextField("Name", text: $user.name) // Two-way binding!
        }
    }
}

Key Differences to Remember

Tool Purpose
@State Used for simple local properties (Strings, Ints) or to initialize an @Observable object for the first time.
@Bindable Used to create bindings to an existing object that was passed into the view.
@Query Used specifically with @Model to fetch data from SwiftData.

Why this is safer

Before @Bindable, we often had to use @State to hold temporary copies of data and then manually save them back to the class. Now, @Bindable lets you edit the class property directly. Because it is a @Model class, those changes are often picked up by the Model Context immediately, keeping your database in sync with your typing.

It’s essentially the "glue" that lets your @Model classes talk to your SwiftUI TextFields.

Where to add the Property Wrapper

QUESTION

What's the difference between: 

// Inline version - recommended

struct EditUserView: View {
    // You receive the model object here
    var user: User

    var body: some View {
        // You wrap it in @Bindable to get access to $ syntax
        @Bindable var user = user
        Form {
            TextField("Name", text: $user.name) // Two-way binding!
        }
    }
}

..and: 

// Property version - old version

struct EditUserView: View {
    // You receive the model object here
    @Bindable var user: User

    var body: some View {
        Form {
            TextField("Name", text: $user.name) // Two-way binding!
        }
    }
}

ANSWER

This is a classic "six of one, half a dozen of the other" situation. Both versions achieve the exact same result: they create a two-way binding so your TextField can update the User model.

However, they differ in where the "binding power" is created and how the view is initialized.

1. The "Inline" Version (Inside body) - Recommended

struct EditUserView: View {
    var user: User 

    var body: some View {
        @Bindable var user = user
        // ...
    }
}
  • How it works: You receive a plain User object. Inside the body property, you "wrap" it in @Bindable right before you use it.
  • The Benefit: It keeps the struct’s initializer very simple. From the outside, you just pass in a User.
  • The Use Case: This is perfect when you receive an object from a @Query or a list and you only need it to be bindable for a specific part of the UI.

2. The "Property" Version (At the top) - Old Technique

struct EditUserView: View {
    @Bindable var user: User 

    var body: some View {
        // ...
    }
}
  • How it works: You declare the property itself as @Bindable.
  • The Benefit: It’s cleaner and more concise. You don't have to redeclare the variable inside the body.
  • The Catch: This version is slightly more "opinionated" about how the view is created. Because the property is @Bindable, the view expects to be linked to an observable object from the moment it's born.

Which one should you use?

Apple officially recommends the "Inline" version (Version 1) in many of their newer tutorials.

Here is why:

When you put @Bindable at the top level of the struct (Version 2), it can sometimes make the Preview and the Initializer a bit more finicky. By using the inline version, you are saying: "I am receiving a User object. I am choosing to treat it as a bindable source specifically within this body."

The subtle technical difference:

  • Version 1 uses @Bindable as a Local Variable.
  • Version 2 uses @Bindable as a Property Wrapper for a stored property.

The Verdict: Stick with Version 1 (the inline version). It’s more flexible, it’s what Paul Hudson typically uses in the later parts of 100 Days of SwiftUI, and it avoids some rare but annoying initialization errors that can pop up with Version 2.

Back to the tutorial

Important: If you want to use Xcode's previews with this, you need to pass a sample object in, which in turn means creating a custom configuration and container. First add an import for SwiftData, then change your preview to this: 

#Preview {
    do {
        let config = ModelConfiguration(isStoredInMemoryOnly: true)
        let container = try ModelContainer(for: User.self, configurations: config)
        let user = User(name: "Taylor Swift", city: "Nashville", joinDate: .now)
        return EditUserView(user: user)
            .modelContainer(container)
    } catch {
        return Text("Failed to create container: \(error.localizedDescription)")
    }
}

We could make a really simple user editing app out of this by adding a new user when a button is pressed, then use programmatic navigation to take the app straight to the new user for editing.

Let's build this step by step. First, open ContentView.swift and add an import for SwiftData, then add properties to get access to the model context, load all our User objects, then store a path we can bind to a NavigationStack: 

@Environment(\.modelContext) var modelContext
@Query(sort: \User.name) var users: [User]
@State private var path = [User]()

Replace the default body property with this: 

NavigationStack(path: $path) {
    List(users) { user in
        NavigationLink(value: user) {
            Text(user.name)
        }
    }
    .navigationTitle("Users")
    .navigationDestination(for: User.self) { user in
        EditUserView(user: user)
    }
}

And now we just need a way to add users. If you think about it, adding and editing are very similar, so the easiest thing to do here is to create a new User object with empty properties, insert it into the model context, then immediately navigate to that by adjusting the path property.

Add this extra modifier below the two navigation modifiers: 

.toolbar {
    Button("Add User", systemImage: "plus") {
        let user = User(name: "", city: "", joinDate: .now)
        modelContext.insert(user)
        path = [user]
    }
}

And that works! In fact, it's pretty much the same approach Apple's own Notes app takes, although they add the extra step of automatically deleting the note if you exit the editing view without actually adding any text.

As you can see, editing with SwiftData objects is no different from editing regular @Observable classes - just with the added bonus that all our data is loaded and saved neatly!
