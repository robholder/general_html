Hacking With Swift - 124 - How to make a NavigationStack return to its root view programmatically

It's common to be several levels deep in a NavigationStack, then to decide to return to the beginning.

For example, perhaps your user is placing an order, and has worked their way through screens - then wants to go back to the root view of NavigationStack.

To demonstrate this, we can create a sandbox that can push to new views endlessly by generating new random numbers each time.

First, here's our DetailView that shows its current number as its title, and has a button that pushes to a new random number whenever it's pressed: 

struct DetailView: View {
    var number: Int

    var body: some View {
        NavigationLink("Go to Random Number", value: Int.random(in: 1...1000))
            .navigationTitle("Number: \(number)")
    }
}

And now we can present that from our ContentView, starting with an initial value of 0 but navigating to a new DetailView every time a new integer is shown: 

struct ContentView: View {
    @State private var path = [Int]()

    var body: some View {
        NavigationStack(path: $path) {
            DetailView(number: 0)
                .navigationDestination(for: Int.self) { i in
                    DetailView(number: i)
                }
        }
    }
}

When you run that back you'll see you can keep pushing your way through views endlessly - you can go 20, 50, or even 500 views deep if you like.

Return to Root

Now, if you're 10 views deep in there and decide you want to return home you have two options:

  • If you're using a simple array for your path, like we're doing in the code above, you can call removeAll() on that array to remove everything in your path, going back to the root view.
  • If you're using NavigationPath for your path, you can set it to a new, empty instance of NavigationPath, like this: path = NavigationPath().

However, there's a bigger problem: how you can do that from the subview, when you don't have access to the original path property?

@Binding

Here you have two options: either store your path in an external class that uses @Observable, or bring in a new property wrapper called @Binding. I've already demonstrated @Observable previously, so let's focus on @Binding here.

You've seen how @State lets us create some storage inside our view, so we can modify values while our app is running. The @Binding property wrapper lets us pass a @State property into another view and modify it from there - we can share a @State property in several places, and changing it in one place will change it everywhere.

In our current code, that means adding a new property to DetailView to get access to the navigation path array: 

@Binding var path: [Int]

And now we need to pass that in from both places in ContentView where DetailView is used, like this: 

DetailView(number: 0, path: $path)
    .navigationDestination(for: Int.self) { i in
        DetailView(number: i, path: $path)
    }

As you can see, we're passing in ＄path because we want to pass the binding in - we want DetailView to be able to read and write the path.

And now we can add a toolbar to DetailView to manipulate the path array: 

.toolbar {
    Button("Home") {
        path.removeAll()
    }
}

And if you're using NavigationPath you'd use this: 

.toolbar {
    Button("Home") {
        path = NavigationPath()
    }
}

Sharing a binding like this is common - it's exactly how TextField, Stepper, and other controls work. In fact, we'll look at @Binding again in project 11 when we create our own custom component, because it's really helpful.

struct DetailView: View {
    // Add @Binding var of 'path'
    @Binding var path: [Int]
    var number: Int

    var body: some View {

            .navigationTitle("Number: \(number)")
            // Add ubiquitous 'Home' button to return to root
            .toolbar {
                Button("home") {
                    // Reset 'path' array
                    path.removeAll()

AI Insight

In this specific code, the path lines feel redundant because the user is doing all the work by tapping links.

However, they represent the shift from Implicit Navigation (SwiftUI does it for you) to Explicit Navigation (You own the data).

1. The Role of the @State path

Without path, navigation is a "black box." With it, you have a two-way mirror:

  • User Action: When a user taps a NavigationLink, SwiftUI automatically appends that Int to your path array.
  • Developer Action: If you change that path array in code, the UI will jump to match it.

2. Why it seems like a placeholder

In your current code, you aren't actually reading or writing to path inside your body. The NavigationStack is just dutifully updating the array in the background every time you tap "Go to Random Number."

It becomes useful the moment you want to do something Programmatic. For example, if you wanted a "Back to Start" button on a view 10 levels deep: 

// Without the path variable, this is impossible. 
// With it, it's one line:
Button("Go Home") {
    path.removeAll() 
}

3. The "Binding" (＄path)

The ＄ is the "Handshake." It tells the NavigationStack:

"Don't just use your own internal memory to track where we are. Use this specific array instead. If the user taps 'Back', remove an item from my array. If I empty the array, pop everything off the screen."

4. Comparison: Implicit vs. Explicit

feature NavigationStack { ... } NavigationStack(path: ＄path) { ... }
Logic Implicit (Internal) Explicit (Programmatic)
Control User-driven only User-driven AND Code-driven
Deep Linking Hard to implement Easy (just pre-fill the array)
Reset to Root Not easily possible path = []
