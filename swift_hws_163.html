SwiftData: Wrap up

Although we've only really scratched the surface of what SwiftData can do, you’ve still seen how it can add, delete, sort, filter, and more, all with relatively simple code. Yes, a few parts are a little murky in Swift - #Predicate, for example, can take some getting used to - but as long as you're careful you should be fine.

Perhaps the most important thing about SwiftData is that it’s guaranteed to be there for all apps, on all of Apple’s platforms. This means you can use it regardless of your needs: maybe it’s for saving important data, maybe it’s just a cache of content you downloaded; it doesn’t matter, because SwiftData will do a great job of managing it for you.

Ai Insight

QUESTION

I have a challenge to complete. I need to take my existing app (iExpense) and utilize SwiftData instead of UserDefaults for app data storage.

... I think the first step is to import SwiftData to the swift files, then add .modelContainer(for: Expenses.self) to the @main struct, but after that I'm clutching at straws. I think I need to change @Observable to @Model. Do I need to@Bindable on the various data variables? As you can see, I'm flailing about a bit. I should be able to understand the steps, but I don't have them in mind to actually implement the steps.

ANSWER

You are actually on the exact right track! Don't let the "flailing" feeling get to you - migrating from UserDefaults to SwiftData is mostly about deleting old, "manual" code and letting SwiftData handle the heavy lifting.

Think of it this way: You are firing the Expenses class from its job as a "Middleman" and letting the ModelContext talk directly to the database.

Here is your step-by-step roadmap to victory:

1. Transform the Model

You don't need a separate Expenses class anymore. Instead, we turn ExpenseItem into a SwiftData Model.

Delete the Expenses class entirely.

Change ExpenseItem to a class and add @Model.

You no longer need Codable (SwiftData handles that) or Identifiable (SwiftData adds a persistent ID automatically). 

import SwiftData

@Model
class ExpenseItem {
    var name: String
    var type: String
    var amount: Double
    
    init(name: String, type: String, amount: Double) {
        self.name = name
        self.type = type
        self.amount = amount
    }
}

2. Prepare the App Entry Point

In iExpensesApp.swift, you tell the app to create storage for your new model. 

@main
struct iExpenseApp: App {
    var body: some Scene {
        WindowGroup {
            ContentView()
        }
        .modelContainer(for: ExpenseItem.self) // Link the database here
    }
}

3. Update ContentView (The Fetcher)

In ContentView, we swap @State private var expenses for a @Query. 

struct ContentView: View {
    @Environment(\.modelContext) var modelContext // The "hand" that moves data
    @Query var items: [ExpenseItem] // The "live stream" of data from the DB

    // Computed properties stay almost the same, just filter the @Query result
    var personalExpenses: [ExpenseItem] {
        items.filter { $0.type == "Personal" }
    }
    var businessExpenses: [ExpenseItem] {
        items.filter { $0.type == "Business" }
    }

    var body: some View {
        NavigationStack {
            List {
                Section("Personal") {
                    ForEach(personalExpenses) { item in
                        ExpenseRow(item: item)
                    }
                    .onDelete { offsets in
                        removeItems(at: offsets, in: personalExpenses)
                    }
                }
                // ... same for Business section ...
            }
            .toolbar {
                NavigationLink {
                    AddView() // We don't need to pass 'expenses' anymore!
                } label: {
                    Label("Add Expense", systemImage: "plus")
                }
            }
        }
    }

    func removeItems(at offsets: IndexSet, in source: [ExpenseItem]) {
        for offset in offsets {
            let item = source[offset]
            modelContext.delete(item) // Just tell the context to delete it!
        }
    }
}

4. Update AddView (The Inserter)

AddView needs to talk to the modelContext to save new items. Because you are using TextField with @State properties, you actually don't need @Bindable here-the standard @State logic works fine until the moment you hit "Save." 

struct AddView: View {
    @Environment(\.dismiss) var dismiss
    @Environment(\.modelContext) var modelContext // Add this!
    
    @State private var name = ""
    @State private var type = "Personal"
    @State private var amount = 0.0
    
    let types = ["Business", "Personal"]

    var body: some View {
        Form {
            TextField("Name", text: $name)
            Picker("Type", selection: $type) {
                ForEach(types, id: \.self) { Text($0) }
            }
            TextField("Amount", value: $amount, format: .currency(code: "USD"))
        }
        .toolbar {
            Button("Save") {
                let item = ExpenseItem(name: name, type: type, amount: amount)
                modelContext.insert(item) // Save to the database
                dismiss()
            }
        }
    }
}

Summary of what changed:

  • Storage: We traded UserDefaults (slow, manual JSON encoding) for SwiftData (fast, automatic).
  • Middleman: We deleted the Expenses class. The @Query in ContentView now acts as the source of truth.
  • The "Hand": We added @Environment(\.modelContext) to both views. This is the "hand" we use to either Insert a new item (in AddView) or Delete one (in ContentView).

When would you use @Bindable?

You mentioned @Bindable. You would only use that if you were passing an existing ExpenseItem into a subview to edit its properties directly (like a "Details" screen). For creating a new item from @State strings and doubles, you don't need it!
