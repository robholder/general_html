Basics

  • 00:00:00 - 00:13:06 : Basics

Most particle systems start with a source (object), piped into a POP Network.

Basic Setup

Inside a Geometry (container) node:

  • Create Sphere
  • Add a POP Network:
    • Connect Sphere into first input
    • Set focus on the DOP/POP Network (select and R key)
    • Playing the timeline, we'll see the particles spawning.
  • Inside the DOP/POP Network:
  • Add a POP Force node (freestanding) and pipe into the Merge node (labeled wire_pops_into_here):
    • In the Force parameter, set to 0, 1, 0 (for an upward direction - y axis)
    • Playback to see the particles rise
    • Experiment with:
      • Noise tab:
        • Amplitude for example (set to 1 perhaps)
        • Look at others
  • Add a POP Color node (and again, pipe to the Merge node):
    • The default Color Type is Constant
    • You can set the Color parameter and see this color in the Viewport particles.
    • Switch Color Type to Ramp:
      • This is already set with a VEXpression based on the age (@nage) attribute:
        • ramp = @nage; which is 'normalized age'
        • change to ramp = @age; to be just age
      • The Range fields are 'seconds' - 0 to 1 is zero to one second:
        • 0 will be the left color on ramp, 1 will become the color on the right (passing through the range as the time moves ahead)
        • change to 0 and 4 and see result
      • On the cog wheel icon to the right of the ramp, there are preset colors to choose from

Setting the 'life' of the particles:

  • On the POP Source node ('source_first_input') node's parameters:
    • Birth tab:
      • Life Expectancy field set's the age (in seconds): try 2
      • Life Variance gives a variance (in seconds) of the value given: try 0.5

Bad news - we need to learn to control things ourselves, not rely on pre-built stacked nodes.

Physically Accurate vs Art Directed Intentions

The point is made that although the 'POP Force' type pre-made POP nodes are accurate and 'realistic', they don't necessarily give the control that an art-directed intention requires.

So we need more control of the behavior of particles, and the many pre-fabricated 'POP' nodes might not be the best option.

The Geometry VOP may be the gateway in to more control.

Geometry VOP

  • 00:13:06 - 00:35:46 : Geometry VOP

One of the differences between a Geometry VOP and an Attribute VOP is in the process of using an OpInput#.

In the Attribute VOP, the nodules on the outside of the node, take input from pipes from input objects. In the Geometry VOP, the inputs must be referenced via an asset's path.

Select the Geometry VOP node, and in its parameters:

  • Inputs tab:
    • Input 1 (Input 2, Input 3 etc.), select an option from the drop-down:
      • Maybe SOP, then the field will become open to paste or type an asset path:
        • Example: /obj/geo2/goal
  • Now, inside the Geometry VOP, the geometryvopglobal node's OpInput1 widget output will reference /obj/geo2/goal

Mini Project

In the example mini-project, our goal is to set a 'target' point in space (Add node), and get the particles to gravitate to that point:

  • Add node is placed in the 'geo' container and set to be a single point:
    • Check on the checkbox to actually set the point.
  • And inside the Geometry VOP:
    • an Import Point Attribute node is dropped and connected to the OpInput1 output into its file input.
    • The Attribute field is set to P on the Import Point Attribute node
    • The Signature drop-down is set to Vector 3
    • Promote the ptnum input to Constant (MMB) and to a value of zero (for the one and only point in the Add node.
  • To get a direction vector, we have to subtract the current particle position from the target position, so we drop a Subtract node:
    • input1 takes in the result from the Import Point Attribute node.
    • input2 takes in the P position from the geometryvopglobal
  • The output needs to be 'normalized':
    • Drop a Normalize node from the diff output of the Subtract node.
    • From the nvect output of the Normalize node, plug into the force input on the geometryvopoutput node.

Problem

  • Playing the result, the particles move from the Sphere geometry, towards the Add point, but then overshoot past.
  • Eventually they turn around an move again towards the target point.

Fix

  • If we plug the nvec into the v (velocity) on the geometryvopoutput instead of the force, the particles behave more like we expected or hoped (honing in on the target point).

Another control

  • You can add a Multiply Constant node after the Normalize to control the speed.

Conclusion on Geometry VOP vs POP nodes

Examples are shown using the POP <blah> nodes and how tweaking and controlling them is difficult compared to the Geometry VOP node.

"The dirty secret" is that inside these various pre-built POP <blah> nodes, is that there is a Geometry VOP inside them. Look inside to see how they are wired.

  • RMB and choose Allow Editing of Contents.

It was noted that a lot of the inner nodes of the Geometry VOP nodes (inside pre-built POP <blah> nodes) look intimitdating, but many are just links to the parameter controls on the outside.

Back to a Geometry VOP

Start afresh inside the POP Network

After the POP Source node, add a Geometry VOP node.

Let's form a plan:

  • First we need to generate a direction vector, and add noise on top:
    • The upward vector is easy - it's a constant 0,1,0 vector.
    • The rotation vector needs to be generated by taking a cross product of a plane that cuts through the torus (one edge towards the center, the other straight 'up').
      • Upward vector:
        • 0,1,0
      • Center Vector:
        • Position of point minus center location:
          • P(x, y, z) - C(x, y, z) (Center is 0, 0, 0 in our case)

Building this calculation inside the 'Geometry VOP'

Jump into the Geometry VOP:

  • Find the vector pointing to the center of the Torus:
    • Create a Constant:
      • Set to 3 Floats (vector)
      • Leave at zero, zero, zero (since that's the center of our Torus)
    • Out of the Value output add a Subtract node:
      • Plug the P (pos) output from the 'geometryvopglobal1' node into the Subtract 'input2'.
    • Out of the Subtract add a Normalize node
      • This is what we need for the cross-product calculation.
    • From the nvec output from the Normalize:
      • Add a Cross Product node (vec1)
    • Inside the Normalize there is facility to hard code a vector.
      • (Otherwise we could add a Constant, but it's not necessary.)
      • Vector2 field to 0, 1, 0 (for an upward normalized vector)
    • The crossprod output can then be plugged into the v of the geometryvopoutput1 node.

Discussion on Geometry Resolution

This discussion centers on the Scatter onto Surface parameter choice on the POP Source node.

NOTE: the resolution of the grid above was set fairly high. If the Rows and Columns of the grid are dialed down, the noise generation is much less defined, and effects the noise attributes significantly. This affects the resulting particle simulations greatly as a result.

The issue is that this method restricts us to the supplied geometry, and introduces an unpredictability to results.

Summing up and Conclusion

Scatter onto Surface can still be a useful approach (especially for curves), but whatever works for the project.

All Geometry

A note on the All Geometry option on the POP Source parameters.

All Geometry actually spawns a new iteration or copy of the input geometry, and affects the primitives by carrying them along with the particle positions. For a geometry grid, it's like a new sheet being spawned each frame, and distorted as if laid on the point carpet.

For Curve inputs (maybe set the grid geometry's Connectivity parameter to Columns, which results in column lines instead of polys to see the result).

HOWEVER, if we are supplying points (via a Scatter node perhaps), All Geometry will still only be the points and the same as All Points.

To prevent multiple spawns of geometry instances

To stop new iterations of the source geometry, you have to animate the values on the POP Source Impulse Activation parameter.

Keyframe Option:

On frame one of the sim, set and keyframe the value to '1', then on the next frame set a keyframe at '0'. Now it will appear to be an animating 'sheet' in the case of a grid.

Expression Option:

Add the string, $F==1.

This is 'if $frame is 1, value is True (1) else the value is False (0)'.

To demonstrate the expression syntax a little further, to spawn only for the first 10 frames it would be $F<=10

<cntl> <shift> LMB on the field to delete the expression.

Fade Source (when using 'Scatter')

To start removing point sources when using a Scatter node to lay down the source points.

  • Add another Attribute VOP, and inside:
    • Add a Random node to the ptnum output on the INPUT node
    • Add a Compare node to the rand output from the Random node (to 'input1'):
      • set the Test to Greater Than
      • adjust the Compare to Float slider to perhaps 0.8
    • Now we can use an If Block from the bool output of the Compare node:
      • From the condition output, add a Remove Point node.
      • This will be connected by its handle input
      • Connect the ptnum from the INPUT node to the ptnum input on the Remove Point node
      • If the 'caution' error icon is persistent, toggle the Q key on the if node to bypass and reinstate.
    • You have to Promote attributes or fields to animate. (You can't animate within the Attribute VOP.):
      • Promote the second input ('input 2') on the Compare node - MMB on the input and choose Promote
      • Now you can animate this value:
        • At frame one set and keyframe a value of 0
        • At frame 48 (for example) set and keyframe 1
        • At frame 96 (for example) set and keyframe 0 again
        • Now on playback the particles start to get emitted, then fade to no more as the sim proceeds.

An Aside

A comparison is made of leaving our Geometry VOP ('set_v') outside the POP Network where it was built and transferring it to the inside of the POP Network. Outside, the velocity is inherited and altered by the POP Network, but if placed inside, it SETS the velocity every frame. Not the same result.

  • As a detail, our setup has a Normalize node, so every frame, the vectors would be reset to 1. Not what we want.

Be mindful of where velocities are set.

Just Born Group

On the parameter panel for the POP Source node (inside of the POP Network) on the Birth tab, there is an Attribute field labeled Just Born Group.

This parameter can be given a name (group name), perhaps born for example. Now, when a particle is 'born' or created it enters the born group, but on the next frame it is removed. So anything we do to this group will only happen on the first frame of its existence.

So if we had a Geometry VOP network after the POP Source (remembering the issue we had in the aside above, where we are re-setting the velocity on every frame - which isn't ideal), we can insert the born group into the Geometry VOP Group field. This would mean that the algorithm inside the Geometry VOP would only get applied on the first frame of each particle's existence.

In our comparison, the Geometry VOP inside the POP Network (with the 'Just Born Group' attribute) would perform the same as if the Geometry VOP (without the 'Just Born Group') were placed outside (up one level) of the POP Network.

So this Just Born Group attribute trick can be useful if you don't have access to place the Geometry VOP at the outer level.

Interpolate Source

  • 01:33:58 - 01:42:31 : Interpolate Source

There is a dedicated lesson on this subject here:

FX Essentials - Fix Particle Stepping - Houdini Tutorial

Meanwhile, this subject covers the emission of particles from moving sources.

The example in this course uses a wielding hammer. The example takes the hammer head as the source, and by default particles are emitted in a single frame moment of birth. So playback reveals a trail of particles that appear to be a stamp of the object as it was as it past through space on each frame.

Interpolate Source is an option that interpolates or samples the birth BETWEEN frames to give a smooth emission of particles throughout the course or trajectory of the emitting object.

There are a few options for the Interpolate Source parameter:

  • None:
    • No interpolation
  • Forward:
    • Predicts the emissions from the 'next frame'
  • Back:
    • A natural trail of particle creation, based on the 'last frame', forming a kind of 'wake'

Interpolation Method:

  • Match Topology:
    • Is consistent if the topology of the source remains the same between frames
    • If the topology changes between frames, it'll leave ugly mismatches between emissions.
  • Use Point Velocity:
    • Uses the general motion of the object, so a more smooth emission.

Just a note on the tutorial content: there is an interesting example for the interpolation of particles using a packed geometry animation. It's unpacked to gain access to the geometry to be able to unpack, and use only certain surfaces or items to be sources for emission.

Controlling the Behavior from the Static Object (Sphere)

We can control the behavior of the particles on contacting the surface by adjusting the settings on the Static Object node.

  • Physical tab:
    • Bounce:
      • How much the particles bounce off the collision surface
    • Bounce Forward:
      • Bouncing works like a normal angle of reflection equals angle of incidence
      • With that in mind, Bounce Forward affects only bounces beyond the perpendicular (90º) to the trajectory of the particle, or continued forward motion.
      • (Backward would be bouncing back toward the source as contrast)
      • Example - if Bounce is set to zero, and Bounce Forward to 1, the particles will splat and slide off the surface.
    • Friction:
      • This is how 'sticky' the collision surface seems
    • Dynamic Friction scale:
      • This is how 'sticky' the collision surface get AFTER the initial impact of the particle.
    • Temperature:
      • Not explained in the tutorial, as the tutor never uses it.
  • Collisions tab:
    • RBD Solver tab:
      • Collsion Detection:
        • Use Solver Default is the default setting
        • Use Volume Collisions
        • Use Surface Collisions

The tutorial now goes into some detail to why it's recommended by the tutor to use Use Volume Collisions:

  • The are faster
  • They are fairly accurate
  • And when a collision object is converted into a volume, the details of the collision mesh aren't that much of a problem if they change or are edited.

If Use Volume Collisions is used, there are some settings to set up:

  • You can switch off the Display Geometry checkbox (as this refers to the SOP Path item and we're changing to Volume.
  • Switch 'On' the Collision Guide checkbox (to show the volume as used in the calculation).
  • Uniform Divisions sets the voxel resolution of the volume object.
