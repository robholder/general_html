rsync

At its core, rsync (Remote Sync) is a utility used for copying and synchronizing files and directories either locally or across a network.

What makes rsync legendary among developers and sysadmins is its delta-transfer algorithm. Instead of blindly overwriting files like cp or scp, rsync compares the source and destination, calculates the exact differences, and only sends the bytes that actually changed.

For your specific workflow (hundreds of text files inside multiple .git repositories), rsync is lightning fast because code diffs are tiny, even if your total directory size is gigabytes.

1. The Core Anatomy of an rsync Command

The basic syntax is always: 

rsync [options] /path/to/source/ /path/to/destination/

The "Trailing Slash" Trap ⚠️

There is one massive quirk with rsync that everyone trips over at least once. The presence or absence of a trailing slash (/) on the source folder completely changes what happens:

  • With a trailing slash (source/): Copies the contents of the folder into the destination. 
rsync -a /my/photos/ /backup/
# Result: /backup/pic1.jpg, /backup/pic2.jpg
  • Without a trailing slash (source): Copies the folder itself into the destination. 
rsync -a /my/photos /backup/
# Result: /backup/photos/pic1.jpg, /backup/photos/pic2.jpg

2. Essential Flags to Know

You rarely use rsync naked. You almost always use these standard flags:

  • -a (Archive Mode): The holy grail flag. It is a shortcut for -rlptgoD. It ensures that symlinks, modification times, permissions, groups, and owners are all preserved perfectly. For Git repos, this is mandatory to keep your metadata intact.
  • -v (Verbose): Prints out exactly what files are currently transferring.
  • -z (Compress): Compresses file data during the transfer. Great for saving bandwidth over a network.
  • -n or --dry-run: Always use this first! It simulates the sync without actually changing any files. It lets you verify that your trailing slashes are correct before committing to the transfer.
  • --delete: Deletes files in the destination directory if they no longer exist in the source directory. Essential for keeping a mirror clean.

3. macOS vs. Linux: The Crucial Differences

While rsync behaves mostly the same on both operating systems, there are two major differences that can bite you if you are syncing between a Mac and a Linux machine.

Difference A: The Engine Version (GPLv2 vs. GPLv3)

  • Linux: Almost all modern Linux distributions ship with rsync version 3.x. Version 3 uses an advanced, incremental file-scanning algorithm that starts transferring files immediately while it's still scanning the rest of the drive.
  • macOS: Apple ships an ancient version of rsync (usually version 2.6.9). Apple froze it there because newer versions of rsync moved to the GPLv3 license, which Apple avoids for legal/corporate reasons.
💡 The Impact: The old Mac version scans the entire directory structure into memory before it sends a single file. If you have hundreds of tutorial repos with thousands of tiny files, a Mac-native rsync might feel sluggish or hog RAM compared to Linux.

The Fix: Install a modern version of rsync on your Mac via Homebrew:

brew install rsync

(Note: You may need to call it via /opt/homebrew/bin/rsync or update your $PATH so it overrides Apple's default).

Difference B: Extended Attributes and Resource Forks (Mac Metadata)

macOS hides a lot of operating system metadata inside files (Finder tags, icons, color labels, and ACL permissions). Linux doesn't natively care about these.

  • On Linux: The -a flag is usually enough.
  • On macOS: To safely copy Mac-specific metadata, you need to explicitly add the -E (or --extended-attributes) flag.

If you are syncing from a Mac to a Linux machine, or Mac-to-Mac, your optimal flag combination changes slightly: 

# Standard Linux-to-Linux
rsync -avz --delete source/ dest/

# macOS (using Homebrew rsync 3.x) to safely handle Mac metadata
rsync -avzE --delete source/ dest/
