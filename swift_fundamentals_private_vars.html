Swift - Private Vars

In Swift, the private keyword is an access control modifier.

When you mark a view property as private var, you are telling the compiler:

"This piece of code belongs strictly to the ContentView struct. No other file, struct, or object in this entire app is allowed to look at it or use it."

By default, if you don't type private, Swift treats the property as internal, meaning any other file inside your app project could theoretically reach inside your view and try to grab or render that sub-view.

Here is exactly what those restrictions mean in practice for your notes:

🔒 What private Restricts (and Protects)

1. It Blocks Scope Leakage

If you have sub-views which have been pared away from the main view code like diceImageSection for example (to call on for different orientation perhaps), it makes sense to put them inside the specific view - ContentView for example.

If you had a second view file in your project (for example, a SettingsView.swift), typing SettingsView().diceImageSection would throw an immediate compiler error. The outside world cannot see it.

2. It Speeds Up Xcode's Auto-Complete

As your app grows, you will have dozens of files. If every button, card, and layout section you extract is accessible globally, your Xcode auto-complete menu will become an unreadable mess of hundreds of choices. Marking layout variables as private hides them from the global menu, keeping your global workspace clean.

3. It Optimizes Compiler Efficiency

When the Swift compiler translates your code into a physical app, it looks for ways to make the app run faster. If it sees a view property marked private, it knows with 100% certainty that no other file will ever try to modify or call it. This allows the compiler to perform hidden speed optimizations (like "dead code elimination" and inline rendering) that make your interface snappier.

🏗️ Extracted View Variables vs. Standalone Structs

When organizing SwiftUI files, you have two ways to break up your code.

The private keyword behaves slightly differently depending on which one you choose:

Method A: Extracted Properties (What we used)

You declare variables inside the parent view. They must be marked private because they are structurally embedded inside the main view's body boundaries. 

struct ContentView: View {
    @State private var sides = 6

    var body: some View {
        VStack { diceImageSection }
    }

    // 🔒 Restricted: Lives inside ContentView, can read 'sides' directly
    private var diceImageSection: some View {
        Image("dice_\(sides)")
    }
}

Method B: Standalone Structs (The Alternative)

If you move the code completely outside of ContentView into its own separate struct block, it can no longer be a private var. It becomes its own independent entity. To limit its scope, you would mark the entire struct as private: 

// 🔒 Restricted: Only files/code inside this specific physical .swift text file can see this struct
private struct DiceImageSection: View {
    let sides: Int // Must pass data in manually now!
    
    var body: some View {
        Image("dice_\(sides)")
    }
}

Rule of Thumb for Your Notes:

Always default to making your extracted layout properties private var. It enforces a clean separation of concerns, ensures your sub-views don't accidentally get misused by other parts of your app, and signals to other developers (and your future self) that this layout block is a specialized piece of design tailored for this file alone.

Scope

When you apply the private keyword to a top-level, free-standing entity (like a struct, class, or global variable written outside of any other curly braces), the physical .swift file itself becomes its bounding container.

In this specific scenario, when adding private the file essentially acts as the outermost "wall" restricting access to that entity.

Here is how that looks visually and structurally in your code:

🏗️ Top-Level private in Practice

Imagine you have a file named CustomComponents.swift: 

// Physical File: CustomComponents.swift

// 🔒 Top-level private: The file is the container.
// Only code written inside this exact text file can see or use this struct.
private struct SecretBackgroundModifier: ViewModifier {
    func body(content: Content) -> some View {
        content.background(Color.purple.opacity(0.2))
    }
}

// 🌐 Public/Internal view: Accessible to the whole app.
struct SharedCardView: View {
    var body: some View {
        Text("Hello World")
            .modifier(SecretBackgroundModifier()) // ✅ ALLOWED: Inside the same file!
    }
}

If you try to go into your main ContentView.swift file and write .modifier(SecretBackgroundModifier()), Xcode will give you a compiler error saying it cannot find that modifier in scope.

📝 The Golden Rule

The scope of the private keyword is always determined by the closest pair of matching curly braces { } surrounding it.

  • If it is inside a struct { ... }, it is locked to that struct.
  • If it is at the root of the file with no braces around it, it locks to the entire file.

Restricting to a File

If you want something to be accessible anywhere within the same physical .swift file, Swift has a different keyword for that called fileprivate.

Here is exactly how the two scopes compare:

🔎 private vs fileprivate

1. private (Scoped to the Curly Braces)

When you declare private var diceImageSection, it can only be seen inside the curly braces of struct ContentView. If you write a completely separate struct further down in the exact same file, that second struct is blocked from seeing it. 

// Physical File: ContentView.swift

struct ContentView: View {
    private var actionWord = "Roll" // 🔒 Completely hidden from the outside world
}

struct SecondaryView: View {
    var body: some View {
        // ❌ COMPILER ERROR: 
        // Even though it's in the same file, SecondaryView cannot see 'actionWord'
        Text(ContentView().actionWord) 
    }
}

Note: Swift does allow private properties to be shared with extensions of that same struct, as long as the extension lives in that same physical file.

2. fileprivate (Scoped to the Text File)

If you change the keyword to fileprivate, you are telling the compiler: "Anyone standing inside this physical text file can share this variable, but keep it hidden from all other files in the app." 

// Physical File: ContentView.swift

struct ContentView: View {
    fileprivate var actionWord = "Roll" 
}

struct SecondaryView: View {
    var body: some View {
        // ✅ ALLOWED!
        // Because it's 'fileprivate' and inside the same physical file.
        Text(ContentView().actionWord) 
    }
}

📝 Summary for your Reference Sheet

  • private = "Only visible inside my specific struct or class container blocks." (This is what you should default to 95% of the time).
  • fileprivate = "Visible to any code written inside this physical .swift text file, but hidden from other files."
