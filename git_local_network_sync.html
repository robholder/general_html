Git - Local Network Sync (Push & Pull)

Ai Insight

QUESTION

I have a git repo for each tutorial project - and it works nicely as I can switch --detach to any commit to see the apps as they stood at various stages of development.

I have copied the directory in which all the repos are stored onto a separate machine (by copying over the network).

Now that I'm adding more commits (and new repos), what's the best method of syncing these folders to maintain a copy on my 2nd machine?

ANSWER

Since you are working with a collection of Git repositories, you are in a great position. You have two excellent paths forward depending on whether you want to sync the entire directory as raw files, or sync via Git itself (which is much cleaner and more robust for development).

Here is a breakdown of the two best methods, along with the pros and cons of each for your specific setup.

Method 1: The Git Native Way (Best for Developers)

Since every tutorial project is already a Git repository, the most reliable method is to treat your second machine as a Git Remote (like a private GitHub).

Instead of copying folders, you just git push your new commits from Machine 1 to Machine 2 over the network.

How to set it up:

  • On Machine 2 (the backup/receiver): Ensure SSH is enabled. Note its IP address and the path to the parent directory (e.g., user@192.168.1.50:/path/to/tutorials/).
  • On Machine 1 (your main workspace): Inside one of your tutorial repos, add Machine 2 as a remote destination named backup: 
git remote add backup user@192.168.1.50:/path/to/tutorials/project-name
  • To Sync: Whenever you make new commits and want to sync, just run: 
git push backup --all --tags

(The --tags flag ensures your step-by-step tutorial tags or branches move over too).

Pros & Cons:

  • Pros: Extremely precise. It only sends the tiny text diffs of your new commits over the network, making it lightning-fast. It won't mess up your --detach states.
  • Cons: You have to manually run git push inside each repo, and for brand new tutorial repos, you have to initialize them on Machine 2 first.

Method 2: The rsync Way (Best for Bulk Syncing)

If you want a single command that completely clones the parent directory-handling new commits, new repositories, and untracked files all at once - you should use rsync.

rsync is a intelligent file-transfer tool built for Unix systems. It scans both machines, compares them, and only copies the files (or parts of files) that have actually changed. Because it copies the hidden .git folders perfectly, all your commit history and detached states remain intact.

The Magic Command (Run from Machine 1):

rsync -avz --delete /path/to/local/tutorials/ user@192.168.1.50:/path/to/remote/tutorials/

What those flags mean:

  • -a (archive): This is critical. It preserves file permissions, timestamps, symlinks, and recursively copies everything.
  • -v (verbose): Shows you exactly which files are flying across the network.
  • -z (compress): Compresses the data during the transfer to save network bandwidth.
  • --delete (Optional but recommended): If you delete a failed tutorial repo on Machine 1, this flag ensures it gets deleted on Machine 2 as well, keeping them perfectly mirrored.

Pros & Cons:

  • Pros: One command syncs everything. It automatically picks up brand-new repository folders without any extra setup.
  • Cons: If you leave a file uncommitted on Machine 1, rsync will copy that uncommitted file over, which might clutter your workspace on Machine 2.

Which one should you choose?

  • Choose rsync (Method 2) if you want a "fire and forget" solution. You can wrap that single rsync command into a bash alias (like alias synctutorials="...") and run it at the end of every study session.
  • Choose Git Remotes (Method 1) if you want to practice professional Git workflows and want absolute control over exactly when commits are pushed to your second machine.

QUESTION

Does Method 1 allow for push in both directions?

ANSWER

Yes, it absolutely does! Because Git is entirely decentralized, both machines can act as remotes for each other. You can push and pull in both directions seamlessly.

However, because you are using Machine 2 as a working computer (and not a cloud service like GitHub), there is a critical Git rule you have to watch out for to prevent errors.

The Golden Rule: Never Push to an Active Branch

By default, Git will block you if you try to git push into a folder on Machine 2 if that machine currently has the exact same branch checked out.

If Machine 2 is currently sitting on the main branch, and you try to git push backup main from Machine 1, Git will spit out an error. It does this to protect Machine 2 from having its underlying files changed out from underneath it while you are working.

How to Sync in Both Directions Safely

Because you mentioned you usually work with detached commits (e.g., git checkout <commit-hash>), you are actually in a safer position than most! However, here are the two best workflows for bidirectional syncing.

Workflow A: The "Fetch and Merge" Strategy (Safest)

Instead of pushing your changes across the network into a working directory, you use the second machine to pull the changes in.

  • You add each machine as a remote to the other:
    • On Machine 1: git remote add mac2 user@machine2:/path
    • On Machine 2: git remote add mac1 user@machine1:/path
  • When you finish working on Machine 1, leave it alone.
  • Walk over to Machine 2 and type: 
git pull mac1 main

(Or if you are in a detached state, run git fetch mac1 to fetch all the tutorial stages, then git checkout the specific commit you want to see).

Workflow B: The "Bare Repository" Hub (Most Professional)

If you truly want a seamless workflow where you can blindly git push from either machine at any time without thinking about what branch the other machine has open, you introduce a third, "neutral" folder called a Bare Repository.

A Bare Repository is a Git folder that contains only the version history data, but has no actual working files visible. You can set this up on a cheap network drive, a shared folder, or just a hidden folder on one of the machines. 

Machine 1 (Work) ----> [ Bare Repo Hub ] <---- Machine 2 (Work)
  (Push/Pull)               (Shared)              (Push/Pull)
  • Create a bare repo on your network or on Machine 2: 
git init --bare /path/to/shared/tutorial.git
  • Both Machine 1 and Machine 2 add this bare repo as their origin: 
git remote add origin user@ip:/path/to/shared/tutorial.git

Now, both machines can git push origin main and git pull origin main whenever they want. There are never any conflicts or blocked pushes because nobody is actively "standing inside" the bare repository.

QUESTION

Would all of these git solutions require a set up for each tutorial repo?

My directory of tutorial repos are many fold; all within an enclosing directory.

ANSWER

Yes, you hit the nail on the head. That is the major drawback of Method 1. Because Git operates strictly at the individual repository level, you would have to manually configure remotes and run push/pull commands for every single tutorial folder individually. If you have dozens of folders inside an enclosing directory, that quickly becomes a maintenance nightmare.

If you want a bidirectional solution that covers the entire enclosing directory all at once without touching individual Git configurations, here are the two best ways to handle a "many-folder" setup.

Solution 1: The rsync Two-Way Sync (Simplest)

While rsync is naturally a one-way mirror, you can easily use it bi-directionally as long as you adhere to one rule: Always sync from the machine you just finished working on.

Because rsync only copies modified files, it will seamlessly update the other machine's entire tree of Git repositories (including all histories and detached head states) in a single command.

The Setup:

You create two distinct bash aliases (one on each machine) inside your ~/.bashrc or ~/.zshrc file:

  • On Machine 1 (Alias to push changes to Machine 2): 
alias sync-to-m2="rsync -avz --delete /path/to/tutorials/ user@machine2:/path/to/tutorials/"
  • On Machine 2 (Alias to push changes back to Machine 1): 
alias sync-to-m1="rsync -avz --delete /path/to/tutorials/ user@machine1:/path/to/tutorials/"

The Workflow:

  • Work on Machine 1. When done, open your terminal and run sync-to-m2.
  • Walk over to Machine 2. All your repos, branches, and commits are exactly as you left them.
  • Work on Machine 2. When done, run sync-to-m1.

Solution 2: Automated Background Syncing via Syncthing (Hands-Free)

If you don't want to remember to type a command every time you switch computers, use Syncthing.

Syncthing is a free, open-source, peer-to-peer file synchronization program. You install it on both machines, point it at your enclosing "Tutorials" directory, and it handles the rest in the background.

Why it fits your workflow perfectly:

  • It copies the hidden .git folders: Because it mirrors everything perfectly, your Git timelines, commits, and detached states are preserved identically on both sides.
  • Real-time bidirectional sync: If you edit a file or make a commit on Machine 1, it silently propagates to Machine 2 within seconds over your local network.
  • No per-repo setup: You only set it up once for the parent enclosing directory. Any new tutorial project you create inside that folder is automatically synced instantly.
⚠️ One Rule for Both Solutions: Because your tutorial repos use a --detach state to view different stages, make sure you don't leave different commits checked out on both machines simultaneously while editing files. Always let the sync complete so both machines are looking at the same state before you start making changes!
