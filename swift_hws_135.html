Hacking With Swift - 135 - Sending and receiving Codable data with URLSession and SwiftUI

iOS gives us built-in tools for sending and receiving data from the internet, and if we combine it with Codable support then it’s possible to convert Swift objects to JSON for sending, then receive back JSON to be converted back to Swift objects. Even better, when the request completes we can immediately assign its data to properties in SwiftUI views, causing our user interface to update.

To demonstrate this we can load some example music JSON data from Apple’s iTunes API, and show it all in a SwiftUI List. Apple’s data includes lots of information, but we’re going to whittle it down to just two types: a Result will store a track ID, its name, and the album it belongs to, and a Response will store an array of results.

So, start with this code: 

// To store track data
struct Result: Codable {
    var trackId: Int
    var trackName: String
    var collectionName: String
}

// For loadData() later
struct Response: Codable {
    var results: [Result]
}

We can now write a simple ContentView that shows an array of results: 

struct ContentView: View {
    //@State private var results = [Result]()
    @State private var foundTracks = [Result]() 

    var body: some View {
        //List(results, id: \.trackId) { item in
        List(foundTracks, id: \.trackId) { track in
            VStack(alignment: .leading) {
                //Text(item.trackName)
                Text(track.trackName)
                    .font(.headline)
                Text(item.collectionName)
            }
        }
    }
}

I renamed results here, as using the same name for two different contexts is making the code unintuitive. I renamed item to track to keep the context more explanatory.

Here is a quick cheat sheet for what can change.

Component Can I rename it? Rule
@State var foundTracks YES This is your private variable. Call it whatever you like!
struct Result property names NO (mostly) trackName must match the JSON key exactly (unless you use CodingKeys).
struct Response property names NO (mostly) results must match the JSON key exactly (unless you use CodingKeys).
{ track in ... } YES This is just a nickname for "the current item in the loop."

AI Interlude

List has a built-in ForEach hidden inside its primary initializer.

In SwiftUI, List is a "smart" container. It was designed to handle data collections so frequently that Apple gave it a shortcut.

1. The Two Ways to Use List

You can think of List as having two "modes" of operation:

Mode A: The "Dynamic List" (The one in your code)

When you pass an array directly into the List parentheses List(results, id: \.trackId) - you are using an initializer that says: "I am going to provide the data and the ID upfront; please loop through this for me."

Mode B: The "Static/Mixed List" (The one requiring ForEach)

If you want a list that has a header, or some buttons at the top followed by data, you use List as a container: 

List {
    Text("My Music") // A static header
    ForEach(results, id: \.trackId) { item in
        Text(item.trackName) // Dynamic rows
    }
    Button("Add More") { } // A static footer
}

2. Why use one over the other?

  • Use your version (the shortcut): When the entire list is just a repetition of the same data array. It's cleaner, shorter, and more readable.
  • Use ForEach inside a List: When you need a "heterogeneous" list - meaning a mix of different types of views, sections, or static buttons alongside your data.

3. Under the Hood: The "Item" Closure

In your code:

  • List(results, id: \.trackId) { item in ... }

The { item in ... } part is a trailing closure. Because you provided the results array at the start, List knows it needs to run that block of code for every single Result in that array. It handles the "for each" logic behind the scenes and hands you one item at a time to design your row.

4. A Note on onDelete

One practical reason developers often switch back to the ForEach version is for Swipe to Delete. The .onDelete modifier only works on a ForEach.

So, if you ever decide your Habit Tracker or Music App needs to let users swipe to remove an item, you will have to "unpack" your current List and wrap your items in a ForEach like this: 

List {
    ForEach(results, id: \.trackId) { item in
        // row content
    }
    .onDelete(perform: removeRows)
}

Summary for your Guide

  • List(data) { ... } is a convenience initializer that includes a built-in loop.
  • List { ForEach(data) { ... } } is the manual version required for complex layouts or swipe-to-delete functionality."

Back to the tutorial

That won’t show anything at first, because the results array is empty. This is where our networking call comes in: we’re going to ask the iTunes API to send us a list of all the songs by Taylor Swift, then use JSONDecoder to convert those results into an array of Result instances.

async & await

However, doing this means you need to meet two important Swift keywords: async and await.

You see, any iPhone capable of running SwiftUI can perform billions of operations every second - it’s so fast that it completes most work before we even realized it started it. On the flip side, networking - downloading data from the internet - might take several hundreds milliseconds, or more, to arrive, which is extremely slow for a computer that’s used to doing literally a billion other things during that time.

Rather than forcing our entire progress to stop while the networking happens, Swift gives us the ability to say “this work will take some time, so please wait for it to complete while the rest of the app carries on running as usual.”

async - Asynchronous Function

This functionality - this ability to leave some code running while our main app code carries on working - is called an asynchronous function. A synchronous function is one that runs fully before returning a value as needed, but an asynchronous function is one that is able to go to sleep for a while, so that it can wait for some other work to complete before continuing. In our case, that means going to sleep while our networking code happens, so that the rest of our app doesn’t freeze up for several seconds.

To make this easier to understand, let’s write it in a few stages. First, here’s the basic method stub - please add this to the ContentView struct: 

func loadData() async {

}

Notice the new async keyword in there - we’re telling Swift this function might want to go to sleep in order to complete its work.

We want that to be run as soon as our List is shown, but we can’t just use onAppear() here because that doesn’t know how to handle sleeping functions - it expects its function to be synchronous.

onAppear() vs task()

SwiftUI provides a different modifier for these kinds of tasks, giving it a particularly easy to remember name: task().

await Keyword

This can call functions that might go to sleep for a while; all Swift asks us to do is mark those functions with a second keyword, await, so we’re explicitly acknowledging that a sleep might happen.

Add this modifier to the List now: 

.task {
    await loadData()
}

Tip: Think of await as being like try - we’re saying we understand a sleep might happen, in the same way try says we acknowledge an error might be thrown.

Inside loadData() we have three steps we need to complete:

  • Creating the URL we want to read.
  • Fetching the data for that URL.
  • Decoding the result of that data into a Response struct.

We’ll add those step by step, starting with the URL.

Step 1 is to set the URL. This needs to have a precise format: “itunes.apple.com” followed by a series of parameters.

You can find the full set of parameters if you do a web search for “iTunes Search API”. In our case we’ll be using the search term “Taylor Swift” and the entity “song”, so add this to loadData() now: 

guard let url = URL(string: "https://itunes.apple.com/search?term=taylor+swift&entity=song") else {
    print("Invalid URL")
    return
}

Step 2 is to fetch the data from that URL, which is where our 'sleep' is likely to happen.

I say “likely” because it might not. iOS will do a little caching of data, so if the URL is fetched twice back-to-back then the data will get sent back immediately rather than triggering a sleep.

Regardless, a sleep is possible here, and every time a sleep is possible we need to use the await keyword with the code we want to run. Just as importantly, an error might also be thrown here - maybe the user isn’t currently connected to the internet, for example.

So, we need to use both try and await at the same time. Please add this code directly after the previous code: 

do {
    let (data, _) = try await URLSession.shared.data(from: url)

    // more code to come
} catch {
    print("Invalid data")
}

That introduced three important things, so let’s break it down:

  • Our work is being done by the data(from:) method, which takes a URL and returns the Data object at that URL. This method belongs to the URLSession class, which you can create and configure by hand if you want, but you can also use a shared instance that comes with sensible defaults.
  • The return value from data(from:) is a tuple containing the data at the URL and some metadata describing how the request went. We don’t use the metadata, but we do want the URL’s data, hence the underscore - we create a new local constant for the data, and toss the metadata away.
  • When using both try and await at the same time, we must write try await - using await try is not allowed. There’s no special reason for this, but they had to pick one so they went with the one that reads more naturally.

So, if our download succeeds our data constant will be set to whatever data was sent back from the URL, but if it fails for any reason our code prints “Invalid data” and does nothing else.

The last part of this method is to convert the Data object into a Response object using JSONDecoder, then assign the array inside to our results property (remember - results is an instance of our Results struct).

This is exactly what we’ve used before, so this shouldn’t be a surprise - add this last code in place of the // more code to come comment now: 

if let decodedResponse = try? JSONDecoder().decode(Response.self, from: data) {
    results = decodedResponse.results
}

If you run the code now you should see a list of Taylor Swift songs appear after a short pause - it really isn’t a lot of code given how well the end result works.

All this only handles downloading data. Later on in this project we’re going to look at how to adopt a slightly different approach so you can send Codable data, but that’s enough for now.

The Visual Flow

If you rename everything for maximum clarity, the "journey" of a piece of data (like a song title) looks like this:

  • The Internet: Sends a JSON "shipping box" labeled results.
  • The Response Struct: Acts as the "receiving dock." It must use the name results to accept the delivery.
  • The Handshake: You manually move the items from the Response dock into your private foundTracks storage.

The View: The List watches the foundTracks storage. The moment it sees a new song arrive, it automatically draws a row.

Why "item" and "results" are so common in tutorials

Tutorials often use generic names like results and item because they want the code to be "reusable." If they used foundTracks, the code looks like it's only for music. By using results, they hope you can copy-paste it into a weather app or a stock market app.

But for learning, that generic naming is a trap! Using specific names like foundTracks or habitList makes the logic "sticky" in your brain.

// Individual data struct
struct Result: Codable {
    var trackId: Int
    var trackName: String
    var collectionName: String
}

// Array of 'Result' objects
struct Response: Codable {
    var results: [Result]
}


struct ContentView: View {
    // Set up an array of Result objects
    //@State private var results = [Result]()
    @State private var foundTracks = [Result]()
    
    var body: some View {
        // This doesn't need a ForEach loop for the results array
        //List(results, id: \.trackId) { item in
        List(foundTracks, id: \.trackId) { track in
            VStack(alignment: .leading) {
                Text(track.trackName)
                    .font(.headline)
                Text(track.collectionName)
            }
        }
        .task {
            await fetchMusic()
        }
        
    }
    
    //func loadData() async {
    func fetchMusic() async {
        // 1. Check the URL
        guard let url = URL(string: "https://itunes.apple.com/search?term=taylor+swift&entity=song") else {
            print("Invalid URL")
            return
        }

            // 2. Fetch the data

        }

More AI Insights

The line:

  • JSONDecoder().decode(Response.self, from: data) is where the Response struct is used.

The results variable inside the Response struct exists purely to act as a "landing pad" for the iTunes data.

How the Data "Moves"

Think of it as a relay race where the data changes hands (and names) to get to the screen:

  • On the Server: The data is sitting in a JSON file under the key "results".
  • During Decoding: Your Response struct catches that data. Because the JSON key is "results", your struct must have a variable named results to catch it.
  • The Handover: In your line foundTracks = decodedResponse.results, you are physically taking the array out of the "Response" container and putting it into your "foundTracks" variable.
  • On the Screen: Your List is only watching foundTracks. It has no idea the Response struct or the word results even existed.

Why this is a "Temporary" Struct

Notice that decodedResponse is created with if let. This means it only exists for a split second inside those curly braces { ... }.

  • Once the line foundTracks = decodedResponse.results finishes, the decodedResponse object is instantly destroyed by the computer to save memory.
  • The only thing that survives is the data you saved into foundTracks.

A Quick Refinement

While your code will work, there is one tiny "gotcha" with @State and async code. SwiftUI expects UI updates (like changing foundTracks) to happen on the Main Thread (the part of the computer that handles drawing the screen).

When you use await, the code might be running on a background "thread." To be safe, it's a good habit to tell Swift to jump back to the main thread for that final assignment: 

// Use MainActor to ensure the UI updates smoothly
await MainActor.run {
    foundTracks = decodedResponse.results
}

The await MainActor.run block is like a "VIP pass" that lets your code jump back to the front of the line where the user interface (UI) is managed.

The Context: Background vs. Foreground

When you use .task and await URLSession, Swift moves your code to a background thread. This is great because it keeps your app from "freezing" while waiting for the internet. However, there is a strict rule in iOS: Only the "Main Thread" (the foreground) is allowed to change the UI.

Since @State private var foundTracks is directly connected to the UI (the List), updating it from a background thread can occasionally cause glitches or purple warnings in Xcode.

Where it fits in your function

You place the MainActor block exactly at the moment you are ready to "hand over" the data to your View's variable. 

func fetchMusic() async {
    guard let url = URL(string: "https://itunes.apple.com/search?term=taylor+swift&entity=song") else {
        return
    }
    
    do {
        // 1. Still on a background thread (waiting for internet)
        let (data, _) = try await URLSession.shared.data(from: url)
        
        // 2. Still on a background thread (crunching JSON numbers)
        if let decodedResponse = try? JSONDecoder().decode(Response.self, from: data) {
            
            // 3. JUMP TO THE FOREGROUND
            await MainActor.run {
                // Now we are safely back on the Main Thread to update the UI
                foundTracks = decodedResponse.results
            }
        }
    } catch {
        print("Invalid data")
    }
}

How it relates to .task

Think of it as a journey with a specific start and end:

  • .task: This is the trigger. It tells the app: "Start this work as soon as the screen appears, and do it in the background so the user can still scroll."
  • await URLSession: This is the heavy lifting. It's done away from the main UI so the app stays responsive.
  • await MainActor.run: This is the completion. It says: "The heavy work is done; let me quickly hop back to the main UI thread to show the user what I found."

A Modern Shortcut

If you want to make your entire class or function run on the Main Actor by default (which is very common for Observable classes or Views), you can actually just put @MainActor at the very top of your function: 

@MainActor // This ensures EVERY line in this function happens on the UI thread
func fetchMusic() async {
    // ... code ...
    foundTracks = decodedResponse.results // No 'MainActor.run' block needed!
}

However, using the manual await MainActor.run block as I showed above is technically more efficient because it only uses the "expensive" UI thread for the one second it takes to update the variable, leaving the background thread to handle the internet and decoding.

Summary of your Logic

  • Response: The "Shipping Manifest" (must match iTunes).
  • Response.results: The "Cargo" inside the ship.
  • foundTracks: Your "Local Warehouse" where you store the cargo to show the customers.
