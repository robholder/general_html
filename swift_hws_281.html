Hacking With Swift - 281 - Working with two side by side views in SwiftUI

You’re already familiar with the basic usage of NavigationStack, which allows us to create views like this one: 

import SwiftUI

struct ContentView: View {
    var body: some View {
        NavigationStack {
            Text("Hello, world!")
                .navigationTitle("Primary")
        }
    }
}

That works great on iPhone, but on iPad it's less than ideal - there we have much larger screens to deal with, and having one large view for navigation is a poor use of the space, and also creates quite a jarring experience when new views slide in all the way across.

NavigationView

NOTE: This is not from the tutorial; this is what I've found elsewhere (see below for full details).

NavigationView was the original navigation container (from 2019), but it's notoriously buggy and it has been deprecated. You should only use NavigationView if you are maintaining older legacy codebases. 

struct ContentView: View {
    var body: some View {
        NavigationView {
            Text("Hello, world!")
                .navigationTitle("Primary")
        }
    }
}

NavigationSplitView

In these situations, it's a better idea to use a different view for navigation, called NavigationSplitView. This lets us specify either two or three columns of data to show side by side, and iPadOS takes care of showing or hiding them at the right times depending on the exact configuration.

Try it out with this simple code: 

NavigationSplitView {
    Text("Primary")
} detail: {
    Text("Content")
}

When you launch the app what you see once again depends on your device and orientation:

  • On iPhone you'll see Primary.
  • On iPad in landscape you'll see Primary in a sidebar along the leading edge of your device, and Content filling the rest of the screen.
  • On iPad in portrait mode you'll see Content filling the screen.
NOTE: I see Primary and Content regardless of orientation on the iPad simulator.

iPadOS does two very clever things here.

First, regardless of your device orientation you'll see a button that shows or hides the primary view, so users can choose what layout they want.

Second, when you activate multi-tasking - if you bring a Safari window alongside, for example - you'll see our app's views adapt so that the Primary view is hidden, because the system recognizes the amount of free space has decreased.

Linking Content to Primary

SwiftUI automatically links the primary and secondary views, which means if you have a NavigationLink in the primary view it will automatically load its content in the secondary view: 

NavigationSplitView {
    NavigationLink("Primary") {
        Text("New view")
    }
} detail: {
    Text("Content")
}

There are some ways you can customize this split view's behavior.

For example, you can tell iOS to prefer to keep the primary view around when space is partially limited like this: 

NavigationSplitView(columnVisibility: .constant(.all)) {
    NavigationLink("Primary") {
        Text("New view")
    }
} detail: {
    Text("Content")
        .navigationTitle("Content View")
}
.navigationSplitViewStyle(.balanced)

That requests that all columns be shown in a balance style, and the result is that iPads in portrait mode will now show the primary view.

Tip: columnVisibility is actually provided as a binding, so you could store your option in some state and update it dynamically.

Second, you can tell the system to prefer the detail view by default, which is helpful on iPhone where the primary view is selected as standard: 

NavigationSplitView(preferredCompactColumn: .constant(.detail)) {
    NavigationLink("Primary") {
        Text("New view")
    }
} detail: {
    Text("Content")
        .navigationTitle("Content View")
}
.navigationSplitViewStyle(.balanced)

Again, you can use that with a binding, so you can change it as your app runs if you want.

Finally, although you can use .toolbar(.hidden, for: .navigationBar) to hide the toolbar in your detail view, be careful because it will hide the button to toggle the sidebar! 

NavigationSplitView {
    NavigationLink("Primary") {
        Text("New view")
    }
} detail: {
    Text("Content")
        .toolbar(.hidden, for: .navigationBar)
}
.navigationSplitViewStyle(.balanced)

Tip: You can even add a third view to NavigationSplitView, which lets you create a sidebar. You’ll see these in apps such as Notes, where you can navigate up from from the list of notes to browse note folders. So, navigation links in the first view control the second view, and navigation links in the second view control the third view - it’s an extra level of organization for times when you need it.

Ai Insight

In SwiftUI, the way we handle moving between screens has evolved significantly. NavigationView, NavigationStack, and NavigationSplitView represent different generations and use cases of SwiftUI's navigation system.

Here is the breakdown of what they are and when to use them.

1. NavigationView (The Legacy Container)

NavigationView was the original navigation container introduced when SwiftUI debuted in 2019.

  • What it does: It provides the top navigation bar, handles pushing and popping views on a stack, and automatically adapts to different device sizes (e.g., showing a sidebar on iPad and a single-screen stack on iPhone).
  • The Problem: It was notorious for being buggy, difficult to control programmatically (like deep-linking or resetting to the root view), and trying to do too much at once by mixing single-column and multi-column layouts.
  • Current Status: Deprecated. Apple split its functionality into two dedicated containers: NavigationStack and NavigationSplitView. You should only use NavigationView if you are maintaining older legacy codebases.

2. NavigationStack (The Modern Single-Column Standard)

Introduced in iOS 16, NavigationStack completely replaced NavigationView for single-column layouts (like standard iPhone apps).

  • What it does: It manages a stack of views where only the top-most view is visible.
  • Why it's better: It introduced data-driven navigation. Instead of binding navigation directly to hardcoded destination views, you can bind the stack to a collection of data (e.g., an array of Product objects). Pushing data onto the array pushes a view onto the screen; clearing the array takes the user back to the home screen instantly.

3. NavigationSplitView (The Multi-Column Powerhouse)

Also introduced in iOS 16, NavigationSplitView is designed specifically for multi-column layouts. It is the go-to choice for iPadOS, macOS, and visionOS apps, though it gracefully collapses into a single column on iPhone.

  • What it does: It organizes your app into two or three distinct vertical columns:
    • Sidebar: The primary navigation (e.g., a list of folders or categories).
    • Content (Optional): A supplementary list based on the sidebar selection (e.g., a list of emails inside the selected folder).
    • Detail: The main area where the actual content is viewed (e.g., the selected email body).
  • How it works: It uses state bindings so that selecting an item in one column automatically updates what is displayed in the next column.

Quick Comparison Table

Feature NavigationView (Legacy) NavigationStack (Modern) NavigationSplitView (Modern)
Primary Use Case Older apps / iOS 15 & lower iPhone apps / Single-column stacks iPad & Mac apps / Multi-column layouts
Layout Style Automatic (Unpredictable) Always a single-column stack 2 or 3 adaptive columns
Programmatic Control Poor / Fragile Excellent (Data-driven) Excellent (State-driven selection)
Status Deprecated Active & Recommended Active & Recommended

Summary Rule of Thumb

If you are building a modern SwiftUI app today, never use NavigationView. Instead, ask yourself: Am I building a multi-column layout for iPad/Mac (NavigationSplitView), or a standard phone-style stack (NavigationStack)?
