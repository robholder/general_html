Hacking With Swift - 192 - Writing data to the documents directory

Previously we looked at how to read and write data to UserDefaults, which works great for user settings or small amounts of JSON, and we also looked at SwiftData, which is a great choice for times when you want relationships between objects, or more advanced sorting and filtering.

In this app we’re going to look at a middle ground: we'll just write our data to a file directly. This isn't because I hate SwiftData, and in fact I think SwiftData would make a good choice here.

Instead, it's so that I can show you the full spread of what's possible in iOS development, because there are lots of apps you'll work on that use exactly this approach to saving their data - it's good that you can at least see how it works.

That being said, using UserDefaults is definitely a bad idea here, because there's no limit to how much data users can create in the app. UserDefaults is better used for simple settings and similar.

Fortunately, iOS makes it very easy to read and write data from device storage, and in fact all apps get a directory for storing any kind of documents we want. Files here are automatically synchronized with iCloud backups, so if the user gets a new device then our data will be restored along with all the other system data - we don’t even need to think about it.

There is a small catch - isn’t there always? - and it’s that all iOS apps are sandboxed, which means they run in their own container with a hard to guess directory name. As a result, we can’t - and shouldn’t try to - guess the directory where our app is installed, and instead need to rely on a special URL that points to our app’s documents directory: 

print(URL.documentsDirectory)

That documents directory is ours to do with as we please, and because it belongs to the app it will automatically get deleted if the app itself gets deleted. Other than physical device limitations there is no limit to how much we can store, although remember that users can use the Settings app to see how much storage your app takes up - be respectful!

Now that we have a directory to work with, we can read and write files there freely. You already met String(contentsOf:) and Data(contentsOf:) for reading data, but for writing data we need to use the write(to:) method. This takes two parameters:

  • A URL to write to.
  • Any additional options to use when saving.

The first of those can be created by combining the documents directory URL with a filename, such as myfile.txt.

As for the second, I prefer to give it two values as an array:

  • .atomic and
  • .completeFileProtection.

They do very different things, but both are important:

.atomic

  • Asking for atomic saving means the whole file should be written in one lump. If this isn't included and we try to write a big file, it’s possible that another part of our app might try and read the file while it’s still being written. This shouldn’t cause a crash or anything, but it does mean that it’s going to read only part of the data, because the other part hasn’t been written yet. Atomic writing causes the system to write our full file to a temporary filename (not the one we asked for), and when that’s finished it does a simple rename to our target filename. This means either the whole file is there or nothing is.

.completeFileProtection

  • Asking for complete file protection means iOS automatically (and transparently) encrypts the file, and only allows our app to read it while the device is unlocked. iOS does a great job of keeping user data secure, but there's no harm being extra safe!

To put all this code into action, we’re going to modify the default code from our template so that it writes a test string to a file in the documents directory, reads it back into a new string, then prints it out - the complete cycle of reading and writing data.

Change the body property of ContentView to this: 

Button("Read and Write") {
    let data = Data("Test Message".utf8)
    let url = URL.documentsDirectory.appending(path: "message.txt")

    do {
        try data.write(to: url, options: [.atomic, .completeFileProtection])
        let input = try String(contentsOf: url, encoding: .utf8)
        print(input)
    } catch {
        print(error.localizedDescription)
    }
}

When that runs you should be able to tap the label to see “Test message” printed to Xcode’s debug output area.

In Context

import SwiftUI

struct ContentView: View {
    var body: some View {
        Button("Read and Write") {
            // Create our example data
            let data = Data("Test Message".utf8)
            // point to app directory and append our file name
            let url = URL.documentsDirectory.appending(path: "message.txt")
            
            do {
                // Write data to our file path
                try data.write(to: url,
                    options: [
                        // write in one go
                        .atomic,
                        // offer extra encryption protection
                        .completeFileProtection
                    ],
                )
                // load in data from file path
                let input = try String(contentsOf: url, encoding: .utf8)
                // print the data to the console
                print(input)
            } catch {
                print(error.localizedDescription)
            }
        }
    }
}

Challenge

Before we move on, here’s a small challenge for you:

Back in project 8 we looked at how to create a generic extension on Bundle that let us find, load, and decode any Codable data from our app bundle.

Can you write something similar for the documents directory, perhaps making it an extension on FileManager?

Ai Insight

This is a massive milestone in an iOS developer's journey. You are taking a highly advanced concept - Generics combined with Protocols - and translating it from one system domain to another.

Before we write the code, let’s demystify what FileManager actually is, because once you understand its purpose, the code will make total sense.

What is FileManager vs. Bundle?

Think of your app like a boxed video game console:

  • Bundle is the Game Disc: It represents your app's main installation package. Everything in your Bundle (like the JSON files for your Astronauts and Missions) was packed tightly at the factory. It is Read-Only. Your app can read files from the Bundle, but it can never write new data back into it.
  • FileManager is the Memory Card: This is iOS's gateway to the device's actual file storage. It handles the "Documents Directory" - a private sandbox folder where your app is allowed to dynamically create, write, edit, and delete files while the user is actively playing the game.

Your previous project used a Bundle extension to read factory-set JSON data. Your current challenge is to use a FileManager extension to read and write user-generated JSON data to that "Memory Card".

The Goal: Universal FileManager Extensions

Right now, your ContentView is manually handling the URL, writing raw Data, and converting it to a string.

We want to build two generic extensions on FileManager:

  • encode(...): Take any Codable struct, turn it into JSON, and write it to the documents directory.
  • decode(...): Go to the documents directory, grab a file, and instantly turn it back into your struct.

Here is how you can write that new extension file (you can name it something like FileManager-Documents.swift): 

import Foundation

extension FileManager {
    
    // 1. THE WRITER: Takes any Codable object and saves it as a file
    func encode<T: Codable>(_ data: T, to file: String) {
        // Find the documents directory URL
        let url = URL.documentsDirectory.appending(path: file)
        
        let encoder = JSONEncoder()
        // Optional: Makes the saved JSON look pretty if you open the file manually
        encoder.outputFormatting = .prettyPrinted 
        
        do {
            // Convert the Swift struct into raw JSON Data
            let encodedData = try encoder.encode(data)
            
            // Write it to disk with atomic writing and encryption
            try encodedData.write(to: url, options: [.atomic, .completeFileProtection])
        } catch {
            fatalError("Failed to encode and save \(file): \(error.localizedDescription)")
        }
    }
    
    // 2. THE READER: Grabs a file and decodes it back into a Swift struct
    func decode<T: Codable>(_ file: String) -> T {
        // Find the file in the documents directory
        let url = URL.documentsDirectory.appending(path: file)
        
        // Try to load the raw data out of that file
        guard let data = try? Data(contentsOf: url) else {
            fatalError("Failed to load \(file) from Documents Directory.")
        }
        
        let decoder = JSONDecoder()
        
        // Map the same detailed error handling you used in your Bundle project
        do {
            return try decoder.decode(T.self, from: data)
        } catch DecodingError.keyNotFound(let key, let context) {
            fatalError("Failed to decode \(file) due to missing key '\(key.stringValue)' - \(context.debugDescription)")
        } catch DecodingError.typeMismatch(_, let context) {
            fatalError("Failed to decode \(file) due to type mismatch - \(context.debugDescription)")
        } catch DecodingError.valueNotFound(let type, let context) {
            fatalError("Failed to decode \(file) due to missing \(type) value - \(context.debugDescription)")
        } catch DecodingError.dataCorrupted(_) {
            fatalError("Failed to decode \(file) because it appears to be invalid JSON.")
        } catch {
            fatalError("Failed to decode \(file): \(error.localizedDescription)")
        }
    }
}

How this cleans up your ContentView

Look how incredibly simple your ContentView button becomes once the FileManager extension is doing the heavy lifting under the hood.

Instead of dealing with URLs, data arrays, and encoding types manually, you just talk to FileManager.default: 

struct ContentView: View {
    var body: some View {
        Button("Read and Write") {
            // Imagine we have a custom profile struct
            let currentMission = Mission(id: 101, launchDate: "2026-05-22", crew: [], description: "Mars Prep")
            
            // WRITE IT: Save the mission to disk as a JSON file
            FileManager.default.encode(currentMission, to: "savedMission.json")
            
            print("Saved successfully!")
            
            // READ IT: Instantly read it back out as a Mission struct
            let loadedMission: Mission = FileManager.default.decode("savedMission.json")
            
            print("Loaded mission ID: \(loadedMission.id)")
        }
    }
}

Notice the Magic Word: FileManager.default

FileManager utilizes a shared instance that lives throughout your app, accessible by calling FileManager.default. By adding your methods to the FileManager extension, you can invoke your clean .encode() and .decode() tools from anywhere inside your codebase.
