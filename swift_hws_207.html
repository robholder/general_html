Hacking With Swift - 207 - Challenge 3

Create another view model, this time for EditView. What you put in the view model is down to you, but I would recommend leaving dismiss and onSave in the view itself - the former uses the environment, which can only be read by the view, and the latter doesn’t really add anything when moved into the model.

Tip: This last challenge will require you to make a State instance in your EditView initializer - remember to use an underscore with the property name!

QUESTION

ANSWER

Don’t worry! This challenge trips up almost everyone the first time they read it. Paul Hudson is introducing a massive architectural concept here, but the phrasing can feel like a riddle.

Let's break down exactly what the goal is, why he wants you to do it, and what that tricky hint about the underscore actually means.

1. What is the goal here?

Right now, your EditView manages its own internal screen state directly using @State variables inside the view (things like var name: String, var description: String, etc.).

The goal of this challenge is to evict all that data logic out of the View and move it into its own dedicated ViewModel, just like you did for ContentView.

Your new file structure will look like this:

  • EditView.swift: Only handles layout elements (TextFields, Buttons, Forms).
  • EditView-ViewModel.swift: Handles holding the raw strings, fetching local coordinate details, or talking to servers.

2. What belongs inside the EditView.ViewModel?

Paul mentions that dismiss and onSave should stay in the view. That leaves the following data to be moved into your new ViewModel:

  • The name string field.
  • The description string field.
  • The original Location data that you are editing.
  • A way to package the changes up when the user clicks save.

3. The Mystery Hint: "Use an underscore with the property name!"

This is the part that causes the most confusion. Let’s look at why he says this.

When ContentView calls your edit sheet, it passes in a location: 

EditView(location: place) { ... }

Currently, your EditView takes that location in its initializer and sets up its local state strings. But now, EditView needs to initialize a ViewModel instead.

Normally, you create a ViewModel like this: 

@State private var viewModel = ViewModel()

But you can't do that here because your EditView.ViewModel needs to know which Location it is editing before it can set up the name and description fields! It can't just start completely empty.

To pass data directly into a @State object during an initializer, Swift requires a very specific syntax using an underscore (_). The underscore lets you bypass the wrapper property and talk directly to the underlying State storage container system.

How to implement the solution

Let's build it step-by-step so you can see how it works.

Step 1: Create EditView-ViewModel.swift

Create a new file and build a custom nested ViewModel extension for EditView using the @Observable macro. We will move the LoadingState enum, your text states, the Wikipedia networking function, and the location creation helper into this class: 

import Foundation

extension EditView {
    @Observable
    class ViewModel {
        enum LoadingState {
            case loading, loaded, failed
        }
        
        // 1. Core properties migrated from the View
        var name: String
        var description: String
        let location: Location
        
        var loadingState = LoadingState.loading
        var pages = [Page]()
        
        // 2. Initializer captures the raw location and populates fields
        init(location: Location) {
            self.location = location
            self.name = location.name
            self.description = location.description
        }
        
        // 3. Helper to build the customized location payload for saving
        func createUpdatedLocation() -> Location {
            var newLocation = location
            newLocation.id = UUID() // Matches your original replacement behavior
            newLocation.name = name
            newLocation.description = description
            return newLocation
        }
        
        // 4. Async Networking moved entirely to background domain logic
        func fetchNearbyPlaces() async {
            let urlString = "https://en.wikipedia.org/w/api.php?ggscoord=\(location.latitude)%7C\(location.longitude)&action=query&prop=coordinates%7Cpageimages%7Cpageterms&colimit=50&piprop=thumbnail&pithumbsize=500&pilimit=50&wbptterms=description&generator=geosearch&ggsradius=10000&ggslimit=50&format=json"
            
            guard let url = URL(string: urlString) else {
                print("Bad URL: \(urlString)")
                return
            }
            
            do {
                let (data, _) = try await URLSession.shared.data(from: url)
                let items = try JSONDecoder().decode(Result.self, from: data)
                
                // Keep the UI update safe on the main thread loop
                await MainActor.run {
                    self.pages = items.query.pages.values.sorted()
                    self.loadingState = .loaded
                }
            } catch {
                await MainActor.run {
                    self.loadingState = .failed
                }
            }
        }
    }
}

Step 2: Update EditView.swift

Now, open your existing view layout file and swap out your old individual @State properties for this new single ViewModel using the underscore initializer syntax. It only acts as a wrapper that displays what the viewModel provides: 

import SwiftUI

struct EditView: View {
    @Environment(\.dismiss) var dismiss
    
    // 1. One single State container managing our new ViewModel
    @State private var viewModel: ViewModel
    var onSave: (Location) -> Void
    
    // 2. Custom Initializer implementing Paul's underscore (_) State trick
    init(location: Location, onSave: @escaping (Location) -> Void) {
        self.onSave = onSave
        self._viewModel = State(initialValue: ViewModel(location: location))
    }
    
    var body: some View {
        NavigationStack {
            Form {
                Section {
                    // 3. Bind text input fields directly to the ViewModel variables
                    TextField("Place name", text: $viewModel.name)
                    TextField("Description", text: $viewModel.description)
                }
                
                Section("Nearby…") {
                    // 4. Read matching cases out of the ViewModel state engine
                    switch viewModel.loadingState {
                    case .loaded:
                        ForEach(viewModel.pages, id: \.pageid) { page in
                            Text("**\(page.title)**: *\(page.description)*")
                        }
                    case .loading:
                        Text("Loading…")
                    case .failed:
                        Text("Please try again later.")
                    }
                }
            }
            .navigationTitle("Place Details")
            .toolbar {
                Button("Save") {
                    // 5. Ask the view model to build the new structural copy
                    let updatedLocation = viewModel.createUpdatedLocation()
                    onSave(updatedLocation)
                    dismiss()
                }
            }
        }
        .task {
            // 6. Trigger the background networking fetch task directly
            await viewModel.fetchNearbyPlaces()
        }
    }
}

Summary of why this is important

By doing this, you've completed a clean MVVM (Model-View-ViewModel) design pattern. Your view files no longer care how data is handled or created; they simply draw the layout on screen and look at the ViewModel for what text should be shown!

What makes this implementation work perfectly:

  • await MainActor.run { ... }: Because ViewModel is a plain background class using async/await, modifying properties like pages or loadingState must explicitly happen on the @MainActor thread so SwiftUI knows to instantly redraw the screen.
  • The _viewModel setup: By passing your custom ViewModel(location: location) directly inside the structural wrapper State(initialValue:), you ensure the network call instantly uses the correct coordinates when .task starts running.
