Hacking With Swift - 207 - Challenge 3

Create another view model, this time for EditView. What you put in the view model is down to you, but I would recommend leaving dismiss and onSave in the view itself - the former uses the environment, which can only be read by the view, and the latter doesn’t really add anything when moved into the model.

Tip: This last challenge will require you to make a State instance in your EditView initializer - remember to use an underscore with the property name!

Ai Insight

QUESTION

The next challenge has me utterly confused. I don't even know what the goal is. I just don't understand the question/challenge:

ANSWER

Don’t worry! This challenge trips up almost everyone the first time they read it. Paul Hudson is introducing a massive architectural concept here, but the phrasing can feel like a riddle.

Let's break down exactly what the goal is, why he wants you to do it, and what that tricky hint about the underscore actually means.

1. What is the goal here?

Right now, your EditView manages its own internal screen state directly using @State variables inside the view (things like var name: String, var description: String, etc.).

The goal of this challenge is to evict all that data logic out of the View and move it into its own dedicated ViewModel, just like you did for ContentView.

Your new file structure will look like this:

  • EditView.swift: Only handles layout elements (TextFields, Buttons, Forms).
  • EditView-ViewModel.swift: Handles holding the raw strings, fetching local coordinate details, or talking to servers.

2. What belongs inside the EditView.ViewModel?

Paul mentions that dismiss and onSave should stay in the view. That leaves the following data to be moved into your new ViewModel:

  • The name string field.
  • The description string field.
  • The original Location data that you are editing.
  • A way to package the changes up when the user clicks save.

3. The Mystery Hint: "Use an underscore with the property name!"

This is the part that causes the most confusion. Let’s look at why he says this.

When ContentView calls your edit sheet, it passes in a location: 

EditView(location: place) { ... }

Currently, your EditView takes that location in its initializer and sets up its local state strings. But now, EditView needs to initialize a ViewModel instead.

Normally, you create a ViewModel like this: 

@State private var viewModel = ViewModel()

But you can't do that here because your EditView.ViewModel needs to know which Location it is editing before it can set up the name and description fields! It can't just start completely empty.

To pass data directly into a @State object during an initializer, Swift requires a very specific syntax using an underscore (_). The underscore lets you bypass the wrapper property and talk directly to the underlying State storage container system.

How to implement the solution

Let's build it step-by-step so you can see how it works.

Step 1: Create EditView-ViewModel.swift

Create a new file and build a custom nested ViewModel extension for EditView using the @Observable macro. We will move the LoadingState enum, your text states, the Wikipedia networking function, and the location creation helper into this class: 

import Foundation

extension EditView {
    @Observable
    class ViewModel {
        enum LoadingState {
            case loading, loaded, failed
        }
        
        // 1. Core properties migrated from the View
        var name: String
        var description: String
        let location: Location
        
        var loadingState = LoadingState.loading
        var pages = [Page]()
        
        // 2. Initializer captures the raw location and populates fields
        init(location: Location) {
            self.location = location
            self.name = location.name
            self.description = location.description
        }
        
        // 3. Helper to build the customized location payload for saving
        func createUpdatedLocation() -> Location {
            var newLocation = location
            newLocation.id = UUID() // Matches your original replacement behavior
            newLocation.name = name
            newLocation.description = description
            return newLocation
        }
        
        // 4. Async Networking moved entirely to background domain logic
        func fetchNearbyPlaces() async {
            let urlString = "https://en.wikipedia.org/w/api.php?ggscoord=\(location.latitude)%7C\(location.longitude)&action=query&prop=coordinates%7Cpageimages%7Cpageterms&colimit=50&piprop=thumbnail&pithumbsize=500&pilimit=50&wbptterms=description&generator=geosearch&ggsradius=10000&ggslimit=50&format=json"
            
            guard let url = URL(string: urlString) else {
                print("Bad URL: \(urlString)")
                return
            }
            
            do {
                let (data, _) = try await URLSession.shared.data(from: url)
                let items = try JSONDecoder().decode(Result.self, from: data)
                
                // Keep the UI update safe on the main thread loop
                await MainActor.run {
                    self.pages = items.query.pages.values.sorted()
                    self.loadingState = .loaded
                }
            } catch {
                await MainActor.run {
                    self.loadingState = .failed
                }
            }
        }
    }
}

Step 2: Update EditView.swift

Now, open your existing view layout file and swap out your old individual @State properties for this new single ViewModel using the underscore initializer syntax. It only acts as a wrapper that displays what the viewModel provides: 

import SwiftUI

struct EditView: View {
    @Environment(\.dismiss) var dismiss
    
    // 1. One single State container managing our new ViewModel
    @State private var viewModel: ViewModel
    var onSave: (Location) -> Void
    
    // 2. Custom Initializer implementing Paul's underscore (_) State trick
    init(location: Location, onSave: @escaping (Location) -> Void) {
        self.onSave = onSave
        self._viewModel = State(initialValue: ViewModel(location: location))
    }
    
    var body: some View {
        NavigationStack {
            Form {
                Section {
                    // 3. Bind text input fields directly to the ViewModel variables
                    TextField("Place name", text: $viewModel.name)
                    TextField("Description", text: $viewModel.description)
                }
                
                Section("Nearby…") {
                    // 4. Read matching cases out of the ViewModel state engine
                    switch viewModel.loadingState {
                    case .loaded:
                        ForEach(viewModel.pages, id: \.pageid) { page in
                            Text("**\(page.title)**: *\(page.description)*")
                        }
                    case .loading:
                        Text("Loading…")
                    case .failed:
                        Text("Please try again later.")
                    }
                }
            }
            .navigationTitle("Place Details")
            .toolbar {
                Button("Save") {
                    // 5. Ask the view model to build the new structural copy
                    let updatedLocation = viewModel.createUpdatedLocation()
                    onSave(updatedLocation)
                    dismiss()
                }
            }
        }
        .task {
            // 6. Trigger the background networking fetch task directly
            await viewModel.fetchNearbyPlaces()
        }
    }
}

#Preview {
    EditView(location: .example) { _ in }
}

Summary of why this is important

By doing this, you've completed a clean MVVM (Model-View-ViewModel) design pattern. Your view files no longer care how data is handled or created; they simply draw the layout on screen and look at the ViewModel for what text should be shown!

What makes this implementation work perfectly:

  • await MainActor.run { ... }: Because ViewModel is a plain background class using async/await, modifying properties like pages or loadingState must explicitly happen on the @MainActor thread so SwiftUI knows to instantly redraw the screen.
  • The _viewModel setup: By passing your custom ViewModel(location: location) directly inside the structural wrapper State(initialValue:), you ensure the network call instantly uses the correct coordinates when .task starts running.

Another Approach

It is completely normal to find this confusing! Moving from "everything in the view" to MVVM (Model-View-ViewModel) requires a shift in how you think about data ownership in SwiftUI.

Let’s break down the conceptual shift (the "why" behind the plain vars and underscores) and then map out a repeatable step-by-step process you can use every time you refactor.

1. The Big Concepts Simplified

Why do @State private var variables become plain var inside the ViewModel?

In structs, @State is a specialized wrapper that tells SwiftUI: "Hey, look after this memory for me, and if it changes, repaint the screen."

In your original view struct, SwiftUI needed to watch those variables directly.

When you move to an @Observable ViewModel, the ViewModel class itself takes over that responsibility. The @Observable macro automatically turns every plain property inside the class into something SwiftUI can track.

  • If you left @State inside the class, it wouldn’t compile or work correctly because @State is specifically designed to be owned by a struct View, not a class.

Demystifying the Underscore (_) Trick

// 1. One single State container managing our new ViewModel
@State private var viewModel: ViewModel

// 2. Custom Initializer implementing Paul's underscore (_) State trick
init(location: Location, onSave: @escaping (Location) -> Void) {
    self.onSave = onSave
    self._viewModel = State(initialValue: ViewModel(location: location))
}

Let’s lift the hood and look at what SwiftUI is actually doing behind the scenes. The confusion usually comes from a hidden piece of "magic" that Swift does when you use the @State keyword.

To understand the connection between the View, the ViewModel, the struct vs. class implications, and that weird underscore, we need to look at how Swift translates your code.

1. What @State Actually Generates

When you write this single line of code inside your struct EditView: 

@State private var viewModel: ViewModel

The Swift compiler secretly expands that line into two separate things under the hood:

  • _viewModel (The Storage Container): A hidden, actual struct of type State<ViewModel>. This is an independent object that manages memory allocation and talks to SwiftUI's rendering engine.
  • viewModel (The Trapdoor/Computed Property): A plain-looking variable name that you use in your code. Whenever you read or write to viewModel, Swift intercepts it and redirects it inside the container (_viewModel.wrappedValue).

Q: I'm not seeing where _viewModel and viewModel are in the process.

I guess:

@State private var viewModel: ViewModel

...sets up viewModel as a computed property - and is that the computed result of the EditView-ViewModel.swift code?

So what is _viewModel? A container with what in it?

A: You are hitting on the exact mechanism that makes this feel like "magic." Let's pull back the curtain entirely and map these exact terms directly to your two new files: EditView.swift (the View) and EditView-ViewModel.swift (the ViewModel).

1. Where do viewModel and _viewModel actually live?

They both live entirely inside EditView.swift. Your ViewModel file (EditView-ViewModel.swift) doesn't know they exist; it is just a plain blueprint for data.

When you write this line in EditView.swift: 

@State private var viewModel: ViewModel

Swift splits that line into two distinct entities behind the scenes. Let's look at exactly what is inside each one:

What is _viewModel? (The Container)

_viewModel is an actual structural container created by SwiftUI.

  • What is inside it? It contains a memory pointer (a reference) pointing to the ViewModel class instance living out in your heap memory. It also contains the internal SwiftUI logic needed to monitor that class.
  • Why do we need it? Because EditView is a struct, it is destroyed and recreated constantly. _viewModel is the anchor that stays attached to SwiftUI's core system, ensuring your actual data doesn't get wiped out when the view blinks.

What is viewModel? (The Computed Property/Facade)

You guessed it perfectly: viewModel acts as a computed property! It is a convenient facade or "trapdoor" that Swift gives you so you don't have to look at the ugly container math.

  • When you read viewModel.name: Swift secretly executes code that goes into the _viewModel container, follows the pointer to your class instance in EditView-ViewModel.swift, and fetches the string.
  • When you write viewModel.name = "Paris": Swift goes through the container, modifies the class instance, and because the class is @Observable, SwiftUI immediately schedules a screen refresh.

2. Tracking Them Through the Code Process

Let's look at how these two hidden variables handle the work inside your code line-by-line.

Phase 1: The Birth (init)

When your app decides to show EditView, it calls the initializer. At this exact moment, the view is a blank slate. The computed property viewModel cannot be used yet because the storage container hasn't been built. 

init(location: Location, onSave: @escaping (Location) -> Void) {
    self.onSave = onSave
    
    // 1. You allocate memory for the blueprint in EditView-ViewModel.swift:
    let newClassInstance = ViewModel(location: location)
    
    // 2. You manually build the container (_viewModel) and drop the instance inside:
    self._viewModel = State(initialValue: newClassInstance)
}

Phase 2: Rendering the UI (body)

Now the view is initialized and ready to draw. From this point forward, you never use _viewModel again. You only use the computed viewModel facade. 

TextField("Place name", text: $viewModel.name)
  • What's happening: The ＄ symbol tells SwiftUI to create a two-way binding.
  • The connection: The TextField reads from the computed viewModel.name to display the text. When the user types a letter, it writes back through the viewModel facade, altering the data inside your EditView-ViewModel.swift class.

Phase 3: Background Tasks (.task)

.task {
    await viewModel.fetchNearbyPlaces()
}
  • The connection: The view uses the computed viewModel property to call the network function located in EditView-ViewModel.swift.
  • When the network data arrives, the code inside the ViewModel updates self.pages. Because the view is watching the class through that initial _viewModel container connection, it spots the change and automatically repaints the screen with your new places.

To Say It In One Sentence:

_viewModel is the invisible physical box holding onto your data's location so it doesn't get lost, while viewModel is the convenient handle on the outside of the box that your View code pulls on to read and write data to EditView-ViewModel.swift.

2. Why the Underscore (_) is Required in init

In SwiftUI, a View is a struct. Structs are value types, meaning they are destroyed and recreated constantly whenever the screen updates.

Because the View struct is temporary, it cannot safely hold onto data. If it did, your data would reset to its initial value every time the view re-renders!

To solve this, SwiftUI holds your data in an external, permanent memory space. The State container (_viewModel) is the bridge that connects your temporary View to that permanent memory.

The Initialization Timeline Dilemma

When your View’s init() is running, the View struct is currently being born. Because of Swift’s strict safety rules:

  • You cannot use the plain viewModel trapdoor yet, because the underlying State storage container hasn't been fully set up or connected to SwiftUI's memory graph.
  • The only way to set things up is to create and initialize the storage container first.

By writing self._viewModel = ..., you are bypassing the trapdoor and directly initializing the hidden State container struct, handing it your brand-new ViewModel class. 

init(location: Location, onSave: @escaping (Location) -> Void) {
    self.onSave = onSave
    
    // You are building the container box (_viewModel) 
    // and putting your ViewModel class instance inside it.
    self._viewModel = State(initialValue: ViewModel(location: location))
}

3. The struct vs. class Connection

This brings us to why your ViewModel must be a class (marked with @Observable) while your View is a struct.

The View is a Struct (Value Type)

If you change a property inside a struct, Swift considers the entire struct to have changed. If your ViewModel were a struct, every time a user typed a single letter into the TextField (viewModel.name), the entire viewModel struct would be recreated, forcing the State container to trigger a heavy re-render loop.

The ViewModel is a Class (Reference Type)

A class lives in a fixed spot in your device's memory. When you pass a class instance around, you aren't copying the data; you are just passing pointers (references) to that same spot in memory.

When we combine them in MVVM:

  • The Container (@State): Lives inside the View struct. It holds a pointer to the ViewModel class.
  • The View Body: Observes the class. Because your ViewModel class is marked with @Observable, it gains the ability to whisper directly to SwiftUI: "Hey, the user just typed a character in name. Only repaint the specific TextFields that care about name."

Putting It All Together

  • The Class (ViewModel): Holds your actual, shifting data and networking logic safely in one persistent spot in memory.
  • The State Wrapper (_viewModel): Acts as the glue anchor inside the View struct, keeping a firm grip on that Class instance so it doesn't vanish when the View re-renders.
  • The Underscore (_): Your one-time pass inside the init() method to build that glue anchor before the rest of the View turns on.

Again

In Swift, property wrappers like @State have two parts:

  • viewModel: The actual data inside (your ViewModel instance).
  • _viewModel: The wrapper itself (the State struct managing that instance).

When you write a custom init method for a View, you cannot assign a value directly to viewModel because SwiftUI hasn't finished setting up its tracking system yet. You have to talk directly to the wrapper.

Analogy: Think of @State as a locked glass display case, and your data is the trophy inside. You can't just throw a new trophy into the case; you have to use the key (_) to interact with the case framework itself (State(initialValue: ...)) to put the trophy in.

2. The Refactoring Process (Step-by-Step)

When you want to convert a standard SwiftUI View into MVVM, don't try to do it all at once. Follow this mechanical 4-step sequence.

Step 1: Create the Empty ViewModel Shell

Start by nesting an @Observable class inside an extension of your view. This keeps your names clean (it will be called EditView.ViewModel). 

extension EditView {
    @Observable
    class ViewModel {
        // Your data and logic will move here
    }
}

Step 2: Migrate State Properties (The "Data")

Look at your original View and cut-and-paste your @State variables into the ViewModel.

  • Rule: Strip away @State private and just leave var.
  • Rule: If the property depends on an input parameter (like location), you need to create an init for the ViewModel to capture it.

Inside ViewModel:

var name: String
var description: String
var loadingState = LoadingState.loading
var pages = [Page]()
let location: Location

init(location: Location) {
    self.location = location
    self.name = location.name
    self.description = location.description
}
Step 3: Migrate Methods (The "Logic")

Cut-and-paste any functions that fetch data, calculate values, or handle button taps into the ViewModel.

  • Note: Because your ViewModel updates properties asynchronously (like pages or loadingState), wrap those specific UI updates in await MainActor.run so they safely change on the main thread. 
func fetchNearbyPlaces() async {
    // ... API call logic ...
    await MainActor.run {
        self.pages = derivedPages
        self.loadingState = .loaded
    }
}

Step 4: Hook up the View to the ViewModel

Now that your View has been emptied of its old @State and functions, connect it to the new source of truth.

  • Declare the ViewModel at the top of your View using @State: 
@State private var viewModel: ViewModel
  • Implement the Underscore Initializer to bootstrap it: 
init(location: Location, onSave: @escaping (Location) -> Void) {
    self.onSave = onSave
    self._viewModel = State(initialValue: ViewModel(location: location))
}

Fix the Compiler Errors in your body by prefixing the old variables with viewModel..

  • text: ＄name becomes text: ＄viewModel.name
  • switch loadingState becomes switch viewModel.loadingState
  • await fetchNearbyPlaces() becomes await viewModel.fetchNearbyPlaces()

Summary Checklist

Action Old Way (View) New Way (MVVM)
State Declaration @State private var name: String var name: String (Inside @Observable class)
View Binding ＄name ＄viewModel.name
Logic/API Calls func doSomething() inside View func doSomething() inside ViewModel
UI Updates (Async) Direct assignment Must use await MainActor.run inside ViewModel

By isolating the data into a plain class, your View becomes a dumb "reflection" of whatever is currently inside that class. It makes your code cleaner, easier to test, and separates the visual layout from your network and business rules!
