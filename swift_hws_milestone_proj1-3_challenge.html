Hacking With Swift - Milestone: - Challenge

You have a basic understanding of arrays, state, views, images, text, and more, so let’s put them together: your challenge is to make a brain training game that challenges players to win or lose at rock, paper, scissors.

So, very roughly:

  • Each turn of the game the app will randomly pick either rock, paper, or scissors.
  • Each turn the app will alternate between prompting the player to win or lose.
  • The player must then tap the correct move to win or lose the game.
  • If they are correct they score a point; otherwise they lose a point.
  • The game ends after 10 questions, at which point their score is shown.

So, if the app chose “Rock” and “Win” the player would need to choose “Paper”, but if the app chose “Rock” and “Lose” the player would need to choose “Scissors”.

To solve this challenge you’ll need to draw on skills you learned in tutorials 1 and 2:

  • Start with an App template, then create a property to store the three possible moves: rock, paper, and scissors.
  • You’ll need to create two @State properties to store the app’s current choice and whether the player should win or lose.
  • You can use Int.random(in:) to select a random move. You can use it for whether the player should win too if you want, but there’s an even easier choice: Bool.random() is randomly true or false. After the initial value, use toggle() between rounds so it’s always changing.
  • Create a VStack showing the player’s score, the app’s move, and whether the player should win or lose. You can use if shouldWin to return one of two different text views.
  • The important part is making three buttons that respond to the player’s move: Rock, Paper, or Scissors.
  • Use the font() modifier to adjust the size of your text. If you’re using emoji for the three moves, they also scale. Tip: You can ask for very large system fonts using .font(.system(size: 200)) – they’ll be a fixed size, but at least you can make sure they are nice and big!

I’m going to provide some hints below, but I suggest you try to complete as much of the challenge as you can before reading them.

Hints:

  • Start off with the simplest logic you can: three buttons, each with logic that says “the player tapped rock, the player was trying to win, and the app chose scissors, so add 1 point.”
  • Once you have that working, look for a way to simplify your logic such as an array of which items beat each move. For example, if your moves array was ["Rock", "Paper", "Scissors"] your array of winning moves would be ["Paper", "Scissors", "Rock"].
  • You don’t need to add graphics if you don’t want to; just some text views and buttons is enough. Why not try emoji?

This should be a fun exercise: there’s a little bit of Swift, a little bit of SwiftUI, and a little bit of logic, all wrapped up in a game where you can really go to town on theming if you want to.

AI Input

Here is a peer review with some "Pro" improvements to make your code cleaner, more robust, and more idiomatic.

1. The "Computed Property" Shortcut

In your code, you have a few places where you are accessing moves[currentMoveIndex]. Instead of typing that out repeatedly, you can create a computed property. This makes the rest of your code much easier to read. 

// Add this inside your ContentView
var computerMove: Move {
    moves[currentMoveIndex]
}

Now, in your VStack, you can just type Text(computerMove.emoji)!

2. Move Logic into the Struct

Right now, your buttonTapped function is doing a lot of heavy lifting with if/else strings. A more "Swift-y" way is to let the Move struct determine if it beats another move. 

struct Move: Equatable {
    let emoji: String
    let name: String
    
    // Logic moved here: "Self" vs "Other"
    func beats(_ other: Move) -> Bool {
        if self.name == "Rock" && other.name == "Scissors" { return true }
        if self.name == "Paper" && other.name == "Rock" { return true }
        if self.name == "Scissors" && other.name == "Paper" { return true }
        return false
    }
}

3. Simplify the buttonTapped Function

With the logic moved into the struct, your function becomes much shorter and easier to debug:

}

4. Layout & UI Polish

The Alert Title: You have alertTitle as a state variable but you never set it. You could set it to "Game Over!" inside the if tries > 3 block.

Button Style Placement: You applied .buttonStyle() inside the ForEach but after the background and frame. For custom ButtonStyles that use scaleEffect, it's often best to put the style modifier first or ensure the background is part of the ButtonStyle label so the whole thing shrinks together.

AI Suggestion - Adding a Flash

Adding a visual "flash" for feedback is a great way to make the game feel responsive. Since you are already using a ZStack with a background color, we can achieve this by creating a new @State variable that changes briefly when a button is tapped.

1. Add the Feedback State

First, add a variable to track the "flash" color. We’ll start it as Color.clear so it doesn't show up until a move is made. 

@State private var feedbackColor = Color.clear

2. Update the Background

The "Fading Flash" Implementation

Since you have a LinearGradient, we want the flash to sit on top of it. We will use a Rectangle with your feedbackColor and put it inside the ZStack. 

ZStack {
    // 1. Your existing Gradient (The bottom layer)
    LinearGradient(colors: [.blue.opacity(0.3), .white], startPoint: .top, endPoint: .bottom)
        .ignoresSafeArea()
    
    // 2. The Flash Layer (The middle layer)
    Rectangle()
        .fill(feedbackColor) // Use .fill() for colors on shapes
        .opacity(0.4)
        .ignoresSafeArea()
        .animation(.default, value: feedbackColor) // Added the missing colon!
    
    // 3. Your UI (The top layer)
    VStack {
        // ... all your game text and buttons ...
    }
}

Why the Rectangle approach is better than not:

By using Rectangle().fill(feedbackColor), you are being very explicit.

  • Rectangle: "Cover the whole screen."
  • Fill: "Paint it this color."
  • Opacity: "Make it see-through so I can still see the gradient underneath."

3. Trigger the Flash in buttonTapped

Now, inside your logic, set the color based on whether the player was correct. We can use DispatchQueue.main.asyncAfter to turn the color back to clear after a fraction of a second. 

func buttonTapped(_ index: Int) {
    let playerMove = moves[index]
    let playerWon = playerMove.beats(computerMove)
    
    if playerWon == win {
        score += 1
        feedbackColor = .green
    } else {
        feedbackColor = .red
    }

    // Reset the background color after 0.3 seconds
    DispatchQueue.main.asyncAfter(deadline: .now() + 0.3) {
        feedbackColor = .clear
    }
    
    // ... existing tries and reset logic ...
}

Why this works well

  • Non-Blocking: DispatchQueue.main.asyncAfter allows the code to keep running while the "timer" for the color reset counts down in the background.
  • Accessibility: Visual cues like color flashes help players understand the "rules" of your specific Win/Lose logic without having to read the score every single time.

Summary

By adding this, you've moved into the realm of Juice. In game design, "Juice" refers to those little animations and effects that make a game feel satisfying to play. A red flash on a mistake is a classic "unconscious" teacher—it tells the player's brain "Oops!" faster than any text alert ever could.

Haptic Bump

To add haptics (vibration) in SwiftUI, we use the UIImpactFeedbackGenerator. It’s the same physical "click" you feel when you toggle a switch in the iPhone Settings or use Apple Pay.

1. The Haptic Helper

Since you might want to use haptics in several places, it's cleanest to create a small function to trigger them. You can place this inside your ContentView or as a standalone utility. 

func triggerHaptic(_ style: UIImpactFeedbackGenerator.FeedbackStyle) {
    let generator = UIImpactFeedbackGenerator(style: style)
    generator.prepare() // Wakes up the haptic engine
    generator.impactOccurred()
}

2. Adding it to your Game Logic

Now, in your buttonTapped function, you can provide different physical "feelings" for a correct answer versus a mistake.

  • .heavy or .medium: Feels like a solid "thump" (Great for scoring).
  • .light: A subtle tap (Good for general button presses). 
    func buttonTapped(_ index: Int) {
    let playerMove = moves[index]
    let playerWon = playerMove.beats(computerMove)
    
    if playerWon == win {
        score += 1
        feedbackColor = .green
        triggerHaptic(.medium) // A satisfying "Good job" tap
    } else {
        feedbackColor = .red
        triggerHaptic(.heavy) // A more noticeable "Oops" vibration
    }

    // ... reset logic ...
}

3. Why prepare() matters

You’ll notice I included generator.prepare(). The haptic engine on an iPhone stays "asleep" to save battery. If you just call impactOccurred(), there might be a tiny millisecond of lag while the engine spins up. Calling prepare() ensures the vibration happens at the exact moment the finger touches the screen.

4. A Pro Touch: "Success" and "Error" Haptics

If you want even more distinct patterns (like the double-pulse you feel when FaceID fails), you can use UINotificationFeedbackGenerator: 

let notification = UINotificationFeedbackGenerator()
notification.notificationOccurred(.success) // A pleasant double-tap
// OR
notification.notificationOccurred(.error) // A rapid triple-pulse

Summary

By adding haptics, you’ve broken through the "Glass Barrier." Before, your game was just light and color; now, it has physical weight. It’s the difference between tapping on a window and tapping on a drum.

Adding haptics is the final step to making your game feel like a professional iOS app. We will use two different types of haptic feedback: Impact (for the physical tap) and Notification (for the final game over result).

        generator.prepare()
        generator.impactOccurred()

        let playerActuallyWon = playerMove.beats(computerMove)

            feedbackColor = .green

            triggerHaptic(.light)
        } else {
            feedbackColor = .red
            // A heavier, more noticeable thump for a mistake
            triggerHaptic(.medium)
        }

            feedbackColor = .clear
        }

            // Success notification haptic (Double tap)
            let generator = UINotificationFeedbackGenerator()
            generator.notificationOccurred(.success)
            complete = true
        } else {
            prepareNextRound()
        }
    }
    
    func prepareNextRound() {
        currentMoveIndex = Int.random(in: 0..<moves.count)
        win = Bool.random()
    }
    
    func resetFullGame() {
        tries = 0
        score = 0
        prepareNextRound()
    }
}

QUESTION

Is it possible to get the haptic feedback on the press rather than the release of the buttons?

Gemini said

Yes, it is! In SwiftUI, the default Button action triggers on the "Touch Up" (when you release your finger). To get that immediate, satisfying haptic "click" the moment your finger hits the glass, we need to change how the button detects the press.

The Solution: _onButtonGesture or SimultaneousGesture

To trigger an action on the "press" (Touch Down), we use a DragGesture with a minimumDistance of zero. This tells SwiftUI: "The second a finger touches this area, do something."

Here is how to update your ForEach loop to make the haptics fire instantly: 

ForEach(moves.indices, id: \.self) { index in
    Button {
        // The main game logic still happens here (on release)
        buttonTapped(index)
    } label: {
        VStack(spacing: 10) {
            Text(moves[index].emoji)
            Text(moves[index].name)
        }
        .font(.system(size: 30))
        .foregroundStyle(.white)
        .frame(width: 110, height: 100)
        .background(.blue)
        .clipShape(RoundedRectangle(cornerRadius: 10))
    }
    .buttonStyle(BouncyButtonStyle())
    // ADD THIS GESTURE:
    .simultaneousGesture(
        DragGesture(minimumDistance: 0)
            .onChanged { _ in
                // This fires the MOMENT the finger touches the screen
                triggerHaptic(.light)
            }
    )
}

Why use simultaneousGesture?

If you used a normal .onTapGesture, it would fight with the Button for control. By using simultaneousGesture, you allow the button to keep its "Bouncy" animation and its main action, while also listening for that initial touch to fire the haptic.

Refined Haptic Logic

If you trigger the haptic on the press, you should remove the triggerHaptic call from inside your buttonTapped function. Otherwise, the user will feel two vibrations (one on press and one on release), which can feel "cluttered."

Updated buttonTapped logic:

  • On Press: Trigger .light haptic (via the gesture above).
  • On Release: If they got it wrong, you could trigger a .heavy haptic inside buttonTapped to signal the error.

Summary

By moving the haptic to the "Press" phase, you’ve synchronized the physical sensation with the visual "squish" of your BouncyButtonStyle. It makes the screen feel like a physical button that you are actually pushing down, rather than a digital one that only reacts once you let go.

Equatable (Protocol)

What does the word Equatable mean in this case? 

struct Move: Equatable {
    let emoji: String
    let name: String

    func beats(_ other: Move) -> Bool {
        if self.name == "Rock" && other.name == "Scissors" { return true }

        if self.name == "Scissors" && other.name == "Paper" { return true }
        return false
    }

Gemini said

Equatable is a Protocol.

In Swift, a Protocol is like a "contract." When you tell Swift that your Move struct is Equatable, you are signing a contract that says: "I promise that any two objects of this type can be compared to see if they are the same."

Is it a SwiftUI keyword?

Not exactly. Equatable is actually part of the Swift Standard Library. This means it works in any Swift program, whether you are building a website, a command-line tool, or a SwiftUI app.

Many SwiftUI features (like animations and list updates) depend on being able to check if a piece of data has changed. If SwiftUI knows how to compare your data, it can run much faster.

Why do we need it?

Imagine you have two Move objects:

  • let moveA = Move(emoji: "🪨", name: "Rock")
  • let moveB = Move(emoji: "🪨", name: "Rock")

To you and me, these are obviously the same. But to a computer, these are just two different "bags of data" sitting in two different spots in the phone's memory.

Without Equatable, if you asked Swift if moveA == moveB, it would get confused:

  • Do you want to know if they are stored in the same memory address?
  • Do you want to know if the names match?
  • What if the names match but the emojis don't?

The "Automatic" Magic

Because your Move struct only contains basic types (String), Swift is smart enough to "synthesize" the comparison for you. When you add : Equatable, Swift automatically writes a hidden function behind the scenes that looks like this: 

// Swift writes this for you for free!
static func == (lhs: Move, rhs: Move) -> Bool {
    return lhs.emoji == rhs.emoji && lhs.name == rhs.name
}

If you had a more complex struct—say, one that included a secret ID number—you could write that function yourself to tell Swift: "Only consider two Moves equal if their IDs match, I don't care if the names are different."

Summary of Terms

  • Move: Your custom Type (the "Blueprint").
  • struct: The Keyword used to define that blueprint.
  • Equatable: The Protocol (the "Contract") that allows the == operator to work.

Summary

Think of Equatable as a "Member of the Comparison Club." Without that badge, your Move struct isn't allowed to play with the == or != operators. By adding that one word, you’ve given your custom data the ability to act like a "real" built-in type, like an Integer or a String.

ANSWER

Swift is a "top-down" compiler, but it’s smart enough to find your functions no matter where you put them. However, for human eyes, we usually organize from General to Specific.

The Standard Layout "Sandwich"

Here is how a professional SwiftUI file is typically structured:

1. The "Data Models" (The Bread - Top)

You usually place your structs (like Move) at the very top, often even outside the main ContentView.

Why? These are the "blueprints" for your data. By putting them at the top, anyone reading your code immediately understands what "tools" the app is using before they see the UI.

2. State & Constants (The Filling - Upper Middle)

Inside the struct ContentView, we list all @State variables and constants (let moves).

Why? This is the "Brain" of the view. You want to see what information the view is tracking before you see how that information is displayed.

3. The body (The Heart - Middle)

The var body: some View is the star of the show. It should be easy to find.

Why? This is the visual "Map." Keeping it in the middle separates the data (above) from the logic (below).

4. The Methods/Functions (The Sauce - Bottom)

We generally put func buttonTapped() and func reset() at the bottom.

Why? These are the "instructions." Once you know what the app looks like (the body), you go to the bottom to see exactly how it behaves when poked.

  • Scannability: If you have a 500-line file, you don't want to hunt through UI code to find a variable. You know it's always "at the top."
  • Scope: By putting struct Move outside the ContentView, you make it "Global." This means if you create a second screen later (like a High Score screen), that screen can also use the Move struct without you having to redefine it.

If you put the instructions before the ingredients, the chef gets confused!

  • Clarity: When another developer looks at your Move.swift file and sees import Foundation, they immediately know: "This file is pure data; I don't need to look for any UI code here."

Think of Foundation as the "Engine" of a car and SwiftUI as the "Dashboard and Paint Job."

  • Controller (The functions in your View): The logic that connects the data to the UI.

More on Refactoring Modularization/Decomposition

The process of separating code into different files is most commonly called Refactoring, specifically through Modularization or Decomposition.

Depending on who you’re talking to, you might hear a few different terms:

1. Modularization

This is the "big picture" term. It means you are breaking a large, monolithic block of code into smaller, independent modules. Each module has a specific job (like your Move.swift handling data and Styles.swift handling visuals).

2. Decomposition (or Functional Decomposition)

This refers to the act of taking a complex problem and "decomposing" it into smaller, more manageable pieces. In your case, you decomposed your single ContentView file into three distinct parts: the Model, the Style, and the View.

3. Separation of Concerns (SoC)

This is a core design principle in software engineering. By moving your code into separate files, you are ensuring that each file addresses a single concern.

  • Move.swift is only concerned with What a move is.
  • Styles.swift is only concerned with How a button looks.
  • ContentView.swift is only concerned with Where things go on the screen.

4. "Extracting"

On a day-to-day basis, developers will often say, "I'm going to extract this struct into its own file." It describes the physical act of cutting code out of one place and creating a new home for it.

Why we do it: The "Spaghetti" Test

When all your code is in one file, it’s often called Spaghetti Code—it’s a tangled mess where changing one thing might accidentally break something else five hundred lines away.

By refactoring your code into separate files, you are creating Clean Code. It makes it easier to:

  • Debug: You know exactly which file to look in when a button looks wrong.
  • Collaborate: Two people can work on the same app at the same time (one on the UI, one on the Data) without crashing into each other's work.
  • Test: You can test your Move logic without having to launch the entire visual app.

Move.swift 
import Foundation

// Data model
struct Move: Equatable {
    let emoji: String
    let name: String
    
    func beats(_ other: Move) -> Bool {
        if self.name == "Rock" && other.name == "Scissors" { return true }
        if self.name == "Paper" && other.name == "Rock" { return true }
        if self.name == "Scissors" && other.name == "Paper" { return true }
        return false
    }
}
ContentView 
import SwiftUI


// Main Interfacej
struct ContentView: View {
    
    let moves = [
        Move(emoji: "🪨", name: "Rock"),
        Move(emoji: "📄", name: "Paper"),
        Move(emoji: "✂️", name: "Scissors")
    ]
    
    @State private var win = Bool.random()
    @State private var currentMoveIndex = Int.random(in: 0...2)
    @State private var score = 0
    @State private var tries = 0
    @State private var complete = false
    @State private var feedbackColor = Color.clear
    
    var computerMove: Move { moves[currentMoveIndex] }
    
    var body: some View {
        ZStack {
            LinearGradient(colors: [.blue.opacity(0.3), .white], startPoint: .top, endPoint: .bottom)
                .ignoresSafeArea()
            
            Rectangle()
                .fill(feedbackColor)
                .opacity(0.4)
                .ignoresSafeArea()
                .animation(.default, value: feedbackColor)
            
            VStack {
                Text("Rock-Paper-Scissors")
                    .font(.largeTitle.bold())
                
                Spacer()
                
                VStack(spacing: 10) {
                    Text(computerMove.emoji)
                    Text(computerMove.name)
                }
                .font(.system(size: 30))
                .bold()
                .frame(width: 120, height: 100)
                .background(.blue.opacity(0.4))
                .clipShape(RoundedRectangle(cornerRadius: 10))
                
                VStack {
                    Text(win ? "Try to Win" : "Try to Lose")
                        .font(.system(size: 50))
                        .bold()
                        .foregroundStyle(win ? .green : .red)
                }
                .padding(.vertical, 40)
                
                HStack(spacing: 20) {
                    ForEach(moves.indices, id: \.self) { index in
                        Button {
                            buttonTapped(index)
                        } label: {
                            VStack(spacing: 10) {
                                Text(moves[index].emoji)
                                Text(moves[index].name)
                            }
                            .font(.system(size: 30))
                            .foregroundStyle(.white)
                            .frame(width: 110, height: 100)
                            .background(.blue)
                            .clipShape(RoundedRectangle(cornerRadius: 10))
                            .shadow(color: .black.opacity(0.3), radius: 5, x: 0, y: 5)
                        }
                        .buttonStyle(BouncyButtonStyle())
                    }
                }
                
                Spacer()
                
                Text("Score: \(score)")
                    .font(.system(size: 40))
                    .bold()
                
                Spacer()
                Text("noddynodster@icloud.com")
            }
            .padding()
        }
        .alert("Game Over", isPresented: $complete) {
            Button("Play Again", action: resetFullGame)
        } message: {
            Text("Final Score is \(score) out of 10")
        }
    }
    
    // MARK: - Haptic Helper Function
    func triggerHaptic(_ style: UIImpactFeedbackGenerator.FeedbackStyle) {
        let generator = UIImpactFeedbackGenerator(style: style)
        generator.prepare()
        generator.impactOccurred()
    }
    
    // MARK: - Logic Functions
    func buttonTapped(_ index: Int) {
        let playerMove = moves[index]
        let playerActuallyWon = playerMove.beats(computerMove)
        
        if playerActuallyWon == win {
            score += 1
            feedbackColor = .green
            // A light, crisp tap for a correct answer
            triggerHaptic(.light)
        } else {
            feedbackColor = .red
            // A heavier, more noticeable thump for a mistake
            triggerHaptic(.medium)
        }
        
        DispatchQueue.main.asyncAfter(deadline: .now() + 0.3) {
            feedbackColor = .clear
        }
        
        tries += 1
        
        if tries >= 10 {
            // Success notification haptic (Double tap)
            let generator = UINotificationFeedbackGenerator()
            generator.notificationOccurred(.success)
            complete = true
        } else {
            prepareNextRound()
        }
    }
    
    func prepareNextRound() {
        currentMoveIndex = Int.random(in: 0..<moves.count)
        win = Bool.random()
    }
    
    func resetFullGame() {
        tries = 0
        score = 0
        prepareNextRound()
    }
}
