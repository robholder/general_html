Hacking With Swift - 125 - How to save NavigationStack paths using Codable

You can save and load the 'navigation stack path' using Codable.

The technique is different based on the kind of path you have:

NavigationPath

  • If you're using NavigationPath to store the active path of your NavigationStack:
    • SwiftUI provides two helpers to make saving and loading your paths easier.

Homogenous Array

  • If you're using a homogenous array - e.g. [Int] or [String]:
    • then you don't need those helpers, and you can load or save your data freely.

The techniques are very similar, so I'll cover them both here.

Both rely on storing your path outside your view, so that all the loading and saving of path data happens invisibly.

An external class takes care of it automatically.

Every time our path data changes - whether that's an array of integers or strings, or a NavigationPath object - we need to save the new path so it's preserved for the future, and we'll also load that data back from disk when the class is initialized.

Homogenous Array

Remember that 'didSet' is a property observer.

Here's how that class would look when our path data is stored as an array of integers: 

@Observable
class PathStore {
    var path: [Int] {
        didSet {
            save()
        }
    }

    private let savePath = URL.documentsDirectory.appending(path: "SavedPath")

    init() {
        if let data = try? Data(contentsOf: savePath) {
            if let decoded = try? JSONDecoder().decode([Int].self, from: data) {
                path = decoded
                return
            }
        }

        // Still here? Start with an empty path.
        path = []
    }

    func save() {
        do {
            let data = try JSONEncoder().encode(path)
            try data.write(to: savePath)
        } catch {
            print("Failed to save navigation data")
        }
    }
}

A short deviation to look at this bit:

@Observable
class PathStore {
    var path: [Int] {
        didSet {
            save()
        }
    }

This is a bit of a "syntax soup" moment because it combines three different Swift concepts in one place: Type Annotation, Property Observers, and the Init requirement.

Let's break down exactly what that block is doing and why the braces are there.

1. The Anatomy of the Block

In Swift, you usually see variables declared simply as var path = [Int](). However, when you want to "attach" logic to a variable, you provide a block { } immediately after the type.

  • var path: [Int]: This is the "What." It tells Swift, "I have a property named path that stores an array of integers."
  • The { ... } block: This is the "How." It houses the Property Observer.
  • didSet { ... }: This is the "When." It tells Swift, "Every single time this array changes (an item is added, removed, or the whole thing is replaced), run the code inside these braces."

2. Why the {} after the type?

Normally, braces { } in Swift indicate a Computed Property (a variable that calculates a value on the fly). However, because you've put didSet inside them, Swift recognizes this as a Stored Property with an observer.

The braces act as a container for "hooks." You are telling the compiler: "Don't just store this data silently; I want to hang some logic off of it."

3. The "Missing" Piece: The Initializer

You might be wondering: "Wait, if I have braces there, where does the actual data go?"

Because you haven't given path an initial value (like = []), Swift requires you to provide one in an init(). Your class is structured like this: 

@Observable
class PathStore {
    var path: [Int] {
        didSet {
            save()
        }
    }

    init() {
        // This is where the actual array is born.
        // Swift loads the data, and path gets its value.
        self.path = [] 
    }
}

4. Why this is great for Navigation

Think of this setup as an Automated Secretary.

In your previous experiments, you were manually managing the array. Here, the PathStore class is designed so that the rest of your app can just say path.append(64) and forget about it. The didSet observer is "watching" from inside the PathStore. The moment that 64 hits the array, the didSet wakes up and calls save().

You don't have to remember to call save() every time you navigate; the variable handles its own chores.

Summary

  • The braces { } are a container for observers.
  • didSet is a trigger that fires after a change.
  • This pattern ensures your JSON saving always stays in sync with what the user is seeing on screen.

More on the init() block

init() {
    if let data = try? Data(contentsOf: savePath) {
        if let decoded = try? JSONDecoder().decode([Int].self, from: data) {
            path = decoded
            return
        }
    }

The Logic Breakdown

  • init(): This is the initializer. It runs automatically the moment you create an instance of this class, ensuring your app starts with its previous state loaded.
  • if let data = try? Data(contentsOf: savePath):
    • try? is the key here. It says: "Try to read this file. If it doesn't exist or is corrupted, don't crash; just return nil."
    • The if let ensures the rest of the code only runs if data was successfully created.
  • if let decoded = try? JSONDecoder().decode([Int].self, from: data):
    • The Syntax [Int].self: This is a bit of "meta" Swift. You aren't passing an array; you are passing the type itself as a parameter so the decoder knows what "shape" to turn the data into.
    • decoded: This isn't actually the decoder object itself; it is the resulting array of integers produced after the decoder finished its job.
  • path = decoded: You’re assigning that freshly minted array to your class property so the UI can update.
  • return: This is a "guard-like" exit. If loading worked, we stop here. If either if let failed, the code skips this whole block and hits your fallback (the empty array).

A Visual Logic Flow

Think of this as a security checkpoint with two gates:

  • Gate 1 (File System): Does a file exist at savePath?
    • No? Exit and use an empty array.
    • Yes? Pass the data to the next gate.
  • Gate 2 (The Decoder): Is this data actually a valid list of Integers?
    • No? Exit and use an empty array.
    • Yes? Success! Assign to path and return.

A Quick Tip on try?

In Day 44, Paul uses try? to keep the code clean. In a "real" production app, you might eventually use do-catch blocks so you can tell the user why it failed (e.g., "Out of disk space" vs "The data was corrupted"), but for navigation persistence, try? is a perfectly elegant way to handle it.

Another Deviation

URL Path

private let savePath = URL.documentsDirectory.appending(path: "SavedPath")

This line is all about finding a safe, permanent "home" for your navigation data on the user's device. Since apps are "sandboxed" in iOS, you can't just save files anywhere; you have to ask the system for a valid address.

Here is the breakdown of the "address" you are building:

1. URL

This is the Type. In modern Swift development, we use URL objects instead of "String" paths (like C:/Users/...) because URLs are more robust. They handle special characters and system differences automatically.

2. .documentsDirectory

This is a System Helper.

  • What it means: It points to a specific folder created just for your app called the Documents folder.
  • Why use it: Files in this folder are private to your app, permanent (they won't be deleted unless the app is uninstalled), and automatically backed up to iCloud by the system.

3. .appending(path: "SavedPath")

This is the File Namer.

  • What it means: It takes that generic folder address and adds your specific filename to the end of it.
  • The Result: If your documents directory was .../Documents/, this method turns it into .../Documents/SavedPath.

Why is it a let and private?

  • private: Only the PathStore class needs to know where its data is hidden. No other part of your app should be "reaching in" to mess with that file.
  • let: The location of your documents directory and the name of your file won't change while the app is running. Setting it as a constant is safer and more efficient.

How it looks in action

If you were to print(savePath), it would look something like this (depending on whether you're on a simulator or a real iPhone): 

file:///Users/username/Library/Developer/CoreSimulator/Devices/.../data/Documents/SavedPath

Vim Tool: The "Path" Check

If you are debugging and want to see exactly where your app is saving data, you can use a quick print statement.

  • Use O in the init() of your class.
  • Type: print(savePath).
  • When the app runs, you can copy that path from the Xcode console and use the command :!open [pasted path] (on macOS) inside Vim/Terminal to jump straight to that folder in Finder!

Summary

  • URL: The format.
  • .documentsDirectory: The "neighborhood" (Safe & Permanent).
  • .appending(path:): The "house number" (Your specific file name).+

NavigationPath

If you're using NavigationPath, you need only four small changes.

First, the path property needs to have the type NavigationPath rather than [Int]: 

var path: NavigationPath {
    didSet {
        save()
    }
}

Second, when we decode our JSON in the initializer we need to decode to a specific type, then use the decoded data to create a new NavigationPath: 

if let decoded = try? JSONDecoder().decode(NavigationPath.CodableRepresentation.self, from: data) {
    path = NavigationPath(decoded)
    return
}

Third, if decoding fails we should assign a new, empty NavigationPath instance to the path property at the end of the initializer: 

path = NavigationPath()

And finally, the save() method needs to write the Codable representation of our navigation path.

This is where things diverge just a little more from using a simple array, because NavigationPath doesn't require that its data types conform to Codable - it only needs Hashable conformance.

As a result, Swift can't verify at compile time that there is a valid Codable representation of the navigation path, so we need to request it and see what comes back.

That means adding a check at the start of the save() method, which attempts to retrieve the Codable navigation path and bails out immediately if we get nothing back: 

guard let representation = path.codable else { return }

That will either return the data ready to be encoded to JSON, or return nil if at least one object in the path cannot be encoded.

Finally, we convert that Codable representation to JSON instead of the original Int array: 

let data = try JSONEncoder().encode(representation)

Here's how the complete class looks: 

@Observable
class PathStore {
    var path: NavigationPath {
        didSet {
            save()
        }
    }

    private let savePath = URL.documentsDirectory.appending(path: "SavedPath")

    init() {
        if let data = try? Data(contentsOf: savePath) {
            if let decoded = try? JSONDecoder().decode(NavigationPath.CodableRepresentation.self, from: data) {
                path = NavigationPath(decoded)
                return
            }
        }

        // Still here? Start with an empty path.
        path = NavigationPath()
    }

    func save() {
        guard let representation = path.codable else { return }

        do {
            let data = try JSONEncoder().encode(representation)
            try data.write(to: savePath)
        } catch {
            print("Failed to save navigation data")
        }
    }
}

Now you can go ahead and write your SwiftUI code as normal, making sure to bind the path of your NavigationStack to the path property of a PathStore (class) instance. For example, this lets you show views with random integers attached - you can push as many views as you like, then quit and relaunch the app to get it exactly back as you left it:

    var number: Int

    var body: some View {
        NavigationLink("Go to Random Number", value: Int.random(in: 1...1000))
            .navigationTitle("Number: \(number)")
    }
}

            DetailView(number: 0)
                .navigationDestination(for: Int.self) { i in
                    DetailView(number: i)
                }
        }
    }
}

Additional AI Notes

"two helpers"

The "two helpers" phrase is referring to two specific properties belonging to the NavigationPath struct that allow you to convert it into a format that can be saved (encoded) and later restored (decoded).

The two helpers are the codable property and the CodableRepresentation type.

1. The "Codable" Property

This is the Exporter. Since NavigationPath can hold many different types of data, you can't just save it to a file directly. You need to turn it into a specific "data-only" format first.

  • What it is: path.codable
  • What it does: It checks if everything you’ve put into your path is actually Codable. If it is, it returns a "Codable Representation" of your path that can be turned into JSON or saved to UserDefaults.

2. The "CodableRepresentation" Type

This is the Importer. When your app restarts, you might have a big chunk of JSON data saved. You need a way to turn that data back into a living, working NavigationPath.

  • What it is: NavigationPath.CodableRepresentation
  • What it does: This is a specific "container" type used during decoding. You decode your saved data into this type first, and then you can initialize a brand new NavigationPath using it.

How they work together in code

Here is the "Big Picture" of those two helpers in action. Imagine you want to save your path to UserDefaults:

Saving (Using the first helper):

if let representation = path.codable {
    // representation is the first "helper" (path.codable)
    if let data = try? JSONEncoder().encode(representation) {
        UserDefaults.standard.set(data, forKey: "SavedPath")
    }
}

Loading (Using the second helper):

if let data = UserDefaults.standard.data(forKey: "SavedPath") {
    // CodableRepresentation is the second "helper" (the type itself)
    if let decoded = try? JSONDecoder().decode(NavigationPath.CodableRepresentation.self, from: data) {
        path = NavigationPath(decoded)
    }
}

Why the text calls them "Helpers"

Without these two, saving a NavigationPath would be a nightmare. Because NavigationPath "erases" the types of the objects inside it (turning everything into a generic Hashable object), you would normally lose the ability to encode it.

These helpers are SwiftUI’s way of saying: "I know I hid the types of your data, but if you use these two tools, I’ll reach inside, find the original types, and safely turn them into JSON for you."
