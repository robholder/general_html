A Complete Guide to Implementing Swipe-to-Delete in SwiftUI

Swipe-to-delete is one of the most intuitive user interaction patterns in iOS. In SwiftUI, enabling this behavior requires a precise marriage of view hierarchy and data management.

This guide breaks down how to implement an onDelete system in two parts: first using a standard, memory-based array, and second, migrating that logic into a SwiftData persistent storage database context.

Part 1: The Core Foundation (Standard In-Memory Arrays)

To understand swipe-to-delete, you must understand a golden rule of SwiftUI layout mechanics:

The Structural Rule: You cannot attach an .onDelete modifier directly to a List. You must attach it to a ForEach loop inside a List.

The List view controls the outer visual container (scroll indicators, row separation lines, and grouping styles). The ForEach view is responsible for managing the dynamic generation of individual rows, meaning it is the only component capable of tracking swipe gestures on specific indices.

Step 1: The Code Implementation

Here is a self-contained layout tracking a basic array of strings. 

import SwiftUI

struct StandardDeleteView: View {
    // 1. The source of truth (Mutable state array)
    @State private var items = ["Apple", "Banana", "Cherry", "Date", "Elderberry"]
    
    var body: some View {
        NavigationStack {
            List {
                // 2. ForEach loops through rows inside the List container
                ForEach(items, id: \.self) { item in
                    Text(item)
                        .font(.body)
                        .padding(.vertical, 4)
                }
                // 3. Attach onDelete directly to the ForEach view
                .onDelete(perform: deleteItem)
            }
            .navigationTitle("Fruit Inventory")
            .toolbar {
                EditButton() // Automatically toggles a multi-select delete mode
            }
        }
    }
    
    // 4. The deletion handler method
    func deleteItem(at offsets: IndexSet) {
        items.remove(atOffsets: offsets)
    }
}

How the Mechanics Work:

  • IndexSet: When a user swipes and confirms a deletion, SwiftUI bundles the numerical positions of the targeted row(s) into an IndexSet collection type.
  • .remove(atOffsets:): Standard Swift arrays feature a built-in method designed specifically to consume an IndexSet and erase those matching positions in a single pass.
  • EditButton: Adding this system component to your toolbar provides an alternate layout mode, adding red minus indicators to the left of your rows automatically.

Part 2: Persistent Storage (The SwiftData Architecture)

When you upgrade your application to use SwiftData, the code layout rules stay exactly the same, but the data mutability mechanics change completely.

Because a @Query property wrapper is a read-only view reflection of an external database on disk, you cannot run photos.remove(atOffsets:) directly on it. Doing so will cause a compiler crash. Instead, you must intercept the index, identify the underlying model instance, and instruct the ModelContext worker to delete it from disk.

The Architecture Workflow

Step 1: The Data Model

Ensure your data object is declared as a @Model class. 

import Foundation
import SwiftData

@Model
class NamedPhoto {
    var name: String
    @Attribute(.externalStorage) var photoData: Data
    
    init(name: String, photoData: Data) {
        self.name = name
        self.photoData = photoData
    }
}

Step 2: The View & Context Management

Here is the complete implementation of the persistent deletion system: 

import SwiftData
import SwiftUI

struct PersistentDeleteView: View {
    // 1. Establish your active database worker
    @Environment(\.modelContext) private var modelContext
    
    // 2. Stream the live, sorted database array
    @Query(sort: \NamedPhoto.name) private var photos: [NamedPhoto]
    
    var body: some View {
        NavigationStack {
            List {
                // 3. ForEach isolates the records for gesture tracking
                ForEach(photos) { item in
                    HStack(spacing: 16) {
                        if let uiImage = UIImage(data: item.photoData) {
                            Image(uiImage: uiImage)
                                .resizable()
                                .scaledToFill()
                                .frame(width: 50, height: 50)
                                .clipShape(.rect(cornerRadius: 6))
                        }
                        Text(item.name)
                            .font(.headline)
                    }
                }
                // 4. Attach onDelete to the ForEach view loop
                .onDelete(perform: removePhoto)
            }
            .navigationTitle("Photo Collector")
        }
    }
    
    // 5. Intercept the offset index and map to database deletion
    func removePhoto(at offsets: IndexSet) {
        for index in offsets {
            // Find the precise model object instance occupying that row position
            let photoToDelete = photos[index]
            
            // Command the database manager to scrub the entry from disk
            modelContext.delete(photoToDelete)
        }
        
        // Note: You do not need to call a save function manually. SwiftData 
        // autosaves this mutation, and @Query automatically cascades the UI 
        // animation updating the list rows.
    }
}

Troubleshooting Quick-Reference

The Error Scenario The Cause The Correction
Value of type 'List<...>' has no member 'onDelete' The .onDelete modifier is attached to the parent List element container. Move the .onDelete modifier down so it hooks directly onto the closing bracket of your ForEach view.
Cannot use mutating member on immutable value: 'photos' is a get-only property You tried to use .remove(atOffsets:) directly on an array fetched via a @Query wrapper. Use the loop approach inside your function to resolve photos[index], and pass that target entity to modelContext.delete().
Swipe-to-delete animation stutters or drops frames. Heavy data (like binary images) is being stored natively inside the main SQL database file block. Add the @Attribute(.externalStorage) macro option wrapper to your data property inside your @Model file schema blueprint to isolate file writes.
