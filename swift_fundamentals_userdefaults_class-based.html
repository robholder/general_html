Swift - Class Based UserDefaults

Blueprint: Class-Based UserDefaults Management with @Observable

This reference guide isolates the architectural pattern for managing lightweight persistent data (such as a Favorites feature) globally outside of your views.

This technique maintains a high-performance Runtime Layer (Set) for instantaneous user interactions, paired with a reliable Storage Layer (UserDefaults) for background persistence.

🚀 Quick Checklist: Class-Based UserDefaults Implementation

  • Step 1: Centralize Keys — Define a raw-value String enum (StorageKeys) to store your identifier strings safely and eliminate typo-driven bugs.
  • Step 2: Create the Class Container — Build an @Observable class (e.g., StorageManager or Favorites) to handle the business logic away from the UI layer.
  • Step 3: Establish a Dual-Layer Pipeline:
    • Runtime Layer: Declare a private Set<String> for lightning-fast, duplicate-free lookups (O(1)) while the app is actively running.
    • Storage Layer: Use UserDefaults.standard to read/write arrays to disk.
  • Step 4: Code the Initialization (init) — Read the data from disk as a string array using your enum key, convert it back into a Set, or fall back to an empty set ([]).
  • Step 5: Define Core Mutation API Methods:
    • Create a contains() method checking the local runtime Set.
    • Create add() and remove() methods that modify the runtime Set and immediately call a private save() routine.
  • Step 6: Encode the Save Mechanism — Inside the save() routine, cast the runtime Set into an Array (Array(itemIDs)) so UserDefaults can serialize and write it to the property list file.
  • Step 7: Inject into Global Environment — Instantiate the manager once at the root level of your App struct using @State, and pass it down the view tree via the .environment() modifier.

Complete Blueprint Code (Copy-Paste Ready)

This class completely encapsulates the data logic. The views simply call add(), remove(), or contains(), unaware of how the data is being saved under the hood. 

import SwiftUI

// 1. Centralized storage key to prevent typo-driven bugs
enum StorageKeys: String {
    case favoritesList
}

// 2. A placeholder model representing your data item
struct CustomItem: Identifiable {
    let id: String
    let name: String
}

@Observable
class StorageManager {
    // High-performance runtime collection (O(1) lookups)
    private var itemIDs: Set<String>
    
    // Resolve the string value from our enum key
    private let storageKey = StorageKeys.favoritesList.rawValue
    
    init() {
        // READ STEP: Attempt to read the array from disk
        if let savedArray = UserDefaults.standard.stringArray(forKey: storageKey) {
            // Convert storage Array back into runtime Set
            self.itemIDs = Set(savedArray)
        } else {
            // Fallback initialization if no history exists
            self.itemIDs = []
        }
    }
    
    /// Checks if an item is stored (Runs in constant time).
    func contains(_ item: CustomItem) -> Bool {
        itemIDs.contains(item.id)
    }
    
    /// Appends an item to the collection and syncs to disk.
    func add(_ item: CustomItem) {
        itemIDs.insert(item.id)
        save()
    }
    
    /// Removes an item from the collection and syncs to disk.
    func remove(_ item: CustomItem) {
        itemIDs.remove(item.id)
        save()
    }
    
    /// Encapsulated persistence engine.
    private func save() {
        // WRITE STEP: Cast Set into an Array for UserDefaults serialization
        UserDefaults.standard.set(Array(itemIDs), forKey: storageKey)
    }
}

View Integration Blueprint

To make this manager accessible across an entire app layout, initialize it once in the main App entry point, inject it into the root view environment, and pull it down wherever needed.

1. Root Level Injection

@main
struct YourApp: App {
    // Instantiate the manager once for the lifetime of the application
    @State private var storageManager = StorageManager()
    
    var body: some Scene {
        WindowGroup {
            MainContentView()
                .environment(storageManager) // Inject into the environment
        }
    }
}

2. Implementation inside a View

struct ItemRowView: View {
    let item: CustomItem
    
    // Read the shared instance from the environment
    @Environment(StorageManager.self) private var storageManager
    
    var body: some View {
        HStack {
            Text(item.name)
            Spacer()
            
            Button {
                if storageManager.contains(item) {
                    storageManager.remove(item)
                } else {
                    storageManager.add(item)
                }
            } label: {
                Image(systemName: storageManager.contains(item) ? "heart.fill" : "heart")
                    .foregroundStyle(.red)
            }
        }
    }
}

3 Core Mechanics to Keep in Mind

  • The Set-to-Array Transformer: UserDefaults cannot natively understand or serialize a Swift Set. Because of this, we utilize a data transformer setup: we run our app logic inside a high-speed Set container, but explicitly cast it down to an Array (Array(itemIDs)) the exact moment it writes to disk.
  • Elimination of View Redraw Overhead: Unlike @AppStorage, which causes a view to invalidate and redraw the moment any key updates, an @Observable class allows SwiftUI to look deep inside your model properties. Your views will now only redraw if they explicitly evaluate properties targeted by a state change.
  • Implicit Main-Thread Sync: Calls to UserDefaults.standard.set() execute immediately on an in-memory dictionary before flushing asynchronously to disk. You do not need to wrap these operations in background tasks or manual threads for small item tracking arrays.

Extra Insight

QUESTION

Can you expand on what is going on here? 

// 1. Centralized storage key to prevent typo-driven bugs
enum StorageKeys: String {
    case favoritesList
}

And how does it relate to: 

private let storageKey = StorageKeys.favoritesList.rawValue

ANSWER

This pattern is a best-practice technique in iOS development called String Key Centralization.

It solves a massive real-world problem: typo-driven bugs that compile perfectly fine but break your app at runtime.

Here is the exact breakdown of how these two code pieces work together under the hood.

Part 1: Defining the Enum (StorageKeys)

enum StorageKeys: String {
    case favoritesList
}

By conforming the enum to String, you are creating a Raw-Value Enum.

Normally, an enum case is just an abstract compiler symbol. However, when you give it a String backing type, Swift activates a feature called Implicit Raw Values. The compiler automatically assigns every case a hidden string value that matches the case name exactly.

Behind the scenes, Swift translates your code into this: 

// What Swift builds for you automatically:
enum StorageKeys: String {
    case favoritesList = "favoritesList" 
}

Part 2: Extracting the Key (.rawValue)

private let storageKey = StorageKeys.favoritesList.rawValue

The .rawValue property is how you ask Swift to pull out that hidden, underlying string value.

So, when the compiler reads StorageKeys.favoritesList.rawValue, it evaluates it simply as the literal string: "favoritesList".

This value is then immediately passed to your storage engine: 

// What actually runs under the hood:
UserDefaults.standard.stringArray(forKey: "favoritesList")

Why go through all this trouble?

If the system ultimately just uses the string "favoritesList", why not just type that directly into UserDefaults?

1. Auto-Complete (No Typo Bugs)

If you type strings manually across your app, you will eventually make a typo: 

// ❌ This compiles perfectly, but returns 'nil' because of the lowercase 'l'
UserDefaults.standard.stringArray(forKey: "favoriteslist")

The compiler cannot check if your text strings are spelled correctly. However, if you use the enum and make a typo, the compiler instantly stops you with an error: 

// ❌ Compiler Error: Value of type 'StorageKeys' has no member 'favoritessList'
private let storageKey = StorageKeys.favoritessList.rawValue

2. Single Source of Truth

Imagine your app grows, and you use this key in 5 different files. Six months from now, you decide you want to rename the key on disk to "user_favorites_v2".

  • Without the enum: You have to do a global Find-and-Replace across your entire codebase and pray you didn't miss a spot.
  • With the enum: You change it in one single line inside your enum, and your entire app updates perfectly: 
enum StorageKeys: String {
    case favoritesList = "user_favorites_v2" // Everything else updates instantly!
}
