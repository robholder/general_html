Hacking With Swift - 197 - Improving our map annotations

Right now we’re using Marker to place locations in our Map view, but SwiftUI lets us place any kind of view on top of our map so we can have complete customizability. So, we’re going to use that to show a custom SwiftUI view containing a custom icon, then take a look at the underlying data type to see what improvements can be made there.

Thanks to the brilliance of SwiftUI, this takes hardly any code at all - replace your existing Marker code with this Annotation block: 

Annotation(
    location.name,
    coordinate: CLLocationCoordinate2D(
            latitude: location.latitude,
            longitude: location.longitude)) {
                Image(systemName: "star.circle")
                .resizable()
                .foregroundStyle(.red)
                .frame(width: 44, height: 44)
                .background(.white)
                .clipShape(.circle)
}

That really helps our location stand out on the map. However, I want to look beyond just the SwiftUI view: I want to look at the Location struct itself, and apply a few improvements that make it better.

Clean Up Code With Computed Property

First, I don’t particularly like having to make a CLLocationCoordinate2D inside our SwiftUI view, and I’d much prefer to move that kind of logic inside our Location struct. So, we can move that into a computed property to clean up our code. First, add an import for MapKit into Location.swift, then add this to Location: 

var coordinate: CLLocationCoordinate2D {
    CLLocationCoordinate2D(latitude: latitude, longitude: longitude)
}

In Context

Location.swift

import Foundation
import MapKit

struct Location: Codable, Equatable, Identifiable {
    let id: UUID
    var name: String
    var description: String
    var latitude: Double
    var longitude: Double
    
    // Computed Property
    var coordinates: CLLocationCoordinate2D {
        CLLocationCoordinate2D(latitude: latitude, longitude: longitude)
    }
}

Now our ContentView code is simpler: 

Annotation(location.name, coordinate: location.coordinate) {

In Context

ContentView.swift

import MapKit
import SwiftUI

let startPosition = MapCameraPosition.region(
    MKCoordinateRegion(
        center: CLLocationCoordinate2D(
            latitude: 56,
            longitude: -3),
        span: MKCoordinateSpan(
            latitudeDelta: 10,
            longitudeDelta: 10)
        )
    )

struct ContentView: View {
    @State private var locations = [Location]()
    
    var body: some View {
        MapReader { proxy in
            Map(initialPosition: startPosition) {
                ForEach(locations) { location in
                    // Swap out Marker for Annotation
                    Annotation(
                        location.name,
                        // Use computed property from Location struct
                        coordinate: location.coordinates) {
                            // Custom 'pin' icon:
                            Image(systemName: "star.circle")
                                .resizable()
                                .foregroundStyle(.red)
                                .frame(width: 44, height: 44)
                                .background(.white)
                                .clipShape(.circle)
                            }
                    }
            }
                .onTapGesture { position in
                    if let coordinate = proxy.convert(position, from: .local) {
                        let newLocation = Location(
                            id: UUID(),
                            name: "New location",
                            description: "",
                            latitude: coordinate.latitude,
                            longitude: coordinate.longitude
                        )
                        locations.append(newLocation)
                    }
                }
        }
    }
}

#Preview {
    ContentView()
}

Add Example Pin (Location)

The second change I want to make is one I encourage everyone to make when building custom data types for use with SwiftUI: add an example! This makes previewing significantly easier, so where possible I would encourage you to add a static example property to your types containing some sample data that can be previewed well.

So, add this second property to Location now: 

static let example = Location(
    id: UUID(),
    name: "Buckingham Palace",
    description: "Lit by over 40,000 lightbulbs.",
    latitude: 51.501,
    longitude: -0.141
    )

Tip: If you wanted, you could wrap the static let example line with #if DEBUG and #endif to avoid it being built into your App Store releases.

The last change I’d like to make here is to add a custom == function to the struct. We already made Location conform to Equatable, which means we can already compare one location to another using ==. Behind the scenes, Swift will write this function for us by comparing every property against every other property, which is rather wasteful - all our locations already have a unique identifier, so if two locations have the same identifier then we can be sure they are the same without also checking the other properties.

So, we can save a bunch of work by writing our own == function to Location, which compares two identifiers and nothing else: 

static func ==(lhs: Location, rhs: Location) -> Bool {
    lhs.id == rhs.id
}

I’m a huge fan of making structs conform to Equatable as standard, even if you can’t use an optimized comparison function like above - structs are simple values like strings and integers, and I think we should extend that same status to our own custom structs too.

With that in place the next step of our project is complete, so please run it now - you should be able to drop a marker and see our custom annotation, but now behind the scenes know that our code is a little bit neater too!

Location.swift

import Foundation
import MapKit

struct Location: Codable, Equatable, Identifiable {
    let id: UUID
    var name: String
    var description: String
    var latitude: Double
    var longitude: Double
    
    // Computed Property
    var coordinates: CLLocationCoordinate2D {
        CLLocationCoordinate2D(latitude: latitude, longitude: longitude)
    }
    
    // Return 'equal' if IDs are the same.
    static func ==(lhs: Location, rhs: Location) -> Bool {
        lhs.id == rhs.id
    }

    static let example = Location(
        id: UUID(),
        name: "Buckingham Palace",
        description: "Lit by over 40,000 lightbulbs.",
        latitude: 51.501,
        longitude: -0.141
    )
    #endif
}

Ai Insight

When you wrap code inside #if DEBUG and #endif, that entire block of code is stripped out completely and never compiled into the final application package that gets sent to the App Store.

Here is a breakdown of what this entails, how it works, and why it is a best practice.

What are #if DEBUG and #endif?

These are called Compiler Directives (specifically, conditional compilation blocks). They act like an if/else statement, but instead of running while your app is executing on a user's phone, they run while Xcode is building your app on your Mac.

Xcode uses a setting called Build Configurations. By default, every project has two:

  • Debug: Used when you click the Play button (Command + R) to test your app on the Simulator or a plugged-in iPhone.
  • Release: Used when you archive your app to submit it to the App Store.

What it Entails in Your Code

In the context of Paul Hudson's tutorial, you are likely creating a mock data structure so you can easily preview your SwiftUI designs in Xcode.

By wrapping it, your code looks like this: 

struct Location {
    let id: UUID
    let name: String
    
    // This code only exists when testing on your machine
    #if DEBUG
    static let example = Location(id: UUID(), name: "Buckingham Palace")
    #endif
}

Or: 

#if DEBUG
static let example = Location(
    id: UUID(),
    name: "Buckingham Palace",
    description: "Lit by over 40,000 lightbulbs.",
    latitude: 51.501,
    longitude: -0.141
    )
#endif

How Xcode processes this during a build:

  • When you are testing (Debug Configuration): Xcode sees that DEBUG is true. It compiles the example property normally so you can use Location.example inside your SwiftUI #Preview blocks.
  • When you build for the App Store (Release Configuration): Xcode sees #if DEBUG, recognizes that DEBUG is false, and completely ignores everything down to #endif. As far as the App Store is concerned, that code simply does not exist.

Why is this useful?

1. Reduces App Size

Mock data often includes massive arrays of hardcoded strings, dummy text, or sample images meant to mimic a real user's database. Leaving that in your app inflates the file size for your real users unnecessarily.

2. Enhances App Security

If you leave testing tools, hardcoded dummy user accounts, or bypass switches in your production code, clever attackers can reverse-engineer your app's binary file and find them. Stripping them out entirely removes that security risk.

3. Prevents Accidental Use

If you accidentally try to use Location.example in a real production view, Xcode will throw a compiler error when you try to archive the app, protecting you from accidentally releasing test data to your actual customers.
