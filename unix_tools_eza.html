eza

eza is very similar to ls but much more powerful. It is a modern replacement for the ls command that includes a built-in tree view.

While the tree command only does one thing, eza combines file listing with several advanced features:

  • Tree View: Use the -T or --tree flag to show your directory hierarchy.
  • Icons & Color: It uses color-coded output and can display icons for different file types (requires a Nerd Font to be active in your terminal).
  • Git Integration: It can show the Git status (modified, new, ignored) for every file directly in the list.
  • Metadata: It displays human-readable file sizes and extended attributes in a much cleaner format than standard ls.

Common eza Tree Commands

  • eza -T: Basic tree view.
  • eza -T --level 2: Limits the tree to 2 levels deep.
  • eza -lT: Combines the tree view with long format (permissions, size, etc.).
  • eza -T --git: Shows the tree with Git status indicators.

Installation on macOS

You can install it easily using the Homebrew Formula: 

brew install eza.

If you already have a Nerd Font set up in WezTerm (or similar), you can alias ls to eza --icons in your .zshrc to get the full visual experience immediately.

To Substitute eza for ls

To turn eza into your primary tool for both listing and tree views, add these aliases to your ~/.zshrc file.

If you have WezTerm (or similar) with Nerd Fonts and Nord colors, eza will look incredible right out of the box.

1. Add these to your ~/.zshrc

Open your config (vim ~/.zshrc) and paste these at the bottom: 

# General replacement for 'ls'
alias ls='eza --icons --group-directories-first'

# Long format (like 'ls -l')
alias ll='eza -lh --icons --group-directories-first --git'

# All files including hidden (like 'ls -a')
alias la='eza -a --icons --group-directories-first'

# THE TREE COMMAND (Replaces 'tree')
# Shows 3 levels deep by default
alias tree='eza -T --level=3 --icons --group-directories-first'

# THE "ULTIMATE" TREE (Long format + Git status)
alias lt='eza -lT --icons --git --group-directories-first'

2. Activate the changes

Run this to make the new commands live: 

source ~/.zshrc

Why these are better for your setup:

  • --icons: This leverages the Nerd Fonts you just set up in WezTerm. You'll see little icons for folders, .lua files, and .vim files. [2, 3]
  • --group-directories-first: This keeps your folders at the top of the list, making the tree view much easier to read. [1]
  • --git: Since you were asking about Git status earlier, this will show a new column in the tree (N for New, M for Modified) so you can see exactly which files in your subdirectories haven't been committed. [4]

Pro-Tip Stow and .dotfiles

Try running lt inside your .dotfiles directory. It will show you exactly where your Stow links are pointing and the Git status of your individual config files in a single view.

Eza and Git Ignore

To verify that a .gitignore is doing its job in a Git repository, and that eza is correctly identifying your repository's state, use the following commands.

1. The "Git Status" Tree View

If you're using eza with the --git flag, you can see a visual map of what is tracked, what is ignored, and what is modified. Run this from your ~/.dotfiles directory: 

# -T (Tree), -a (All), --git (Status), --git-ignore (Apply .gitignore rules)
eza -Ta --git --git-ignore --icons
  • If it's working: You should not see ignored entities in the tree at all.
  • The Git Column: Look for M (Modified) or N (New) next to files.

2. The "Ignored" Audit

If you want to see exactly what is being ignored (to make sure you didn't accidentally ignore something important): 

# This shows ONLY the files that are currently ignored by Git
git clean -ndX

3. The "Sync Check" Alias

To make this a one-word command add this to your ~/.zshrc: 

# 'check' will show you exactly what is about to be backed up
alias check='eza -la --git --header --icons --group-directories-first'

Why this is a "Pro" move:

  • Safety: If you see a .DS_Store file in your check output, you know your .gitignore has a typo.
  • Cleanliness: By using --git-ignore in your tree view, your project structure looks exactly how it will appear on GitHub—no "junk" files cluttering the view.
  • Quick Tip: If you find that a file you want to ignore is still showing up with an N (New) status, it might already be "staged." Run git rm --cached to tell Git to stop tracking it and start ignoring it.

More on Viewing Git Flags

When you use eza --git, the indicators actually appear in a two-character column. The first character represents the Staging Area (what’s ready to be committed), and the second represents the Workdir (what’s changed on disk but not staged).

Here is the full decoder ring for eza and Git:

The Primary Indicators

Symbol Meaning Description
N New An untracked file that Git hasn't seen before.
M Modified An existing file that has been changed.
D Deleted A file that existed in the last commit but is now gone.
R Renamed You moved or renamed a file (Git tracks this as a pair).
I Ignored Matches a rule in .gitignore or .git/info/exclude.
U Unmerged A conflict! Both you and a branch changed this file.
. / - Unchanged Everything is quiet; no changes detected.

The "Two-Column" Secret

Because eza mimics git status --short, you will often see combinations. If you see M (a space then an M), it means the file is modified but not staged. If you see M (an M then a space), it means you’ve already run git add and it’s staged.

Specialized Indicators

If you are doing advanced operations, you might see these rarer birds:

  • S (Type Changed): You turned a file into a symlink, or vice versa.
  • ! (Ignored Path): Usually shows up when a directory is ignored, but you're forcing eza to look inside.

Pro-Tip: The "Dirty" Directory

When using --tree, look at the Folders.

  • If a folder shows an M, it doesn't mean the folder itself changed (folders are just containers). It means at least one file inside that tree is modified.
  • This "Status Bubbling" is the fastest way to see if you have uncommitted work hidden deep in your subdirectories.

Think of these letters as the "Health Bar" for your project. A sea of . means you’re safe and synced. A splash of N and M means you’re mid-sprint. A red U means it’s time to put on the helmet—you've got merge conflicts to resolve!
