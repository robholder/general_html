Hacking With Swift - 223 - MapKit

Yesterday you built a new app that imports photos from the user’s library, and hopefully you’re pleased with the finished product - or at least making great progress towards the finished product.

But your boss has come in and demanded a new feature:

  • When you’re viewing a picture that was imported, you should show a map with a pin that marks where they were when that picture was added.
  • It might be on the same screen side by side with the photo,
  • It might be shown or hidden using a segmented control, or perhaps
  • It’s on a different screen - it’s down to you.

Regardless, you know how to drop pins, and you also know how to use the center coordinate of map views, so the only thing left to figure out is how to get the user’s location to save alongside their text and image.

Although I do want you to push your skills, I’m not cruel. So, here’s a class that fetches the user’s location: 

import CoreLocation

class LocationFetcher: NSObject, CLLocationManagerDelegate {
    let manager = CLLocationManager()
    var lastKnownLocation: CLLocationCoordinate2D?

    override init() {
        super.init()
        manager.delegate = self
    }

    func start() {
        manager.requestWhenInUseAuthorization()
        manager.startUpdatingLocation()
    }

    func locationManager(_ manager: CLLocationManager, didUpdateLocations locations: [CLLocation]) {
        lastKnownLocation = locations.first?.coordinate
    }
}

To use that, start by adding a new key in the Info tab for your app's target, just like we did when we wanted Face ID permission. This key is called “Privacy - Location When In Use Usage Description”, then give it some sort of value explaining to the user why you need their location.

Now you can use it inside SwiftUI view like this: 

struct ContentView: View {
    let locationFetcher = LocationFetcher()

    var body: some View {
        VStack {
            Button("Start Tracking Location") {
                locationFetcher.start()
            }

            Button("Read Location") {
                if let location = locationFetcher.lastKnownLocation {
                    print("Your location is \(location)")
                } else {
                    print("Your location is unknown")
                }
            }
        }
    }
}

If you’re using the simulator rather than a real device, you can fake a location by going to the Debug menu and choosing Location > Apple.

NOTE: Current version of Simulator - Features > Location > Apple (or choice of 5 options)

And now it’s over to you: can you add the feature your boss wants, and bring MapKit and SwiftUI together in a single app?

Ai Insight - Class (first line)

QUESTION

In the class definition above, are those protocols on the first line of the class? 

class LocationFetcher: NSObject, CLLocationManagerDelegate {

ANSWER

You have spotted a very specific architectural detail of how Swift handles inheritance and protocols on the declaration line!

The short answer is: One of them is a base class, and one of them is a protocol. In Swift, when you are defining a new class, everything after the colon : is written in a comma-separated list. However, that list follows a strict hierarchy: The Superclass must always come first, followed by any Protocols.

Here is the breakdown of exactly what those two items are doing on that first line: 

class LocationFetcher: NSObject, CLLocationManagerDelegate
//                     └──┬───┘  └──────────┬───────────┘
//                     1. Class        2. Protocol

1. NSObject (The Superclass)

NSObject is a Class, not a protocol. Specifically, it is the granddaddy foundational base class for almost all Objective-C and Apple system frameworks.

By inheriting from NSObject, your LocationFetcher class gains access to the core runtime plumbing of Apple's ecosystem (which includes things like memory management hooks and key-value observing).

Why do you need it here?

Notice further down in your code where you override the initializer: override init() { super.init() }. You can only use the override and super keywords if your class is inheriting from a parent class!

2. CLLocationManagerDelegate (The Protocol)

This one is a Protocol.

In Apple's frameworks, whenever you see the word Delegate at the end of a type name, it is almost always a protocol. A delegate protocol is basically a job description or a contract.

By listing CLLocationManagerDelegate on the first line, your class is signing a contract saying: "I promise that I know how to handle location events sent to me by the phone's GPS hardware."

Why are both required to work together?

This is where the plot thickens. The reason you are forced to include the NSObject class just to use the CLLocationManagerDelegate protocol comes down to Apple's internal rules.

If you look at the hidden source code definition for CLLocationManagerDelegate, Apple wrote it like this: 

protocol CLLocationManagerDelegate : NSObjectProtocol

The protocol itself requires whoever adopts it to be an Objective-C compatible object (NSObjectProtocol). Because a pure, native Swift class doesn't speak Objective-C out of the box, you have to inherit from NSObject to give your class that compatibility.

The Lifecycle Chain Reaction:

  • You make your class inherit from NSObject so it can speak the system language.
  • You adopt CLLocationManagerDelegate to declare you can handle GPS updates.
  • Inside your initializer, you type manager.delegate = self. This tells the GPS manager, "Hey, look at me, I'm the one handling the events now!"

When the phone receives a new GPS coordinate, it looks at the delegate (self) and calls your locationManager(_:didUpdateLocations:) function to deliver the data!

Argument Labels

What you are looking at above is one of Swift’s most defining syntactic features: Argument Labels (the external name used when calling the function) versus Parameter Names (the internal name used inside the function's code block).

Let's break down exactly how both of these pairs are behaving under the hood.

1. The First Parameter: _ manager:

func locationManager(_ manager: CLLocationManager, ...)
//                   ┬ ───┬───
//                   │    └─ Internal Parameter Name
//                   └────── External Argument Label (Omitted)

The External Label (_)

The underscore _ is a special wildcard symbol in Swift. It tells the compiler: "When someone calls this function, they don't need to type an argument label at all."

The Internal Name (manager)

Inside the curly braces {} of this function, you use the word manager to talk to the specific CLLocationManager object that triggered this update.

2. The Second Parameter: didUpdateLocations locations:

func locationManager(..., didUpdateLocations locations: [CLLocation])
//                             ───────┬────────── ───┬─────
//                                    │              └─ Internal Parameter Name
//                                    └──────────────── External Argument Label

The External Label (didUpdateLocations)

This is the name used from the outside perspective. It is designed to make the function read out loud like a natural, smooth English sentence when called.

The Internal Name (locations)

This is the name you use inside your code block to work with the data array. Notice that on the very next line, you type: 

lastKnownLocation = locations.first?.coordinate
//                  └───┬───┘
//                      Using the internal name!

Why does Apple double up like this?

This dual-naming convention is Apple's solution to a classic API design problem: Making code highly readable when called, but clean and concise when written.

To illustrate this, look at how the code shifts perspective between the "Outside World" and the "Inside World":

The Outside Perspective (How it reads to the system)

When the iOS operating system wants to hand your class some new GPS data, it looks for a signature that reads beautifully:

"Hey LocationFetcher, tell the locationManager that you didUpdateLocations."

The Inside Perspective (How it reads to you)

Once you are inside the function, you don't care about the historical context phrase didUpdateLocations. You just want to grab the array of data and do some math.

Because Apple split the labels, you get to work with the ultra-clean variable name locations instead of being forced to write something repetitive or clunky: 

// ❌ Clunky and repetitive without split labels:
lastKnownLocation = didUpdateLocations.first?.coordinate 

//  Clean and concise with split labels:
lastKnownLocation = locations.first?.coordinate

It gives you the best of both worlds: elegant, expressive sentences for the API architecture, and short, practical variable names for your actual logic!


Two Buttons?

QUESTION

In this proposed demonstration of the location features, there are two buttons - one of which 'starts' the location tracking. Does this need to be explicit? Can it be combined with the location access action?

ANSWER

No, it absolutely does not need to be explicit, and in a real production app, you almost never want to make the user press one button just to authorize/start tracking and a second button to actually read the data. That creates unnecessary friction.

You can absolutely combine these actions. To understand how to do it cleanly, we need to look at how iOS handles the requestWhenInUseAuthorization() method inside your locationFetcher.start() function.

How iOS Handles Authorization Behind the Scenes

When your code calls requestWhenInUseAuthorization(), iOS checks the current permission status of your app:

  • If the user has never been asked: iOS pauses your app's execution flow, pops up the native system alert asking for permission, and then immediately lets the code continue once they tap "Allow".
  • If the user already granted permission previously: iOS ignores the request completely and lets the code sail straight through instantly.

Because iOS is smart about caching that permission, you can treat your .start() function as a safe "Ensure Tracking is Active" command and run it automatically right before you read the data.

Combined Solution 1: The "All-in-One" Button

If you want a single button that triggers the authorization, starts the hardware sensors, and displays the coordinates all at once, you can compress your code down to this: 

struct ContentView: View {
    let locationFetcher = LocationFetcher()
    @State private var locationMessage = "Location unknown"

    var body: some View {
        VStack(spacing: 20) {
            Text(locationMessage)
                .font(.headline)
            
            Button("Get My Location") {
                // 1. Kick off authorization and hardware tracking
                locationFetcher.start()
                
                // 2. Immediately try to read the coordinate
                if let location = locationFetcher.lastKnownLocation {
                    locationMessage = "Lat: \(location.latitude), Lon: \(location.longitude)"
                } else {
                    // Note: The very first time a user taps this, the hardware 
                    // takes a split-second to warm up, landing here.
                    locationMessage = "Fetching GPS lock... tap again in a second."
                }
            }
            .buttonStyle(.borderedProminent)
        }
    }
}

The "First-Tap" Speed Bump (And How to Fix It)

If you use the combined button approach above, you will notice a slight quirk the very first time you test it: when you tap the button, it might say "Fetching GPS lock...", and you have to tap it a second time to see the numbers.

This happens because manager.startUpdatingLocation() runs asynchronously. It takes the physical GPS chip inside the iPhone a few hundred milliseconds to wake up, talk to satellites, and stream the coordinates back to your lastKnownLocation variable.

If you want a truly seamless app experience where the user doesn't have to push any tracking buttons, the industry standard is to trigger the setup automatically using SwiftUI's .onAppear modifier.

Combined Solution 2: Automated Activation (The Industry Standard)

By moving locationFetcher.start() into .onAppear, the prompt appears the moment the screen loads. Once authorized, the GPS chip tracks the location silently in the background, making the "Read Location" button instantly responsive on the very first tap. 

struct ContentView: View {
    let locationFetcher = LocationFetcher()
    @State private var locationMessage = "Press below to check coordinates"

    var body: some View {
        VStack(spacing: 20) {
            Text(locationMessage)
                .font(.subheadline)
            
            Button("Read Location") {
                if let location = locationFetcher.lastKnownLocation {
                    locationMessage = "Your location is \(location.latitude), \(location.longitude)"
                } else {
                    locationMessage = "GPS signal not acquired yet."
                }
            }
            .buttonStyle(.borderedProminent)
        }
        // As soon as this view appears on screen, prepare the engine
        .onAppear {
            locationFetcher.start()
        }
    }
}
