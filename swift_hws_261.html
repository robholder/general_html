Hacking With Swift - 261 - Challenge 2

Challenge 2

  • If you drag a card to the right but not far enough to remove it, then release, you see it turn red as it slides back to the center. Why does this happen and how can you fix it? (Tip: think about the way we set offset back to 0 immediately, even though the card hasn’t animated yet. You might solve this with a ternary within a ternary, but a custom modifier will be cleaner.)

Ai Insight

This is one of the trickiest bugs in the entire 100 Days of SwiftUI course because it involves a race condition between state changes and UI animations.

Your intuition about abs is very close, but the actual culprit is a sudden math flip when the card snaps back to the center. Here is exactly why it turns red, and the cleanest way to fix it.

Why It Happens (The Bug)

Look closely at how you color the background based on the drag direction: 

.fill(offset.width > 0 ? .green : .red)

When you drag the card to the right, offset.width is a positive number (e.g., +120). The ternary check sees 120 > 0, so it colors the background green. Perfect.

When you let go of the card without dragging it far enough, your .onEnded closure triggers this line instantly: 

offset = .zero

Because of your .animation(.bouncy, value: offset) modifier at the bottom, the card does not teleport back to the center; it physically slides back using a smooth spring animation.

However, the offset variable itself drops to exactly 0 the absolute millisecond you let go. As the card is physically animating through space back to the center, it re-evaluates the background color block. It checks your ternary condition: offset.width > 0. Because offset.width is now exactly 0, the condition 0 > 0 evaluates to false.

Because it's false, the ternary drops into the else block, which is .red.

⚠️ The Flash: As the card slides home, its state says its offset is 0, failing the "greater than zero" test, causing a split-second flash of red right before the animation finishes.

Solution 1: The Ternary within a Ternary (Quick Fix)

Paul mentions you can solve this with a nested ternary. If the width is exactly 0, we can default it to a neutral color (like white or clear) so it doesn't flash red or green:

Find this line in your background: 

.fill(offset.width > 0 ? .green : .red)

And replace it with this: 

.fill(offset.width == 0 ? .clear : (offset.width > 0 ? .green : .red))

Why this works: If the offset is exactly zero (which happens instantly when you release), it becomes .clear, hiding the underlying red asset entirely while it slides home.

Solution 2: A Custom View Modifier (The Clean Way)

Paul's tip mentions that a custom modifier is much cleaner than messy nested inline ternaries. We can create a dedicated modifier that handles coloring the card background defensively based on the swipe state.

Add this extension wrapper at the very bottom of your Swift file (outside the CardView struct): 

struct CardBackgroundModifier: ViewModifier {
    let offset: CGFloat

    func body(content: Content) -> some View {
        content
            .background(
                RoundedRectangle(cornerRadius: 25)
                    .fill(backgroundColor)
            )
    }

    // Read-only property to isolate the coloring math cleanly
    private var backgroundColor: Color {
        if offset > 0 {
            return .green
        } else if offset < 0 {
            return .red
        } else {
            return .clear // Safely handles exactly 0 without flashing red!
        }
    }
}

// Optional: Make it easy to call like a native modifier
extension View {
    func cardBackground(for offset: CGFloat) -> some View {
        modifier(CardBackgroundModifier(offset: offset))
    }
}

Then, look at your CardView and replace your messy inline background property block: 

// Replace this whole section:
// .background(
//     accessibilityDifferentiateWithoutColor
//     ? nil : RoundedRectangle(cornerRadius: 25)
//         .fill(offset.width > 0 ? .green : .red)
// )

// With this clean implementation:
.background(
    accessibilityDifferentiateWithoutColor 
    ? nil 
    : RoundedRectangle(cornerRadius: 25)
        .fill(offset.width == 0 ? .clear : (offset.width > 0 ? .green : .red))
)

Or if you implement the custom extension version, you can simply write .cardBackground(for: offset.width) if you prefer to decouple accessibility.

Decoupling

When your code is "coupled," it means too many different features are tangled up in the exact same line of code.

The Problem with the Original Code

Look at how many things your original .background() modifier is trying to do at the exact same time: 

.background(
    accessibilityDifferentiateWithoutColor
    ? nil : RoundedRectangle(cornerRadius: 25).fill(offset.width > 0 ? .green : .red)
)
  • It checks the accessibility settings (accessibilityDifferentiateWithoutColor).
  • It handles the layout structure (RoundedRectangle).
  • It performs the math to calculate the drag direction (offset.width > 0).
  • It assigns the color (.green or .red).

Because everything is mashed together, it turns into a massive, hard-to-read block of code right in the middle of your visual layout.

What "Decoupling" Looks Like

To "decouple" means to separate those responsibilities. We pull the complex logic out of your main body view and hide it inside a clean, reusable custom modifier.

If you use the custom modifier approach, we can shift the accessibility check inside the modifier itself.

1. The Custom Modifier (The Hidden Logic)

You would place this code at the very bottom of your file. Notice how it cleanly handles both the accessibility check and the color math in one isolated place: 

struct CardBackground: ViewModifier {
    let offset: CGFloat
    
    // We move the environment check inside here!
    @Environment(\.accessibilityDifferentiateWithoutColor) var differentiateWithoutColor

    func body(content: Content) -> some View {
        content
            .background(
                Group {
                    if differentiateWithoutColor {
                        // If accessibility is ON, don't show any background color
                        Color.clear 
                    } else {
                        // If accessibility is OFF, run our safe color logic
                        RoundedRectangle(cornerRadius: 25)
                            .fill(swipeColor)
                    }
                }
            )
    }

    // Isolated math property
    private var swipeColor: Color {
        if offset > 0 { return .green }
        if offset < 0 { return .red }
        return .clear
    }
}

// Create a nice, clean shortcut name for it
extension View {
    func cardFeedbackBackground(for offset: CGFloat) -> some View {
        modifier(CardBackground(offset: offset))
    }
}

2. The Main View (The Clean Result)

Now that the logic is decoupled, look how incredibly clean your actual CardView body becomes. You can delete that giant block of nested ternary code completely and replace it with just one line: 

RoundedRectangle(cornerRadius: 25)
    .fill(.white.opacity(1 - Double(abs(offset.width / 50))))
    .cardFeedbackBackground(for: offset.width) // 👈 Everything is handled here now!
    .shadow(radius: 10)

Why do this?

By "decoupling" it, your main UI layout reads like a simple list of instructions, and your complex logic is safely tucked away where it won't clutter your sight.

Both tricks eliminate that annoying false-negative check when the card structural value snaps to zero!
