Hacking With Swift - 179 - How to let the user share content with ShareLink

SwiftUI's ShareLink view lets users export content from our app to share elsewhere, such as saving a picture to their photo library, sending a link to a friend using Messages, and more.

We provide the content we want to share, and iOS takes care of showing all the apps that can handle the data we're sending. For example, we can share a URL like this: 

ShareLink(item: URL(string: "https://www.hackingwithswift.com")!)

That's going to make a button saying "Share" with an icon attached to it, and pressing it will bring up the iOS share sheet. If you're in the simulator you'll only see a few things there as samples, and some might not even work, but if you use a real device you'll see you can share that URL just fine.

If you want more control over the data, you have several options.

First, you can attach a subject and message to the shared data like this: 

ShareLink(item: URL(string: "https://www.hackingwithswift.com")!, subject: Text("Learn Swift here"), message: Text("Check out the 100 Days of SwiftUI!"))

Refactored to multiline: 

ShareLink(
    item: URL(
        string: "https://www.hackingwithswift.com"
    )!,
    subject: Text(
        "Learn Swift here"
    ),
    message: Text(
        "Check out the 100 Days of SwiftUI!"
    )
)

How that information is used depends on the app users share to – the URL will always be attached, because that's the most important thing, but some apps will use the subject, some the message, and others will use both.

Second, you can customize the button itself by providing whatever label you want: 

ShareLink(item: URL(string: "https://www.hackingwithswift.com")!) {
    Label("Spread the word about Swift", systemImage: "swift")
}

Refactored to multiline: 

ShareLink(
    item: URL(
        string: "https://www.hackingwithswift.com"
    )!
) {
    Label(
        "Spread the word about Swift",
        systemImage: "swift"
    )
}

And third, you can provide a preview to attach, which is important when you're sharing something more complex – it's possible to share entirely custom data here, so the preview is helpful for giving the recipient some idea of what's inside.

Annoyingly, this is needed even for data that is its own preview, such as an image. To avoid making your code repetitive, I'd suggest assigning the image to a local constant then using that: 

let example = Image(.example)

ShareLink(item: example, preview: SharePreview("Singapore Airport", image: example)) {
    Label("Click to share", systemImage: "airplane")
}

Refactored to multiline: 

let example = Image(
    .example
)

ShareLink(
    item: example,
    preview: SharePreview(
        "Singapore Airport",
        image: example
    )
) {
    Label(
        "Click to share",
        systemImage: "airplane"
    )
}

Letting users share data from your app is really important, because without it your data can feel quite isolated from the rest of their life!

Ai Insight

The Exclamation Mark

In this line... 

ShareLink(item: URL(string: "https://www.hackingwithswift.com")!) {
    Label("Spread the word about Swift", systemImage: "swift")

That exclamation mark is the Force Unwrapping operator (often called the "bang" operator in programming).

In Swift, it is the coding equivalent of telling the compiler: "I know for an absolute fact that this item is not empty, so rip open the safety packaging and use it right now."

To understand why it is there, we have to look at what happens when you create a web address from a piece of text.

The Problem: Not All Text is a Valid URL

When you write URL(string: "..."), you are passing a piece of plain text into Apple's URL builder.

Because text can contain typos, spaces, or illegal characters, the URL builder might fail. For example, if you accidentally typed URL(string: "this is not a website"), Apple's system wouldn't be able to turn that into a real internet address.

Because it might succeed or it might fail, the URL(string:) initializer returns an Optional URL (written in Swift as URL?). Think of an Optional as a sealed cardboard box that either contains a valid URL or contains nothing (nil).

Why the Exclamation Mark is Necessary Here

SwiftUI’s ShareLink is strict. It refuses to accept a mystery cardboard box; it demands a real, guaranteed web address to show the share sheet.

By placing the ! at the very end of the line, you are forcefully ripping open that optional box: 

URL(string: "https://www.hackingwithswift.com")!
                                              ▲
                                    "Trust me, it's in here!"

Because Paul Hudson hardcoded a perfectly valid web address ("[https://www.hackingwithswift.com](https://www.hackingwithswift.com)"), there is a 100% chance the box contains a valid URL. The exclamation mark tells Swift, "Don't make me write extra safety code to check this; I know it's valid."

The Danger of the Exclamation Mark

While convenient for hardcoded links, the force unwrap operator comes with a massive warning label: If the box turns out to be empty, your app will instantly crash.

If you ever generate URLs dynamically - such as using text that a user typed into a TextField - you should never use the exclamation mark. Instead, you use a safe unwrap method like if let or guard let: 

// ❌ DANGEROUS: If the user types a space or emoji, the app crashes instantly!
let userURL = URL(string: userInputText)! 

//  SAFE: If the text is a bad link, the app handles it gracefully without crashing.
if let safeURL = URL(string: userInputText) {
    ShareLink(item: safeURL)
} else {
    // Show an alert saying "Invalid web address"
}

Summary

Use the ! for hardcoded, static strings that you know are perfect URLs. For everything else, stick to your guard let or if let safety nets!

OK, But Why not Omit the Exclamation Mark?

If you write it without the exclamation mark like this: 

let userURL = URL(string: userInputText)

The code will compile perfectly fine on that specific line. However, the catch is that userURL is no longer a standard URL. It is now an Optional URL (written as URL? in Swift).

Because it is an Optional, Swift wraps a safety layer around it. The moment you try to actually use that variable later in your code - like passing it to a ShareLink or trying to load a webpage with it - the compiler will stop you with a red error message:

❌ Value of optional type 'URL?' must be unwrapped to a value of type 'URL'

The Cardboard Box Analogy

Think of an Optional as a sealed cardboard box.

  • URL(string: ...)! rips the box open immediately. You get the raw URL inside, but if the box was empty (nil), the app explodes (crashes).
  • URL(string: ...) leaves the box sealed. It is perfectly safe, but you can't actually look at the web address or use it while it's locked inside the box.

How to use it safely without crashing - Unwrapping the Optional

To use the safe version, Swift forces you to "unwrap" the box first using a conditional statement. This proves to the compiler that you have a plan in place just in case the user typed complete nonsense.

There are two standard, safe ways to do this:

Approach 1: The if let strategy (Local Safety)

This opens the box. If there is a real URL inside, it temporarily assigns it to safeURL and runs the code inside the brackets. If the box is empty, it skips it entirely. 

let userURL = URL(string: userInputText) // This is an Optional (URL?)

if let safeURL = userURL {
    // This code ONLY runs if the URL is valid. 
    // 'safeURL' is a regular URL, so ShareLink is happy!
    ShareLink(item: safeURL)
} else {
    // Run this if the user typed an invalid link
    print("That wasn't a real website address!")
}

Approach 2: The guard let strategy (Early Exit)

This is preferred if the rest of your entire function requires a valid URL to proceed. It checks the box right away; if it's empty, it exits the function immediately. 

    guard let safeURL = URL(string: userInputText) else {
        // Stop right here if the URL is bad
        return 
    }
    
    // Everything below this point can safely use 'safeURL' as a guaranteed, non-optional URL
    print("Loading \(safeURL.absoluteString)...")
}

Summary

You can absolutely drop the exclamation mark - and in real-world production apps, you should drop it for anything a user types! You just have to add an if let or guard let check right afterward before you try to hand that URL over to SwiftUI views.
