Better Commit Messages

Tips to construct Commit Messages

  • Try to avoid the '-m' option:
    • Use the pop up editor
  • Use 'imperative mood' (as if commanding the computer):
    • 'feat: update style names'
  • Use Header (maybe all lowercase):
    • Followed by a Gap line (leave a gap line between header and body)
    • Reason: to utilize --oneline flag
  • Body:
    • Bullet points - (or * )
    • Body text (use regular punctuation and case)
  • Why - focus on 'why', not 'how' or 'what'
  • Use 'types', perhaps...
    • 'build' - set up and builds
    • 'chore' - changes that affect Git tooling
    • 'docs' - documentation change
    • 'feat' - new feature or enhancement
    • 'fix' - but fixes
    • 'perf' - performance edit
    • 'refactor' - refactor edit
    • 'test' - test additions
  • Add ticket reference if applicable:
    • '(#6130)'

Structure

type: description
gap
optional body text

# Example:

feat: add cancel button

- Cancel button to close panel
- Custom style

If you need to change a commit message, run git commit --amend. 

# To amend the commit of the last commit:
git commit --amend -m "amended message"

Index Organization

With the idea of commit 'types' in mind, think about the batch of files to 'stage' (put into Index) before a commit.

Branch Names

Just like commits, why not structure 'branch' names?

And each branch should be for one task only! 

<initials>/<ticket #>/<Imperative Mood description>

Example:

rh/1512/remove-typos-in-documentation

Reminder!

You can rename branches using the '--move' or '-m' flag. 

# To rename a branch from the branch:
git branch -m <my_new_branch_name>


# To rename a branch without switching (from another branch):
git branch -m <old_name> <new_name>
