Hacking With Swift - Challenge Day 1

Your challenge

Build an app that handles unit conversions: users will select an input unit and an output unit, then enter a value, and see the output of the conversion.

Which units you choose are down to you, but you could choose one of these:

  • Temperature conversion: users choose Celsius, Fahrenheit, or Kelvin.
  • Length conversion: users choose meters, kilometers, feet, yards, or miles.
  • Time conversion: users choose seconds, minutes, hours, or days.
  • Volume conversion: users choose milliliters, liters, cups, pints, or gallons.

Example

If you were going for length conversion you might have:

  • A segmented control for meters, kilometers, feet, yard, or miles, for the input unit.
  • A second segmented control for meters, kilometers, feet, yard, or miles, for the output unit.
  • A text field where users enter a number.
  • A text view showing the result of the conversion.

So, if you chose meters for source unit and feet for output unit, then entered 10, you’d see 32.81 as the output.

You know almost everything you need to build this. In fact, if you go for a simple solution – and I hope you do – the only actually new thing you’ll find is that you need to use a different format option for your text field, because .currency and .percent don’t make sense here. You should choose .number instead, which is suitable for standard numerical input.

Tips

You know everything you need to complete this project, but if you’re hitting problems then I have some tips that might help.

First, and most importantly: keep it simple! You’ll have lots of time in your future career to think up clever coding architectures and more, but in this initial challenge I want you to go for the simplest solution you can find.

Second, all our unit conversions are simple mathematics, but you shouldn’t try to write conversions to go from every source unit to very other unit. A better idea is to convert the user’s input to a single base unit, then convert from there to the target unit.

So, rather than writing code to convert from liters to milliliters, and from liters to cups, and from liters to pints, and so on, a better idea is to convert the user’s input into milliliters (the lowest common denominator), then convert from there to whatever output unit they want.

Third, as we have three user values here – their input number, their input unit, and their output unit – you need to have three `@State properties to store them all. You’ll need a TextField`, two pickers, and a text view to show your output, and that’s about it. You can break your form up into sections if you want, but it’s not required.

Fourth, the easiest way to store your conversion units is to have a simple string array, which you can loop over using something like ForEach(units, id: \.self).

Finally, if you’d like to be a bit fancy with the output number you display, try calling .formatted() on it – e.g. someDouble.formatted(). This will cause iOS to format the number for printing, adding thousands separator and removing a lot of the noise from unnecessary decimal places.

Go ahead and get started now. Don’t worry if you need to refer back to the WeSplit project – that’s perfectly normal, and is all part of the learning process.

Good luck! You can do this. And once you’re done, tell other people: you’ve built another SwiftUI app, and this one was entirely designed by you.

You should be proud of what you’ve accomplished.

Tip: Even though you can absolutely complete this project using simple arithmetic, you might be interested to know that Apple does give us dedicated functionality for doing unit conversion – see my article:

  • How to convert units using Unit and Measurement

...if you’re curious. However, I should reiterate that it is not required to complete this project: you can convert from liters to pints (for example) just by multiplying the input by 2.11338.

ANSWER - Gemini said

This is a great moment to simplify your logic. In SwiftUI, we often try to avoid creating extra "middle-man" variables like @State private var fromCelsius.

Since inTempScale already holds the value "ºC" or "ºF", you don't need a separate Boolean to track it. You can just check that string directly inside your computed variables.

The "Computed Logic" Fix

Instead of trying to manually set fromCelsius and toCelsius (which would require complex observers), just update your variables to check the scale on the fly: 

// Change your computed variables to this:

var convertedTemp: Double {
    let input = inTemp ?? 0
    
    // 1. Convert everything to a "Base" (Celsius)
    let tempInCelsius: Double
    if inTempScale == "ºC" {
        tempInCelsius = input
    } else {
        // Fahrenheit to Celsius formula: (F - 32) * 5/9
        tempInCelsius = (input - 32) * 5 / 9
    }
    
    // 2. Convert from the "Base" to the desired output
    if outTempScale == "ºC" {
        return tempInCelsius
    } else {
        // Celsius to Fahrenheit formula: (C * 9/5) + 32
        return (tempInCelsius * 9 / 5) + 32
    }
}

Why this is better than using Booleans

  • Automatic Updates: Because convertedTemp is a computed property, SwiftUI watches inTempScale and outTempScale. Whenever the user taps the picker, SwiftUI sees the change and recalculates the math automatically. You don't have to write any "if picker changed, set bool" code.

Cleaning up your celsiusToFahrenheit function

Your current function uses convertedStart inside it, which is a bit circular. It’s cleaner to make it a "Pure Function" that just takes a number and returns a number: 

func celsiusToFahrenheit(_ temp: Double) -> Double {
    return (temp * 9 / 5) + 32
}

func fahrenheitToCelsius(_ temp: Double) -> Double {
    return (temp - 32) * 5 / 9
}

Final Look at the Logic Flow

  • User types 100 in the TextField (inTemp).
  • User selects ºF for input and ºC for output.
  • convertedTemp fires:
    • It sees inTempScale is ºF, so it runs the (100 - 32) * 5/9 math.
    • tempInCelsius becomes 37.77.
    • It sees outTempScale is ºC, so it returns 37.77 directly.
  • The Text at the bottom updates instantly.

One Wit-y Observation

If you plan to add Kelvin or Rankine later, this "Convert to Base" (Celsius) method is a lifesaver. You just convert any input to Celsius first, then convert that Celsius to whatever the user wants. It prevents you from having to write a formula for every possible combination (F to K, K to F, etc.)!

QUESTION

I like the streamlined example, but I don't see how the 'Cleaning up your celsiusToFahrenheit' section fits in (since the code above seems to make them obsolete). Where should I consider these functions.

ANSWER - Gemini said

You’ve got a sharp eye for clean code. You are exactly right: in that specific "streamlined" example, those functions are technically obsolete because the math is simple enough to do inline.

However, as your app grows, you’ll find that "Inline Math" eventually becomes a mess. Here is where and why you should consider using those functions instead of the raw formulas.

1. Readability (The "Self-Documenting" Code)

When you look at (input - 32) * 5 / 9, your brain has to pause for a microsecond to remember, "Wait, is that Celsius to Fahrenheit or the other way around?"

When you use a function, the code reads like English: 

let tempInCelsius: Double
if inTempScale == "ºC" {
    tempInCelsius = input
} else {
    tempInCelsius = fahrenheitToCelsius(input) // Much clearer!
}

2. The "Single Point of Truth"

Imagine you find a bug in your math, or you decide to use a more precise version of the formula (e.g., using 1.8 instead of 9/5).

  • Inline: You have to hunt through your convertedTemp variable and change it in multiple places.
  • Functions: You change it once inside the function, and every part of your app is instantly fixed.

3. Where to put them?

In Swift, you have two great options for where these functions "live":

Option A: Inside the Struct (Private Helpers)

Put them at the bottom of your ContentView, just above the body. This keeps them "local" to this screen. 

private func fahrenheitToCelsius(_ temp: Double) -> Double {
    (temp - 32) * 5 / 9
}

Option B: A "Global" Utility (The Pro Way)

If you plan to have a "Weather" tab and a "Cooking" tab later, you don't want to copy-paste those functions. Create a new Swift file (Cmd + N) called TemperatureMath.swift and put them there. They will be available to your entire app.

Refined Logic using the Functions

Here is how that convertedTemp looks using the functions. Notice how it separates the Logic (what to do) from the Math (how to do it): 

var convertedTemp: Double {
    let input = inTemp ?? 0
    
    // Step 1: Normalize to Celsius
    let tempInCelsius = (inTempScale == "ºC") ? input : fahrenheitToCelsius(input)
    
    // Step 2: Convert to Target
    return (outTempScale == "ºC") ? tempInCelsius : celsiusToFahrenheit(tempInCelsius)
}

// MARK: - Calculation Helpers
func fahrenheitToCelsius(_ temp: Double) -> Double {
    (temp - 32) * 5 / 9
}

func celsiusToFahrenheit(_ temp: Double) -> Double {
    (temp * 9 / 5) + 32
}

A Quick Swift Syntax Tip

Notice I removed the word return inside the functions. In Swift, if a function is only one line long, it automatically returns the result of that line. It’s a nice way to keep your "Math" section looking very lean.

QUESTION

What's the significance of making a private function? I notice your last code block didn't use them.

ANSWER

In Swift, the private keyword is all about Encapsulation. Think of it like the "Internal Affairs" of your code—it defines who is allowed to see and use a specific function or variable.

When you mark a function as private inside a struct or class, you are telling the rest of your app: "This function is for my internal use only. Nobody else needs to worry about how I’m doing this math."

The Significance of private

Feature Internal (Default) Private
Visibility Any file in your app can call it. Only the code inside the same struct can see it.
Autocomplete Shows up everywhere in Xcode. Only shows up when you are typing inside that specific file.
Safety Someone could accidentally use it elsewhere. Guarantees that if you change the math, you only have to check one file.

Why my last code block didn't use them

I left them "Internal" (the default in Swift) so they would be easier for you to move around. If you decide to move those functions to a separate file (like TemperatureMath.swift), they cannot be private, or your ContentView won't be able to see them!

The Rule of Thumb:

  • Use private if the function is a "helper" that only makes sense inside that one specific View (like a special way to format a string just for that screen).
  • Use Default (no keyword) if you want to be able to use that logic in other parts of your app later.

Why "Vimmers" Love Private Functions

As your projects get bigger, your Autocomplete list (Ctrl + N or Cmd + .) gets cluttered.

  • If you have 500 functions in your app and they are all "Internal," searching for the right one is a nightmare.
  • By marking things private, you keep your "Global Space" clean. You only see what you actually need to see.

The "Clean Code" approach for your App

Since temperature math is something you might want to use in a "History" view or a "Settings" view later, keeping them Internal (without the private keyword) is actually the smarter move for now. It keeps your options open.

Pro Tip: In SwiftUI, we almost always make our @State variables private (e.g., @State private var inTemp). This is because those variables are "Owned" by that specific view's UI, and letting another part of the app touch them directly would cause major "State" bugs!

QUESTION

Your last question reminded me of something I wanted to ask. Right now, the result (conversion) shows '32' since the input field is blank, and I guess a zero input results in 32 ºF. That makes sense, but can we make the SwiftUI code leave the result blank if there are no actual figures input into the user field?

ANSWER

That is a great intuition for User Experience (UX). In a professional app, showing a "0" or "32" when the user hasn't even typed anything yet can feel like the app is "guessing" or, worse, giving a "false" result.

Since you defined inTemp as an Optional (Double?), you have the perfect tool to handle this. An optional is either a value or it is nil (nothing).

The "Optional Logic" Fix

You can update your convertedTemp variable to return an Optional Double as well. This way, the math only happens if there is something to calculate.

Update your convertedTemp like this: 

var convertedTemp: Double? { // Change return type to Double?
    // 1. Check if we actually have an input. If not, return nil immediately.
    guard let input = inTemp else { return nil }
    
    // 2. Perform the "Base" conversion
    let tempInCelsius = (inTempScale == "ºC") ? input : fahrenheitToCelsius(input)
    
    // 3. Perform the "Target" conversion
    return (outTempScale == "ºC") ? tempInCelsius : celsiusToFahrenheit(tempInCelsius)
}

Updating the UI to handle "Nothing"

Now that convertedTemp can be nil, your Text view needs to know what to display when there's no result. You can use Nil Coalescing (??) or an if let statement right in your SwiftUI body.

Option A: The "Placeholder" Text (Cleanest)

This will show the number if it exists, or the word "---" if it doesn't. 

Section("Converted Temperature") {
    // ... your picker ...
    
    // If convertedTemp is nil, show "---"
    Text(convertedTemp?.formatted(.number) ?? "---")
        .font(.title) // Make the result pop!
}

Option B: Hiding the result entirely

If you don't want to see "32" or "---", you can wrap the text in an if statement: 

Section("Converted Temperature") {
    // ... your picker ...
    
    if let result = convertedTemp {
        Text(result, format: .number)
    } else {
        Text("Waiting for input...")
            .foregroundColor(.secondary)
    }
}

Why guard let is your friend

In the code above, I used guard let input = inTemp else { return nil }.

  • In English: "Look at inTemp. If it's a real number, call it input and keep going. If it's empty (nil), stop right here and return 'nothing'."
  • Why it's better than Python's if: It "unwraps" the optional safely. Inside the rest of that function, input is a regular Double, so you don't have to keep checking if it's nil.

A "Wit-y" UI Detail

By using Double? = nil as your starting state, the very first time the user opens the app, the "Converted" section will be empty or show your placeholder. It looks much more "premium" than an app that defaults to 0.0.

ANSWER

To insert this, you’ll replace the static Text(convertedTemp, format: .number) line you had earlier. SwiftUI is designed to handle this kind of "conditional views" inside a Section block perfectly.

Here is exactly where it goes in your ContentView.swift: 

Section("Converted Temperature") {
    Picker("Converted Temperature", selection: $outTempScale) {
        ForEach(tempUnits, id: \.self) {
            Text($0)
        }
    }
    .pickerStyle(.segmented)

    // --- INSERT THE BLOCK HERE ---
    if let result = convertedTemp {
        // This only shows up if convertedTemp is NOT nil
        Text(result, format: .number)
            .font(.largeTitle) // Optional: make it look like a result!
    } else {
        // This shows up when the input field is empty
        Text("Waiting for input...")
            .foregroundColor(.secondary)
            .italic()
    }
    // ------------------------------
}

What’s happening under the hood?

Because convertedTemp is now a computed property that returns an optional (Double?), SwiftUI re-runs the logic inside your body every time the user types or changes a picker.

  1. Initial State: inTemp is nil. convertedTemp hits that guard let and returns nil. SwiftUI sees the else block and displays "Waiting for input...".
  2. User Types "10": inTemp becomes 10.0. convertedTemp calculates the result. The if let result = convertedTemp check succeeds, and SwiftUI swaps the "Waiting..." text for the actual number.

By adding the else block with .foregroundColor(.secondary), you’ve moved from a "functional" app to a "polished" one. It gives the user a clear hint that the app is active and waiting for them, rather than just showing a static 0 that might look like a bug.

QUESTION

I've just thought of two things I'd like to change on the temperature interface:

  1. Restrict results (conversion) to 2 decimal places.
  2. Have the user input field become blank if re-entered for another figure input. (Right now, the input has "Enter Temp Value" before entering values. I'd like that to appear the same if the field is re-activeated for another value input.

ANSWER

Those are two excellent "quality of life" improvements that will make your app feel much more like a finished product. Both involve slightly more advanced ways of using SwiftUI’s TextField and FormatStyle.

1. Restricting to 2 Decimal Places

In SwiftUI, the .number format is very powerful, but it defaults to showing as many digits as it needs. To force exactly two decimal places, you modify the format parameter of the Text view where the result is displayed.

Change your result Text to this: 

Text(result.formatted(.number.precision(.fractionLength(2))))
  • .precision(.fractionLength(2)): This tells SwiftUI, "I only want to see two digits after the decimal point." It will also handle rounding for you (e.g., 10.556 becomes 10.56).

2. Clearing the Field on Re-entry

This is a bit of a "pro" move. SwiftUI doesn't have a simple toggle for "clear on edit," but we can achieve this using the @FocusState property wrapper. This allows the app to know exactly when the user has clicked into (focused) the text field.

Step A: Add the FocusState at the top

Add this line near your other @State variables: 

@FocusState private var isInputFocused: Bool

Step B: Attach it to the TextField

Add the .focused modifier to your TextField: 

TextField("Enter Temp Value", value: $inTemp, format: .number)
    .keyboardType(.decimalPad)
    .focused($isInputFocused) // <--- Add this

Step C: Clear the value when focus changes

Use the .onChange modifier at the bottom of your Form (or on the TextField itself) to detect when the user clicks the field: 

.onChange(of: isInputFocused) {
    if isInputFocused {
        inTemp = nil // This clears the field the moment they tap it
    }
}

How the Updated Section Looks

If you want the result to only show decimals if they are needed (e.g., show 32 instead of 32.00), you can use: .precision(.fractionLength(0...2))

This tells SwiftUI: "Show between 0 and 2 decimal places depending on the number."

QUESTION

If I was expanding this app to take into account more temperature scales, this wouldn't be valid, but since I'm not planning to do that, is there a way of making the two pickers mutually exclusive; so that they toggle each other when they are changed? For example if the input is changed, the output would change and visa versa?

ANSWER

That is a very logical "UI shortcut" for a two-unit app! If there are only two choices, it feels redundant to have to tap both pickers. If I'm changing the input to Celsius, I almost certainly want the output to be Fahrenheit.

In SwiftUI, we can handle this by "watching" the state variables and reacting when they change.

The .onChange Solution

You can attach an .onChange modifier to your Form. This modifier acts like a "listener" that fires a small block of code whenever a specific variable is modified.

Here is how you would implement the "Mutual Exclusion" logic: 

Form {
    // ... your sections ...
}
.onChange(of: inTempScale) { oldValue, newValue in
    // If Input becomes Celsius, force Output to Fahrenheit
    if newValue == "ºC" {
        outTempScale = "ºF"
    } else {
        outTempScale = "ºC"
    }
}
.onChange(of: outTempScale) { oldValue, newValue in
    // If Output becomes Celsius, force Input to Fahrenheit
    if newValue == "ºC" {
        inTempScale = "ºF"
    } else {
        inTempScale = "ºC"
    }
}

Why this works

  1. User Taps "ºC" on the Input Picker: The variable inTempScale updates.
  2. The Listener Fires: The first .onChange sees the change and instantly updates outTempScale to "ºF".
  3. The UI Updates: Because outTempScale is a @State variable, the bottom Picker physically slides to the other side automatically.

A "Wit-y" Warning: The Infinite Loop?

You might wonder: "If changing Input changes Output, won't that change to Output then trigger the Input change again, creating an infinite loop?"

SwiftUI is smart enough to handle this. It checks if the value actually changed. If outTempScale was already "ºF", setting it to "ºF" again won't trigger the second .onChange, so the "ping-pong" effect stops immediately.

"One-Line" Pro Version

If you want to keep your code extra lean (and impress your future self), you can write those if/else statements as single lines using the Ternary Operator: 

.onChange(of: inTempScale) { _, newValue in
    outTempScale = (newValue == "ºC") ? "ºF" : "ºC"
}
.onChange(of: outTempScale) { _, newValue in
    inTempScale = (newValue == "ºC") ? "ºF" : "ºC"
}

*(Note: In Swift, the _ in the closure just means "I don't care what the old value was, just give me the new one.")*

Where to put it?

The best place to put these .onChange modifiers is right after the closing brace of your Form { ... }, but before the .navigationTitle.

ANSWER

Ah, the classic SwiftUI "Compiler Conundrum." You’ve just discovered one of the most frustrating quirks of the language: The Type-Check Timeout.

Strictly speaking, your second code block is *syntactically valid*. In an ideal world, moving .navigationTitle to the bottom of that chain shouldn't change anything. However, Swift is a "Statically Typed" language, meaning the compiler tries to calculate the exact "Type" of your view by looking at every single modifier in the chain.

Why the error happens when you move it

When .navigationTitle is at the *top*, the compiler sees:

  1. A Form.
  2. A Title attached to it (Simple).
  3. Some logic listeners (.onChange) attached after that.

When you move it to the *bottom*, you are asking the compiler to:

  1. Look at the Form.
  2. Apply three complex logic closures (.onChange) that modify other state variables.
  3. Then apply a Title to the "Result" of all those logic closures.

Because each .onChange adds a layer of complexity to the "Type" definition, the compiler basically gets a "math headache." It’s trying to verify that the view coming out of the third .onChange is still a valid view that can accept a .navigationTitle, and it runs out of its allotted "thinking time."

The Fix (The "Clean Code" Way)

To stop the compiler from timing out, we want to make the "Chain" as simple as possible.

*Keep the .navigationTitle at the top.* It is standard practice in SwiftUI to put "Structural" modifiers (like Titles and Toolbars) first, and "Behavioral" modifiers (like .onChange) last.

One Wit-y Observation

The Swift compiler is like a very brilliant, very exhausted librarian.

  • If you say, "Give me the book Convert Temperature and then add these three sticky notes to the back," it’s fine.
  • If you say, "Take this stack of papers, apply these three complex logic puzzles to them, and then tell me if this pile of paper is a book called Convert Temperature," the librarian throws the papers in the air and quits.

Summary of your Logic

Since your code works perfectly with the title at the top, stick with that! It’s actually better for readability anyway. You want someone (or "Future You") to see the *Name* of the screen immediately, before they have to scroll past all the .onChange logic.

QUESTION

Here's another thing I'd like to improve. First I reduce the temperature decimal point accuracy from 2 to 1 place. But I notice a new issue. If the value has a zero as the decimal fraction, I'd like it to be omitted completely. e.g., 104.1 is ok, but when it's 104.0, I'd like it to show '104'. Is this possible?

ANSWER

Yes, this is absolutely possible! SwiftUI’s .number format has a specific way to handle "flexible" precision. You want to tell the app: "You can have up to 1 decimal place, but if it's a zero, don't show it at all."

The "Flexible Precision" Fix

Instead of forcing a fixed length, you use a *range* for the fraction length.

Update your result Text to this: 

Text(result.formatted(.number.precision(.fractionLength(0...1))))

How it works:

  • 0...1: This is a Swift Range. It tells the formatter: "The minimum number of decimal places is 0, and the maximum is 1."
  • The Logic: If the result is 104.1, it uses the 1 allowed decimal. If the result is 104.0, the formatter realizes it can satisfy the "minimum of 0" requirement without showing the zero, so it truncates it to 104.

One Wit-y Observation

You have just discovered "The Principle of Least Astonishment." In UI design, users are "astonished" (and annoyed) when they see 104.0 because it implies a level of precision that isn't actually adding new information. By using 0...1, you’re making the app "smarter"—it only speaks when it has something meaningful to say (like that .1).

A Quick Recap of your "Polished" Result Section

Your "Converted Temperature" section now looks like a high-end calculator: 

Section("Converted Temperature") {
    // ... Picker ...

    if let result = convertedTemp {
        // Only shows .1 if it's not a zero!
        Text(result.formatted(.number.precision(.fractionLength(0...1))))
            .font(.largeTitle)
    } else {
        Text("Waiting for input...")
            .foregroundColor(.secondary)
    }
}
