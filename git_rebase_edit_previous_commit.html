Rebase - Editing a previous commit (back in History)

AI Guide

QUESTION

I want to edit a commit in git.

This is my git (comment) history: 

latest (HEAD)
previous - but needs changing
another
earlier

...I want to edit 'previous - but needs changing', but leave 'latest' unchanged and the most recent commit.

Is there any way to achieve this?

ANSWER

Yes, there is a standard way to do this using Interactive Rebase. It’s essentially a way to rewrite history by pausing the timeline at a specific point, letting you make your changes, and then "replaying" the subsequent commits on top of that change.

The Game Plan

Since you want to reach back one step behind your current HEAD, you will use the command: 

git rebase -i HEAD~2

This tells Git: "I want to look at the last 2 commits and interactively decide what to do with them."

  • Using HashNumber - Commit Number

Step-by-Step Execution

1. Start the Rebase

Run the command above. Your default text editor (likely Vim or Nano) will open with a list of commits that looks like this: 

pick e1a2b3c another
pick f4g5h6i previous - but needs changing
pick j7k8l9m latest (HEAD)

2. Change the Instruction

Change the word pick to edit (or just e) next to the commit you want to fix. Leave latest as pick. 

pick e1a2b3c another
edit f4g5h6i previous - but needs changing
pick j7k8l9m latest (HEAD)

Save and close the editor.

3. Make Your Changes

Git will now "travel back in time." You are currently sitting at the previous commit.

  • Modify your files to fix whatever was wrong.
  • Stage the changes: git add <filename>
  • Update the commit: git commit --amend (You can also change the commit message here if you want with the -m flag).

4. Return to the Present

Once the edit is done, tell Git to finish the process: 

git rebase --continue

Git will take your modified previous commit and then try to "tack" the latest commit back on top of it.

A Word of Caution

Because you are changing a commit that has already happened, you are technically rewriting history.

  • If you haven't pushed yet: You're golden. No one will ever know.
  • If you have already pushed: You will need to use git push --force to update the remote server. Be careful doing this if you are working on a team, as it can mess up your colleagues' local histories.

Errors After Edits

QUESTION

I tried to edit a commit (using git rebase -i HEAD~4 etc) and after editing and running: 

git rebase --continue

I get... 

Auto-merging SwiftDataProject/ContentView.swift
CONFLICT (content): Merge conflict in SwiftDataProject/ContentView.swift
error: could not apply 8c13ec7... Filter #Predicate example - 'logical and'
hint: Resolve all conflicts manually, mark them as resolved with
hint: "git add/rm <conflicted_files>", then run "git rebase --continue".
hint: You can instead skip this commit: run "git rebase --skip".
hint: To abort and get back to the state before "git rebase", run "git rebase --abort".
hint: Disable this message with "git config set advice.mergeConflict false"
Could not apply 8c13ec7... # Filter #Predicate example - 'logical and'

What is wrong, and how should I proceed?

ANSWER

Don't worry - this is the most intimidating part of Git, but it's actually very logical once you see what’s happening.

What went wrong?

When you "edited" the previous commit to remove the duplicate button, you changed the file. Now, when Git tries to move to the next commit in your history (8c13ec7), it’s confused. It’s trying to apply the changes from that next commit onto a file that no longer looks the way it expected.

Think of it like this: You changed page 5 of a book, and now Git is struggling to figure out how to glue the original page 6 onto your newly edited page 5.

How to proceed (Step-by-Step)

1. Find the Conflict

Open the conflicted file(s). You will see some very ugly "conflict markers" that Git inserted. They look like this: 

<<<<<<< HEAD
// This is the "Clean" version you just edited
=======
// This is the version from the next commit (8c13ec7) 
// that still has the double-button or conflicting logic
>>>>>>> 8c13ec7... Filter #Predicate example

2. Clean it up

You need to manually delete the markers (<<<<<<<, =======, >>>>>>>) and keep only the code that is correct.

  • Look closely at the code.
  • Fix the issues (usually deleting one section or the other).

3. Tell Git you fixed it

Once the file(s) look perfect, go back to your terminal and run: 

git add <files>

..or 

git add .

4. Continue the journey

Now that you’ve "resolved" the conflict for this specific stop in time, tell Git to keep going: 

git rebase --continue

What if it happens again?

If you have 4 commits and you edited the first one, Git might stop and ask for help at each subsequent commit.

  • If it stops again: Fix the code again → git add . → git rebase --continue.
  • If you get totally overwhelmed and things feel broken, remember your Emergency Exit:
    • git rebase --abort

Why did this happen?

This happened because your subsequent commits also contained changed code (or were based on it). By removing it in an early commit, you created a "divergence." Git is just being cautious and asking you: "Hey, since you changed the past, how should I handle these specific lines in the future commits?"

Pro Tip: This is exactly why we suggest safety-backup branch later down the page! You can take risks here knowing that your original code is safe elsewhere.

Aborting Edit

QUESTION

If I decide NOT to edit the file after all, how do I revert to how we were before: git rebase -i HEAD~2

ANSWER

If you haven't finished the rebase yet and you're feeling that "I've made a huge mistake" panic - don't worry! Getting out of a rebase is actually very straightforward.

1. If you are still in the text editor (where it says pick or edit):

Just close the editor without saving, or delete all the text in the file and save an empty file. Git will see the empty file and say "Nothing to do," cancelling the rebase.

2. If you have already started the rebase (Git is waiting for you to fix a commit):

Run this command in your terminal: 

git rebase --abort

This is the "Emergency Exit" button. It tells Git to throw away everything it just tried to do and put your history exactly back to the way it was before you typed the rebase command.

How to tell if you are "in" a rebase

If you aren't sure if you're still mid-rebase, look at your terminal prompt. Many terminals will show something like (REBASE 1/2) or (master|REBASE-i). You can also type: 

git status

If it says "You are currently editing a commit while rebasing," then the --abort command is your best friend.

Why sometimes it's better NOT to edit history

Interactive rebasing is powerful, but it's also "destructive" because it changes commit timestamps and IDs (hashes). If the code is just for your own learning:

  • Just make a new commit called "Cleanup" where you fix the line.
  • Move on.

In the real world, developers often value a "clean history," but they value not breaking the repo even more. If the rebase feels like it's becoming a headache, just abort and make a fresh commit with the fix.

Rebase to a Hash Number

You can rebase using a Commit Hash, and it is often safer than counting with HEAD~n because it’s precise and doesn't require you to manually count back through your history.

How it works

When you use a hash, you are telling Git: "I want to rewrite everything that happened after this specific commit."

The command looks like this: 

git rebase -i <commit-hash>

The "Target" Rule (Crucial!)

There is one detail that trips people up: The commit hash you specify is the "base" or the "anchor." It will not be included in the list of commits you can edit.

If you want to edit Commit A, you must provide the hash for the commit before A (Commit B).

Git will then open the editor starting with the first commit that happened after the hash you provided.

Pro-Tip: Finding the Hash

Instead of squinting at a long git log, use this command to get a "one-line" view of your history with short hashes: 

git log --oneline -n 10

You'll see something like: 

a1b2c3d Fixed the double button
e4f5g6h Added Job model
i7j8k9l Initial commit

If you want to edit "Added Job model", you would use the hash from the commit below it:

  • git rebase -i i7j8k9l

Why use hashes instead of HEAD~?

  • No Counting Errors: You don't have to worry about whether you meant HEAD~4 or HEAD~5.
  • Clarity: If you have dozens of commits, counting is tedious. Copying a hash is instant.
  • Specific Snapshots: If you know exactly where the "trouble" started (like when you first introduced that double-button bug), you just grab that hash and go.

Combining with the "Safety Branch"

See below.

Since you’re using the Safety Branch technique, using a hash is even better. You can look at your safety-backup branch, find the hash of the last "perfect" commit, and rebase your main branch onto that exact point.

A Quick Tip for the Future

Before doing anything "experimental" with Git history, you can create a temporary "safety branch": 

git branch safety-copy

If the rebase goes wrong, you can just delete your mess and go back to the safety-copy branch. It's like a manual save point in a video game!

Step by Step

The "Safety Branch" technique is the ultimate stress-reducer. It’s like creating a "Save As..." copy of your entire project before you start messing with the timeline.

If you want to go back and fix those commits using the safety branch method, here is exactly how to do it step-by-step.

1. Create your "Save Point"

Before you touch the history, create a new branch that stays exactly where you are right now. 

git branch safety-backup
  • What this does: It creates a bookmark called safety-backup at your current commit. If you accidentally delete your whole project during the rebase, this branch will still have everything.

2. Start the Surgery

Now, stay on your main branch (usually main or master) and start the interactive rebase to fix those doubled-up lines. 

git rebase -i HEAD~3

(Replace 3 with however many commits back the error goes.)

3. The "Edit" Loop

As we discussed before:

  • Change pick to edit for the messy commits.
  • When Git stops at a commit, fix the code in Xcode (delete the duplicate line).
  • Run git add .
  • Run git commit --amend --no-edit
  • Run git rebase --continue

4. Compare the Results

Once the rebase is finished, you can actually compare your "Fixed" version with your "Safety" version to make sure you didn't accidentally delete something important.

In Xcode (or using git diff), you can compare the two: 

git diff safety-backup

You should only see the deletions of those duplicate lines. If you see a bunch of other code missing, you’ll know something went wrong.

5. The "I Messed Up" Recovery

If the rebase ends and your code is a total mess, you can instantly "teleport" your main branch back to the safety point: 

git reset --hard safety-backup

The Result: It’s like the rebase never happened. Your main branch is perfectly restored to its pre-surgery state.

6. Cleanup

Once you are 100% happy that the double-lines are gone and the app still runs perfectly: 

git branch -D safety-backup

This deletes the backup, leaving you with a beautiful, clean history.

Why this is better than just "fixing it later"

By doing this, you keep your Git history honest. If you ever have to go back to an old version of the app to test something, you won't be frustrated by "Add User" failing because of a syntax error you fixed "later" in the timeline.

Clarification on Creating a Safety-Backup Branch But Working On Main

Think of it like this:

  • The Safety Branch: Is a snapshot of the "Broken" version (with the double-button lines).
  • The Main Branch: Is where you are performing the "Surgery" (the rebase).

The Workflow Visualized

  • Before Rebase: Both main and safety-backup point to the same (broken) commit.
  • During Rebase: You are on main. Git takes main back in time. You fix the commits one by one.
  • After Rebase: main has now "diverged." It has new, clean commits. safety-backup is still pointing to the old, messy commits.

Why the Comparison works

Because you are still on main when the rebase finishes, you can run: 

git diff safety-backup

This tells Git: "Show me the difference between where I am now (Clean Main) and that snapshot I took earlier (Messy Safety)."

Summary of Branch Status

  • Do you switch back? No, you never left main. You just created a bookmark elsewhere.
  • What are you comparing? You are comparing the "New Past" (Main) against the "Old Past" (Safety).

If the diff shows that the only things gone are those duplicate Button lines, you’ve won! You can then safely delete the safety-backup branch. If you see that you accidentally deleted the entire toolbar or something vital, you use that:

  • reset --hard safety-backup command to teleport back to the start and try again.

It’s a very professional way to work - it gives you the freedom to make mistakes without the fear of losing your progress!
