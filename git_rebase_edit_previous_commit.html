Rebase - Editing a previous commit (back in History)

AI Guide

QUESTION

I want to edit a commit in git.

This is my git (comment) history: 

latest (HEAD)
previous - but needs changing
another
earlier

...I want to edit 'previous - but needs changing', but leave 'latest' unchanged and the most recent commit.

Is there any way to achieve this?

ANSWER

Yes, there is a standard way to do this using Interactive Rebase. It’s essentially a way to rewrite history by pausing the timeline at a specific point, letting you make your changes, and then "replaying" the subsequent commits on top of that change.

The Game Plan

Since you want to reach back one step behind your current HEAD, you will use the command: 

git rebase -i HEAD~2

This tells Git: "I want to look at the last 2 commits and interactively decide what to do with them."

Step-by-Step Execution

1. Start the Rebase

Run the command above. Your default text editor (likely Vim or Nano) will open with a list of commits that looks like this: 

pick e1a2b3c another
pick f4g5h6i previous - but needs changing
pick j7k8l9m latest (HEAD)

2. Change the Instruction

Change the word pick to edit (or just e) next to the commit you want to fix. Leave latest as pick. 

pick e1a2b3c another
edit f4g5h6i previous - but needs changing
pick j7k8l9m latest (HEAD)

Save and close the editor.

3. Make Your Changes

Git will now "travel back in time." You are currently sitting at the previous commit.

  • Modify your files to fix whatever was wrong.
  • Stage the changes: git add <filename>
  • Update the commit: git commit --amend (You can also change the commit message here if you want with the -m flag).

4. Return to the Present

Once the edit is done, tell Git to finish the process: 

git rebase --continue

Git will take your modified previous commit and then try to "tack" the latest commit back on top of it.

A Word of Caution

Because you are changing a commit that has already happened, you are technically rewriting history.

  • If you haven't pushed yet: You're golden. No one will ever know.
  • If you have already pushed: You will need to use git push --force to update the remote server. Be careful doing this if you are working on a team, as it can mess up your colleagues' local histories.
