Hacking With Swift - 227 - Understanding Swift's Result type

Today you’re going to tackle a tricky concept in the form of Swift’s Result type, but to balance things out we’re also going to cover two easier ones too so hopefully you don’t find today too much work.

Swift’s Result type is designed to solve the problem when you know thing A might be true or thing B might be true, but exactly one can be true at any given time. If you imagine those as Boolean properties, then each has two states (true and false), but together they have four states:

  • A is false and B is false
  • A is true and B is false
  • A is false and B is true
  • A is true and B is true

If you know for sure that options 1 and 4 are never possible - that either A or B must be true but they can’t both be true - then you can immediately halve the complexity of your logic.

American author Ursula K Le Guin once said that:

"The only thing that makes life possible is permanent, intolerable uncertainty; not knowing what comes next."

The absolute opposite is of this adage is true of good software: the more certainty we can enforce and the more constraints we can apply, the safer our code is and the more work the Swift compiler can do on our behalf.

So, although Result requires you to think about escaping closures being passed in as parameters, the pay off is smarter, simpler, safer - totally worth it.

Today you have three topics to work through, in which you’ll learn about Result, image interpolation, and context menus.

Result

Swift provides a special type called Result.

It allows us to encapsulate either a successful value or some kind of error type, all in a single piece of data.

So, in the same way that an 'optional' might hold a string or might hold nothing at all, Result might hold a string or might hold an error.

The syntax for using it is a little odd at first, but it does have an important part to play in our projects.

To see Result in action, we could start by writing a method that downloads an array of data readings from a server, like this: 

struct ContentView: View {
    @State private var output = ""

    var body: some View {
        Text(output)
            .task {
                await fetchReadings()
            }
    }

    func fetchReadings() async {
        do {
            let url = URL(string: "https://hws.dev/readings.json")!
            let (data, _) = try await URLSession.shared.data(from: url)
            let readings = try JSONDecoder().decode([Double].self, from: data)
            output = "Found \(readings.count) readings"
        } catch {
            print("Download error")
        }
    }
}

The output of the above is a string across the screen saying: 

Found 1000 readings

That code works just fine, but it doesn’t give us a lot of flexibility - what if we want to stash the work away somewhere and do something else while it’s running? What if we want to read its result at some point in the future, perhaps handling any errors somewhere else entirely? Or what if we just want to cancel the work because it’s no longer needed?

Task

Well, we can get all that by using Result, and it’s actually available through an API you’ve met previously: Task. We could rewrite the above code to this: 

func fetchReadings() async {
    let fetchTask = Task {
        let url = URL(string: "https://hws.dev/readings.json")!
        let (data, _) = try await URLSession.shared.data(from: url)
        let readings = try JSONDecoder().decode([Double].self, from: data)
        return "Found \(readings.count) readings"
    }
}

We’ve used Task before to launch pieces of work, but here we’ve given the Task object the name of fetchTask - that’s what gives us the extra flexibility to pass it around, or cancel it if needed. And notice how our Task closure returns a value now? That value gets stored in our Task instance so that we can read it in the future when we’re ready.

More importantly, that Task might have thrown an error if the network fetch failed, or if the data decoding failed, and that’s where Result comes in: the result of our task might be a string saying “Found 10000 readings”, but it might also contain an error. The only way to find out is to look inside - it’s very similar to optionals.

To read the result from a Task, read its result property like this: 

let result = await fetchTask.result

Notice how we haven’t used try to read the Result out? That’s because Result holds it inside itself - an error might have been thrown, but we don’t have to worry about it now unless we want to.

If you look at the type of result, you’ll see it’s a Result<String, Error> - if it succeeded it will contain a string, but it might also have failed and will contain an error.

You can read the successful value directly from the Result if you want, but you’ll need to make sure and handle errors appropriately, like this: 

do {
    output = try result.get()
} catch {
    output = "Error: \(error.localizedDescription)"
}

Alternatively, you can switch on the Result, and write code to check for both the success and failure cases. Each of those cases have their values inside (the string for success, and an error for failure), so Swift lets us read those values out using a specially crafted case match: 

switch result {
    case .success(let str):
        output = str
    case .failure(let error):
        output = "Error: \(error.localizedDescription)"
}

Regardless of how you handle it, the advantage of Result is that it lets us store the whole success or failure of some work in a single value, pass that around wherever we need, and read the error only when we’re ready.

Ai Insight - .task

QUESTION

I tried an example from Paul Hudson's 100 Days of SwiftUI, and I was surprised that it ran without having to tap anything to trigger the function. I assumed '.task' needed to be triggered. I guess I'm wrong. What is going on in fact?

ANSWER

You are definitely not the first person to be surprised by this! It is a very common point of confusion when learning SwiftUI.

Your assumption that it needs a user action (like a tap) to trigger is actually how buttons or gesture modifiers work, but .task belongs to a different family of modifiers: lifecycle modifiers.

Here is exactly what is going on behind the scenes.

The Short Answer

In SwiftUI, the .task modifier triggers its code automatically as soon as the view appears on the screen. You don't need to tap anything because the mere act of the ContentView loading and rendering is what fires the function.

How .task Actually Works

The .task modifier is essentially a modern, asynchronous version of the older .onAppear modifier.

When SwiftUI builds your user interface, it goes through a specific lifecycle:

  • Initialization: The view struct (ContentView) is created.
  • Appearance: The view is rendered on the screen.
  • The Trigger: The moment the Text view appears on the screen, SwiftUI says, "Hey, I see a .task attached to this view. Let me spin up a background task and run this code right now."
  • Disappearance: If the user navigates away and the view leaves the screen, SwiftUI will automatically cancel the task if it hasn't finished yet.

Because your ContentView is the very first thing that loads when your app opens, fetchReadings() runs immediately on launch.

Why use .task instead of a button?

You would use .task for data that your view absolutely needs the moment the user looks at it.

  • Good use cases for .task: Fetching a user's profile info, loading a news feed, or downloading current weather data on launch.
  • Good use cases for a button/tap: Submitting a form, liking a post, or deleting an item.

If you wanted it to trigger on a tap...

If your goal actually was to wait for a user to tap the text before downloading the data, you would remove .task and use a .onTapGesture combined with a manual Task block like this: 

Text(output.isEmpty ? "Tap to load data" : output)
    .onTapGesture {
        // Manually start an async task when tapped
        Task {
            await fetchReadings()
        }
    }

So, you aren't wrong about how interactive triggers work - you just stumbled onto a lifestyle trigger instead! Paul Hudson loves using .task early in his courses because it's the cleanest way to get data onto your screen the second the app opens.

.task vs .onAppear

For the vast majority of use cases, .task has effectively superseded .onAppear when it comes to fetching data or running asynchronous code.

Apple introduced .task in iOS 15 specifically to solve the clunkiness of using .onAppear with Swift's modern Concurrency (async/await).

Here is a breakdown of why .task is now the go-to choice, and the few edge cases where .onAppear is still relevant.

Why .task is Better (The "Superpowers")

Before .task, if you wanted to load data when a view appeared, you had to use .onAppear. Because .onAppear is synchronous, you had to manually bridge it to the async world.

Here is how they compare side-by-side:

1. Automatic Cancellation (The Biggest Win)

If a user visits a screen that triggers a slow network download, and then they immediately hit the "Back" button, what happens to that download?

  • With .onAppear: The network request keeps running in the background, wasting data and battery power, because you have to manually write code to cancel it.
  • With .task: SwiftUI automatically cancels the asynchronous task the exact millisecond the view disappears from the screen.

2. Cleaner, Modern Syntax

Look at how much boilerplate code .task removes: 

// ❌ The Old Way (.onAppear)
Text("Hello")
    .onAppear {
        Task { // Forced to manually create a Task block
            await fetchData()
        }
    }

//  The New Way (.task)
Text("Hello")
    .task {
        await fetchData() // Works out of the box
    }

3. It Listens for ID Changes

The .task modifier has a special superpower: it can watch a variable. If that variable changes, .task will cancel its current work and restart itself. This is incredibly useful for things like search bars: 

// Every time the user types a new search query, 
// the old search task is canceled and a new one starts automatically.
Text("Search Results")
    .task(id: searchQuery) {
        await performSearch(for: searchQuery)
    }

Does .onAppear still have a purpose?

Yes! .onAppear is not deprecated, because it was never meant to be a data-fetching tool; it was meant for UI lifecycle events.

You should still use .onAppear for synchronous, non-async UI actions, such as:

  • Triggering a Haptic Feedback vibration when a view shows up.

*Starting a standard UI animation.

  • Analytics tracking (e.g., logging Analytics.logEvent("opened_homescreen")).
  • Focusing a text field automatically when the screen loads.

Summary Cheat Sheet

Use Case Best Modifier Why?
Fetching JSON from the web .task Built-in async support and auto-cancellation.
Loading data from a database .task Keeps the UI responsive during heavy reads.
Making the phone vibrate on load .onAppear It's instant, synchronous, and purely visual/tactile.
Restarting a fetch when a variable changes .task(id:) Automatically handles the reset logic for you.

The Bottom Line: If you are writing async/await code inside a view lifecycle, always reach for .task. It writes less code, prevents memory/network leaks, and is the modern standard in SwiftUI.
