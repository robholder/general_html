Hacking With Swift - 269 - Resizing images to fit the screen using GeometryReader

SwiftUI lets us create views with exact sizes like this: 

Image(.example)
    .resizable()
    .scaledToFit()
    .frame(width: 300, height: 300)

All this works great if we want fixed-sized views, but very often you want images that automatically scale up to fill more of the screen in one or both dimensions. That is, rather than hard-coding a width of 300, what you really want to say is "make this image fill 80% of the width of the screen."

One option is to use the containerRelativeFrame() modifier, which we covered back in project 8. But SwiftUI also gives us a dedicated type for this work called GeometryReader, and it’s remarkably powerful.

We’ll go into much more detail on GeometryReader shortly, but for now we’re going to use it for one job: to make sure our image fills some percentage of its container's width.

GeometryReader

GeometryReader is a view just like the others we’ve used, except when we create it we’ll be handed a GeometryProxy object to use. This lets us query the environment: how big is the container? What position is our view? Are there any safe area insets? And so on.

In principle that seems simple enough, but in practice you need to use GeometryReader carefully because it automatically expands to take up available space in your layout, then positions its own content aligned to the top-left corner.

For example, we could make an image that’s 80% the width of the screen, with a fixed height of 300: 

GeometryReader { proxy in
    Image(.example)
        .resizable()
        .scaledToFit()
        .frame(width: proxy.size.width * 0.8, height: 300)
}

You can even remove the height from the image, like this: 

GeometryReader { proxy in
    Image(.example)
        .resizable()
        .scaledToFit()
        .frame(width: proxy.size.width * 0.8)
}

We’ve given SwiftUI enough information that it can automatically figure out the height: it knows the original width, it knows our target width, and it knows our content mode, so it understands how the target height of the image will be proportional to the target width.

Now, you're probably wondering how this is any different from using containerRelativeFrame(). Well, the problem is that containerRelativeFrame() has a very precise definition of what constitutes a "container": it might be the whole screen, it might be a NavigationStack, it might be a List or a ScrollView, and so on, but it won't consider a HStack or a VStack a container.

This causes problems when using views in stacks, because you can't easily subdivide them using containerRelativeFrame(). For example, the code below places two views in a HStack, with one being given a fixed width and the other using a container relative frame: 

HStack {
    Text("IMPORTANT")
        .frame(width: 200)
        .background(.blue)

    Image(.example)
        .resizable()
        .scaledToFit()
        .containerRelativeFrame(.horizontal) { size, axis in
            size * 0.8
        }
}

That's not going to lay out well at all, because the containerRelativeFrame() will read the whole screen width for its size, meaning that image will be 80% the screen width despite 200 points of the screen being a text view.

On the other hand, using a GeometryReader will subdivide the space correctly: 

GeometryReader { proxy in
    Image(.example)
        .resizable()
        .scaledToFit()
        .frame(width: proxy.size.width * 0.8)
}

Of course, that introduces a different problem: our image is now aligned to the top-left corner of the GeometryReader!

Fortunately, this is easily solved. If you ever want to center a view inside a GeometryReader, rather than aligning to the top-left corner, add a second frame that makes it fill the full space of the container, like this: 

GeometryReader { proxy in
    Image(.example)
        .resizable()
        .scaledToFit()
        .frame(width: proxy.size.width * 0.8)
        .frame(width: proxy.size.width, height: proxy.size.height)
}

Ai Insight

GeometryReader

Once again, the tutorial is presenting multiple structural concepts, layout bugs, and syntax steps simultaneously. Let’s separate the mental model (what is actually happening visually) from the code mechanics so you can see the core lesson clearly.

1. The Core Principle: What is a GeometryReader?

Normally, SwiftUI layouts are conversational negotiations:

  • The Parent View says: "Here is all the space I have. How much do you want?"
  • The Child View responds: "I only need exactly 150x50 pixels, thanks."

The problem is that a standard child view cannot see the space it was offered. It just says what it needs.

A GeometryReader is a special container view that acts like a spy. It intercepts that offer from the parent, reads the exact measurements, and hands those numbers to you inside a variable called proxy (which is a GeometryProxy object).

2. Visualizing the 2 Big "Quirks" of GeometryReader

To use GeometryReader successfully, you have to understand its two unique, built-in personality traits. This is where most beginners get confused.

Quirk A: It is a "Greedy" View

Most views in SwiftUI shrink to fit their content (like a Text view wrapping tightly around its words).

GeometryReader does the opposite. It is greedy. The second you place it on the screen, it automatically expands to consume every single pixel its parent container offers it.

Quirk B: It Aligns to the Top-Left

When you put a child view (like an Image) inside a standard VStack, SwiftUI centers it by default.

When you put a child view inside a GeometryReader, it completely dumps that rule and pins your view directly to the top-left corner.

3. Demystifying the "Double Frame" Center Trick

At the very end of the lesson, Paul presents this code to fix the top-left alignment issue, which looks like total wizardry: 

GeometryReader { proxy in
    Image(.example)
        .resizable()
        .scaledToFit()
        .frame(width: proxy.size.width * 0.8) // Frame 1
        .frame(width: proxy.size.width, height: proxy.size.height) // Frame 2
}

Why apply .frame twice? This is a brilliant SwiftUI layout technique when broken down step-by-step:

  • Frame 1 (The Math Frame): width: proxy.size.width * 0.8
    • This shrinks the image itself down to exactly 80% of the available space. Because it's inside a GeometryReader, this small image immediately snaps to the top-left corner.
  • Frame 2 (The Invisible Box Trick): width: proxy.size.width, height: proxy.size.height
    • This wraps an invisible box around that image. This new invisible box is forced to match the exact, full size of the GeometryReader.

Here is the trick: Standard frames in SwiftUI center their content by default. By placing an invisible full-sized frame around our 80% width image, the frame naturally forces the smaller image to sit perfectly in its absolute center.

4. GeometryReader vs. ContainerRelativeFrame

Paul explains that containerRelativeFrame() only looks at "major" containers like the whole device screen or a ScrollView.

If you put an image inside an HStack next to a sidebar, containerRelativeFrame() ignores the sidebar entirely and calculates its width based on the entire screen, causing your layout to smash off the edge of the device.

A GeometryReader calculates its size based on where it sits in the hierarchy. If it is placed inside an HStack alongside a 200-point text view, the GeometryReader will only measure the remaining space left over, preventing layout clipping.

Formula for Writing Your Own

Whenever you need a view to be a direct percentage of its local container, use this blueprint:

{{swift GeometryReader { proxy in 

YourView()
// 1. Size your actual content using the proxy dimensions
.frame(width: proxy.size.width * 0.50) // 50% of available width

// 2. Put it in a full-sized invisible box to center it
.frame(width: proxy.size.width, height: proxy.size.height)

} }}}

Proxy/Geo Variable

proxy is completely arbitrary. It is a variable name of convenience.

You can name it absolutely anything you want. The only requirement is that whatever name you choose on the left side of the in keyword must match the name you use to call .size.width inside the closure.

For example, all three of these code blocks do the exact same thing: 

// Option 1: Using 'geo' (A very common shorthand in the SwiftUI community)
GeometryReader { geo in
    Text("Hello")
        .frame(width: geo.size.width * 0.5)
}

// Option 2: Using a descriptive name
GeometryReader { geometry in
    Text("Hello")
        .frame(width: geometry.size.width * 0.5)
}

// Option 3: Using something completely ridiculous
GeometryReader { banana in
    Text("Hello")
        .frame(width: banana.size.width * 0.5)
}

Why does Paul use proxy?

He uses it because the actual, official Swift data type that SwiftUI hands you here is called GeometryProxy.

A "proxy" in English is an agent or representative authorized to act on behalf of someone else. In this case, that object is acting as a representative for the view's physical dimensions.

Most developers prefer geo or geometry simply because it's easier to read at a glance, but proxy is just Paul's personal stylistic choice.
