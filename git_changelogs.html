Git - Changelogs

When recording changes in a project, ideally, you should record these changes in three distinct places, because they serve three different audiences:

  • yourself (the code),
  • your users (the README), and
  • the software ecosystem (the changelog/versioning).

Here is the industry-standard way to track these kinds of subtle updates:

1. Within the Python Script (The "Self-Documenting" Level)

The code itself should explain why this logic exists so you (or someone else) don't accidentally undo it later.

  • Inline Comments: Add a brief comment explaining the empty sequence check.
  • Update the Version: Since you modified the program's behavior, increment the script's version. Since this is a minor fix, bump the patch version (e.g., from 2.0.1 to 2.0.2 in your __version__ string).

Example: 

__version__ = "2.0.2"  # Bumped from 2.0.1

# ... 

# ---> NEW CHECK <---
# If files were found but none parsed as valid sequences (e.g., only .txt or .py files),
# warn the user rather than exiting silently.
if not sequences:
    print(f"Warning: No valid sequential files found in '{testPath}'", file=sys.stderr)
    sys.exit(0)

2. In the README.md (The User Level)

The README is for the person using your tool. They don't need a line-by-line code change history here, but they do need to know how the tool behaves.

  • Expected Behavior Section: Under a "Usage" or "Behavior" section, make a quick note of this safety feature:
    • "If the directory is empty or contains no numbered sequences, the utility will print a warning to stderr and exit cleanly."
  • Troubleshooting Section: If someone complains, "Why am I getting a warning for my folder full of PDFs?", the README explains that lseq specifically filters for numbered sequence files.

3. In a CHANGELOG.md (The "System" You Might Be Unaware Of)

In professional software development, you don't clutter the README with a list of every bug you've ever fixed. Instead, you use a dedicated file in the root of your repository called CHANGELOG.md.

This is the system designed specifically to track history. The standard format is called Keep a Changelog and looks like this: 

# Changelog

All notable changes to this project will be documented in this file.

## [2.0.2] - 2026-07-16
### Fixed
- Fixed a bug where directories with only non-sequential files (like `.txt` or `.py`) would cause the script to exit silently. It now correctly outputs a `No valid sequential files found` warning to `stderr`.

## [2.0.1] - 2026-05-10
### Added
- Initial release of the optimized sequence parser.

Semantic Versioning

The common system used for version numbers is called Semantic Versioning (often abbreviated as SemVer). It is the universal standard for versioning software.

The format is strictly defined as three numbers separated by dots:

MAJOR.MINOR.PATCH

For example, in your script's version 2.0.1, 2 is the Major version, 0 is the Minor version, and 1 is the Patch version.

Here is exactly what each section is used for, and when you should increment them:

The SemVer Rules

Position Type Increment When You... Example
MAJOR Breaking Changes Make incompatible API changes. Old code using your script will break. 1.0.0 → 2.0.0
MINOR New Features Add functionality in a backward-compatible way. Nothing breaks. 2.0.1 → 2.1.0
PATCH Bug Fixes Make backward-compatible bug fixes or minor maintenance. 2.0.1 → 2.0.2

1. PATCH (The Third Digit)

  • When to change: When you fix a bug, optimize performance, or make a minor adjustment that doesn't add new features.
  • Rule: It must be backward-compatible. A user should be able to drop in the new patch version and have everything work exactly as it did before, just with fewer bugs.
  • In an example case: Changing a silent exit to a reassuring warning message is a classic patch because it fixes a behavioral bug without changing how users run the command.

2. MINOR (The Second Digit)

  • When to change: When you add a brand-new feature or capability to your script, but it is still backward-compatible.
  • Rule: Any existing scripts or automation that use your tool will still run perfectly without modification, but they now have access to new options.
  • Example: If you added a new command-line flag like --csv to export the sequence list to a spreadsheet, you would bump the version from 2.0.2 to 2.1.0. (Note: Whenever you increment the Minor version, the Patch version resets to 0).

3. MAJOR (The First Digit)

  • When to change: When you make "breaking changes." This means your script’s interface has changed so much that older scripts calling it will break or fail.
  • Rule: If a user cannot upgrade without potentially rewriting some of their own code/commands, it must be a Major bump.
  • Example: If you decided to completely change the output format, or if you renamed the mandatory directory argument to path, you would bump the version from 2.1.5 to 3.0.0. (Note: When the Major version bumps, both Minor and Patch reset to 0).

A Real-World Evolution

To see how this looks in practice, imagine the life cycle of a utility:

  • 1.0.0 – Your initial stable release.
  • 1.0.1 – You fix a typo in the help text (Patch).
  • 1.1.0 – You add a feature to highlight a line in a feedback string
  • 1.1.1 – You fix a bug where negative frame numbers crashed the script (Patch).
  • 2.0.0 – You rewrite the tool to use a completely different output layout that existing pipeline scripts can't read without being updated (Major).
