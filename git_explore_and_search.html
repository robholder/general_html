    Blame Game!

    Examine the what, who, and when a file was edited

    Blame

    The 'git blame' command works at line level (it looks at one line at a time).

    It can tell you when a line changed, but not what changed. 

    # to show what's changed, by whom and at what commit:
git blame <file_name>

# The default is to look at HEAD:
git blame HEAD <file_name>
# ...is the same as:
git blame <file_name>
 
# to show at a given commit hash:
git blame <commit ID> <file_name>

    Flags for Blame

    -s

    # Suppress time stamps:
git blame -s <file_name>

    Grep

    Git grep

    Not to be confused with 'git log --grep' options

    Git grep only searches files that have been staged or committed, and not untracked files in the working area.

    Git grep can use expressions and expanded expressions. 

    # To search through repo for a search-term
git grep <search-term>
# To search for multi-word terms (surround with quotes):
git grep "multi word term"

    Grep Options

    # Case insensitive -i or --ignore-case
git grep -i <search-term>
# Display line numbers -n or --line-number
git grep -n <search-term>
# Display only filenames -l or --name-only
git grep -l <search-term>

# Mixing flags:
git grep -ni <search-term>
# or:
git grep -n -i <search-term>

    Log's Diff Options

    Git log w/Pickaxe S

    -S

    The 'git log' command can be given the '-S' flag to add the pickaxe option.

    '-S' must immediately precede the search-string.

    The 'pickaxe' option will show a commit if a given string term was added or removed.

    Pickaxe doesn't accept regular expressions - only strings. 

    # To show commits for a file where the word "test" was added or removed:
git log -S test

    Git log w/Pickaxe G

    -G

    The 'git log' command can be given the '-G' flag to add the pickaxe option that looks for search-strings where they've been *changed* (in commits).

    '-G' must immediately precede the search-string.

    The '-G' flag will show any commit that has a CHANGE on the same LINE as the search term.

    The '-G' flag CAN receive regular expressions.

    Git log w/Patch

    -p

    Adding a 'patch' option to 'git log' will show you a diff display of a log output. 

    git log -p

    Git log w/Word-Diff

    --word-diff

    To show the actual word/s on a line use the --word-diff flag. 

    # To show individual word changes per line:
git log --word-diff <search-term>

# With other flags:
git log -p --oneline -S <string-term> --word-diff

    Example: 

    $ git log -p -S walnut -i --word-diff

    Log options Combined!

    Combine pickaxe and patch for an even better result. 

    # To show the commit with the search-term and how it changed:
git log -p -S <string-term>
# Or move the patch option (however, -S has to be followed by the <search-term>:
git log -S <string-term> -p

    Example: Here we see that all the lines were added in this commit (it was a new file!):

    Searching Through Commit Messages

    Git log w/Grep (Commit Messages)

    --grep

    The '--grep' flag searches commit messages only.

    No regular expression facility with --grep; just strings. 

    # To search through commit messages for a search-term:
git log --grep <string-term>

EXAMPLE:
$ git log --grep nuts
commit e9beff319ce7a28fe4abb062f50b026e70668fc4
Author: Armstrong <armstrong@eventplannerz.com>
Date:   Sat Jul 17 15:12:03 2021 -0400

    remove nuts from all items due to allergy concerns

    Bisect

    git bisect

    • Bisect is a binary search workflow:
      • It looks at the start end, then moves to middle (rinse and repeat)
    • git bisect starts a session of binary search starting at HEAD
    • git bisect <commit_ID> start a session at the given commit

    Bisect Process

    CAUTION! The bisect process leaves you in a 'detached HEAD' state.

    You will not be on a branch - DO NOT COMMIT! 

    # To start a binary process:
git bisect start # starts at HEAD
# or
git bisect <commit_ID> # start at the commit given

    This kicks off the process by checking out the middle commit (between first and <commit_ID> given.

    You can inspect the repo (which now has the commit's snapshot in the working directory). If the commit is 'bad' (has a bug or mistake), you can move to the next search: 

    # To move to the next commit to examine:
git bisect bad  # If the bug/error is present
# or
git bisect good  # If the bug/error is not present

    Eventually git will tell you which is the commit that introduced the error or but: 

    <commit_ID> is the first bad commit

    IMPORTANT!

    To end the binary search 'bisect' session, you must end: 

    # To end the 'bisect' session:
git bisect reset

    Full Example:

    # Starting at the latest commit (HEAD):
$ git bisect start
>status: waiting for both good and bad commits

$ git status
>On branch master
>Your branch is up to date with 'origin/master'.
>
>You are currently bisecting, started from branch 'master'.
>  (use "git bisect reset" to get back to the original branch)

# Check working directory for bugs (in code editor for example).

# If bad, indicate to git
$ git bisect bad
>status: waiting for good commit(s), bad commit known

# Check working directory for bugs (in code editor for example).

$ git bisect good

# Check working directory for bugs (in code editor for example).

$ git bisect bad

# Check working directory for bugs (in code editor for example).
 
# Bisect finds the culprit commit!
$ git bisect good
>

$ git bisect reset

    If you end up with a detached head after a reset, and you don't care about losing your current file conditions, run a checkout on the master: 

    git checkout <branch-name>

# e.g.,
git checkout main
