Hacking With Swift - 77 - Validating words with UITextChecker

Now that our game is all set up, the last part of this project is to make sure the user can’t enter invalid words. We’re going to implement this as four small methods, each of which perform exactly one check:

  • is the word original (it hasn’t been used already)
  • is the word possible (they aren’t trying to spell “car” from “silkworm”)
  • is the word real (it’s an actual English word).
  • the fourth method will be there to make showing error messages easier.

Anyway, let’s start with the first method: this will accept a string as its only parameter, and return true or false depending on whether the word has been used before or not. We already have a usedWords array, so we can pass the word into its contains() method and send the result back like this: 

func isOriginal(word: String) -> Bool {
    !usedWords.contains(word)
}

That’s one method down!

The next one is slightly trickier: how can we check whether a random word can be made out of the letters from another random word?

There are a couple of ways we could tackle this, but the easiest one is this: if we create a variable copy of the root word, we can then loop over each letter of the user’s input word to see if that letter exists in our copy. If it does, we remove it from the copy (so it can’t be used twice), then continue. If we make it to the end of the user’s word successfully then the word is good, otherwise there’s a mistake and we return false.

So, here’s our second method: 

func isPossible(word: String) -> Bool {
    var tempWord = rootWord

    for letter in word {
        if let pos = tempWord.firstIndex(of: letter) {
            tempWord.remove(at: pos)
        } else {
            return false
        }
    }

    return true
}

The final method is harder, because we need to use UITextChecker from UIKit. In order to bridge Swift strings to Objective-C strings safely, we need to create an instance of NSRange using the UTF-16 count of our Swift string. This isn’t nice, I know, but I’m afraid it’s unavoidable until Apple cleans up these APIs.

So, our last method will make an instance of UITextChecker, which is responsible for scanning strings for misspelled words. We’ll then create an NSRange to scan the entire length of our string, then call rangeOfMisspelledWord() on our text checker so that it looks for wrong words. When that finishes we’ll get back another NSRange telling us where the misspelled word was found, but if the word was OK the location for that range will be the special value NSNotFound.

So, here’s our third method: 

func isReal(word: String) -> Bool {
    let checker = UITextChecker()
    let range = NSRange(location: 0, length: word.utf16.count)
    let misspelledRange = checker.rangeOfMisspelledWord(in: word, range: range, startingAt: 0, wrap: false, language: "en")

    return misspelledRange.location == NSNotFound
}

Breakdown Of The Function

This block is basically a "bridge" between two different eras of Apple programming. You’re using a modern Swift string, but you’re asking a very old, very powerful library (written in Objective-C) to check the spelling.

Let's break it down line by line, like a translator.

1. let checker = UITextChecker()

The Librarian: This line creates an instance of UITextChecker. Think of this as opening a massive dictionary. It’s an object that knows how to scan text and find mistakes using the system's built-in dictionaries.

2. let range = NSRange(location: 0, length: word.utf16.count)

The Tape Measure: This is the most confusing line.

  • NSRange: This is an old-school way of saying "Start here and go this far."
  • location: 0: Start at the very first letter.
  • length: word.utf16.count: This tells the checker exactly how long the word is.

Why .utf16? Modern Swift counts characters differently than the old spell-checker. If you use a simple .count, and the user enters an emoji or a complex accent, the "old" spell-checker will get the math wrong and potentially crash. .utf16 ensures they are both speaking the same "math language."

3. let misspelledRange = ...

The Search Party: This is where the work happens. You are calling a method on your "Librarian" (checker) called rangeOfMisspelledWord.

  • in: word: The string you want to check.
  • range: range: The "tape measure" we just made (check the whole word).
  • startingAt: 0: Don't skip any letters.
  • wrap: false: If you reach the end, don't start over at the beginning.
  • language: "en": Use the English dictionary.

4. return misspelledRange.location == NSNotFound

The Verdict: This is the clever part.

  • When UITextChecker looks for a mistake, if it finds one, it returns the location of that mistake (e.g., "I found a typo at character 3").
  • If it doesn't find any mistakes, it returns a special value called NSNotFound.
  • The Logic: If the location of the typo is "Not Found," then the word must be spelled correctly! So, we return true.

The Big Picture

Think of it like handing a slip of paper to a professional editor:

  • Line 1: Hire the editor.
  • Line 2: Tell the editor exactly where on the paper to look (from the start to the end).
  • Line 3: Ask the editor to circle any typos.
  • Line 4: Look at the paper. If there are no circles (NSNotFound), tell the app the word is "Real."

Why do we need this?

Swift doesn't have a simple word.isSpelledCorrectly property because spelling is hard! It depends on the dictionary, the language, and the context. This little bit of "clunky" code is just the price we pay to use Apple's incredibly accurate system dictionary. You can think of UITextChecker as a massive, pre-indexed digital dictionary that is built directly into the iOS operating system.

When you call that function, you aren't manually looping through a list of words; you are handing the word over to the system’s "Spell Check Engine"—the same one that underlines your text in red when you're writing an iMessage or an Email.

The "Over-the-Shoulder" View

Here is the high-level logic of how that block functions in the background:

  • The Hand-off: You give the function a string (e.g., "swift").
  • The Map: You provide an NSRange. This is like putting a highlighter over the word so the system knows which characters to inspect.
  • The Look-up: The UITextChecker doesn't just "compare" word-by-word like a simple list. It uses a trie (a complex data tree) of the English language. It follows the letters s -> w -> i -> f -> t.
  • The Result: * If it reaches the end of the letters and finds a "valid word" marker, it tells you: "Nothing found" (meaning no errors found).
    • If it gets to s -> w -> i -> f -> z, it realizes that path doesn't exist in its dictionary. It stops and returns the location where the mistake started.

Why this is better than a simple list

If you tried to load a text file of every single English word and compare them one by one, your app would be incredibly slow and use a ton of battery.

UITextChecker is:

  • Fast: It can check a word in microseconds.
  • Smart: It knows about grammar, common typos, and even some proper nouns.
  • Localized: It automatically adjusts if the user changes their phone language to French, Spanish, etc. (though we hardcoded "en" for your specific game).

How it fits into your game

In WordScramble app, this is your "Final Boss" of validation. You've already checked:

  • Is the word long enough?
  • Is it a real word from the "root" word?
  • Have they used it before?

This function is the final check to make sure they didn't just type in gibberish like "asdfg" that happens to use the right letters.

A Note on the "Magic" of NSNotFound

The reason the code looks for NSNotFound is a bit of old-school logic. In the programming world of Objective-C, you don't usually get a "True/False" back from a search. Instead, the computer says: "I searched for a mistake. I didn't find one, so here is a 'Nothing' value." By checking if the result is that "Nothing" value, you are effectively proving the word is correct.

Back to the Tutorial

@State private var errorTitle = ""
@State private var errorMessage = ""
@State private var showingError = false

Now we can add a method that sets the title and message based on the parameters it receives, then flips the showingError Boolean to true: 

func wordError(title: String, message: String) {
    errorTitle = title
    errorMessage = message
    showingError = true
}

We can then pass those directly on to SwiftUI by adding an alert() modifier below .onAppear(): 

.alert(errorTitle, isPresented: $showingError) {
    Button("OK") { }
} message: {
    Text(errorMessage)
}

We’ve done that several times now, so hopefully it’s becoming second nature! In fact, now that you're familiar with basic alerts like that one, here's a pro tip for you: if you don't include any buttons in the code, you automatically get a simple "OK" button that dismisses the alert.

So, if you prefer you can write this: 

.alert(errorTitle, isPresented: $showingError) { } message: {
    Text(errorMessage)
}

Yes, that looks a bit odd – choose whichever style you prefer.

At long last it’s time to finish our game: replace the // extra validation to come comment in addNewWord() with this: 

guard isOriginal(word: answer) else {
    wordError(title: "Word used already", message: "Be more original")
    return
}

guard isPossible(word: answer) else {
    wordError(title: "Word not possible", message: "You can't spell that word from '\(rootWord)'!")
    return
}

guard isReal(word: answer) else {
    wordError(title: "Word not recognized", message: "You can't just make them up, you know!")
    return
}

If you run the app now you should find that it will refuse to let you use words if they fail our tests – trying a duplicate word won’t work, words that can’t be spelled from the root word won’t work, and gibberish words won’t work either.
