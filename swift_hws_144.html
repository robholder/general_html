Challenge 1: Whitespace check

AI Insights

In Swift, there isn't a single property like .isNotWhitespace, but the standard way to handle this is to trim the whitespace and then check if the resulting string .isEmpty.

Here are the two best ways to handle it:

1. The Direct Approach

You can use trimmingCharacters(in:) to strip out all whitespace and newlines before checking the count. 

if myString.trimmingCharacters(in: .whitespacesAndNewlines).isEmpty {
    print("It's either empty or just whitespace.")
}

The trimmingCharacters(in:) method does not strip out all whitespace regardless of placement. It only removes whitespace from the start and the end of the string.

How it behaves:

  • " Hello " becomes "Hello" (Leading/trailing removed)
  • "Hello World" remains "Hello World" (Internal space is preserved)
  • " " (just spaces) becomes "" (An empty string)

Other Built-in Options

Think of a CharacterSet as a giant bucket of specific characters. When you pass that bucket to trimmingCharacters(in:), Swift looks at the start and end of your string and says, "If the character I'm looking at is inside this bucket, toss it away."

Because it uses CharacterSet, you have a lot of options beyond just spaces. Here are the other "buckets" you can use:

  • .whitespaces: Only removes standard spaces and tabs. Unlike .whitespacesAndNewlines, it will leave "Enter" key presses (newlines) alone.
  • .decimalDigits: Removes numbers (0-9).
  • .letters: Removes all letters, but leaves numbers and symbols.
  • .punctuationCharacters: Removes marks like !, ?, ., and ,.
  • .uppercaseLetters / .lowercaseLetters: Self-explanatory, but very handy for specific formatting.

Mixing and Matching

You aren't limited to the presets. You can combine them using "unions" or even create your own custom set.

To remove both punctuation AND whitespace: 

let combinedSet = CharacterSet.whitespaces.union(.punctuationCharacters)
let cleanString = myString.trimmingCharacters(in: combinedSet)

To remove specific characters you define: 

let customSet = CharacterSet(charactersIn: "0123456789#*")
let filtered = myString.trimmingCharacters(in: customSet)

Why this is perfect for your project:

For an address field in Cupcake Corner, you want to allow internal spaces (like "123 Main St"). You only want to prevent the user from:

  • Leaving the field entirely empty.
  • Typing a bunch of spaces to "trick" the .isEmpty check.

If you actually wanted to strip ALL whitespace

If you ever need to remove every single space in a string (e.g., for a username or a credit card number), you would use a different approach: 

let messyString = " 1 2 3 4 5 "
let cleanString = messyString.replacingOccurrences(of: " ", with: "")
// result: "12345"

2. The "Paul Hudson" Way (Recommended)

Since you are following 100 Days of SwiftUI, Paul often encourages writing extensions to make your code more readable. This allows you to keep using a simple property name throughout your project.

Add this extension to your project (usually at the bottom of the file or in a String-Extensions.swift file): 

extension String {
    var isReallyEmpty: Bool {
        self.trimmingCharacters(in: .whitespacesAndNewlines).isEmpty
    }
}

Now you can use it just like .isEmpty: 

if name.isReallyEmpty {
    print("Please enter a name that isn't just spaces.")
}

Why this matters for Cupcake Corner

In Project 10, users might try to "cheat" your validation by entering a single space for their name or address. Using .isReallyEmpty (or the trimming method) ensures that the Place Order button stays disabled until they provide actual text.

If you are using this inside your Order class for a computed property like hasValidAddress, it would look like this: 

var hasValidAddress: Bool {
    if name.isReallyEmpty || streetAddress.isReallyEmpty || city.isReallyEmpty || zip.isReallyEmpty {
        return false
    }
    return true
}

Challenge 2

AI Insights

You’ve hit on a common UI challenge: making a single alert "dynamic" so it reacts to the state of your data.

To solve this, you need a way to change the title string based on whether the network request succeeded or failed.

The Strategy

You already have a @State property for the message. You just need to add another @State property for the title.

Add a new State property:

@State private var alertTitle = ""

Update your placeOrder() function:

In the do block (Success): 

alertTitle = "Thank you!"
confirmationMessage = "Your order for..."
showingConfirmation = true

In the catch block (Failure): 

alertTitle = "Oops!"
confirmationMessage = "Check your internet connection and try again."
showingConfirmation = true

Update the Alert View:

Change the hardcoded "Thank you!" to your new variable: 

.alert(alertTitle, isPresented: $showingConfirmation) {
    Button("OK") { }
} message: {
    Text(confirmationMessage)
}

Why this is better than two separate alerts

You could create two different .alert modifiers (one for success, one for error), but that gets messy quickly. By using variables for both the title and the message, your code follows the Single Source of Truth principle-a core concept in SwiftUI.

A Quick "Pro" Tip for the Challenge

Paul’s challenge often implies catching actual network errors.

When you comment out request.httpMethod = "POST", URLSession defaults to a GET request.

httpbin.org/post will actually throw an error because it requires a POST. However, if you want to test a "No Internet" scenario without touching your code, you can simply:

  1. Turn off your Mac's Wi-Fi.
  2. Run the app.
  3. The catch block will trigger with a "The Internet connection appears to be offline" error automatically!

Challenge 3

AI Insight

This is definitely the "final boss" of Project 10. The reason it's tricky is that you're trying to sync a Class (which is a reference type) with UserDefaults (which is a data store), while keeping Codable (which expects specific keys) happy.

Even thought the data is collected there, this functionality should not live in AddressView.swift.

In SwiftUI, views should be "dumb" - they just display data. The logic for saving and loading should live inside your Order class.

The "Middle Ground" Strategy

The conflict Paul Hudson mentions is this:

If you use @Published or @Observable properties, and then try to add get and set blocks to them to talk to UserDefaults, Swift’s automatic Codable synthesis breaks.

The best "middle ground" is to use Property Observers (didSet). This allows you to keep the class simple and Codable, while triggering a "save" every time a value changes.

Step 1: Create a "Load" Initializer

In your Order class, you need to check UserDefaults when the app starts. 

@Observable
class Order: Codable {
    // 1. Define your keys to avoid typos
    enum CodingKeys: String, CodingKey {
        case _name = "name"
        case _streetAddress = "streetAddress"
        case _city = "city"
        case _zip = "zip"
        // Add your other properties here (type, quantity, etc.)
    }

    var name: String {
        didSet { UserDefaults.standard.set(name, forKey: "name") }
    }
    var streetAddress: String {
        didSet { UserDefaults.standard.set(streetAddress, forKey: "streetAddress") }
    }
    var city: String {
        didSet { UserDefaults.standard.set(city, forKey: "city") }
    }
    var zip: String {
        didSet { UserDefaults.standard.set(zip, forKey: "zip") }
    }

    init() {
        // 2. Load existing data or use empty defaults
        self.name = UserDefaults.standard.string(forKey: "name") ?? ""
        self.streetAddress = UserDefaults.standard.string(forKey: "streetAddress") ?? ""
        self.city = UserDefaults.standard.string(forKey: "city") ?? ""
        self.zip = UserDefaults.standard.string(forKey: "zip") ?? ""
    }
}

Why this works:

  • The View is happy: AddressView doesn't change at all. It just binds to order.name like it always did.
  • UserDefaults is happy: Every time the user types a character, the didSet triggers and saves the newest version to disk.
  • Codable is happy: Since you aren't using complex custom getters/setters, Swift can still encode and decode the class easily for your placeOrder() network call.

The Advanced "Middle Ground": A Wrapper

If you have a lot of properties, the didSet approach gets repetitive. A more "engineered" solution is to create a separate struct just for the address: 

struct Address: Codable {
    var name = ""
    var street = ""
    var city = ""
    var zip = ""
}

Then, in your Order class, you save the entire struct as a single piece of Data in UserDefaults whenever any part of it changes.

How to Include the Address Struct

The Address struct idea is actually a very elegant "middle ground." It allows you to group all the data you want to persist into one container, making it much easier to save and load from UserDefaults in one go.

However, it does change how your CodingKeys and your AddressView bindings work. Here is how you can refactor the Order class to make this work.

1. The Address Struct

First, define a simple struct at the top of your file (outside the class). Since it's a struct, it gets Codable for free. 

struct Address: Codable {
    var name = ""
    var streetAddress = ""
    var city = ""
    var zip = ""
}

2. Updating the Order Class

You will replace the four individual string variables with one address property. To solve the challenge, we use didSet on that property to save it to UserDefaults whenever any part of the address changes. 

@Observable
class Order: Codable {
    static let types = ["Vanilla", "Strawberry", "Chocolate", "Rainbow"]
    
    var type = 0
    var quantity = 3
    var extraFrosting = false
    var addSprinkles = false
    var specialRequestEnabled = false {
        didSet {
            if specialRequestEnabled == false {
                extraFrosting = false
                addSprinkles = false
            }
        }
    }

    // GROUPED ADDRESS PROPERTY
    var address: Address {
        didSet {
            if let encoded = try? JSONEncoder().encode(address) {
                UserDefaults.standard.set(encoded, forKey: "SavedAddress")
            }
        }
    }

    enum CodingKeys: String, CodingKey {
        // Properties used by the @Observable macro start with an underscore internally
        case _type = "type"
        case _quantity = "quantity"
        case _extraFrosting = "extraFrosting"
        case _addSprinkles = "addSprinkles"
        case _specialRequestEnabled = "specialRequestEnabled"
        case _address = "address" 
    }

    init() {
        // Load from UserDefaults if available, otherwise use a blank Address
        if let savedData = UserDefaults.standard.data(forKey: "SavedAddress") {
            if let decoded = try? JSONDecoder().decode(Address.self, from: savedData) {
                self.address = decoded
                return
            }
        }
        self.address = Address()
    }
}

3. Impact on AddressView

Because you moved the properties into a struct, your TextField bindings in AddressView.swift need to drill down one level deeper: 

Section {
    TextField("Name", text: $order.address.name)
    TextField("Street", text: $order.address.streetAddress)
    TextField("City", text: $order.address.city)
    TextField("Zip", text: $order.address.zip)
}

How this solves the "Middle Ground"

  • Codable Support: By using _address = "address" in your CodingKeys, the JSON sent to the server will look like this: {"type": 0, "address": {"name": "...", ...}}.
    • Note: If your server (like reqres.in) expects a "flat" JSON (no nested address object), you would need to write a custom encode(to:) method. But for most modern APIs, nesting is preferred!
  • UserDefaults: Since Address is a single struct, you only need one key in UserDefaults instead of four.
  • Validation: You’ll need to update your hasValidAddress property to look at address.name, etc.

A Note on the "Flat JSON" issue

If you find that your server call fails because it doesn't want the address inside a sub-folder (nested), you have two choices:

  • Stick to individual variables: Keep your current name, streetAddress, etc., but add a didSet to each one individually that saves to its own UserDefaults key.
  • Custom Encoding: Keep the struct for cleanliness, but tell Order how to "flatten" it when sending to the web.

The individual variables with didSet is arguably the "simplest" middle ground because it keeps your JSON structure exactly as it was before.

Troubleshooting the "Codable" issue

If you use the new @Observable macro (introduced in iOS 17), remember that it adds an underscore to your variable names behind the scenes (e.g., _name). When you use JSONEncoder, your JSON keys will have those underscores.

To fix that and make the reqres.in or httpbin server happy, you must include the enum CodingKeys shown in Step 1 to map the underscored names back to clean ones.
