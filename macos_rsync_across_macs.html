rsync - Two Macs

Ethernet

Connecting two Macs directly via Ethernet is one of the fastest, most reliable ways to transfer massive amounts of data or run automated backups.

Here is a complete, step-by-step guide to setting up your connection, mounting the volumes, navigating them in the Terminal, automating backups with rsync, and optimizing performance.

Step 1: Physical Setup & Network Configuration

  • Connect both Macs using an Ethernet cable. (Modern Macs automatically detect direct connections, so a standard Cat5e or Cat6 cable works perfectly—no crossover cable needed).
  • On both Macs, go to System Settings > Network.
  • Look at the Ethernet status. They should automatically assign themselves local IP addresses (Self-Assigned IPs) and show a green dot or a "Connected" status.
    💡 Tip: To ensure the Macs talk directly over Ethernet rather than your slower Wi-Fi network, make sure Ethernet is at the top of your network service order, or temporarily turn off Wi-Fi on both machines while setting this up.

Step 2: Enable File Sharing on the Target Mac (Mac Mini)

Before you can connect, the "host" Mac (your Mac Mini) needs to allow incoming connections:

  • On the Mac Mini, go to System Settings > General > Sharing.
  • Turn on File Sharing and click the small "i" (info) icon next to it.
  • Note the exact local hostname listed at the bottom (e.g., macmini.local).
  • Ensure the folders you want to share are listed under Shared Folders, and your user account has Read & Write permissions.

Step 3: Mount the Volume via SMB (Command+K)

Now, move to your main MacBook:

  • Click on your desktop to focus on Finder, then press <Cmd> + k (or go to Go > Connect to Server in the menu bar).
  • In the Server Address field, type: smb://macmini.local (adjust for the name of the mac!) and click Connect.
  • A prompt might ask for a username and password. Enter the login credentials for the Mac Mini.
  • A pop-up will appear asking you to choose the volume/mount folder you want to mount (e.g., Macintosh HD, User, or a specific shared folder). Select it and click OK.

The volume will now appear in your Finder sidebar under Locations, and on your Desktop (if you have "Connected Servers" visible in Finder settings).

Step 4: Accessing the Connected Volume in Terminal

In macOS, all external drives, network shares, and mounted volumes are located in a hidden root directory called /Volumes.

To find and navigate your mounted Mac Mini in the Terminal:

  • Open Terminal.
  • Type the following command to see all mounted drives: 
ls -la /Volumes
  • You will see your Mac Mini's shared folder listed there. To enter it, use the cd command (wrap the path in quotes if the folder name has spaces): 
cd "/Volumes/Your Mounted Folder Name"

Step 5: Setting Up the rsync Automation & Alias

Since you are running rsync over a mounted network volume, you don't actually need to pass the network username and SSH address (<username>@<MACHINE-1-NAME>.local) inside the rsync command itself. Because the drive is already mounted via SMB, the Mac treats it like a local hard drive located in /Volumes.

Here is how to structure your rsync command, along with a breakdown of your requested syntax:

The rsync Command Explained

rsync−avz−−delete/Source/Directory//Destination/Directory/
  • -a (archive mode): Preserves permissions, modification times, and symlinks.
  • -v (verbose): Shows the files being transferred in real-time.
  • -z (compress): Compresses data during transfer (great for networks).
  • --delete: Caution! This deletes files in the destination directory if they no longer exist in the source directory. It keeps both folders perfectly mirrored.

Creating the Permanent Alias

  • Open your shell configuration file (modern macOS uses Zsh by default) by running: 
alias sync-to-m1="rsync -avz --delete /path/to/local/tutorials/ \"/Volumes/Your Shared Folder Name/path/to/remote/tutorials/\""
  • Reload your Terminal profiles to activate the alias. 
    source ~/.zshrc

Now, whenever you type sync-to-m1 in Terminal, your sync will run instantly.

Step 6: Pro-Tips for Power Users

How to Auto-Connect at Login

If you don't want to press <Cmd> + k every time you boot your Mac, you can force macOS to mount the Mac Mini automatically when you log in:

  • Go to System Settings > General > Login Items.
  • Under the Open at Login section, click the + (plus) icon.
  • In the Finder sidebar, look under Locations, select your mounted Mac Mini folder, and click Open.
  • Now, macOS will automatically attempt to mount this share every time you log in (as long as the Ethernet cable is plugged in).

Turn Off Spotlight on the Mounted Drive (Highly Recommended)

When you connect a network share, macOS will often try to index the entire drive using Spotlight. This can cause massive lag, high CPU usage, and slow down your rsync file transfers.

  • Go to System Settings > Siri & Spotlight.
  • Scroll all the way to the bottom and click Spotlight Privacy.
  • Drag and drop your mounted Mac Mini volume from your desktop (or Finder) directly into this list, or click the + icon to browse and add it.
  • Click Done. Spotlight will now ignore that directory completely.

When you use an SSH key, you bypass the slow interactive password challenge entirely. Instead, your Macs perform an instantaneous, cryptographic handshake. This shaves seconds off the initial connection time, which makes scripts and rsync aliases fire off almost instantly.

Here is how to set up lightning-fast, passwordless SSH keys between your two Macs in under two minutes.

Step 1: Generate a Key Pair on Machine 1

We will use an Ed25519 key, which is the most modern, secure, and computationally fast encryption algorithm available.

Open a terminal on Machine 1 (your main Mac) and run: 

ssh-keygen -t ed25519 -C "my-mac-sync-key"
  • Press Enter three times to accept the defaults (leave the passphrase blank, as adding a passphrase forces you to type it in anyway, defeating our automated speed goal).

This creates two files in a hidden directory:

  • id_ed25519 (Your Private Key — keep this safe on Machine 1).
  • id_ed25519.pub (Your Public Key — the "lock" we are going to copy to the Mac Mini).

Step 2: Copy the Key to the Mac Mini

macOS doesn't come with the standard Linux ssh-copy-id tool by default, but you can copy the key over securely with this one-liner.

Run this command from Machine 1 (replace user and macmini.local with your actual details): 

cat ~/.ssh/id_ed25519.pub | ssh user@macmini.local "mkdir -p ~/.ssh && chmod 700 ~/.ssh && cat >> ~/.ssh/authorized_keys && chmod 600 ~/.ssh/authorized_keys"
  • Type your Mac Mini account password one last time to authorize the transfer.

Step 3: Switch Your Alias Back to the Speed Engine (SSH)

Because SSH keys are incredibly fast and secure, you can abandon the slow SMB File Sharing method entirely. Go into your ~/.zshrc file on Machine 1 and change your alias back to using native SSH, matching your new setup: 

alias sync-to-m2="rsync -avz --delete ~/Desktop/xcode_projects/ user@macmini.local:/Users/user/Desktop/xcode_projects/"

Run source ~/.zshrc to reload.

The Result

Test it out by running your alias. You will notice:

  • It will never ask you for a password again.
  • The connection initialization is instantaneous.
  • Because rsync is communicating natively via SSH rather than scanning individual security signatures on a mounted file server (SMB), your sync speeds will be blazingly fast.

Step-by-Step Guide: Syncing Git Repos via rsync and Local Hostnames

This guide establishes a seamless, two-way manual syncing routine between your primary machine (Machine 1) and secondary machine (Machine 2) over your home Wi-Fi network.

Architectural Overview

By using Apple's built-in Bonjour network service, your Macs can discover each other using a persistent domain extension (.local) instead of relying on shifting IP addresses. 

[Machine 1]  --- (rsync over SSH via Name.local) --->  [Machine 2]

Part 1: Initial Setup (One-Time Configuration)

You must configure the receiving machine so it has a readable network name and permission to accept secure files.

Step 1.2: Enable Remote Login Permissions

  • While still in System Settings > General > Sharing on Machine 2, look for the Remote Login toggle.
  • Toggle the switch to ON.

  • Note down your macOS account shortname (the name of your home user folder).

File Sharing on Both Machines

Step 1: Grant Permission on Machine 1 (Your Main Mac)

Part 2: Testing the Connection (The Dry Run)

Before modifying actual data, use rsync's simulation mode to verify that your directory paths match perfectly.

Step 2.1: Run a Simulation from Machine 1

Open your terminal or a Tmux pane on Machine 1 and run the following command.

⚠️ The Trailing Slash Rule: Note the trailing slash on the source directory /tutorials/. This ensures that rsync copies the contents of your tutorial directory into the target destination, rather than copying the parent folder itself. 

rsync -avzn --delete /path/to/local/tutorials/ username@TARGET-NAME.local:/path/to/remote/tutorials/
  • Replace username with Machine 2's shortname.
  • Replace TARGET-NAME.local with the hostname found in Step 1.1.
  • The -n flag tells rsync to execute a Dry Run.

Test the hostname with a basic "Ping"

If the command fails, try using Ping. We need to see if Machine 1 can even look up <machine_name>.local. Open your terminal on Machine 1 and type: 

ping macmini.local

If it says ping: cannot resolve <machine_name>.local: Unknown host: Your home Wi-Fi router isn't properly broadcasting Apple's Bonjour names between devices.

If it shows lines of numbers like 64 bytes from...: Great news! The network name works perfectly, meaning something is blocking the SSH connection itself.

Press <Ctrl> + C to stop the ping if it keeps going.

Step 2.2: Establish SSH Trust

When you execute the test command above, your terminal will display a security warning:

  • Type yes and press Enter. This permanently saves Machine 2 to your Mac's known hosts list.
  • Type Machine 2's user login password when prompted.
  • Review the terminal output. It will display a list of all files and hidden .git historical metrics it intends to sync without modifying any real files.

To remove the friction of typing long paths and server addresses every time you study, map your synchronization commands to short command-line aliases.

Step 3.1: Add the Push Shortcut to Machine 1

  • On Machine 1, open your shell layout file (usually ~/.zshrc or ~/.bashrc).

Step 3.2: Add the Pull Shortcut to Machine 2

To achieve true bidirectional synchronization, reverse the path logic on your secondary machine so you can push your updates back whenever you work away from your primary desk.

  • Walk over to Machine 2, open its shell configuration file (~/.zshrc).
  • Add a matching reflection alias pointing back to Machine 1's local hostname:

Part 4: Daily Operational Workflow Rule

Because rsync with a --delete flag completely mirrors one environment to another, adopt this strict operational rule to prevent accidentally erasing code updates:

💡 The Golden Rule: Always run your sync command from the machine you just finished coding on immediately before switching desks.
  • Finish work on Machine 1: Type sync-to-m2 in your terminal. Wait for the file transfer to hit 100%.
  • Move to Machine 2: Wake it up. Open your tutorial folders or check out specific Git stage milestones (git checkout <commit>). Your workspaces are instantly aligned.
  • Finish work on Machine 2: Type sync-to-m1 in your terminal to save your progress back to your primary workstation drive.

The Delete Flag

The --delete flag is what turns rsync from a simple file copier into a true mirroring engine.

Without it, rsync only looks at what needs to be added or updated. With it, rsync also looks at what needs to be cleaned up.

Here is exactly what that means for your tutorial repositories and why it is so significant for your specific workflow.

The Default Behavior (Without --delete)

If you run rsync without the --delete flag, the sync is strictly additive.

Imagine you are working in your tutorials folder on Machine 1:

  • You decide a project called Bad-Experiment was a mistake, and you delete the entire folder from your computer.
  • You run your rsync command without the --delete flag.
  • The Result: rsync copies over your new work, but it completely ignores the deleted folder. Bad-Experiment will remain sitting on Machine 2 forever. Over months of learning, Machine 2 will become a "ghost town" cluttered with old, abandoned files and folders that you deleted weeks ago on your primary machine.

The Mirror Behavior (With --delete)

When you include --delete, you are telling rsync: "Make the destination look exactly like the source. If a file doesn't exist on my main machine anymore, destroy it on the backup machine too." 

Source (Machine 1)         Destination (Machine 2)
  ├── Project A              ├── Project A
  └── Project B              ├── Project B
                             └── Bad-Experiment  ❌ (rsync deletes this)

Now, your backup machine remains perfectly pristine, matching your primary workspace file-for-file.

⚠️ Why it matters for Git repos (The "Detached HEAD" Gotcha)

Because you use git checkout <commit-hash> to jump back and forth between different stages of your tutorials, the --delete flag is absolutely critical.

When you jump back to a very early commit in Git, newer files and folders physically disappear from your hard drive (Git hides them in its internal history). If you sync your machines while sitting on an early commit without the --delete flag, the second machine won't remove those newer files. Your working directory on Machine 2 will become a confusing hybrid mix of early commits and late commits mashed together, which can break your code builds.

The --delete flag ensures that whatever state your Git repository is currently sitting in on Machine 1 is perfectly, identically replicated on Machine 2.

The Golden Safety Rule

Because --delete has the power to erase files on the destination machine, you must always adhere to the operational workflow rule: Only sync from the machine you just finished coding on. If you accidentally code for three hours on Machine 2, walk over to Machine 1, and run sync-to-m2 (which pushes Machine 1's old state to Machine 2), the --delete flag will instantly wipe out those three hours of new work on Machine 2 because those files don't exist on Machine 1 yet!

(This is why we used the -n dry-run flag earlier—it lets you double-check exactly what files --delete intends to erase before any real damage can happen).

Not working

When an SSH server abruptly closes a connection the exact millisecond you authenticate, it isn't a problem with your password, network, or rsync. It means the sshd-session binary on the Mac Mini is crashing or intentionally killing its own process because macOS is denying it permission to spawn the environment.

This happens if there is a corrupted permission cache or if macOS security gets trapped in a loop. Let's force macOS to reset its security database for these specific tools.

Step 1: Nuclear Reset of Full Disk Access Caches

We need to tell the Mac Mini's internal security database (tccutil) to wipe out and reset the permissions for Remote Login and the Shell, forcing macOS to re-evaluate them cleanly.

Walk over to the Mac Mini (or screen share into it), open its terminal, and run these two commands:

Bash sudo tccutil reset SystemPolicyAllFiles com.apple.Terminal sudo tccutil reset SystemPolicyAllFiles /usr/libexec/sshd-session (It will prompt you for your Mac Mini password to authorize the reset).

Step 2: The Re-Apply Sequence

Now that the database is clean, let's re-apply the permissions in the specific sequence macOS expects.

On the Mac Mini:

Go to System Settings > Privacy & Security > Full Disk Access.

If sshd-session, Terminal, or zsh are still in the list, highlight them and click the - (minus) button to delete them entirely.

Go to System Settings > General > Sharing.

Turn Remote Login completely OFF.

Turn Remote Login back ON. (This triggers macOS to automatically re-add the fresh sshd-session architecture back into the system privacy database).

Go back to Privacy & Security > Full Disk Access, locate Remote Login (sshd) or sshd-session, and ensure it is toggled ON.

Step 3: Run the Direct Diagnostic Test

Before trying rsync, let's use the simplest baseline test to see if the Mac Mini will let you in at all. From Machine 1, run a completely empty SSH connection:

Bash ssh user@macmini.local If it logs you in and gives you a command prompt (user@macmini ~ %): We won! The security loop is broken. Type exit to close it, and your rsync alias will now run perfectly.

If it still says Connection closed: There is a deeper configuration mismatch in the SSH daemon configuration file itself. If this happens, please run ssh -vvv user@macmini.local and paste the last 10 lines of output.
