Hacking With Swift - 243 - Challenge 2 - Add Editing Screen

  • Add an editing screen, so users can adjust the name and email address of someone they scanned previously. (Tip: Use the simple form of NavigationLink rather than navigationDestination(), to avoid your list selection code confusing the navigation link.)

Add Editing Screen

Ai Insight

This is a classic Hacking with Swift challenge! Let’s break down exactly what this looks like to the user, why Paul Hudson gives that specific NavigationLink tip, and how to write the code.

What it looks like in the app

  • The Trigger: When a user taps on any prospect row in your List, the row will display a native chevron (>) disclosure indicator on the right.
  • The Transition: Tapping the row slides a new "Edit Prospect" detail screen into view from the right side.
  • The Screen: This screen displays a clean, native Form containing two text fields: one initialized with the prospect's current name, and one with their email address.
  • The Autosave: Because you are using SwiftData, any changes the user types into those text fields will mutate the @Model properties directly and automatically save back to the database the moment they hit the back button.

Understanding the Tip: Why avoid navigationDestination?

Normally, modern SwiftUI apps use .navigationDestination(for:). However, your list has multi-selection active (selection: ＄selectedProspects).

If you use .navigationDestination, SwiftUI gets confused about your tap intent. It can't figure out if tapping a row means "add this item to the edit/delete selection batch" or "navigate to the detail screen".

By wrapping your row content inside a simple form NavigationLink(destination:), SwiftUI creates a dedicated disclosure button area for navigation that neatly coexists with the multi-selection edit checkmarks.

QUESTION

In the original tutorial challenge, the 'Tip' section elude to some of these issues? And have we applied it by the end of this page?

ANSWER

Yes, Paul Hudson's tip points exactly to the core of the nightmare we fought through to create this solution!

In fact, the tip acknowledges a known architectural tension in SwiftUI: when a List has multi-selection enabled (selection: ＄selectedProspects), any complex navigation mechanics attached to the rows can get their touch-targets and state tracking completely crossed.

Here is the breakdown of what that tip means, why it matters, and how we ultimately applied it.

What the Tip Means: NavigationLink vs navigationDestination()

In modern SwiftUI, there are two primary ways to handle stack navigation:

  • The Simple/Older Form (Direct Binding): 
NavigationLink {
    EditProspectView(prospect: prospect) // View is declared directly right here
} label: {
    Text(prospect.name)
}
  • The Modern Form (Type-Driven Deferral): 
NavigationLink(value: prospect) // Only passes a data token
// ...placed somewhere down on the List view:
.navigationDestination(for: Prospect.self) { prospect in
    EditProspectView(prospect: prospect)
}

Why navigationDestination() Fails Here

When you use the modern .navigationDestination(for:) pattern, SwiftUI relies entirely on row selection hashes to push views onto the stack. However, when you tap Edit to select multiple items for deletion, the List also starts manipulating row selection hashes to build your selectedProspects Set.

If you use .navigationDestination(), SwiftUI gets completely confused. It can't distinguish between "the user wants to select this checkbox to batch-delete" and "the user wants to navigate to the edit screen." This conflict is what triggers the rapid layout glitches where checkboxes flash open and instantly snap shut.

Have We Applied It?

Yes, we used the simple form exactly as instructed. In our working code, you can see the direct declaration layout: 

NavigationLink {
    EditProspectView(prospect: prospect) // Direct destination injection
} label: {
    HStack { ... }
}

By sticking to the simple form, the destination view (EditProspectView) is tied tightly to the literal button label of that specific row.

The Missing Piece We Had to Solve

Even with Paul's tip, SwiftUI still has trouble when a row contains a working hyperlink and a native checkbox at the same time. While the simple NavigationLink kept our state from crashing, the row link was still "stealing" the user's touch. Tapping a row during edit mode would accidentally push the user into the EditProspectView instead of checking the multi-select circle.

To completely bridge the gap and make the tip functional, we added this modifier to the row: 

.allowsHitTesting(!isEditing)

By adding this, we fulfilled the spirit of the tutorial's tip:

  • When isEditing is false: The simple NavigationLink handles standard detail navigation natively.
  • When isEditing is true: The NavigationLink gesture engine is completely deactivated. It remains visible on screen, but it steps out of the way so the List container can intercept your tap and cleanly check the selection circle.

The Battle

Step 1: Create the Edit Screen View

Create a brand new SwiftUI file named EditProspectView.swift. It needs a bindable or directly mutable reference to the Prospect being edited. 

import SwiftUI
import SwiftData

struct EditProspectView: View {
    // Bindable allows TextField to write updates directly back to the SwiftData model object
    @Bindable var prospect: Prospect
    
    var body: some View {
        Form {
            TextField("Name", text: $prospect.name)
                .textContentType(.name)
            
            TextField("Email Address", text: $prospect.emailAddress)
                .textContentType(.emailAddress)
                .keyboardType(.emailAddress)
                .autocorrectionDisabled()
        }
        .navigationTitle("Edit Prospect")
        .navigationBarTitleDisplayMode(.inline)
    }
}

Step 2: Update ProspectsView.swift

Now, wrap your row's current horizontal layout inside the simple form of NavigationLink.

Find the List loop inside your ProspectsView and modify it to look like this: 

List(prospects, selection: $selectedProspects) { prospect in
    // Wrap the row's existing visual content inside a NavigationLink
    NavigationLink {
        EditProspectView(prospect: prospect)
    } label: {
        // Your exact row design goes here entirely unchanged
        HStack {
            VStack(alignment: .leading) {
                Text(prospect.name)
                    .font(.headline)
                Text(prospect.emailAddress)
                    .foregroundStyle(.secondary)
            }
            
            Spacer()
            
            Image(
                systemName: prospect.isContacted ?
                "person.crop.circle.fill.badge.checkmark" :
                "person.crop.circle.badge.xmark"
            )
            .font(.title2)
            .foregroundStyle(prospect.isContacted ? .blue : .red)
        }
    } // NavigationLink ends here
    // Swipe actions attach outside the NavigationLink, to the row container
    .swipeActions {
        Button("Delete", systemImage: "trash", role: .destructive) {
            modelContext.delete(prospect)
        }
        if prospect.isContacted {
            Button("Mark Uncontacted", systemImage: "person.crop.circle.badge.xmark") {
                prospect.isContacted.toggle()
            }
            .tint(.blue)
        } else {
            Button("Mark Contacted", systemImage: "person.crop.circle.fill.badge.checkmark") {
                prospect.isContacted.toggle()
            }
            .tint(.green)
            
            Button("Remind Me", systemImage: "bell") {
                addNotification(for: prospect)
            }
            .tint(.orange)
        }
    }
    .tag(prospect)
} // List ends

How the Magic Happens Automatically

Because Prospect is marked with the SwiftData @Model macro, any object passed into EditProspectView remains tracked by your container's context. When a user changes the text in the edit fields, they are editing the live database entry. When they tap the back arrow to return to ProspectsView, the main list will immediately animate to reflect the new name or email change without any extra save() code required!

#Preview

To make a SwiftUI #Preview work with a view that depends on a SwiftData model object like Prospect, you have to provide it with an in-memory database container.

If you just try to pass a loose Prospect() instance without a container, SwiftData will crash because the model isn't hooked up to a data context.

Here is how you populate the #Preview enclosure using an in-memory container and dummy data.

The Solution

Add this code to the bottom of your file: 

#Preview {
    // We wrap the setup in a closure and call it immediately with ()
    {
        let config = ModelConfiguration(isStoredInMemoryOnly: true)
        let container = try! ModelContainer(for: Prospect.self, configurations: config)
        
        // Creating sample data with your new isContacted parameter
        let sampleProspect = Prospect(name: "John Doe", emailAddress: "john.doe@example.com", isContacted: false)
        
        container.mainContext.insert(sampleProspect)
        
        return EditProspectView(prospect: sampleProspect)
            .modelContainer(container)
    }() // <--- These parentheses execute the code block and return the configured View safely
}

Why this works:

  • isStoredInMemoryOnly: true: This ensures that any edits you make inside the canvas preview are temporary. It prevents your dummy preview data from polluting your live app's real database.
  • try!: While using try! is generally a bad idea in production code, it is perfectly acceptable—and standard practice—in previews because if the preview container fails to initialize, you want the build to fail immediately so you can fix it.
  • .modelContainer(container): Even though your view only takes a @Bindable var prospect, SwiftData objects require an active context in the environment to track changes correctly. Injecting the container at the root of the preview keeps the canvas stable and interactive.

Now your preview will fully render, and you'll be able to type into the text fields interactively in Xcode!

import SwiftUI

struct ProspectsView: View {
    @Environment(\.modelContext) var modelContext
    @Query(sort: \Prospect.name) var prospects: [Prospect]

        case none, contacted, uncontacted

                        Button("Mark Contacted", systemImage: "person.crop.circle.fill.badge.checkmark") {
                            prospect.isContacted.toggle()
                        }
                        .tint(.green)
                        
                        Button("Remind Me", systemImage: "bell") {
                            addNotification(for: prospect)
                        }
                        .tint(.orange)
                    }
                }
                .tag(prospect)

                /// When `isEditing` is true, we pass `false` to `.allowsHitTesting()`.
                /// This completely freezes the row's child interaction. Because the NavigationLink is 
                /// now transparent to taps, user touches bypass it entirely and hit the underlying List frame,
                /// cleanly selecting the native checkmark circles instead of opening the details screen.
                .allowsHitTesting(!isEditing)

            /// Instead of letting the container handle EditMode, we explicitly inject our own rule into the List.
            /// When `isEditing` toggles, this tells the List to instantly slide out its checkbox margin.

                        /// ignoring Apple's automatic swap to "Done". We change the tint color to red

                        Button("Edit") {

                        /// protects it from crashing or getting cut off by the system Tab Bar layout.
                        /// It dynamically renders if we are in edit mode AND at least one selection is marked.
                        if isEditing && !selectedProspects.isEmpty {
                            Button(role: .destructive, action: deleteSelected) {
                                Label("Delete Selection", systemImage: "trash.fill")
                                    .foregroundStyle(.red)
                            }

                            .transition(.move(edge: .leading).combined(with: .opacity))

                    Button("Scan", systemImage: "qrcode.viewfinder") {
                        isShowingScanner = true
                    }
                }

        }
    }
    
    // MARK: - Core Logic Functions
    
    init(filter: FilterType) {
        self.filter = filter
        
        if filter != .none {
            let showContactedOnly = filter == .contacted
            
            _prospects = Query(filter: #Predicate {
                $0.isContacted == showContactedOnly
            }, sort: [SortDescriptor(\Prospect.name)])
        }
    }
    
    func handleScan(result: Result<ScanResult, ScanError>) {
        isShowingScanner = false
        
        switch result {
        case .success(let result):
            let details = result.string.components(separatedBy: "\n")
            guard details.count == 2 else { return }
            
            let person = Prospect(
                name: details[0],
                emailAddress: details[1],
                isContacted: false)
            modelContext.insert(person)
            
        case .failure(let error):
            print("Scanning failed: \(error.localizedDescription)")
        }
    }
    
    /// LEARNING POINT: Processing the Batch Delete
    /// This runs through our selected set, removes them from the SwiftData ModelContext,
    /// and resets the view state back to normal.
    func deleteSelected() {
        withAnimation {
            for prospect in selectedProspects {
                modelContext.delete(prospect)
            }
            selectedProspects.removeAll()
            isEditing = false // Cleanly shut down editing mode once deletion finishes
        }
    }
    
    func addNotification(for prospect: Prospect) {
        let center = UNUserNotificationCenter.current()
        
        let addRequest = {
            let content = UNMutableNotificationContent()
            content.title = "Contact \(prospect.name)"
            content.subtitle = prospect.emailAddress
            content.sound = UNNotificationSound.default
            
            let trigger = UNTimeIntervalNotificationTrigger(
                timeInterval: 5,
                repeats: false
            )
                
            let request = UNNotificationRequest(
                identifier: UUID().uuidString,
                content: content,
                trigger: trigger
            )
            center.add(request)
        }
        
        center.getNotificationSettings { settings in
            if settings.authorizationStatus == .authorized {
                addRequest()
            } else {
                center.requestAuthorization(options: [.alert, .badge, .sound]) { success, error in
                    if success {
                        addRequest()
                    } else if let error {
                        print(error.localizedDescription)
                    }
                }
            }
        }
    }
}
