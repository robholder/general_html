Hacking With Swift - 91 - Next: Challenge

Challenge

Before we proceed onto more complex projects, it’s important you have lots of time to stop and use what you already have. So, today you have a new project to make entirely by yourself, with no help from me other than some hints below. Are you ready?

Your goal is to build an “edutainment” app for kids to help them practice multiplication tables – “what is 7 x 8?” and so on. Edutainment apps are educational at their core, but ideally have enough playfulness about them to make kids want to play.

Breaking it down:

  • The player needs to select which multiplication tables they want to practice. This could be pressing buttons, or it could be an “Up to…” stepper, going from 2 to 12.
  • The player should be able to select how many questions they want to be asked: 5, 10, or 20.
  • You should randomly generate as many questions as they asked for, within the difficulty range they asked for.

If you want to go fully down the “education” route then this is going to be some steppers, a text field and a couple of buttons. I would suggest that’s a good place to start, just to make sure you have the basics covered.

Once you have that, it’s down to you how far you want to take the app down the “entertainment” route – you could throw away fixed controls like Stepper entirely if you wanted, and instead rely on colorful buttons to get the same result.

This is one of those challenges that is best approached step by step: get something working first, then improve it as far as you want. Maybe you’re happy with a simple app, or maybe you really want to spend some time crafting a fun design. It’s down to you!

Important: It’s really easy to get sucked into these challenges and spend hours fighting with a particular bug that only exists because you wanted to get an exact effect. Don’t overload yourself with work, because you’ll just burn out! Instead, start with the simplest possible code that works, then build up slowly.

If you have lots of time on your hands, you could use something like

  • Kenney’s Animal Pack

...(which is public domain, by the way!) to add a fun theme to make it into a real game. Don’t be afraid to add some animations, too – it needs to appeal to kids 9 and under, so bright and colorful is a good idea!

To solve this challenge you’ll need to draw on skills you learned in all the projects so far, but if you start small and work your way forward you stand the best chance of success. At its core this isn’t a complicated app, so get the basics right and expand only if you have the time.

At the very least, you should:

  • Start with an App template, then add some state to determine whether the game is active or whether you’re asking for settings.
  • Generate a range of questions based on the user’s settings.
  • Show the player how many questions they got correct at the end of the game, then offer to let them play again.

Once you have your code working, try and see if you can break up your layouts into new SwiftUI views rather than putting everything in ContentView. This requires passing data between views, which isn’t something we’ve covered in detail yet, so in the meantime send data using closures – the button action from your settings view would call a function passed in by the parent view that starts the game with the user’s settings, for example.

I’m going to provide some hints below, but I suggest you try to complete as much of the challenge as you can before reading them.

Hints:

  • You should generate all your questions as soon as your game starts, storing them as an array of questions.
  • Those questions should probably be their own Swift struct, Question, storing the text and the answer.
  • When it comes to asking questions, use another state property called questionNumber or similar, which is an integer pointing at some position in your question array.
  • You can get user input either using buttons on the screen, like a calculator, or using a number pad text field – whichever you prefer.
  • If you intend to pass a closure into a view’s initializer for later use, Xcode will force you to mark it as @escaping. This means “will be used outside of the current method.”

At its simplest, this is not a hard app to build. Get that core right – get the fundamental logic of what you’re trying to do – then think about how to bring it to life. Yes, I know that part is the fun part, but ultimately this app needs to be useful, and it’s better to get the core working than try for everything at once and find you get bored part-way through.

AI Assistance

The standard .segmented Picker style cannot wrap onto multiple lines. The UISegmentedControl (which powers this in SwiftUI) is strictly designed as a single-row component. If you try to jam 12 items into it, the segments become tiny and almost impossible to tap, which is likely why you're looking for a two-row solution.

To achieve a "2 x 6" look while still acting as a single picker, you have two great options:

Option 1: The "Clean" Native Way (LazyVGrid)

Instead of using a Picker, you can build a custom grid. This gives you total control over the layout while maintaining the single selection logic. This is usually the best "modern" SwiftUI approach. 

struct ContentView: View {
    let tables = Array(1...12)
    @State private var selectedTable = 6
    
    // Define 6 flexible columns
    let columns = Array(repeating: GridItem(.flexible()), count: 6)

    var body: some View {
        NavigationStack {
            Form {
                Section("Select a Table") {
                    LazyVGrid(columns: columns, spacing: 10) {
                        ForEach(tables, id: \.self) { table in
                            Button {
                                selectedTable = table
                            } label: {
                                Text("\(table)")
                                    .frame(maxWidth: .infinity)
                                    .frame(height: 40)
                                    .background(selectedTable == table ? Color.blue : Color(.systemGray5))
                                    .foregroundColor(selectedTable == table ? .white : .primary)
                                    .cornerRadius(8)
                            }
                            .buttonStyle(.plain) // Keeps the button from highlighting the whole row
                        }
                    }
                    .padding(.vertical, 10)
                }
            }
            .navigationTitle("Times Tables")
        }
    }
}

Option 2: The "Simple" Way (WheelPicker)

If you want to keep the native Picker logic but 12 items are too cramped for a horizontal line, the .wheel style is the intended alternative for larger datasets. It's not the "two rows" you asked for, but it handles the space much better. 

Picker("Tables", selection: $selectedTable) {
    ForEach(tables, id: \.self) {
        Text("\($0)")
    }
}
.pickerStyle(.wheel)

Why the Grid is likely your best bet:

  • Touch Targets: In a segmented picker with 12 items, the tap area is about 30px wide. In a 6-column grid, it's significantly larger and more accessible.
  • Visual Hierarchy: Two rows of 6 look very balanced and "game-like," which fits your "player" theme perfectly.
  • Customization: You can easily add icons or change colors for different difficulty levels (e.g., make 12 red and 2 green).

A quick tip for your code: You can replace let tables = [1, 2, 3...] with let tables = Array(1...12) to keep things concise!

Multiple Tile Selections

QUESTION

Is the LazyVGrid able to accept multiple selected 'tiles'? Yes, absolutely! Since the LazyVGrid approach uses Buttons rather than a single Picker binding, you can easily switch from tracking one integer to tracking a Set or Array of integers.

ANSWER

Here is how you would refactor your code to allow a player to toggle multiple tables at once:

The "Multi-Select" Grid Logic

Instead of @State private var selectedTable = 6, you will use a Set. Using a Set is better than an Array here because it automatically prevents duplicates and makes it very fast to check if a specific number is "selected." 

struct ContentView: View {
    let tables = Array(1...12)
    
    // 1. Change to a Set to hold multiple values
    @State private var selectedTables: Set<Int> = [2, 5, 10] 
    
    let columns = Array(repeating: GridItem(.flexible()), count: 6)

    var body: some View {
        NavigationStack {
            Form {
                Section("Pick your tables") {
                    LazyVGrid(columns: columns, spacing: 10) {
                        ForEach(tables, id: \.self) { table in
                            Button {
                                toggleSelection(for: table)
                            } label: {
                                Text("\(table)")
                                    .frame(maxWidth: .infinity)
                                    .frame(height: 40)
                                    // 2. Check if the set contains the number
                                    .background(selectedTables.contains(table) ? Color.blue : Color(.systemGray5))
                                    .foregroundColor(selectedTables.contains(table) ? .white : .primary)
                                    .cornerRadius(8)
                            }
                            .buttonStyle(.plain)
                        }
                    }
                    .padding(.vertical, 10)
                }
                
                Section {
                    Text("Selected: \(selectedTables.sorted().map(String.init).joined(separator: ", "))")
                        .font(.caption)
                        .foregroundStyle(.secondary)
                }
            }
            .navigationTitle("Times Tables")
        }
    }

    // 3. Logic to add/remove from the Set
    private func toggleSelection(for table: Int) {
        if selectedTables.contains(table) {
            selectedTables.remove(table)
        } else {
            selectedTables.insert(table)
        }
    }
}

Why this is better than a standard Picker

  • Logical Flexibility: In a game context, you might want to allow the player to "Select All" or "Clear All." You can do this simply by modifying the Set with a single button press.
  • Visual Feedback: You can use different colors for different states (e.g., highlighting "Even" numbers in one color and "Odd" in another) which a standard Picker can't do easily.

Validation: You can easily add a rule that requires at least one table to be selected by checking if selectedTables.count > 1 inside your toggleSelection function.

A Small UI Polish Tip

If this is for a game, you might want to add a subtle animation so the tiles "pop" when selected. Just add .animation(.default, value: selectedTables) to the LazyVGrid.

General Overview

QUESTION

_I'm going to make a friendly looking game that lets the user select one or more 'times tables' (1 to 12), then the user will be presented with 'number' x 'number' and offered either a text box to enter the answer (or preferably a scroll wheel or something more interactive) to give the integer answer. I figure a bank of two rows for picking the times tables along the top, then a big (possible art directed) presentation of the multiplication (8 x 6 for example). If possible, I'd like these to by visual assets. What do you think? Is this possible in a reasonably manageable way for someone starting out?_

ANSWER

This is a fantastic project for someone starting out! It touches on all the "big" SwiftUI concepts—State management, Layout (Grids), and User Input—but it's self-contained enough that you won't get overwhelmed.

It is absolutely manageable. In fact, building it piece-by-piece will teach you more about "App Flow" than any tutorial.

The Game Architecture

To keep it manageable, think of your app in three distinct layers:

1. The Selection "Dashboard" (The Grid)

You already have the logic for this! Using the LazyVGrid with a Set<Int> is perfect. It’s clean, responsive, and easy to code.

2. The Question Engine (The Logic)

You’ll need a simple function that:

  • Picks a random number from your selectedTables set.
  • Picks another random number from 1 to 12.
  • Calculates the product so it knows if the user is right.

3. The "Art Directed" Presentation

For the big visual assets, you have two cool options:

  • System Fonts: Use Font.system(size: 80, weight: .heavy, design: .rounded). On macOS/iOS, this looks incredibly professional and "app-like" without needing actual image files.
  • SF Symbols: You can use symbols like xmark for the "times" sign or equal for the equals sign.

An Interactive "Input" Idea

Since you mentioned a scroll wheel or something interactive, a Picker with the .wheel style works, but for a game, you might want something more "tactile."

What about a Number Pad or a Horizontal Slider? For a "friendly" game, a custom number pad allows for big, chunky buttons that feel satisfying to tap.

A Starting Template

Here is how you might structure the "Big Question" area to make it look art-directed and professional: 

VStack(spacing: 20) {
    HStack {
        NumberCard(number: currentTable)
        Text("×").font(.system(size: 60, weight: .bold, design: .rounded)).foregroundColor(.orange)
        NumberCard(number: multiplier)
    }
    
    // The Input Area
    HStack {
        Text("=")
            .font(.system(size: 60, weight: .bold))
        
        TextField("?", text: $userAnswer)
            .keyboardType(.numberPad)
            .font(.system(size: 60, design: .rounded))
            .multilineTextAlignment(.center)
            .frame(width: 120)
            .background(Color(.systemGray6))
            .cornerRadius(15)
    }
}
.padding()

Why this is a great "Starter" project:

  • Iterative Growth: Start with just text. Next week, add the grid. The week after, add a "Score" counter.
  • No Assets Needed: You can do 100% of the "Art Direction" using SwiftUI's Colors, Gradients, and Shadows. You don't need to be a graphic designer to make it look "friendly."
  • Instant Feedback: You can use a simple .alert to tell the user they got it right, which is very satisfying to code.

A Little Design Tip

For that "friendly" look, try using a Linear Gradient as the background of your main view and use the Rounded font design. It instantly moves the app from "utility" to "toy."
