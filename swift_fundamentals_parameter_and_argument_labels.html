Swift Function Parameters and Argument Labels

When writing functions in Swift, a key feature of the language's expressive syntax is the clear distinction between how a function is called (the external interface) and how it is executed (the internal implementation).

This article explores the mechanics of parameter naming, argument label omission using the underscore (_), and how to leverage these patterns for clean, readable code.

1. Anatomy of a Swift Parameter

By default, every parameter in a Swift function has two names:

  • The Argument Label: Used by the caller when invoking the function.
  • The Parameter Name: Used inside the function body to reference the passed value. 
func greet(person name: String) {
    // 'name' is the internal parameter name
    print("Hello, \(name)!")
}

// 'person' is the external argument label
greet(person: "Taylor")

If you don't explicitly provide an argument label, Swift duplicates the parameter name so that it serves as both the internal and external name: 

func greet(name: String) {
    print("Hello, \(name)!")
}

// 'name' is used both inside and outside
greet(name: "Taylor")

2. Argument Label Omission with the Underscore _

There are many scenarios where an explicit argument label feels redundant and disrupts the natural flow of reading code. To omit an argument label entirely at the call site, you prepend the parameter name with an underscore (_) followed by a space.

The Syntax Comparison

Without the Underscore (Explicit Label Required)

func decode(file: String) -> Data {
    // Internal usage
    print("Reading \(file)...") 
    return Data()
}

// Call Site: Forced to use 'file:'
let data = decode(file: "users.json")

With the Underscore (Label Omitted)

func decode(_ file: String) -> Data {
    // Internal usage remains exactly the same
    print("Reading \(file)...") 
    return Data()
}

// Call Site: No external label needed
let data = decode("users.json")

Using the underscore effectively shifts the word file from being an external requirement to an internal-only variable, dramatically cleaning up the call site.

3. When to Omit vs. When to Label

Choosing whether to use an underscore is a balance between keeping code concise and ensuring it remains self-explanatory.

When to use _ (Omission)

  • The function name implies the argument: If the function name is a verb that naturally transitions into the first argument, labels are redundant.
    • Good: Bundle.main.decode("data.json")
    • Good: print("Hello World") (Instead of print(text: "Hello World"))
  • Mathematical or purely functional operations:
    • Good: abs(-5) (Instead of abs(number: -5))

When to keep explicit labels

  • To provide essential context: If removing the label makes it unclear what the argument actually represents.
    • Bad: setAlarm(8, 30) (What do 8 and 30 mean?)
    • Good: setAlarm(hour: 8, minute: 30)
  • To differentiate multiple arguments of the same type:
    • Good: func move(from source: URL, to destination: URL)

4. Summary Cheat Sheet for Reference Notes

Syntax Blueprint __________________________ Call Site Example _________________ Best Used For
func action(_ param: Type) action(value) Highly expressive, conversational API designs where the function name makes the target obvious.
func action(param: Type) action(param: value) Standard operations where the parameter name serves well as both internal variable and external description.
func action(label param: Type) action(label: value) Advanced API design where the external phrasing needs to match plain English, but the internal variable needs to remain concise.
