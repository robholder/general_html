Vim - Args And Buffers (And Other Batch Concepts)

Multi-File Operations

Operation Types

Vim has 8 different means to execute commands across multiple files.

  • :argdo --> arg list files
  • :bufdo --> buffers
  • :windo --> windows in the current tab
  • :tabdo --> tabs
  • :cdo --> items in the quickfix list
  • :cfdo --> files in the quickfix list
  • :ldo --> items in the location list
  • :lfdo --> files in the location list

Creating Lists

These commands require you to make a list in their respective categories (argument list, buffer list, window list, tab list, quickfix list, and location list) before you can run the command.

  • argdo: To create an argument list, run :args file1.txt file2.txt file3.txt
  • bufdo: To create a buffer list, just open more files (like :e file1.txt)
  • windo: To create a window list, create more split windows (like :vsplit file1.txt)
  • tabdo: To create a tab list, create more tabs (like :tabnew tab1)
  • cdo & cfdo: To create a quickfix list, use a command that uses quickfix (like :vimgrep /searchkeyword/ *.txt)
  • ldo and lfdo: To create a location list, use a command that uses location list (like :lvimgrep /searchkeyword/ *.txt)

On this page, :argdo will be used predominantly, but overall the flow is the same for all of them.

On Quickfix Lists and Location Lists

You can gather the same 'locations' across the same available file material, but it's what happens to that list (restricted to a window or not) that makes the difference to Quickfix lists and Location Lists.

This is the distinction.

Both quickfix lists and location lists can gather similar "locations" across the same material, such as search matches or error reports from multiple files. The key difference lies in how the list is scoped and managed:

  • Quickfix Lists: These are global to the entire Vim session. You can access and navigate them from any window or buffer. They're ideal for tasks spanning multiple files, like compiling code or running a search over a directory.
  • Location Lists: These are local to a specific window. Even if the list includes references to multiple files, the list itself is tied to the window where it was created. This makes location lists better suited for localized or file-specific tasks, while still allowing file-spanning entries.

In other words, it's less about the data gathered and more about the scope of the list's availability and control

Which to Utilize - Quickfix (Global) or Location (Local) Lists

Here's a quick example to illustrate when you'd favor one over the other:

When to Use a Quickfix List

  • Scenario: You're working on a codebase with multiple files, and you want to search for all instances of T0DO comments across the entire project to clean them up.
    • You'd use a quickfix list because:
    • The task involves multiple files across the project, not just the one you're currently editing.
    • With a quickfix list, you can jump between files using :cnext and :cprev, and the quickfix list remains accessible no matter which window or file you're in.

Workflow: 

:grep T0DO **/*.py
:copen

This creates a quickfix list with all T0DO entries across .py files, letting you navigate and clean them up project-wide.

When to Use a Location List

  • Scenario: You're focused on fixing linting issues in just one file, and your linter outputs warnings like "Unused variable" or "Trailing whitespace."
    • You'd use a location list because:
    • It's scoped to the window displaying the file you're working on, so it keeps your focus on the file in question.
    • You might have multiple windows open for different tasks, and each window can have its own location list specific to that task.

Workflow: 

:lopen

After your linter outputs errors to the location list, you can navigate them (:lnext, :lprev) and resolve the issues in just that file or window.

Why Choose One Over the Other?

  • Quickfix List: Best for global tasks across many files.
  • Location List: Best for focused, window-specific tasks, especially when you're juggling multiple windows or working on localized problems.

Conclusion

It boils down to workspace organization:

  • Quickfix List: Great for overarching, multi-file tasks like cleaning up search results across the entire project.
  • Location List: Helps you stay focused on tasks scoped to the current window, even if they involve referencing multiple files.

:cdo & and :cfdo AND :ldo & :lfdo

:cdo & :cfdo

The difference between :cdo and :cfdo commands, and how they interact with quickfix lists and files:

Let’s break them down and highlight the distinctions.

:cdo - Acts on Quickfix List Entries

  • What It Does:
    • The :cdo command applies a given action (like search, substitute, or any Ex command) to each entry in the quickfix list.
    • The action is carried out at the specific locations stored in the quickfix list, which may span across multiple files and line numbers.
  • Use Case:
    • Best for working on individual matches or specific locations across files.
  • Example: Let’s say you searched for T0DO across your project: 
:grep T0DO **/*.py
:cdo s/T0DO/DONE/g | update
  • This substitutes T0DO with DONE at each location in the quickfix list.

update saves the changes after substitution.

:cfdo - Acts on Files Referenced in Quickfix List

  • What It Does:
    • The :cfdo command applies the given action to every file referenced in the quickfix list, regardless of the specific locations within those files.
    • It loops through the files, executing the command on each one in its entirety.
  • Use Case:
    • Best for performing file-wide actions or tasks unrelated to specific locations (like formatting or batch updates across files).
  • Example: Suppose you want to format all files that have entries in the quickfix list: 
:cfdo normal gg=G | update
  • This formats (gg=G) the entire file for each file in the quickfix list.
  • update saves the changes to all files after formatting.

Key Differences Between :cdo and :cfdo

Command Scope Operates On Use Case
:cdo Quickfix list entries Specific locations (line/col) Targeted edits at matches  
:cfdo Quickfix list files Whole files File-wide actions  

Why Choose One Over the Other?

  • Use :cdo:
    • When you need precision—making edits at specific locations like search results, errors, or compiler warnings.
    • Example: Replacing text at each location across multiple files.
  • Use :cfdo:
    • When you need to perform a file-wide operation across all affected files, such as applying formatting, cleaning up blank lines, or saving all files.
    • Example: Running normal gg=G on every file found in the quickfix list.

If you’ve got tasks that need both, you could even use them in combination—working at individual locations first with :cdo, then handling broader file-wide cleanup with :cfdo.

:ldo and lfdo

:ldo and :lfdo are counterparts to :cdo and :cfdo, but they operate on the location list instead of the quickfix list. Here's a breakdown of what they do and when to use them:

:ldo - Acts on Location List Entries

  • What It Does:
    • Similar to :cdo, the :ldo command applies a given action (like search, replace, or any Ex command) to each entry in the location list.
    • The action is executed at the specific line and column locations stored in the location list.
  • Use Case:
    • Use :ldo when you want to apply edits to specific matches or issues reported in a location list.
  • Example: Let's say a linting tool populates the location list with unused variable warnings in the current file: 
:lopen
:ldo s/unused_variable/cleaned_variable/g | update

This substitutes all instances of unused_variable with cleaned_variable across all locations in the location list and saves the file.

:lfdo - Acts on Files Referenced in the Location List

  • What It Does:
    • Similar to :cfdo, the :lfdo command applies the given action to each file referenced in the location list. It works on the entire file, not just the specific matches in the list.
  • Use Case:
    • Use :lfdo for file-wide operations, such as formatting, applying bulk edits, or saving changes, for all files referenced in the location list.
  • Example: If the location list includes matches from multiple files, you could format all of them: 
:lfdo normal gg=G | update

This reformats each file from the location list and saves the changes.

Comparison of :ldo and :lfdo

Command Scope Operates On Use Case
:ldo Location list entries Specific locations (line/col) Targeted edits at matches
:lfdo Location list files Whole files File-wide actions

When to Use Each

  • Use ldo
    • When working with specific locations identified in the location list (e.g., resolving linting errors, replacing matched patterns).
    • It's precise and localized.
  • Use lfdo
    • When performing bulk file-wide actions (e.g., formatting, saving, or applying changes) across all files referenced in the location list.

Practical Perspective

The difference between :ldo and :lfdo mirrors the difference between :cdo and :cfdo, but the scope is narrower since location lists are tied to a specific window. While :ldo focuses on individual matches in the list, :lfdo tackles every file associated with the list, even if it's not currently displayed.

Quickfix or Quick Lists

What is a Quickfix List?

A quick list or quickfix list (same thing) is essentially a collection of locations in files - each entry corresponds to a specific file, line, and column. These entries are typically generated as a result of:

  • Running a compiler (:make) to capture errors or warnings.
  • Performing a search across multiple files.
  • Using custom scripts or commands to populate the list.

You can navigate these locations using commands like :cnext, :cprev, or display the list using :copen.

How to Create a Quickfix List

  • Running a Compiler
    • The quickfix list is often generated automatically when you run commands like :make: 
:make

This compiles the code and captures errors into the quickfix list.

Grep Results

  • You can populate the quickfix list with search results using :grep or :vimgrep: 
:grep pattern **/*.c

This searches for pattern in all .c files and populates the quickfix list with matches.

When you use the :grep command with **/*.c, it searches for files matching the given pattern (**/*.c) in the present working directory (pwd) and its subdirectories. Here’s how it works:

How :grep Functions

  • Pattern (pattern): The text you want to search for within the files.
  • Files (**/*.c): This glob pattern (**/*.c) instructs Vim to search all .c files in the current directory and all subdirectories.

Scope of Search

The **/*.c uses the shell's globbing capabilities, which means:

  • Current Directory: The search starts in the present working directory.
  • Subdirectories: The ** wildcard ensures that all subdirectories are included recursively.

Testing/Customizing

If you want to confirm the search is restricted to a specific folder or adjust its scope:

  • Use an absolute path: 
:grep pattern /path/to/directory/**/*.c
  • Limit to only the current directory (non-recursive): 
:grep pattern *.c

Manually Adding Entries

  • You can create or add to a quickfix list programmatically using the setqflist() function: 
:call setqflist([], 'a', {'title': 'My Quickfix List', 'lines': ['file1.c:10:Warning', 'file2.c:20:Error']})

This allows you to define your own list of entries.

Navigating a Quickfix List

  • Once you have a quickfix list, you can navigate through it using these commands:
    • :copen - Open the quickfix window to view all entries.
    • :cnext - Jump to the next entry in the list.
    • :cprev - Jump to the previous entry in the list.
    • :cfirst / :clast - Jump to the first/last entry.
    • :cclose - Close the quickfix window.

Within the quickfix window, pressing <Enter> on an entry will jump to that location.

Managing the Quickfix List

  • View the Current List: To see the current quickfix list in the command line: 
:copen
  • Clear the Quickfix List: You can clear the quickfix list by resetting it: 
:call setqflist([])
  • Filter or Modify the List: You can filter or manipulate the quickfix list with setqflist() and getqflist(): 
:call setqflist(filter(getqflist(), {idx, val -> val.valid == 1}))

Difference Between Quickfix and Location Lists

  • Quickfix List (:c* commands):
    • Global and applies to all files in the Vim session.
    • Used for compilers, global searches, etc.
  • Location List (:l* commands):
    • Local to a specific window.
    • Useful for file-specific searches or local errors.

Useful Commands for Workflow

  • Quickly search for a pattern and populate the quickfix list: 
vim
:vimgrep /pattern/ **/*.py
  • Apply an action to all entries using :cdo: 
:cdo s/foo/bar/g | update

This replaces foo with bar in all locations and saves the files.

When to Use It

Quickfix lists are invaluable for:

  • Error fixing after running compilers or tests.
  • Searching and editing across multiple files.
  • Automating repetitive tasks (e.g., formatting code, applying replacements).

Location Lists

What is a Location List?

A location list is a set of file locations tied to a specific window in Vim. Each window can have its own independent location list, containing entries such as:

  • Search results within the window's file.
  • Diagnostics (e.g., linting errors) related to the file.
  • Custom-defined lists.

Location lists allow precise navigation and actions within a single file or window.

How to Create a Location List

Searching with :lgrep

  • You can populate a location list by searching for a pattern: 
:lgrep pattern %

This searches for pattern within the current file and stores the matches in the location list.

Running Commands with :lvimgrep

  • To search multiple files and store the results in a location list for the current window: 
:lvimgrep /pattern/ **/*.c

This searches for pattern across all .c files and creates a location list for the current window.

Manually Adding Entries

  • You can manually add entries to a location list using the setloclist() function: 
:call setloclist(0, [{'filename': 'file1.c', 'lnum': 10, 'col': 5, 'text': 'Error here'}], 'a')
  • Here:
    • 0 refers to the current window.
    • The list is appended ('a') to the existing location list.

Navigating a Location List

Once you’ve populated a location list, you can navigate its entries using these commands:

  • :lopen - Open the location list window for the current window.
  • :lnext - Move to the next entry in the location list.
  • :lprev - Move to the previous entry.
  • :lfirst / :llast - Jump to the first/last entry in the location list.
  • :lclose - Close the location list window.

Like with the quickfix list, pressing <Enter> on an entry in the location list window will jump to the location.

Managing the Location List

  • View the Current List
    • To see the current window’s location list: 
:lopen
  • Clear the Location List
    • You can reset the location list for the current window: 
:call setloclist(0, [])
  • Filter or Modify the List
  • You can manipulate location lists programmatically using getloclist() and setloclist(): 
:call setloclist(0, filter(getloclist(0), {idx, val -> val.valid == 1}))

This filters the current location list to only include valid entries.

Difference Between Location Lists and Quickfix Lists

Feature Quickfix List (:c*) Location List (:l*)
Scope Global (applies to all files) Local (specific to one window)
Commands :copen, :cnext, :cprev :lopen, :lnext, :lprev
Use Case Compilers, global searches File-specific errors/searches

Useful Commands for Workflow

Search Within a File and Edit Results:

:lgrep /pattern/ %
:lfirst
:lnext

Use location list navigation (:lnext, :lprev) to edit matched results.

Apply Commands to All Entries:

Similar to :cdo for quickfix lists, you can use :ldo for location lists: 

:ldo s/foo/bar/g | update

This replaces foo with bar for all entries in the location list.

When to Use Location Lists

Use location lists for localized tasks:

  • When focusing on a single file or window.
  • When running diagnostics or linting tools for the current file.
  • When handling search results within the file.

argdo

The workflow below, reads in all files ending with '.srt' into Vim as args. A batch command (:argo) says to delete all lines 1 thru 2, then save the each arg/file.

"update" writes the arg/buffer IF the file has been altered. 

vim *.srt

:argdo 1,2d | update

Silent

If we run a command in argdo that would evoke an error, Vim will raise an error. There area couple of ways of suppressing this issue.

Example - if this substitute command doesn't find a match in one of the files in the 'arg' list, it will fail: 

:argdo %s/\a/*/g

We can add an 'e' at the end to fix (but this still offers a report): 

:argdo %s/\a/*/ge

A workaround to avoid the report is to use the :silent prefix: 

:silent argdo %s/\a/*/ge

However, to not have any error or report, use the :silent! prefix: 

:silent! argdo %s/\a/*/g

Using a previously typed command:

Pasting the last Ex command into the command line

External Webpage - Vimcasts.org

On the command line, you can paste the contents of a register by pressing <cntl> r{register}.

In this context, the quote: register comes in handy: it contains the last executed Ex command. For example, if we start out by running a simple substitute command: 

:%s/\a/*/g

To prefix this with argdo, simply type :argdo, then a space, then <cntl> r : which will produce: 

:argdo %s/\a/*/g

After executing this command, the : register will contain argdo %s/\a/*/g. We could prefix this with silent by typing :silent <space> <cntl> r: to produce: 

`:silent argdo %s/\a/*/g`

:argdo Vs :bufdo

External Webpage - Vimcasts.org

Just as we can use :argdo to execute a command on each of the files in the arglist, we can use :bufdo to execute a command on each of the files in the buffer list.

In some circumstances, the buffer list and arglist may refer to the same set of files. In this case, it would make no difference whether you used :bufdo or :argdo.

It’s easy to add files to the buffer list (perhaps unintentionally), so if you use :bufdo there’s always a risk that it might end up touching a file that you didn’t mean to change. Whereas the arglist only changes when you tell it to. For that reason, it may be preferable to use :argdo instead of :bufdo.

Argdo If

:argdo if &ft=='python' | <do something> | endif
