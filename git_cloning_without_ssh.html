Git - Cloning Repo Without SSH

QUESTION:

I'd like to know more about the HTTPS with PAT technique for cloning GitHub repos. Very often, I have limited access on employers' company machines, and I can't install anything. Plus, it would be nice to know of a way to clone without having to create and SSH key.

Using HTTPS with a Personal Access Token (PAT) is a really handy workaround when you’re on a locked‑down machine and can’t install SSH tools or generate keys. It’s also a perfectly valid long‑term alternative to SSH—GitHub treats PAT‑over‑HTTPS as a first‑class authentication method.

Let me walk you through how it works, why it’s useful, and a few tricks to make it less painful.

🔐 What “HTTPS with PAT” Actually Means

When you clone or push to a GitHub repo over HTTPS, Git will normally prompt you for a username and password. Since GitHub removed password authentication for Git operations, you now use:

  • Username: your GitHub username
  • Password: a Personal Access Token (PAT)

That’s it. No SSH keys, no extra software, no agents.

🧰 Why This Is Great for Locked‑Down Machines

This method shines when:

  • You can’t install GitHub CLI, SSH tools, or key managers.
  • You can’t write to your user profile (some companies block SSH key creation).
  • You’re on a temporary machine and don’t want to leave keys behind.
  • You need a quick, disposable credential you can revoke later.

All you need is:

  • Git (usually already installed)
  • A browser to generate the PAT

🛠️ How to Use HTTPS + PAT for GitHub

1. Create a Personal Access Token

On GitHub:

  • Go to Settings → Developer settings → Personal access tokens → Tokens (classic)
  • Click Generate new token
  • Give it a name and expiration date
  • Select scopes:
    • For cloning/pulling: repo:read
    • For pushing: repo (full repo access)
  • Generate and copy the token

Important: You won’t be able to see the token again, so store it somewhere safe or regenerate it later.

2. Clone a Repo Using HTTPS

Use the HTTPS URL from GitHub: 

git clone https://github.com/owner/repo.git

Git will prompt: 

Username: <your GitHub username>
Password: <paste your PAT here>

That’s it. You’re in.

3. Avoid Re‑Entering the PAT Every Time

Git can cache credentials in memory: 

git config --global credential.helper cache

Or for a longer timeout: 

git config --global credential.helper "cache --timeout=36000"

If the machine is shared or monitored, you may prefer not to cache credentials at all.

🧼 A Cleaner Way: Embed the PAT in the Remote URL (Optional)

If you’re on a disposable machine and want zero prompts: 

git clone https://USERNAME:PAT@github.com/owner/repo.git

But be careful:

  • The PAT will appear in shell history unless you prefix with a space or disable history.
  • It will be stored in .git/config unless you edit it out.

This is convenient but not recommended on machines you don’t fully control.

🛡️ Security Tips

  • Always set an expiration date on PATs.
  • Prefer fine‑grained PATs (GitHub’s newer model) when possible.
  • Revoke the token when you’re done.
  • Avoid storing PATs in plain text on company machines.

🆚 PAT vs SSH: When to Use Which

Feature HTTPS + PAT SSH
Works without installing anything ✔️ ❌ (requires keygen + agent)
Good for restricted machines ✔️ ❌
Persistent, secure auth ✔️ ✔️
Easier to rotate credentials ✔️ ❌
Safer for long‑term use ✔️ ✔️

PATs are often the best option in corporate environments.

Let’s go deeper into every angle of using HTTPS + Personal Access Tokens (PATs) with GitHub, especially for locked‑down corporate environments. I’ll keep things practical, opinionated in a helpful way, and focused on real‑world workflows that make your life easier.

🔐 1. Fine‑Grained PATs (The Newer, Safer Way)

GitHub now offers fine‑grained PATs, which are more secure than the older “classic” tokens because:

  • They apply to specific repositories (not your whole account)
  • They have explicit permissions (read‑only, read/write, admin)
  • They have forced expiration
  • You can revoke them per‑repo without touching others

How to create one

  • Go to Settings → Developer settings → Personal access tokens → Fine‑grained tokens
  • Click Generate new token
  • Choose:
  • Resource owner (your account)
  • Repository access (single repo or multiple)
  • Permissions:
    • For cloning:
      • Repository contents → Read
    • For pushing:
      • Repository contents → Read and write
  • Set an expiration date
  • Generate and copy the token

Why this matters

If your employer audits your machine or your token leaks, the blast radius is tiny. You can revoke just that one token and keep working.

🛡️ 2. Storing PATs Safely on Restricted Machines

Corporate machines often block:

  • SSH key creation
  • Credential managers
  • GitHub CLI
  • Writing to certain directories

But you still have options.

Option A — Memory‑only caching (safe on shared machines)

git config --global credential.helper cache

This stores your PAT only in RAM and forgets it after a timeout.

Extend the timeout: 

git config --global credential.helper "cache --timeout=36000"

Option B — No caching at all (most secure)

Just enter the PAT every time Git prompts you.

Annoying, but safest on machines you don’t trust.

Option C — Environment variable (use only on ephemeral machines)

export GITHUB_PAT="yourtoken"

Then use it in commands: 

git clone https://$GITHUB_PAT@github.com/owner/repo.git

This avoids storing the PAT in .git/config, but it will appear in process lists on some systems, so use with caution.

🧼 3. Avoiding PAT Leakage in Logs, Shell History, and Config Files

This is where people get burned. Here’s how to avoid it.

Never paste a PAT directly into a command

Bad: 

git clone https://username:pat@github.com/owner/repo.git

This leaks into:

  • Shell history
  • Process lists
  • .git/config

If you must embed it, do this:

A. Prefix the command with a space (bash/zsh)

git clone https://username:pat@github.com/owner/repo.git

Leading spaces prevent history logging if HISTCONTROL=ignorespace is set.

B. Immediately scrub .git/config

If you accidentally leaked it: 

sed -i '' 's/:.*@github.com/@github.com/' .git/config

C. Revoke the token afterward

Always revoke a token used this way.

🔄 4. Using PATs for Push, Pull, and Remote Management

Once you clone with HTTPS, Git stores the remote URL in .git/config.

You can inspect it: 

git remote -v

If you want to switch an existing repo to HTTPS: 

git remote set-url origin https://github.com/owner/repo.git

Then Git will prompt for username + PAT on the next push/pull.

Pushing with PAT

git push origin main

Git prompts: 

Username: yourusername
Password: <paste PAT>

Pulling with PAT

git pull

Same prompt.

🧭 5. A Clean Workflow for Corporate Machines

Here’s a workflow that balances convenience and safety.

Step 1 — Create a fine‑grained PAT with:

  • Repo‑specific access
  • Read/write permissions (if you need to push)
  • Short expiration (7–30 days)

Step 2 — Clone using HTTPS

git clone https://github.com/owner/repo.git

Step 3 — When prompted:

Username → your GitHub username
Password → your PAT

Step 4 — Use memory‑only caching

git config --global credential.helper cache

Step 5 — Revoke the PAT when you’re done

This keeps your footprint minimal.

🧩 6. When PATs Are Better Than SSH Keys

  • Even outside corporate environments, PATs can be superior:
  • Easier to rotate
  • Easier to revoke
  • No key files to manage
  • No SSH agent issues
  • Work seamlessly behind corporate proxies
  • Work on machines where you can’t write to ~/.ssh

SSH is great for long‑term personal machines. PATs are great for everything else.

🧠 7. Bonus: Using GitHub CLI Without Installing It (Portable Mode)

If you ever are allowed to run binaries but not install them, GitHub CLI (gh) can run in portable mode:

  • Download the binary
  • Put it in a folder you control
  • Run it directly

But since you said you often can’t install anything, PATs remain the most universal solution.
