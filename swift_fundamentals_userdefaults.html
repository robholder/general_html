Swift - UserDefaults

(Note the new class-based approach in Section 3.)

What is UserDefaults?

UserDefaults is a lightweight, persistent key-value store built into iOS and macOS. Think of it as a digital ledger where your app can quickly write down small pieces of information and read them back the next time the app launches.

It is designed for saving simple user configuration settings, such as:

  • App themes (e.g., Dark Mode vs. Light Mode)
  • Volume sliders or sound toggle preferences
  • "Remember Me" switches or onboarding completion flags
    ⚠️ The Golden Rule of UserDefaults: Only store small amounts of non-sensitive data (numbers, strings, booleans, or simple arrays). Because the entire file is loaded into your phone's memory when the app launches, storing large items like images or PDFs will severely degrade your app's performance.

How it Works Under the Hood

Behind the scenes, iOS creates a simple, unencrypted XML text file (known as a .plist or Property List file) inside your app’s secure sandbox directory.

When you save something, you must provide a Key (a unique string that acts as the address) and a Value (the piece of data). If you write a new value to an existing key, the old value is instantly overwritten.

Implementing UserDefaults in SwiftUI

There are two primary ways to interact with UserDefaults in SwiftUI:

  • The View Approach (@AppStorage): Perfect for simple UI toggles directly inside a single view.
  • The Class Approach (UserDefaults.standard): Necessary when managing data logic globally outside of views, using modern state systems like @Observable.

1. The Standard View Implementation (@AppStorage)

The @AppStorage property wrapper completely automates reading and writing to UserDefaults while forcing your SwiftUI views to update instantly whenever the saved data changes 

import SwiftUI

struct SettingsView: View {
    // 1. Declare variables using the @AppStorage property wrapper
    // Format: @AppStorage("UNIQUE_KEY") var variableName = DEFAULT_VALUE
    @AppStorage("isSoundEnabled") private var isSoundEnabled = true
    @AppStorage("themeColor") private var themeColor = "Blue"
    
    var body: some View {
        NavigationStack {
            Form {
                Section("Audio Settings") {
                    // 2. Interacting with the variable updates UserDefaults automatically
                    Toggle("Enable Sound Effects", isOn: $isSoundEnabled)
                }
                
                Section("Theme Customization") {
                    Picker("Theme Color", selection: $themeColor) {
                        Text("Blue").tag("Blue")
                        Text("Green").tag("Green")
                        Text("Red").tag("Red")
                    }
                    .pickerStyle(.segmented)
                }
                
                Section("Debug Info") {
                    Text("Saved values: Sound is \(isSoundEnabled ? "ON" : "OFF"), Theme is \(themeColor)")
                        .font(.caption)
                        .foregroundStyle(.secondary)
                }
            }
            .navigationTitle("Preferences")
        }
    }
}

How to Read from @AppStorage Anywhere Else

If you need to check a value on a completely different screen, you just declare the exact same @AppStorage key combination again: 

struct GamePlayView: View {
    // Re-declaring this hooks into the exact same file location on disk
    @AppStorage("isSoundEnabled") private var isSoundEnabled = true

    var body: some View {
        VStack {
            Text("Game Screen")
            if isSoundEnabled {
                Text("🔊 [Playing background music...]")
            } else {
                Text("🔇 [Audio Muted]")
            }
        }
    }
}

2. Advanced Tip: Best Practices for Keys

Typing hardcoded string keys like "isSoundEnabled" in multiple files is risky; a single typo (like "issoundEnabled") will cause your app to read from the wrong location without showing a compiler error.

The Enum Solution (Recommended)

Using an enum creates a strict, isolated container that is completely separated from normal text strings. To use an enum with @AppStorage, your enum must conform to String (known as a Raw-Value Enum). 

// Step 1: Define the Keys Enum
enum StorageKeys: String {
    case isSoundEnabled // Automatically stands for the string "isSoundEnabled"
    case themeColor     // Automatically stands for the string "themeColor"
    case favorites      // Automatically stands for the string "favorites"
}

// Step 2: Use it inside your Views
struct SettingsView: View {
    // We pass the string value wrapped inside the enum case's .rawValue
    @AppStorage(StorageKeys.isSoundEnabled.rawValue) private var isSoundEnabled = true
    @AppStorage(StorageKeys.themeColor.rawValue) private var themeColor = "Blue"

    var body: some View {
        Toggle("Sound", isOn: $isSoundEnabled)
    }
}

Implicit Raw Values: If you conform an enum to String, Swift automatically assumes that the string value matches the case name exactly. Writing case themeColor = "themeColor" is redundant; the compiler handles it behind the scenes.

3. The Modern Class Implementation (@Observable + UserDefaults.standard)

When building data layers - like a Favorites tracking system - you shouldn't leak raw storage data directly into your UI views. Instead, encapsulate the data logic inside an @Observable class using UserDefaults.standard.

Because UserDefaults cannot natively save a Swift Set, this architecture maps a high-performance runtime collection (Set) into a storage-friendly type (Array) seamlessly behind the scenes. 

import SwiftUI

// Assume a basic Model exists for your items
struct Resort: Identifiable {
    let id: String
    let name: String
}

@Observable
class Favorites {
    // The actual raw data stored as a high-performance Set
    private var resorts: Set<String>
    
    // Explicitly reference centralized storage keys
    private let key = StorageKeys.favorites.rawValue
    
    init() {
        // 1. READ: Attempt to read from disk as an Array of Strings
        if let savedItems = UserDefaults.standard.stringArray(forKey: key) {
            // 2. Hydrate the runtime Set using the array elements
            resorts = Set(savedItems)
        } else {
            // 3. Fallback to an empty Set if no previous ledger exists
            resorts = []
        }
    }
    
    // Constant time lookup checking if an item is favorited
    func contains(_ resort: Resort) -> Bool {
        resorts.contains(resort.id)
    }
    
    func add(_ resort: Resort) {
        resorts.insert(resort.id)
        save()
    }
    
    func remove(_ resort: Resort) {
        resorts.remove(resort.id)
        save()
    }
    
    private func save() {
        // 4. WRITE: Cast the Set back into an Array so UserDefaults can serialize it
        UserDefaults.standard.set(Array(resorts), forKey: key)
    }
}

When to Upgrade from UserDefaults to SwiftData

Because UserDefaults is so easy to use, it can be tempting to use it for everything. Use this checklist to decide if it's the right tool for your dataset:

Data Requirement Best Tool Why?
Simple switches, configurations, and numbers UserDefaults (@AppStorage) Lightweight, rapid layout implementation, zero configuration setup.
Global collections (e.g., Favorites lists) UserDefaults (UserDefaults.standard) Clean, rapid indexing when paired with custom @Observable classes.
Sensitive user tokens or passwords Keychain Services UserDefaults saves in plain, unencrypted text. Anyone with physical backup access can parse it.
Complex data models, relations, or deep sorting SwiftData UserDefaults cannot easily sort or link multi-property structural items together without writing cumbersome custom coders.
