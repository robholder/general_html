Swift - UserDefaults

What is UserDefaults?

UserDefaults is a lightweight, persistent key-value store built into iOS and macOS. Think of it as a digital ledger where your app can quickly write down small pieces of information and read them back the next time the app launches.

It is designed for saving simple user configuration settings, such as:

  • App themes (e.g., Dark Mode vs. Light Mode)
  • Volume sliders or sound toggle preferences
  • "Remember Me" switches or onboarding completion flags
    ⚠️ The Golden Rule of UserDefaults: Only store small amounts of non-sensitive data (numbers, strings, booleans). Because the entire file is loaded into your phone's memory when the app launches, storing large items like images or PDFs will severely degrade your app's performance.

How it Works Under the Hood

Behind the scenes, iOS creates a simple, unencrypted XML text file (known as a .plist or Property List file) inside your app’s secure sandbox directory.

When you save something, you must provide a Key (a unique string that acts as the address) and a Value (the piece of data). If you write a new value to an existing key, the old value is instantly overwritten.

Implementing UserDefaults in SwiftUI

SwiftUI provides a dedicated property wrapper called @AppStorage. This wrapper completely automates reading and writing to UserDefaults while forcing your SwiftUI views to update instantly whenever the saved data changes.

1. The Standard Implementation

Here is a self-contained layout demonstrating how to track an onboarding status and a sound preference: 

import SwiftUI

struct SettingsView: View {
    // 1. Declare variables using the @AppStorage property wrapper
    // Format: @AppStorage("UNIQUE_KEY") var variableName = DEFAULT_VALUE
    @AppStorage("isSoundEnabled") private var isSoundEnabled = true
    @AppStorage("themeColor") private var themeColor = "Blue"
    
    var body: some View {
        NavigationStack {
            Form {
                Section("Audio Settings") {
                    // 2. Interacting with the variable updates UserDefaults automatically
                    Toggle("Enable Sound Effects", isOn: $isSoundEnabled)
                }
                
                Section("Theme Customization") {
                    Picker("Theme Color", selection: $themeColor) {
                        Text("Blue").tag("Blue")
                        Text("Green").tag("Green")
                        Text("Red").tag("Red")
                    }
                    .pickerStyle(.segmented)
                }
                
                Section("Debug Info") {
                    Text("Saved values: Sound is \(isSoundEnabled ? "ON" : "OFF"), Theme is \(themeColor)")
                        .font(.caption)
                        .foregroundStyle(.secondary)
                }
            }
            .navigationTitle("Preferences")
        }
    }
}

2. How to Read from UserDefaults Anywhere Else

If you need to check a value on a completely different screen (for example, reading the sound preference on your main game loop screen), you just declare the exact same @AppStorage key combination again: 

struct GamePlayView: View {
    // Re-declaring this hooks into the exact same file location on disk
    @AppStorage("isSoundEnabled") private var isSoundEnabled = true

    var body: some View {
        VStack {
            Text("Game Screen")
            if isSoundEnabled {
                Text("🔊 [Playing background music...]")
            } else {
                Text("🔇 [Audio Muted]")
            }
        }
    }
}

Advanced Tip: Best Practices for Keys

Typing hardcoded string keys like "isSoundEnabled" in multiple files is risky; a single typo (like "issoundEnabled") will cause your app to read from the wrong location without showing a compiler error.

To solve this, organize your keys inside an enum or a struct extension to make them safe and auto-completable:

Extension Solution

// Create an extension for centralized management
extension String {
    static let isSoundEnabled = "isSoundEnabled"
    static let themeColor = "themeColor"
}

// Usage inside your views:
@AppStorage(.isSoundEnabled) private var isSoundEnabled = true

Enum Solution

While the String extension trick is clever, many iOS developers actually prefer using an enum because it creates a strict, isolated container that is completely separated from normal text strings.

To use an enum with @AppStorage, your enum must conform to String (known as a Raw-Value Enum). This tells Swift that every case represents a matching string under the hood.

Step 1: Define the Keys Enum

enum StorageKeys: String {
    case isSoundEnabled // Automatically stands for the string "isSoundEnabled"
    case themeColor     // Automatically stands for the string "themeColor"
}

Step 2: Use it inside your Views

Because @AppStorage natively expects a raw String, not a custom StorageKeys type, you have to read the enum case's .rawValue property when assigning it: 

struct SettingsView: View {
    // We pass the string value wrapped inside the enum case
    @AppStorage(StorageKeys.isSoundEnabled.rawValue) private var isSoundEnabled = true
    @AppStorage(StorageKeys.themeColor.rawValue) private var themeColor = "Blue"

    var body: some View {
        Toggle("Sound", isOn: $isSoundEnabled)
    }
}

Implicit Raw Values

Why couldn't we just declare the enum case explicitly with a string assignment like this? 

enum StorageKeys: String {
    case isSoundEnabled = "isSoundEnabled"
}

You absolutely can write it that way, but Swift has a built-in feature called Implicit Raw Values.

If you conform an enum to String, Swift automatically assumes that the string value matches the case name exactly. Writing = "isSoundEnabled" is redundant because the compiler writes that code for you behind the scenes.

Extension vs. Enum: Which is better?

Both approaches are designed to solve the exact same problem: preventing you from making a spelling typo that breaks your app configuration file layout.

  • The Extension Way (.isSoundEnabled): Cleaner to type inside your views because you get to use the ultra-short dot notation syntax, but it pollutes the global String namespace with keys meant only for storage.
  • The Enum Way (StorageKeys.isSoundEnabled.rawValue): Slightly more verbose to type, but it is vastly safer and more organized because all your app settings keys are securely boxed inside their own custom data type!

When to Upgrade from UserDefaults to SwiftData

Because UserDefaults is so easy to use, it can be tempting to use it for everything. Use this checklist to decide if it's the right tool for the job:

Data Requirement Best Tool Why?
Simple switches, numbers, strings UserDefaults (@AppStorage) Lightweight, zero setup required.
Sensitive user tokens or passwords Keychain storage UserDefaults is saved in plain, unencrypted text. Anyone with physical backup access can read it.
Complex objects, lists, or relational items SwiftData UserDefaults cannot easily filter, sort, or link multi-property datasets together without slow, custom encoding.
