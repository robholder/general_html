Blueprint: Implementing a Searchable List in SwiftUI

Adding search functionality to a List in SwiftUI is incredibly streamlined. It requires no complex delegation - just a piece of state, a modifier, a quick computed property, and swapping out your data source.

Here is the decoupled, 4-step framework you can copy and paste into any project.

The 4-Step Architecture

Step 1: Create the Search Text State

Declare a private @State string inside your view. This variable will actively listen to and store whatever the user types into the search bar. 

@State private var searchText = ""

Step 2: Attach the .searchable Modifier

Apply the .searchable modifier to the inner view structure containing your list (typically right beneath your .navigationTitle or navigation destination modifiers). Pass it a two-way binding (＄) to your state string. 

.searchable(text: $searchText, prompt: "Search...")

Step 3: Write a Filtered Computed Property

Create a computed property that sits right alongside your body variable. This property handles the actual filtering logic. If the search string is empty, it returns everything; otherwise, it filters the array using localizedStandardContains(), which handles uppercase/lowercase matching cleanly for users. 

// Replace 'YourModelType' with your actual Struct/Class name
var filteredItems: [YourModelType] {
    if searchText.isEmpty {
        return allItems // Return your baseline, unfiltered array
    } else {
        // Replace '.name' with the text property you want to search against
        return allItems.filter { $0.name.localizedStandardContains(searchText) }
    }
}

Step 4: Swap the List's Data Source

Change your List initializer so that it reads from your new computed property (filteredItems) instead of the raw, static data array (allItems). 

List(filteredItems) { item in
    // Render your row content here
}

Standard Code Blueprint (Copy-Paste Template)

Here is how those four parts assemble inside a completely standard, project-agnostic view template: 

import SwiftUI

// 1. A Sample Model for reference
struct CustomItem: Identifiable {
    let id = UUID()
    let title: String
}

struct SearchableListView: View {
    // Standard baseline data array
    let allItems = [
        CustomItem(title: "Apples"),
        CustomItem(title: "Bananas"),
        CustomItem(title: "Cherries")
    ]
    
    // [STEP 1] State to store the active query
    @State private var searchText = ""
    
    // [STEP 3] Computed property handling the filtering logic
    var filteredItems: [CustomItem] {
        if searchText.isEmpty {
            return allItems
        } else {
            return allItems.filter { $0.title.localizedStandardContains(searchText) }
        }
    }

    var body: some View {
        NavigationStack {
            // [STEP 4] List reads from the computed property, NOT the raw array
            List(filteredItems) { item in
                Text(item.title)
            }
            .navigationTitle("Inventory")
            // [STEP 2] Searchable modifier bound to state attached to the view hierarchy
            .searchable(text: $searchText, prompt: "Search inventory")
        }
    }
}

3 Pillars to Remember for Later

  • localizedStandardContains vs contains: Always prefer localizedStandardContains(). It is case-insensitive, ignores diacritics (like accents), and respects the user's localized language settings automatically.
  • Placement Matters: The .searchable modifier requires a surrounding navigation view (NavigationStack or NavigationSplitView) to physically render the bar in the user interface. If you place it on a view that isn't inside a navigation container, the search bar will not appear.
  • Performance Optimization: Because computed properties calculate on the fly during view redraws, this structural approach handles arrays of several thousand items beautifully. (If filtering massive datasets or database queries, you would instead shift this filtering logic into a background actor/thread via a ViewModel).

Revealing the Search Bar

SwiftUI's default behavior on iOS is to tuck the search bar away under the navigation title, revealing it only when the user scrolls down. This is done to save precious screen real estate, but it can easily lead to users missing the feature entirely.

You can change this behavior using the placement parameter inside the .searchable modifier.

The Solution: .navigationBarDrawer(displayMode: .always)

By explicitly setting the search bar placement configuration, you can force iOS to pin the search bar persistently on the screen so it is visible immediately upon opening the view - no pulling down required.

Code Update

Modify your .searchable modifier to include the placement argument like this: 

.searchable(
    text: $searchText, 
    placement: .navigationBarDrawer(displayMode: .always), // Forces visibility
    prompt: "Search for a resort"
)

Visual Comparison of Placement Choices

SwiftUI gives you a few different options for how that search bar behaves depending on your UI design goals:

Placement Value Behavior Best Used For
.automatic (Default) Hidden by default. Reveals itself only when the user drags the list downwards. Standard clean look; when search is a secondary feature.
.navigationBarDrawer(displayMode: .always) Permanently pinned right beneath the navigation title. Visible the second the view loads. Primary features; apps where searching is the main way users interact with data.
.navigationBarDrawer(displayMode: .automatic) Matches system default behavior (hidden until pulled down). Explicitly declaring default platform behavior.

Pro-Tip: Adding Focus to the Search Bar

If you want to go a step further and actively force the keyboard and search bar to be active the moment the screen appears, you can pair the search bar with SwiftUI's @FocusState or use the .searchable environment value, though simply setting the display mode to .always is usually the most elegant and least disruptive way to signpost the feature to your users!
