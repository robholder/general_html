Hacking With Swift - 164 - Challenge 1 Comparison

What Challenge 1 involved

The transition you've just made is a classic modernization step in iOS development. You went from manually managing a file-based cache (UserDefaults) to using a declarative, schema-driven database engine (SwiftData).

Because SwiftData integrates natively with SwiftUI's architecture, you ended up deleting a massive amount of "boilerplate" management code.

Here is the structural breakdown of exactly what changed and why.

1. Data Model: From Plain Data to Database Managed

In your original approach, ExpenseItem was a lightweight structure. Because UserDefaults doesn't natively understand custom objects, you had to conform it to Codable to manually serialize it into JSON. You also had to manually generate a UUID().

The Core Shift:

Old: struct ExpenseItem: Identifiable, Codable { ... }
New: @Model class ExpenseItem { ... }
  • The @Model macro turns your simple Swift class into a persistent database schema. SwiftData now implicitly manages the database identity, tracking, and columns for you.
  • Why Identifiable and Codable vanished: @Model automatically conforms your class to Identifiable under the hood. It also handles database migrations natively, completely eliminating the need for Codable JSON serialization.

2. Architecture: Deleting the Middleware

The most dramatic change in your codebase is the complete disappearance of the @Observable class Expenses.

In your old architecture, you had to build a custom "middleman" class to watch over your data. It had a dirty flag (didSet) that triggered a manual re-write to disk every single time an item was added or deleted. 

               OLD METHOD (Manual Pipeline)
[Data Object] ──> [JSON Encoder] ──> [UserDefaults Storage]

               NEW METHOD (Direct Pipeline)
[Data Object] ──────────────────────> [@Model Context Engine]

SwiftData completely removes the need for this manager. Instead of writing custom encoding/decoding loops, you initialize the entire infrastructure with a single line at the root of your application (iExpensesApp.swift): 

.modelContainer(for: ExpenseItem.self)

This sets up the SQLite persistent store in the background for your entire app ecosystem.

3. Reading Data: From Static State to Live Query

Because the data manager class was removed, the way ContentView reads data changed entirely.

  • Old: @State private var expenses = Expenses()
    • This loaded the data exactly once during initialization. If data changed somewhere else, you had to manually handle sync alerts.
  • New: @Query var items: [ExpenseItem]
    • This is the real magic of SwiftData. The @Query macro asks the database container for all ExpenseItem objects. Crucially, it sets up a live database listener. If an item is added or deleted anywhere in the app, the UI automatically updates without your intervention.

4. Writing and Deleting: Interacting with the Context

Instead of appending or removing elements inside a standard Swift array, you now execute mutations through the database context (@Environment(\.modelContext)), which behaves like a staging area for data changes.

Action Old (UserDefaults Array) New (SwiftData Context)
Pass Reference AddView(expenses: expenses) None required (Context is global)
Insert Data expenses.items.append(item) modelContext.insert(item)
Delete Data Loop array → find match → .remove(at:) modelContext.delete(item)

Notice how parsing logic shrank inside removeItems. You no longer have to locate the matching index across different filtered arrays; you simply hand the exact object to modelContext.delete(item), and SwiftData finds it in the database and cleans it up instantly.

Summary of Benefits

By shifting to SwiftData, your code became:

  • More Thread-Safe: Database transactions handle concurrent reads/writes safely.
  • More Performant: Instead of loading a massive JSON blob of all expenses into memory at boot time, SwiftData efficiently fetches only what it needs.
  • Cleaner: You traded complex data cycle management for declarative macros (@Model, @Query).
