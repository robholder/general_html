Hacking With Swift - 169 - Challenge

It’s time for you to build an app from scratch, and it’s a particularly expansive challenge today.

Your job is to

  • use URLSession to download some JSON from the internet,
  • use Codable to convert it to Swift types,
  • then use NavigationStack, List, and more to display it to the user.

Your first step should be to examine the JSON. The URL you want to use is this:

  • ExternalLink: FriendFace

...that’s a massive collection of randomly generated data for example users.

As you can see, there is an array of people, and each person has

  • an ID,
  • name,
  • age,
  • email address,
  • and more.

They also have an array of tag strings, and an array of friends, where each friend has a name and ID.

How far you implement this is down to you, but at the very least you should:

  • Fetch the data and parse it into User and Friend structs.
  • Display a list of users with a little information about them, such as their name and whether they are active right now.
  • Create a detail view shown when a user is tapped, presenting more information about them, including the names of their friends.
  • Before you start your download, check that your User array is empty so that you don’t keep starting the download every time the view is shown.

If you’re not sure where to begin, start by designing your types:

  • Build a User struct with properties for name, age, company, and so on,
  • Then a Friend struct with id and name.
  • After that, move onto some URLSession code to fetch the data and decode it into your types.

You might notice that the date each user registered has a very specific format: 2015-11-10T01:47:18-00:00. This is known as ISO-8601, and is so common that there’s a built-in dateDecodingStrategy called .iso8601 that decodes it automatically.

While you’re building this, I want you to keep one thing in mind: this kind of app is the bread and butter of iOS app development - if you can nail this with confidence, you’re well on your way to a full-time job as an app developer.

Tip: As always, the best way to solve this challenge is to keep it simple - write as little code as you can to solve the challenge, and for you to feel comfortable that it works well.

ANSWER

It is completely normal to feel overwhelmed by this challenge. Paul Hudson calls this "expansive" for a reason - it is the first time you are connecting multiple distinct pillars of iOS development (Networking, Data Parsing, UI layout, and State Management) completely from scratch.

Take a deep breath. We are going to achieve this by applying the "Divide and Conquer" principle. We will treat each requirement as a completely isolated micro-task so you don't have to think about the whole app all at once.

Here is your step-by-step game plan to push past this roadblock, kept as simple and minimal as possible.

Phase 1: Establish Your Data Blueprint (Models)

Before you write any UI or networking code, you need Swift types that perfectly mirror the JSON structure.

What to do:

  • Open the JSON URL in your browser and look at the structure. You will see an array of objects. Inside each of these objects is another sub-array called friends.
  • Create a new Swift file named User.swift and define your structures.

The Blueprint:

import Foundation

// The main object. It needs Identifiable so it plays nice with SwiftUI Lists.
struct User: Codable, Identifiable {
    let id: UUID
    let isActive: Bool
    let name: String
    let age: Int
    let company: String
    let email: String
    let address: String
    let about: String
    let registered: Date // We will use Paul's .iso8601 tip for this!
    let tags: [String]
    let friends: [Friend] // Nested relationship
}

// The smaller object inside the user's friend array.
struct Friend: Codable, Identifiable {
    let id: UUID
    let name: String
}

If you look closely at that raw JSON file, the id value is wrapped in quotation marks, making it a string: 

"id": "50a48c60-75b2-4cd3-9391-4c556b40523b"

So, why do we declare it as a UUID type instead of a String in Swift?

1. The Magic of JSONDecoder

JSONDecoder is highly intelligent. A UUID (Universally Unique Identifier) has a very strict, standardized format: exactly 36 characters long, broken up by four hyphens in a specific pattern (8-4-4-4-12).

Because the string in the JSON matches this exact pattern, JSONDecoder recognizes it automatically. When you tell it to decode into a UUID type, it parses that text string, verifies that it is a valid unique ID format, and converts it into a native Swift UUID object behind the scenes.

2. Why prefer UUID over String?

While your app would compile and work perfectly fine if you used let id: String, changing it to UUID offers two massive advantages:

Data Safety (Type Safety)

If you use String, any text is allowed. If the web server accidentally glitched and sent back "id": "hello-world", a String type would accept it blindly. If you use UUID, the decoder ensures the data is a legitimate, properly formatted identifier.

SwiftUI List Optimization

You are using a List(users) to display this data. For a List to be smooth and performant, SwiftUI must uniquely identify every row.

Comparing two native UUID structures in memory is computationally faster for iOS than comparing two random text strings character-by-character. It ensures your list scrolling remains buttery smooth as your data footprint grows.

What if the JSON IDs were simple numbers?

If the JSON data looked like "id": 1 or "id": "user_01", then you could not use UUID. In those cases, you would have to use Int or String respectively, because they don't match the universal 36-character format.

But since Paul Hudson's data generator used standard UUID strings, we get to take advantage of Swift's built-in UUID type for free!

Phase 2: Create a Dummy UI First

Do not try to fetch live data yet. That is a recipe for debugging nightmares. Instead, build your layout using static, fake data so you know your UI works.

What to do:

  • In ContentView, create a temporary @State array with one fake user inside it.
  • Build a simple List and a NavigationLink that pushes to a detail view. 
    struct ContentView: View {
    // Start with empty array as requested, but put a dummy item here temporarily to test your UI!
    @State private var users = [User]() 
    
    var body: some View {
        NavigationStack {
            List(users) { user in
                NavigationLink {
                    // This will be your detail view
                    UserDetailView(user: user)
                } label: {
                    HStack {
                        VStack(alignment: .leading) {
                            Text(user.name).font(.headline)
                            Text(user.company).font(.subheadline).foregroundStyle(.secondary)
                        }
                        Spacer()
                        // Status indicator
                        Circle()
                            .fill(user.isActive ? .green : .gray)
                            .frame(width: 10, height: 10)
                    }
                }
            }
            .navigationTitle("FriendFace")
        }
    }
}

Create a basic companion view UserDetailView.swift that takes a let user: User and displays their details, along with a ForEach loop over their user.friends.

Phase 3: The Data Pipeline (URLSession)

Now that your UI is ready to display users, it's time to go get them. We will add a single network function to ContentView and use the built-in async/await syntax to keep it clean.

What to do:

  • Write a function called loadData() inside ContentView.
  • Use Paul's tip to check if users.isEmpty before running the download.
  • Trigger this function using the .task modifier on your List.

The Code:

func loadData() async {
    // Guard check: Don't re-download if we already have data
    guard users.isEmpty else { return }
    
    // 1. Create the URL
    guard let url = URL(string: "https://www.hackingwithswift.com/samples/friendface.json") else {
        print("Invalid URL")
        return
    }
    
    do {
        // 2. Fetch the raw data
        let (data, _) = try await URLSession.shared.data(from: url)
        
        // 3. Configure the Decoder (Crucial step for Dates!)
        let decoder = JSONDecoder()
        decoder.dateDecodingStrategy = .iso8601 // Handles "2015-11-10T01:47:18-00:00" automatically
        
        // 4. Decode into our Swift types
        let decodedUsers = try decoder.decode([User].self, from: data)
        
        // 5. Update our UI state
        users = decodedUsers
        
    } catch {
        print("Decoding failed: \(error)")
    }
}

Hooking it up to the UI:

Attach the task to your list in ContentView: 

List(users) { user in
    // ... your list row rows
}
.task {
    await loadData()
}

Why This Plan Will Keep You Sane

By splitting the challenge into these distinct milestones, you isolate your errors:

  • If the app fails to compile in Phase 1, your properties don't match the JSON keys.
  • If the app looks wrong in Phase 2, your SwiftUI code needs layout adjustments.
  • If the app stays blank in Phase 3, your networking URL or your dateDecodingStrategy is hitting a snag.

You are fully capable of doing this. Take it one phase at a time. Don't worry about making it visually stunning right now; focus on getting the data to flow from the web server onto your screen. You've got this!
