Hacking With Swift - 169 - Challenge

It’s time for you to build an app from scratch, and it’s a particularly expansive challenge today.

Your job is to

  • an ID,
  • name,
  • age,
  • email address,

They also have an array of tag strings, and an array of friends, where each friend has a name and ID.

How far you implement this is down to you, but at the very least you should:

  • Fetch the data and parse it into User and Friend structs.
  • Display a list of users with a little information about them, such as their name and whether they are active right now.
  • Create a detail view shown when a user is tapped, presenting more information about them, including the names of their friends.
  • Before you start your download, check that your User array is empty so that you don’t keep starting the download every time the view is shown.

If you’re not sure where to begin, start by designing your types:

  • Build a User struct with properties for name, age, company, and so on,
  • Then a Friend struct with id and name.
  • After that, move onto some URLSession code to fetch the data and decode it into your types.

You might notice that the date each user registered has a very specific format: 2015-11-10T01:47:18-00:00. This is known as ISO-8601, and is so common that there’s a built-in dateDecodingStrategy called .iso8601 that decodes it automatically.

While you’re building this, I want you to keep one thing in mind: this kind of app is the bread and butter of iOS app development - if you can nail this with confidence, you’re well on your way to a full-time job as an app developer.

Tip: As always, the best way to solve this challenge is to keep it simple - write as little code as you can to solve the challenge, and for you to feel comfortable that it works well.

ID - UUID or String?

If you look closely at that raw JSON file, the id value is wrapped in quotation marks, making it a string: 

"id": "50a48c60-75b2-4cd3-9391-4c556b40523b"

So, why do we declare it as a UUID type instead of a String in Swift?

1. The Magic of JSONDecoder

JSONDecoder is highly intelligent. A UUID (Universally Unique Identifier) has a very strict, standardized format: exactly 36 characters long, broken up by four hyphens in a specific pattern (8-4-4-4-12).

Because the string in the JSON matches this exact pattern, JSONDecoder recognizes it automatically. When you tell it to decode into a UUID type, it parses that text string, verifies that it is a valid unique ID format, and converts it into a native Swift UUID object behind the scenes.

2. Why prefer UUID over String?

While your app would compile and work perfectly fine if you used let id: String, changing it to UUID offers two massive advantages:

Data Safety (Type Safety)

If you use String, any text is allowed. If the web server accidentally glitched and sent back "id": "hello-world", a String type would accept it blindly. If you use UUID, the decoder ensures the data is a legitimate, properly formatted identifier.

SwiftUI List Optimization

You are using a List(users) to display this data. For a List to be smooth and performant, SwiftUI must uniquely identify every row.

Comparing two native UUID structures in memory is computationally faster for iOS than comparing two random text strings character-by-character. It ensures your list scrolling remains buttery smooth as your data footprint grows.

What if the JSON IDs were simple numbers?

If the JSON data looked like "id": 1 or "id": "user_01", then you could not use UUID. In those cases, you would have to use Int or String respectively, because they don't match the universal 36-character format.

But since Paul Hudson's data generator used standard UUID strings, we get to take advantage of Swift's built-in UUID type for free!

Generating UUID (In Cases Where None Exist)

If the JSON didn't include an id property at all, you would have to tell Swift to ignore the JSON for that specific property and generate a brand-new UUID() yourself when the object is created.

To do this, you change how you declare the variable and provide a default value.

1. The Dynamic: Changing the Property Declaration

In your original struct, you declared it like this: 

let id: UUID // Look for a UUID inside the JSON file

If there is no id key in the JSON, you change it to this: 

var id = UUID() // Ignore the JSON; create a brand-new, random identifier right now

By giving it an initial value (= UUID()), you tell Swift: "If you are initializing this struct and nobody hands you an ID, just generate a fresh one automatically."

2. The Catch: You Must Keep Codable Happy

There is a subtle catch here. If you just add var id = UUID() to a Codable struct, the automatic Codable system will still look into the JSON file searching for a key named "id". When it doesn't find it, the decoder will crash with a keyNotFound error.

To prevent this crash, you must use the CodingKeys master checklist we discussed earlier to explicitly exclude id.

Here is exactly what your struct would look like: 

struct User: Codable, Identifiable {
    // 1. Give the ID a default value so it populates itself automatically
    var id = UUID() 
    
    let name: String
    let age: Int
    let company: String

    // 2. Explicitly leave 'id' OUT of this enum checklist!
    enum CodingKeys: String, CodingKey {
        case name
        case age
        case company
    }
}

How this works under the hood:

  • JSONDecoder looks at your custom CodingKeys enum. It sees name, age, and company. It completely ignores id because it isn't on the checklist.
  • The decoder successfully parses the name, age, and company from your JSON string.
  • When it builds the final User object, it notices that id wasn't part of the JSON stream, so it uses your fallback default value (UUID()) to generate a unique identifier on the spot.

This keeps your struct completely compatible with SwiftUI's List (via the Identifiable protocol) even if the web server never provided an ID to begin with!

The UserDetailView

UserDetailView is a completely separate view designed to focus entirely on one single user at a time.

When a user taps a row in your ContentView list, the NavigationStack pushes this detail view onto the screen. It receives the specific User object that was tapped and spreads out all of their individual properties (like their email, address, bio, and friends) in a clean, readable layout.

What Should Be Shown Here?

A good detail view breaks up a massive wall of text into distinct, organized sections. For the FriendFace app, your layout should aim to show:

  • The Hero/Header Section: The user's name, their company, and a prominent badge showing if they are currently active or offline.
  • Contact & Info Cards: Their age, email address, physical address, and registration date.
  • The "About" Section: A larger block of text for their biography/profile description.
  • The Friends List: A dedicated section or list showing the names of all the friends connected to them.

How to Build the UserDetailView

Create a new SwiftUI View file named UserDetailView.swift. To keep things incredibly simple, use a static let user: User property. Because it's a constant, this view doesn't need to fetch anything; it just layout-binds whatever data the parent view hands it.

Here is a clean, minimal implementation you can copy directly into your project: 

import SwiftUI

struct UserDetailView: View {
    // This view receives exactly on user from ContentView
    let user: User
    
    var body: some View {
        List {
            // Section 1: Status & Company
            Section("Status") {
                HStack {
                    Text("Account Status:")
                    Spacer()
                    Text(user.isActive ? "Active Now" : "Offline")
                        .fontWeight(.bold)
                        .foregroundStyle(user.isActive ? .green : .secondary)
                }
                // Two versions with the same result: HStack with Text, and LabeledContent
                // HStack version:
                HStack {
                    Text("Age")
                    Spacer()
                    Text("\(user.age) years old")
                        //.fontWeight(.bold)
                        .foregroundStyle(.secondary)
                }
                // LabeledContent version:
                LabeledContent("Age", value: "\(user.age) years old")
                LabeledContent("Company", value: "\(user.company)")
            }
            
            // Section2: Contact Information
            Section("Contact Details") {
                LabeledContent("Email", value: user.email)
                LabeledContent("Address", value: user.address)
                
                // Formatted Date display
                LabeledContent("Registered") {
                    Text(user.registered, style: .date)
                }
            }
            
            // Section 3: Bio / About
            Section("About") {
                Text(user.about)
                    .padding(.vertical, 5)
            }
            
            // Section 4: Friends List (Nested Array)
            Section("Friends (\(user.friends.count))") {
                ForEach(user.friends) { friend in
                    HStack {
                        Image(systemName: "person.crop.circle")
                            .foregroundStyle(.blue)
                        Text(friend.name)
                    }
                }
            }
        }
        .navigationTitle(user.name)
        .navigationBarTitleDisplayMode(.inline) // Keeps the title neatly centered in the bar
    }
}

The Master View to Detail View Pipeline

Now you can visually see how the entire app links up. The data flow travels seamlessly from the internet, down through your views: 

  [Internet JSON] 
         │
         ▼ (via URLSession & Codable)
 ┌────────────────┐
 │  ContentView   │ ─── (Holds the full [User] array in a List)
 └───────┬────────┘
         │ (User taps "Andrew Glouberman")
         ▼ 
 ┌────────────────┐
 │ UserDetailView │ ─── (Receives exactly ONE User object and maps it out)
 └────────────────┘

By separating your code this way, ContentView has one job (managing the big list and fetching data), and UserDetailView has one job (styling a single person's profile). This makes your app highly organized and incredibly easy to maintain!

Phase 3: The Data Pipeline (URLSession)

Now that your UI is ready to display users, it's time to go get them. We will add a single network function to ContentView and use the built-in async/await syntax to keep it clean.

What to do:

  • Write a function called loadData() inside ContentView.
  • Use Paul's tip to check if users.isEmpty before running the download.
  • Trigger this function using the .task modifier on your List.

The Code:

func loadData() async {
    // Guard check: Don't re-download if we already have data
    guard users.isEmpty else { return }
    
    // 1. Create the URL
    guard let url = URL(string: "https://www.hackingwithswift.com/samples/friendface.json") else {
        print("Invalid URL")
        return
    }
    
    do {
        // 2. Fetch the raw data
        let (data, _) = try await URLSession.shared.data(from: url)
        
        // 3. Configure the Decoder (Crucial step for Dates!)
        let decoder = JSONDecoder()
        decoder.dateDecodingStrategy = .iso8601 // Handles "2015-11-10T01:47:18-00:00" automatically
        
        // 4. Decode into our Swift types
        let decodedUsers = try decoder.decode([User].self, from: data)
        
        // 5. Update our UI state
        users = decodedUsers
        
    } catch {
        print("Decoding failed: \(error)")
    }
}

Hooking it up to the UI:

Attach the task to your list in ContentView: 

List(users) { user in
    // ... your list row rows
}
.task {
    await loadData()
}

Why This Plan Will Keep You Sane

By splitting the challenge into these distinct milestones, you isolate your errors:

  • If the app fails to compile in Phase 1, your properties don't match the JSON keys.
  • If the app looks wrong in Phase 2, your SwiftUI code needs layout adjustments.
  • If the app stays blank in Phase 3, your networking URL or your dateDecodingStrategy is hitting a snag.

You are fully capable of doing this. Take it one phase at a time. Don't worry about making it visually stunning right now; focus on getting the data to flow from the web server onto your screen. You've got this!
