Hacking With Swift - 95 - Showing and hiding views

There are several ways of showing views in SwiftUI, and one of the most basic is a sheet: a new view presented on top of our existing one. On iOS this automatically gives us a card-like presentation where the current view slides away into the distance a little and the new view animates in on top.

Sheets work much like alerts, in that we don’t present them directly with code such as mySheet.present() or similar.

Instead, we define the conditions under which a sheet should be shown, and when those conditions become true or false the sheet will either be presented or dismissed respectively.

Example View

Let’s start with a simple example, which will be showing one view from another using a sheet. First, we create the view we want to show inside a sheet, like this: 

struct SecondView: View {
    var body: some View {
        Text("Second View")
    }
}

There’s nothing special about that view at all - it doesn’t know it’s going to be shown in a sheet, and doesn’t need to know it’s going to be shown in a sheet.

Next we create our initial view, which will show the second view. We’ll make it simple, then add to it: 

struct ContentView: View { 
    var body: some View {
        Button("Show Sheet") {
            // show the sheet
        }
    }
}

Show the Sheet

Filling that in requires four steps, and we’ll tackle them individually.

First, we need some state to track whether the sheet is showing. Just as with alerts, this can be a simple Boolean, so add this property to ContentView now: 

@State private var showingSheet = false

Second, we need to toggle that when our button is tapped, so replace the // show the sheet comment with this: 

showingSheet.toggle()

Third, we need to attach our sheet somewhere to our view hierarchy. If you remember, we show alerts using isPresented with a two-way binding to our state property, and we use something almost identical here: sheet(isPresented:).

sheet() is a modifier just like alert(), so please add this modifier to our button now: 

.sheet(isPresented: $showingSheet) {
    // contents of the sheet
}

Fourth, we need to decide what should actually be in the sheet. In our case, we already know exactly what we want: we want to create and show an instance of SecondView. In code that means writing SecondView(), then… er… well, that’s it.

So, the finished ContentView struct should look like this: 

struct ContentView: View {
    @State private var showingSheet = false

    var body: some View {
        Button("Show Sheet") {
            showingSheet.toggle()
        }
        .sheet(isPresented: $showingSheet) {
            SecondView()
        }
    }
}

Passing Parameters

When you create a view like this, you can pass in any parameters it needs to work. For example, we could require that SecondView be sent a name it can display, like this: 

struct SecondView: View {
    let name: String

    var body: some View {
        Text("Hello, \(name)!")
    }
}

And now just using SecondView() in our sheet isn’t good enough - we need to pass in a name string to be shown. For example, we could pass in my Twitter username like this: 

.sheet(isPresented: $showingSheet) {
    SecondView(name: "@twostraws")
}

Break Down

Swift is doing a ton of work on our behalf here: as soon as we said that SecondView has a name property, Swift ensured that our code wouldn’t even build until all instances of SecondView() became SecondView(name: "some name"), which eliminates a whole range of possible errors.

Alternative to Swipe Away

Before we move on, there’s one more thing I want to demonstrate, which is how to make a view dismiss itself. Yes, you’ve seen that the user can just swipe downwards, but sometimes you will want to dismiss views programmatically - to make the view go away because a button was pressed, for example.

To dismiss another view we need another property wrapper - and yes, I realize that so often the solution to a problem in SwiftUI is to use another property wrapper.

Anyway, this new one is called @Environment, and it allows us to create properties that store values provided to us externally. Is the user in light mode or dark mode? Have they asked for smaller or larger fonts? What timezone are they on? All these and more are values that come from the environment, and in this instance we’re going to ask the environment to dismiss our view.

Yes, we need to ask the environment to dismiss our view, because it might have been presented in any number of different ways. So, we’re effectively saying “hey, figure out how my view was presented, then dismiss it appropriately.”

To try it out add this property to SecondView, which creates a property called dismiss based on a value from the environment: 

@Environment(\.dismiss) var dismiss

Now replace the text view in SecondView with this button: 

Button("Dismiss") {
    dismiss()
}

In Context

import SwiftUI

struct SecondView: View {
    @Environment(\.dismiss) var dismiss
    
    let name: String
    
    var body: some View {
        Text("Hello, \(name)")
        Button("Dismiss") {
            dismiss()
        }
    }
}

More on @Environment

This is a great place to stop and look under the hood. The @Environment property wrapper is one of SwiftUI's most powerful "magic" features. It allows a view to reach out and grab information or tools from the system without you having to pass them down manually.

Let's break down that specific line of code like a linguist.

1. Breaking down the Syntax

@Environment(\.dismiss) var dismiss

  • @Environment (The Property Wrapper): This tells SwiftUI: "Don't look inside this View for this value; look at the surroundings (the environment) that this view is living in."
  • (\.dismiss) (The KeyPath): This is the "address" of the specific tool you want.
    • The Backslash (\) signifies a KeyPath. It’s like a GPS coordinate to a specific property in the SwiftUI library.
    • The Dot (.) is shorthand. The full address is actually \EnvironmentValues.dismiss, but because SwiftUI knows you are looking for an Environment value, you can leave the first part out.
  • var dismiss (Your Variable Name): This is just the name you are giving the tool so you can use it later. You could technically call it var beatIt, and then call beatIt() in your button, but dismiss is the standard convention.

2. What is it actually doing?

When you show a sheet using .sheet, SwiftUI creates a "Parent-Child" relationship.

The ContentView (the Parent) is the one in charge of the showingSheet boolean. However, SecondView (the Child) doesn't have access to that boolean. Instead of "begging" the parent to close the door, the SecondView simply calls the dismiss action from the Environment.

SwiftUI sees this call and says: "Ah, this view is currently being presented as a sheet. I know how to close that!" and it flips the showingSheet toggle back to false for you.

3. The "Manager" Analogy

Think of the @Environment as a Utility Belt provided by the hotel (iOS).

  • You are a guest in a room (SecondView).
  • You don't own the light switch in the hallway, and you didn't bring your own plumbing.
  • By writing @Environment(\.dismiss), you are basically picking up the "Room Service" phone.
  • When you execute dismiss(), you are telling the hotel: "I'm checking out now." The hotel then handles the logic of clearing the room.

4. Other things in the Environment

To help you see the "logic goal," know that dismiss isn't the only thing in there. You can grab all sorts of system-wide settings using the same \. syntax:

KeyPath What it tells the View
   
\.colorScheme Is the phone in Light Mode or Dark Mode?
\.editMode Is the user currently clicking "Edit" on a list?
\.locale What language/region is the user in?
\.dynamicTypeSize How big did the user set their system font?

Why use this instead of a simple Boolean?

You could pass your $showingSheet variable into SecondView as a @Binding, but @Environment(\.dismiss) is much cleaner. It allows SecondView to be "independent." It doesn't care how it was shown (a sheet, a popover, or a full-screen cover); it just knows how to ask the system to go away.
