Hacking With Swift - 278 - Key points

Key Points

Before we get on to the challenges for this project, there are two points I want to explore in more depth to make sure you’ve understood them thoroughly: how map() and filter() fit into the larger world of functional programming, and Swift’s Result type.

Functional programming

Although I cover functional programming a lot in my book Pro Swift, I want to touch on it here too because we used it in briefly project 16 with the map() method. This method is designed to let us specify what we want rather than how we get there, and both are part of a wider programming approach called functional programming.

To demonstrate how this approach differs from a common alternative, called imperative programming, take a look at this code: 

let numbers = [1, 2, 3, 4, 5]
var evens = [Int]()

for number in numbers {
    if number.isMultiple(of: 2) {
        evens.append(number)
    }
}

That creates an array of integers, loops over them one by one, and adds those that are multiples of two to a new array called evens - we need to spell out exactly how we want the process to happen.

That code is easy to read, easy to write, and works great, but we can use a functional programming method called filter() and rewrite it as this: 

let numbers = [1, 2, 3, 4, 5]
let evens = numbers.filter { $0.isMultiple(of: 2) }

Now we don’t need to spell out how things should happen, and instead focus on what we want to happen: we provide filter() with a test it can perform, and it does the rest. This means our code is shorter, which is awesome, but it’s also improved in three other ways:

  • It’s no longer possible to put a surprise break inside the loop - filter() will always process every element in the array, and this extra simplicity means we can focus on the test itself.
  • Rather than providing a closure we can call a shared function instead, which is great for code reuse.
  • The resulting evens array is now constant, so we can’t modify it by accident afterwards.

Writing less code is always nice, but writing code that is simpler, more reusable, and less variable is even better!

Functions that accept a function as a parameter, or send back a function as their return value, are called higher-order functions, and both map() and filter() are examples of them. Swift has many more like them, but one of the most useful is compactMap(), which:

  • Runs a transformation function over every item in an array, just like map().
  • Unwraps any optionals returned by that transformation function, and puts the result into a new array to be returned.
  • Any optionals that are nil get discarded.

So, while map() will create a new array containing the same number of items as the array it took in, compactMap() might return the same amount, fewer items, or even none at all!

To see the difference between map() and compactMap() in action, try this example: 

let numbers = ["1", "2", "fish", "3"]
let evensMap = numbers.map(Int.init)
let evensCompactMap = numbers.compactMap(Int.init)

That creates an array of strings, then converts it to an array of integers using map() then compactMap(). When that code runs, evensMap will contain two optional integers, then nil, then another optional integer, whereas evensCompactMap will contain three real integers - no optionality, and no nil. Much better!

See Ai Insight below for more insight into map() and compactMap().

Result

We used Swift’s Result type as a simple way of returning a single value that either succeeded or failed, but there are a few important features I think you’ll find useful in your own code.

First, if you think about it, Result is like a slightly more advanced form of optionals. Optionals either contain some sort of value - an integer, a string, etc - or they contain nothing at all, and Result also contains some sort of value, but now rather than nothing at all for the alternative case it must contain some sort of error.

Behind the scenes, Optional and Result are both implemented as a Swift enum with two cases.

For optionals the enum is called Optional and the cases are

  • .none for nil and
  • .some with an associated value of your integer/string/etc

The enum Result has the cases:

  • .success with an associated value or
  • .failure with another associated value.

The only real difference between the two is that Swift has syntactic sugar in place for optionals - special syntax designed to make our life easier, because optionals are so common. So, things like if let and optional chaining exist for optionals, whereas Result doesn’t have any special code around it.

Second, as you’ve seen, a Result contains some sort of success value or some sort of error value, but if you ever need it, there are ways of using Result and throwing functions interchangeably.

If you have a Result and want to get back to do/catch territory, just call the get() method of your Result - this will return the successful value if it exists, or throws its error otherwise.

As an example, consider code like this: 

enum NetworkError: Error {
    case badURL
}

func createResult() -> Result<String, NetworkError> {
    return .failure(.badURL)
}

let result = createResult()

That defines some sort of error, creates a function that returns either a string or an error (but in practice always returns an error), then calls that function and puts its return value into result. If you wanted to use do/catch with that value, you could use get() like this: 

do {
    let successString = try result.get()
    print(successString)
} catch {
    print("Oops! There was an error.")
}

To go the other way - to create a Result value from throwing code - you’ll find that Result has an initializer that accepts a throwing closure. If the closure returns a value successfully that gets used for the success case, otherwise the thrown error is placed into the failure case.

For example: 

let result = Result { try String(contentsOf: someURL) }

In that code, result will be a Result<String, Error> - it doesn’t have a specific kind of Error because String(contentsOf:) doesn’t send one back.

The last thing you should know about Result is that it has functional methods you’re already used to, including map() and mapError(). For example, the map() method looks inside the Result, and transforms the success value into a different kind of value using a closure you specify - for example, it might transform a string into an integer. However, if it finds failure instead, it just uses that directly and ignores your transformation. Alternatively, mapError() transforms the error from one type to another, which can be helpful if you want to homogenize error types in one place.

This is one of the many things to love about functional programming: once you understand the "takes a closure and uses it to transform stuff" nature of map(), you’ll find it exists on arrays, Result, and even Optional!

Let's strip away terms like "higher-order functions" and look at this like a real-world sorting machine.

The Big Picture: A Box of Strings

Imagine you have a conveyor belt carrying a box of strings: ["1", "2", "fish", "3"]. Your goal is to turn them into actual numbers so you can do math with them. To do that, you use a tool called Int(), which tries to convert strings to numbers.

But Int() is cautious. If you hand it "1", it says, "Great, that's the number 1."

If you hand it "fish", it panics and says, "I don't know what this is, here is nothing (nil)."

Here is how map() and compactMap() handle that conveyor belt differently.

1. How map() handles it (The Rigid Machine)

map() has one strict rule: Whatever goes in, the exact same number of items must come out. It doesn't care if the transformation failed.

When you pass your box through map(Int.init), it processes every item blindly:

  • "1" becomes Optional(1)
  • "2" becomes Optional(2)
  • "fish" fails and becomes nil
  • "3" becomes Optional(3)

Your final result: [Optional(1), Optional(2), nil, Optional(3)]

The Problem: Because map() is forced to return something for "fish", it spits out a nil. Because there is a nil in the box, Swift forces all the other numbers to wear protective suits called "Optionals." You can't do math with these numbers yet because Swift doesn't trust that the box is safe.

2. How compactMap() handles it (The Smart Filter)

compactMap() is the smart upgrade. It does the exact same conversion work as map(), but it adds a built-in quality control step: If an item turns into nil, throw it in the trash. If it succeeds, take off its protective optional suit.

When you pass the exact same box through compactMap(Int.init):

  • "1" becomes 1
  • "2" becomes 2
  • "fish" fails (nil) → Thrown in the trash!
  • "3" becomes 3

Your final result: [1, 2, 3]

Summary Table: Side-by-Side Comparison

Machine Input Count Output Count What's inside the output?
map() 4 items 4 items A messy mix of wrapped numbers and nil.
compactMap() 4 items 3 items Pure, clean integers ready to be used.

Use map() when you want a 1-to-1 transformation where every single element matters. Use compactMap() when you are converting data that might fail, and you just want a clean array of the successful results.

Can you do the same reanalysis of the next section which covers 'Result' and its various aspects?

ANSWER

That text is explaining Result, which is essentially a specialized delivery box for code that has a chance of blowing up (like downloading a file or fetching data from a database).

Here is the concept stripped of the textbook jargon.

The Big Picture: Optionals vs. Result

Think of an Optional as a mystery gift box. When you open it, you either get a prize (.some), or the box is completely empty (.none / nil). It tells you that something failed, but it can never tell you why.

A Result box is the upgraded version. It can never be empty. When you open a Result box, you are guaranteed to find one of two things:

  • .success: A shiny green card with your data (like a String or an Int).
  • .failure: A bright red card explaining exactly what went wrong (the specific error).
Box Type What's inside Case A? What's inside Case B? Syntactic Sugar?
Optional A real value Absolutely nothing (nil) Yes (if let, ??, ?.)
Result A real value A specific error message No (Must use switch or methods)
Note: The text mentions "syntactic sugar" - that just means Swift gives optionals nice shortcuts like if let because we use them constantly. Result doesn't get those shortcuts, so it feels a bit more manual.

Feature 1: Turning a Result into a do/catch (The get() method)

Sometimes you have a Result box, but you prefer working with standard Swift error handling (do, try, catch).

You can rip open the box using .get().

  • If the box contains a .success, .get() hands you the value directly.
  • If the box contains a .failure, .get() immediately throws the error, kicking you straight into the catch block. 
// Opening the box with get()
do {
    let finalData = try myResultBox.get() 
    print("Success: \(finalData)")
} catch {
    print("The box contained an error, so I ended up here!")
}

Feature 2: Turning a throwing function into a Result

You can also go the exact opposite direction. If you have a sketchy function that throws errors, you can wrap it in a Result box right at the birth site. 

// This automatically creates a Result box for you
let myResultBox = Result { try String(contentsOf: websiteURL) }

Swift runs the code inside the braces. If the website loads, it automatically labels the box .success(text). If the internet drops, it catches the error and labels the box .failure(internetError). No do/catch required on this line!

Feature 3: Transforming inside the box (map and mapError)

Just like arrays, Result boxes have a map() tool. It allows you to modify what's inside the box without actually opening it yet.

map() changes the Success

Imagine you have a Result box containing the string "100". You want to turn it into the integer 100.

  • If it's a .success: map reaches inside, converts "100" to 100, and seals the box back up as a success.
  • If it's a .failure: map looks inside, sees an error, closes the lid, and ignores your conversion entirely. The error remains untouched.

Use map when you want to change the successful data inside the box, but leave any errors completely untouched. 

// A Result box containing a String ("42"), or a NetworkError
let stringResult: Result<String, NetworkError> = .success("42")

// map() reaches inside and converts the String "42" into the Int 42
let integerResult: Result<Int, NetworkError> = stringResult.map { stringValue in
    Int(stringValue) ?? 0
}

// NOTE: If stringResult had been a .failure, map() would have ignored 
// the integer conversion entirely and just passed the error along.

mapError() changes the Failure

This does the exact opposite. It ignores successful data, but if it finds an error, it lets you translate a messy system error (like Code 404) into a clean, readable app error (like .pageNotFound).

Use mapError when you don't care about the successful data, but you want to take a messy, low-level error and translate it into a clean, custom error that your app understands. 

// Custom, clean errors for our app's UI
enum AppUIError: Error {
    case connectionLost
}

// A Result box containing a String, but holding a messy system URLSession error
let systemResult: Result<String, Error> = .failure(URLError(.notConnectedToInternet))

// mapError() ignores success, but catches the system error and swaps it out
let cleanResult: Result<String, AppUIError> = systemResult.mapError { systemError in
    // Turn any internet-related system error into our clean .connectionLost case
    return AppUIError.connectionLost
}
