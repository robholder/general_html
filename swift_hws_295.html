QUESTION 10

Which is true?

  • Option 1: Apple recommends @State properties should use public access control.
  • Option 2: We can trigger haptic effects using the sensoryFeedback() modifier.

Option 2 is true.

Here is a quick breakdown of why this is correct and why Option 1 is incorrect.

Why Option 2 is True

The .sensoryFeedback() modifier is the standard, modern way to trigger haptics in SwiftUI. It allows you to watch a specific piece of state and play a physical vibration (like a gentle tap, success notification, or selection click) on the user's device whenever that state changes. 

Button("Purchase") {
    isPurchased.toggle()
}
// Triggers a success haptic tap whenever isPurchased becomes true
.sensoryFeedback(.success, trigger: isPurchased)

Why Option 1 is False

Apple actually recommends the exact opposite: @State properties should almost always be marked as private.

@State is specifically designed to manage a view's internal private storage. Because SwiftUI manages the memory lifecycle of that variable behind the scenes, allowing outside views to directly modify a view's raw @State can break things and cause unexpected UI bugs. If an outside view needs access, you should use @Binding or pass it down via an environment instead.

QUESTION 11

Option 1: Once a Timer has started, it can't be stopped. Option 2: We can use the dynamicTypeSize() modifier to declare which Dynamic Type sizes a view supports.

Option 2 is true.

Here is the quick breakdown of why this is correct and why Option 1 is incorrect.

Why Option 2 is True

The .dynamicTypeSize() modifier allows you to control exactly how much your text can scale when a user adjusts their system font size settings. You can pass a range to define a minimum and maximum cap, or specify a single fixed size to declare exactly what the view supports: 

// Limits the text scaling to a maximum of 'xxxLarge' 
// so it doesn't break your layout boundaries
Text("Resort Details")
    .dynamicTypeSize(...DynamicTypeSize.xxxLarge)

Why Option 1 is False

A Timer absolutely can be stopped. If you are using a standard Foundation Timer, you stop it by calling its .invalidate() method, which halts the timer entirely and removes it from the run loop.

If you are using a modern SwiftUI-style publishing timer (Timer.publish), you stop it by calling .connect().cancel(), which shuts down the data stream and stops the repeats instantly.

  • Option 2: EditButton will automatically switch between Edit and Done when tapped.

Option 2 is true.

Here is the quick breakdown of why this is correct and why Option 1 is incorrect.

Why Option 2 is True

The EditButton is a built-in SwiftUI view specifically designed to manage container editing states (like deleting or reordering rows in a List).

When you place it in a toolbar, SwiftUI hooks it directly into the environment's editMode state. When a user taps it, the button automatically flips its text between "Edit" and "Done," while simultaneously telling the list below to reveal or hide its delete red circles and reordering handles.

JSON stands for JavaScript Object Notation (not Networking).

It is a lightweight, human-readable text format used for storing and transporting structured data. Even though it originated in the JavaScript ecosystem, it is completely language-independent.

QUESTION 13

  • Option 1: SwiftData queries must have a sort order specified.
  • Option 2: The Codable protocol is actually a combination of Encodable and Decodable.

Option 2 is true.

Here is the quick breakdown of why this is correct and why Option 1 is incorrect.

Why Option 2 is True

The Codable protocol is not an independent protocol written from scratch. Instead, it is a typealias - a convenient nickname—that glues two distinct protocols together.

When you declare that a struct conforms to Codable, you are telling the Swift compiler to conform to both parts of the contract simultaneously:

  • Decodable: Allows data (like a JSON string) to be parsed into your native Swift object.
  • Encodable: Allows your native Swift object to be converted out into data (like a JSON string).

Under the hood, Apple defines it exactly like this: 

typealias Codable = Encodable & Decodable

Why Option 1 is False

SwiftData queries (@Query) do not require a sort order.

If you write a simple @Query var resorts: [Resort], SwiftData will compile perfectly and pull your records out of the database using its default internal sequencing (typically the order in which they were originally saved to disk). Providing a specific property sorting order (like sorting alphabetically by name) is entirely optional.

QUESTION 14

  • Option 1: If a VStack has a foreground style and some text inside also has a foreground style, the VStack's foreground style is used.
  • Option 2: Decorative images are images that are merely there to make the UI look nicer.

Option 2 is true.

Here is the quick breakdown of why this is correct and why Option 1 is incorrect.

Why Option 2 is True

Decorative images are assets that serve purely as visual styling or "eye candy" for the layout—like background gradients, accent shapes, or abstract graphics.

Because they provide zero actual information or context to the app's contents, you declare them in SwiftUI using Image(decorative: "name"). This tells the system's VoiceOver accessibility engine to completely ignore the asset, preventing the screen reader from reading out meaningless image names to visually impaired users.

Why Option 1 is False

In SwiftUI, the inner-most, closest modifier always wins. If you apply a broad style modifier to a container like a VStack, that style acts as a fallback default for everything inside it. However, if a child element (like a Text view) explicitly declares its own .foregroundStyle(), it overrides the parent layout's style for itself. 

VStack {
    Text("I will be Blue") // Fallback inherited style
    
    Text("I will be Red!") // Inner modifier overrides the parent!
        .foregroundStyle(.red) 
}
.foregroundStyle(.blue)

QUESTION 15

  • Option 1: When creating a custom alignment guide we must provide a default value.
  • Option 2: Arrays cannot be used with @State.

Option 1 is true.

Here is the quick breakdown of why this is correct and why Option 2 is incorrect.

Why Option 1 is True

When you define a custom alignment guide by conforming a new type to the AlignmentID protocol, Swift requires you to implement a static defaultValue(in:) function.

This default value acts as a fallback plan. It tells SwiftUI exactly how to align any other view in your layout hierarchy that hasn't explicitly specified its own custom alignment instructions for that guide.

Why Option 2 is False

Arrays can absolutely be used with @State (in fact, you are doing exactly this in your ContentView with strings and properties!).

You can use @State with any Swift type - including arrays, dictionaries, integers, booleans, or custom data structs. Whenever you append, remove, or reorder items inside a @State private var myItems: [String] array, SwiftUI detects the mutation and automatically schedules a view redraw to update your UI.

QUESTION 16

  • Option 1: Ease in animations start slow and end fast.
  • Option 2: SwiftUI's views cannot be created as @State properties.

Option 1 is true.

Here is the quick breakdown of why this is correct and why Option 2 is incorrect.

Why Option 1 is True

An "Ease In" animation curve starts at a zero or near-zero velocity and accelerates as it moves forward, meaning it starts slow and ends fast. It is modeled after physical objects that need time to build momentum from a standstill (like a car accelerating from a stoplight).

Why Option 2 is False

While it is an anti-pattern (bad practice), SwiftUI views can technically be created as @State properties, meaning the statement "cannot be created" is false.

You could write @State private var myView = Text("Hello"), and it will compile. However, you should never do this. Views in SwiftUI are lightweight blueprint structures designed to be recreated constantly by the rendering engine. Storing a view instance directly inside @State forces it to persist in memory, which bypasses SwiftUI's lifecycle engine and causes major UI rendering and update bugs.

QUESTION 17

  • Option 1: Images from SF Symbols can be shown larger or small by using the font() modifier.
  • Option 2: Each modifier can be applied only once to a given view.

Option 1 is true.

Here is the quick breakdown of why this is correct and why Option 2 is incorrect.

Why Option 1 is True

Apple designed SF Symbols to behave exactly like system typography. Because they are vector-based typographic glyphs rather than static pixel bitmaps, you can scale them flawlessly using the .font() modifier.

Whether you use semantic sizes like .font(.largeTitle) or precise numeric sizes like .font(.system(size: 48)), the symbol will scale up or down smoothly without any blurriness or pixelation.

Why Option 2 is False

You can absolutely apply the same modifier multiple times to a single view, and SwiftUI will layer them sequentially from the inside out.

A classic example of this is creating layered borders or compound padding gaps by chaining modifiers back-to-back: 

Text("Hello World")
    .padding(10)          // Inner padding gap
    .background(.blue)    // Inner blue background
    .padding(10)          // Outer padding gap
    .background(.green)   // Outer green frame layer

QUESTION 18

  • Option 1: We always need to bind text fields to strings.
  • Option 2: SwiftUI can animate several properties changing at the same time.

Option 2 is true.

Here is the quick breakdown of why this is correct and why Option 1 is incorrect.

Why Option 2 is True

SwiftUI handles multiple simultaneous animations beautifully. When you wrap state changes inside a single withAnimation() block, or when a view tracks a state variable modified by a .animation() modifier, SwiftUI automatically animates every single affected layout property at the same time.

For example, a view can smoothly change its size, rotate, change its color, and fade its opacity all at once during a single state change transition.

Why Option 1 is False

While TextField views are most commonly bound to strings, you are not restricted to strings alone.

SwiftUI text fields have a built-in format parameter that allows you to bind them directly to other data types like numbers or dates. SwiftUI automatically manages the typing conversions for you: 

// Binds a TextField directly to a numeric Double property!
@State private var price: Double = 0.0

var body: some View {
    TextField("Enter price", value: $price, format: .currency(code: "USD"))
        .keyboardType(.decimalPad)
}

QUESTION 19

  • Option 1: The Identifiable protocol has no requirements.
  • Option 2: Creating a property using @Environment(\.horizontalSizeClass) will keep the value updated when the size class changes.

Option 2 is true.

Here is the quick breakdown of why this is correct and why Option 1 is incorrect.

Why Option 2 is True

The @Environment property wrapper creates an active, live dependency on the system's environment values.

When you read \.horizontalSizeClass, SwiftUI subscribes your view to the device's current layout state. If a user rotates their iPhone from portrait to landscape, or enters split-screen multitasking on an iPad, the size class dynamically flips between .compact and .regular. SwiftUI intercepts this event, updates the property value automatically, and forces the view to redraw to adapt to the new layout.

Why Option 1 is False

The Identifiable protocol has one very strict, non-negotiable requirement: any type conforming to it must provide a unique id property.

The type of id can vary (it is commonly a UUID or a String), but it must conform to the Hashable protocol so that SwiftUI can uniquely track, update, and animate individual elements inside collections like a List or a ForEach. You can see this directly in your own SortType enum from earlier, where you had to explicitly declare var id: String { self.rawValue } to satisfy the compiler!

QUESTION 20

  • Option 1: The NavigationBar view lets us show new views and also place text at the top of the screen.
  • Option 2: We can send nil to the animation() modifier.

Option 2 is true.

Here is the quick breakdown of why this is correct and why Option 1 is incorrect.

Why Option 2 is True

You can pass nil to the .animation(_:value:) modifier to explicitly disable animations for a specific view or a specific section of your view hierarchy.

This is incredibly useful when you want a broad container (like a VStack) to animate all of its layout changes, but you have one specific element inside it (like a text field or a profile picture) that you want to instantly snap into place without any transition effect. 

VStack {
    Text("I animate smoothly!")
    
    Text("I snap instantly without any animation!")
        .animation(.none, value: changeCount) // Passing .none (which evaluates to nil) turns it off!
}
.animation(.easeInOut, value: changeCount)

Why Option 1 is False

There is no actual view type called NavigationBar in SwiftUI.

Instead, SwiftUI uses containers like NavigationStack or NavigationSplitView to manage showing new views. To place text or buttons at the top of the screen, you attach modifiers inside those containers, such as .navigationTitle("Title") or .toolbar().

QUESTION 21

  • Option 1: QR codes are just barcodes with more colors.
  • Option 2: Using scaledToFill() with an image might mean parts of an image lie outside its container's frame.

Option 2 is true.

Here is the quick breakdown of why this is correct and why Option 1 is incorrect.

Why Option 2 is True

When you apply the .scaledToFill() modifier to an image, SwiftUI forces the image to expand until it completely fills the dimensions of its frame, regardless of the aspect ratio.

If the image's original shape doesn't match the shape of the container frame, the image will spill over the borders. The parts that lie outside the frame boundaries remain visible and overlap surrounding UI elements unless you explicitly cut them off by adding a modifier like .clipped() or .clipShape().

Why Option 1 is False

QR codes are not defined by color; they are defined by their physical structure and layout. Traditional barcodes are one-dimensional (1D) linear codes that store data horizontally using lines of varying widths.

A QR code is a two-dimensional (2D) matrix code that stores data both horizontally and vertically using geometric modules (typically black and white squares). This grid-based structure allows a QR code to hold hundreds of times more data than a standard linear barcode, entirely independent of color.

QUESTION 22

  • Option 1: The sequenced(before:) modifier lets us create chains of gestures.
  • Option 2: @Binding cannot be used with a private property.

Option 1 is true.

Here is the quick breakdown of why this is correct and why Option 2 is incorrect.

Why Option 1 is True

The .sequenced(before:) modifier is specifically used to chain two gestures together in a strict chronological order. It forces SwiftUI to wait for the first gesture to finish completely before it activates and listens for the second gesture.

A classic use case is requiring a user to press and hold an item (a LongPressGesture) before they are allowed to start dragging it around the screen (a DragGesture).

Why Option 2 is False

You can absolutely mark a @Binding property as private. In fact, it is common practice to do so if that binding is only used internally within that specific subview's layout body.

The private keyword simply prevents other parts of your code from accessing that property directly from the outside, but it has no impact on @Binding's core ability to establish a two-way data link with a parent view's state. 

struct CustomToggleView: View {
    // Completely valid! Accessible only inside this struct,
    // but still safely linked to the parent view's data.
    @Binding private var isOn: Bool 

    var body: some View {
        Toggle("Status", isOn: $isOn)
    }
}

  • Option 2: We can use @Environment only once per view.

SwiftUI gives you total control over how screen readers (like VoiceOver) interpret your layout. You can use the .accessibilityElement(children: .combine) modifier to merge multiple distinct views - such as a flag image, a resort title, and a subtitle—into a single, unified accessible item.

Instead of forcing a visually impaired user to swipe three separate times to hear the card's contents, VoiceOver will select the entire row layout as a single block and read out the combined details naturally.

QUESTION 24

  • Option 1: We can bind an alert() modifier to an optional value.
  • Option 2: SF Symbols don't have any default accessibility labels.

Option 1 is true.

Here is the quick breakdown of why this is correct and why Option 2 is incorrect.

Why Option 1 is True

SwiftUI provides a specific overload of the .alert() modifier that binds directly to an optional source of truth (an item).

When that optional state contains data (is not nil), SwiftUI unpacks the object, passes it directly into your alert configuration block, and presents the alert popup. As soon as the user dismisses the alert, SwiftUI automatically sets that optional value back to nil. 

// A completely valid alert bound to an optional struct
.alert("Error Occurred", presenting: activeError) { error in
    Button("Retry") { /* Handle retry */ }
} message: { error in
    Text(error.localizedDescription)
}

Why Option 2 is False

SF Symbols actually do come with smart, built-in default accessibility labels provided by Apple.

For instance, if you place Image(systemName: "trash") into your view, VoiceOver doesn't just stay silent or read out the filename "trash" - it automatically announces it to the user as "Trash" or "Delete". However, you should still override these defaults with custom accessibility labels (.accessibilityLabel()) if the symbol serves a unique semantic purpose in your specific layout (e.g., if a trash can icon is being used specifically to mean "Clear Cache").

QUESTION 25

  • Option 1: Swift's arrays use generics.
  • Option 2: "Conic gradient" is another name for a radial gradient.

Option 1 is true.

Here is the quick breakdown of why this is correct and why Option 2 is incorrect.

Why Option 1 is True

Swift's arrays are built entirely on generics, meaning they can hold any type of data, but they must be specialized to hold exactly one specific type at a time.

When you write [String] or [Int], you are using syntactical shorthand for the true generic type syntax: Array<String> or Array<Int>. The <Element> placeholder allows the array to handle your custom data models safely without losing type safety.

Why Option 2 is False

Conic gradients and radial gradients are completely different visual patterns.

  • Radial Gradient: Radiates color outward from a central point in a circular or elliptical pattern, like ripples in water or a glowing light bulb.
  • Conic Gradient: Sweeps colors 360 degrees around a central point like a clock hand or a color wheel. It is also frequently called an angular gradient.

QUESTION 27

  • Option 1: A view is always and exactly the same size as its body.
  • Option 2: Child views must always use less than or equal to the amount of space the parent offers them.

Both options are technically false, but Option 1 is significantly closer to a fundamental truth of SwiftUI - it is considered a core layout rule, though there is a technical exception.

Let's break down the rules of SwiftUI's layout engine to see how both work.

The Nuance of Option 1 (The Core Rule)

One of the three absolute rules of the SwiftUI layout engine is: A view chooses its own size, and the parent must respect it. Because a container view wraps its content tightly, a view's final dimensions are almost always exactly the size of its underlying body.

The tiny catch: Modifiers like .frame(width: 100, height: 100) actually wrap your view inside a new, invisible layout layer. In those cases, the parent sees the size of the modifier, which might differ from the raw, unfiltered size of the inner body content.

Why Option 2 is Flat Out False

Child views do not have to stay within the bounds offered by their parents.

In SwiftUI, layout is a negotiation:

  • The parent offers a size (e.g., "You have 100x100 points of space").
  • The child decides its own size based on its content ("Thanks, but I am a large image and I need 300x300 points").
  • The parent must accept that size.

If a child view demands more space than the parent offers, it will simply break out of the parent's boundaries and layout as an overflow, spilling onto the surrounding UI unless the parent explicitly uses a modifier like .clipped() to chop off the excess.

QUESTION 28

  • Option 1: Changing any @State property of a view causes SwiftUI to reinvoke the body property.
  • Option 2: Arrays cannot use the @State property wrapper.

Option 1 is true.

Here is the quick breakdown of why this is correct and why Option 2 is incorrect.

Why Option 1 is True

This is the core foundation of how SwiftUI works: Data is the single source of truth for your user interface. When you mark a property with @State, SwiftUI establishes a watch on that memory address. The moment that variable changes (even by a tiny bit, like toggling a boolean or incrementing an integer), SwiftUI detects the mutation, invalidates the current layout, and instantly re-runs (re-invokes) the body property of that view to compute the brand-new UI layout.

Why Option 2 is False

As we touched on earlier, arrays can absolutely use the @State property wrapper.

@State works flawlessly with any standard Swift value type, including arrays. When you append a new item, remove an element, or sort an array that is wrapped in @State, Swift handles it as a structural mutation. SwiftUI intercepts this mutation and immediately triggers a UI refresh to display the updated list.

QUESTION 29

  • Option 1: contentShape() allows us to control the tap area for a view.
  • Option 2: We can show an alert by calling its show() method.

Option 1 is true.

Here is the quick breakdown of why this is correct and why Option 2 is incorrect.

Why Option 1 is True

By default, interactive elements like hit testing and tap gestures only trigger when a user taps on pixels that actually contain content (like text or an icon). If you have a container with empty space, tapping that empty space does nothing.

The .contentShape() modifier allows you to define a custom hit-test geometry for a view. For example, applying .contentShape(Rectangle()) to a row container makes the entire boundary tappable, including all the transparent, empty spaces between your text and images.

Why Option 2 is False

SwiftUI does not use imperative code commands like a .show() method to present UI elements.

Instead, SwiftUI is completely declarative and state-driven. To show an alert, you attach the .alert() modifier to a view and bind it to a boolean or optional @State property. You trigger the alert simply by changing that state variable to true or giving it a value, and SwiftUI handles the presentation automatically. 

@State private var showAlert = false

var body: some View {
    Button("Trigger Alert") {
        showAlert = true // This state change triggers the alert!
    }
    .alert("Hello!", isPresented: $showAlert) {
        Button("OK", role: .cancel) { }
    }
}

QUESTION 30

  • Option 1: Custom view modifiers must conform to the ViewModifier protocol.
  • Option 2: ForEach views can't loop over more than 10 items, because SwiftUI doesn't allow it.

Option 1 is true.

Here is the quick breakdown of why this is correct and why Option 2 is incorrect.

Why Option 1 is True

To build a custom, reusable view modifier in SwiftUI, your structure must conform to the ViewModifier protocol.

Conforming to this protocol requires you to implement a single method: body(content: Content) -> some View. Inside this method, you take the existing content (the view the modifier is being attached to) and wrap or decorate it with whatever paddings, backgrounds, shapes, or animations you need.

Why Option 2 is False

ForEach loops in SwiftUI can handle hundreds, thousands, or even millions of items without a problem.

The confusion about a "10-item limit" comes from structural container views like VStack, HStack, or Group. SwiftUI's view builder limits you to passing 10 static, individual child views directly inside a single layout group. However, a ForEach loop counts as just one single child view to the parent container, regardless of how many dynamic items it generates internally.

QUESTION 31

  • Option 1: If we place views inside a Group the parent view decides how those views should be laid out.
  • Option 2: We can have only one @State property in a view.

Option 1 is true.

Here is the quick breakdown of why this is correct and why Option 2 is incorrect.

Why Option 1 is True

A Group view in SwiftUI doesn't have any layout behavior of its own - it doesn't stack views horizontally or vertically, and it doesn't add spacing. Instead, it is a structural tool used to group views together (often to bypass the 10-view limit or apply a single modifier to multiple elements).

Because Group is transparent to layout, the responsibility falls completely to the parent container.

  • If you place a Group inside a VStack, the parent lays the items out vertically.
  • If you place that same Group inside an HStack, the parent lays them out horizontally.

Why Option 2 is False

You can have as many @State properties inside a single view as your application needs.

It is incredibly common for a view to track multiple moving pieces of data independently. For example, a login screen might use three separate @State properties to track the user's input and loading status simultaneously: 

struct LoginView: View {
    // Declaring multiple @State properties is completely valid and standard practice
    @State private var username = ""
    @State private var password = ""
    @State private var isLoading = false
    
    var body: some View {
        // View layout code here...
    }
}

QUESTION 32

  • Option 1: SwiftUI stores view positions and sizes as integers.
  • Option 2: The containerRelativeFrame() modifier lets us size a child view relative to its nearest container.

Option 2 is true.

Here is the quick breakdown of why this is correct and why Option 1 is incorrect.

Why Option 2 is True

The .containerRelativeFrame() modifier gives a view precise information about its parent container's layout size. It allows you to dynamically scale a child view relative to its nearest container (like a ScrollView, a List, or the window itself) using an axis and a sizing closure.

For example, you can use it to explicitly state: "I want this card to always take up exactly three-quarters (3/4) of the width of the scroll view it lives inside." 

Image("hero-banner")
    .resizable()
    // Scales the image width to 75% of its parent container's width
    .containerRelativeFrame(.horizontal) { size, axis in
        size * 0.75
    }

Why Option 1 is False

SwiftUI does not use integers (Int) to store layout positions or sizes. Instead, it uses CGFloat (Core Graphics Float), which is a floating-point type capable of handling precise decimal numbers.

This precision is required because modern Apple devices rely on high-density Retina displays. Points on a screen do not map 1:1 with physical pixels; a position might sit perfectly at a fractional coordinate like 10.5 points to remain crisp, smooth, and accurately aligned across varying display scales (@2x or @3x pixel densities).

QUESTION 33

  • Option 1: View modifiers must return the same view struct they were given.
  • Option 2: Context menus are triggered when users long press on a view.

Option 2 is true.

Here is the quick breakdown of why this is correct and why Option 1 is incorrect.

Why Option 2 is True

In iOS and iPadOS, context menus are designed to reveal secondary actions or previews without cluttering the main screen layout. They are triggered standardly when a user performs a long-press gesture (or a right-click/haptic force press depending on the platform and hardware) on a view that has the .contextMenu() modifier attached to it.

Why Option 1 is False

View modifiers do not have to return the exact same view struct they were given. In fact, most modifiers explicitly don't!

When you apply a modifier like .padding() or .background(.blue) to a view (e.g., a Text view), SwiftUI doesn't modify the properties inside that Text struct directly. Instead, it wraps your original view inside a brand new structural layout layer, such as ModifiedContent<Text, _PaddingLayout>.

This structural layering is the exact reason why the order of your view modifiers matters so much in SwiftUI.

QUESTION 34

  • Option 1: A GeometryReader always takes up all available space in its parent.
  • Option 2: Swift macros rewrite our code at build time.

Both options are true. This is a rare case where both statements are accurate. Here is the breakdown of how each works.

Why Option 1 is True

A GeometryReader behaves as a greedy layout container. When SwiftUI's layout engine asks a GeometryReader how much space it wants, it doesn't look at its inner children to decide. Instead, it instantly claims 100% of the available space offered by its parent view.

Once it expands to fill that maximum boundary, it passes those exact dimensions down to its coordinate proxy so you can measure and position your inner views relative to that frame.

Why Option 2 is True

Swift macros (introduced in Swift 5.9) are designed to eliminate boilerplate code by rewriting your source code at compile/build time.

When you use a macro (like @Observable or #Predicate), the Swift compiler passes your code to a separate macro plug-in before compilation finishes. The plug-in generates additional, valid Swift code out of sight and injects it right back into your program. You can even right-click a macro in Xcode and select "Expand Macro" to see the exact code it wrote for you behind the scenes.

QUESTION 35

  • Option 1: Before trying to stretch the contents of an image view, we should use aspectRatio(contentMode: .resize).
  • Option 2: We can align text in a HStack using the baseline of the first or last text views.

Option 2 is true.

Here is the quick breakdown of why this is correct and why Option 1 is incorrect.

Why Option 2 is True

When aligning elements horizontally in an HStack, standard alignments like .center or .top can look awkward if your text views use different font sizes or styles.

To fix this, SwiftUI provides specialized text alignment guides: .firstTextBaseline and .lastTextBaseline. Using these ensures that the actual invisible baseline that the text sits on lines up perfectly across the row, regardless of font variations or multi-line wraps. 

HStack(alignment: .firstTextBaseline) {

Once the image is made resizable, you can then control how it stretches using either .aspectRatio(contentMode: .fit) or .aspectRatio(contentMode: .fill).

QUESTION 36

  • Option 1: One environment object can be shared in up to two views.
  • Option 2: When a URLSession download completes, it will send back the downloaded data plus any additional metadata.

Option 2 is true.

Here is the quick breakdown of why this is correct and why Option 1 is incorrect.

Why Option 2 is True

When you use URLSession to download a file (for example, using URLSession.shared.download(for:)), the completion handler or async return tuple delivers two distinct pieces of information:

  • The downloaded data/file location: A temporary file URL pointing to where the downloaded data is safely staged on the device's local disk.
  • The metadata: A URLResponse object (which you can safely cast to HTTPURLResponse). This contains vital server metadata, such as the HTTP status code (e.g., 200 OK or 404 Not Found), content length, MIME type, and HTTP headers.

Why Option 1 is False

There is no limit to how many views can share a single environment object.

An @EnvironmentObject is injected at a high level in your view hierarchy (like the root navigation view or the entire app window) using the .environmentObject() modifier. Once injected, that single instance is made available to any and all child views, grandchild views, or deep subviews down the entire layout tree that explicitly declare a dependency on it. Whether that is two views or two hundred views, they will all safely read from and react to the same shared object.

QUESTION 37

  • Option 1: We can use a ForEach view inside a List.
  • Option 2: We only need to add custom Voice Control labels to our views when we want to support languages other than English.

Option 1 is true.

Here is the quick breakdown of why this is correct and why Option 2 is incorrect.

Why Option 1 is True

Using a ForEach inside a List is standard practice in SwiftUI and unlocks a lot of flexibility.

While you can pass an array directly into a List to create a simple scrollable layout, embedding a ForEach inside the List allows you to mix static and dynamic content. For example, you can have a static header row at the top, followed by a dynamic list of items generated by ForEach, followed by a static footer button. Furthermore, operations like .onDelete (swipe-to-delete) and .onMove (drag-to-reorder) must be attached directly to a ForEach view inside the list to function properly.

Why Option 2 is False

Voice Control labels are not about translation; they are about semantic clarity for users navigating their devices entirely via voice commands.

Even in English, a button might display a visual icon (like a heart icon) instead of text. A person using Voice Control needs to know exactly what phrase to speak out loud to activate that button. By providing an explicit accessibility label or a .accessibilityInputLabels(["Like", "Favorite"]), you tell the operating system exactly what spoken English words should trigger that specific interactive element.

QUESTION 38

  • Option 1: Calling data(from:) on a URLSession must be done with await.
  • Option 2: Parent views can force a size on one of their chidren.

Option 1 is true.

Here is the quick breakdown of why this is correct and why Option 2 is incorrect.

Why Option 1 is True

The data(from:) method on URLSession is an asynchronous networking operation that fetches data from the web. Because it takes time to connect to a server and download content, it is marked as async in the Foundation library.

In Swift, whenever you invoke an asynchronous function, you must use the await keyword to mark a suspension point, letting the system know your code will yield its thread while waiting for the download to finish. Furthermore, because network requests can fail, you generally pair it with try: 

// This operation is asynchronous and throwing, requiring both try and await
let (data, response) = try await URLSession.shared.data(from: url)

Why Option 2 is False

As we established in a previous layout discussion, parent views in SwiftUI cannot actually force a size on their children.

The SwiftUI layout system operates entirely on negotiation:

  • The parent offers a size.
  • The child calculates its own size based on its unique content and rules, and then tells the parent what size it is going to be.
  • The parent must accept that size and position the child accordingly.

Modifiers like .frame(width: 100, height: 100) might look like a parent forcing a size on a child, but under the hood, that modifier simply wraps the child view in an invisible wrapper. The wrapper obeys the 100x100 constraint, but the child inside it can still choose to ignore that size and spill out as an layout overflow if its content demands it.

QUESTION 39

  • Option 1: Unless we ask for a custom alignment, most parents always place their child views in the top-left corner of their available space.
  • Option 2: Text views automatically fit the size required to display all their lines.

Option 2 is true.

Here is the quick breakdown of why this is correct and why Option 1 is incorrect.

Why Option 2 is True

In SwiftUI, Text views are flexibly sized based entirely on their string content and the font size applied to them. By default, a Text view will automatically expand or shrink to perfectly fit the exact width and height required to render all of its text lines without clipping, wrapping onto multiple lines as needed if space allows.

Why Option 1 is False

Most layout containers in SwiftUI do not default to the top-left corner. Instead, they default to the center.

  • Containers like ZStack place all child views layered directly on top of each other, centered vertically and horizontally.
  • Layout frames (like .frame(width: 300, height: 300)) default to a center alignment (Alignment.center) for whatever child view is placed inside them.
  • Stacks like VStack and HStack center their children along their minor axes by default (a VStack centers its children horizontally; an HStack centers them vertically).
