QUESTION 10

Which is true?

  • Option 1: Apple recommends @State properties should use public access control.
  • Option 2: We can trigger haptic effects using the sensoryFeedback() modifier.

Option 2 is true.

Here is a quick breakdown of why this is correct and why Option 1 is incorrect.

Why Option 2 is True

The .sensoryFeedback() modifier is the standard, modern way to trigger haptics in SwiftUI. It allows you to watch a specific piece of state and play a physical vibration (like a gentle tap, success notification, or selection click) on the user's device whenever that state changes. 

Button("Purchase") {
    isPurchased.toggle()
}
// Triggers a success haptic tap whenever isPurchased becomes true
.sensoryFeedback(.success, trigger: isPurchased)

Why Option 1 is False

Apple actually recommends the exact opposite: @State properties should almost always be marked as private.

@State is specifically designed to manage a view's internal private storage. Because SwiftUI manages the memory lifecycle of that variable behind the scenes, allowing outside views to directly modify a view's raw @State can break things and cause unexpected UI bugs. If an outside view needs access, you should use @Binding or pass it down via an environment instead.

QUESTION 11

Option 1: Once a Timer has started, it can't be stopped. Option 2: We can use the dynamicTypeSize() modifier to declare which Dynamic Type sizes a view supports.

Option 2 is true.

Here is the quick breakdown of why this is correct and why Option 1 is incorrect.

Why Option 2 is True

The .dynamicTypeSize() modifier allows you to control exactly how much your text can scale when a user adjusts their system font size settings. You can pass a range to define a minimum and maximum cap, or specify a single fixed size to declare exactly what the view supports: 

// Limits the text scaling to a maximum of 'xxxLarge' 
// so it doesn't break your layout boundaries
Text("Resort Details")
    .dynamicTypeSize(...DynamicTypeSize.xxxLarge)

Why Option 1 is False

A Timer absolutely can be stopped. If you are using a standard Foundation Timer, you stop it by calling its .invalidate() method, which halts the timer entirely and removes it from the run loop.

If you are using a modern SwiftUI-style publishing timer (Timer.publish), you stop it by calling .connect().cancel(), which shuts down the data stream and stops the repeats instantly.

QUESTION 13

  • Option 1: SwiftData queries must have a sort order specified.
  • Option 2: The Codable protocol is actually a combination of Encodable and Decodable.

Option 2 is true.

Here is the quick breakdown of why this is correct and why Option 1 is incorrect.

Why Option 2 is True

The Codable protocol is not an independent protocol written from scratch. Instead, it is a typealias - a convenient nickname—that glues two distinct protocols together.

When you declare that a struct conforms to Codable, you are telling the Swift compiler to conform to both parts of the contract simultaneously:

  • Decodable: Allows data (like a JSON string) to be parsed into your native Swift object.
  • Encodable: Allows your native Swift object to be converted out into data (like a JSON string).

Under the hood, Apple defines it exactly like this: 

typealias Codable = Encodable & Decodable

Why Option 1 is False

SwiftData queries (@Query) do not require a sort order.

If you write a simple @Query var resorts: [Resort], SwiftData will compile perfectly and pull your records out of the database using its default internal sequencing (typically the order in which they were originally saved to disk). Providing a specific property sorting order (like sorting alphabetically by name) is entirely optional.

QUESTION 14

  • Option 1: If a VStack has a foreground style and some text inside also has a foreground style, the VStack's foreground style is used.
  • Option 2: Decorative images are images that are merely there to make the UI look nicer.

Option 2 is true.

Here is the quick breakdown of why this is correct and why Option 1 is incorrect.

Why Option 2 is True

Decorative images are assets that serve purely as visual styling or "eye candy" for the layout—like background gradients, accent shapes, or abstract graphics.

Because they provide zero actual information or context to the app's contents, you declare them in SwiftUI using Image(decorative: "name"). This tells the system's VoiceOver accessibility engine to completely ignore the asset, preventing the screen reader from reading out meaningless image names to visually impaired users.

    Text("I will be Red!") // Inner modifier overrides the parent!
        .foregroundStyle(.red) 
}

QUESTION 16

  • Option 1: Ease in animations start slow and end fast.
  • Option 2: SwiftUI's views cannot be created as @State properties.

Option 1 is true.

Here is the quick breakdown of why this is correct and why Option 2 is incorrect.

Why Option 1 is True

An "Ease In" animation curve starts at a zero or near-zero velocity and accelerates as it moves forward, meaning it starts slow and ends fast. It is modeled after physical objects that need time to build momentum from a standstill (like a car accelerating from a stoplight).

Why Option 2 is False

While it is an anti-pattern (bad practice), SwiftUI views can technically be created as @State properties, meaning the statement "cannot be created" is false.

You could write @State private var myView = Text("Hello"), and it will compile. However, you should never do this. Views in SwiftUI are lightweight blueprint structures designed to be recreated constantly by the rendering engine. Storing a view instance directly inside @State forces it to persist in memory, which bypasses SwiftUI's lifecycle engine and causes major UI rendering and update bugs.

QUESTION 19

  • Option 1: The Identifiable protocol has no requirements.
  • Option 2: Creating a property using @Environment(\.horizontalSizeClass) will keep the value updated when the size class changes.

Option 2 is true.

Here is the quick breakdown of why this is correct and why Option 1 is incorrect.

Why Option 2 is True

The @Environment property wrapper creates an active, live dependency on the system's environment values.

When you read \.horizontalSizeClass, SwiftUI subscribes your view to the device's current layout state. If a user rotates their iPhone from portrait to landscape, or enters split-screen multitasking on an iPad, the size class dynamically flips between .compact and .regular. SwiftUI intercepts this event, updates the property value automatically, and forces the view to redraw to adapt to the new layout.

The Identifiable protocol has one very strict, non-negotiable requirement: any type conforming to it must provide a unique id property.

The type of id can vary (it is commonly a UUID or a String), but it must conform to the Hashable protocol so that SwiftUI can uniquely track, update, and animate individual elements inside collections like a List or a ForEach. You can see this directly in your own SortType enum from earlier, where you had to explicitly declare var id: String { self.rawValue } to satisfy the compiler!

QUESTION 21

  • Option 1: QR codes are just barcodes with more colors.
  • Option 2: Using scaledToFill() with an image might mean parts of an image lie outside its container's frame.

Option 2 is true.

Here is the quick breakdown of why this is correct and why Option 1 is incorrect.

Why Option 2 is True

When you apply the .scaledToFill() modifier to an image, SwiftUI forces the image to expand until it completely fills the dimensions of its frame, regardless of the aspect ratio.

If the image's original shape doesn't match the shape of the container frame, the image will spill over the borders. The parts that lie outside the frame boundaries remain visible and overlap surrounding UI elements unless you explicitly cut them off by adding a modifier like .clipped() or .clipShape().

Why Option 1 is False

QR codes are not defined by color; they are defined by their physical structure and layout. Traditional barcodes are one-dimensional (1D) linear codes that store data horizontally using lines of varying widths.

A QR code is a two-dimensional (2D) matrix code that stores data both horizontally and vertically using geometric modules (typically black and white squares). This grid-based structure allows a QR code to hold hundreds of times more data than a standard linear barcode, entirely independent of color.

QUESTION 27

  • Option 1: A view is always and exactly the same size as its body.
  • Option 2: Child views must always use less than or equal to the amount of space the parent offers them.

Both options are technically false, but Option 1 is significantly closer to a fundamental truth of SwiftUI - it is considered a core layout rule, though there is a technical exception.

Let's break down the rules of SwiftUI's layout engine to see how both work.

The Nuance of Option 1 (The Core Rule)

One of the three absolute rules of the SwiftUI layout engine is: A view chooses its own size, and the parent must respect it. Because a container view wraps its content tightly, a view's final dimensions are almost always exactly the size of its underlying body.

The tiny catch: Modifiers like .frame(width: 100, height: 100) actually wrap your view inside a new, invisible layout layer. In those cases, the parent sees the size of the modifier, which might differ from the raw, unfiltered size of the inner body content.

Why Option 2 is Flat Out False

Child views do not have to stay within the bounds offered by their parents.

In SwiftUI, layout is a negotiation:

  • The parent offers a size (e.g., "You have 100x100 points of space").
  • The child decides its own size based on its content ("Thanks, but I am a large image and I need 300x300 points").
  • The parent must accept that size.

If a child view demands more space than the parent offers, it will simply break out of the parent's boundaries and layout as an overflow, spilling onto the surrounding UI unless the parent explicitly uses a modifier like .clipped() to chop off the excess.

QUESTION 28

  • Option 1: Changing any @State property of a view causes SwiftUI to reinvoke the body property.
  • Option 2: Arrays cannot use the @State property wrapper.

Option 1 is true.

Here is the quick breakdown of why this is correct and why Option 2 is incorrect.

Why Option 1 is True

This is the core foundation of how SwiftUI works: Data is the single source of truth for your user interface. When you mark a property with @State, SwiftUI establishes a watch on that memory address. The moment that variable changes (even by a tiny bit, like toggling a boolean or incrementing an integer), SwiftUI detects the mutation, invalidates the current layout, and instantly re-runs (re-invokes) the body property of that view to compute the brand-new UI layout.

Why Option 2 is False

As we touched on earlier, arrays can absolutely use the @State property wrapper.

@State works flawlessly with any standard Swift value type, including arrays. When you append a new item, remove an element, or sort an array that is wrapped in @State, Swift handles it as a structural mutation. SwiftUI intercepts this mutation and immediately triggers a UI refresh to display the updated list.

QUESTION 29

  • Option 1: contentShape() allows us to control the tap area for a view.
  • Option 2: We can show an alert by calling its show() method.

Option 1 is true.

Here is the quick breakdown of why this is correct and why Option 2 is incorrect.

Why Option 1 is True

By default, interactive elements like hit testing and tap gestures only trigger when a user taps on pixels that actually contain content (like text or an icon). If you have a container with empty space, tapping that empty space does nothing.

The .contentShape() modifier allows you to define a custom hit-test geometry for a view. For example, applying .contentShape(Rectangle()) to a row container makes the entire boundary tappable, including all the transparent, empty spaces between your text and images.

Why Option 2 is False

SwiftUI does not use imperative code commands like a .show() method to present UI elements.

Instead, SwiftUI is completely declarative and state-driven. To show an alert, you attach the .alert() modifier to a view and bind it to a boolean or optional @State property. You trigger the alert simply by changing that state variable to true or giving it a value, and SwiftUI handles the presentation automatically. 

@State private var showAlert = false

var body: some View {
    Button("Trigger Alert") {
        showAlert = true // This state change triggers the alert!
    }
    .alert("Hello!", isPresented: $showAlert) {
        Button("OK", role: .cancel) { }
    }
}

Option 1 is true.

Here is the quick breakdown of why this is correct and why Option 2 is incorrect.

Why Option 1 is True

To build a custom, reusable view modifier in SwiftUI, your structure must conform to the ViewModifier protocol.

Conforming to this protocol requires you to implement a single method: body(content: Content) -> some View. Inside this method, you take the existing content (the view the modifier is being attached to) and wrap or decorate it with whatever paddings, backgrounds, shapes, or animations you need.

QUESTION 31

QUESTION 32

  • Option 1: SwiftUI stores view positions and sizes as integers.
  • Option 2: The containerRelativeFrame() modifier lets us size a child view relative to its nearest container.

Option 2 is true.

Here is the quick breakdown of why this is correct and why Option 1 is incorrect.

Why Option 2 is True

The .containerRelativeFrame() modifier gives a view precise information about its parent container's layout size. It allows you to dynamically scale a child view relative to its nearest container (like a ScrollView, a List, or the window itself) using an axis and a sizing closure.

For example, you can use it to explicitly state: "I want this card to always take up exactly three-quarters (3/4) of the width of the scroll view it lives inside." 

Image("hero-banner")
    .resizable()

QUESTION 34

  • Option 1: A GeometryReader always takes up all available space in its parent.
  • Option 2: Swift macros rewrite our code at build time.

Both options are true. This is a rare case where both statements are accurate. Here is the breakdown of how each works.

Why Option 1 is True

A GeometryReader behaves as a greedy layout container. When SwiftUI's layout engine asks a GeometryReader how much space it wants, it doesn't look at its inner children to decide. Instead, it instantly claims 100% of the available space offered by its parent view.

Once it expands to fill that maximum boundary, it passes those exact dimensions down to its coordinate proxy so you can measure and position your inner views relative to that frame.

Why Option 2 is True

Swift macros (introduced in Swift 5.9) are designed to eliminate boilerplate code by rewriting your source code at compile/build time.

When you use a macro (like @Observable or #Predicate), the Swift compiler passes your code to a separate macro plug-in before compilation finishes. The plug-in generates additional, valid Swift code out of sight and injects it right back into your program. You can even right-click a macro in Xcode and select "Expand Macro" to see the exact code it wrote for you behind the scenes.
