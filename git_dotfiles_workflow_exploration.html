Git - Dotfiles Workflow Exploration

QUESTION

Ok, some git questions.

  • I have this dotfile setup on a machine specific branch.
  • I have a branch for each of my known machines.
  • Let's say I have these 3 branches (after main):
    • main
    • macbook-m3
    • macbook-intel
    • macmini

At the moment, branch 'macbook-m3' has all these development changes.

I'd like to transfer these changes to 'main', and then pull these changes in turn to the other two 'machine' branches.

What is a good strategy?

ANSWER FROM GEMINI

To transfer your development changes from macbook-m3 back to main and then out to your other machines, the most effective strategy is a "Feature Graduation" workflow. This treats your specific machine branch as a staging ground where you've perfected the logic before "graduating" it to the shared main branch.

Step 1: Graduate Changes to main

First, move the shared logic (the zshrc updates, the sed trick, and the new Starship custom blocks) from your macbook-m3 branch into main.

  • Switch to main: git checkout main
  • Merge the machine branch: git merge macbook-m3
    • Note: If you only want specific commits (not the whole branch), use git cherry-pick instead.
  • Resolve Conflicts: If main had older logic, Git will ask you to choose. Keep the new universal logic.
  • Push: git push origin main

Step 2: Propagate to Other Machines

Now that main is the "Source of Truth," you can pull those improvements into your other machine-specific branches.

  • For macbook-intel:
    • git checkout macbook-intel
    • git merge main (or git rebase main if you want a cleaner, linear history).
  • For macmini:
    • git checkout macmini
    • git merge main
  • Push all:
    • git push --all

The "Clean History" Pro-Tip

If you want to keep your machine branches very tidy, use git rebase main instead of merge in Step 2.

  • Merge: Creates a "Merge commit" (e.g., Merged branch main into macmini).
  • Rebase: Moves your machine-specific palette changes to the very top, as if you just wrote them today on top of the brand-new main logic.

Strategy Checklist

  • Shared Files: Keep universal logic (like the KNOWN_HOSTS array) in the top/middle of files so they merge easily.
  • Machine Files: Keep machine-specific blocks (like [palettes.macbook-intel]) at the bottom of the TOML file. Git is less likely to see these as "conflicts" when merging from main.
  • Testing: After merging into macmini, run starship explain to ensure the new custom modules are triggering correctly on that hardware.

QUERY

Follow-up: Would you like to see how to use a .gitattributes file to tell Git "never merge this specific line" so your palettes stay private to their branches?

ANSWER

To keep your machine-specific palettes from ever "bleeding" into the main branch during a merge, you can use a Git Merge Strategy called "ours". This tells Git: "Whenever there is a conflict in this specific file, always keep the version that is already on the current branch."

Git Attributes - Merge Strategies.

Step 1: Define the custom merge driver

Git doesn't have a "keep mine" strategy enabled by default for specific files. You have to "register" it in your global git config once on each machine: 

git config --global merge.ours.driver true

This creates a dummy driver that always exits with success without changing the file, effectively telling Git "don't touch my version." How to make Git ignore a file during merge.

2. Create the .gitattributes file

In the root of your dotfiles repository (on the main branch), create a file named .gitattributes and add your Starship and Zsh configs to it: 

# ~/.config/starship/starship.toml merge=ours
# ~/.zshrc merge=ours

3. How this works during your workflow

  • Graduate to main: When you git checkout main and git merge macbook-m3, Git will see the .gitattributes. If there are changes in the [palettes] section, it will ignore the M3's specific colors and keep whatever is on main.
  • Propagate to others: When you git checkout macmini and git merge main, the Mac Mini will keep its own palette and only pull in the new custom modules or logic you added to the rest of the file.

The "Selective" Alternative (Better for your setup)

If you want to merge most of the file but keep the palette lines unique, .gitattributes is a bit "all or nothing." A better strategy for your specific setup is to use Environment Variables for the colors:

  • In .zshrc: Define export MACHINE_COLOR="#306998" inside your hostname case statement. Zsh Environment Variables.
  • In starship.toml: Use style = "bold $MACHINE_COLOR". Starship Style Strings.
  • Result: The starship.toml is now 100% identical on every branch/machine. Only the .zshrc (which already has your hostname logic) handles the color.

Which approach do you prefer?

  • The .gitattributes way: Harder to set up, but lets you keep different text in the same files.
  • The Variable way: Much cleaner; allows you to have one single branch for all machines because the logic handles the differences dynamically. Starship Configuration.

Which one should YOU use?

Since you are the only one working on these dotfiles:

  • Use git merge macbook-m3 when moving changes UP to main. This preserves the fact that the work was done on that specific machine.
  • Use git rebase main when moving changes DOWN to your machine branches. This keeps your machine-specific palettes at the "tip" of the branch, making them easy to edit and keep separate from the core logic. StackOverflow: When to use merge vs rebase

Summary for your workflow:

  • git checkout main -> git merge macbook-m3 (Graduation)
  • git checkout macmini -> git rebase main (Syncing)

More on git push --force

In Git, a push --force is not just "advisable" after a rebase—it is technically required if you have already pushed that branch to a remote server.

Why you must force push

  • Commit Rewriting: A rebase doesn't just move your commits; it re-creates them with new unique IDs (hashes). Even if the code is identical, Git sees the "old" commits on the server and the "new" ones on your machine as completely different history.
  • Fast-Forward Protection: By default, Git only allows "fast-forward" pushes, where your local work is a direct extension of the remote work. Since rebasing creates a divergent history, the remote server will reject a standard push to prevent you from accidentally overwriting data.
  • Syncing History: The --force flag tells the server, "I know what I'm doing; replace the old history with this new one".

Why it is "safe" in your specific setup

The "Golden Rule of Rebasing" is: Never rebase (and force push) branches shared with others. However, for your dotfiles:

  • Private Machine Branches: Since your macbook-m3 and macmini branches are only used by you, there are no other developers whose work you would break by rewriting history.
  • A "Safer" Force: If you want to be extra careful, use git push --force-with-lease. This command will fail if the remote branch has new commits that you haven't seen yet, ensuring you don't accidentally overwrite changes you made from a different machine.

A Cleaner Workflow

If you find the constant force-pushing annoying, you can use git pull --rebase when syncing between machines. This pulls the latest changes and automatically reapplies your local tweaks on top, often avoiding the need for a manual rebase/force-push cycle.

In your specific dotfile cycle, you should run git push --force immediately after completing a local rebase on your machine-specific branch.

Because a rebase rewrites your commit history, your local machine branch will no longer match the version already stored on your remote server

Updated Workflow (actually shown above)

Your workflow effectively uses main as the "source of truth" and treats your machine branches as "layers" sitting on top of that truth.

A few "Quality of Life" tweaks:

Automate

Automating this workflow is a great way to reduce the repetitive "plumbing" of switching and rebasing. Below is a simple Bash script designed to handle your specific cycle safely.

Dotfile Sync Script

Save this as sync-dotfiles.sh in your repository and give it execution permissions (chmod +x sync-dotfiles.sh). 

#!/bin/bash

# 1. Automatic Directory Discovery & Validation
DOTFILES_PATH="$HOME/dotfiles"

# If we aren't in a git repo named 'dotfiles', try to go there
REPO_NAME=$(basename "$(git rev-parse --show-toplevel 2>/dev/null)" 2>/dev/null)

if [[ "$REPO_NAME" != "dotfiles" ]]; then
    if [[ -d "$DOTFILES_PATH" ]]; then
        echo "Not in dotfiles repo. Jumping to $DOTFILES_PATH..."
        cd "$DOTFILES_PATH" || exit 1
    else
        echo "ERROR: Not in 'dotfiles' repo and could not find it at $DOTFILES_PATH."
        exit 1
    fi
fi

# 2. Safety Check: Uncommitted changes
if ! git diff-index --quiet HEAD --; then
    echo "ERROR: You have uncommitted changes. Commit or stash them first."
    exit 1
fi

# ... [Remainder of your rebase/merge logic] ...

# Configuration
MAIN_BRANCH="main"
MACHINE_BRANCHES=("macbook-m3" "macmini" "macbook-intel")
PUSH_CMD="git push --force-with-lease"

# Save starting point
START_BRANCH=$(git branch --show-current)

echo "--- Starting dotfile sync ---"

# 1. Safety Check: Ensure no uncommitted changes
if ! git diff-index --quiet HEAD --; then
    echo "ERROR: You have uncommitted changes on $START_BRANCH."
    echo "Please commit or stash them before running the sync."
    exit 1
fi

# 2. Update main and merge the current machine's work
echo "Switching to $MAIN_BRANCH and pulling..."
git checkout $MAIN_BRANCH
git pull origin $MAIN_BRANCH

if [ "$START_BRANCH" != "$MAIN_BRANCH" ]; then
    echo "Propagating changes from $START_BRANCH to $MAIN_BRANCH..."
    if git merge "$START_BRANCH"; then
        git push origin $MAIN_BRANCH
    else
        echo "ERROR: Merge into $MAIN_BRANCH failed. Resolve conflicts and try again."
        exit 1
    fi
fi

# 3. Process each machine branch
for branch in "${MACHINE_BRANCHES[@]}"; do
    echo "Rebasing $branch onto latest $MAIN_BRANCH..."
    
    if git show-ref --verify --quiet "refs/heads/$branch"; then
        git checkout "$branch"
        
        if git rebase "$MAIN_BRANCH"; then
            $PUSH_CMD origin "$branch"
            echo "SUCCESS: $branch is rebased and pushed."
        else
            echo "ERROR: Rebase failed on $branch. Resolve manually."
            exit 1
        fi
    else
        echo "SKIP: $branch not found locally."
    fi
done

# 4. Return to where you started
git checkout "$START_BRANCH"
echo "--- Sync complete! Back on $START_BRANCH ---"

Why this script helps:

  • The "Dirty Check": Added git diff-index --quiet HEAD --. If you have modified or staged files that aren't committed, the script will stop immediately to prevent messy rebases or accidental loss of work.
  • Propagate to Main: It now successfully merges your current machine's work into main before rebasing the other branches, ensuring everyone gets the newest updates.
  • Atomic Failure: If a merge or rebase fails, the script exits with an error code rather than continuing and potentially corrupting other branches.
  • Safety First: It uses --force-with-lease, which prevents you from accidentally overwriting remote work you haven't fetched yet.
  • Branch Awareness: It checks if a branch exists locally before trying to switch, preventing errors if a specific machine doesn't have all branches checked out.
  • Conflict Handling: If a rebase hits a conflict, the script stops immediately so you can fix it manually before continuing.

Git Alias - UPDATED VERSION

This updated version is 'better' as it checks the location (the dotfiles repo): 

[alias]
    sync-dots = "!f() { \
        DIR=\"$HOME/dotfiles\"; \
        MAIN='main'; \
        MACHINES=('macbook-m3' 'macmini' 'macbook-intel'); \
        if [ ! -d \"$DIR\" ]; then echo \"Folder $DIR not found\"; return 1; fi; \
        REPO=$(basename \"$(git -C \"$DIR\" rev-parse --show-toplevel)\"); \
        if [ \"$REPO\" != 'dotfiles' ]; then echo 'Not a dotfiles repo'; return 1; fi; \
        START=$(git -C \"$DIR\" branch --show-current); \
        if ! git -C \"$DIR\" diff-index --quiet HEAD --; then echo 'Uncommitted changes found'; return 1; fi; \
        echo \"Syncing from $DIR...\"; \
        git -C \"$DIR\" checkout $MAIN && git -C \"$DIR\" pull origin $MAIN; \
        if [ \"$START\" != \"$MAIN\" ]; then \
            git -C \"$DIR\" merge \"$START\" && git -C \"$DIR\" push origin $MAIN || return 1; \
        fi; \
        for br in \"${MACHINES[@]}\"; do \
            git -C \"$DIR\" checkout $br && git -C \"$DIR\" rebase $MAIN && git -C \"$DIR\" push --force-with-lease || return 1; \
        done; \
        git -C \"$DIR\" checkout $START; \
    }; f"

Why this is better:

  • Location Independent: You can run git sync-dots from your ~/Downloads folder, and it will still perform the work inside ~/dotfiles.
  • Zero Accidents: If you rename your dotfiles folder or delete it, the script identifies the mismatch and stops before running dangerous commands like git push --force.
  • Clean Exit: If any single rebase or merge fails, the script uses return 1 to halt immediately, leaving you exactly where the error occurred so you can fix it.

Pro-Tip: Add a Git Alias

If you don't want to run a script file every time, you can add this logic as a Git alias in your ~/.gitconfig or simply run the script via a shorthand like alias dsync='./sync-dotfiles.sh'.

Git Alias Addition

To add a multi-line script as a Git alias, you wrap the entire script in a shell function inside your .gitconfig. By prefixing the command with an exclamation mark (!), Git passes the command directly to your system's shell instead of trying to run it as a standard Git subcommand.

Method 1: The "Function Wrap" (Direct in .gitconfig)

Open your ~/.gitconfig file (e.g., git config --global --edit) and add the following under the [alias] section: 

[alias]
    sync-dots = "!f() { \
        MAIN='main'; \
        MACHINES=('macbook-m3' 'macmini' 'macbook-intel'); \
        START=$(git branch --show-current); \
        if ! git diff-index --quiet HEAD --; then \
            echo 'ERROR: Uncommitted changes found. Commit or stash first.'; return 1; \
        fi; \
        echo \"Propagating $START to $MAIN...\"; \
        git checkout $MAIN && git pull origin $MAIN; \
        if [ \"$START\" != \"$MAIN\" ]; then \
            git merge \"$START\" && git push origin $MAIN || { echo 'Merge failed'; return 1; }; \
        fi; \
        for br in \"${MACHINES[@]}\"; do \
            echo \"Updating $br...\"; \
            git checkout $br && git rebase $MAIN && git push --force-with-lease || { echo \"Rebase failed on $br\"; return 1; }; \
        done; \
        git checkout $START; \
        echo '--- Sync Complete ---'; \
    }; f"
  • The Dirty Check: git diff-index --quiet HEAD -- checks for any local modifications before starting.
  • The Propagation: It now correctly merges the START branch into MAIN and pushes it to the server.
  • Error Handling: Added || { ... return 1; } logic. If any step (merge, rebase, or push) fails, the alias stops immediately so you don't end up in a broken state across multiple branches.
  • Refined Scope: It uses return 1 instead of exit 1 because aliases run within the shell context of the git command; this ensures it exits the function cleanly.
  • The !: Tells Git to run this in the shell.
  • The f() { ... }; f: Defines a temporary shell function and immediately executes it.
  • Backslashes (\): Required at the end of each line in the config file to tell Git the command continues on the next line.

Method 2: The "External Script" (Cleaner & Easier)

If the script is too long, the standard practice is to save it as a standalone file and point the alias to it.

  • Save the script: Create a file named git-sync-dots (no extension needed) in a folder that is in your $PATH (like ~/bin/ or /usr/local/bin/).
  • Make it executable: Run chmod +x ~/bin/git-sync-dots.
  • Add the simple alias: 
git config --global alias.sync-dots "!git-sync-dots"
