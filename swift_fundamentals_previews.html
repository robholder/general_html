Swift - Previews (Orientation)

Regular Simple Preview

The following is the standard simple preview syntax, where ContentView reflects the name of the View Struct: 

#Preview {
    ContentView()
}

Configuring Xcode Previews with Macro Traits

When building user interfaces in SwiftUI, testing layouts across different device configurations—such as landscape and portrait orientations—is essential for ensuring a responsive design.

While Xcode provides visual dropdowns to override the canvas orientation, these manual toggles are temporary and often disconnect when switching the preview to Live Mode. To create permanent, interactive, and side-by-side orientation previews, you must configure the #Preview macro using Traits.

1. What is a Preview Trait?

A Trait is a configuration argument passed directly into the #Preview macro. Instead of modifying the view after it is built, a trait instructs Xcode on how to construct the surrounding simulator sandbox environment from the very first frame. 

#Preview(traits: .landscapeLeft) {
    MyView()
}

By defining the environment at the macro level, Xcode respects the configuration natively. This prevents the canvas from snapping back to default vertical orientations when you activate interactive live previews.

2. The Old Way vs. The Modern Macro Way

Prior to the introduction of the unified #Preview macro, developers adjusted canvas layouts using standard view modifiers. Attempting to use those old patterns inside a macro now triggers compiler errors.

The Incorrect Pattern (View Modifier)

Attaching configuration directly to the view instance inside the macro results in a deprecation or ignore warning: 

// ❌ COMPILER ERROR / WARNING
#Preview {
    CardView(card: .example)
        .previewInterfaceOrientation(.landscapeLeft) 
}
Error: PreviewInterfaceOrientation is ignored in a #Preview macro. Use the traits argument of the macro to provide the desired orientation...

The Correct Pattern (Macro Argument)

The orientation instruction must be moved inside the parentheses of the #Preview declaration itself: 

//  VALID MULTI-MODE CONFIGURATION
#Preview(traits: .landscapeLeft) {
    CardView(card: .example)
}

3. Implementation: Multi-Orientation Reference Blueprint

To test both portrait and landscape behaviors simultaneously without manual switching, you can declare multiple independent #Preview blocks at the bottom of a single Swift file.

Adding an optional String Name as the first argument provides clear labeling in the Xcode Canvas tabs. 

import SwiftUI

// MARK: - Previews

#Preview("Standard Portrait") {
    CardView(card: .example)
}

#Preview("Landscape Left Alignment", traits: .landscapeLeft) {
    CardView(card: .example)
}

When this file is active, Xcode parses both macros and renders two distinct canvas elements side-by-side or stacked in your preview panel.

4. Key Orientation Traits Reference

The PreviewTrait structure offers several presets for testing layout flows:

Trait Argument Behavior Best Used For
Default (No argument) Uses the target device's default portrait setup. Standard vertical layout testing.
traits: .landscapeLeft Rotates the canvas 90° counter-clockwise. Testing horizontal layouts, split-views, and wide aspect ratios.
traits: .landscapeRight Rotates the canvas 90° clockwise. Verifying symmetrical UI behaviors when the device is inverted.

5. Troubleshooting the Interactive Canvas

If your preview orientation is locked but the application elements do not respond to clicks, taps, or state updates:

  • Verify Live Mode: Look at the toolbar at the bottom of the canvas panel. Ensure the Live Preview icon (the Play triangle) is active. You can toggle this state using the keyboard shortcut ⌥⌘P (<Option> + Command + p).
  • Hit-Testing Bounds: If tap gestures behave erratically in wide landscape layouts, append .contentShape(Rectangle()) directly above your view's gesture modifiers to ensure the entire expanded coordinate space accepts user interaction.
