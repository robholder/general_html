Challenge 2

  • Fill in the loading and saving methods for Favorites.

Ai Insight

To bridge the Favorites class with UserDefaults, we have to navigate one minor limitation: UserDefaults does not natively store Swift Set types. However, it is incredibly happy storing Array types.

The most elegant way to solve this is to quickly convert your Set<String> into an Array<String> right when saving, and convert it back into a Set when loading. This gives you the speed benefits of a Set during runtime (O(1) lookups) with the convenience of UserDefaults storage.

Here is your completed class with the init() and save() logic fully implemented: 

import SwiftUI

@Observable
class Favorites {
    // the actual resorts the user has favorited
    private var resorts: Set<String>
    
    // the key we're using to read/write in UserDefaults
    private let key = "Favorites"
    
    init() {
        // 1. Attempt to load the saved data as an array of Strings
        if let savedItems = UserDefaults.standard.stringArray(forKey: key) {
            // 2. If successful, convert the array back into a Set for fast lookups
            resorts = Set(savedItems)
        } else {
            // 3. Fallback to an empty set if no previous data exists
            resorts = []
        }
    }
    
    // returns true if our set contains this resort
    func contains(_ resort: Resort) -> Bool {
        resorts.contains(resort.id)
    }
    
    // adds the resort to our set and saves the change
    func add(_ resort: Resort) {
        resorts.insert(resort.id)
        save()
    }
    
    // removes the resort from our set and saves the change
    func remove(_ resort: Resort) {
        resorts.remove(resort.id)
        save()
    }
    
    func save() {
        // 4. Convert the Set to an Array so UserDefaults can serialize it
        UserDefaults.standard.set(Array(resorts), forKey: key)
    }
}

Why this approach works perfectly:

  • Performance: Keeping the runtime collection as a Set is ideal because checking if an item is favorited (contains) runs in O(1) constant time, whereas an Array lookup grows slower as the list grows (O(n) linear time).
  • stringArray(forKey:): This built-in UserDefaults helper safely extracts the data and guarantees that you receive an array containing only strings, eliminating the need for manual type-casting.

Challenge 3

  • For a real challenge, let the user sort the resorts in ContentView either using the default order, alphabetical order, or country order.

Ai Insight

To solve this challenge elegantly, we can expand on the exact architectural concepts you've already mastered: using an Enum for type safety and utilizing a Computed Property to pipeline your data dynamically.

Instead of writing separate filtering and sorting blocks, we will enhance filteredResorts to become a continuous data pipeline: Raw Array → Filtered Array → Sorted Array.

The 3-Step Strategy

  • The Sort Type Enum: Create an enum containing our three options (defaultOrder, alphabetical, country). Conforming it to CaseIterable and Identifiable makes it exceptionally easy to loop through in a picker or menu.
  • The State Tracking: Add a @State variable to store the user's active choice.
  • The Pipeline Upgrade: Modify the computed property. First, it will filter based on searchText. Then, it will pass that filtered result through a .sorted { ... } block matching the chosen enum state before handing it over to the List.

Fully Documented Implementation

Here is your updated ContentView.swift containing the sorting menu integrated directly into the navigation toolbar. 

import SwiftUI

// 1. Create a type-safe representation of our sorting options.
// CaseIterable allows us to effortlessly list all cases in a loop.
// Identifiable allows SwiftUI to differentiate options without manual IDs.
enum SortType: String, CaseIterable, Identifiable {
    case defaultOrder = "Default"
    case alphabetical = "Alphabetical"
    case country = "Country"
    
    // Satisfy Identifiable using the raw value string
    var id: String { self.rawValue }
}

struct ContentView: View {
    let resorts: [Resort] = Bundle.main.decode("resorts.json")
    
    @State private var searchText = ""
    @State private var favorites = Favorites()
    
    // 2. Track the active sorting method selected by the user
    @State private var sortOrder: SortType = .defaultOrder
    
    // 3. This computed property acts as our master data pipeline:
    // It filters the array first, then sorts it on the fly.
    var filteredResorts: [Resort] {
        // Phase A: Filtering (Your existing logic)
        let intermediateList: [Resort]
        if searchText.isEmpty {
            intermediateList = resorts
        } else {
            intermediateList = resorts.filter { $0.name.localizedStandardContains(searchText) }
        }
        
        // Phase B: Sorting based on the active state
        switch sortOrder {
        case .defaultOrder:
            // Return the list exactly as it came from the JSON bundle
            return intermediateList
            
        case .alphabetical:
            // Sort ascending by the resort name string
            return intermediateList.sorted { $0.name < $1.name }
            
        case .country:
            // Sort ascending by country name. 
            // If countries match, it falls back to sorting by name so it looks neat.
            return intermediateList.sorted {
                if $0.country == $1.country {
                    return $0.name < $1.name
                }
                return $0.country < $1.country
            }
        }
    }
    
    var body: some View {
        NavigationSplitView {
            // The List remains completely untouched because it automatically 
            // streams the finished array out of the updated computed property!
            List(filteredResorts) { resort in
                NavigationLink(value: resort) {
                    HStack {
                        Image(resort.country)
                            .resizable()
                            .scaledToFill()
                            .frame(width: 40, height: 25)
                            .clipShape(.rect(cornerRadius: 5))
                            .overlay(
                                RoundedRectangle(cornerRadius: 5)
                                    .stroke(.black, lineWidth: 1)
                            )
                        
                        VStack(alignment: .leading) {
                            Text(resort.name)
                                .font(.headline)
                            Text("\(resort.runs) runs")
                                .foregroundStyle(.secondary)
                        }
                    }
                    if favorites.contains(resort) {
                        Spacer()
                        Image(systemName: "heart.fill")
                            .accessibilityLabel("This is a favorite resort")
                            .foregroundStyle(.red)
                    }
                }
            }
            .navigationTitle("Resorts")
            .navigationDestination(for: Resort.self) { resort in
                ResortView(resort: resort)
            }
            .searchable(text: $searchText, prompt: "Search for a resort")
            // 4. Add a toolbar button containing a contextual Menu to swap the sort state
            .toolbar {
                ToolbarItem(placement: .topBarTrailing) {
                    Menu {
                        // Picker wraps our choices into an interactive checkmarked list
                        Picker("Sort Order", selection: $sortOrder) {
                            ForEach(SortType.allCases) { type in
                                Text(type.rawValue).tag(type)
                            }
                        }
                    } label: {
                        Label("Sort resorts", systemName: "arrow.up.arrow.down")
                    }
                }
            }
        } detail: {
            WelcomeView()
        }
        .environment(favorites)
    }
}

Architectural Breakdown: How it Works

1. The Two-Tiered Country Sort

Look closely at the .country sorting case: 

if $0.country == $1.country { return $0.name < $1.name }

If you only sort by \(0.country < \)1.country, any resorts belonging to the same country will appear in a random, unpredictable order. Adding the secondary condition checks if the countries match, and if they do, alphabetizes them by name. It adds a polish that native Apple apps use.

2. The .toolbar and Menu Layout

Instead of plastering buttons directly onto the screen, placing a Menu inside a .topBarTrailing toolbar matches standard iOS design guidelines. The Picker automatically embeds checkmarks next to whichever SortType is matching the active ＄sortOrder state.

3. UI Redraw Economy

Because filteredResorts is a computed property, the sorting and filtering calculations are incredibly low-overhead. The second the user taps an option in the menu, ＄sortOrder mutates, forcing SwiftUI to evaluate filteredResorts again and gracefully animate the list items into their new positions.

CaseIterable

The CaseIterable protocol is a built-in Swift feature that does one specific job: it automatically counts and organizes all the cases of your enum into an array sequence.

Normally, enums are distinct, individual choices. You can’t naturally loop through them or count them. By adding : CaseIterable, Swift automatically generates a special static property on your enum called allCases.

The Core Transformation

Imagine your enum is a collection of separate puzzle pieces. Adding CaseIterable collects those pieces and places them neatly into a standard Swift array. 

// WITHOUT CaseIterable:
enum SortType {
    case defaultOrder, alphabetical, country
}
// You cannot loop. You cannot count. They are just isolated symbols.

// WITH CaseIterable:
enum SortType: CaseIterable {
    case defaultOrder, alphabetical, country
}
// Swift secretly writes this code for you under the hood:
// static var allCases: [SortType] = [.defaultOrder, .alphabetical, .country]

Why is this useful in SwiftUI?

In the ContentView code, we used this line to build our dropdown menu: 

ForEach(SortType.allCases) { type in
    Text(type.rawValue).tag(type)
}

Because of CaseIterable, SortType.allCases behaves exactly like a standard array (e.g., ["Default", "Alphabetical", "Country"]).

The Major Benefits:

  • Automation: If you add a fourth sorting option next week (like case favoriteFirst), you only have to type it once inside the enum definition. You don't have to update your UI layout code because SortType.allCases will automatically include it on the next app build.
  • Array Power: Because allCases is an array, you get access to all standard array features instantly:
  • SortType.allCases.count (Returns 3)
  • SortType.allCases.first (Returns .defaultOrder)
  • SortType.allCases.isEmpty (Returns false)

In short, CaseIterable saves you from manually typing out arrays like let options = [.defaultOrder, .alphabetical, .country] and keeping them manually synchronized whenever your data requirements shift.
