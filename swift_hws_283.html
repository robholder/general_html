Hacking With Swift - 283 - Using groups as transparent layout containers

SwiftUI’s Group view might seem odd at first, because it doesn’t actually affect our layout at all. However, it performs an important purpose as a transparent layout container: it gives us the ability to add SwiftUI modifiers to multiple views without changing their layout, or send back multiple views without using @ViewBuilder.

Flip Stack Example

For example, this UserView has a Group containing three text views: 

struct UserView: View {
    var body: some View {
        Group {
            Text("Name: Paul")
            Text("Country: England")
            Text("Pets: Luna and Arya")
        }
        .font(.title)
    }
}

That group contains no layout information other than applying a font all three text views, so we don’t know whether the three text views will be stacked vertically, horizontally, or by depth. This is where the transparent layout behavior of Group becomes important: whatever parent places a UserView gets to decide how its text views get arranged.

For example, we could create a ContentView like this: 

struct ContentView: View {
    @State private var layoutVertically = false

    var body: some View {
        Button {
            layoutVertically.toggle()
        } label: { 
            if layoutVertically {
                VStack {
                    UserView()
                }
            } else {
                HStack {
                    UserView()
                }
            }
        }
    }
}

That flips between vertical and horizontal layout every time the button is tapped.

Size Classes

You might wonder how often you need to have alternative layouts like this, but the answer might surprise you: it’s really common!

You see, this is exactly what you want to happen when trying to write code that works across multiple device sizes - if we want layout to happen vertically when horizontal space is constrained, but horizontally otherwise.

Apple provides a very simple solution called size classes, which is a thoroughly vague way of telling us how much space we have for our views.

When I say "thoroughly vague" I mean it: we have only two size classes horizontally and vertically, called "compact" and "regular". That’s it - that covers all screen sizes from the largest iPad Pro in landscape down to the smallest iPhone in portrait. That doesn’t mean it’s useless - far from it! - just that it only lets us reason about our user interfaces in the broadest terms.

To demonstrate size classes in action, we could create a view that has a property to track the current size class so we can switch between VStack and HStack automatically: 

struct ContentView: View {
    @Environment(\.horizontalSizeClass) var horizontalSizeClass

    var body: some View {
        if horizontalSizeClass == .compact {
            VStack {
                UserView()
            }
        } else {
            HStack {
                UserView()
            }
        }
    }
}
NOTE: This next tip doesn't play well in modern SwiftUI. See Ai Insight below.

Tip: In situations like this, where you have only one view inside a stack and it doesn’t take any parameters, you can pass the view’s initializer directly to the VStack to make your code shorter: 

if sizeClass == .compact {
    VStack(content: UserView.init)
} else {
    HStack(content: UserView.init)
}

I know short code isn’t everything, but this technique is pleasingly concise when you’re using this approach to grouped view layout.

What you see when that code runs depends on the device you’re using. For example, an iPhone 15 Pro will have a compact horizontal size class in both portrait and landscape, whereas an iPhone 15 Pro Max will have a regular horizontal size class when in landscape.

Regardless of whether we’re toggling our layout using size classes or buttons, the point is that UserView just doesn’t care - its Group simply groups the text views together without affecting their layout at all, so the layout arrangement UserView is given depends entirely on how it’s used.

View That Fits

Before I'm done, I want to mention that SwiftUI does provide a view that goes some way towards making this behavior easier. It's called ViewThatFits, and you can provide it with several different layouts - it will automatically try each one in order until it finds one that can be fitted into the available space.

For example, this will attempt to show a 500x200 rectangle by default, but if that can't fit into the available space it will show a 200x200 circle instead: 

ViewThatFits {
    Rectangle()
        .frame(width: 500, height: 200)

    Circle()
        .frame(width: 200, height: 200)
}

If you can use ViewThatFits then you should, because if you show the same view in several different forms then SwiftUI will correctly preserve that view's state as the layout changes. However, it does limit how much control you get!

In Context

ContentView.swift

import SwiftUI


struct ContentView: View {
    
    var body: some View {
        ViewThatFits {
            Rectangle()
                .frame(width: 500, height: 200)
            
            Circle()
                .frame(width: 200, height: 200)
        }
    }
}

Ai Insight

QUESTION

In the example of passing the view's initializer into a stack... 

if sizeClass == .compact {
    VStack(content: UserView.init)
} else {
    HStack(content: UserView.init)
}

...I get this error: 

Call to main actor-isolated initializer 'init()' in a synchronous nonisolated context

What's wrong?

ANSWER

The error occurs because of stricter concurrency safety rules introduced in recent Swift updates.

In SwiftUI, all View types are automatically isolated to the @MainActor (meaning they are strictly forced to run on the main user-interface thread to prevent crashes).

When you write UserView.init as a direct function reference, you are passing the initializer into the VStack and HStack as a generic, un-isolated piece of code. The compiler panics because it cannot guarantee that this naked init() function won't be accidentally triggered on a dangerous background thread.

How to Fix It

To make the compiler happy, you just need to explicitly instantiate the view inside a standard SwiftUI @ViewBuilder closure (using curly braces { }). This tells Swift exactly which context the initializer is running in.

Here is the updated, error-free code (which is the same code given initially in the tutorial - but here with comments): 

import SwiftUI

struct UserView: View {
    var body: some View {
        Group {
            Text("Name: Paul")
            Text("Country: England")
            Text("Pets: Luna and Arya")
        }
        .font(.title)
    }
}

struct ContentView: View {
    @Environment(\.horizontalSizeClass) var horizontalSizeClass
    
    var body: some View {
        // Wrap the layout choices inside a container like a VStack or Group
        // so SwiftUI can handle the conditional rendering safely.
        Group {
            if horizontalSizeClass == .compact {
                // FIX: Use traditional closure braces instead of 'content: UserView.init'
                VStack {
                    UserView()
                }
            } else {
                // FIX: Use traditional closure braces instead of 'content: UserView.init'
                HStack {
                    UserView()
                }
            }
        }
    }
}

Why did Paul Hudson's code use UserView.init?

When Paul Hudson originally wrote that tutorial, older versions of Swift allowed you to pass initializers as arguments (UserView.init) as a clever shorthand.

However, as Apple finalized Swift 6 Data Isolation, they locked down initializer references to prevent multithreading bugs. Wrapping your views inside standard trailing closures (VStack { UserView() }) ensures the code inherits the proper @MainActor main-thread safety automatically.

QUESTION

What about this syntax? Does this work in Swift 6? 

ViewThatFits {
Rectangle()
.frame(width: 500, height: 200)

Circle()
.frame(width: 200, height: 200)}

ANSWER

Yes, this syntax works perfectly in Swift 6!

In fact, using ViewThatFits is the modern, preferred way to handle this exact layout challenge without having to manually inspect the @Environment(\.horizontalSizeClass) like you did in the previous example.

Why it works safely in Swift 6

In your previous example, the compiler threw an error because UserView.init passed a naked function pointer across thread boundaries.

With ViewThatFits, you are using standard trailing closure syntax ({ ... }). This closure is implicitly marked with @ViewBuilder and correctly inherits @MainActor isolation. Swift 6 is perfectly happy because it knows every view inside those braces is safely bound to the main user-interface thread.

How ViewThatFits works behind the scenes

ViewThatFits acts like an automated triage system for your layouts:

  • It evaluates your views in the exact order you list them (from top to bottom).
  • It looks at the first view (Rectangle) and tests: "Can a 500-point wide box fit onto the current screen without clipping or overflowing?"
  • If it fits (like on a large iPad or Mac screen), it renders the Rectangle and completely ignores the code below it.
  • If it doesn't fit (like on a vertical iPhone screen), it seamlessly drops down to the next option (Circle) and checks if a 200-point wide shape fits instead.

When to use this over horizontalSizeClass

Approach Best Used For
ViewThatFits Sizing-specific fixes. (e.g., preventing a long string of text or a button group from clipping on smaller iPhones). It responds purely to physical pixel constraints.
horizontalSizeClass Structural design shifts. (e.g., completely changing an app's layout from a bottom tab bar on iPhone to a side navigation bar on iPad).
