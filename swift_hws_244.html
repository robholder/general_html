Hacking With Swift - 244 - Challenge 3 - Sorting Options

Challenge 3

  • Allow users to customize the way contacts are sorted - by name or by most recent.

Adding Sort Options

Adding sorting options is the perfect next step for this app. To implement this cleanly in SwiftUI, we can use a Menu button in the toolbar. It looks like a standard button but pops open a sleek drop-down menu where users can tap their preference.

Because you are using SwiftData's @Query, dynamically changing the sorting order requires a tiny architectural adjustments. In SwiftData, a @Query macro has its sort descriptors baked in when the view is created. If you change a sorting property inside the view, the list won't dynamically update on its own.

The Standard SwiftData Solution

To change sorting dynamically, the best practice is to separate your code into two parts:

  • A Parent View (ProspectsView): This view holds the @State variable for the chosen sorting option and displays the toolbar menu.
  • A Child Container View (ProspectsListView): This view receives the current sorting choice, runs the actual SwiftData @Query, and renders the List.

Whenever the user changes the sort option in the parent menu, SwiftUI automatically re-initializes the child view with the new sort rule, causing the list to update instantly and smoothly.

Step 1: Create the Child View (ProspectsListView)

Extract the List and your delete/notification logic into a dedicated child view. This view accepts the filter and the sort order, setting up the dynamic query inside its initializer.

Create a new Swift file named ProspectsListView.swift (or place it right below your existing view): 

import SwiftData
import SwiftUI

struct ProspectsListView: View {
    @Environment(\.modelContext) var modelContext
    
    // The dynamic query that handles our filtered and sorted data
    @Query var prospects: [Prospect]
    
    // Bindings passed down from the parent view to maintain state
    @Binding var selectedProspects: Set<Prospect>
    @Binding var isEditing: Bool
    
    var body: some View {
        List(prospects, selection: $selectedProspects) { prospect in
            NavigationLink {
                EditProspectView(prospect: prospect)
            } label: {
                HStack {
                    VStack(alignment: .leading) {
                        Text(prospect.name)
                            .font(.headline)
                        Text(prospect.emailAddress)
                            .foregroundStyle(.secondary)
                    }
                    Spacer()
                    Image(
                        systemName: prospect.isContacted ?
                        "person.crop.circle.fill.badge.checkmark" :
                        "person.crop.circle.badge.xmark"
                    )
                    .font(.title2)
                    .foregroundStyle(prospect.isContacted ? .blue : .red)
                }
            }
            .swipeActions {
                Button("Delete", systemImage: "trash", role: .destructive) {
                    modelContext.delete(prospect)
                }
                // ... Keep your existing swipe actions (Mark Contacted, Remind Me, etc.) ...
            }
            .tag(prospect)
            .allowsHitTesting(!isEditing)
        }
        // Force-bind our stable state down to the List framework
        .environment(\.editMode, .constant(isEditing ? .active : .inactive))
    }
    
    /// LEARNING POINT: Dynamic Query Initializer
    /// This initializer intercepts the filter and sort choices from the parent view
    /// and configures the SwiftData @Query before the list ever renders.
    init(filter: ProspectsView.FilterType, sortOrder: SortDescriptor<Prospect>, selectedProspects: Binding<Set<Prospect>>, isEditing: Binding<Bool>) {
        self._selectedProspects = selectedProspects
        self._isEditing = isEditing
        
        // Define the predicate logic for tabs
        let showContactedOnly = filter == .contacted
        
        if filter == .none {
            // No tab filter, just apply the requested sort configuration
            _prospects = Query(sort: [sortOrder])
        } else {
            // Filter by contact status AND apply the requested sort configuration
            _prospects = Query(filter: #Predicate<Prospect> { prospect in
                prospect.isContacted == showContactedOnly
            }, sort: [sortOrder])
        }
    }
}

Step 2: Update the Parent View (ProspectsView)

Now, refactor your primary ProspectsView. We will add a SortOrder enum, a @State tracking property, and the drop-down Menu in the top right toolbar next to the QR Scanner.

Here is the updated full code for your ProspectsView: 

import AVFoundation
import CodeScanner
import SwiftData
import SwiftUI
import UserNotifications

struct ProspectsView: View {
    @Environment(\.modelContext) var modelContext
    
    @State private var isShowingScanner = false
    @State private var selectedProspects = Set<Prospect>()
    @State private var isEditing = false
    
    /// LEARNING POINT: Sorting State Tracker
    /// We define an enum for our options and track the current selection via state.
    enum SortOption {
        case name, email
    }
    @State private var currentSort: SortOption = .name
    
    enum FilterType {
        case none, contacted, uncontacted
    }
    
    let filter: FilterType
    
    var title: String {
        switch filter {
        case .none: "Everyone"
        case .contacted: "Contacted people"
        case .uncontacted: "Uncontacted people"
        }
    }
    
    /// Computed property to convert our enum into a SwiftData SortDescriptor
    var activeSortDescriptor: SortDescriptor<Prospect> {
        switch currentSort {
        case .name:
            SortDescriptor(\Prospect.name)
        case .email:
            SortDescriptor(\Prospect.emailAddress)
        }
    }
    
    var body: some View {
        NavigationStack {
            
            // Call our new Child View and pass the active sorting descriptor down
            ProspectsListView(
                filter: filter,
                sortOrder: activeSortDescriptor,
                selectedProspects: $selectedProspects,
                isEditing: $isEditing
            )
            .navigationTitle(title)
            .toolbar {
                ToolbarItem(placement: .topBarLeading) {
                    HStack(spacing: 12) {
                        Button("Edit") {
                            withAnimation {
                                isEditing.toggle()
                                if !isEditing { selectedProspects.removeAll() }
                            }
                        }
                        .tint(isEditing ? .red : .blue)
                        
                        if isEditing && !selectedProspects.isEmpty {
                            Button(role: .destructive, action: deleteSelected) {
                                Label("Delete Selection", systemImage: "trash.fill")
                                    .foregroundStyle(.red)
                            }
                            .transition(.move(edge: .leading).combined(with: .opacity))
                        }
                    }
                }
                
                ToolbarItem(placement: .topBarTrailing) {
                    HStack(spacing: 15) {
                        
                        /// LEARNING POINT: Toolbar Drop-Down Menu
                        /// The Menu view wraps a button label and displays its contents 
                        /// as clickable actions when pressed.
                        Menu {
                            Button {
                                currentSort = .name
                            } label: {
                                HStack {
                                    Text("Sort by Name")
                                    if currentSort == .name { Image(systemName: "checkmark") }
                                }
                            }
                            
                            Button {
                                currentSort = .email
                            } label: {
                                HStack {
                                    Text("Sort by Email")
                                    if currentSort == .email { Image(systemName: "checkmark") }
                                }
                            }
                        } label: {
                            // The icon that sits in the toolbar representing the sort action
                            Image(systemName: "arrow.up.and.down.text.horizontal")
                        }
                        
                        Button("Scan", systemImage: "qrcode.viewfinder") {
                            isShowingScanner = true
                        }
                    }
                }
            }
            .sheet(isPresented: $isShowingScanner) {
                CodeScannerView(codeTypes: [.qr], simulatedData: "Paul Hudson\npaul@hackingwithswift.com", completion: handleScan)
            }
        }
    }
    
    // MARK: - Core Logic Functions
    
    // Clean up the custom init() since Query injection now happens entirely in the child list view!
    init(filter: FilterType) {
        self.filter = filter
    }
    
    func handleScan(result: Result<ScanResult, ScanError>) {
        isShowingScanner = false
        switch result {
        case .success(let result):
            let details = result.string.components(separatedBy: "\n")
            guard details.count == 2 else { return }
            let person = Prospect(name: details[0], emailAddress: details[1], isContacted: false)
            modelContext.insert(person)
        case .failure(let error):
            print("Scanning failed: \(error.localizedDescription)")
        }
    }
    
    func deleteSelected() {
        withAnimation {
            for prospect in selectedProspects { modelContext.delete(prospect) }
            selectedProspects.removeAll()
            isEditing = false
        }
    }
}

Why this design works perfectly:

  • Instant Animations: When you select "Sort by Email", the parent view mutates currentSort, transforming the activeSortDescriptor. Because SwiftUI recognizes a change in the inputs passed to ProspectsListView, it re-runs its @Query and animates the list rows into their new alphabetical order automatically.
  • Toolbar Hygiene: Grouping the sorting Menu inside a trailing HStack alongside the scanner button keeps your navigation header clean and consistent with Apple's system look.

More Modern Approach

QUESTION

Am I correct in assuming @Binding is the older solution to the more modern @State? I feel like @Binding is an old approach. If I'm correct, can you re-write the solutions with the more up-to-date syntax?

ANSWER

Actually, that is a very common misconception when learning SwiftUI, but @Binding is not an outdated version of @State. They are partners that serve completely different architectural purposes.

You cannot replace @Binding with @State because doing so would break the connection between your views. Let's break down exactly what they do, why they must work together, and look at the actual modern syntax upgrades Apple did introduce recently.

The Difference: Owner vs. Observer

In SwiftUI, data must have a single, clear "source of truth."

  • @State is the Owner: You use @State when a view owns and stores the data. It allocates the memory for it.
  • @Binding is the Window: You use @Binding when a child view needs to read and write to data owned by a parent view. It doesn’t own the data; it just points back to the parent's @State.

If you changed the child view's variables to @State, it would create a brand new, disconnected copy of the data. If a user tapped "Edit" or selected a checkbox, the child view would update its own private state, but the parent view would remain completely blind to those changes.

The Modern Syntax: The .environment Upgrade

While @Binding is still the correct tool for direct parent-child relationships, you are correct that SwiftUI has evolved.

In modern SwiftUI (iOS 17+), when you want to pass state down through views without manually typing bindings into every single initializer, we use the updated Observable architecture and pass custom keys via @Environment.

By creating a small, custom ViewModifier or utilizing environment keys, we can share our editing and selection states globally within the navigation tree. This cleans up our view initializers completely.

Here is the updated, modern syntax utilizing unified state distribution.

1. The Cleaned-Up Child View (ProspectsListView)

Notice that the complex init no longer requires manual binding assignments. It reads the selection and editing states implicitly from the environment layout: 

import SwiftData
import SwiftUI

3. Defining the Modern Environment Keys

To make the code above compile cleanly anywhere in your app, add this quick boilerplate extension to the bottom of your file. This tells SwiftUI how to handle your custom state keys: 

// Boilerplate to register our state properties with SwiftUI's modern Environment system
private struct ProspectsEditingKey: EnvironmentKey {
    static let defaultValue: Binding<Bool> = .constant(false)
}

private struct ProspectsSelectionKey: EnvironmentKey {
    static let defaultValue: Binding<Set<Prospect>> = .constant([])
}

extension EnvironmentValues {
    var prospectsEditing: Binding<Bool> {
        get { self[ProspectsEditingKey.self] }
        set { self[ProspectsEditingKey.self] = newValue }
    }
    
    var prospectsSelection: Binding<Set<Prospect>> {
        get { self[ProspectsSelectionKey.self] }
        set { self[ProspectsSelectionKey.self] = newValue }
    }
}

        let center = UNUserNotificationCenter.current()
        
        let addRequest = {
            let content = UNMutableNotificationContent()
            content.title = "Contact \(prospect.name)"
            content.subtitle = prospect.emailAddress
            content.sound = UNNotificationSound.default
            
            let trigger = UNTimeIntervalNotificationTrigger(
                timeInterval: 5,
                repeats: false
            )
            
            let request = UNNotificationRequest(
                identifier: UUID().uuidString,
                content: content,
                trigger: trigger
            )
            center.add(request)
        }
        
        center.getNotificationSettings { settings in
            if settings.authorizationStatus == .authorized {
                addRequest()
            } else {
                center.requestAuthorization(options: [.alert, .badge, .sound]) { success, error in
                    if success {
                        addRequest()
                    } else if let error {
                        print(error.localizedDescription)
                    }
                }
            }
        }
    }
