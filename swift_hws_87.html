Hacking With Swift - 87 - Building custom transitions using ViewModifier

It’s possible - and actually surprisingly easy - to create wholly new transitions for SwiftUI, allowing us to add and remove views using entirely custom animations.

This functionality is made possible by the .modifier transition, which accepts any view modifier we want. The catch is that we need to be able to instantiate the modifier, which means it needs to be one we create ourselves.

Creating a View Modifier

To try this out, we could write a view modifier that lets us mimic the Pivot animation in Keynote - it causes a new slide to rotate in from its top-left corner. In SwiftUI-speak, that means creating a view modifier that causes our view to rotate in from one corner, without escaping the bounds it’s supposed to be in. SwiftUI actually gives us modifiers to do just that: rotationEffect() lets us rotate a view in 2D space, and clipped() stops the view from being drawn outside of its rectangular space.

rotationEffect() is similar to rotation3DEffect(), except it always rotates around the Z axis. However, it also gives us the ability to control the anchor point of the rotation - which part of the view should be fixed in place as the center of the rotation. SwiftUI gives us a UnitPoint type for controlling the anchor, which lets us specify an exact X/Y point for the rotation or use one of the many built-in options - .topLeading, .bottomTrailing, .center, and so on.

Let’s put all this into code by creating a CornerRotateModifier struct that has an anchor point to control where the rotation should take place, and an amount to control how much rotation should be applied: 

struct CornerRotateModifier: ViewModifier {
    let amount: Double
    let anchor: UnitPoint

    func body(content: Content) -> some View {
        content
            .rotationEffect(.degrees(amount), anchor: anchor)
            .clipped()
    }
}

The addition of clipped() there means that when the view rotates the parts that are lying outside its natural rectangle don’t get drawn.

Structural Breakdown

  • struct - keyword
  • CornerRotateModifier - user name for struct
  • ViewModifier (The Protocol): It is a protocol. In Swift, protocols define a "contract." By saying : ViewModifier, you are promising the compiler: "I will provide a function called body that transforms a view."
  • let amount: Double - user constant of Double type
  • let anchor: UnitPoint - user constant of UnitPoint type
  • func body - required method for a ViewModifier struct
  • content: Content: This is a parameter. Think of Content as a placeholder for "whatever view this modifier is being applied to." If you apply this to a Text, content is that Text. If you apply it to an Image, content is that Image.
  • content (inside the function): This is the usage of that parameter. You take the original view (content) and wrap it in new layers (rotation and clipping).
  • -> some View: This is an Opaque Return Type. It tells SwiftUI: "I’m going to return something that follows the View protocol, but I’m not going to make you worry about the specific, complex underlying type (like ClippedView<RotatedView<Content>>)."

Method Inheritance: Where do .rotationEffect and .clipped() come from?

This is a common point of confusion. These methods are not inherited from a parent class in the traditional Object-Oriented sense. Instead, they are Protocol Extensions.

Method Source What it does
.rotationEffect View Protocol Extension Takes the amount and anchor and calculates a transformation matrix to tilt the pixels.
.clipped() View Protocol Extension It tells the rendering engine to ignore any part of the view that sits outside its original bounding box (frame).

How it works together:

SwiftUI provides a massive library of functions (modifiers) that are available to any type that conforms to the View protocol. Because the input content is a type of Content (which is a View), you can chain those built-in modifiers directly onto it.

The "Feeding" Process

The .rotationEffect is being "fed" the new properties:

  • You pass -90 and .topLeading into your Extension (.pivot).
  • The extension initializes your Struct (CornerRotateModifier) with those values.
  • Your struct then "feeds" those values into the built-in SwiftUI method (.rotationEffect).

The Final Visual

Think of a ViewModifier like a custom filter for a camera.

  • content: The raw image coming through the lens.
  • rotationEffect: A lens attachment that tilts the image.
  • clipped: A frame that cuts off the edges.
  • body: The assembly line where you put the lens and frame together.

Extension

We can try that straight away using the .modifier transition, but it’s a little unwieldy. A better idea is to wrap that in an extension to AnyTransition, making it rotate from -90 to 0 on its top leading corner: 

extension AnyTransition {
    static var pivot: AnyTransition {
        .modifier(
            active: CornerRotateModifier(amount: -90, anchor: .topLeading),
            identity: CornerRotateModifier(amount: 0, anchor: .topLeading)
        )
    }
}

AI Insights

This is a Type Extension adding a custom static property. In SwiftUI, this is the standard "clean" way to create reusable transitions that look just like the built-in ones (e.g., .transition(.opacity)).

Here is the dissection of what is happening under the hood.

1. The Extension Target

extension AnyTransition

You are extending the AnyTransition struct. By adding a static var here, you enable dot syntax in your views. Instead of typing a long constructor, you can simply write:

view.transition(.pivot)

2. The Static Property

static var pivot: AnyTransition { ... }

  • Static: This belongs to the type itself, not a specific instance of a transition.
  • Computed Property: Every time you call .pivot, it returns a new AnyTransition instance configured with your custom logic.

3. The .modifier Method

This is the "engine" of your transition. While SwiftUI has basic transitions like opacity or move, .modifier allows you to create completely custom geometric changes by wrapping a ViewModifier.

It requires two states:

  • active: This is how the view looks when it is "not there" (either just before it appears or just after it disappears). In your code, it's rotated -90 degrees.
  • identity: This is how the view looks when it is fully on screen. Here, the rotation is 0 degrees.

4. CornerRotateModifier

CornerRotateModifier(amount: ..., anchor: .topLeading)

This isn't a built-in SwiftUI tool; it's a Custom ViewModifier that we created above.

Putting it all together: The "Visual" Flow

When you trigger this transition (e.g., inside an if block with withAnimation):

  • Insertion: The view is added to the hierarchy. SwiftUI looks at the active state. The view starts at -90° pinned to the top-leading corner. It then animates to the identity state (0°). It looks like the view is "swinging" down into place.
  • Removal: SwiftUI animates the view from identity (0°) back to active (-90°) before removing it from the screen. It looks like the view is "swinging" up and out.

Why do it this way?

This pattern is powerful because it encapsulates complexity. Your View code stays clean: 

if isShowing {
    Rectangle()
        .fill(.blue)
        .transition(.pivot) // Clean and readable!
}

Back to the tutorial...

Putting it into Use

With that in place we now attach the pivot animation to any view using this: 

.transition(.pivot)

For example, we could use the onTapGesture() modifier to make a red rectangle pivot onto the screen, like this: 

struct ContentView: View {
    @State private var isShowingRed = false

    var body: some View {
        ZStack {
            Rectangle()
                .fill(.blue)
                .frame(width: 200, height: 200)

            if isShowingRed {
                Rectangle()
                    .fill(.red)
                    .frame(width: 200, height: 200)
                    .transition(.pivot)
            }
        }
        .onTapGesture {
            withAnimation {
                isShowingRed.toggle()
            }
        }
    }
}

More AI Insights

The CornerRotateModifier is the "engine" that makes the pivot transition possible. While it looks simple, there are three critical architectural details that explain why it works and how it behaves during an animation.

Here it is repeated from above: 

struct CornerRotateModifier: ViewModifier {
    let amount: Double
    let anchor: UnitPoint

    func body(content: Content) -> some View {
        content
            .rotationEffect(.degrees(amount), anchor: anchor)
            .clipped()
    }
}

1. The "Animatable" Magic

Even though you don’t see an animatableData property here, this struct works because it wraps a built-in SwiftUI modifier that is animatable: .rotationEffect.

When SwiftUI transitions between the active state (-90°) and the identity state (0°), it doesn't just jump between the two. Because this struct is used inside an AnyTransition, SwiftUI looks at the amount property and interpolates the values. It calculates every fractional degree in between (e.g., -45°, -10°) to create a smooth swinging motion.

2. The Anchor Point (UnitPoint)

The anchor property is what defines the "hinge" of the pivot.

  • By using .topLeading, you are telling SwiftUI to pin the top-left corner of the view.
  • If you changed that anchor to .center, the view would spin like a fan.
  • By pinning it to a corner, you create a "gate" or "page-turn" effect.

3. The Role of .clipped()

This is a small but vital inclusion in your body function.

  • When a view rotates, its corners often extend past its original bounding box.
  • .clipped() ensures that the view doesn't overlap other UI elements during its "swing."
  • Note: Depending on your design, you might actually want to remove .clipped() if you want to see the view swinging "over" the surrounding content.

How it fits into the "Identity" Concept

In SwiftUI transitions, Identity is a crucial concept.

  • When amount is 0, the modifier is in its "Identity" state—meaning it has zero effect on the view's appearance.

8 This is why the extension defines identity: CornerRotateModifier(amount: 0, ...). It tells SwiftUI: "This is exactly how the view should look when it is fully settled on the screen."

A Quick Experiment

To see how powerful this struct is, you can actually use it without the transition just to test it: 

Rectangle()
    .fill(.orange)
    .frame(width: 100, height: 100)
    .modifier(CornerRotateModifier(amount: -45, anchor: .topLeading))

This will show the rectangle permanently tilted at 45 degrees, hanging from its top-left corner.

Summary Table: The Pivot Logic

Property Value in active Value in identity Visual Result
amount -90 0 Swings from "standing up" to "laying flat."
anchor .topLeading .topLeading The top-left corner never moves.
clipped Active Active Keeps the rotation tidy within its frame.
