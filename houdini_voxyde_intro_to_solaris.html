Solaris

Solaris is built on USD.

USD is a file format, a data structure and a framework.

TIP: A quick way to switch to Solaris is hold n key in the 'Network Editor' pane, and choose LOPs option from the pop-up radial menu.

Virtual synonyms for USD context:

  • LOPs
  • Stage
  • Solaris

Working in Solaris

Don't forget that there are a few Solaris builds (desktops) available. These will give access to the Geometry Spreadsheet, Scene Graph and the Network Editor, which is where a lot of the structure of USD will be displayed.

Geometry Spreadsheet

The Geometry Spreadsheet is context sensitive to whether we have selected an item in Solaris or in the regular 'obj' context.

  • In the Solaris context:
    • You'll see all the USD scene structure and hierarchy
  • In the '/obj' context:
    • You'll see the table of parameters and attributes

TIP: Don't forget you can nip back and forth from Stage to /obj contexts with the n key and radial menu, and selecting:

  • LOPs (For USD/Solaris/Stage)
  • Object (For '/obj')
  • etc.

'/obj' vs 'Solaris'

You can create assets/objects in either '/obj' or 'Solaris' contexts.

If you create in the '/obj' context:

  • Import to Solaris:
    • Sub-layering with a sublayer node

If you create directly in the 'Solaris' context:

  • You can just drop direct geometry

Primitive Path

When in Solaris/Stage/USD context, you can organize the hierarchy using Primitive Path strings/paths:

  • Add the Primitive Path string in the item's Parameter Panel to arrange in the USD hierarchy:
    • '/' is the root of the USD hierarchy, therefore *all paths must start with '/'*
    • '/blah/blah/$OS'
    • e.g.,:
      • '/environment/assets/cube'
      • '/environment/assets/$OS'

NOTE: if the same 'Primitive Path' is applied to multiple objects, the concept of overrides (or Opinions in USD-speak) will come into play, and only the latest item with the given path will appear in the hierarchy. (The previous one will be 'overwritten' and will effectively disappear, or be merged with the last item with the same path.

This concept is very useful if used correctly (to combine versions - maybe shaded and animated versions - but can be confusing if not managed correctly.

$OS

The $OS Variable is the 'name' attribute of the item/object. This may be the item name (as named in the Network Editor) or the 'name' attribute (which overrides the actual typed name) if present.

This is useful for populating the Primitive Path field of an object.

However, you can name the object anything without having to reference the actual item name in the Network Editor.

Combining items in Solaris/Stage/USD

You can chain items together along a chained spine, or you can use Merge nodes to join items. There must be some connection to create the overall hierarchy of items (as displayed in the Geometry Spreadsheet or Scene Graph).

This is more like regular SOPs workflow: 


cube   cylinder
 \     /
  \   /
   \ /
   Merge

But this works too (a long chain of elements): 


cube
 |
 |
cylinder

Long chains can be organized more like a node based system using merges (like Nuke): 


item
 |
 |
Merge ---- item2
 |
 |
Merge ---- item3
 |
 |
Merge ---- item4
 |
 |
etc.

Workflow to import /obj geometry:

  • In Stage/Solaris/LOPs/USD:
    • SOP Import node:
      • Set SOP Path parameter on the 'SOP Import' node to the '/obj' item:
        • example:
          • /obj/hero/OUT_hero
      • Set Import Path Prefix to set the USD hierarchy path:
        • example:
          • /characters/asset/$OS:
            • This might give a generic 'mesh_0' or similar is 'name' not specifies in '/obj' land
            • To give the item a name, in '/obj' land, add a Name SOP/node, and add a name as an attribute:
              • Leave Attribute as 'name'
              • Add a string (name) to the Name attribute field

Protected Vars for 'name' and 'path'

If you use the Name SOP/node, you can add:

  • Attribute 'name'
  • and 'Name' field with USD hierarchy name.

However, you can add:

  • Attribute 'path'
  • and 'Name' field with USD hierarchy path

'path' attribute overrides 'name' attribute

The important thing is that:

  • 'path' overrides 'name' (if the path ends with an explicit item name: i.e., doesn't end in '/')
  • 'path overrides 'Import Path Prefix' in the SOP Import node back in Stage/Solaris/USD world

OUTPUT nodes in Solaris/Stage/USD

In '/obj' world, it's common to drop a Null node at the end of a network and call it something like 'OUT_thing'.

In Solaris/Stage/USD, you can drop a dedication Output node. Same function as the 'Null' in '/obj' contexts.

SOP Create Node

The SOP Create node acts just like a geometry container in the '/obj' context. It is arguably the 'preferred' method of bringing or making items (rather than importing).

Create an empty 'SOP Create' node, and you can drop geometry subnodes as you would in '/obj' world.

Example:

  • SOP Create:
    • Inside:
      • Drop a 'rubber toy' item
      • Add a Match Size set to 'justify min' to zero, to set it on the ground level
      • Add an Output node as the 'OUT_rubber_toy' node

NOTE: SOP Create has its own 'Import Path Prefix' for the Solaris/Stage/USD hierarchy. Include the object name at the end of the path to negate the need of adding a 'Name' node (for adding 'name' and/or 'path' attributes).

In the case of the rubber toy (a Houdini item), the object is already set up with a material from the SHOPs context. This is imported when using a 'SOP Create Node', while not when using the 'SOP Import' node.

To switch off this import of material:

This might be just for Houdini default objects (like the Rubber Toy):

  • On the 'SOP Create' parameters:
    • Partition Attributes field:
      • Either delete the path string
      • Or switch off attribute with checkbox
  • Or, you can just disable the 'Add Shader' checkbox on the rubber_toy node since it's offered.

Karma Render Settings

Add the Karma Render Settings node to set the settings for the rendering of the scene.

Random TIPS:

  • <shift> R over viewport to get Karma Renderer (using 'viewport display' settings if no Karma Render Settings node added)
  • Remember to choose the camera in the Karma Render Settings:
    • default is /camera/camera1
  • Viewing the network ABOVE the Karma Render Settings node, will force the Viewport to use the 'Display' settings rather than the Karma Render Settings.
  • D in Viewport to open display settings.
  • Disable the DOF setting if there is a load of blur.

USD Render ROP

Add the USD Render ROP to conclude the network.

  • Set Render Deligate to:
    • Karma CPU
    • Karma XPU
    • etc.
