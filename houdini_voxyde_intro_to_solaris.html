Solaris

Solaris is built on USD.

USD is a file format, a data structure and a framework.

TIP: A quick way to switch to Solaris is hold n key in the 'Network Editor' pane, and choose LOPs option from the pop-up radial menu.

Virtual synonyms for USD context:

  • LOPs
  • Stage
  • Solaris

Working in Solaris

Don't forget that there are a few Solaris builds (desktops) available. These will give access to the Geometry Spreadsheet, Scene Graph and the Network Editor, which is where a lot of the structure of USD will be displayed.

Geometry Spreadsheet

The Geometry Spreadsheet is context sensitive to whether we have selected an item in Solaris or in the regular 'obj' context.

  • In the Solaris context:
    • You'll see all the USD scene structure and hierarchy
  • In the '/obj' context:
    • You'll see the table of parameters and attributes

TIP: Don't forget you can nip back and forth from Stage to /obj contexts with the n key and radial menu, and selecting:

  • LOPs (For USD/Solaris/Stage)
  • Object (For '/obj')
  • etc.

'/obj' vs 'Solaris'

You can create assets/objects in either '/obj' or 'Solaris' contexts.

If you create in the '/obj' context:

  • Import to Solaris:
    • Sub-layering with a sublayer node

If you create directly in the 'Solaris' context:

  • You can just drop direct geometry

Primitive Path

When in Solaris/Stage/USD context, you can organize the hierarchy using Primitive Path strings/paths:

  • Add the Primitive Path string in the item's Parameter Panel to arrange in the USD hierarchy:
    • '/' is the root of the USD hierarchy, therefore *all paths must start with '/'*
    • '/blah/blah/$OS'
    • e.g.,:
      • '/environment/assets/cube'
      • '/environment/assets/$OS'

NOTE: if the same 'Primitive Path' is applied to multiple objects, the concept of overrides (or Opinions in USD-speak) will come into play, and only the latest item with the given path will appear in the hierarchy. (The previous one will be 'overwritten' and will effectively disappear, or be merged with the last item with the same path.

This concept is very useful if used correctly (to combine versions - maybe shaded and animated versions - but can be confusing if not managed correctly.

$OS

The $OS Variable is the 'name' attribute of the item/object. This may be the item name (as named in the Network Editor) or the 'name' attribute (which overrides the actual typed name) if present.

This is useful for populating the Primitive Path field of an object.

However, you can name the object anything without having to reference the actual item name in the Network Editor.

Combining items in Solaris/Stage/USD

You can chain items together along a chained spine, or you can use Merge nodes to join items. There must be some connection to create the overall hierarchy of items (as displayed in the Geometry Spreadsheet or Scene Graph).

This is more like regular SOPs workflow: 


cube   cylinder
 \     /
  \   /
   \ /
   Merge

But this works too (a long chain of elements): 


cube
 |
 |
cylinder

Long chains can be organized more like a node based system using merges (like Nuke): 


item
 |
 |
Merge ---- item2
 |
 |
Merge ---- item3
 |
 |
Merge ---- item4
 |
 |
etc.

Scene Assembly

Workflow to import /obj geometry:

  • In Stage/Solaris/LOPs/USD:
    • SOP Import node:
      • Set SOP Path parameter on the 'SOP Import' node to the '/obj' item:
        • example:
          • /obj/hero/OUT_hero
      • Set Import Path Prefix to set the USD hierarchy path:
        • example:
          • /characters/asset/$OS:
            • This might give a generic 'mesh_0' or similar is 'name' not specifies in '/obj' land
            • To give the item a name, in '/obj' land, add a Name SOP/node, and add a name as an attribute:
              • Leave Attribute as 'name'
              • Add a string (name) to the Name attribute field

Protected Vars for 'name' and 'path'

If you use the Name SOP/node, you can add:

  • Attribute 'name'
  • and 'Name' field with USD hierarchy name.

However, you can add:

  • Attribute 'path'
  • and 'Name' field with USD hierarchy path

'path' attribute overrides 'name' attribute

The important thing is that:

  • 'path' overrides 'name' (if the path ends with an explicit item name: i.e., doesn't end in '/')
  • 'path overrides 'Import Path Prefix' in the SOP Import node back in Stage/Solaris/USD world

OUTPUT nodes in Solaris/Stage/USD

In '/obj' world, it's common to drop a Null node at the end of a network and call it something like 'OUT_thing'.

In Solaris/Stage/USD, you can drop a dedication Output node. Same function as the 'Null' in '/obj' contexts.

SOP Create Node

The SOP Create node acts just like a geometry container in the '/obj' context. It is arguably the 'preferred' method of bringing or making items (rather than importing).

Create an empty 'SOP Create' node, and you can drop geometry subnodes as you would in '/obj' world.

Example:

  • SOP Create:
    • Inside:
      • Drop a 'rubber toy' item
      • Add a Match Size set to 'justify min' to zero, to set it on the ground level
      • Add an Output node as the 'OUT_rubber_toy' node

NOTE: SOP Create has its own 'Import Path Prefix' for the Solaris/Stage/USD hierarchy. Include the object name at the end of the path to negate the need of adding a 'Name' node (for adding 'name' and/or 'path' attributes).

In the case of the rubber toy (a Houdini item), the object is already set up with a material from the SHOPs context. This is imported when using a 'SOP Create Node', while not when using the 'SOP Import' node.

To switch off this import of material:

This might be just for Houdini default objects (like the Rubber Toy):

  • On the 'SOP Create' parameters:
    • Partition Attributes field:
      • Either delete the path string
      • Or switch off attribute with checkbox
  • Or, you can just disable the 'Add Shader' checkbox on the rubber_toy node since it's offered.

Stage Manager

YouTube - timecode: 00:29:25

The Stage Manager is a way of moving stuff about in the assembled Solaris/Stage/USD scene.

Moving Items

If you edit an item in the Viewport while in Solaris, you will get an Edit node appear, which records the edits.

You can also add regular Transform nodes, and edit the transformations (even at an object by object level by pasting the object path into the 'parameters' panel (Primitive field).

The Stage Manager

The Stage Manager is a better way to edit the scene within the Solaris/Stage/USD context.

Make sure all items that are to be edited are ABOVE the Stage Manager node in the network.

LMB to select items while in the Stage Manager, and upon release of the mouse button, the transform handle will be invoked. (Insert key to move the handle independently of object, then Insert again to proceed.)

You can duplicate (<cntl> D) in addition to transform edits on objects.

You can make Groups by:

  • <shift> LMB to select and
  • <cntl> G to make a group.
  • The items are now grouped inside a Group level in the hierarchy.
  • You can rename the group in the Scene Graph or the sub-scene graph in the Stage Manager panel.

Transforms can be applied on Groups as if they were a single item.

Like all nodes, the Stage Manager is additive, so you can deactivate the node (Q key) to switch off the edits.

You can layer these Stage Manager nodes along the network, for incremental or organized edits.

NOTE: Previewing edits in the Viewport can slow down if you aren't previewing in the actual 'Stage Manager' node. Try previewing the Stage Manager node itself to get more rapid Viewport response.

CAUTION: If you hit <esc> for example, you can lose focus of the Stage Manager node (even thought it's selected), and any edits will spawn an Edit node.

Materials

YouTube - timecode: 00:39:30

Methods of applying materials in Solaris/Stage/USD:

Material Library

The Material Library can be dropped down after the item import or build.

  • Double LLM click to enter the 'Material Library'
  • Drop a Material Builter node (dependent on your renderer):
    • Karma Material Builder
    • Redshift Material Builder
    • Arnold Material Builder
  • There will be a Material X node inside (mtlX node).

By default the Material Path Prefix on the Material Library node will be set to '/materials/'. This will set the hierarchy of the 'Scene Graph' to 'root/materials/<my_material_name>'. This can be edited of course.

Material Library Fields

  • Number of Materials - set number of materials you are utilizing):
    • or Auto Fill Materials:
      • Detects the number of materials in the Library
      • Populates fields with existing materials
      • NOTE: this method populates the Material Path, which might not be what you want
  • Material Path Prefix - sets path in Scene Graph
  • Material VOP - material nodes to actually reference outside (default is * for all).
  • Material Path - this is a custom path that will override the 'Material Path Prefix':
    • Leave empty to just use the 'Material Path Prefix' path
  • Assign to Geometry - checkbox
  • Geometry - set path to geometry to assign to

Material Linker

The Material Linker node is an alternative way (to the 'Number of Materials' fields) to assign materials.

  • Drop after a Material Library node
  • The parameter panel has three panes:
    • Materials
    • Assignment area
    • Scene graph style hierarchy
  • Drag the material into the center
  • Drag items from the right panel onto the center panel material assignment listings.

Exclusion

There are two areas on the center Assignment panel material:

  • Assigned (the material cube icon):
    • Add items of groups
  • Exclude (the 'x' or cross icon):
    • Exclude individual items from an assigned group perhaps

Catalog

The center panel also has a Catalog tab. This contains Houdini pre-made materials, which can be dragged onto the left panel to bring as materials to use in the scene.

You can bring in (download) even more by clicking on the 'cog wheel' in the 'Catalog' tab, and choose:

  • Open AMD MaterialX Library
  • Set download location and 'Accept'

Assign Material

The Assign Material node is another alternative way of assigning materials.

  • Primitives - path to geometry to assign to
  • Specify Material Using - allows for various ways of auto assignment (VEX for example)
  • Material Path - path to material to assign
  • Material Strength - USD override order (if item already has material for example)

Quick Surface Material

The Quick Surface Material node is a rapid way of making a mtlX material - useful for testing and prototyping.

  • Primitive Path - set path and name in the Solaris/Stage/USD hierarchy

Assign the Quick Surface Material

Assign using an Assign Material node (see above):

  • Set to the Primitive Path location of theQuick Surface Material as created.

Component Builder

YouTube - timecode: 00:49:19

The Component Builder node is actually a whole network for authoring content in the Solaris/Stage/USD context.

The 'Component Builder' network is made up of:

  • Component Geometry node
  • Material Library node
  • Component Material node (where the previous two merge)
  • Component Output node:
    • You can write out the result here to be brought in as a reference object if desired.
    • There is a 'Save to Disk' button in there
    • Bring back in with a 'Reference' node

The first node, the Component Geometry is where the mesh objects are created.

The Material Node contains all the material creation. (Perhaps with a Karma Material Builder network within.)

The Component Material node merges and applies the geometry and the materials.

The given name of the last node, the Component Output becomes the name given to the container in the Scene Graph hierarchy. 


`Component Geometry`  `Material Library` 
componentgeometry1  materiallibrary1
            \         /
             \       /
              \     /
               \   /
          `Component Material`
            componentmaterial1
                 |
                 |
                 |
          `Component Output`
            componentoutput1

Component Geometry

The Component Geometry node has several drop-down modes:

  • Internal SOP Network:
    • build Meshes inside the node
  • Imported Files
  • External SOP Network:
    • Import from the '/obj' (SOPs) context
    • Pipe into the Default/Render field.
  • Referenced Files

Inside the 'Component Geometry' node, there are pre-build nodes already. There are three output nodes:

  • default:
    • Required
    • The base mesh
  • proxy:
    • Optional
    • Low resolution version of the mesh
    • Add a Polyreduce node after the original geo perhaps
  • simproxy:
    • Optional
    • Low resolution and often simplified, for simulations (collisions etc.)
    • Perhaps add a Shrinkwrap node to simplify and a Remesh node to clean up the resulting geo.

These outputs reflect in the 'Scene Graph', with the subsections of 'default', 'proxy' and 'simproxy'.

NOTE: You can preview the proxys using a visualization tool on the right Viewport vertical panel on the right. RMB on the HOUDINI GL icon, and select which versions to show ('Render', 'Proxy' and 'Guide' checkboxes).

Material Library

Standard Material Library

Component Material

With the Component Material, you can choose meshes in the Primitives field and Material Path field for the materials. By default, it pulls in all meshes and all materials:

  • 'lopinputprims('.', 0)'
  • 'lopinputprim('.',0)'

Add a Stage Manager

To the end of the Component Builder network, you could attach a Scene Manager node.

FUN STUFF: You can switch on Use Physics (top of viewport panel) and:

  • Activate Physics
  • Per object, click on Add Physics:
    • NOTE: If you duplicate an object, you have to reapply 'Add Physics'
  • Moving objects will use collision and will result in interaction feedback between components.
  • You can turn on Display Proxies to see what geometries are defining the collision.

Component Geometry Variants

If you create multiple Component Geometry setups, you can join the with a Component Geometry Variants node to enable you to select a variant with the Set Variant node (see below).

Set Variant

NOTE: you can use a Stage Manager to achieve the results of the Set Variant node too (see below)

Variants are set up with the Component Geometry Variants node (above) for geometry, and with multiple Component Material nodes (in series) for materials.

The Set Variant node can be added to the end of the network (after the 'Component Output' node for example).

  • Variant Set field to choose which aspect to set variant on (from drop-down at end of field or explicitly entered) - example:
    • geo
    • mtl
  • Variant Name field to specify which variant to use:
    • Use the drop-down menu to select options or
    • Type name in the field
    • The name is the given name of:
      • the Component Geometry node for geo (geometry)
      • the Component Material node for mtl (material)

You can use multiple Set Variant nodes in series to do all the variant selections (for geo and mtl for example).

If you saved the Component Output node result out to a reference USD item, you can bring in using a Reference node, and put the Set Variant node(s) after this to use the same functionality of variant selection.

Stage Manager to select Variants

You can attach a Stage Manager node after the Component Builder setup (or a read/reference node).

  • RMB click on a selected asset/object
  • Go to geo or mtl and choose a variant from the radio button list.

Light Type & Light Mixer

Karma Physical Sky

The Karma Physical Sky node creates an environmental light, using the components of:

  • Sun:
    • Control scale of the sun with Angular Size
    • Angle in elevation with Solar Altitude
    • Direction around scene with Solar Azimuth
    • You can choose whether to render the sun disc if visible in shot
  • Sky:
    • The Sky tab has numerous controls for:
      • Color
      • Appearance of phenomena
  • Atmospheric:
    • You can also switch from sky to Atmospheric
    • This gives volumetric scattering that can be controlled with numerous dials:
      • Scatter
      • Absorption (TIL It's not spelled 'Absorbtion')
    • different scatter sections:
      • Rayleigh
      • Mie
      • Ozone
    • Ground

Dome Light

The Dome Light allows you to use HDRI maps to light the scene.

  • Base Properties:
    • Load HDRI into Texture field.
    • Adjust Intensity or Exposure
  • Transform Tab:
    • Rotate etc.

NOTE: To control the display of the HDRI as a background, open the Viewport Display options (D key):

  • Background:
    • Display Environmental Lights as Backgrounds (checkbox)
    • Disabling will control render status too.

Light

By default, a new (simple) Light is dropped as a Point light (with sharp shadows from an infinitely small area).

Multiple lights can be combined with a Merge node in the same way as other components.

There are a selection of light types to choose from:

  • Cylinder
  • Distant
  • Disk
  • Point
  • Rectangle (area light)
  • Sphere (orb light)

Shadows are affected by the light type selection. A Sphere type will act like an 'orb' of light, and the shadows will get softer as the orb get's larger.

Scaling a Sphere light will affect the intensity by default, but to prevent this variation with scaling, you can activate the Normalize Power checkbox.

The Karma tab has grayed out controls (which when grayed out won't get saved to the USD description - hence saving space. Activate them to use them. To activate:

  • Click on icon next to parameter: on the drop-down:
    • Set or Create
  • Deactivate with:
    • Do Not Use (on drop-down)

Spread is an example of a parameter that is grayed out. This will adjust the focus of rays.

Contribute is useful for setting contribution to different AOV type light components:

  • Any
  • Diffuse
  • Reflection
  • Coaat
  • Volume
  • etc.

This can be used to turn off volume lighting for a light for example. RegEx is used here:

  • asterix for all
  • Carot (^) to negate
  • Example:
    • "* ^volume" will illuminate all components EXCEPT volumetrics.

Spotlight

To activate the Spotlight (on the Rectangle type for example) you have to go to the light's Shaping tab, and activate the 'Spotlight' checkbox. This opens controls for adjusting the angle, softness and the barn-doors.

Placement mode

When a light is selected, there will be a bar across the top of the Viewport, allowing you to options.

  • Set to Diffuse:
    • Click on items in your scene - the light will jump to where it would illuminate most strongly at the point on the item where clicked. (Aligned to the normal of the point on the surface clicked.)
  • Set to Shadow:
    • Click to set the light position
    • Click again in an area where you'd like the original light position to cast shadows to.
  • Various shortcuts:
    • <cntl> LMB drag to adjust the distance of the light from subject.
    • <cntl><shift> LMB drag to adjust the light intensity
    • B key to toggle through light size adjust mode (LMB drag to adjust)

Light Mixer

The Light Mixer node can be dropped after the lights have been created and incorporated into a network.

In the Light Mixer parameter panel, you can drag existing lights from the left to the center area, where you'll see the attributes to adjust. There are three tabs:

  • Attributes:
    • Adjust parameters
    • Columns include the same controls as the sliders below
  • Sliders:
    • A fader desk style control hub for all lights
    • Selecting lights in the fader bank still allows the 'Light' controls in the Viewport.
    • The left slider is Intensity
    • The right slider is Exposure
    • The Star is Solo
    • The strike-through icon is Mute for the light

Light Linker

The Light Linkder node controls which lights affect which items or surfaces.

The layout is similar to the Material Linker node as described earlier in these notes.

  • Drag lights from the left pane to the center
  • Drag items from the right onto the light item in the center panel:
    • Sun icon of the light to include
    • Moon icon of the light to exclude
  • It seems that this applies to primary rays only (LOOK INTO THIS!)

Volumetric Lights & Emissive Lights

YouTube - timecode: 01:19:06

Volumetric Lighting

To achieve Volumetric Lighting, you need to create a volume for the lights to illuminate.

For this, you can use a Karma Fog Box node.

  • Scale to encompass the whole scene
  • TIP: Turn off item outline on the icons to the right of the Viewport.

Emissive Material

Create a mtlX with Emission turned on and up.

  • Material Builder
  • Set MaterialX node:
    • Emission: to a value greater than zero (maybe 10 if you like)
  • Assign with Auto-fill Material button (or set in fields)
  • Click Assign to Geometry

Objects with this material will now illuminate the scene.

NOTE: The viewport will use the headlight mode if there is only an emission material illuminating the scene. Add any ordinary light of any intensity to kick the viewport into previewing a more accurate lighting scenario with emission.

Render Geometry Settings

The Render Geometry Settings node is useful for controlling Karma render settings per object.

In the case of an Emission Material on an object for example:

  • you can add a Render Geometry Settings node after the item
  • activate the Light > Treat As Light Source attribute:
  • switch to Yes.
  • This will speed up the rendering of the object (treating it as a light instead of full surface geometry).
  • This will also result in less noise

This can be used to set any render settings for a particular item/object for optimal results.

Instancer

YouTube - timecode: 01:28:56

The Instancer node is similar to the copy to points node in SOPs. You can easily scatter millions of polygons across your scene.

The two inputs for the 'Instancer' node are different to the 'copy to points' node.

  • Left input - to plug into scene network:
    • You don't actually need anything plugged into the left input
  • Right input - to plug in geometry to be 'instanced'

Inside Instancer

You can create points inside, maybe with an add node, and set the number of points etc.

The objects are instanced in-place of these points.

Randomize Attributes

The same operations and nodes as those with 'copy to points' can be used inside the Instancer.

The Randomize Attributes allows you to set randomize any attribute or var:

  • P
  • Cd
  • pscale
  • N
  • etc, etc.

You can layer a number of Randomize Attributes nodes to build the scene items.

There are ton of controls on this node's parameter panel. For example the Distribution drop-down has:

  • Constant Value (not random)
  • Two Values
  • Uniform (Continuous)
  • Direction or Orientation
  • Inside Sphere
  • Normal (Gaussian)
  • Exponential
  • Log-Normal
  • Cauchy-Lorentz
  • Custom Ramp
  • Custom Discrete

Using with a Merge

You can use a Merge node to merge multiple geometries before you pipe into the Instancer, and this allows you to use some in-built controls on the Instancer parameter panel to control what get's used.

It's a good idea to keep your USD paths in shape for each geometry too, so that the everything makes sense in the Scene Graph views.

  • Prototype Primitives:
    • Add a string to reflect the input geos:
      • example: '/assets/geo/*'
  • Prototype Index:
    • Random:
      • get random geometry show up from all the pre-merge candidates
    • Index:
      • where the index refers to the input (so input zero is the first item in the merge)
    • Index Attribute
    • Name Attribute
    • Path Attribute

Using Instancer with a Component Builder

Instead of using the Instancer with just a merge and input geometries, you can use it with the Component Builder networks to have a more powerful implementation.

  • Attach after a Component Builder network (after a Component Output node).
  • Or
  • Attach to a reference node where you've saved and read in a Component Builder network.

There are a number of caveats to circumvent to get the system to work:

  • Add an Explore Variants node:
    • Mode --> 'Explore Variants'
    • To use with Instancer or Copy to Points set 'spacing' to zero
    • This node will create an item in the USD hierarchy:
      • explorevariants1:
        • item will be 'green' indicating it needs to be 'flattened' (see step below)
      • within will be items reflecting each 'variant'
    • Update the 'Prototype Primitives' path:
      • /explorevariants1/<any other levels>/*
  • Add a Configure Layer node:
    • This is to 'flatten' the explorevariants1 level:
      • Flatten Input drop-down:
        • 'Flatten Input Stage'
  • This will enable the whole Component Builder or reference network to work with the Instancer 
    
    Reference (or Component Builder network)
         |
         |
    Explore Variants (set to zero 'Spacing', and update paths)
         |
         |
    Configure Layer (set to 'Flatten Input')
         |
         |
    |    |
  Instancer

Good Workflow

Pin the Viewport at the outer Stage level and jump into the Instancer to work.
(This allows you to see the 'results' while tinkering with the setup inside.)
