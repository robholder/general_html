Challenge 1

I tried to mimic the CupcakeCorner project solution to block the save button if fields are empty. I added a hasValidEntries in the Book.swift data.

Ai Insight

The Problem

Your hasValidEntries property lives inside the Book class. To check that property, you would first have to create a newBook instance. But you don't want to create the book until the user presses "Save"—which is exactly what you're trying to disable!

In SwiftUI, if the data lives in the View's @State, the validation logic should generally live there too.

Strategy 1: The Computed Property (Recommended)

The cleanest way to do this is to add a computed property directly to your AddBookView. This property will look at your @State variables and decide if the form is ready.

Add this inside your AddBookView (before the body): 

var isFormInvalid: Bool {
    title.trimmingCharacters(in: .whitespaces).isEmpty || 
    author.trimmingCharacters(in: .whitespaces).isEmpty || 
    review.trimmingCharacters(in: .whitespaces).isEmpty
}

Then, simply attach the .disabled() modifier to your button: 

Button("Save") {
    let newBook = Book(title: title, author: author, genre: genre, review: review, rating: rating)
    modelContext.insert(newBook)
    dismiss()
}
.disabled(isFormInvalid) // Here is the magic

Strategy 2: The "Validation" Logic in the Model

If you really want to keep the validation logic inside the Book class (perhaps because you want to use it in multiple places), you can make it a static function. This allows you to call it without needing an actual book instance first.

In your Book class: 

static func validate(title: String, author: String, review: String) -> Bool {
    if title.trimmingCharacters(in: .whitespaces).isEmpty || 
       author.trimmingCharacters(in: .whitespaces).isEmpty || 
       review.trimmingCharacters(in: .whitespaces).isEmpty {
        return false
    }
    return true
}

In your View: 

.disabled(Book.validate(title: title, author: author, review: review) == false)

Which one should you use?

Go with Strategy 1.

Since AddBookView "owns" the temporary strings (title, author, etc.) before they become a permanent book, it's the View's job to decide if its own text fields are satisfactory.

One quick tip: Notice I added .trimmingCharacters(in: .whitespaces). This prevents users from "cheating" your validation by just typing a single space!

In Context

import SwiftData
import SwiftUI


struct AddBookView: View {
    @Environment(\.modelContext) var modelContext
    @Environment(\.dismiss) var dismiss
    
    let genres = ["Fantasy", "Horror", "Kids", "Mystery", "Poetry", "Romance", "Thriller"]
    
    @State private var title = ""
    @State private var author = ""
    @State private var rating = 3
    @State private var genre = "Fantasy"
    @State private var review = ""
    
    // Computed property
    var isBookValid: Bool {
        title.trimmingCharacters(in: .whitespacesAndNewlines).isEmpty ||
        author.trimmingCharacters(in: .whitespacesAndNewlines).isEmpty ||
        review.trimmingCharacters(in: .whitespacesAndNewlines).isEmpty
    }
    
    var body: some View {
        NavigationStack {
            Form {
                
                Section {
                    TextField("Name of book", text: $title)
                    TextField("Author's name", text: $author)
                    
                    Picker("Genre", selection: $genre) {
                        ForEach(genres, id: \.self) {
                            Text($0)
                        }
                    }
                }
                
                Section("Write a review") {
                    TextEditor(text: $review)
                    RatingView(rating: $rating)
                }
                
                Section {
                    Button("Save") {
                        // add the book
                        let newBook = Book(title: title, author: author, genre: genre, review: review, rating: rating)
                        modelContext.insert(newBook)
                        dismiss()
                    }
                    // Check if fields are completed
                    .disabled(isBookValid)
                    
                }
            }
            .navigationTitle("Add Book")
        }
    }
}

import SwiftData
import SwiftUI


struct ContentView: View {
    @Environment(\.modelContext) var modelContext
    @Query(sort: [
        SortDescriptor(\Book.title),
        SortDescriptor(\Book.author)
    ]) var books: [Book]
    
    @State private var showingAddScreen = false
    
    var body: some View {
        NavigationStack {
            List {
                ForEach(books) { book in
                    NavigationLink(value: book) {
                        HStack {
                            EmojiRatingView(rating: book.rating)
                                .font(.largeTitle)
                            
                            VStack(alignment: .leading) {
                                Text(book.title)
                                    .font(.headline)
                                    // Color book title red if bad rating
                                    .foregroundStyle(book.rating < 2 ? .red : .primary)
                                Text(book.author)
                                    .foregroundStyle(.secondary)
                            }
                        }
                    }
                }
                // Add delete functinoality (call func)
                .onDelete(perform: deleteBooks)
            }
            .navigationTitle("Bookworm")
            .navigationDestination(for: Book.self) { book in
                DetailView(book: book)
            }
            .toolbar {
                // Add an edit button for the book list
                ToolbarItem(placement: .topBarLeading) {
                    EditButton()
                }
                
                ToolbarItem(placement: .topBarTrailing) {
                    Button("Add Book", systemImage: "plus") {
                        showingAddScreen.toggle()
                    }
                }
            }
            .sheet(isPresented: $showingAddScreen) {
                AddBookView()
            }
        }
    }
    
    func deleteBooks(at offsets: IndexSet) {
        for offset in offsets {
            // find this book in our query
            let book = books[offset]
            
            // delete it from the context
            modelContext.delete(book)
        }
    }
}

Challenge 3 - Add Date (Now)

QUESTION

I want to return a string representation of the Date in this function. How do I do that? 

func getDate() -> Date {
    let dateNow = Date.now
        
    return dateNow
}

ANSWER

Here is how you would write that function to return a nicely formatted string: 

func getDate() -> String {
    let dateNow = Date.now
    
    // This turns the Date into a String based on your requirements
    return dateNow.formatted(date: .abbreviated, time: .omitted)
}

Breaking Down the Formatting Options

Swift provides a few "presets" that are much easier than manual math. If you want specific control like you mentioned (day, month, year), you use the dateTime configuration:

  • For "Oct 12, 2026":
    • Date.now.formatted(date: .abbreviated, time: .omitted)
  • For "10/12/2026":
    • Date.now.formatted(date: .numeric, time: .omitted)
  • For custom control (your original idea):
    • Date.now.formatted(.dateTime.day().month().year())

Use it in your View

Since you are likely using this in a SwiftUI Text view, you don't actually need a function at all! SwiftUI can format dates directly inside a string interpolation: 

Text("Added on \(Date.now, format: .dateTime.day().month().year())")

Pro Tip for Bookworm

If you are adding a "Date Added" feature to your Book model, remember to:

  • Add var dateAdded: Date = Date.now to your @Model class.
  • In your DetailView, use the .formatted() logic above to show the user when they finished the book.

While it’s tempting to save the date as a String (because that's how we want to see it), you almost always want to save it as a Date object in your @Model.

Here is why Date and not String, and how to coordinate it.

1. The "Why": Data vs. Presentation

Think of a Date object as the "truth" and a String as just one "translation" of that truth.

  • Sorting: If you save dates as Strings (like "January 5" and "December 1"), a database will sort them alphabetically. "December" comes before "January," which is a mess! If you save them as Date, SwiftData knows exactly which one happened first in time.
  • Localization: A user in the US sees 10/12/26 (Oct 12), while a user in the UK sees 12/10/26 (Dec 10). If you save a String, you force one format on everyone. If you save a Date, the iPhone automatically translates it to the user's preferred local format.
  • Math: If you want to show "Added 3 days ago," you can do math with Date objects. You can't do math with the word "Monday."

2. How to coordinate the property

You should store the "Raw Data" in the Model and do the "Formatting" in the View.

Step A: Update the Model

Add a Date property to your Book class. Give it a default value of Date.now so it automatically stamps the book the moment it's created. 

@Model
class Book {
    var title: String
    var author: String
    var dateAdded: Date // Store the 'Truth'
    
    init(title: String, author: String, dateAdded: Date = .now) {
        self.title = title
        self.author = author
        self.dateAdded = Date.now
    }
}

Step B: Use it in the View

When you want to display the date in your DetailView, that is when you convert it to a String using the .formatted() method we discussed. 

Section("Book Details") {
    Text("Added on \(book.dateAdded.formatted(date: .long, time: .omitted))")
}

The Architecture Workflow

This is the "Golden Rule" of handling dates in apps:

  • Storage: Always Date.
  • Logic/Sorting: Always Date.
  • Display: Convert to String at the very last second in the body of your SwiftUI view.

A Wittier Perspective

Storing a date as a String is like taking a photo of a clock and saving the photo. You know what time it was then, but the photo can’t tell you how many seconds have passed since, and it’s hard to compare two photos to see which was taken first. Storing the Date object is like having the actual clock—it’s much more useful!

Tailored To Bookworm

Since the Book model already handles the "creation" of the date automatically (if you set a default value), coordinating it across your views is actually much simpler than you might expect.

You don't need to create a @State for the date in AddBookView. Instead, you let the Model stamp the date when the book is born, and then you just display it in the DetailView.

Step 1: Update your Model

First, ensure your Book class is ready to store the date. By setting the default to .now in the initializer, you don't even have to pass a date when you create a book—it just "knows" what time it is. 

@Model
class Book {
    var title: String
    var author: String
    var genre: String
    var review: String
    var rating: Int
    var dateAdded: Date // The new property

    init(title: String, author: String, genre: String, review: String, rating: Int, dateAdded: Date = .now) {
        self.title = title
        self.author = author
        self.genre = genre
        self.review = review
        self.rating = rating
        self.dateAdded = Date.now
    }
}

Step 2: The "Save" Logic in AddBookView

In your AddBookView, you don't need a text field for the date. When the user hits "Save," the Book initializer handles the timestamp for you. 

Button("Save") {
    // Note: We don't need to pass dateAdded here because it defaults to .now
    let newBook = Book(title: title, author: author, genre: genre, review: review, rating: rating)
    
    modelContext.insert(newBook)
    dismiss()
}

Step 3: Displaying in DetailView

Now that the date is saved in the database, you can display it anywhere in your DetailView. A nice place for it is usually right under the author's name or in a new section. 

// Inside DetailView.swift body:

Text(book.author)
    .font(.title)
    .foregroundStyle(.secondary)

// Add the formatted date here:
Text("Added on \(book.dateAdded.formatted(date: .long, time: .shortened))")
    .font(.caption)
    .foregroundStyle(.secondary)

Why this "Coordination" works so well:

  • Single Source of Truth: The dateAdded lives inside the book object. Because your DetailView has let book: Book, it has access to everything about that book, including the date it was created.
  • No Manual Entry: Since you likely want to know when the book was added to the app, having the Model do it automatically is safer than asking the user to type it in.
  • Formatting is local: By using .formatted(), your app will automatically handle different time zones and calendar formats (like Day/Month vs Month/Day) without you writing any extra code.

A "Pro" DetailView Layout Tip

If you want it to look a bit more "official," you can wrap the date in a Section or a VStack with specific alignment: 

VStack(alignment: .leading) {
    Text(book.author)
        .font(.title)
    
    Text(book.dateAdded.formatted(date: .abbreviated, time: .omitted))
        .font(.subheadline)
        .foregroundStyle(.secondary)
}

Challenge 4

Ai Insight - Add Option To Edit Existing Book

This is a perfect evolution of your app. In SwiftUI, the most common way to handle "Editing" is to reuse your AddBookView but pass it an existing book to fill in the fields, rather than starting with empty strings.

However, there is a small architectural catch: AddBookView is currently designed to insert a new book. To make it an "Edit View," we need to tweak it slightly so it knows whether it's creating something new or updating something old.

1. Update AddBookView to handle Editing

We need to tell the view: "If I give you a book, use its data. If I don't, start fresh." 

struct AddBookView: View {
    @Environment(\.modelContext) var modelContext
    @Environment(\.dismiss) var dismiss
    
    // 1. Add an optional book property
    var bookToEdit: Book? 

    @State private var title = ""
    @State private var author = ""
    @State private var rating = 3
    @State private var genre = "Fantasy"
    @State private var review = ""

    var body: some View {
        NavigationStack {
            Form {
                // ... your existing textfields/pickers ...
            }
            .navigationTitle(bookToEdit == nil ? "Add Book" : "Edit Book")
            .toolbar {
                Button("Save") {
                    if let book = bookToEdit {
                        // 2. Update existing book
                        book.title = title
                        book.author = author
                        book.genre = genre
                        book.review = review
                        book.rating = rating
                    } else {
                        // 3. Create new book (your existing logic)
                        let newBook = Book(title: title, author: author, genre: genre, review: review, rating: rating)
                        modelContext.insert(newBook)
                    }
                    dismiss()
                }
                .disabled(isBookValid)
            }
            // 4. Fill the fields if we are editing
            .onAppear {
                if let book = bookToEdit {
                    title = book.title
                    author = book.author
                    rating = book.rating
                    genre = book.genre
                    review = book.review
                }
            }
        }
    }
}

2. Add the Edit Button to DetailView

Now, in your DetailView, you need a state variable to show the sheet and a button to trigger it. 

struct DetailView: View {
    let book: Book
    @State private var showingEditSheet = false // Toggle for the sheet

    var body: some View {
        ScrollView {
            // ... your existing layout ...
        }
        .toolbar {
            // Add the Edit button next to the Trash button
            Button("Edit") {
                showingEditSheet = true
            }
            
            Button("Delete this book", systemImage: "trash") {
                showingDeleteAlert = true
            }
        }
        // Show the AddBookView as an editor
        .sheet(isPresented: $showingEditSheet) {
            AddBookView(bookToEdit: book)
        }
    }
}

Why this works

  • The Sheet: By passing bookToEdit: book, the AddBookView initializes its @State properties with the existing book's data inside .onAppear.
  • SwiftData Magic: Because Book is a class marked with @Model, when you change book.title = title, SwiftData automatically watches for that change and saves it to the database. You don't even need to call a "save" or "update" function on the context; simply modifying the properties of the object is enough!

One Little Polish Tip

In your AddBookView's isBookValid logic, remember that it currently checks if the fields are empty to disable the button.

If you want to be extra fancy, you could change the button text based on the mode: 

Button(bookToEdit == nil ? "Add" : "Update") { ... }

Where to place:

You'll want to swap out the static "Save" string for the ternary operator inside the Button label. You can also apply this same logic to the .navigationTitle so the top of the screen says "Edit Book" when appropriate.

Here is where those changes go: 

// ... inside body -> NavigationStack -> Form
Section {
    // Check if bookToEdit is nil to decide the button text
    Button(bookToEdit == nil ? "Save Book" : "Update Details") {
        if let book = bookToEdit {
            book.title = title
            book.author = author
            book.genre = genre
            book.review = review
            book.rating = rating
        } else {
            let newBook = Book(title: title, author: author, genre: genre, review: review, rating: rating)
            modelContext.insert(newBook)
        }
        dismiss()
    }
    .disabled(isBookValid)
}

// ... then further down, change the Navigation Title too:
.navigationTitle(bookToEdit == nil ? "Add Book" : "Edit Book")
.navigationBarTitleDisplayMode(.inline)

Two Small Refinements

  • The .onAppear Placement: In your current code, you have .onAppear attached to a Section. While that works, it’s safer to attach it to the Form or the NavigationStack. If a Section is off-screen (scrolled away), sometimes .onAppear behaves inconsistently.
  • NavigationStack: If you are calling AddBookView from a .sheet in your DetailView, the DetailView is likely already inside a NavigationStack. If you include another NavigationStack inside the sheet, you’ll end up with "Double Nav Bars" (two large titles stacked on top of each other).

The fix: Usually, you remove the NavigationStack from AddBookView and just keep the Form. Then, wrap the call inside the .sheet in the DetailView with a NavigationStack instead.

Final Check on DetailView

When you trigger this from your DetailView, ensure it looks like this: 

.sheet(isPresented: $showingEditSheet) {
    NavigationStack {
        AddBookView(bookToEdit: book)
    }
}

This way, AddBookView stays lightweight and flexible—it works as a standalone "Add" screen or a "Pop-up" edit screen!
