Hacking With Swift - 231 - Scheduling local notifications

UserNotifications

iOS has a framework called UserNotifications that does pretty much exactly what you expect: lets us create notifications to the user that can be shown on the lock screen.

We have two types of notifications to work with, and they differ depending on where they were created:

  • Local notifications - scheduled locally
  • Remote notifications (commonly called push notifications) - sent from a server somewhere.

Remote notifications require a server to work, because you send your message to Apple’s push notification service (APNS), which then forwards it to users.

But local notifications are nice and easy in comparison, because we can send any message at any time as long as the user allows it.

To try this out, start by adding an extra import near the top of ContentView.swift: 

import UserNotifications

Next we’re going to put in some basic structure that we’ll fill in with local notifications code.

Using local notifications requires:

  • Asking the user for permission, then
  • Registering the notification we want to show.

We’ll place each of those actions into separate buttons inside a VStack, so please put this inside your ContentView struct now: 

VStack {
    Button("Request Permission") {
        // first
    }

    Button("Schedule Notification") {
        // second
    }
}

Request Authorization

OK, that’s our setup complete so let’s turn our focus to the first of two important pieces of work: requesting authorization to show alerts.

Notifications can take a variety of forms, but the most common thing to do is ask for permission to show alerts, badges, and sounds. That doesn’t mean we need to use all of them at the same time, but asking permission up front means we can be selective later on.

When we tell iOS what kinds of notifications we want:

  • It will show a prompt to the user - so they have the final say on what our app can do.
  • When they make their choice, a closure we provide will get called and tell us whether the request was successful or not.

So, replace the // first comment with this: 

UNUserNotificationCenter.current().requestAuthorization(options: [.alert, .badge, .sound]) { success, error in
    if success {
        print("All set!")
    } else if let error {
        print(error.localizedDescription)
    }
}

If the user grants permission, then we’re all clear to start scheduling notifications. Even though notifications might seem simple, Apple breaks them down into three parts to give it maximum flexibility:

  • The content is what should be shown, and can be a title, subtitle, sound, image, and so on.
  • The trigger determines when the notification should be shown, and can be a number of seconds from now, a date and time in the future, or a location.
  • The request combines the content and trigger, but also adds a unique identifier so you can edit or remove specific alerts later on. If you don’t want to edit or remove stuff, use UUID().uuidString to get a random identifier.

When you’re just learning notifications the easiest trigger type to use is UNTimeIntervalNotificationTrigger, which lets us request a notification to be shown in a certain number of seconds from now. So, replace the // second comment with this: 

let content = UNMutableNotificationContent()
content.title = "Feed the cat"
content.subtitle = "It looks hungry"
content.sound = UNNotificationSound.default

// show this notification five seconds from now
let trigger = UNTimeIntervalNotificationTrigger(timeInterval: 5, repeats: false)

// choose a random identifier
let request = UNNotificationRequest(identifier: UUID().uuidString, content: content, trigger: trigger)

// add our notification request
UNUserNotificationCenter.current().add(request)

If you run your app now, press the first button to request notification permission, then press the second to add an actual notification.

Now for the important part: once your notification has been added press Cmd+L in the simulator to lock the screen. After a few seconds have passed the device should wake up with a sound, and show our message - nice!

Ai Insight

It is completely normal to find UserNotifications overwhelming at first! SwiftUI is highly declarative ("make a button look like this"), whereas iOS's notification system relies on an older, imperative framework. You are essentially shifting gears into telling iOS to handle a multi-step background task.

To make sense of it, you only need to understand two core concepts:

  • asking for a hall pass (Permissions) and
  • building a package to ship later (Scheduling).

Here is the high-level flow of what your code is doing under the hood, followed by the fully commented version to tie it all together.

The High-Level Flow

[ Your App ]
     │
     ├── Step 1: Request Permission ──> (System shows popup to User)
     │                                        │
     │                            If approved, you can now send notifications!
     │
     └── Step 2: Schedule Notification
               │
               ├── 1. Content  (What does it say? "Feed the cat")
               ├── 2. Trigger  (When does it fire? 5 seconds from now)
               └── 3. Request  (Pack Content + Trigger together with a unique ID)
                       │
                       └──> Sent to UNUserNotificationCenter (iOS takes over)

1. Requesting Permission

Before your app can send any alerts, sounds, or badges, Apple forces you to ask the user explicitly.

  • Your first button speaks to UNUserNotificationCenter.current() - this is the iOS background manager in charge of all notifications.
  • It throws up the system alert. If the user clicks "Allow", your app receives a success flag, unlocking the ability to schedule alerts.

2. Scheduling the Notification

Think of scheduling a notification like mailing a package to the future. You have to build three components:

  • The Content (UNMutableNotificationContent): The physical box and what's inside it (Title, subtitle, sound).
  • The Trigger (UNTimeIntervalNotificationTrigger): The delivery date. In your case, a countdown timer set to 5 seconds.
  • The Request (UNNotificationRequest): The shipping label. You take your Content, your Trigger, and slap a unique tracking number (UUID) on it so iOS can keep track of it.

Finally, you hand that completed request over to UNUserNotificationCenter.current().add(request). At that exact moment, iOS takes the package. Even if the user closes your app completely, iOS will count down the 5 seconds and fire the alert.

Your Code, Fully Commented

Here is your exact code with inline explanations pulling all of these concepts together: 

import SwiftUI
// Import the Apple framework responsible for handling local and push notifications
import UserNotifications

struct ContentView: View {
    @State private var backgroundColor = Color.red
    
    var body: some View {
        VStack {
            // MARK: - BUTTON 1: ASK FOR PERMISSION
            Button("Request Permission") {
                // Fetch the shared iOS Notification Manager and ask the user for permission
                UNUserNotificationCenter.current().requestAuthorization(
                    options: [.alert, .badge, .sound] // What types of notifications we want to send
                ) { success, error in
                    // This closure (completion handler) runs AFTER the user makes a choice
                    if success {
                        print("All set! User allowed notifications.")
                    } else if let error {
                        print("Permission denied or error occurred: \(error.localizedDescription)")
                    }
                }
            }
            
            // MARK: - BUTTON 2: SCHEDULE THE ALERT
            Button("Schedule Notification") {
                // 1. Create the PACKAGE (Content)
                // We use "Mutable" because we want to modify its properties (title, subtitle, etc.)
                let content = UNMutableNotificationContent()
                content.title = "Feed the cat"
                content.subtitle = "It looks hungry"
                content.sound = UNNotificationSound.default // Use the standard iOS notification ding
                
                // 2. Create the TIMING (Trigger)
                // Tell iOS to wait 5 seconds. 'repeats: false' means it only happens once.
                let trigger = UNTimeIntervalNotificationTrigger(
                    timeInterval: 5,
                    repeats: false
                )
                
                // 3. Create the SHIPPING LABEL (Request)
                // We combine our Content and Trigger together.
                // We give it a random UUID string as a unique ID so we could cancel it later if we wanted to.
                let request = UNNotificationRequest(
                    identifier: UUID().uuidString, 
                    content: content, 
                    trigger: trigger
                )
                
                // 4. HAND IT OVER TO THE SYSTEM
                // We pass our completed request to iOS. The system takes over counting down the 5 seconds.
                UNUserNotificationCenter.current().add(request)
            }
        }
    }
}

By default, iOS will not display a banner notification if the app is already open and running in the foreground!

Ai Insight - UUID -> String

Why do we need the .uuidString method?

In short, UUID().uuidString generates a guaranteed unique, random 36-character tracking number for that specific notification.

Here is exactly how to break that tiny piece of code down into two parts:

1. UUID()

This creates a brand new Universally Unique Identifier. It is an industry-standard cryptographic random number generator. Every single time you call UUID(), Apple’s system generates a sequence that is so mathematically random, the chances of it ever generating the same number twice anywhere on earth is effectively zero.

2. .uuidString

The UUID() function actually spits out a complex data object. Adding .uuidString simply translates that object into a standard, readable Swift text string (String) that looks something like this:

  • "E621E1F8-C36C-495A-93FC-0C247A3E6E5F"

Why does iOS need this? (The Tracking Number)

iOS requires every notification request to have a unique identifier string. Think of it like a package tracking number.

Because you used a random UUID, if a user mashes your "Schedule Notification" button 5 times in a row, iOS will successfully queue up 5 separate notifications because they all have different tracking numbers.

When would you NOT use a random UUID?

You only use UUID().uuidString if you want a "fire and forget" notification. If you want to be able to update or cancel a notification later, you should use a fixed, custom string instead.

  • Updating a notification: If you send a notification with the identifier "daily-reminder", and then send another one 2 minutes later with the exact same identifier "daily-reminder", iOS will instantly overwrite the old one.
  • Canceling a notification: If the user feeds the cat early, you can tell iOS: 'Hey, delete the pending notification with the ID "daily-reminder".' If you used a random UUID().uuidString, you wouldn't easily know what tracking number to tell iOS to look for!

UUID Format

A UUID (Universally Unique Identifier) follows a very strict, standardized format known as UUID Version 4 (which means it is generated using completely random numbers).

Here is exactly how it breaks down:

The Formation

A UUID string is always 36 characters long in total. It consists of 32 hexadecimal characters (the numbers 0-9 and lowercase letters a-f) separated by 4 hyphens grouped into 5 distinct blocks.

The exact layout always follows a 8-4-4-4-12 character formation:

XXXXXXXX-XXXX-XXXX-XXXX-XXXXXXXXXXXX

An actual example generated by UUID().uuidString looks like this:

6ec0bd7f-11c0-43da-975e-2a8ad9ebae0b

What do the blocks mean?

While a Version 4 UUID is almost entirely random, a few specific bits are reserved to tell the system what kind of UUID it is:

  • Time-Low (8 characters): 6ec0bd7f
  • Time-Mid (4 characters): 11c0
  • Time-High & Version (4 characters): 43da (In a random UUID, this block always starts with a 4 to indicate it is a Version 4 UUID).
  • Clock-Skew & Variant (4 characters): 975e (This block always starts with 8, 9, a, or b to indicate the variant standard being used).
  • Node ID (12 characters): 2a8ad9ebae0b

Why this specific size?

With 32 hexadecimal characters, there are 2**128 (or about 340 undecillion) possible combinations.

To put that in perspective for your app: if you generated 1 billion UUIDs every single second for the next 100 years, the chances of your app accidentally creating two identical strings as notification IDs is lower than the chance of a meteor hitting your computer at this exact moment. It is safe to use as a completely unique identifier!
