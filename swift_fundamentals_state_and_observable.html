Swift - @Observable & @State

The Core Concept: Value Types vs. Reference Types

To understand how state works in modern SwiftUI, you must first ask: What kind of data am I tracking?

  • @State (on its own): Used for Value Types (structs, strings, integers, arrays, enums). The data lives directly inside the view.
  • @Observable + @State: Used for Reference Types (classes). The data lives in a separate shared object, and the view watches it for changes.

1. Local Value State: @State (with Structs/Primitives)

When your data is simple and belongs entirely to a single view, use a struct or a primitive type combined with the @State property wrapper.

The Rule

@State tells SwiftUI to manage the storage for this data. When the value changes, SwiftUI automatically redraws any view reading that data.

Code Blueprint

import SwiftUI

struct CounterView: View {
    // Storing a simple value type directly inside the view
    @State private var count = 0
    @State private var userName = ""

    var body: some View {
        VStack(spacing: 20) {
            Text("\(userName)'s Count: \(count)")
            
            Button("Increment") {
                count += 1 // Directly modifying triggers a redraw
            }
            
            // The '$' creates a two-way binding so the Textfield can write back to 'userName'
            TextField("Enter name", text: $userName)
                .textFieldStyle(.roundedBorder)
        }
        .padding()
    }
}

2. Object Observation: @Observable + @State (with Classes)

When your data is complex, manages business logic, or needs to be shared across multiple views, use a Class marked with the @Observable macro. To instantiate it inside a view, you pair it with @State.

The Rule

The @Observable macro tracks property reads and writes automatically. You no longer need to type @Published on properties or conform to ObservableObject. When a view reads a property of an @Observable class, SwiftUI takes note and redraws only if that exact property changes.

Code Blueprint

import SwiftUI
import Observation // Implicitly imported by SwiftUI, but good to know

// 1. Define your data model or model facade using the macro
@Observable
class UserProfile {
    var name: String = "Taylor"
    var age: Int = 26
    var status: String = "Active"
    
    func celebrateBirthday() {
        age += 1 // Changes to 'age' will only redraw views reading 'age'
    }
}

struct ProfileView: View {
    // 2. Instantiate the reference type using @State to keep it alive
    @State private var profile = UserProfile()

    var body: some View {
        VStack(spacing: 16) {
            // Reading 'name' and 'age' registers this view for updates
            Text("Name: \(profile.name)")
            Text("Age: \(profile.age)") 
            
            Button("Birthday Bash") {
                profile.celebrateBirthday()
            }
            
            // Pass the managed object down to a child view
            ChildProfileEditor(profile: profile)
        }
    }
}

3. Passing @Observable Objects to Child Views

When you want a sub-view to read or modify your @Observable object, pass it as a regular property. You do not need a property wrapper on the child view to read it.

Code Blueprint

struct ChildProfileEditor: View {
    // 1. Just a standard, plain property! No @ObservedObject or @Binding needed to read.
    var profile: UserProfile

    var body: some View {
        VStack {
            Text("Editing: \(profile.name)")
            
            // 2. CRITICAL: If you need a two-way binding for UI controls (like Textfields),
            // use the @Bindable wrapper locally right inside the body block.
            @Bindable var bindableProfile = profile
            TextField("Change Name", text: $bindableProfile.name)
                .textFieldStyle(.roundedBorder)
        }
        .padding()
        .background(Color(.secondarySystemBackground))
    }
}

3 Pillars of Modern Observation to Remember

  • Precision Updates: Unlike the older ObservableObject system, which forced the entire view hierarchy to redraw whenever any property changed, @Observable is precise. If ProfileView reads profile.name but doesn't read profile.status, a change to profile.status will not trigger a redraw of that view.
  • Implicit Binding Creation: If you don't need a text field or toggle to mutate a property, just read the property normally (e.g., Text(profile.name)). Only use @Bindable when interacting with input controls.
  • Memory Safety: Always use @State when instantiating your class inside a view (@State private var profile = UserProfile()). This ensures SwiftUI creates the object exactly once and doesn't accidentally destroy and recreate it when the parent view updates.
