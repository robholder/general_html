Challenge 1: iExpense with NavigationLink

This is a classic SwiftUI refactor. You are moving from a Modal presentation (a sheet that slides up over everything) to a Push presentation (a new screen that slides in from the right within the navigation stack).

To achieve this, you need to make three specific changes to your ContentView.

1. Remove the Sheet State

Since we are no longer "showing" or "hiding" a modal, you can delete the @State variable that controls the sheet.

  • Delete this line:

@State private var showingAddExpense = false

2. Swap the Toolbar Button for a NavigationLink

In your existing code, the "Add Expense" button manually sets a Boolean to true. We want to replace that Button with a NavigationLink.

  • Change your .toolbar block to this: 
.toolbar {
    NavigationLink {
        AddView(expenses: expenses)
    } label: {
        Label("Add Expense", systemImage: "plus")
    }
}

3. Clean up the .sheet Modifier

Since the NavigationLink now handles the transition, you must remove the .sheet modifier entirely.

  • Delete this block: 
.sheet(isPresented: $showingAddExpense) {
    AddView(expenses: expenses)
}

4. Update the AddView (The Hint)

Inside your AddView.swift Paul's hint becomes important. When a view is inside a NavigationStack, it automatically gets a "Back" button. Paul suggests hiding it to force the user to use your "Save" or "Cancel" logic.

In your AddView, add this modifier to the top-most view (usually a Form or VStack): 

.navigationBarBackButtonHidden(true)

Why do this?

If the user just hits the default "Back" button, the data they typed might be lost or the app might not behave as you intended. By hiding it, you ensure they either hit "Save" (which adds the expense and calls dismiss()) or "Cancel" (which just calls dismiss()).

Summary of the "Push" Workflow

  • Trigger: User taps NavigationLink.
  • Transition: AddView slides in from the right.
  • Completion: Your "Save" button in AddView calls dismiss(), which pops the view off the stack and returns the user to the list.

import SwiftUI

struct AddView: View {
    // MARK: - ENVIRONMENT & STATE
    
    // @Environment: Accesses system-wide values.
    // Here, we grab the "dismiss" action, which allows the view to close itself.
    @Environment(\.dismiss) var dismiss
    
    // @State: Local "Single Source of Truth" for the form fields.
    // These hold the data temporarily until the user hits "Save."
    @State private var name = ""
    @State private var type = "Personal"
    @State private var amount = 0.0
    
    // Add NavigationStack title variable for user editing
    //@State private var expenseTitle: String
    //@State private var expenseItem =  "Expense item"
    
    // Data source for our Picker.
    let types = ["Business", "Personal"]
    
    // MARK: - EXTERNAL DATA
    
    // This receives the 'Expenses' class instance from ContentView.
    // Because it is a class (Reference Type), adding an item here
    // updates the same list seen in ContentView automatically.
    var expenses: Expenses
    
    // MARK: - USER INTERFACE
    var body: some View {
        NavigationStack {
            // Form: A specialized container that automatically styles rows
            // to look like standard iOS settings or input screens.
            Form {
                // $name: The '$' creates a "Two-Way Binding."
                // 1. The TextField shows what's in the variable.
                // 2. Typing in the field updates the variable instantly.
                TextField("Expense item", text: $name)
                
                // Picker: Linking the 'selection' to our 'type' state.
                Picker("Type", selection: $type) {
                    // \.self: Tells SwiftUI to use the String itself as the unique ID.
                    ForEach(types, id: \.self) {
                        Text($0) // '$0' is shorthand for "the current string in the loop"
                    }
                }
                
                // TextField for numbers: Uses 'value' and 'format' instead of 'text'.
                TextField("Amount", value: $amount, format: .currency(code: "USD"))
                    .keyboardType(.decimalPad) // Forces the numeric keypad with a decimal point
            }
            .navigationTitle("Add new expense")
            
            // MARK: - ACTIONS
            // Set up ToolbarItems - Cancel & Save
            .toolbar {
                ToolbarItem(placement: .topBarTrailing) {
                    Button("Save") {
                        // 1. Package the local @State into a new ExpenseItem struct.
                        let item = ExpenseItem(name: name, type: type, amount: amount)
                        
                        // 2. Append the new item to the shared Expenses class.
                        expenses.items.append(item)
                        
                        // 3. Dismiss the view.
                        // This triggers the 'isPresented' binding in ContentView to become false.
                        dismiss()
                    }
                }
                ToolbarItem(placement: .topBarLeading){
                    Button("Cancel") {
                        dismiss()
                    }
                    
                }
            }
            // HIDE BACK BUTTON
            .navigationBarBackButtonHidden(true)
        }
    }
}

ContentView.swift

import SwiftUI

// MARK: - DATA MODEL
// Structs are "Value Types." Use them to define the shape of your data.
struct ExpenseItem: Identifiable, Codable {
    // Identifiable: Lets the List know which item is which using a unique ID.
    // Codable: Allows this data to be converted to JSON for saving (encoding/decoding).
    
    var id = UUID() // Generates a unique "Universally Unique Identifier."
    let name: String
    let type: String
    let amount: Double
}

// MARK: - DATA STORAGE (THE "BRAIN")
// @Observable: A macro that tells SwiftUI to watch this class for changes.
// Classes are "Reference Types." Use them to share data across different views.
@Observable
class Expenses {
    // The "Source of Truth" array.
    var items = [ExpenseItem]() {
        // didSet is a "Property Observer." It runs every time the array changes.
        didSet {
            // Convert the array into JSON data (encoding).
            if let encoded = try? JSONEncoder().encode(items) {
                // Save that JSON to the phone's permanent storage (UserDefaults).
                UserDefaults.standard.set(encoded, forKey: "Items")
            }
        }
    }
    
    // The initializer runs when the app first creates this object.
    init() {
        // Check if there is data saved under the key "Items."
        if let savedItems = UserDefaults.standard.data(forKey: "Items") {
            // Try to convert the JSON data back into an array of ExpenseItems (decoding).
            if let decodedItems = try? JSONDecoder().decode([ExpenseItem].self, from: savedItems) {
                items = decodedItems
                return
            }
        }
    }
}

// MARK: - COMPONENT VIEW (A REUSABLE ROW)
// Extracting logic into a sub-view keeps the main layout clean.
struct ExpenseRow: View {
    let item: ExpenseItem
    
    // Computed Property: Logic that returns a specific type (Color).
    var amountColor: Color {
        switch item.amount {
        case ..<10: return .primary      // Standard text color
        case ..<100: return .green      // Mid-range
        default: return .red            // High expense
        }
    }
    
    var body: some View {
        HStack {
            VStack(alignment: .leading) {
                Text(item.name)
                    .font(.headline)
                Text(item.type)
                    .font(.subheadline)
                    .foregroundStyle(.secondary) // Faded grey look
            }
            
            Spacer() // Pushes the following content to the far right
            
            // Format the Double as a currency based on the user's phone settings.
            Text(item.amount, format: .currency(code: Locale.current.currency?.identifier ?? "USD"))
                .foregroundStyle(amountColor) // Applies our custom switch-case color
        }
    }
}

// MARK: - MAIN INTERFACE
struct ContentView: View {
    // @State: Tells SwiftUI that this view "owns" this data.
    // If 'expenses' changes, the view will re-draw.
    @State private var expenses = Expenses()
    
    // CONVERT TO NAVIGATIONLINK - DELETE THIS SHEET BASED VAR:
    // Controls whether the AddView sheet is currently visible.
    //@State private var showingAddExpense = false
    
    // Filtered Computed Properties:
    // These create "virtual lists" based on the master 'items' array.
    var personalExpenses: [ExpenseItem] {
        expenses.items.filter { $0.type == "Personal" }
    }
    var businessExpenses: [ExpenseItem] {
        expenses.items.filter { $0.type == "Business" }
    }
    
    var body: some View {
        // NavigationStack: Provides the header area and manages view transitions.
        NavigationStack {
            // List: Provides the scrolling, grouped look and swipe actions.
            List {
                // Section: Groups rows with a header.
                Section("Personal") {
                    ForEach(personalExpenses) { item in
                        ExpenseRow(item: item)
                    }
                    .onDelete { offsets in
                        // Pass the offsets and the specific filtered list.
                        removeItems(at: offsets, in: personalExpenses)
                    }
                }
                
                Section("Business") {
                    ForEach(businessExpenses) { item in
                        ExpenseRow(item: item)
                    }
                    .onDelete { offsets in
                        removeItems(at: offsets, in: businessExpenses)
                    }
                }
            }
            .navigationTitle("iExpense") // Sets the text in the NavigationStack
            .toolbar {
                NavigationLink {
                    AddView(expenses: expenses)
                } label: {
                    Label("Add Expense", systemImage: "plus")
                }
            }
        }
    }
    
    // FUNCTION: Handles item deletion across sections.
    func removeItems(at offsets: IndexSet, in source: [ExpenseItem]) {
        for offset in offsets {
            let item = source[offset]
            // We find the ID of the filtered item within the MASTER array.
            if let index = expenses.items.firstIndex(where: { $0.id == item.id }) {
                expenses.items.remove(at: index)
            }
        }
    }
}

// MARK: - PREVIEW
#Preview {
    ContentView()
}

Challenge 2: User Editable NavigationStack Title

AI Insights

QUESTION

I added a @State private var to this view, but didn't assign an initial value. The view errored. If I assign a string, it's OK. Why is this?

  • This is the unassigned version version that errors:
    • @State private var expenseItem: String
  • This is the assigned version that seems OK:
    • @State private var expenseItem = "Expense Item"

ANSWER

The short answer is: Swift is a "safety-first" language. Every property in a struct or class must have a valid value by the time the object is finished being created.

When you use @State, you are telling SwiftUI to manage some storage for you. If you don't provide an initial value, Swift doesn't know what should be in that storage when the view first appears.

The "Missing Initializer" Error

When you write @State private var expenseItem: String, you haven't given it a value. Because ContentView is a struct, Swift automatically tries to create a "memberwise initializer."

It essentially thinks you want to do this:

  • let view = ContentView(expenseItem: "Milk")

However, in SwiftUI, we almost never want to pass values into @State from the outside (that's what @Bindable or regular let properties are for). Because @State is marked private, the rest of your app can't see it to give it a value, and since you didn't give it one, the compiler gets stuck.

Three Ways to Fix It

1. The Default Value (Most Common)

This is what you did. It tells SwiftUI: "Start with this string." 

@State private var expenseItem = ""

Note: Usually, for a text field, you'll want an empty string "" rather than placeholder text.

2. The Explicit Type (If you don't want to infer)

If you want to be very specific about the type but still give it a starting value: 

@State private var expenseItem: String = ""

3. The Optional (Rare for @State)

If you truly want it to start with nothing, you make it an "Optional" using a question mark. 

@State private var expenseItem: String?

Warning: If you do this, you’ll have to unwrap it later, which usually makes SwiftUI code much messier.

Why does this matter for iExpense?

In your AddView, you are likely using TextField("Name", text: $expenseItem).

The TextField requires a binding to a string. It needs to know that even before the user types a single letter, there is a valid string (even an empty one) to look at. If expenseItem was unassigned, the TextField would be trying to display "nothingness," which would cause the app to crash or fail to compile.

Pro-Tip: Always initialize your @State variables with a sensible "blank" value (like "", 0.0, or false). It keeps the compiler happy and your views predictable!

QUESTION

I want a NavigationStack title to be user editable in this case.

ANSWER

Ah, an editable title! That’s a great way to make an app feel more personal. In SwiftUI, the "magic" trick is that .navigationTitle() can actually take a Binding to a string instead of just a static string.

When you pass a binding (using the $ sign), SwiftUI automatically turns the title into an editable text field when the user taps on it.

The Code Implementation

Here is how you adjust your ContentView to make the "iExpense" title something the user can change: 

struct ContentView: View {
    @State private var expenses = Expenses()
    
    // 1. Create a @State property to hold the title text
    @State private var accountName = "iExpense"

    var body: some View {
        NavigationStack {
            List {
                // ... your existing Section and ForEach code ...
            }
            // 2. Pass a BINDING to the navigationTitle
            .navigationTitle($accountName) 
            
            // 3. Optional: Set a display mode
            .navigationBarTitleDisplayMode(.inline) 
            
            .toolbar {
                NavigationLink {
                    AddView(expenses: expenses)
                } label: {
                    Label("Add Expense", systemImage: "plus")
                }
            }
        }
    }
}

How it works in the UI

When you use .navigationTitle($accountName):

  • Standard View: It looks like a normal title.
  • Interaction: When the user taps the title, it transforms into a text entry field.
  • Persistence: Any change the user types is instantly saved back into your accountName variable.

A Quick "Day 48" Catch

Since you are currently learning about Persistence (saving to UserDefaults), keep in mind that if you change the title to "Bob's Budget" and then close the app, it will reset to "iExpense" unless you also save that accountName string to UserDefaults.

Why the error happened earlier

This connects perfectly to your previous question about unassigned variables. If you tried to do:

  • @State private var accountName: String

...and then used it in .navigationTitle($accountName), the app wouldn't know what to show in the header the moment it loaded. By assigning = "iExpense", you give the UI its "starting frame.

Making The Navigationtitle Sit Below The Button Row

QUESTION

I have this view struct, and right now, the cancel and save buttons are on the top, but the 'name' field (Expense item) is in between the buttons. I'd like this editable inline navigationtitle to be below them, and clearly an editable field. Is this possible?

ANSWER

Human input! This doesn't work. Only '.inline' is able to be user edited.

I see what's happening! By using .navigationBarTitleDisplayMode(.inline), you are forcing the title to squeeze into that narrow space between the "Cancel" and "Save" buttons. Because that space is so tight, it doesn't look like a clear editable field - it just looks like a cramped label.

To get the look you want-where the editable title is large, prominent, and sits below the buttons - you need to change the Display Mode.

The Solution: Use .large Display Mode

Switch your display mode from .inline to .large. In iOS, "Large" titles sit in the space below the navigation bar but above the Form, providing plenty of room for the user to tap and edit. 

// Change this line:
.navigationBarTitleDisplayMode(.large)

Why this works:

  • Placement: .large moves the title out of the "button row" and gives it its own dedicated line.
  • Visual Cue: Because the text is much bigger, it is much more obvious to the user that they can tap it to change the name of the expense.
  • Standard iOS Design: This follows the look of apps like Contacts or Reminders, where the name of the entry is the most prominent thing on the screen.

Important: Watch out for "Double Stacks"

In your code snippet, you have a NavigationStack inside your AddView: 

var body: some View {
    NavigationStack { // <--- This might be your issue
        Form { ... }
    }
}

Since you are navigating from ContentView using a NavigationLink, you are already inside a NavigationStack.

If you put a second NavigationStack inside AddView, you are "nesting" them. This often causes the navigation bar to look glitchy, titles to disappear, or buttons to behave strangely.

Try this cleanup:

  • Remove the NavigationStack { and its closing } from AddView.
  • Keep the Form and all the modifiers (.navigationTitle, .toolbar, etc.) attached to the Form.
  • The AddView will "inherit" the navigation environment from ContentView, and your .navigationTitle($name) will work much more reliably!

One Final Touch

If you want to make it really clear that the title is editable, some developers like to add a prompt. While navigationTitle doesn't have a built-in placeholder, you can initialize your state with a more "instructional" string: 

@State private var name = "New Expense"

Challenge 3: Moonshot

This challenge marks a major shift from the "old" way of navigating in SwiftUI to the modern, data-driven NavigationStack.

In the original Moonshot project, you likely used the "simple" NavigationLink:

  • NavigationLink(destination: MissionView(mission: mission)) { ... }

Paul wants you to switch to the decoupled version. You don't have to convert every single view in your app, but you do need to convert the main path: ContentView → MissionView.

What is the "NavigationLink(value:)" System?

Instead of telling the link exactly which View to go to, you tell the link: "Here is a piece of data. If the user taps this, just remember this data." Then, elsewhere in the code, you tell the NavigationStack: "If you ever receive a Mission object, show a MissionView."

Step 1: Conforming to Hashable

To use value:, the data you pass must be Hashable so SwiftUI can uniquely identify it in the navigation history.

  • Go to Mission.swift and Astronaut.swift.
  • Update the struct definitions: struct Mission: Codable, Identifiable, Hashable.
    • Make sure any structs inside Mission and Astronaut structs are also updated with Hashable.
    • Typical error if not is: Type 'Mission' does not conform to protocol 'Equatable'
  • (Note: Since these structs only contain basic types like Strings and Ints, Swift will provide the conformance automatically once you add the word.)

Step 2: Updating the Links

In your GridLayout or ListLayout, find where you created the NavigationLink.

Old way:

NavigationLink(destination: MissionView(mission: mission, astronauts: astronauts)) {
    // Label code...
}

New way:

NavigationLink(value: mission) {
    // Label code...
}

Step 3: Using navigationDestination()

This is where the magic happens. In ContentView, inside the NavigationStack (usually attached to the Group or List), you add a "listener": 

.navigationDestination(for: Mission.self) { mission in
    MissionView(mission: mission, astronauts: astronauts)
}

Why is Paul making you do this?

It feels like more work for the same result, right? But there are three huge benefits to this "Modern" approach:

  • Deep Linking: It makes it much easier to programmatically jump to a specific mission (e.g., from a search result or a notification).
  • Performance: MissionView is now only created the moment the user taps the link. In the old way, SwiftUI sometimes initialized the destination views before they were even needed.
  • Clean Code: Your GridLayout no longer needs to know how to build a MissionView. It just says "I have a mission," and the NavigationStack handles the rest.

Do I need to do this for Astronauts too?

Yes! If you have a link inside MissionView that goes to AstronautView, you should:

  • Make Astronaut conform to Hashable.
  • Change the link to NavigationLink(value: astronaut).
  • Add a second .navigationDestination(for: Astronaut.self) to your NavigationStack.

Astronauts

The actual syntax for a nested struct like CrewMember is where the rubber meets the road.

Here is how you adapt your MissionView.

Step 1: The "Small Print" (Hashable CrewMember)

Because you are looping through crew and want to pass crewMember.astronaut into the link, the Astronaut struct must be Hashable. But there's a tiny detail: if you ever wanted to pass the whole CrewMember struct instead, that would need to be Hashable too.

For now, since we are just passing the astronaut property, just ensure your Astronaut struct in Astronaut.swift has : Hashable.

Step 2: Update the NavigationLink

You’re going to swap your "View-based" link for a "Value-based" link.

Find this section in your code:

ForEach(crew, id: \.role) { crewMember in
    // OLD WAY
    // NavigationLink {
    //    AstronautView(astronaut: crewMember.astronaut)
    // } label: {
    //    CrewView(crewMember: crewMember)
    // }

    // NEW WAY
    NavigationLink(value: crewMember.astronaut) {
        CrewView(crewMember: crewMember)
    }
}

Step 3: Ensure the "Kitchen" is Ready

Even though you are inside MissionView, this link will send an Astronaut object up to the NavigationStack in your ContentView.

Make sure your ContentView.swift has both listeners attached to the stack: 

.navigationDestination(for: Mission.self) { mission in
    MissionView(mission: mission, astronauts: astronauts)
}
.navigationDestination(for: Astronaut.self) { astronaut in
    AstronautView(astronaut: astronaut)
}

Why this is actually cleaner

Notice something cool? In your init, you are doing a lot of work to map the mission's crew names to actual Astronaut objects.

In the "Old" way, you were doing all that work, then also having to tell the NavigationLink exactly how to build an AstronautView.

In the "New" way, MissionView doesn't even need to know AstronautView exists. It just says: "Here is the astronaut data I already spent time finding in my init. You handle the screen change, Boss (NavigationStack)."

One detail to watch: id: \.role

In your ForEach, you use id: \.role. This works fine as long as no mission has two people with the exact same role title. Since you've already made Astronaut conform to Hashable, you could also use the astronaut itself as the identifier if you wanted:

ForEach(crew, id: \.astronaut.id) { ... }

If you run this code right now, nothing will happen when you tap a row. It will highlight, but it won't navigate.

This is because ListLayout is now just shouting: "Hey, someone clicked Mission 11!" into the void. You need a listener.

The "Listener" in ContentView

Go back to your ContentView.swift and add the .navigationDestination modifier to your NavigationStack (or the Group inside it). 

NavigationStack {
    Group {
        if showingGrid {
            GridLayout(astronauts: astronauts, missions: missions)
        } else {
            ListLayout(astronauts: astronauts, missions: missions)
        }
    }
    // ADD THIS HERE:
    .navigationDestination(for: Mission.self) { mission in
        MissionView(mission: mission, astronauts: astronauts)
    }
    .navigationTitle("Moonshot")
    // ... other modifiers
}

Why this is a win for your project:

  • The "Astronaut" Problem: In your original ListLayout, you had to pass the astronauts dictionary into the view just so you could pass it again into the MissionView.
  • Decoupling: Now, ListLayout doesn't need to know that MissionView requires an astronauts dictionary. It just passes the mission, and ContentView (the boss) decides how to build the next screen.

The "Value" System Mental Model

To help with the challenge logic, think of the new navigation system like a Restaurant Order:

  • The Link (NavigationLink(value:)): This is the Waiter. They don't cook the food (the View); they just write down the order (the Data/Mission) and take it to the kitchen.
  • The Stack (.navigationDestination): This is the Kitchen. It stays in one place and waits for orders to arrive. When it sees an order for "Mission," it knows exactly which recipe (MissionView) to cook.

SwiftUI Modern Navigation Guide

The "Value" System (NavigationStack)

The goal is to separate the Trigger (tapping a link) from the Destination (creating the new view).

1. Data Preparation

All data passed through a NavigationLink must be Hashable.

  • Add : Hashable to your structs (e.g., Mission, Astronaut, CrewRole).
  • Swift handles the logic automatically if the internal properties are also Hashable.

2. The Trigger (Link)

In your List or Grid layouts, remove destination views from the link. 

// Modern Style
NavigationLink(value: mission) {
    Text(mission.displayName)
}

3. The Destination (Stack)

Attach the "listener" to the NavigationStack in your main view. 

NavigationStack {
    Group {
        // Your List or Grid
    }
    .navigationDestination(for: Mission.self) { mission in
        MissionView(mission: mission, astronauts: astronauts)
    }
}
