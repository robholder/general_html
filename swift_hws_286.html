Hacking With Swift - 286 - Building a primary list of items

Today we’re going to implement the first half of our program, which means we’ll get a list of ski resorts, a detail view to show more information, and a NavigationSplitView that can show them side by side.

That by itself shouldn’t present a problem for you, but along the way you’ll also learn about loading static example data from the bundle, controlling how NavigationSplitView should show primary and secondary views on iPhone, and even how to format lists of strings more neatly.

Although you already know so much of what is needed to make this code work, we’re still here on Day 97 introducing new things to learn. I hope you’re not discouraged by that - learning is an important skill, and in programming as well as many other industries you can find yourself constantly trying new things throughout your whole career.

"I am always doing that which I cannot do, in order that I may learn how to do it."

Building A Primary List Of Items

In this app we’re going to display two views side by side, just like Apple’s Mail and Notes apps.

In SwiftUI this is done by placing two views into a NavigationSplitView, then using a NavigationLink in the primary view to control what’s visible in the secondary view.

Primary View

So, we’re going to start off our project by building the primary view for our app, which will show:

  • a list of all ski resorts
  • which country they are from
  • how many ski runs it has
  • how many pistes you can ski down (sometimes called "trails" or just "slopes")

I’ve provided some assets for this project in the GitHub repository for this book, so if you haven’t already downloaded them please do so now.

You should drag resorts.json into your project navigator, then copy all the pictures into your asset catalog.

You might notice that I’ve included 2x and 3x images for the countries, but only 2x pictures for the resorts. This is intentional: those flags are going to be used for both retina and Super Retina devices, but the resort pictures are designed to fill all the space on an iPad Pro - they are more than big enough for a Super Retina iPhone even at 2x resolution.

List Layout

To get our list up and running quickly, we need to define a simple Resort struct that can be loaded from our JSON. That means it needs to conform to Codable, but to make it easier to use in SwiftUI we’ll also make it conform to both Hashable and Identifiable.

The actual data itself is mostly just strings and integers, but there’s also a string array called facilities that describe what else there is on the resort - I should add that this data is mostly fictional, so don’t try to use it in a real app!

Create a new Swift file called Resort.swift, then give it this code: 

import SwiftUI

struct Resort: Codable, Hashable, Identifiable {
    var id: String
    var name: String
    var country: String
    var description: String
    var imageCredit: String
    var price: Int
    var size: Int
    var snowDepth: Int
    var elevation: Int
    var runs: Int
    var facilities: [String]
}

Loading JSON Arrays

As usual, it’s a good idea to add an example value to your model so that it’s easier to show working data in your designs. This time, though, there are quite a few fields to work with and it’s helpful if they have real data, so I don’t really want to create one by hand.

Instead, we’re going to load an array of resorts from JSON stored in our app bundle, which means we can re-use the same code we wrote for project 8 - the Bundle-Decodable.swift extension. If you still have yours, you can drop it into your new project, but if not then create a new Swift file called Bundle-Decodable.swift and give it this code:

NOTE: See commented version further below. 
extension Bundle {
    func decode<T: Decodable>(_ file: String) -> T {
        guard let url = self.url(forResource: file, withExtension: nil) else {
            fatalError("Failed to locate \(file) in bundle.")
        }

        guard let data = try? Data(contentsOf: url) else {
            fatalError("Failed to load \(file) from bundle.")
        }

        let decoder = JSONDecoder()

        do {
            return try decoder.decode(T.self, from: data)
        } catch DecodingError.keyNotFound(let key, let context) {
            fatalError("Failed to decode \(file) from bundle due to missing key '\(key.stringValue)' - \(context.debugDescription)")
        } catch DecodingError.typeMismatch(_, let context) {
            fatalError("Failed to decode \(file) from bundle due to type mismatch - \(context.debugDescription)")
        } catch DecodingError.valueNotFound(let type, let context) {
            fatalError("Failed to decode \(file) from bundle due to missing \(type) value - \(context.debugDescription)")
        } catch DecodingError.dataCorrupted(_) {
            fatalError("Failed to decode \(file) from bundle because it appears to be invalid JSON.")
        } catch {
            fatalError("Failed to decode \(file) from bundle: \(error.localizedDescription)")
        }
    }
}

Making Example Data

With that in place, we can add some properties to Resort to store our example data, and there are two options here. The first option is to add two static properties: one to load all resorts into an array, and one to store the first item in that array, like this: 

static let allResorts: [Resort] = Bundle.main.decode("resorts.json")
static let example = allResorts[0]

The second is to collapse all that down to a single line of code. This requires a little bit of gentle typecasting because our decode() extension method needs to know what type of data it’s decoding: 

static let example = (Bundle.main.decode("resorts.json") as [Resort])[0]

Of the two, I prefer the first option because it’s simpler and has a little more use if we wanted to show random examples rather than the same one again and again.

In case you were curious, when we use static let for properties, Swift automatically makes them lazy - they don’t get created until they are used. This means when we try to read Resort.example Swift will be forced to create Resort.allResorts first, then send back the first item in that array for Resort.example. This means we can always be sure the two properties will be run in the correct order - there’s no chance of example going missing because allResorts wasn’t called yet.

Loading the Data to the App

Now that our simple Resort struct is done, we can also use that same Bundle extension to add a property to ContentView that loads all our resorts into a single array: 

let resorts: [Resort] = Bundle.main.decode("resorts.json")

Formatting the List Layout

For the body of our view, we’re going to use a NavigationSplitView with a List inside it, showing all our resorts. In each row we’re going to show:

  • A 40x25 flag of which country the resort is in.
  • The name of the resort.
  • How many runs it has.

40x25 is smaller than our flag source image, and also a different aspect ratio, but we can fix that by using resizable(), scaledToFill(), and a custom frame. To make it look a little better on the screen, we’ll use a custom clip shape and a stroked overlay.

When the row is tapped we’re going to push to a detail view showing more information about the resort, but we haven’t built that yet so instead we’ll just push to a temporary text view as a placeholder.

Replace your current body property with this: 

NavigationSplitView {
    List(resorts) { resort in
        NavigationLink(value: resort) {
            HStack {
                Image(resort.country)
                    .resizable()
                    .scaledToFill()
                    .frame(width: 40, height: 25)
                    .clipShape(
                        .rect(cornerRadius: 5)
                    )
                    .overlay(
                        RoundedRectangle(cornerRadius: 5)
                            .stroke(.black, lineWidth: 1)
                    )

                VStack(alignment: .leading) {
                    Text(resort.name)
                        .font(.headline)
                    Text("\(resort.runs) runs")
                        .foregroundStyle(.secondary)
                }
            }
        }
    }
    .navigationTitle("Resorts")
} detail: {
    Text("Detail")
}

Go ahead and run the app now and you should see it looks good enough - make sure you try it in both portrait and landscape, on both iPhone and iPad.

Ai Insight

An Explanation Of The Bundle Extension

Here is the Bundle extension, fully annotated with line-by-line comments. Following the code, we’ll deeply break down how the generic parameters and protocol requirements work so you have a solid conceptual reference. 

import Foundation

// We extend 'Bundle' so we can call this helper function anywhere in the app 
// using 'Bundle.main.decode("filename.json")'.
extension Bundle {
    
    // --- THE FUNCTION SIGNATURE ---
    // '<T: Decodable>' is a GENERIC PARAMETER with a PROTOCOL REQUIREMENT.
    // It means: "This function can return ANY type of data you want (T), 
    // as long as that type conforms to the Decodable protocol."
    // It accepts a file name string and returns that type 'T'.
    // The underscore (_) is called an argument label omission. It tells the
    // compiler: "When someone calls this function, they don't need to type out
    // the parameter name before passing the string."
    func decode<T: Decodable>(_ file: String) -> T {
        
        // --- STEP 1: LOCATE THE FILE ---
        // 'self.url(forResource:...)' looks inside your app's compiled folder (the bundle) 
        // to find where the physical file lives on the device disk.
        guard let url = self.url(forResource: file, withExtension: nil) else {
            // If the file spelling is wrong or missing from the target, crash explicitly
            // with a helpful error message instead of letting the app fail mysteriously later.
            fatalError("Failed to locate \(file) in bundle.")
        }
        
        // --- STEP 2: LOAD THE RAW DATA ---
        // 'Data(contentsOf: url)' attempts to read the raw 0s and 1s bytes from that file path.
        // 'try?' means "return nil if this throws an error" (e.g., if the file is locked or unreadable).
        guard let data = try? Data(contentsOf: url) else {
            fatalError("Failed to load \(file) from bundle.")
        }
        
        // --- STEP 3: INITIALIZE THE DECODER ---
        // Create an instance of JSONDecoder, Swift's built-in engine for turning JSON into Swift objects.
        let decoder = JSONDecoder()
        
        // --- STEP 4: ATTEMPT DECODING WITH EXPLICIT ERROR CATCHING ---
        // We use a do-catch block to specifically triage EXACTLY why a JSON file might fail to decode.
        do {
            // 'T.self' tells the decoder: "Parse this data into whatever Swift type (T) 
            // the calling code asked for."
            return try decoder.decode(T.self, from: data)
            
        } catch DecodingError.keyNotFound(let key, let context) {
            // Triggers if your Swift model expects a property, but it's missing entirely from the JSON text.
            fatalError("Failed to decode \(file) from bundle due to missing key '\(key.stringValue)' - \(context.debugDescription)")
            
        } catch DecodingError.typeMismatch(_, let context) {
            // Triggers if the JSON has a String (e.g., "age": "twenty"), but your Swift struct expects an Int.
            fatalError("Failed to decode \(file) from bundle due to type mismatch - \(context.debugDescription)")
            
        } catch DecodingError.valueNotFound(let type, let context) {
            // Triggers if a key exists but holds a 'null' value, and your Swift model marked it as non-optional.
            fatalError("Failed to decode \(file) from bundle due to missing \(type) value - \(context.debugDescription)")
            
        } catch DecodingError.dataCorrupted(_) {
            // Triggers if the JSON is malformed (e.g., a missing comma or broken curly brace).
            fatalError("Failed to decode \(file) from bundle because it appears to be invalid JSON.")
            
        } catch {
            // A fallback catch for any other unforeseen system or filesystem errors.
            fatalError("Failed to decode \(file) from bundle: \(error.localizedDescription)")
        }
    }
}

Concept Breakdown

1. What is the Generic Parameter ?

The T is a placeholder name for Type. It acts as a blank variable for a data type.

Without generics, if you had a JSON file full of User objects and another file full of Product objects, you would have to write two separate functions:

  • func decodeUsers(_ file: String) -> [User]
  • func decodeProducts(_ file: String) -> [Product]

By using <T>, you tell Swift: "I don't know what kind of data structure is in this JSON file yet. Figure out what type the developer wants based on how they assign the result."

Swift infers what T is automatically when you use it: 

// Swift looks at the variable type definition and automatically sets T = [User]
let users: [User] = Bundle.main.decode("users.json")

// Swift looks at this variable and automatically sets T = ConfigurationSettings
let settings: ConfigurationSettings = Bundle.main.decode("settings.json")

2. What is the Protocol Requirement : Decodable?

If you just wrote func decode<T>(_ file: String), the compiler would stop you with an error.

Why? Because inside the function, you call: 

try decoder.decode(T.self, from: data)

Swift's JSONDecoder can't just parse data into anything. It doesn't know how to turn JSON data into a custom UIKit Image or a CoreMedia Video stream. It only knows how to parse data into types that have explicitly opted-in to being decoded.

By adding the Protocol Constraint : Decodable, you are drawing a line in the sand. You are telling Swift: "The developer can use this function for any data type they want, BUT that type must conform to the Decodable (or Codable) protocol."

If someone tries to use your extension on a custom class or struct that isn't Decodable, Xcode will safely reject it at compile time rather than letting the app crash in production.
