chmod

Here is a concise summary of common chmod commands using the numeric (octal) mode to set useful file permission states.

Permissions State (Numeric Code) Symbolic Representation Who Can Do What Use Case
644 -rw-r--r-- Owner: Read/Write. Group/Others: Read only. Standard for public files (e.g., website files) that only the owner should edit.
600 -rw------- Owner: Read/Write. Group/Others: No access. Highly private files (e.g., SSH private keys, local config files).
755 -rwxr-xr-x Owner: Read/Write/Execute. Group/Others: Read/Execute. Standard for executable files (scripts, programs) and directories on a web server.
700 -rwx------ Owner: Read/Write/Execute. Group/Others: No access. Highly private directories or scripts that should only run for the owner.
400 -r-------- Owner: Read only. Group/Others: No access. Read-only configuration or credential files.
664 -rw-rw-r-- Owner/Group: Read/Write. Others: Read only. Shared documents where group members must also be able to edit.

Concise Instructions for Setting Permissions

To set the permissions, use the chmod command followed by the numeric code and the filename (or directory name).

For Standard Files (No Execution Needed)

To allow the owner to read and write, and everyone else to just read: 

chmod 644 filename

For Private Files

To allow only the owner to read and write: 

chmod 600 filename

For Scripts or Directories (Execution/Access Needed)

To allow the owner full control, and everyone else to read and execute (crucial for accessing a directory's contents): 

chmod 755 script_or_directory

For Unchanging (Immutable) Protection

As a supplemental step for critical files, use chattr to prevent even the root user from deleting the file. This is a file attribute, not a permission: 

sudo chattr +i filename

(The file must be mutable for the command to succeed.)

System Admin Impact (or Will it cause problems?)

If you set a file to read-only (e.g., using chmod 444), system administrators can generally still delete it as part of their day-to-day duties. You won't cause problems with their regular cleanup or admin tasks.

Here's why:

  • Permissions to Delete: The ability to delete a file is governed by the Write (w) and Execute (x) permissions on the directory that contains the file, not the file itself. By making the file read-only (444), you only prevent modifications to its content.
  • The Root Override: Since system administrators operate as the root user (or with sudo), they have the power to override standard file permissions. Even if the containing directory somehow restricted deletion, a sysadmin could simply:
    • Temporarily change the directory permissions.
    • Change the file permissions (e.g., from 444 back to 644).
    • Delete the file.

In short, setting a file to read-only using chmod provides a basic level of protection against accidental modification by non-root users, but it is not a barrier to deletion by a system administrator. This is standard and expected behavior in Linux/Unix environments.
