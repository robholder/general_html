chmod

Here is a concise summary of common chmod commands using the numeric (octal) mode to set useful file permission states.

Permissions State (Numeric Code) Symbolic Representation Who Can Do What Use Case
644 -rw-r--r-- Owner: Read/Write. Group/Others: Read only. Standard for public files (e.g., website files) that only the owner should edit.
600 -rw------- Owner: Read/Write. Group/Others: No access. Highly private files (e.g., SSH private keys, local config files).
755 -rwxr-xr-x Owner: Read/Write/Execute. Group/Others: Read/Execute. Standard for executable files (scripts, programs) and directories on a web server.
700 -rwx------ Owner: Read/Write/Execute. Group/Others: No access. Highly private directories or scripts that should only run for the owner.
400 -r-------- Owner: Read only. Group/Others: No access. Read-only configuration or credential files.
664 -rw-rw-r-- Owner/Group: Read/Write. Others: Read only. Shared documents where group members must also be able to edit.

Concise Instructions for Setting Permissions

To set the permissions, use the chmod command followed by the numeric code and the filename (or directory name).

For Standard Files (No Execution Needed)

To allow the owner to read and write, and everyone else to just read: 

chmod 644 filename

For Private Files

To allow only the owner to read and write: 

chmod 600 filename

For Scripts or Directories (Execution/Access Needed)

To allow the owner full control, and everyone else to read and execute (crucial for accessing a directory's contents): 

chmod 755 script_or_directory

For Unchanging (Immutable) Protection

As a supplemental step for critical files, use chattr to prevent even the root user from deleting the file. This is a file attribute, not a permission: 

sudo chattr +i filename

(The file must be mutable for the command to succeed.)

System Admin Impact (or Will it cause problems?)

If you set a file to read-only (e.g., using chmod 444), system administrators can generally still delete it as part of their day-to-day duties. You won't cause problems with their regular cleanup or admin tasks.

Here's why:

  • Permissions to Delete: The ability to delete a file is governed by the Write (w) and Execute (x) permissions on the directory that contains the file, not the file itself. By making the file read-only (444), you only prevent modifications to its content.
  • The Root Override: Since system administrators operate as the root user (or with sudo), they have the power to override standard file permissions. Even if the containing directory somehow restricted deletion, a sysadmin could simply:
    • Temporarily change the directory permissions.
    • Change the file permissions (e.g., from 444 back to 644).
    • Delete the file.

In short, setting a file to read-only using chmod provides a basic level of protection against accidental modification by non-root users, but it is not a barrier to deletion by a system administrator. This is standard and expected behavior in Linux/Unix environments.

Understanding the Chmod Numbers

Understanding how these numbers are derived is key to mastering file permissions in Unix-like systems.

Octal Permission Derivation (The Numbers)

File permissions are set for three distinct categories of users: User (the owner), Group (users belonging to the file's group), and Other (everyone else).

The three-digit octal number (like 755 or 644) is the shorthand for setting these permissions. Each digit corresponds to one user category, and that digit is a sum derived from a three-bit binary system:

Permission Binary Value Decimal Value
read 100 4
write 010 2
execute 001 1
No Permission 000 0

To determine the final digit for a category, you add the decimal values for the permissions you want to grant.

Example: chmod 755

Digit Position → First Digit Second Digit Third Digit
Category User (Owner) Group Other
Octal Digit 7 5 5
Breakdown 4+2+1 4+1 4+1
Permissions rwx r-x r-x
Meaning Owner can read, write, and execute. Group can read and execute. Others can read and execute.

Summary of Common Octal Codes

Octal Code Permissions Symbolic Equivalent Common Use Case
777 rwxrwxrwx a=rwx World-writable (rarely recommended)
755 rwxr-xr-x u=rwx,go=rx Scripts, programs, and directories
700 rwx------ u=rwx,go= Private, executable script/directory
664 rw-rw-r-- ug=rw,o=r Data files for collaborators
644 rw-r--r-- u=rw,go=r Standard data files (owner writes, others read)
600 rw------- u=rw,go= Private configuration or data files
400 r-------- u=r,go= Private, read-only data file

Symbolic Permission Syntax (The Alternatives)

Instead of using octal numbers, you can use symbolic mode to modify permissions. This is often more intuitive when you only want to change one specific permission without affecting the others (e.g., adding execution without touching read/write).

The symbolic syntax follows the structure: u/g/o/a → +/-/= → r/w/x

A. User Categories

Symbol Category Description
u User The file owner
g Group Users belonging to the file's group
o Other All other users
a All All three categories (u, g, and o). This is the default if no category is specified.

B. Action Operators

Operator Action Description
+ Add Adds the specified permissions to the existing ones.
- Remove Removes the specified permissions from the existing ones.
= Set Sets the exact permissions, overwriting any existing ones.

C. Permission Types

Symbol Permission
r Read
w Write
x Execute

Symbolic Syntax Examples

Symbolic Command Explanation Equivalent Octal
chmod +x filename Adds execute permission for all (a) categories. Varies (e.g., 644→755)
chmod u+w filename Adds write permission only for the user (owner). Varies (e.g., 444→644)
chmod go-r filename Removes read permission from the group and other. Varies (e.g., 644→600)
chmod u=rw,go=r filename Sets the owner to have read/write and the group/other to have read. 644
chmod a-wx,a+r filename Removes write/execute, then adds read for all. 444

Using the symbolic mode like chmod +x is particularly useful for making a script executable without needing to remember its current octal code.
