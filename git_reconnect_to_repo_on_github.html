Git - Reconnecting a Repo to a Remote Repo

QUESTION

Here's a Git question.

  • I have disengaged a local git repo from the origin on GitHub.
  • I then created a new blank/empty repo on GitHub (I used SSH).
  • Now, I want to disengage the same local git repo from the obsolete 'new' repo on GitHub, and RECONNECT to the ORIGINAL repo on GitHub, but also keep current changes and create a new branch to keep any local changes separate.

How do I do this?

ANSWER

To reconnect your local repository to the original GitHub repository while preserving your current changes in a new branch, follow these steps:

1. Update the Remote URL

First, "disengage" from the obsolete repository and point back to the original one using the git remote set-url command: 

git remote set-url origin git@github.com:USERNAME/ORIGINAL-REPO.git

Verify the change by running git remote -v to ensure the URLs now point to the original repository.

Why didn't we do: git remote remove origin ...before: git remote set-url origin git@github.com:USERNAME/ORIGINAL-REPO.git

While git remote remove origin followed by git remote add origin achieves the same end goal, git remote set-url is generally preferred because it is cleaner and safer for your existing workflow.

Here is why we skipped the "remove" step:

  • Preserves Configuration: git remote remove is destructive. It deletes all remote-tracking branches (like origin/main or origin/dev) and any specialized configuration settings you might have manually added for that remote in your .git/config.
  • Maintains Continuity: git remote set-url simply swaps the "address" while keeping the "house" intact. Your local branches will still know they are supposed to track a remote named origin, even if the destination of origin has moved.
  • Efficiency: It’s a single atomic command rather than two. If you use remove and then forget to add it back immediately, your repository is left in a "headless" state where it has no connection to GitHub at all.
  • Prevents Errors: If you try to run git remote add origin when an origin already exists, Git will throw a fatal error. Using set-url avoids this collision entirely.

In short, set-url is the "surgical" way to change a repo's location, while remove/add is the "demolish and rebuild" way.

2. Create and Switch to a New Branch

To keep your local changes separate from the existing work in the original repo, create a new branch and switch to it: 

git checkout -b your-new-branch-name
# or:
git switch -c your-new-branch-name

All your current uncommitted or local-only changes will now reside on this new branch.

3. Fetch Original History

Since the local repo was disengaged, your local knowledge of the original repo's state might be outdated. Update your local references without merging them yet: 

git fetch origin

4. Commit and Push Changes

Stage and commit your local changes to the new branch, then push it to the original repository on GitHub: 

git add .
git commit -m "Describe your current changes"
# either:
git push -u origin your-new-branch-name
# or same but more verbose:
git push --set-upstream orignin your-new-branch-name

The -u flag links your local branch to the new remote branch, making future pushes simpler.

5. Handle Diverged History (If Necessary)

If your local main branch has diverged significantly from the original origin/main, you may want to reset your local main to match the remote exactly after you have safely pushed your changes to the new branch: 

git checkout main
git reset --hard origin/main
Warning: Only do this after you are sure your changes are safe on your new branch, as this will overwrite your local main with the version from GitHub.

QUERY

Do you need help resolving merge conflicts if you eventually decide to merge this new branch back into your original main branch?

ANSWER

When you merge your new branch back into main, Git may find changes on the same lines of code that it cannot automatically resolve. Here is how to handle those conflicts:

1. Identify the Conflicts

When a merge fails, Git will tell you which files are "unmerged." You can see a list at any time by running: 

git status

Files with conflicts will be listed under "Unmerged paths".

2. Understand Conflict Markers

Open a conflicted file in any text editor. Git inserts special markers to show you exactly where the "clash" is occurring: 

<<<<<<< HEAD: # Start of the changes on your current branch (e.g., main).
=======: #The divider between the two versions.
>>>>>>> # your-branch-name: End of the changes from the branch you are merging in.

3. Resolve Manually

To resolve the conflict, you must edit the file to what it should look like.

  • Choose one side: Delete the markers and the code from the side you don't want.
  • Combine both: Edit the code to include parts of both versions.
  • Clean up: Ensure you delete all three marker lines (<<<<, ====, >>>>).

4. Finish the Merge

Once the file looks correct, tell Git you have resolved it: 

git add <filename>
git commit -m "Fixed merge conflicts"

If you are in the middle of a merge, simply running git commit without the -m flag will often open a pre-filled merge message for you to save.

Pro-Tip: Use a Visual Tool (Mac)

Since you are on macOS, you can use built-in or third-party tools to make this much easier:

  • VS Code: Provides "Accept Current," "Accept Incoming," or "Compare" buttons directly above the markers.
  • git mergetool: Run this in your terminal to launch a visual side-by-side editor like FileMerge (part of Xcode) or opendiff.
  • Dedicated Apps: High-quality Mac clients like Fork or Tower offer dedicated merge conflict wizards that visualize the "Our version" vs "Their version" side-by-side.

QUERY

Would you like a recommendation for a free visual merge tool that works well with your specific code editor?

ANSWER

Since you are already on macOS, there are several powerful free options. The "best" one depends on whether you prefer staying in your current code editor or using a standalone app.

1. Visual Studio Code (The Most Popular Choice)

If you already use VS Code, you don't need to install anything new. It has a built-in "3-Way Merge Editor" that is excellent for conflict resolution.

  • Setup: First, ensure the code command is in your path (Open VS Code, press Cmd+Shift+P, and run Shell Command: Install 'code' command in PATH).
  • Configure Git: Run these commands in your terminal to set it as your default: 
    git config --global merge.tool vscode
git config --global mergetool.vscode.cmd 'code --wait --merge $REMOTE $LOCAL $BASE $MERGED'

2. Meld (Best Open-Source Standalone)

Meld is a favorite in the open-source community because of its simple, three-pane layout that clearly shows "Mine," "Theirs," and the "Merged" result.

  • Installation: The easiest way to install it on a Mac is via Homebrew: 
brew install --cask meld
  • Setup: Tell Git to use it: 
git config --global merge.tool meld

3. P4Merge (Best for Complex Conflicts)

P4Merge (from Perforce) is widely considered the "gold standard" for three-way merging because of how it handles complex code moves and signature changes that confuse other tools.

  • Cost: It is completely free even if you don't use the rest of the Perforce suite.
  • Visuals: It uses unique icons and color-coding to show exactly where each line of code originated.

4. FileMerge (The "No-Install" Option)

This is already on your Mac if you have Xcode installed. While its interface is more basic, it is stable and requires zero setup other than running git mergetool.

Summary Recommendation:

  • Use VS Code if you want to stay in your editor and keep your workflow unified.
  • Use Meld if you want a dedicated, lightweight visual tool.
  • Use P4Merge if you are dealing with a massive, "nightmare" merge conflict.

QUERY

Which code editor do you currently use as your primary driver? I can give you the specific configuration lines for it.

ANSWER

For a Vim-centric workflow on macOS, you have two primary paths: using the built-in vimdiff (no plugins required) or the popular vim-fugitive plugin.

1. The Built-in Way: vimdiff

This is the standard method that requires zero installation. It opens a 4-panel layout: LOCAL (your changes), BASE (common ancestor), REMOTE (their changes), and MERGED (the file you edit).

Configuration:

Run these commands in your terminal to set it up: 

git config --global merge.tool vimdiff
git config --global merge.conflictstyle diff3
git config --global mergetool.prompt false
  • diff3 is highly recommended because it shows the "BASE" version, which helps you understand why a conflict occurred.

How to use it:

  • Run git mergetool.
  • Navigate to the MERGED pane at the bottom (usually Ctrl-w then j).
  • Use these commands while your cursor is on a conflict:
    • :diffget LO — Pull from your local branch.
    • :diffget RE — Pull from the remote branch.
    • :diffget BA — Pull from the base (ancestor).
  • Type :wqa to save all and exit once all conflicts are resolved.

2. The Plugin Way: vim-fugitive

If you prefer a 3-way vertical split (LOCAL | MERGED | REMOTE), vim-fugitive is the gold standard.

Configuration:

Add this to your .gitconfig: 

[merge]
    tool = fugitive
[mergetool "fugitive"]
    cmd = vim -f -c "Gdiffsplit!" "$MERGED"

How to use it:

  • Open the conflicted file in Vim and run :Gdiffsplit!.
  • You will see three vertical panes.
  • In the center (MERGED) pane, use these shortcuts:
    • d2o — Diff 2 Obtain (Get changes from the left pane/LOCAL).
    • d3o — Diff 3 Obtain (Get changes from the right pane/REMOTE).
  • Jump between conflicts using ]c (next) and [c (previous).

Pro-Tip for MacVim Users

If you use the GUI version (MacVim), you may need to ensure it's in your $PATH to use mvimdiff as your tool. You can check this by running which mvim in your terminal.

QUERY

Would you like the exact mapping lines for your .vimrc to make jumping between these conflicts even faster?

ANSWER

To make conflict resolution lightning-fast in Vim, add these mappings to your .vimrc. They focus on jumping between conflicts and "grabbing" the version you want without typing long commands.

1. Navigation & Basic "Gets"

These keys allow you to hop through conflicts and pull code from the left (Local) or right (Remote) panes while staying in the center (Merged) file. 

" Navigation: Jump to next/previous conflict
nnoremap <leader>cn ]c
nnoremap <leader>cp [c

" Resolution: 'Obtain' from specific panes (Works in 3-way split)
" 'gh' = Get Here (Local/Left)
" 'gl' = Get Light (Remote/Right)
nnoremap <leader>gh :diffget //2<CR>
nnoremap <leader>gl :diffget //3<CR>

2. The "Fugitive" Power Flow

If you use vim-fugitive (highly recommended for Git users), these mappings are even more intuitive: 

" Quick-open the 3-way merge view
nnoremap <leader>gs :Gvdiffsplit!<CR>

" Toggle the fold (close/open) of non-conflicted code to focus
nnoremap <leader>zf :setlocal foldmethod=diff<CR>

3. Essential "Panic" Mappings

Sometimes you make a mess while merging. These help you exit or refresh quickly: 

" Save all and exit (Tells Git you're done with this file)
nnoremap <leader>gw :wqa<CR>

" Update the diff highlights if they get stuck
nnoremap <leader>du :diffupdate<CR>

How to use this workflow:

  • Open the file with git mergetool or :Gvdiffsplit!.
  • Jump to the first mess with \cn.
  • Decide:
    • Want your local changes? Hit \gh.
    • Want their remote changes? Hit \gl.
  • Save: Hit \gw to move to the next file.

A quick check: Do you currently have a leader key defined in your .vimrc (usually let mapleader = "," or " ")? If not, these shortcuts will default to starting with the backslash \ key.
