T0DO - Convention and Syntax

NOTE: A lot of examples are show in 'C' or 'Java', but can be adapted to Python.

There isn't one "official" standard enforced across all projects, but there are widely accepted conventions and best practices that many teams and IDEs follow when using TODO comments. Here’s what you need to know:

  • Basic Syntax—Use Uppercase and a Consistent Format: The most common approach is as simple as writing a comment like: 
// TODO: Refactor this function to handle edge cases

Using uppercase TODO (often followed by a colon) is a de facto standard. Many IDEs and code editors highlight these comments automatically and even allow you to navigate between them easily2.

  • Additional Context Is Helpful: While the basic TODO tag is enough for many cases, it can be beneficial to be a bit more descriptive. For example, you might include:
    • A brief explanation of what needs to be done.
    • An identifier or initials of the person responsible (especially in a team setting).
    • A date or reference to an issue tracker. For instance: 
// TODO (J.Doe, 2025-06-01): Replace the temporary algorithm with the one from the optimization ticket #1234.
  • Distinguish Between Different Types of Work: In addition to TODO, you might see comments like FIXME (for things that are known to be broken and require fixing) or XXX (to mark areas of potential hazard or instability). This differentiation helps signal to developers not only that something needs to be done but also what kind of attention it requires.
  • Integrate with Your Workflow: Many development environments (like IntelliJ IDEA or VSCode) come with built-in support for listing and managing TODOs. This makes it easier to track outstanding work directly from your editor, but it also means that you might want to stick to a consistent format that your whole team agrees on. Some teams even have style guidelines stating how and when to use TODOs so they don’t accumulate into an unmanageable backlog.
  • Temporary vs. Permanent: Remember, TODO comments are a temporary measure. They’re great for noting work that needs to be done later, but relying on them as your only tracking method can lead to "hidden" technical debt. Ideally, they should either be linked to an issue in your project’s tracking system or be resolved within a reasonable time frame.

In summary, while you’re usually just "slapping the word TODO" into a comment, there is a de facto standard:

  • Use uppercase TODO (with an optional colon)
  • Add relevant, concise context when possible
  • Use additional tags (like FIXME or XXX) for different kinds of tasks

Following this approach will not only make your code more self-documenting but also provide better integration with code editors and help your team manage pending tasks more effectively.

Examples in Use

Below are a few examples and best practices for using TODO comments in your code. While there isn’t a single “official” standard, many teams follow similar patterns that both developers and IDEs recognize.

Basic Usage

A simple TODO comment is usually marked in all uppercase with a colon. For example, in C/C++ or Java: 

// TODO: Refactor this method to handle edge cases.
public void processData() {
    // ...
}

Or in Python: 

def fetch_data(url):
    # TODO: Handle network exceptions properly.
    response = requests.get(url)
    return response.json()

This standard notation lets developers quickly search—and many IDEs automatically index—TODO comments.

Enhanced Troubleshooting and Traceability

It’s often very helpful to add more context. For example, if you’re working on a team, include your initials or name along with a date, or reference an issue number. For example, in Java: 

// TODO (JD, 2025-06-01, #1234): Replace the temporary algorithm with the optimized version.
public int calculateScore(int[] data) {
    // temporary implementation
    return 0;
}

In Python, you could do something similar: 

def process_file(filename):
    # TODO (Jane, 2025-06-01): Add proper error handling for file not found.
    with open(filename) as file:
        return file.read()

Adding these details ensures that when you—or someone else—come back later, you’ll have some background on why the task exists and who is responsible for resolving it.

Using Preprocessor Directives

Some languages allow you to use preprocessor directives to surface warnings during compilation. For example, in C# you can write: 

#warning TODO: Optimize the sorting algorithm before release.
public void SortData() {
    // sort implementation
}

This method forces the compiler to emit a warning, making it less likely that the comment (and the associated work) is forgotten.

Integration with IDEs

Many IDEs, such as Visual Studio, Eclipse, or VS Code, can automatically detect these TODO comments and list them as tasks:

  • Visual Studio: The Task List pane will automatically show lines with // TODO: or other designated tokens. This lets you navigate directly to the code that needs attention.
  • VS Code: Extensions such as “Todo Tree” can scan your project for TODO comments and display them in a navigable tree within the UI.
  • Eclipse: Eclipse can be configured to recognize custom tags in comments (like TODO or FIXME) and list them in a dedicated view.

These features help maintain a dynamic to-do list that’s directly linked to your source code, ensuring that technical debt or work-in-progress items are visible.

Other Related Tags

In addition to TODO, you might come across:

  • FIXME: Indicates something is broken and needs to be fixed.
  • HACK: Denotes a workaround or temporary solution that should eventually be replaced.
  • XXX: Often used to warn about a potential problem or something that is especially messy.

For example, in a JavaScript file: 

// FIXME: This function crashes when input is empty.
function processInput(input) {
    // ...
}

// HACK: Temporary fix until refactoring is done.

These tags serve different purposes, and using them consistently makes it easier for anyone reviewing your code to prioritize what needs to be done.

These examples should give you a good idea of how to annotate and manage TODO comments within your codebase. Many developers and teams adopt a similar approach to ensure that uncompleted tasks are both visible and trackable through integrated development tools.

Using Vim to Manage T0DOs

Vim can be set up to work with TODOs and help you manage them more efficiently. There are a couple of approaches you can take:

Using Vim’s Built-In Tools

Even without plugins, Vim has powerful search and navigation features that can help you work through TODO comments:

  • Syntax Highlighting: You can configure Vim to highlight TODO comments. For example, adding the following to your ~/.vimrc will highlight any occurrence of TODO in your files: 
autocmd Syntax * syn match Todo /\<TODO\>/
highlight link Todo WarningMsg

This makes TODOs stand out in your code.

  • Finding TODOs with Search: You can use Vim’s search command to jump between TODO comments. For example, type: 
/TODO

And then press n to jump to the next occurrence.

  • Quickfix List: To list all TODO comments in your project, you can use :vimgrep along with the quickfix window: 
:vimgrep /TODO/ **/*
:copen

This will search for TODOs across files (using Vim’s globbing) and open the quickfix window so you can navigate through matches.

Using Dedicated Plugins

If you’d like more robust management—like creating, checking off, and organizing tasks—there are several plugins designed for this:

vim-todo-lists

  • Overview: vim-todo-lists is a Vim plugin specifically designed to manage TODO lists. It supports a Markdown-like syntax for tasks and provides commands to create new items, navigate between items, and mark items as done.
  • Installation with vim-plug: Add the following to your ~/.vimrc: 
Plug 'aserebryakov/vim-todo-lists'

Then reload Vim and run :PlugInstall.

  • Usage Examples: Once installed, items marked with the default syntax (e.g., - [ ] Task description for pending tasks and - [X] Task description for completed ones) will be recognized by the plugin. It provides commands such as:
    • :VimTodoListsCreateNewItemAbove
    • :VimTodoListsCreateNewItemBelow
    • :VimTodoListsGoToNextItem
    • :VimTodoListsGoToPreviousItem

You can customize mappings in your configuration to jump between tasks or toggle their status.

Other Plugins

  • todo.txt-vim: If you work with a plain text TODO format (the popular Todo.txt standard), then todo.txt-vim might be a better match. It facilitates managing your tasks within a dedicated file following that format.
  • Task-Wiki or similar integrations: For more elaborate setups, plugins like taskwiki integrate task management into Vim by connecting with task tracking systems like Taskwarrior.

Best Practices and Integration

  • Consistent Commenting Style: Adopt a standard such as always using TODO: in your code comments. That way both your search tools and plugins can recognize these markers.

Mapping Keys: Create custom mappings in your Vim configuration to quickly navigate to the next TODO. For example: 

nnoremap <Leader>t /TODO<CR>

This maps <Leader>t to a search for TODO.

  • Use of Quickfix/Location List: Even if you’re not using a dedicated plugin, combining Vim’s search with the quickfix window can give you a task-like interface to navigate through TODOs.

In summary, Vim is very flexible in how you handle TODOs. Whether you prefer using plain search commands and custom mappings or fully featured plugins like vim-todo-lists or todo.txt-vim, you can tailor Vim to be a powerful tool for managing and navigating your tasks.
