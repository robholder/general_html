Hacking With Swift - 266 - Alignment and alignment guides

SwiftUI gives us a number of valuable ways of controlling the way views are aligned, and I want to walk you through each of them so you can see them in action.

Frame & Alignment

The simplest alignment option is to use the alignment parameter of a frame() modifier.

Remember, a text view always uses the exact width and height required to show its text, but when we place a frame around it that can be any size. As the parent doesn’t have a say in the final size of the child, code like this will create a 300 x 300 frame with a smaller text view centered inside it: 

Text("Live long and prosper")
    .frame(width: 300, height: 300)

If you don’t want the text to be centered, use the alignment parameter of the frame(). For example, this code places the view in the top-left corner when running on a left-to-right environment 

    .frame(width: 300, height: 300, alignment: .topLeading)

You can then use offset(x:y:) to move the text around inside that frame. 

import SwiftUI

struct ContentView: View {
    var body: some View {
        Text("Live long and prosper")
            .offset(x: 108, y: 121)
            .frame(width: 300, height: 300, alignment: .topLeading)
            //.offset(x: 108, y: 121) // Same result as placing above .frame
            .background(.red)
            //.offset(x: 108, y: 121) // Offsets background also!
    }
}

#Preview {
    ContentView()
}

Stacks & Alignment

The next option up is to use the alignment parameter of a stack. For example, here are four text views of varying sizes arranged in a HStack: 

HStack {
    Text("Live")
        .font(.caption)
    Text("long")
    Text("and")
        .font(.title)
    Text("prosper")
        .font(.largeTitle)
}

We haven’t specified an alignment there, so they will be centered by default. That doesn’t look great, so you might think to align them all to one edge to get a neater line, like this: 

HStack(alignment: .bottom) {

However, that also looks bad: because each of the text views has a different size, they also have a different baseline - that’s the name for where letters such as "abcde" sit on a line, which excludes letters that go below the line such as "gjpy". As a result, the bottom of the small text sits lower than the bottom of the bigger text.

Fortunately, SwiftUI has two special alignments that align text on the baseline of either the first child or the last child. This will cause all views in a stack to be aligned on a single unified baseline, regardless of their font: 

HStack(alignment: .lastTextBaseline) {

View Alignment

Moving on, for more fine-grained control we can customize what "alignment" means for each individual view. To get a really good idea of how this works we’re going to start with this code: 

struct ContentView: View {
    var body: some View {
        VStack(alignment: .leading) {
            Text("Hello, world!")
            Text("This is a longer line of text")
        }
        .background(.red)
        .frame(width: 400, height: 400)
        .background(.blue)
    }
}

When that runs you’ll see the VStack sits tightly around its two text views with a red background. The two text views have different lengths, but because we used the .leading alignment they will both be aligned to their left edge in a left-to-right environment. (You might have expected that the text would be against the left edge of the screen. It doesn't work like that. The alignment aligns to the left most edge of the longest text line.)

Outside of that there’s a larger frame that has a blue background. Because the frame is larger than the VStack (the text), the VStack is centered in the middle.

Now, when the VStack comes to aligning each of those text views, it asks them to provide their leading edge. By default this is obvious: it uses either the left or right edge of the view, depending on the system language. But what if we wanted to change that - what if we wanted to make one view have a custom alignment?

Alignment Guide

SwiftUI provides us with the alignmentGuide() modifier for just this purpose. This takes two parameters: the guide we want to change, and a closure that returns a new alignment. The closure is given a ViewDimensions object that contains the width and height of its view, along with the ability to read its various edges.

By default, the .leading alignment guide for a view is its leading alignment guide - I know that sounds obvious, but its effectively equivalent to this: 

VStack(alignment: .leading) {
    Text("Hello, world!")
        .alignmentGuide(.leading) { d in d[.leading] }
    Text("This is a longer line of text")
}

We could rewrite that alignment guide to use the view’s trailing edge for its leading alignment guide, like this: 

VStack(alignment: .leading) {
    Text("Hello, world!")
        .alignmentGuide(.leading) { d in d[.trailing] }
    Text("This is a longer line of text")
}

And now you’ll see why I added colors: the first text view will move to the left so that its right edge sits directly above the left edge of the view below, the VStack will expand to contain it, and the whole thing will still be centered within the blue frame.

This result is different from using the offset() modifier: if you offset a text, its original dimensions don’t actually change, even though the resulting view is rendered in a different location. If we had offset the first text view rather than changing its alignment guide, the VStack wouldn’t expand to contain it.

Although the alignment guide closure is passed your view’s dimensions, you don’t need to use them if you don’t want to - you can send back a hard-coded number, or create some other calculation. For example, this creates a tiered effect for 10 text views by multiplying their position by -10: 

var body: some View {
    VStack(alignment: .leading) {
        ForEach(0..<10) { position in
            Text("Number \(position)")
                .alignmentGuide(.leading) { _ in Double(position) * -10 }
        }
    }
    .background(.red)
    .frame(width: 400, height: 400)
    .background(.blue)
}

For complete control over your alignment guides you need to create a custom alignment guide. And I think that deserves a mini chapter all of its own...

Ai Insight

Reinterpretation

The Golden Rule of Alignment Guides

To understand alignment guides, you have to look at them as a contract between a Container (like a VStack) and its Child Views (like the Text).

The Rule: The parent container draws a single, invisible vertical line down the screen (the alignment line). Every child view must pick a point on itself to attach to that line.

If you tell a VStack(alignment: .leading) to build a stack, it creates a vertical alignment line. By default, every child view says: "I will attach my own .leading (left) edge to that line." As a result, all their left edges line up.

Deciphering Paul’s Examples

Example 1: The Default (Obvious) Behavior

VStack(alignment: .leading) {
    Text("Hello, world!")
        .alignmentGuide(.leading) { d in d[.leading] }
    Text("This is a longer line of text")
}
  • .alignmentGuide(.leading): The container is asking, "Where is your leading anchor?"
  • { d in d[.leading] }: The Text view looks at its own dimensions (d) and replies, "Use my actual .leading (left) edge."
  • Result: Normal, boring left alignment.

Example 2: The Mind-Bending Swap

VStack(alignment: .leading) {
    Text("Hello, world!")
        .alignmentGuide(.leading) { d in d[.trailing] }
    Text("This is a longer line of text")
}

This is where the magic happens. The VStack asks the Text view, "Where is your leading anchor so I can line it up with the others?" The Text view lies! It replies, "My leading anchor is actually sitting over here at my .trailing (right) edge!"

Because SwiftUI is forced to believe the child, it takes the right edge of "Hello, world!" and snaps it to the VStack’s left alignment line.

To make that happen visually, the "Hello, world!" text has to shift completely out to the left. It shifts so far out that its right edge now hovers directly over the left edge of the text below it.

Alignment Guide vs. Offset: The "Physical" Difference

Paul mentions that this is completely different from using .offset(x: -100). This distinction is crucial:

  • .offset() is a ghost/illusion: If you offset a view, its physical body stays exactly where it was in the VStack, but its visual image is pushed to the side. The VStack will not stretch to fit it, and it might clip off the edge of the screen.
  • .alignmentGuide() is real physical movement: Changing the guide actually moves the view's physical boundaries. The VStack looks at the newly shifted view, says "Wow, you're sticking way out to the left now," and widens its own background frame to safely encircle it.

The Hardcoded Numbers (The Tiered Effect)

In his final example, Paul uses this loop: 

.alignmentGuide(.leading) { _ in Double(position) * -10 }

In Context

var body: some View {
    VStack(alignment: .leading) {
        ForEach(0..<10) { position in
            Text("Number \(position)")
                .alignmentGuide(.leading) { _ in Double(position) * -10 }
        }
    }
    .background(.red)
    .frame(width: 400, height: 400)
    .background(.blue)
}

When you return a raw number from an alignment guide closure, you are telling SwiftUI: "Shift my alignment anchor X points away from my default 0 position."

  • When position is 0, the guide is 0. The first card sits normally on the line.
  • When position is 1, the guide is -10. The view shifts its anchor, causing the text to move right by 10 points.
  • When position is 2, the guide is -20. The text shifts right by 20 points.

Because every card shifts its anchor slightly differently relative to that one single VStack alignment line, it creates a beautiful, cascading staircase (or tiered) layout.

Note on Double

In Swift, you cannot multiply different types of numbers together.

  • The position variable comes from ForEach(0..<10). Because it represents a count or loop index, Swift infers its type as an Int (Integer).
  • The alignment guide closure, however, expects you to return a CGFloat or a Double (decimal numbers), because layouts in iOS require precise decimal values for crisp sub-pixel rendering.

If you tried to write: 

position * -10

...Swift would see an Int multiplied by an Int, resulting in an Int. The compiler would then throw an error saying:

"Cannot convert value of type 'Int' to expected argument type 'Double'."

By wrapping it in Double(position), you manually convert that whole number (0, 1, 2) into a decimal format (0.0, 1.0, 2.0), allowing Swift to multiply it cleanly and return the decimal type the layout engine expects.

Breaking Down the Math

Now let's look at what that returned value actually tells the VStack to do.

Remember: returning a custom value from an alignment guide overrides where the child view tells the parent its "anchor point" is. The math creates a unique anchor offset for each row:

Position (Int) Converted Double The Math (* -10) What SwiftUI Hears
0 0.0 0.0 * -10 = 0 "Line up my normal left edge exactly with the VStack line."
1 1.0 1.0 * -10 = -10 "My anchor is 10 points to the left of my normal edge."
2 2.0 2.0 * -10 = -20 "My anchor is 20 points to the left of my normal edge."

The Visual Result

Because SwiftUI must line up every single one of those calculated anchor points perfectly with the VStack's central invisible alignment line, it forces the physical text views to step outward:

  • Row 0 stays put.
  • Row 1 has to shift right by 10 points to get its "cheated" anchor point back onto the vertical alignment line.
  • Row 2 has to shift right by 20 points to do the same.

This cumulative shifting is what creates that perfect staircase or "tiered" cascade of text views across your red and blue frames!

QUESTION

I still don't quite understand where the guides are 'drawn' by the stacks. For example, if: 

VStack(alignment: .leading) {

I would expect the guide line to be vertical 'edge' down the far left of the stacks bounds, but I see that the Text within, while ranged left, is still centered on the stack's frame.

ANSWER

You have just stumbled onto the final boss of SwiftUI layouts: The difference between a Stack's internal alignment line and its external frame bounding box.

Your intuition is entirely logical - you would expect the alignment line to sit permanently at the far left edge of the stack. But in SwiftUI, it works in reverse: The child views determine where the alignment line is first, and then the stack builds its outer bounds around them.

Here is exactly how SwiftUI draws that line and why it still looks centered inside your frame.

1. Step One: Drawing the Invisible "Clothesline"

When you write VStack(alignment: .leading), SwiftUI doesn't care about the screen width or the parent frame yet. It only looks at the immediate children inside the stack.

  • It measures the width of every Text view inside the loop.
  • It looks at the longest line of text in the stack.
  • It draws its vertical alignment line right down the left edge of that longest text view.
  • It then snaps the left edges of all the other smaller text views to that exact same line.

At this point, the VStack shrinks tightly to wrap around the children. The left edge of the VStack and its internal alignment line are perfectly overlapping.

2. Step Two: The Frame Explosion (Why it Looks Centered)

The confusion happens when you add this modifier to the outside of the stack: 

.frame(width: 400, height: 400)

By default, a .frame() modifier with a fixed width acts like a giant layout box that forces whatever is inside it to align to its center.

So, this is the exact sequence of events on your screen:

  • The VStack creates a tight layout box based on your staircase text. Its internal alignment line is on its left edge.
  • The .frame(width: 400) modifier comes along and says, "I am forcing this entire VStack box to be placed right in the dead-center of my 400-point width."
  • The VStack moves to the center. Because the internal alignment line is welded to the inside of the VStack, it moves to the center along with it.

Summary

The alignment line is at the left edge of the VStack's actual content. However, because the entire VStack container is being centered inside a much larger 400 x 400 frame, that "left edge" ends up sitting near the middle of your blue screen!

If you wanted that alignment line to truly sit at the absolute far-left edge of the 400-point blue box, you would have to tell the frame to align its contents to the left, like this: 

.frame(width: 400, height: 400, alignment: .leading)
